## Summary

- **Solid:** The parser/runtime ownership boundary is crisply drawn and consistently enforced across all three contract documents. The "What This Contract Must Not Claim" sections are excellent defensive documentation.
- **Solid:** Completion semantics requiring progress evidence plus quiescence is well-justified and prevents the stale-idle false-positive class.
- **Solid:** Staged migration plan with rollback strategy is realistic.
- **Risky:** The RxPY adoption is a significant architectural commitment for a codebase that currently has zero reactive code; the dismissed imperative alternative deserves stronger justification.
- **Risky:** Transport fingerprint normalization is critically underspecified — false-positive change detection directly impacts latency, and there is no contract for the normalization algorithm.
- **Risky:** Gemini backend is absent from all contracts despite being a supported `shadow_only` backend in the codebase.
- **Risky:** Several subtle state-machine edge cases in the turn-monitor contract have gaps or ambiguities (detailed below).

## Findings

### Must Fix / Blocking

- **B1: Gemini backend is absent.** The codebase ships `gemini_headless` as a backend (`src/gig_agents/agents/brain_launch_runtime/backends/`), and the `ShadowParserStack` supports it. None of the contracts mention Gemini. If Gemini uses `shadow_only` parsing, it needs its own state contract at the same level as Claude and Codex. If it does not, the proposal and design must explicitly state that Gemini is out of scope and explain why. Silently omitting a supported provider creates an untested contract gap.

- **B2: `evt_projection_changed` naming collision across contract layers.** In `claude-state-contracts.md` and `codex-state-contracts.md`, `evt_projection_changed` is defined as a *parser transition event* (projected dialog text changes between successive snapshots). In `turn-monitor-contracts.md`, `evt_projection_changed` is defined as a *runtime event* meaning "post-submit projected dialog differs from the pre-submit baseline or later observation." These are semantically different things — one is snapshot-to-snapshot delta, the other is submit-baseline comparison. Using the same name for both creates implementer confusion. Recommend renaming the runtime variant to `evt_projection_diverged_from_baseline` or similar.

- **B3: `blocked_waiting_user` → `completed` transition is unsafe without justification.** The turn-monitor transition graph allows `Block --> Done: evt_ready_after_submit and progress_seen and evt_completion_quiet_elapsed`. This means: a user-answer prompt appeared, then the surface returned to `ready_for_input` with progress evidence, and the output went quiet — so the turn completes. But what caused the user prompt to disappear? Either the agent answered its own question (the runtime never intervened), or the prompt timed out silently. The contract must specify whether this is an intentional happy-path (agent self-resolution) or should instead emit an anomaly/warning. Without clarification, silent completion after an unanswered user prompt is a correctness risk.

### Should Fix / Important

- **S1: Transport fingerprint normalization needs a contract.** The design says "normalized ANSI-stripped text" and mentions "avoid treating ANSI-only repaint noise as meaningful activity," but provides no specification for the normalization algorithm. Critical open questions: Is whitespace normalized? Are cursor-positioning escape sequences stripped? Are tmux status-line artifacts excluded? Is the comparison hash-based or string-equality? Given that false-positive transport changes directly extend readiness/completion latency by restarting quiet windows, the normalization algorithm is a first-order correctness concern. At minimum, the turn-monitor contract should define what "normalized" means with concrete examples of what is and is not a transport change.

- **S2: Quiet-window preconditions vs. restart semantics have a subtle gap.** The turn-monitor contract defines `evt_ready_quiet_elapsed` as "no new transport change for `ready_quiet_window_seconds` while `accepts_input=true`." The quiescence contract section says "any `evt_transport_changed` before a quiet-window threshold is reached restarts the active quiet countdown." But: what happens when the surface leaves `accepts_input=true` during an active readiness quiet countdown (e.g., activity transitions to `working`)? Is the countdown cancelled, paused, or does it continue running (impossible to fire because the precondition fails)? The contract should explicitly state that a readiness quiet countdown is only active while `accepts_input=true` holds, and that losing `accepts_input` cancels/resets it.

- **S3: Polling interval quantization is unacknowledged.** With a 0.4s polling interval, the system can only observe quiescence at poll boundaries. A `ready_quiet_window_seconds` of 0.5s means the actual observed quiet period is between 0.5s and 0.9s. The contracts should acknowledge this quantization and either (a) specify that quiet-window values represent minimum observed quiet time, or (b) define how the implementation should compensate. This also constrains the choice of default quiet-window values — values smaller than the polling interval are meaningless.

- **S4: RxPY adoption justification is weak.** Design Decision 3 commits to RxPY as the lifecycle timing engine. The alternative ("preserve the existing imperative monitor and replace `time.monotonic()` comparisons with more timestamps") is dismissed because "it would keep transition logic scattered across loops and make restartable quiet-window tests harder." But the current `_TurnMonitor` is ~200 lines of focused, readable code. Adding restartable timers imperatively (a `last_transport_change_at` field + comparison in `observe_*`) is straightforward. The reactive approach introduces:
  - A new programming paradigm in an otherwise entirely imperative codebase
  - Debugging difficulty (reactive stack traces are notoriously opaque)
  - A learning curve for every contributor touching shadow lifecycle
  - `reactivex` is a dependency today but has zero usage — this would be the inaugural adoption

  The mitigation ("keep the poll source imperative, isolate RxPY to quiescence/timer composition") is vague. Recommend either: (a) strengthening the justification with a concrete side-by-side complexity comparison showing why imperative is intractable, or (b) reconsidering a phased approach where the initial implementation is imperative (proving the quiescence semantics first) and RxPY is adopted in a follow-up once the semantics are validated.

- **S5: Claude and Codex contracts are >90% structural duplication.** The two provider contracts differ only in: naming (`trust_prompt` vs `approval_prompt`), parser class names, and minor dialog-projection wording. This duplication means a contract amendment must be applied twice, creating drift risk. Consider factoring out a shared `provider-state-contract-base.md` with provider-specific addenda, or at minimum cross-referencing the shared sections and calling out which sections are provider-specific.

- **S6: The spec bundles orthogonal concerns.** The scenario "Historical slash-command output does not block recovered shadow readiness" and "Active slash-command surface remains non-ready" are about slash-command/model-switch recovery semantics. These are real concerns but are not quiescence-specific — they describe how `ui_context` and `accepts_input` interact with readiness. Bundling them into the quiescence spec makes the change scope harder to review and test. Consider whether these should be a separate requirement or at least clearly tagged as pre-existing behavior clarification vs. new quiescence behavior.

### Nice To Have

- **N1: Default quiet-window values are not proposed.** The design says "positive built-in defaults" but never suggests concrete values. The open question acknowledges "separate vs shared defaults" but doesn't propose a starting point. Having concrete proposed defaults (even marked as provisional) would make the design more reviewable and would anchor test expectations. Consider adding e.g. `ready_quiet_window_seconds=1.0`, `completion_quiet_window_seconds=1.5` as initial proposals based on the 0.4s polling interval.

- **N2: The design open question about caller-facing `mode_diagnostics` should be resolved before implementation.** If diagnostics are log-only, the implementation is simpler. If they are caller-facing, the payload shape must be designed now.

- **N3: Tasks 4.1 and 4.2 overlap.** Task 4.1 ("runtime unit tests for readiness settling...") and Task 4.2 ("reactive-timing tests using TestScheduler...") are likely the same tests. If the implementation is reactive, the settling tests will use TestScheduler. If it's imperative, Task 4.2 is moot. Consider merging or making 4.2 conditional on RxPY adoption.

## Design Alignment With Existing System

- **Existing pattern observed:** `_TurnMonitor` in `cao_rest.py` (lines ~301-494) is a ~200-line imperative state machine using `time.monotonic()` timestamps, boolean flags (`m_saw_working_after_submit`, `m_saw_projection_change_after_submit`), and direct field comparison. The polling loops in `_wait_for_shadow_ready_status()` and `_wait_for_shadow_completion()` are simple while-sleep-poll loops.

- **Existing pattern observed:** `_ShadowStallPolicy` (lines ~271-276) is a frozen dataclass resolved from `launch_plan.metadata`. The proposed `ShadowTimingPolicy` extension follows this pattern well.

- **Existing pattern observed:** Anomalies use `ShadowParserAnomaly` dataclasses with code/message/details — the proposed diagnostic surface for quiescence timing fits naturally.

- **Alignment concern:** The RxPY proposal (Design Decision 3) is a paradigm mismatch. No existing module in `src/gig_agents/` uses reactive programming. The `scan`, `debounce`, `switch_latest`, `merge` operator composition style is foreign to the codebase's imperative + dataclass + Pydantic conventions. The split into "stream/observation module" + "reducer-style monitor" (Decision 5) is a good decomposition regardless of whether RxPY is used, but the reactive implementation would be the first of its kind in this repo.

- **Alignment positive:** The observation/change-signal dataclasses (Decision 1), the policy shape extension, and the reducer concept all align well with the existing frozen-dataclass and explicit-state patterns.

## Open Questions (With Proposed Defaults)

### Q1) Should the readiness quiet window be evaluated only while `accepts_input=true` continuously, or should it persist across brief `accepts_input` interruptions?

- Decision: `Blocking`

#### Why this matters

If the surface briefly flickers from `ready_for_input` → `unknown` → `ready_for_input` (a single poll cycle), should the quiet window reset from scratch? The answer determines whether noisy but fundamentally-ready surfaces experience pathological latency.

#### Examples

- Claude surface shows prompt, next poll shows spinner for one frame, next poll shows prompt again. With strict "continuous" semantics, the quiet window restarts twice. With "persistent" semantics, only transport changes restart it.
- Codex briefly shows an unknown surface during a repaint, then returns to prompt.

#### Why this question arises in code

- `_TurnMonitor.observe_readiness()` currently returns immediately on `accepts_input=true`. The new contract says readiness requires `accepts_input=true` AND quiet. But the interaction between losing `accepts_input` and the quiet countdown is not specified.
- The turn-monitor contract defines `evt_ready_quiet_elapsed` with the condition "while `accepts_input=true`" but the quiescence contract section only says "any `evt_transport_changed` ... restarts the active quiet countdown" without referencing the `accepts_input` precondition.

#### Options

**Option A: Strict continuous — quiet window only runs while `accepts_input=true`; any loss cancels and resets.**

- Code impact:
  - Quiet timer has a precondition gate that cancels on every non-ready observation
- User-facing difference:
  - More conservative; noisy surfaces take longer to become ready
- Pros:
  - Simplest mental model; impossible to submit during a non-ready flicker
- Cons:
  - Pathological latency if surface is noisy but fundamentally ready

**Option B: Transport-change-only restart — quiet window counts transport silence regardless of activity state, but submit still requires `accepts_input=true` at the moment of submission.**

- Code impact:
  - Quiet timer runs independently; submit gate checks `accepts_input` only at fire time
  - A transport change (which accompanies most activity transitions) restarts the timer anyway
- User-facing difference:
  - Slightly faster for noisy-but-ready surfaces
- Pros:
  - Transport change already restarts the timer, so activity flickers that don't change tmux content are naturally absorbed
  - Simpler timer logic (one restart trigger: transport change)
- Cons:
  - Edge case: `accepts_input` could be false when the timer fires; need a "wait for next ready poll" fallback

#### Proposal (Recommended)

Option B: transport-change-only restart with `accepts_input` gate at fire time. A transport change accompanies virtually all activity transitions (activity changes require visible TUI change), so the timer naturally restarts. The submit gate provides the safety net.

#### Pros / Cons (Proposal)

- Pros:
  - Single restart trigger simplifies implementation
  - Transport changes are the quiescence primitive anyway — activity flickers without transport change are benign
  - Avoids pathological latency from noisy-but-ready surfaces
- Cons:
  - Requires documenting the fire-time gate clearly to prevent submit during non-ready states

### Q2) How should the transport fingerprint be computed to avoid false-positive change detection from benign tmux repaints?

- Decision: `Blocking`

#### Why this matters

Every false-positive transport change restarts quiet windows, directly increasing latency. The normalization algorithm is the single most important tuning knob for real-world quiescence performance.

#### Examples

- Cursor blink on the prompt line: tmux output differs in a cursor-position escape sequence. Should this be a transport change?
- Clock or status-bar update in tmux status line: raw output includes status line. Should this be filtered?
- Trailing whitespace difference from terminal resize: same content, different wrapping.

#### Why this question arises in code

- `DialogProjection` already has `normalized_text` (ANSI-stripped). The proposed transport fingerprint is a separate concept over the full tmux output, not just the dialog portion.
- The current codebase uses `_ansi_strip()` in parser code but has no concept of "transport normalization" as a distinct layer.

#### Options

**Option A: ANSI-strip + whitespace-normalize the full tmux capture, compare as string.**

- Code impact:
  - New normalization function in the observation module
  - Full-text comparison (potentially large strings)
- User-facing difference:
  - Broad noise filtering; may miss real changes that only differ in whitespace layout
- Pros:
  - Simple implementation; catches cursor/ANSI noise
- Cons:
  - May over-normalize (e.g., indentation changes in output could be real)

**Option B: ANSI-strip only, hash-compare for efficiency, with optional exclusion zones (e.g., first/last N lines for status bars).**

- Code impact:
  - Hash computation per observation; configurable zone exclusion
- User-facing difference:
  - Efficient and tunable per-provider
- Pros:
  - Hash comparison is O(1) for inequality; exclusion zones handle status-bar noise
  - Can be tuned per-provider via tool-adapter config
- Cons:
  - Exclusion zone config adds complexity; wrong zones could mask real changes

#### Proposal (Recommended)

Option B with conservative defaults (ANSI-strip + hash, no exclusion zones initially). Add exclusion zone support as a follow-up if real-session telemetry shows status-bar churn is a problem.

#### Pros / Cons (Proposal)

- Pros:
  - Hash comparison is efficient for the hot polling path
  - ANSI-strip is already proven in the parser stack
  - No premature configuration complexity
- Cons:
  - May need follow-up tuning if status-bar noise causes latency in practice

### Q3) Should the initial implementation use RxPY or an imperative approach, given zero existing reactive code in the codebase?

- Decision: `Blocking`

#### Why this matters

This is the highest-risk architectural decision in the change. It determines the programming model for all future shadow lifecycle work and sets a precedent for the codebase.

#### Examples

- Restartable ready timer, imperative:
  ```python
  if transport_changed:
      last_change_at = now
  if accepts_input and (now - last_change_at) >= ready_quiet_window:
      return settled_ready
  ```
- Restartable ready timer, reactive:
  ```python
  observations.pipe(
      filter(lambda o: o.transport_changed),
      switch_map(lambda _: timer(ready_quiet_window, scheduler=scheduler)),
      filter(lambda _: latest_assessment.accepts_input),
  )
  ```

#### Why this question arises in code

- `cao_rest.py` lines ~301-494: `_TurnMonitor` is imperative, ~200 lines
- `pyproject.toml` line 21: `reactivex>=4.1.0,<5` is declared but unused
- Design Decision 3 commits to RxPY without demonstrating that the imperative approach is intractable

#### Options

**Option A: RxPY from the start (as proposed in design)**

- Code impact:
  - New reactive module, TestScheduler-based tests, reactive operator chain
  - First reactive code in the repo; all contributors must learn RxPY idioms
- User-facing difference:
  - None (implementation detail)
- Pros:
  - Clean timer composition; TestScheduler enables precise virtual-time tests
  - Natural fit for debounce/restart semantics
- Cons:
  - Paradigm mismatch with 100% of existing code
  - Opaque stack traces; harder to debug in production
  - High bus-factor risk if few contributors know RxPY

**Option B: Imperative first, RxPY later if justified**

- Code impact:
  - Extend `_TurnMonitor` with `last_transport_change_at` and quiet-window comparisons
  - Extract into observation/reducer modules (Decision 5) without reactive operators
  - Tests use `time.monotonic()` patching or a simple `FakeClock` injectable
- User-facing difference:
  - None
- Pros:
  - Consistent with codebase idioms; lower reviewer/contributor friction
  - Proves quiescence semantics before committing to reactive paradigm
  - Easier to debug
- Cons:
  - Manual timer management; slightly more boilerplate for restart logic
  - May need refactoring if reactive approach is later proven superior

#### Proposal (Recommended)

Option B: implement imperatively first, with the observation/reducer decomposition (Design Decision 5). Inject a clock abstraction (`Protocol` with `monotonic()`) so tests are deterministic. If the imperative timers prove unwieldy, adopt RxPY in a follow-up with real complexity evidence.

#### Pros / Cons (Proposal)

- Pros:
  - Zero paradigm risk; validates semantics before committing to reactive
  - The observation/reducer decomposition is valuable regardless
  - Injectable clock gives most of TestScheduler's benefits for this use case
- Cons:
  - Defers potential RxPY benefits; may require later refactor

### Q4) What are the concrete default values for `ready_quiet_window_seconds` and `completion_quiet_window_seconds`?

- Decision: `Deferrable`

#### Why this matters

Default values directly determine user-perceived latency. Too small and quiescence provides no protection. Too large and every turn is artificially delayed.

#### Examples

- Polling at 0.4s: a `ready_quiet_window_seconds` of 0.5s means 1-2 poll cycles of quiet. 1.0s means 2-3 cycles.
- Claude completion with streaming: output may trickle for 2-3s after the last substantive line. A `completion_quiet_window_seconds` of 1.0s might be too short; 2.0s adds noticeable latency.

#### Why this question arises in code

- The design says "positive built-in defaults" but proposes none
- The spec says "SHALL use positive built-in defaults selected by runtime" — no guidance on magnitude
- Real-session telemetry for Claude/Codex TUI settling times is not documented

#### Options

**Option A: Shared default (both 1.0s)**

- Pros: Simple; one fewer tuning parameter
- Cons: Completion may need longer (post-stream settling)

**Option B: Split defaults (ready=0.8s, completion=1.5s)**

- Pros: Accounts for completion needing longer settling
- Cons: More config surface; may still be wrong without telemetry

#### Proposal (Recommended)

Start with split defaults: `ready_quiet_window_seconds=0.8`, `completion_quiet_window_seconds=1.5`. These give 2-4 poll cycles of quiet, account for completion needing longer settling, and are easy to tune down once telemetry confirms TUI settling profiles.

#### Pros / Cons (Proposal)

- Pros:
  - Conservative but not punitive; provides meaningful debounce
  - Split allows independent tuning
- Cons:
  - Values are provisional; need validation against real Claude/Codex sessions
