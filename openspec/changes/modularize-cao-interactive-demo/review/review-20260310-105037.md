## Summary

- The six-module split is directionally solid and matches package/export patterns already used in `src/gig_agents/cao/__init__.py` and `src/gig_agents/agents/brain_launch_runtime/__init__.py`.
- The current backward-compatibility story is not implementable as written: a thin re-export shim will not preserve the legacy module-global monkeypatch behavior used by the demo tests.
- The new canonical package contract (`gig_agents.demo.cao_interactive_demo`) is underspecified in validation: the tasks only verify the legacy path, so the new API could regress without detection.
- The change should explicitly decide whether legacy compatibility means import compatibility only, or full legacy patch-target compatibility, because that choice materially changes the implementation shape.

## Findings

### Must Fix / Blocking

- The shim design in `openspec/changes/modularize-cao-interactive-demo/design.md:57-72` and `openspec/changes/modularize-cao-interactive-demo/specs/cao-interactive-demo-module-structure/spec.md:48-62` is not sufficient to preserve the current test behavior. Today, `main()` resolves `inspect_demo` from the monolith's own globals (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:368-433`), `_ensure_cao_server()` resolves `_loopback_port_is_listening` and `shutil.which` from the same module globals (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:1462-1499`), and `inspect_demo()` resolves `CaoRestClient` / `ClaudeCodeShadowParser` from module globals (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:687-739`, `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:2068-2103`). The existing tests patch those names on the legacy module path (`tests/unit/demo/test_cao_interactive_full_pipeline_demo.py:299-356`, `tests/unit/demo/test_cao_interactive_full_pipeline_demo.py:895-960`). After extraction, `cli.main`, `commands.inspect_demo`, and `cao_server._ensure_cao_server` will keep resolving globals in their defining submodules, not in the shim module. That means task `7.2` in `openspec/changes/modularize-cao-interactive-demo/tasks.md:28-29` cannot meet the spec with a pure re-export file; the change must either narrow the compatibility contract or adopt a different implementation strategy.

### Should Fix / Important

- The validation plan does not exercise most of the newly added requirements. The spec adds structural guarantees for exact module layout, module boundaries, acyclic imports, the canonical `gig_agents.demo.cao_interactive_demo` import path, and explicit `__all__` re-exports (`openspec/changes/modularize-cao-interactive-demo/specs/cao-interactive-demo-module-structure/spec.md:3-46`, `openspec/changes/modularize-cao-interactive-demo/specs/cao-interactive-demo-module-structure/spec.md:64-74`), but the tasks only call for `typecheck`, `lint`, existing tests through the legacy shim, and `python -m gig_agents.demo.cao_interactive_full_pipeline_demo --help` (`openspec/changes/modularize-cao-interactive-demo/tasks.md:31-36`). A repository-wide search found no non-OpenSpec callers of `gig_agents.demo.cao_interactive_demo`, so without targeted tests the canonical path can silently drift.
- The compatibility surface is under-inventoried. The design and tasks talk mainly about `shutil.which` and `_loopback_port_is_listening` (`openspec/changes/modularize-cao-interactive-demo/design.md:59-72`, `openspec/changes/modularize-cao-interactive-demo/tasks.md:28-35`), but the current unit suite patches at least eight legacy globals: `shutil`, `_loopback_port_is_listening`, `_run_timestamp_slug`, `_prompt_yes_no`, `_find_listening_pids_for_port`, `inspect_demo`, `CaoRestClient`, and `ClaudeCodeShadowParser` (`tests/unit/demo/test_cao_interactive_full_pipeline_demo.py:299-356`, `tests/unit/demo/test_cao_interactive_full_pipeline_demo.py:582-638`, `tests/unit/demo/test_cao_interactive_full_pipeline_demo.py:895-960`). If the change keeps manual `__all__` and shim exports, it needs an explicit inventory of which names remain supported on the legacy path and why.

### Nice To Have

- Add a short compatibility matrix to the design mapping each legacy name to its new owning module and whether the legacy path guarantees import-only compatibility or patch-target compatibility. That would make later maintenance much easier than relying on two hand-maintained export lists plus scattered test knowledge.

## Design Alignment With Existing System

- Existing pattern(s) observed: `src/gig_agents/cao/__init__.py:1-66` and `src/gig_agents/agents/brain_launch_runtime/__init__.py:1-69` already use explicit package-level re-exports with hand-maintained `__all__`, so introducing `src/gig_agents/demo/cao_interactive_demo/__init__.py` is consistent with the rest of the repository.
- Existing pattern(s) observed: tests elsewhere patch the owning implementation module directly, not a package shim or re-export layer. See `tests/unit/cao/test_server_launcher.py:257-280` patching `gig_agents.cao.server_launcher.shutil.which`, and `tests/unit/agents/brain_launch_runtime/test_headless_base.py:257-264` patching `gig_agents.agents.brain_launch_runtime.backends.headless_base.shutil.which`.
- Proposed design alignment: the concern-based split aligns well with those package patterns, but the "thin shim preserves test patch targets" idea does not. A more native approach is either to treat the new subpackage modules as the patching surface for tests, or to add explicit compatibility wrappers that intentionally look up dependencies through a shared compatibility layer instead of relying on simple re-exports.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What backward-compatibility contract does the legacy module path actually need to preserve?

- Decision: `Blocking`

#### Why this matters

- This decides whether the refactor can stay a pure extraction or must introduce wrapper logic / dependency indirection.
- The current design assumes "same import path" and "same patch targets" are equivalent, but the current demo tests depend on both.

#### Examples

- `tests/unit/demo/test_cao_interactive_full_pipeline_demo.py:299-356` patches `gig_agents.demo.cao_interactive_full_pipeline_demo.shutil.which`, `_loopback_port_is_listening`, and `_run_timestamp_slug`, then calls `start_demo()` / `main()`.
- `tests/unit/demo/test_cao_interactive_full_pipeline_demo.py:895-960` patches `gig_agents.demo.cao_interactive_full_pipeline_demo.inspect_demo`, `CaoRestClient`, and `ClaudeCodeShadowParser`, then calls `main()` / `inspect_demo()`.

#### Why this question arises in code

- `main()` directly calls `inspect_demo()` as a module global today (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:368-433`).
- `_ensure_cao_server()` directly reads `_loopback_port_is_listening` and `shutil.which` as module globals (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:1462-1499`).
- `inspect_demo()` and `_best_effort_output_text_tail()` directly read `CaoRestClient` and `ClaudeCodeShadowParser` as module globals (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:687-739`, `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:2068-2103`).
- With a plain re-export shim, those lookups move to submodule globals:

```python
# legacy shim
from .cao_interactive_demo.cli import main
from .cao_interactive_demo.commands import inspect_demo

# cli.py
from .commands import inspect_demo

def main(...):
    inspect_demo(...)
```

- Patching `legacy_shim.inspect_demo` would not change `cli.inspect_demo`.

#### Options

**Option A: Preserve full legacy patch-target compatibility.**

- Code impact:
  - The legacy module cannot remain a trivial `from ... import ...` shim.
  - You would need wrappers or a compatibility indirection layer that resolves collaborators through the legacy module at call time, or you would need to keep more behavior in the legacy file.

```python
def main(...):
    return _compat.inspect_demo(...)
```

- User-facing difference:
  - Existing tests and any downstream monkeypatch-based tooling keep working unchanged.
- Pros:
  - Meets the current spec wording and non-goal of leaving tests untouched.
  - Minimizes immediate churn for current test helpers.
- Cons:
  - This is no longer a "pure move, no rewrite"; it introduces compatibility plumbing that the rest of the repo generally avoids.
  - It permanently couples the new package internals to the legacy module.

**Option B: Narrow legacy compatibility to imports and `python -m`, and migrate tests to the owning modules / new package.**

- Code impact:
  - Keep the shim limited to import compatibility plus `__main__`.
  - Update demo tests to patch `cao_server.py`, `commands.py`, `runtime.py`, or `cli.py` directly, depending on ownership.

```python
monkeypatch.setattr(
    "gig_agents.demo.cao_interactive_demo.cao_server.shutil.which",
    lambda _: "/usr/bin/fake",
)
```

- User-facing difference:
  - Normal imports and CLI behavior stay stable, but monkeypatch-based callers must retarget the new owning modules.
- Pros:
  - Matches the repo's existing testing style in `tests/unit/cao/test_server_launcher.py:257-280` and `tests/unit/agents/brain_launch_runtime/test_headless_base.py:257-264`.
  - Keeps the new package ownership clear and avoids compatibility indirection leaking into production code.
- Cons:
  - Requires changing the existing demo unit tests and relaxing the current spec / non-goal language.
  - Any out-of-tree callers patching the legacy module would need to adjust.

> **PROPOSED: Choose Option B and explicitly redefine legacy compatibility as import/CLI compatibility, not legacy monkeypatch compatibility.**
> Rationale: The repository already treats the owning implementation module as the patching surface (`tests/unit/cao/test_server_launcher.py:257-280`, `tests/unit/agents/brain_launch_runtime/test_headless_base.py:257-264`), and the new package structure would fit that pattern naturally. Preserving full legacy patch semantics would require wrapper indirection or keeping behavior in the shim, which conflicts with the stated "pure move" goal in `openspec/changes/modularize-cao-interactive-demo/design.md:21-25` and would make the refactor less native to this codebase.
>
> **DECISION: Remove `gig_agents.demo.cao_interactive_full_pipeline_demo` after the split and update all scripts/tests/imports to the split modules instead of keeping a legacy shim.**
> Rationale: The legacy monolith should not remain as a supported import surface. This avoids shim complexity entirely and makes the new package the only supported path, so the change artifacts should be revised to remove backward-compatibility requirements and add migration work for all in-repo callers.

### Q2) How should the change validate the new canonical package contract?

- Decision: `Deferrable`

#### Why this matters

- The spec adds new behavior that is independent of the legacy shim: a canonical import path, explicit `__all__`, exact module structure, and no circular imports.
- Existing validation mostly proves "the old module still works," not "the new package is correctly established."

#### Examples

- `openspec/changes/modularize-cao-interactive-demo/specs/cao-interactive-demo-module-structure/spec.md:66-74` requires `from gig_agents.demo.cao_interactive_demo import start_demo` to work and requires an explicit `__all__`.
- `openspec/changes/modularize-cao-interactive-demo/tasks.md:33-36` does not include any assertion over the new import path or export list.

#### Why this question arises in code

- Today, all concrete imports still target the legacy module (`tests/unit/demo/test_cao_interactive_full_pipeline_demo.py`, `tests/integration/demo/test_cao_interactive_full_pipeline_demo_cli.py`, and `scripts/demo/cao-interactive-full-pipeline-demo/scripts/verify_report.py`).
- The only in-repo references to `gig_agents.demo.cao_interactive_demo` are in the OpenSpec documents themselves.
- That leaves several spec branches effectively untested:

```text
canonical import path works
explicit __all__ is complete
split modules import acyclically
module/file structure matches the spec
```

#### Options

**Option A: Add targeted structural and import tests for the new package.**

- Code impact:
  - Add a small unit test module that imports `gig_agents.demo.cao_interactive_demo`, asserts a few representative symbol identities, and verifies the explicit `__all__`.
  - Optionally add a lightweight structural assertion for expected files / module count; if the `<500 line` rule is meant to be enforced mechanically, add a focused check for that too.

```python
from gig_agents.demo import cao_interactive_demo as demo_pkg

assert "start_demo" in demo_pkg.__all__
assert demo_pkg.start_demo is commands.start_demo
```

- User-facing difference:
  - No behavior change; better confidence that the new package stays healthy.
- Pros:
  - Directly covers the new requirements instead of relying on incidental coverage through the legacy path.
  - Makes future refactors safer once callers start using the canonical package.
- Cons:
  - Adds a small amount of test maintenance overhead.
  - Structural checks can become noisy if they overfit implementation details.

**Option B: Keep validation limited to typecheck + existing tests + manual inspection.**

- Code impact:
  - No additional test files.
- User-facing difference:
  - Faster to implement initially.
- Pros:
  - Minimal short-term work.
  - Avoids adding tests for constraints the team may later relax.
- Cons:
  - Leaves the new package contract largely unverified.
  - Makes it easy to accidentally satisfy the old-path behavior while violating the new spec.

> **PROPOSED: Choose Option A and add one focused unit test module for canonical imports and explicit exports, then decide whether the line-count rule is a hard requirement or a design guideline.**
> Rationale: The new capability introduced by this change is the package structure itself, so it deserves direct coverage rather than only indirect validation through the shim. This also fits the repo's existing preference for deterministic unit tests over manual verification and will catch incomplete `__all__` lists or accidental import cycles early.
>
> **DECISION: Add direct validation for the new canonical package contract.**
> Rationale: Follow the proposed approach: add focused tests for canonical imports and explicit exports, and clarify whether the line-count rule is enforced as a hard requirement or treated as design guidance.
