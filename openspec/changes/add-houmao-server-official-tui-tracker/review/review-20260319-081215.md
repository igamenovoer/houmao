## Summary

- The architectural direction is strong: moving live parsing and watch ownership into `houmao-server` matches the current pain points in `src/houmao/server/service.py:418`, the runtime-local monitor logic in `src/houmao/agents/realm_controller/backends/cao_rest.py:211`, and the demo-only tracker in `src/houmao/demo/cao_dual_shadow_watch/monitor.py:57`.
- Reusing the shared parser stack instead of inventing a second parser path aligns well with the existing abstraction in `src/houmao/agents/realm_controller/backends/shadow_parser_stack.py:38`.
- The main gap is not direction but contract precision: discovery authority, public lookup identity, and tracked-state shape are still too open for an implementation this invasive.
- The change also needs one explicit decision about how it supersedes the existing server-local `registration.json` bridge and the overlapping `add-shadow-watch-state-stability-window` change, otherwise implementation teams can still pull the design in conflicting directions.

## Findings

### Must Fix / Blocking

1. **The change leaves the authoritative known-session discovery contract unresolved even though startup rediscovery and worker supervision depend on it.** The design still asks “What exact discovery sources and precedence rules should define the known-session registry on startup?” in `openspec/changes/add-houmao-server-official-tui-tracker/design.md:182`, while the specs already require continuous tracking for every known tmux-backed session in `openspec/changes/add-houmao-server-official-tui-tracker/specs/official-tui-state-tracking/spec.md:3` and the tasks already require startup rediscovery in `openspec/changes/add-houmao-server-official-tui-tracker/tasks.md:23`. The repository already has multiple plausible sources of truth: runtime-owned manifests and shared-registry publication from delegated launches in `src/houmao/srv_ctrl/commands/runtime_artifacts.py:145`, fresh-record resolution in `src/houmao/agents/realm_controller/registry_storage.py:94`, tmux-published manifest recovery in `src/houmao/demo/cao_interactive_demo/runtime.py:365`, and the server-local bridge file written by `src/houmao/server/service.py:370`. Without an explicit precedence and staleness policy, the implementation can duplicate workers, keep watching stale sessions, or silently diverge between manifest, registry, and server-local registration views.**

> **DECISION: Accept.**
> Rationale: The gap is real. Startup rediscovery and supervisor lifecycle need one authoritative known-session contract, and leaving it open would produce conflicting implementations. For this change, that contract should be explicit and server-scoped: `houmao-server` should seed known sessions from its own registration bridge for sessions it manages, then enrich and verify with manifest-backed metadata and live tmux liveness. Shared-registry records are useful compatibility evidence, but the current `houmao-server` delta spec already classifies them as transitional compatibility state, so they should not become the primary watch authority.

2. **The artifacts move tracking authority away from CAO `terminal_id`, but they do not yet define the public API identity that will address tracked state after that shift.** The design explicitly says the internal target is keyed by Houmao session identity with only an optional compatibility `terminal_id` in `openspec/changes/add-houmao-server-official-tui-tracker/design.md:75`, but the current server routes, client, and models are all terminal-id keyed in `src/houmao/server/app.py:224`, `src/houmao/server/client.py:68`, and `src/houmao/server/models.py:109`. That mismatch becomes concrete as soon as the system tracks a session rediscovered from manifests or tmux env where CAO metadata is only “a compatibility aid,” or when a TUI-down session remains tracked even though the CAO-shaped live-terminal abstraction is no longer the right authority. The change needs to say whether the primary query surface becomes agent/session keyed with `terminal_id` as an alias, or whether `terminal_id` remains a mandatory compatibility key for all tracked sessions in v1.**

> **DECISION: Refine.**
> Rationale: The reviewer is right that the public lookup contract must be stated, but I am not taking a new public route family in this change. The internal tracker should use Houmao-owned identity and treat `terminal_id` as compatibility metadata, while the v1 external query surface remains the existing terminal-keyed extension routes resolved through a server-owned alias map. That is narrower than the reviewer’s proposed API migration, but it closes the ambiguity without forcing client and route churn into the same architectural rewrite.

### Should Fix / Important

3. **The tracked-state schema is still too implicit for the amount of reducer, route, and test work being proposed.** The design correctly calls out states such as “tmux alive but tool process down” and “tool process alive but parser failed” in `openspec/changes/add-houmao-server-official-tui-tracker/design.md:45`, and the specs require transport state, TUI liveliness, parsed surface, operator-facing state, and stability metadata in `openspec/changes/add-houmao-server-official-tui-tracker/specs/houmao-server/spec.md:94` and `openspec/changes/add-houmao-server-official-tui-tracker/specs/official-tui-state-tracking/spec.md:49`. But the repository’s current server contract still only exposes coarse CAO-shaped operator statuses in `src/houmao/server/models.py:93` and `src/houmao/server/service.py:487`, while the shared parser stack is explicitly limited to supported tools and raises for unsupported tools in `src/houmao/agents/realm_controller/backends/shadow_parser_stack.py:49`. Before implementation, the change should define the minimum response-model fields and enum-like states for transport, process, parse status, parse error, stability metadata, and recent-transition entries so tasks `3.2`, `4.2`, and `5.2` can converge on one contract instead of inventing local interpretations.**

> **DECISION: Accept.**
> Rationale: The tracked-state schema needs to be made explicit before implementation. The change should define minimum response fields for `transport_state`, `process_state`, `parse_status`, optional `probe_error` or `parse_error`, nullable `parsed_surface`, derived `operator_state`, stability metadata, and bounded recent-transition entries. Otherwise tasks `3.2`, `4.2`, and `5.2` will drift into local interpretations of the same pipeline.

4. **The design should say more explicitly which code stays shared versus which code becomes server-local, otherwise the likely implementation will duplicate existing runtime helpers.** Decision 6 proposes a new `src/houmao/server/tui/` subtree in `openspec/changes/add-houmao-server-official-tui-tracker/design.md:117`, which is directionally fine, but the repo already has reusable tmux primitives in `src/houmao/agents/realm_controller/backends/tmux_runtime.py:82`, parser abstraction in `src/houmao/agents/realm_controller/backends/shadow_parser_stack.py:38`, manifest helpers in `src/houmao/agents/realm_controller/manifest.py:87`, and shared-registry access in `src/houmao/agents/realm_controller/registry_storage.py:94`. The change should explicitly call out that the server owns probe/track orchestration while shared tmux, manifest, registry, and parser-selection primitives remain centralized, otherwise “official parser” work can turn into copy-paste or parallel helper stacks.**

> **DECISION: Accept.**
> Rationale: The server should own orchestration, worker lifecycle, and in-memory tracking, but the shared primitives should remain shared. This change should explicitly call out reuse of `tmux_runtime` helpers, the shared parser stack, manifest helpers, and any registry access that remains necessary for compatibility or enrichment, so implementation does not create a parallel helper stack under `src/houmao/server/` without reason.

### Nice To Have

5. **The change hygiene section should record the intended disposition of the overlapping stability-window change more concretely than “reconcile or narrow.”** The proposal says this change should supersede that direction in `openspec/changes/add-houmao-server-official-tui-tracker/proposal.md:15`, the design repeats that this server-owned design should become the primary architecture in `openspec/changes/add-houmao-server-official-tui-tracker/design.md:168`, and task `6.2` says to reconcile or narrow the overlap in `openspec/changes/add-houmao-server-official-tui-tracker/tasks.md:34`. That is probably enough for implementation planning, but not yet enough to prevent two active changes from continuing to define competing stability semantics. A short design note stating whether the demo change is being narrowed to demo-only visualization, superseded entirely, or expected to consume the new server state later would make review and later archive work cleaner.**

> **DECISION: Accept.**
> Rationale: The overlap with `add-shadow-watch-state-stability-window` should be resolved more concretely in the artifacts. The server-owned tracker defined here is the primary stability contract. The overlapping change should be narrowed to demo-only consumption or visualization work, or otherwise marked as superseded for contract-setting purposes, so two active changes do not continue to define competing stability semantics.

## Design Alignment With Existing System

- Existing pattern(s) observed: shared tmux primitives already live in `src/houmao/agents/realm_controller/backends/tmux_runtime.py:82`; parser selection already lives behind `ShadowParserStack` in `src/houmao/agents/realm_controller/backends/shadow_parser_stack.py:38`; runtime-owned durable identity lives in session manifests via `src/houmao/agents/realm_controller/manifest.py:87`; transitional discovery already uses the shared registry in `src/houmao/agents/realm_controller/registry_storage.py:94`; delegated `houmao-srv-ctrl launch` already materializes manifests and registry records in `src/houmao/srv_ctrl/commands/runtime_artifacts.py:145`.
- Proposed design alignment: the change aligns well when it reuses those shared primitives and moves only the watch-plane orchestration into `houmao-server`. The main mismatch is that the current public server surface is still terminal-id and file-mirror shaped in `src/houmao/server/app.py:224`, `src/houmao/server/models.py:109`, `docs/reference/system-files/houmao-server.md:15`, and `docs/reference/houmao_server_pair.md:59`, while the new design is session-identity and memory-primary shaped. The artifacts should make that boundary migration explicit instead of leaving it implicit in implementation.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What should be the authoritative startup discovery contract for the known-session registry?

- Decision: `Blocking`

#### Why this matters

- Worker lifecycle, restart recovery, and stale-session cleanup all depend on the same answer.
- The repo already has multiple plausible discovery sources, and they are not equivalent in freshness or authority.

#### Examples

- A delegated `houmao-srv-ctrl launch` session has a runtime-owned manifest and fresh shared-registry record, but the server also has an older `sessions/<session-name>/registration.json` bridge file.
- A tmux session is still alive and publishes `AGENTSYS_MANIFEST_PATH`, but the shared-registry lease has already expired.

#### Why this question arises in code

- Delegated launches currently create runtime-owned manifests and publish shared-registry records in `src/houmao/srv_ctrl/commands/runtime_artifacts.py:145`.
- The server currently writes its own registration bridge in `src/houmao/server/service.py:370`.
- Freshness-aware shared-registry lookup already exists in `src/houmao/agents/realm_controller/registry_storage.py:94`.
- Tmux manifest-path recovery already exists in `src/houmao/demo/cao_interactive_demo/runtime.py:365`.

```text
manifest + fresh registry + live tmux env
        ↓ reconcile
 known-session registry
        ↓ supervise
    watch workers
```

#### Options

**Option A: Make runtime-owned manifests plus freshness-aware shared-registry records the primary discovery contract, with tmux env used to verify liveness and server-local registration kept only as a narrow compatibility backfill.**

- Code impact:
  - Reuse `manifest.py`, `registry_storage.py`, and tmux environment lookup as the normal server discovery path.
  - Narrow `registration.json` to a v1 bridge instead of a competing authority.
- User-facing difference:
  - Restart recovery and state lookup stay aligned with the repo’s existing runtime-owned identity model.
- Pros:
  - Matches how delegated launches already publish durable identity.
  - Gives explicit stale-record handling through existing registry freshness rules.
  - Avoids inventing a second server-local authority for the same sessions.
- Cons:
  - Requires the design to define reconciliation order and stale/fallback behavior precisely.

**Option B: Make `houmao-server`’s own `sessions/<session-name>/registration.json` the primary discovery source and treat manifests/registry only as optional metadata.**

- Code impact:
  - Keeps most discovery local to `src/houmao/server/service.py`.
  - Pushes manifest and registry reuse to best-effort enrichments.
- User-facing difference:
  - Server-local registration becomes the main persistence contract for what the server knows.
- Pros:
  - Simpler server-local implementation in the short term.
  - Avoids initial dependency on registry freshness semantics.
- Cons:
  - Duplicates runtime-owned identity that the repo already persists elsewhere.
  - Creates more drift risk between server, manifest, registry, and tmux reality.

> **PROPOSED: Use runtime-owned manifests plus freshness-aware shared-registry records as the primary discovery contract, with tmux env as liveness verification and `registration.json` reduced to a narrow compatibility bridge.**
> Rationale: Delegated launches already materialize the durable identity bundle the rest of the repo uses in `src/houmao/srv_ctrl/commands/runtime_artifacts.py:145`, and freshness-aware shared-registry lookup already exists in `src/houmao/agents/realm_controller/registry_storage.py:94`. Re-centering discovery there fits the existing architecture better than promoting `src/houmao/server/service.py:370` into a second primary authority, while tmux env recovery in `src/houmao/demo/cao_interactive_demo/runtime.py:365` gives a concrete liveness cross-check for restart rebuilds.
>
> **DECISION: Reject PROPOSED. Use server-owned registration as the primary discovery seed for this server instance, with manifest-backed metadata and tmux liveness as verification; shared registry remains compatibility evidence, not primary authority.**
> Rationale: The proposed manifest-plus-shared-registry-first contract is plausible, but it conflicts with the current `houmao-server` spec direction that treats shared registry state as transitional compatibility rather than authoritative live-tracking truth. `POST /houmao/launches/register` already materializes the server-scoped identity bridge this server needs, including `terminal_id`, `manifest_path`, `agent_id`, and `tmux_session_name`. For this server-owned watch feature, the cleaner contract is: the server watches sessions it knows through its own registration path, uses the manifest referenced by that registration to enrich identity, and uses live tmux presence to rebuild or discard state after restart.

### Q2) What should be the primary public lookup identity for live tracked state?

- Decision: `Blocking`

#### Why this matters

- The design moves tracking authority to Houmao session identity, but today all server extension routes are addressed by `terminal_id`.
- Without a clear answer, the implementation can build an internal model that the public API cannot address cleanly.

#### Examples

- A tracked session remains known and watchable even when the TUI process is down, but the caller wants to inspect it by agent/session identity rather than a CAO-shaped terminal id.
- A manifest-discovered session is known to Houmao before the child CAO metadata is fully refreshed.

#### Why this question arises in code

- The design makes `terminal_id` only optional compatibility metadata in `openspec/changes/add-houmao-server-official-tui-tracker/design.md:77`.
- The current extension routes and client remain terminal-id keyed in `src/houmao/server/app.py:224` and `src/houmao/server/client.py:68`.
- The persisted server manifest section also still carries `terminal_id` as a first-class field in `src/houmao/agents/realm_controller/manifest.py:165`.

```text
tracked session identity
  ├─ agent/session identity
  ├─ tmux session name
  └─ terminal_id? (compatibility)
```

#### Options

**Option A: Add a session/agent-keyed live-state route as the primary contract and keep `terminal_id` routes as compatibility aliases when a CAO terminal id exists.**

- Code impact:
  - Add explicit identity models and lookup paths for tracked sessions.
  - Keep current terminal-id routes as a thin compatibility view or redirect layer.
- User-facing difference:
  - Callers can query tracked state by the Houmao identity that actually owns the worker.
- Pros:
  - Matches the design’s declared identity model.
  - Works for rediscovered known sessions even when CAO metadata is secondary.
  - Makes the CAO-to-Houmao migration path explicit instead of implicit.
- Cons:
  - Requires adding or revising server/client route contracts in this change.

**Option B: Keep `terminal_id` as the only public lookup key for v1 and require all tracked sessions to retain one compatible terminal id.**

- Code impact:
  - Minimizes public API churn.
  - Forces the server-owned tracker to maintain terminal-id mapping for every known session.
- User-facing difference:
  - Callers continue to think in CAO-shaped terminal identity even as the internal tracker stops using it as authority.
- Pros:
  - Easier short-term compatibility with current client code.
  - Avoids adding a second route family immediately.
- Cons:
  - Keeps the public contract CAO-shaped after the design explicitly says tracking authority is Houmao-owned.
  - Makes manifest/registry rediscovery semantics less natural.

> **PROPOSED: Add a session/agent-keyed live-state route as the primary contract and keep terminal-id lookup as a compatibility alias where a CAO terminal id still exists.**
> Rationale: The design already re-centers authority on Houmao identity in `openspec/changes/add-houmao-server-official-tui-tracker/design.md:75`, while the existing terminal-id routes in `src/houmao/server/app.py:224` and `src/houmao/server/client.py:68` are purely CAO-shaped. Keeping `terminal_id` as an alias preserves compatibility, but making session identity primary is the only shape that stays coherent once the server continuously tracks known tmux sessions independent of CAO polling.
>
> **DECISION: Reject PROPOSED as the v1 public contract. Keep terminal-id lookup as the v1 public route shape, backed by a Houmao-owned internal identity and alias mapping.**
> Rationale: I do not want this change to both replace the watch architecture and introduce a second public route family. The internal worker registry and in-memory store should be Houmao-identity keyed, but the external HTTP and client surface can remain the existing `/houmao/terminals/{terminal_id}/...` compatibility routes for v1. Because the server-scoped registration seed remains authoritative in this design, every tracked session the server owns can still retain a compatible `terminal_id` alias.

### Q3) How explicit should the live tracked-state contract be about parser availability and parser failure?

- Decision: `Deferrable`

#### Why this matters

- The specs require transport, TUI liveliness, parsed surface state, operator-facing state, and stability metadata, but they do not yet say how parser-unavailable or parser-failed cycles appear on the wire.
- This affects reducer behavior, route schemas, and tests for supported-vs-unsupported tools.

#### Examples

- The tmux session exists and the TUI process is running, but the parser throws for a malformed snapshot.
- The session runs a tool outside the supported parser stack, which currently supports Claude and Codex lanes directly.

#### Why this question arises in code

- The server contract today exposes only coarse operator states in `src/houmao/server/models.py:93`.
- The shared parser stack only supports specific tools and raises otherwise in `src/houmao/agents/realm_controller/backends/shadow_parser_stack.py:49`.
- The design already expects states like “tool process alive but parser failed” in `openspec/changes/add-houmao-server-official-tui-tracker/design.md:48`.

```text
transport_state + process_state + parse_status + parsed_surface? + operator_state + stability
```

#### Options

**Option A: Define explicit fields for `transport_state`, `process_state`, `parse_status`, optional parse-error detail, and nullable `parsed_surface`, with operator-facing state derived on top.**

- Code impact:
  - Requires new response models and test fixtures.
  - Lets reducers and history entries report parser-unavailable cycles without pretending a parsed surface exists.
- User-facing difference:
  - Callers can distinguish “TUI down,” “unsupported tool,” and “parser failed” instead of seeing a generic unknown/error.
- Pros:
  - Honest contract for the new multi-stage pipeline.
  - Makes supported-tool scope and parse-failure semantics testable.
- Cons:
  - More API detail to define up front.

**Option B: Keep the public contract mostly at the operator-state level and let transport/process/parser distinctions remain implementation detail.**

- Code impact:
  - Smaller model churn.
  - Pushes nuance into logs or internal diagnostics instead of the main API.
- User-facing difference:
  - Clients mainly see one summarized state and limited diagnostics.
- Pros:
  - Simpler first API.
  - Faster short-term implementation.
- Cons:
  - Loses much of the value of splitting probe, parse, and track into separate stages.
  - Makes failure analysis and future automation harder.

> **PROPOSED: Define explicit transport/process/parse fields with nullable parsed surface and parser-error detail, and derive the operator-facing state on top of that structure.**
> Rationale: The whole point of the new design is to stop compressing watch semantics into a coarse status call, as described in `openspec/changes/add-houmao-server-official-tui-tracker/design.md:37`. The current coarse server model in `src/houmao/server/models.py:93` and the parser-stack support limits in `src/houmao/agents/realm_controller/backends/shadow_parser_stack.py:49` make it clear that a summarized operator state alone will hide the exact distinctions this change is trying to surface.
>
> **DECISION: Accept PROPOSED with one narrowing: define explicit transport, process, and parse fields, and keep `operator_state` as a derived summary rather than the only public contract.**
> Rationale: The reviewer is right that the new probe/parse/track architecture loses most of its value if the wire contract collapses back to one coarse status. The response model should explicitly distinguish transport state, process state, parser availability or failure, nullable parsed surface, and derived operator state. That still lets clients consume the summary field, but it does not hide the distinctions this change is meant to surface.
