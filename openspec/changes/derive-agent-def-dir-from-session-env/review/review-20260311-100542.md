## Summary

- **Well-scoped change**: The proposal, design, specs, and tasks are internally coherent and narrowly targeted at name-based session control. The decision to use tmux session environment (alongside existing `AGENTSYS_MANIFEST_PATH`) is architecturally sound and avoids manifest schema churn.
- **Strong spec coverage**: All three delta specs include concrete scenarios for happy path, explicit override, and failure modes (missing/blank/relative/stale). The agent-identity delta spec correctly extends the existing name-resolution requirement with steps 4-5 for `AGENTSYS_AGENT_DEF_DIR`.
- **Publication gap in backend constructors**: Neither `HeadlessInteractiveSession` nor `CaoRestSession` currently receives `agent_def_dir`, so the publication site (`_publish_tmux_manifest_pointer` / launch_env composition) cannot publish it without constructor/call-site changes. The tasks and design do not specify this wiring path.
- **CLI help text becomes misleading**: The `--agent-def-dir` help string on control commands still describes a single precedence chain; after this change, name-based control has a different resolution model. Task 3.3 covers docs but not CLI `--help` text itself.
- **Demo `_stop_remote_session` wrapper is not covered**: Task 2.4 mentions interactive demo prompt/control/stop flows, but only `send_turn` and `send_control_input` are mentioned explicitly in the spec scenarios. The `_stop_remote_session` helper in `runtime.py:288-320` also passes explicit `--agent-def-dir` and must be updated.

## Findings

### Must Fix / Blocking

1. **Publication sites lack `agent_def_dir` access (implementation gap in tasks)**

   Both tmux-env publication sites need `agent_def_dir` to publish `AGENTSYS_AGENT_DEF_DIR`, but neither currently receives it:

   - `HeadlessInteractiveSession.__init__` (`headless_base.py:45-78`) does not accept `agent_def_dir`. Its `_publish_tmux_manifest_pointer` method (`headless_base.py:289-319`) composes `launch_env` but has no way to set `AGENTSYS_AGENT_DEF_DIR`.
   - `CaoRestSession` launch-env composition (`cao_rest.py:~1430-1476`) builds `launch_env` and sets `AGENTSYS_MANIFEST_PATH` at line 1437, but `agent_def_dir` is not in scope.
   - The `_create_backend_session` factory (`runtime.py:368-481`) does not accept or forward `agent_def_dir`.
   - `start_runtime_session` (`runtime.py:185-271`) receives `agent_def_dir` but only uses it for `load_role_package` at line 201.

   **Action required:** Task 1.1 ("Update tmux-backed launch paths to publish `AGENTSYS_AGENT_DEF_DIR`") needs an explicit sub-step for threading `agent_def_dir` through `_create_backend_session` into backend constructors, or publishing at the `start_runtime_session` level after session creation. This wiring is non-trivial because `HeadlessInteractiveSession._publish_tmux_manifest_pointer` is called from multiple sites (constructor via `_ensure_tmux_container`, and per-turn in `send_prompt`).

### Should Fix / Important

1. **`--agent-def-dir` CLI help text on control commands becomes stale**

Task 3.3 covers reference docs but not the inline `argparse` help strings.

4. **`AgentIdentityResolution` needs an `agent_def_dir` field**

The current `AgentIdentityResolution` dataclass (`runtime.py:70-86`) has no `agent_def_dir` field. The delta `brain-launch-runtime` spec adds the concept but does not explicitly name the return type.    **Suggestion:** Make this explicit in the delta spec or design so the type change is traceable.

## Design Alignment With Existing System

- **Existing pattern: tmux env publication** (`headless_base.py:289-319`, `cao_rest.py:~1437`): Both backends already publish `AGENTSYS_MANIFEST_PATH` into tmux session environment. The proposed `AGENTSYS_AGENT_DEF_DIR` publication follows the same mechanism and namespace convention. **Aligns well.**

- **Existing pattern: tmux env recovery for name resolution** (`runtime.py:629-710`): `_resolve_manifest_path_from_tmux_session` reads `AGENTSYS_MANIFEST_PATH` from tmux env, validates it (absolute, existing file), and returns the path. The proposed `AGENTSYS_AGENT_DEF_DIR` recovery should follow the same validation pattern (absolute, existing directory). The delta agent-identity spec correctly specifies this. **Aligns well.**

- **Existing pattern: resolution ordering in CLI handlers** (`cli.py:286-347`): All control commands currently call `_resolve_agent_def_dir` (ambient resolution) *before* `resolve_agent_identity` (name resolution). The change needs to invert this for name-based identity so that `resolve_agent_identity` runs first and can return the tmux-recovered `agent_def_dir`. This is a deliberate reordering that breaks the current uniform pattern. **Aligned by design intent; implementation must carefully preserve the manifest-path branch behavior.**

- **Existing pattern: demo wrapper CLI construction** (`commands.py:198-209`, `commands.py:266-274`, `runtime.py:288-320`): Demo helpers build explicit CLI arg lists with `"--agent-def-dir", str(env.agent_def_dir)`. Removing these for control commands is clean and follows the change's intent. Build/start wrappers (`runtime.py:148-184`, `runtime.py:187-253`) should keep explicit `--agent-def-dir` as the spec states. **Aligns with proposed split.**

- **Naming and layering**: The proposed constant `AGENTSYS_AGENT_DEF_DIR` follows the existing `AGENTSYS_*` namespace for tmux session environment variables. The `agent_identity.py` module is the appropriate home for the constant. **Aligns well.**

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should `agent_def_dir` reach backend publication sites during `start-session`?

- Decision: `Blocking`

#### Why this matters

- The two tmux-env publication sites (`HeadlessInteractiveSession._publish_tmux_manifest_pointer` and `CaoRestSession` launch-env composition) need the resolved `agent_def_dir` value to publish `AGENTSYS_AGENT_DEF_DIR`. The design says "publish alongside `AGENTSYS_MANIFEST_PATH`" but doesn't specify the wiring path through the constructor chain.

#### Examples

- `start-session --agent-def-dir /repo/.agentsys/agents --brain-manifest ...` must result in `AGENTSYS_AGENT_DEF_DIR=/repo/.agentsys/agents` appearing in the tmux session environment.
- For headless backends, `_publish_tmux_manifest_pointer` is called both at init (via `_ensure_tmux_container`) and per-turn (in `send_prompt` when `session_id` is first captured). The value must be available at both call sites.

#### Why this question arises in code

- `start_runtime_session` (`runtime.py:185-271`) receives `agent_def_dir` at line 187 but only forwards it to `load_role_package` at line 201.
- `_create_backend_session` (`runtime.py:368-481`) is the factory that instantiates all backends but does not accept `agent_def_dir`.
- `HeadlessInteractiveSession.__init__` (`headless_base.py:45-78`) does not accept `agent_def_dir`. Its `_publish_tmux_manifest_pointer` method (`headless_base.py:289-319`) would need the value.

```python
# Current _create_backend_session signature (runtime.py:368)
def _create_backend_session(
    *,
    launch_plan: LaunchPlan,
    role_name: str,
    role_prompt: str,
    api_base_url: str | None = None,
    cao_profile_store_dir: Path | None,
    resume_state: SessionManifestPayloadV2 | None = None,
    session_manifest_path: Path | None = None,
    agent_identity: str | None = None,
    cao_parsing_mode: CaoParsingMode | None = None,
) -> InteractiveSession:
    # No agent_def_dir parameter
```

#### Options

**Option A: Thread `agent_def_dir` through `_create_backend_session` into backend constructors**

- Code impact:
  - Add `agent_def_dir: Path` to `_create_backend_session`, `HeadlessInteractiveSession.__init__`, and `CaoRestSession.__init__`.
  - Store as `self._agent_def_dir` in each backend and reference it in publication methods.
  ```python
  # headless_base.py _publish_tmux_manifest_pointer addition:
  launch_env["AGENTSYS_AGENT_DEF_DIR"] = str(self._agent_def_dir)
  ```
- User-facing difference:
  - None (internal wiring only).
- Pros:
  - Backends own their publication contract end-to-end.
  - Per-turn re-publication in headless backends automatically includes the value.
- Cons:
  - Constructor signature change touches many call sites (4 headless backends + CAO REST).
  - `agent_def_dir` is not otherwise used by backends, making it a pass-through-only parameter.

**Option B: Publish `AGENTSYS_AGENT_DEF_DIR` at the `start_runtime_session` level after backend creation**

- Code impact:
  - After `_create_backend_session` returns, add a tmux `set-environment` call from `start_runtime_session` using the existing `set_tmux_session_environment_shared` helper.
  ```python
  # runtime.py start_runtime_session, after backend_session creation:
  if resolved_agent_identity:
      set_tmux_session_environment_shared(
          session_name=resolved_agent_identity,
          env_vars={"AGENTSYS_AGENT_DEF_DIR": str(agent_def_dir)},
      )
  ```
- User-facing difference:
  - None (internal wiring only).
- Pros:
  - Minimal code change; no backend constructor signature changes needed.
  - Single publication site is easy to audit.
- Cons:
  - Headless `_publish_tmux_manifest_pointer` re-publishes *all* env on each turn (line 311-318). If `AGENTSYS_AGENT_DEF_DIR` is only set once from `start_runtime_session`, a per-turn full re-publish from the headless backend would overwrite the session env without `AGENTSYS_AGENT_DEF_DIR` (since the backend's `launch_env` dict doesn't include it). This could *erase* the published value.
  - For CAO REST, the full env is set once at session creation (`cao_rest.py:1473-1476`), so a post-creation set is safe.

> **PROPOSED: Option A (thread through constructors).**
> Rationale: The headless backend's `_publish_tmux_manifest_pointer` (`headless_base.py:289-319`) calls `set_tmux_session_environment_shared(session_name=session_name, env_vars=launch_env)` with a freshly composed `launch_env` dict on both init and per-turn. This full-dict overwrite means any env var not in `launch_env` gets replaced. Option B's post-creation one-time set would be silently erased by the first `send_prompt` turn. Threading `agent_def_dir` into the constructor ensures the value is present in every publication. The constructor change is mechanical (add one `Path` parameter to 5 call sites) and consistent with how `session_manifest_path` is already threaded.
>

> Rationale: _To be filled by developer._

### Q2) Should demo startup-cleanup (`_reset_demo_startup_state`) continue passing explicit `--agent-def-dir` for backward compatibility?

- Decision: `Deferrable`

#### Why this matters

- The startup-reset flow in `commands.py:593-652` calls `_stop_remote_session` with `tolerate_stale=True` for each cleanup identity. If the cleanup targets a pre-change session lacking `AGENTSYS_AGENT_DEF_DIR` in tmux env, the hard-error behavior from design decision 3 would cause cleanup to fail rather than tolerating it like a stale session.

#### Examples

- Operator upgrades to new code, runs `start` which triggers `_reset_demo_startup_state`. The previous session's tmux env has `AGENTSYS_MANIFEST_PATH` but not `AGENTSYS_AGENT_DEF_DIR`. The cleanup `stop-session` call (without `--agent-def-dir`) hits a hard error from missing tmux pointer.
- With `tolerate_stale=True` the failure might be caught at the stale-detection level, but the error message ("agent definition pointer missing") doesn't match `STALE_STOP_MARKERS` in `models.py`.

#### Why this question arises in code

- `_stop_remote_session` (`runtime.py:288-320`) passes `--agent-def-dir str(env.agent_def_dir)` currently.
- `_reset_demo_startup_state` (`commands.py:619-629`) calls `_stop_remote_session` in a loop.
- `STALE_STOP_MARKERS` (`models.py`) likely contains markers like "does not exist" or "not found" but probably not "agent definition pointer missing".

```python
# commands.py:619-629 cleanup loop
for index, agent_identity in enumerate(
    _dedup_preserve_order(cleanup_agent_identities), start=1
):
    _stop_remote_session(
        paths=paths, env=env,
        agent_identity=agent_identity,
        run_command=run_command,
        tolerate_stale=True,
        log_prefix=f"replacement-stop-{index}",
    )
```

#### Options

**Option A: Keep explicit `--agent-def-dir` only for the cleanup/reset path**

- Code impact:
  - `_stop_remote_session` keeps its `--agent-def-dir` argument for cleanup contexts.
  - The `stop_demo` function's main stop path omits it (uses the new default).
  - Or: split `_stop_remote_session` into cleanup-mode (with `--agent-def-dir`) and normal-mode (without).
- User-facing difference:
  - Cleanup of pre-change sessions works without error.
- Pros:
  - No backward-compatibility breakage during upgrade.
  - Cleanup is a best-effort path anyway.
- Cons:
  - Two code paths for stop in the demo; slightly more complex.
  - The cleanup path remains on the old resolution model indefinitely.

**Option B: Omit `--agent-def-dir` everywhere and extend stale-stop tolerance**

- Code impact:
  - Add "agent definition pointer missing" (or the specific error substring) to `STALE_STOP_MARKERS`.
  - All demo stop paths omit `--agent-def-dir`.
- User-facing difference:
  - Cleanup of pre-change sessions is tolerated as stale.
- Pros:
  - Uniform code path; all demo control commands use the new default.
  - Consistent with "pre-change sessions fail clearly" design intent.
- Cons:
  - Extends stale markers, which could mask genuine errors.

> **PROPOSED: Option A (keep explicit `--agent-def-dir` only for cleanup/reset).**
> Rationale: The startup-reset flow in `commands.py:593-652` is a best-effort cleanup path that must tolerate arbitrary prior-session states. Since `_stop_remote_session` is already a distinct helper (`runtime.py:288-320`), keeping its `--agent-def-dir` argument for the `_reset_demo_startup_state` caller path is a one-line difference and avoids broadening `STALE_STOP_MARKERS` to include errors that could mask real issues. The main `stop_demo` flow (`commands.py:444-485`) can use the new default since it always addresses the current session (which will have `AGENTSYS_AGENT_DEF_DIR` published).
>

### Q4) Should `AGENTSYS_AGENT_DEF_DIR` validation in the recovery path reuse the existing `_resolve_manifest_path_from_tmux_session` pattern or be a separate helper?

- Decision: `Deferrable`

#### Why this matters

- The existing `_resolve_manifest_path_from_tmux_session` (`runtime.py:629-710`) is a ~80-line function that handles tmux session existence check, env var read, missing/blank/non-absolute/stale validation, and output parsing. The new `AGENTSYS_AGENT_DEF_DIR` recovery needs the same validation pattern but for a directory instead of a file.

#### Examples

- Both recovery paths: (1) check tmux session exists, (2) read env var, (3) validate non-blank, absolute, exists. The only difference is `is_file()` vs `is_dir()`.

#### Why this question arises in code

- `_resolve_manifest_path_from_tmux_session` (`runtime.py:629-710`) duplicates tmux session existence check + env read + validation in one monolithic function.
- Task 1.2 says "Add shared validation/helpers for reading `AGENTSYS_AGENT_DEF_DIR`" which suggests extraction, but doesn't say whether to also refactor the existing manifest-path helper.

```python
# runtime.py:629-710 - existing monolithic pattern
def _resolve_manifest_path_from_tmux_session(*, session_name: str) -> Path:
    # 1. has_tmux_session check
    # 2. show_tmux_environment for AGENTSYS_MANIFEST_PATH
    # 3. parse output, validate absolute + is_file
    ...
```

#### Options

**Option A: Extract a shared tmux-env-read-and-validate helper, use for both vars**

- Code impact:
  - Extract common tmux env read + validate logic into a generic helper (e.g., `_read_validated_tmux_env_path`).
  - Call it twice (once for manifest path with `is_file` check, once for agent-def-dir with `is_dir` check), or once with both vars in a combined read.
- User-facing difference:
  - None.
- Pros:
  - DRY; reduces risk of validation inconsistency between the two env vars.
  - Easier to add future tmux env vars.
- Cons:
  - Refactoring existing working code; mild churn risk.

**Option B: Write a parallel standalone helper for `AGENTSYS_AGENT_DEF_DIR` only**

- Code impact:
  - New function `_resolve_agent_def_dir_from_tmux_session` alongside the existing `_resolve_manifest_path_from_tmux_session`.
  - Duplicates tmux session check + env read + validation pattern.
- User-facing difference:
  - None.
- Pros:
  - No churn on existing working code.
  - Each function is self-contained and easy to read.
- Cons:
  - ~60 lines of near-duplicate validation logic.
  - Maintenance risk if tmux env handling changes.

> **PROPOSED: Option A (extract a shared helper).**
> Rationale: The existing `_resolve_manifest_path_from_tmux_session` (`runtime.py:629-710`) already contains the tmux existence check (`has_tmux_session_shared` at line 633, `show_tmux_environment_shared` at line 650) that would be duplicated verbatim. Since design decision 2 establishes that both variables are read from the same tmux session in the same resolution flow (steps 3-5 in the delta agent-identity spec), extracting a shared helper like `_read_tmux_env_var(session_name, var_name) -> str` that handles existence/blank/parse and then validating the returned string separately (absolute + is_file vs absolute + is_dir) avoids ~60 lines of duplication. The tmux session existence check only needs to run once per resolution, further favoring a combined approach.
>
