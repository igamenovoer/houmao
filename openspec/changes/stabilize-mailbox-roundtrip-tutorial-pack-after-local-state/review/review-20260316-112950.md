## Findings

### Must Fix / Blocking

- **None identified.** The change is well-formed and internally consistent at the artifact level.

### Should Fix / Important

1. **Tutorial delta spec missing launcher cleanup scenario.**
   The `mailbox-roundtrip-tutorial-pack/spec.md` specifies "stop launcher-managed CAO on cleanup when the current run started it" as a requirement clause but provides no explicit scenario covering cleanup behavior — particularly for interrupted runs, SIGINT during a mail command, or partial-start failures where CAO was started but no agent sessions were created. The existing `run_demo.sh` has a `cleanup()` trap (line 166-182) that handles session teardown; the new launcher management needs analogous trap-level coverage in the spec.

2. **Pydantic validator `_validate_home_dir` currently rejects empty strings — spec should reference this explicitly.**
   The `cao-server-launcher` delta spec adds the "Empty home_dir selects the launcher-derived default home" scenario, which is correct. However, the current validator at `server_launcher.py:87-94` raises `ValueError("must not be empty when provided")` for blank strings. While task 1.2 covers this, the spec scenario should make clear that this is a breaking change to the config validation contract — previously valid configs with `home_dir = ""` would have been rejected, now they will be accepted. This avoids confusion if the spec is read as documenting existing behavior.

3. **Tutorial runner's CAO profile-store alignment lacks a concrete scenario.**
   Decision 2 states the runner SHALL "align `--cao-profile-store` with the launcher-managed CAO home for the selected base URL." The current `run_demo.sh` already detects the profile store via `detect-cao-profile-store` (line 250-257). The delta spec should include a scenario that explicitly verifies profile-store detection uses the demo-local launcher context (not the repo-wide `config/cao-server-launcher/local.toml`) when the runner started its own CAO.

4. **The runtime startup reorder needs a spec constraint for bootstrap failure cleanup.**
   The new scenario "Second mailbox-enabled session joins an initialized shared mailbox root without manual pre-registration" covers the success path. But if `bootstrap_resolved_mailbox` succeeds (creating a registration) and then `build_launch_plan` fails for an unrelated reason (e.g., invalid env contract), the mailbox root now has a dangling registration with no running session. The spec should either acknowledge this as acceptable or add a rollback constraint.

### Nice To Have

1. **Task 2.2 (external-CAO escape hatch) could be more concrete about failure modes.**
   The task says "fail or skip with clear profile-store guidance." Listing the specific expected error messages or `SKIP:` reasons would make implementation and test coverage more straightforward.

2. **Design Decision 5 could reference the exact Pydantic validator line for implementer clarity.**
   The decision correctly describes the desired contract but doesn't cite the specific code path (`_LauncherConfigModel._validate_home_dir` at `server_launcher.py:87-94`) that needs modification.

3. **Tutorial delta spec could add a scenario for `--snapshot-report` with the new launcher-managed layout.**
   The current `run_demo.sh` supports `--snapshot-report` (line 407-409). After the tutorial refresh, the sanitized report will need to mask launcher-managed CAO artifacts. A spec scenario would prevent regressions in snapshot mode.

## Design Alignment With Existing System

### Runtime startup ordering fix (Decision 4)

- **Existing pattern observed**: `start_runtime_session()` in `runtime.py:546-690` follows a strict sequence: resolve config → build launch plan → compute manifest path → bootstrap mailbox.
- **Proposed design alignment**: The fix reorders to: resolve config → compute manifest path → bootstrap mailbox → build launch plan. This is a clean reorder because `default_manifest_path()` at line 664 depends only on `effective_runtime_root`, `selected_backend`, and `session_id` — all computed before line 653. The `mailbox_env_bindings()` call inside `build_launch_plan()` at `launch_plan.py:83` is what triggers `resolve_active_mailbox_inbox_dir()` at `filesystem.py:443-456`, which calls `load_active_mailbox_registration()` on initialized roots. Moving bootstrap before launch-plan construction directly fixes this call ordering. **Aligns well.**

### Launcher `home_dir` portability fix (Decision 5)

- **Existing pattern observed**: `_resolve_home_dir()` at `server_launcher.py:1072-1087` already handles `None` by returning the derived default `server_root / "home"`. The `_validate_home_dir` Pydantic validator at lines 87-94 rejects empty strings.
- **Proposed design alignment**: Normalizing empty string to `None` in the validator (or treating it equivalently in `_resolve_home_dir`) follows the established pattern where the function already has the right fallback. The checked-in `local.toml` change from `home_dir = "/data/agents/cao-home"` to omitting `home_dir` directly resolves the known issue at `context/issues/known/issue-cao-server-local-config-hardcodes-nonportable-home-dir.md`. **Aligns well.**

### Tutorial pack launcher management (Decision 2)

- **Existing pattern observed**: The current `run_demo.sh` at `scripts/demo/mailbox-roundtrip-tutorial-pack/run_demo.sh` assumes ambient CAO (line 247-257: detects profile store from existing launcher config). It does not start or stop CAO. The `cao.tools.cao_server_launcher` module at `src/houmao/cao/tools/cao_server_launcher.py` provides the launcher CLI surface.
- **Proposed design alignment**: Having the tutorial runner generate a demo-local `launcher.toml` and use `houmao.cao.tools.cao_server_launcher` to manage CAO lifecycle follows the repo's existing launcher pattern. The proposed `cao/` subdirectory in the demo output layout is new but consistent with keeping all demo artifacts under one root. **Aligns, but adds significant new bash-level orchestration complexity.** The current script is ~416 lines of bash; launcher management (config generation, start, ownership check, stop on cleanup, error recovery) could add 80-120 lines. Consider whether a Python-based launcher helper (scoped to the tutorial pack under `scripts/`) would be cleaner than pure bash.

### Mailbox state model documentation (Decision 3)

- **Existing pattern observed**: The `add-mailbox-local-state-and-gateway-notifier` design established the `index.sqlite` / `mailbox.sqlite` split. The mailbox env bindings in `mailbox_runtime_support.py:272-294` already publish `AGENTSYS_MAILBOX_FS_LOCAL_SQLITE_PATH` and `AGENTSYS_MAILBOX_FS_MAILBOX_DIR`. The current tutorial README at `scripts/demo/mailbox-roundtrip-tutorial-pack/README.md` makes no mention of `mailbox.sqlite` or the local-state split.
- **Proposed design alignment**: Updating the README and verification flow to surface the split is straightforward documentation work that fills a real gap. **Aligns well.**

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should `mailbox_env_bindings()` add defense-in-depth tolerance for missing registrations, in addition to the bootstrap reorder?

- Decision: `Deferrable`

#### Why this matters

- The bootstrap reorder (Decision 4) fixes the immediate ordering bug, but the fix depends on a precise code-flow ordering invariant. If a future refactor or new call site invokes `mailbox_env_bindings()` before bootstrap, the same failure reappears silently.

#### Examples

- A future `resume-session` path calls `mailbox_env_bindings()` against a persisted config without re-running bootstrap.
- A test helper constructs a `LaunchPlanRequest` with a `MailboxResolvedConfig` without first bootstrapping the mailbox root.

#### Why this question arises in code

- `mailbox_env_bindings()` at `mailbox_runtime_support.py:272-294` unconditionally calls `resolve_active_mailbox_dir()`, which at `filesystem.py:459-472` calls `load_active_mailbox_registration()` when the root is initialized.
- There is no documented contract that callers must bootstrap before calling `mailbox_env_bindings()`.

```python
# filesystem.py:468-472 — strict path when root is initialized
if not paths.protocol_version_file.exists() or not paths.sqlite_path.exists():
    return paths.mailbox_entry_path(normalized_address)
# ↓ requires active registration to exist
registration = load_active_mailbox_registration(paths.root, address=normalized_address)
return registration.mailbox_path
```

#### Options

**Option A: Bootstrap reorder only (as currently designed)**

- Code impact:
  - Move `bootstrap_resolved_mailbox()` before `build_launch_plan()` in `runtime.py`. No changes to `mailbox_env_bindings()` or `filesystem.py`.
- User-facing difference:
  - The second-agent startup bug is fixed. No change to error messages or behavior when the invariant holds.
- Pros:
  - Minimal code change. Clean separation of concerns — bootstrap owns registration creation, env bindings assume it exists.
- Cons:
  - Fragile ordering invariant. No compile-time or runtime guard prevents future regressions.

**Option B: Bootstrap reorder + defensive fallback in `resolve_active_mailbox_dir()`**

- Code impact:
  - Same reorder as Option A. Additionally, wrap `load_active_mailbox_registration()` in a try/except in `resolve_active_mailbox_dir()` and `resolve_active_mailbox_inbox_dir()` that falls back to `paths.mailbox_entry_path(normalized_address)` when the registration is not found.
- User-facing difference:
  - Same fix. Additionally, `mailbox_env_bindings()` would produce a best-effort result even without bootstrap, preventing hard failures on stale or test-constructed configs.
- Pros:
  - Defense-in-depth. Future callers of `mailbox_env_bindings()` won't hit the same trap.
- Cons:
  - Slightly blurs the contract for `resolve_active_mailbox_dir()`. Could mask genuine registration errors.

> **PROPOSED: Option A for this change, with a `# NOTE:` comment in `mailbox_env_bindings()` documenting the bootstrap-first invariant.**
> Rationale: The reorder is the correct primary fix and the one the design explicitly chose. Adding a defensive fallback risks masking legitimate registration errors in production. A code comment at `mailbox_runtime_support.py:272` creates a low-cost breadcrumb for future maintainers. If the invariant is violated later, the failure will be loud and diagnosable rather than silently producing wrong paths.
>
> **DECISION: Accept the proposed direction, with a narrow refinement: the reorder is the behavioral fix for this change, and we should not add fallback path synthesis.**
> Rationale: The current runtime already has a single clear ownership point for registration creation in `bootstrap_resolved_mailbox()`, and the delta runtime spec now makes bootstrap-before-binding the intended contract. Adding fallback behavior in `resolve_active_mailbox_dir()` or `resolve_active_mailbox_inbox_dir()` would weaken that contract and could hide real registration problems. I do want one small defense-in-depth breadcrumb: document the bootstrap-first invariant near `mailbox_env_bindings()` or the reordered startup path so a future caller does not accidentally reintroduce this bug.

### Q2) Should the tutorial runner's new CAO lifecycle management be implemented in bash or in a scoped Python helper?

- Decision: `Blocking`

#### Why this matters

- The current `run_demo.sh` is already 416 lines of bash. Adding launcher config generation, CAO start/stop with ownership verification, and cleanup integration will add substantial logic. Bash makes error handling, JSON manipulation, and conditional flows harder to test and maintain.

#### Examples

- Generating a demo-local `launcher.toml` from template values requires either `cat <<EOF` heredoc construction or a Python helper call.
- Ownership verification on reuse requires parsing the launcher result JSON and checking status fields.
- The cleanup trap needs to conditionally stop CAO only when the current run started it, which requires tracking launcher state across the script.

#### Why this question arises in code

- The current script already delegates structured work to Python helpers (`tutorial_pack_helpers.py`) for JSON parsing, path resolution, parameter validation, report building, and worktree provisioning (`run_demo.sh:121-135`, `run_demo.sh:396-405`).
- The existing `houmao.cao.tools.cao_server_launcher` CLI at `src/houmao/cao/tools/cao_server_launcher.py` provides `start`, `stop`, and `status` commands that return structured JSON.
- The question is whether the runner should call `cao_server_launcher` CLI directly from bash and parse results, or delegate the entire lifecycle (config generation + start + ownership check + cleanup) to a new function in `tutorial_pack_helpers.py`.

#### Options

**Option A: Direct bash + `cao_server_launcher` CLI calls**

- Code impact:
  - Add 80-120 lines to `run_demo.sh` for config generation, `pixi run python -m houmao.cao.tools.cao_server_launcher start/stop/status` calls, JSON result parsing via `json_value`, and cleanup trap updates.
- User-facing difference:
  - All CAO lifecycle steps visible in bash script. Operator can read the script linearly.
- Pros:
  - Everything visible in one file. No new Python helper surface.
- Cons:
  - Bash error handling is fragile. JSON parsing via `json_value` shell calls is verbose. Ownership verification logic becomes hard to test in unit tests.

**Option B: Python helper function in `tutorial_pack_helpers.py`**

- Code impact:
  - Add a `manage_demo_cao_lifecycle()` function (or similar) to `tutorial_pack_helpers.py` that generates the launcher config, calls the launcher programmatically (importing `houmao.cao.server_launcher` directly), and returns a structured result. Bash calls one helper command for start and one for stop.
- User-facing difference:
  - Bash script stays concise. CAO lifecycle is a single helper call. Cleanup trap calls the stop helper.
- Pros:
  - Testable in Python. Reuses launcher module directly instead of CLI-to-JSON-to-bash round-trips. Ownership verification is a Python conditional, not a bash `json_value` chain.
- Cons:
  - Adds ~60-80 lines to the Python helper. Tutorial bash script no longer shows all CAO steps inline.

> **PROPOSED: Option B — delegate CAO lifecycle management to `tutorial_pack_helpers.py`.**
> Rationale: The current `run_demo.sh` already delegates structured work to `tutorial_pack_helpers.py` (JSON parsing at line 121, worktree provisioning at line 223, report building at line 396). CAO lifecycle management involves config generation, JSON parsing of launcher results, and conditional ownership logic — all of which are cleaner and testable in Python. The script already calls `pixi run python "$HELPER_SCRIPT" <subcommand>` for each structured operation; adding `start-demo-cao` and `stop-demo-cao` subcommands follows the established pattern. This also enables task 4.1's "unit and integration tests for launcher-managed CAO startup, ownership-mismatch recovery" since the logic will be in a testable Python function.
>
> **DECISION: Accept the proposed direction: implement demo CAO lifecycle in `tutorial_pack_helpers.py`, with `run_demo.sh` kept as a thin orchestration wrapper.**
> Rationale: This matches the current pack structure much better than expanding the bash script into a second launcher implementation. The tutorial already uses the helper for structured path, JSON, and report logic, and CAO lifecycle adds exactly the kinds of ownership checks, launcher-result parsing, and cleanup-state handling that are easier to reason about and test in Python. I am not locking the implementation to a specific import-vs-CLI call path yet; the important architectural decision is that helper-owned Python logic, not bash, should own the lifecycle behavior.

### Q3) Should the checked-in `local.toml` omit `home_dir` entirely or set it to an explicit empty string?

- Decision: `Deferrable`

#### Why this matters

- The design says both are equivalent. But the checked-in config is documentation-by-example — operators will copy it as a starting point.

#### Examples

- `config/cao-server-launcher/local.toml` with `home_dir` omitted: operators who want to set a custom home must know the key name from docs.
- `config/cao-server-launcher/local.toml` with `home_dir = ""`: operators see the key exists, understand it's optional, and can fill it in.

#### Why this question arises in code

- The current file at `config/cao-server-launcher/local.toml` has 5 keys. Removing `home_dir` entirely makes it 4 keys. Keeping it as `home_dir = ""` preserves discoverability but requires the new validator normalization.
- TOML files support commenting out keys: `# home_dir = "/custom/path"` is a third option that preserves discoverability without requiring code changes for blank-string handling.

#### Options

**Option A: Omit `home_dir` entirely**

- Code impact:
  - Only the TOML file changes. `_validate_home_dir` still needs the blank-string normalization for user-authored configs but the checked-in file doesn't exercise it.
- User-facing difference:
  - Clean minimal config. Less obvious that `home_dir` is configurable.
- Pros:
  - Simplest. No ambiguity about what "empty" means.
- Cons:
  - Operators must discover `home_dir` from docs or spec.

**Option B: Set `home_dir = ""` with a TOML comment**

- Code impact:
  - TOML file uses `home_dir = ""`. Validator must normalize blank to None.
- User-facing difference:
  - Key is visible. Operators know it exists and can fill it in.
- Pros:
  - Self-documenting config file.
- Cons:
  - Empty-string semantics are less conventional in TOML. Slightly more code in the validator.

**Option C: Comment out with example value**

- Code impact:
  - TOML file uses `# home_dir = "/path/to/custom/cao-home"`. No validator change needed beyond the blank-string normalization.
- User-facing difference:
  - Key is visible as a comment. Operator uncomments and edits to customize.
- Pros:
  - Most discoverable. Familiar TOML pattern. Default behavior is clear (omitted = system default).
- Cons:
  - TOML comment is not machine-parseable. Operators must manually uncomment.

> **PROPOSED: Option C — comment out with an example value.**
> Rationale: This is the most conventional TOML pattern for optional-with-default settings and is already used in many well-known config files. It preserves discoverability without requiring the validator to handle a blank-string edge case in the checked-in config itself (though blank-string normalization should still be implemented for user-authored configs per the spec). The current `local.toml` at `config/cao-server-launcher/local.toml` is intentionally minimal (5 lines); a commented-out `home_dir` with a hint like `# home_dir = "/path/to/custom/cao-home"  # omit for launcher-derived default` adds documentation without adding code complexity.
>
> **DECISION: Reject the proposed option C. Use explicit `home_dir = ""` in the checked-in config.**
> Rationale: This change intentionally introduces the contract that blank `home_dir` means "use the launcher-derived system-defined home," and the repo-owned `local.toml` should exercise that contract directly rather than only describing it in a comment. That makes the portable default visible, copyable, and testable, which fits both the promoted system issue and the delta spec's explicit blank-string scenarios. Omission should still work, but the checked-in example should demonstrate the new portable-default behavior instead of hiding it.

### Q4) Should the tutorial verification flow check `mailbox.sqlite` existence via filesystem inspection or via a runtime/helper query?

- Decision: `Deferrable`

#### Why this matters

- Task 3.3 says "Add verification checks that both tutorial mailboxes materialize mailbox-local `mailbox.sqlite` state." The delta spec says the verification flow SHALL confirm mailbox-local state artifacts exist. The question is the mechanism.

#### Examples

- Direct filesystem check: `test -f "$MAILBOX_ROOT/mailboxes/<address>/mailbox.sqlite"` in bash.
- Runtime query: extend `tutorial_pack_helpers.py` `build-report` to include a `mailbox_local_state_present` check using `resolve_active_mailbox_local_sqlite_path()`.

#### Why this question arises in code

- The current report builder at `run_demo.sh:396-404` delegates to `tutorial_pack_helpers.py build-report` which already receives `--mailbox-root`. That helper could use the mailbox module's path-resolution functions to check for `mailbox.sqlite`.
- Direct bash `test -f` is simpler but hardcodes the internal layout path (`mailboxes/<address>/mailbox.sqlite`), coupling the tutorial to filesystem internals.

#### Options

**Option A: Direct filesystem check in bash**

- Code impact:
  - Add 2-4 lines to `run_demo.sh` per mailbox address, using known paths.
- User-facing difference:
  - Visible in script. Fails with a clear `FAIL:` message if missing.
- Pros:
  - Simple. No Python dependency for this check.
- Cons:
  - Hardcodes mailbox internal layout in the tutorial script.

**Option B: Python helper using `resolve_active_mailbox_local_sqlite_path()`**

- Code impact:
  - Add a `verify-mailbox-local-state` subcommand to `tutorial_pack_helpers.py` that uses `houmao.mailbox.filesystem.resolve_active_mailbox_local_sqlite_path()`.
- User-facing difference:
  - Same verification, but routed through the mailbox module's path contract.
- Pros:
  - Decoupled from internal layout. If the path changes, only the mailbox module needs updating.
- Cons:
  - Slightly more complex for a simple file-existence check.

> **PROPOSED: Option B — use the mailbox module's path resolution.**
> Rationale: The `resolve_active_mailbox_local_sqlite_path()` function at `filesystem.py:475-478` is the authoritative API for this path. The tutorial already delegates structured checks to `tutorial_pack_helpers.py`, and adding a small verification subcommand keeps the bash script free of internal layout knowledge. This follows the same pattern as `detect-cao-profile-store` and `render-mailbox-root` — structured queries go through Python helpers.
>
> **DECISION: Accept the proposed direction, with the verification folded into the existing helper/report path rather than a bash-side file probe.**
> Rationale: `resolve_active_mailbox_local_sqlite_path()` is the authoritative contract for mailbox-local state paths, and the tutorial already routes structured verification through `tutorial_pack_helpers.py`. Keeping this check in Python avoids hardcoding `mailboxes/<address>/mailbox.sqlite` into `run_demo.sh` and keeps the demo aligned with mailbox-module path resolution if layout details evolve later.
