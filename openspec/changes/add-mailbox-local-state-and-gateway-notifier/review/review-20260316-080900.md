## Summary

- The overall split is directionally strong: mailbox truth stays mailbox-owned, gateway polling stays gateway-owned, and the queue-backed execution model fits the existing gateway runtime well.
- The change is currently strongest on intent and weakest on persisted contracts. A few repo-critical boundaries still need to be made explicit before implementation starts.
- The biggest gaps are around how the new mailbox-local paths survive launch-plan/session-manifest persistence, and how the existing shared `thread_summaries.unread_count` is retired or repurposed.
- Gateway observability and idle-only scheduling align with current storage/runtime patterns, but notifier persistence and mailbox-capability discovery need tighter repo-native wording.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should the gateway discover whether a session is mailbox-enabled for notifier control?

- Decision: `Blocking`

#### Why this matters

- Task `4.5` requires explicit rejection for mailbox-disabled sessions, but the current gateway attach path only consumes mailbox-adjacent manifest information indirectly.
- If this contract is left implicit, notifier enablement will likely grow ad hoc manifest parsing in one codepath and duplicate checks somewhere else in runtime/recovery.

#### Examples

- A session has `manifest_path` but its `launch_plan.mailbox` is `null`; notifier enablement must reject deterministically.
- A session resumes after rebind; the notifier should re-discover mailbox support the same way the gateway already re-discovers the live CAO terminal.

#### Why this question arises in code

- Current gateway manifest reload only extracts `payload.cao` in [src/houmao/agents/realm_controller/gateway_service.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/houmao/agents/realm_controller/gateway_service.py#L116).
- Mailbox enablement already exists in persisted launch-plan/session-manifest models via `LaunchPlanPayloadV1.mailbox` and `SessionManifestPayloadV3.launch_plan` in [src/houmao/agents/realm_controller/boundary_models.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/houmao/agents/realm_controller/boundary_models.py#L87) and [src/houmao/agents/realm_controller/boundary_models.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/houmao/agents/realm_controller/boundary_models.py#L239).

```text
attach_contract.manifest_path
  -> load_session_manifest(...)
  -> parse_session_manifest_payload(...)
  -> payload.launch_plan.mailbox ? notifier_supported : reject
```

#### Options

**Option A: Reuse the existing session manifest as the mailbox-capability source.**

- Code impact:
  - Add an explicit helper on the gateway side that reads `payload.launch_plan.mailbox` from the manifest already referenced by `attach_contract.manifest_path`.
  - Add tests for missing manifest, missing launch plan, and `mailbox is None`.
- User-facing difference:
  - No new persisted gateway contract is introduced; notifier support tracks the same manifest/runtime truth operators already have.
- Pros:
  - Reuses the repo’s existing persisted source of truth.
  - Avoids duplicating mailbox metadata into another file.
- Cons:
  - Requires explicit design wording about manifest-read failure behavior.

**Option B: Extend the attach contract with explicit mailbox-capability fields.**

- Code impact:
  - Add new fields to `GatewayAttachContractV1`, update writers/readers, and version or evolve that contract.
  - Keep attach-contract mailbox data in sync with session manifest changes.
- User-facing difference:
  - Gateway capability inspection becomes more direct, but now operators must reason about two persisted sources.
- Pros:
  - Simplifies enablement checks at runtime.
  - Reduces repeated manifest parsing.
- Cons:
  - Introduces duplicated persisted truth and more drift risk.

> **PROPOSED: Use Option A and make `payload.launch_plan.mailbox` the explicit notifier-support contract.**
> Rationale: The gateway already reloads the runtime-owned manifest through `attach_contract.manifest_path` in [src/houmao/agents/realm_controller/gateway_service.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/houmao/agents/realm_controller/gateway_service.py#L116), and mailbox capability already exists in the manifest-boundary models in [src/houmao/agents/realm_controller/boundary_models.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/houmao/agents/realm_controller/boundary_models.py#L87). Reusing that path keeps one authoritative persisted source, fits existing architecture better than attach-contract duplication, and only needs explicit failure semantics plus tests.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._

### Q2) What should the primary identity shape be inside each mailbox-local SQLite database?

- Decision: `Blocking`

#### Why this matters

- Migration, repair, and helper-script behavior all depend on whether mailbox-local state is mailbox-scoped or still redundantly tied back to shared registration ids.
- Leaving this implicit will spread schema decisions across delivery, repair, and query implementations instead of giving them one stable target.

#### Examples

- A symlink-registered mailbox moves to a new resolved path; local state should move with the mailbox directory rather than require registration-id rewrites in every row.
- Repair rebuilds one mailbox-local DB from shared catalog data; the rebuild path is simpler if the local DB is intrinsically mailbox-scoped.

#### Why this question arises in code

- The current shared-root `mailbox_state` table is keyed by `(registration_id, message_id)` because one database serves many mailboxes in [src/houmao/mailbox/filesystem.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/houmao/mailbox/filesystem.py#L122).
- The new design moves that mutable state into one `mailbox.sqlite` per resolved mailbox directory, which changes the natural scope of the database.

```text
shared index.sqlite:
  mailbox_state(registration_id, message_id, ...)

candidate local mailbox.sqlite:
  message_state(message_id, ...)
```

#### Options

**Option A: Make each local mailbox DB mailbox-scoped and key message state by `message_id`.**

- Code impact:
  - Define a mailbox-local `message_state` table keyed by `message_id` (plus any local thread/cache tables).
  - Migration maps shared `(registration_id, message_id)` rows into mailbox-scoped local rows.
- User-facing difference:
  - Repair and inspection are easier because each local DB naturally represents one mailbox.
- Pros:
  - Matches the new directory-level ownership model.
  - Removes redundant identity from local rows.
- Cons:
  - Cross-check tooling must join back to shared registration data externally when needed.

**Option B: Keep `registration_id` in each local DB and preserve the shared-table identity shape.**

- Code impact:
  - Local schema mirrors the old shared-root row shape even though each DB only serves one mailbox.
  - Repair and migration must carry registration ids through every local mutation path.
- User-facing difference:
  - Some joins are more direct, but local DBs become less self-contained.
- Pros:
  - Feels closer to the current schema.
  - May simplify some migration code mechanically.
- Cons:
  - Duplicates external identity in a mailbox-scoped database.
  - Makes moved/symlinked mailbox ownership less clean.

> **PROPOSED: Use Option A and make each local mailbox DB mailbox-scoped with `message_id` as the row identity for mutable message state.**
> Rationale: The current `(registration_id, message_id)` key exists because `index.sqlite` is shared across all mailboxes in [src/houmao/mailbox/filesystem.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/houmao/mailbox/filesystem.py#L122). Once the authority moves to one `mailbox.sqlite` per resolved mailbox directory, keeping `registration_id` in every row adds redundant coupling to shared-root identity and makes repair/move scenarios harder than necessary.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._
