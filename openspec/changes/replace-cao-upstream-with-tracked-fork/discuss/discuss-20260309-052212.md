# Discussion: CAO Fork Reference Policy Open Questions

- Feature/Change: `replace-cao-upstream-with-tracked-fork`
- Timestamp: `2026-03-09T05:22:12Z`
- Context: This doc captures unresolved questions found while scoping the `gig-agents` change to replace orphan/upstream CAO references and installation guidance with a fork-backed policy.

Key constraints:

- `gig-agents` is a standalone repo, so active guidance should not rely on a containing workspace’s orphan checkout path.
- Runtime behavior remains unchanged: `gig-agents` still requires `cao-server` on `PATH`.
- Active docs, issue notes, specs, and launcher/demo messages currently mix orphan paths, upstream URLs, and generic package-name installs.

---

## Summary of recommended defaults

1) Standardize active install and `uvx --from` guidance on the fork Git URL pinned to the tracked branch rather than the package-name install. (Q1)
2) Treat orphan/upstream cleanup as an active-guidance migration with explicit exceptions for provenance/archive/licensing text. (Q2)
3) Update active issue notes to use fork-oriented actionable references while preserving provenance wording when an issue was first observed in upstream-derived CAO. (Q3)
4) Split normal operational links from governance/security/release links instead of rewriting every self-reference the same way. (Q4)

## Q1) What exact fork-backed install contract should active `gig-agents` guidance use?

- Decision: `Blocking`

### Why this matters

- `README.md` currently mixes upstream Git install guidance and generic package-name install guidance.
- Launcher and demo remediation strings need one concrete command family, not a vague “install CAO” message.
- `uvx --from` snippets in CAO profile examples must follow the same source policy as executable installation.

### Examples

- `README.md` uses upstream install guidance at lines 73-76 and the package-name install at lines 424-427.
- `docs/reference/cao_server_launcher.md`, `docs/reference/brain_launch_runtime.md`, `scripts/demo/cao-server-launcher/README.md`, `scripts/demo/cao-server-launcher/run_demo.sh`, and `src/gig_agents/cao/server_launcher.py` all need the same answer.

### Why this question arises in code

- The new proposal says “fork-backed” but does not define the canonical command.
- Without one answer, implementation will drift across duplicated docs and help strings.

### Options

**Option A: Keep `uv tool install cli-agent-orchestrator`**

- User-facing difference:
  - Shortest command, but the installed source is ambiguous.
- Pros:
  - Minimal text churn.
- Cons:
  - Does not guarantee the fork is what users install.

**Option B: Standardize on the fork Git URL pinned to the tracked branch**

- User-facing difference:
  - Longer command, but it explicitly installs the intended fork.
- Pros:
  - Aligns install guidance with the fork policy.
  - Works consistently for both `uv tool install` and `uvx --from`.
- Cons:
  - More verbose.

**Option C: Prefer local-source workflows only**

- User-facing difference:
  - Works for contributors already inside a checkout, but not for general operators.
- Pros:
  - Strong local-source alignment.
- Cons:
  - Poor fit for launcher/demo/operator guidance that expects `cao-server` on `PATH`.

### Proposal (Recommended)

Take **Option B** for active operational guidance. Use a fork Git URL command family for install and `uvx --from` guidance, and reserve local-source workflows for fork-internal development docs only.

### Pros / Cons (Proposal)

- Pros:
  - Removes ambiguity from every active operator-facing surface.
  - Keeps runtime behavior unchanged.
- Cons:
  - Requires a coordinated text sweep.

> **DECISION: Split CAO install guidance by audience: end users install the fork from `hz-release`, while repo developers use Pixi with a repo-local editable CAO install from `hz-dev`.**
> Rationale: Operator-facing docs and launcher/demo remediation strings need one explicit fork-backed command family, so official guidance should use `uv tool install git+https://github.com/imsight-forks/cli-agent-orchestrator.git@hz-release --upgrade` and matching `uvx --from ...@hz-release ...` examples. For contributors working inside this repo, the workflow should stay Pixi-managed and avoid a separately installed system CAO; `gig-agents` already uses editable Pixi installs for local development (`pyproject.toml` has `gig_agents = { path = \".\", editable = true }`), so CAO development should follow the same pattern with a repo-local editable install that tracks the fork's `hz-dev` branch.

### Notes / Follow-ups

- Current repo state does not yet satisfy the developer half of this decision: `extern/tracked/` inside `gig-agents` contains only scaffold files, and there is no local `extern/tracked/cli-agent-orchestrator` checkout in this repo today.
- The forked CAO source currently exists only in the parent workspace at `/data1/huangzhe/code/agent-system-dissect/extern/tracked/cli-agent-orchestrator`, because the submodule was added to the parent repo rather than to `gig-agents`.
- Implementation needs a concrete repo-local source strategy for contributors, most likely adding the fork under `gig-agents/extern/tracked/cli-agent-orchestrator` and wiring Pixi activation to install it as an editable package.

## Q2) Which upstream/orphan references are allowed to remain after the migration?

- Decision: `Blocking`

### Why this matters

- The change wants to clean up active guidance, but some files are provenance or licensing records rather than operator docs.
- Verification needs a reviewable exception policy instead of an implicit “grep until zero.”

### Examples

- Active docs and specs should change.
- Licensing/provenance text and archived history may need explicit upstream wording to remain accurate.

### Why this question arises in code

- Proposal/design language already implies scoped exceptions, but the tasks do not enumerate them.
- Files like `NOTICE` and active issue notes sit close to the boundary between “current guidance” and “historical record.”

### Options

**Option A: Rewrite everything except obvious legal notices**

- Pros:
  - Simplest sweep rule.
- Cons:
  - Overwrites useful provenance and historical context.

**Option B: Rewrite active guidance and keep an explicit exception allowlist**

- Pros:
  - Matches the actual intent of the change.
  - Makes review boundaries clear.
- Cons:
  - Requires reviewers to agree on the allowlist.

**Option C: Only update the most obvious docs and ignore the rest**

- Pros:
  - Lowest initial effort.
- Cons:
  - Leaves the repo inconsistent.

### Proposal (Recommended)

Take **Option B**. Define active operational guidance as in-scope, and treat archive/provenance/licensing text as explicit exceptions that must be reviewed deliberately rather than rewritten by default.

### Pros / Cons (Proposal)

- Pros:
  - Keeps current operator guidance coherent.
  - Preserves intentional provenance.
- Cons:
  - Verification becomes policy-aware rather than purely mechanical.

## Q3) How should active issue notes preserve provenance while dropping orphan-path references?

- Decision: `Blocking`

### Why this matters

- Active issue notes are part of current repo guidance, but many of them describe behavior first observed in upstream-derived CAO.
- Rewriting them too aggressively could erase why a note exists; leaving them untouched keeps orphan paths alive in active docs.

### Examples

- `context/issues/known/issue-cao-server-fixed-port-9889.md`
- `context/issues/known/issue-cao-claude-code-output-mode-last-marker-mismatch.md`

### Why this question arises in code

- Those files currently mix actionable source pointers with provenance wording.
- The migration tasks need a rule for whether to update wording, pointers, or both.

### Options

**Option A: Rewrite issue notes as purely fork-current**

- Pros:
  - Maximum consistency with the new fork policy.
- Cons:
  - Can erase historical provenance and overstate what the fork currently guarantees.

**Option B: Preserve provenance wording, but update actionable references**

- Pros:
  - Keeps the notes useful and honest.
- Cons:
  - Leaves some intentional upstream wording in active files.

**Option C: Treat all issue notes as exceptions**

- Pros:
  - Lowest risk of historical distortion.
- Cons:
  - Leaves active docs inconsistent with the new policy.

### Proposal (Recommended)

Take **Option B**. Update actionable CAO source/install references to the fork policy, but keep explicit provenance wording when the note is documenting an upstream-derived observation.

### Pros / Cons (Proposal)

- Pros:
  - Balances usability and accuracy.
- Cons:
  - Requires file-by-file editorial judgment.

## Q4) Should governance/security/release links move to the fork too, or stay explicit?

- Decision: `Blocking`

### Why this matters

- Normal operational links such as clone/install/docs/raw-content should clearly move to the fork.
- Security, advisory, and release links may represent governance ownership rather than ordinary navigation.

### Examples

- `README.md` and provider docs clearly need fork-owned repository/docs/install links.
- Security/advisory/release surfaces may need a separate decision before they can be rewritten safely.

### Why this question arises in code

- The change wants a fork-owned CAO story, but not every link inside CAO-facing docs is the same kind of link.
- A blanket rewrite could accidentally promise fork-owned support processes that do not exist yet.

### Options

**Option A: Move every link to the fork**

- Pros:
  - Most visually consistent.
- Cons:
  - Risks misrouting support or vulnerability reporting.

**Option B: Split operational links from governance/support links**

- Pros:
  - Keeps everyday usage aligned with the fork while avoiding unsupported promises.
- Cons:
  - Less uniform.

**Option C: Leave governance/support docs untouched**

- Pros:
  - Lowest risk.
- Cons:
  - Makes the migration feel incomplete.

### Proposal (Recommended)

Take **Option B**. Rewrite operational source/install/docs links to the fork, but treat governance/security/release ownership as an explicit sub-decision rather than rewriting those links automatically.

### Pros / Cons (Proposal)

- Pros:
  - Avoids over-claiming fork ownership.
  - Still gives operators a coherent fork-backed usage story.
- Cons:
  - Requires reviewers to accept intentional asymmetry in some docs.

### Notes / Follow-ups

- If Option B is chosen, implementation/review should explicitly note which governance/support links stay unchanged and why.
