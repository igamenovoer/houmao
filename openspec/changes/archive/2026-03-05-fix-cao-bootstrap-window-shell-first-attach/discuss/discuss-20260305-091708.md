# Discussion: CAO bootstrap window cleanup + first attach UX

- Feature/Change: `fix-cao-bootstrap-window-shell-first-attach`
- Timestamp: `2026-03-05T09:17:11Z`
- Context: Reviewed change artifacts plus current CAO backend implementation to enumerate remaining decisions needed before implementing bootstrap-window pruning and “first attach lands on agent window” behavior.

Key constraints / goals:
- Preserve existing tmux env propagation semantics (`AGENTSYS_MANIFEST_PATH` published in tmux env).
- Avoid upstream CAO API changes; runtime behavior must be robust even if cleanup fails.
- Prefer no extra CAO REST calls during startup unless they materially reduce risk.

---

## Summary of recommended defaults

1) Record the bootstrap tmux window by `window_id` immediately after `tmux new-session` (Q1).
2) Use `create_terminal(...)` response `terminal.name` as the CAO agent tmux window name; resolve it to a `window_id` via `tmux list-windows` (bounded retry) (Q2, Q3).
3) Best-effort `tmux select-window` to the agent window before pruning so first attach lands on it even if pruning fails (Q2).
4) Surface cleanup failures as a `warning:` line on stderr (keep `start-session` JSON stable) (Q4).
5) Keep tmux cleanup helpers colocated in `backends/cao_rest.py` for now; revisit after `unify-headless-tmux-backend` refactor lands (Q5).

## Q1) How should we record the bootstrap tmux window identity?

- Decision: `Blocking`

### Why this matters

- The runtime must not assume bootstrap is “window 0”; tmux `base-index` is configurable, and users can also rename windows.
- Pruning must be targeted: killing the wrong window could terminate the agent terminal window and break the session.

### Examples

- A user has `set -g base-index 1` in tmux config. The bootstrap window is index `1`, not `0`.
- A session contains windows with duplicate names (possible in tmux); selecting/killing by name may be ambiguous.

### Why this question arises in code

- CAO backend currently creates the session via `tmux new-session` and does not capture window identity: `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1184` → `_create_tmux_session` at `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1399`.
- `_create_tmux_session` does not set a deterministic bootstrap window name and returns no metadata: `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1399`.

### Options

**Option A: Record the bootstrap `window_id` via `tmux list-windows` (recommended)**

- Code impact (pseudo code, if appropriate):
```python
# Immediately after tmux new-session
bootstrap = tmux_list_windows(session, format="#{window_id}|#{window_name}|#{window_index}")[0]
bootstrap_window_id, bootstrap_name, bootstrap_index = bootstrap.split("|", 2)
```
- User-facing difference:
  - No user-visible change; this only affects which window is pruned/selected.
- Pros:
  - Robust to `base-index` and window renames.
  - `window_id` (`@123`) is unambiguous and safe to target for `kill-window`.
- Cons:
  - Requires at least one extra tmux call during startup (`list-windows`).

**Option B: Set a deterministic bootstrap window name (e.g. `-n agent-bootstrap`) and target by name**

- Code impact (pseudo code, if appropriate):
```python
tmux new-session -d -s <session> -n agent-bootstrap -c <workdir>
tmux kill-window -t <session>:agent-bootstrap
```
- User-facing difference:
  - If pruning fails, the leftover window name is recognizable (`agent-bootstrap`) instead of the default shell name.
- Pros:
  - Useful diagnostics; easier to recognize leftover bootstrap windows.
- Cons:
  - Still ambiguous if name collisions occur (less likely, but possible).
  - Adds a new visible naming convention (even if best-effort pruned).

**Option C: Assume bootstrap is the “first window” by index**

- Code impact: trivial (`kill-window -t session:0` / `session:1`).
- User-facing difference: none (when it works); catastrophic if wrong.
- Pros:
  - No extra tmux call.
- Cons:
  - Known brittle; contradicts change design constraints (tmux index config is variable).

### Proposal (Recommended)

Adopt Option A: record `window_id` immediately after `_create_tmux_session`, and keep `window_name`/`window_index` only for diagnostics. Do not rely on index for correctness. Optionally consider renaming bootstrap window later if diagnostics are insufficient.

### Pros / Cons (Proposal)

- Pros:
  - Most robust targeting mechanism available in tmux.
  - Keeps implementation independent of user tmux configuration.
- Cons:
  - Adds a tmux introspection call.

> **DECISION: Use bootstrap `window_id` as the pruning identity (store name/index only for diagnostics).**
> Rationale: CAO startup currently creates tmux sessions via `_create_tmux_session` without capturing a window handle (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1184`, `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1399`); `window_id` targeting avoids brittle index assumptions and makes later `kill-window` unambiguous.

## Q2) How should we resolve/select the CAO agent tmux window?

- Decision: `Blocking`

### Why this matters

- Selecting the agent window as current fixes the “shell-first attach” UX even if pruning fails.
- Pruning must ensure the recorded bootstrap window is not the agent window.

### Examples

- CAO returns `terminal.name="gpu-kernel-coder_abcd-1a2b"`. The runtime must select the corresponding tmux window so `tmux attach -t AGENTSYS-gpu` lands there.
- If the CAO window name is not yet visible to `tmux list-windows` due to a race, selection/prune logic needs a defined fallback.

### Why this question arises in code

- CAO terminal creation already returns a `Terminal` object with `name` (`src/gig_agents/cao/rest_client.py:122`) that corresponds to CAO’s tmux window name (vendored API: `extern/orphan/cli-agent-orchestrator/docs/api.md` “Terminal object”).
- The runtime currently throws away `terminal.name` and returns only `terminal.id`: `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1192`–`1202`.

### Options

**Option A: Use `terminal.name` directly as a tmux target**

- Code impact (pseudo code, if appropriate):
```python
tmux select-window -t f"{session_name}:{terminal.name}"
```
- User-facing difference:
  - First attach lands on the agent window when `select-window` succeeds.
- Pros:
  - Simple; no need to map name → id.
- Cons:
  - Requires careful escaping/quoting if CAO ever emits window names with spaces or special characters.
  - If multiple windows share the same name, name targeting can be ambiguous.

**Option B: Resolve `terminal.name` to `window_id` via `tmux list-windows` and select by id (recommended)**

- Code impact (pseudo code, if appropriate):
```python
agent_window_id = find_window_id_by_name(session_name, terminal.name)
tmux select-window -t agent_window_id
```
- User-facing difference:
  - First attach lands on the agent window more reliably (unambiguous targeting).
- Pros:
  - Unambiguous tmux targeting (`window_id`).
  - Naturally pairs with Q1 (id-based bootstrap targeting).
- Cons:
  - Adds a tmux introspection call (and perhaps retry logic).

**Option C: Call `GET /terminals/{id}` to resolve name, then proceed with Option A/B**

- Code impact:
  - Adds extra CAO REST call during startup.
- User-facing difference:
  - None, except potential latency/extra failure mode on startup.
- Pros:
  - Could help if `create_terminal(...)` response shape drifts.
- Cons:
  - Conflicts with current runtime “shadow_only must not call GET /terminals/{id} status” expectations in tests (see `tests/unit/agents/brain_launch_runtime/test_cao_client_and_profile.py:473`–`477` for the strictness around `shadow_only` flows).

### Proposal (Recommended)

Adopt Option B: treat `create_terminal(...)` response `terminal.name` as the window name, but select/prune by `window_id` after resolving via `tmux list-windows` (with bounded retry per Q3).

### Pros / Cons (Proposal)

- Pros:
  - Unambiguous and robust tmux targeting.
  - Avoids adding an extra CAO REST round-trip.
- Cons:
  - Slightly more startup complexity (name→id resolution).

> **DECISION: Resolve CAO `terminal.name` to tmux `window_id`, then select/prune by `window_id`.**
> Rationale: `create_terminal` already returns a typed `CaoTerminal` including `name` (`src/gig_agents/cao/rest_client.py:122`), while CAO startup currently discards the name and only returns `terminal.id` (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1192`, `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1202`); using the create response avoids an extra CAO `GET /terminals/{id}` dependency and stays compatible with current shadow-only expectations (`tests/unit/agents/brain_launch_runtime/test_cao_client_and_profile.py:473`).

## Q3) Do we need bounded retry/polling to resolve the CAO window before selection/prune?

- Decision: `Blocking`

### Why this matters

- Without retry, a transient race can silently skip selection/prune, leaving the original “shell-first attach” behavior intact.
- Too much waiting can slow down startup; too little can be flaky under load.

### Examples

- `create_terminal(...)` returns successfully but `tmux list-windows -t <session>` briefly shows only the bootstrap window; selection/prune fails if attempted once.
- CAO server is under load; the tmux window exists but takes a moment to appear in `list-windows` output.

### Why this question arises in code

- Startup sequence is a mix of local tmux calls and remote CAO calls: `_start_terminal` in `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1159`–`1202`.
- There is currently no retry/backoff scaffolding in startup (contrast with the polling loops used for turn readiness/completion elsewhere in the same backend).

### Options

**Option A: No retry; single best-effort attempt**

- Code impact:
  - One `list-windows` call; if resolution fails, skip selection/prune and warn.
- User-facing difference:
  - Under races, first attach may still land on bootstrap shell window.
- Pros:
  - Minimal complexity and fastest startup.
- Cons:
  - Risk of flakiness and non-deterministic attach UX.

**Option B: Bounded retry with short sleeps (recommended)**

- Code impact (pseudo code, if appropriate):
```python
for _ in range(10):  # e.g. 10 * 0.1s = 1s total
    agent_window_id = find_window_id_by_name(...)
    if agent_window_id:
        break
    time.sleep(0.1)
```
- User-facing difference:
  - More reliable “first attach lands on agent window”, at the cost of ≤O(1s) additional startup delay in the rare retry path.
- Pros:
  - Significantly reduces flake risk without materially impacting normal-case startup.
- Cons:
  - Adds sleep/retry behavior that needs deterministic unit tests (fake clock or injected sleep).

**Option C: Wait for “terminal ready” via CAO API before cleanup**

- Code impact:
  - Additional CAO polling during startup (for status or output readiness).
- User-facing difference:
  - Potentially longer startup; additional failure modes.
- Pros:
  - Couples cleanup timing to “terminal is actually alive”.
- Cons:
  - Risks entangling cleanup with parsing-mode-specific readiness semantics.

### Proposal (Recommended)

Adopt Option B: bounded retry to resolve `terminal.name` → `window_id` (for selection and prune safety), but keep it short and deterministic (small fixed attempt count and sleep interval). If the agent window cannot be resolved within the bound, keep session start successful, emit a warning, and skip pruning.

### Pros / Cons (Proposal)

- Pros:
  - Stabilizes attach UX and reduces chance of “no-op cleanup”.
  - Keeps best-effort, non-fatal semantics intact.
- Cons:
  - Adds timing behavior that must be tested carefully.

> **DECISION: Add bounded retry (short fixed attempts + sleep) for CAO window resolution before select/prune.**
> Rationale: `_start_terminal` currently has no retry phase between `create_terminal(...)` and return (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1159`), so a transient tmux visibility race would skip cleanup; bounded retry improves determinism while preserving non-fatal startup behavior.

## Q4) How should cleanup failures be surfaced to users/operators?

- Decision: `Deferrable`

### Why this matters

- Cleanup is best-effort; operators still need visibility when it fails so they can diagnose unexpected two-window sessions.
- We should avoid breaking `start-session` JSON output contracts without a clear need.

### Examples

- `tmux kill-window` fails because the window was already closed; user should not see a hard failure, but a warning would be useful.
- `tmux select-window` fails because the agent window couldn’t be resolved within the bound; attach UX may regress and should be diagnosable.

### Why this question arises in code

- The `start-session` CLI already prints warnings for agent identity normalization to stderr while keeping JSON output stable: `src/gig_agents/agents/brain_launch_runtime/cli.py:208`–`210`.
- CAO backend currently has no logging and no “startup events” output path comparable to `send-prompt` events.

### Options

**Option A: `logging.warning(...)` only**

- Code impact:
  - Add a module logger and emit warnings on cleanup failure paths.
- User-facing difference:
  - Depends on runtime logging configuration; may be invisible in default CLI usage.
- Pros:
  - Minimal surface change.
- Cons:
  - Easy to miss warnings in practice.

**Option B: Print a `warning:` line to stderr during `start-session` (recommended)**

- Code impact:
  - Ensure cleanup helpers return a structured result that the CLI can surface as a warning, or print directly from backend with a consistent prefix.
- User-facing difference:
  - Users immediately see “cleanup failed” without breaking the JSON output.
- Pros:
  - Mirrors existing CLI warning conventions.
- Cons:
  - Requires plumbing warning text across the backend/CLI boundary (or accepting backend stderr printing).

**Option C: Add a structured field to `start-session` JSON output**

- Code impact:
  - Extend CLI output payload (breaking/compat risk, even in unstable repo).
- User-facing difference:
  - Machine-readable cleanup status in the main output.
- Pros:
  - Best for automation.
- Cons:
  - Output contract churn; requires updating any downstream consumers/tests.

**Option D: Persist cleanup status in the session manifest**

- Code impact:
  - Extend manifest schema/payload with a diagnostic field.
- User-facing difference:
  - Indirect; users must inspect manifest to see status.
- Pros:
  - Great for post-mortems.
- Cons:
  - Manifest schema churn for a best-effort cosmetic issue.

### Proposal (Recommended)

Adopt Option B: surface cleanup failures as `warning:` lines on stderr and keep `start-session` JSON unchanged. Treat adding structured JSON/manifest fields as a follow-up only if operational needs demand machine-readability.

### Pros / Cons (Proposal)

- Pros:
  - Immediate operator feedback with minimal contract churn.
- Cons:
  - Requires deciding how to plumb warnings (backend vs CLI ownership).

> **DECISION: Surface cleanup failures as `warning:` lines on stderr and keep `start-session` JSON unchanged.**
> Rationale: The CLI already has an established stderr warning pattern (`for warning in controller.agent_identity_warnings`) and keeps JSON payload stable (`src/gig_agents/agents/brain_launch_runtime/cli.py:206`, `src/gig_agents/agents/brain_launch_runtime/cli.py:209`), so this path gives operator visibility without output-contract churn.

## Q5) Where should tmux cleanup helper logic live given `unify-headless-tmux-backend` work?

- Decision: `Deferrable`

### Why this matters

- We want to avoid duplicating tmux primitives across backends, but we also don’t want to block this narrow CAO fix on a larger refactor.

### Examples

- `unify-headless-tmux-backend` proposes broader shared tmux session contracts; it may introduce a shared utility module that CAO cleanup could naturally use.

### Why this question arises in code

- CAO backend owns tmux session creation and env publication today in `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py` (`_create_tmux_session`, `_set_tmux_session_environment`).
- There is an active OpenSpec change for unifying tmux-backed headless behavior: `openspec/changes/archive/2026-03-05-unify-headless-tmux-backend/`.

### Options

**Option A: Implement cleanup helpers in `cao_rest.py` now (recommended)**

- Code impact:
  - Add small private helpers near existing tmux functions.
- User-facing difference: none (aside from improved attach UX).
- Pros:
  - Fastest path; minimal coordination.
- Cons:
  - Likely some duplication once headless tmux utilities land.

**Option B: Introduce a shared tmux utility module now and use it from CAO backend**

- Code impact:
  - New module + refactor CAO backend to call into it.
- User-facing difference: none.
- Pros:
  - Sets the stage for headless unification.
- Cons:
  - Scope creep; higher risk and review cost for this small change.

**Option C: Block this change until `unify-headless-tmux-backend` lands**

- Pros:
  - Avoids duplication.
- Cons:
  - Delays a user-facing correctness fix with no direct dependency.

### Proposal (Recommended)

Adopt Option A: implement the cleanup helpers in `cao_rest.py` for this change. Treat shared tmux utility refactoring as follow-up work tied to `unify-headless-tmux-backend` (where duplication can be retired intentionally).

### Pros / Cons (Proposal)

- Pros:
  - Keeps this fix small and reviewable.
- Cons:
  - Accepts short-term duplication.

> **DECISION: Implement tmux cleanup helpers in `cao_rest.py` now; defer shared helper extraction to `unify-headless-tmux-backend`.**
> Rationale: Existing tmux lifecycle functions for CAO already live in `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1399` and `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1422`; colocating this change minimizes scope while preserving a clean future refactor point under `openspec/changes/archive/2026-03-05-unify-headless-tmux-backend/`.
