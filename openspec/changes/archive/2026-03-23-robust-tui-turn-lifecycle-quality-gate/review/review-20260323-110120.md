- **Solid problem identification**: The three bugs (stale `last_turn` carry, Claude status scope bleed, Claude prompt-marker style degradation) are well-evidenced by the existing code in `session.py` (lines 364-376 only clear success, not interrupted/known_failure) and `prompt_behavior.py` (lines 231-265 where non-dim/non-inverse SGR codes set `active_other_style = True`).
- **Well-structured tasks**: The 4 task groups follow a logical dependency chain (shared fix → Claude fix → fixtures → docs/verify) and map cleanly to the 4 delta specs.

- **Risk: "Latest-turn region" is under-specified**: Task 2.1 introduces a fundamentally new concept (turn-region scoping) with no design detail on _how_ Claude's latest-turn boundary is derived from pane content. This is the highest-risk area.

## Findings

### Must Fix / Blocking

1. **The design does not specify how Claude derives the "latest-turn region" from pane content.** The design (lines 46-56) and the `versioned-tui-signal-profiles` delta spec (scenario: "Claude status interpretation can ignore stale transcript terminal text") both say the profile should "derive the latest turn region" and "scope terminal evidence" to it, but neither describes the algorithm. The current Claude detector in `apps/claude_code/profile.py` has no concept of turn regions — `INTERRUPTED_RE` (line 25: `r"^⎿\s+Interrupted · What should Claude do instead\?$"`) matches on any visible line. Implementing region scoping requires either (a) tracking the prompt-marker line position as a region boundary, (b) scanning for Claude-specific turn delimiters in the pane text, or (c) using the existing `PromptAreaSnapshot.region_lines` from `build_prompt_area_snapshot()` (`prompt_behavior.py`, lines 120-154) as a spatial anchor. Without specifying the approach, Task 2.1 is ambiguous enough to produce divergent implementations or to become blocked during coding.

> **DECISION: Accept.**
> Rationale: The reviewer identified a real artifact gap. The later revision should specify a repo-native, stateless boundary rule built on the existing prompt anchor helpers in [surface.py](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/surface.py) and [prompt_behavior.py](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/apps/claude_code/signals/prompt_behavior.py), rather than leaving turn-region derivation implicit.

2. **The shared reducer's stale-clear trigger list needs explicit definition beyond prose.** The `official-tui-state-tracking` delta spec lists three kinds of "authoritative newer-turn evidence" (visible non-placeholder draft, explicit input submission, newer active-turn evidence), but the design and tasks don't specify _where_ in the reducer's signal-processing flow these checks fire. Currently, success invalidation lives at `session.py` lines 364-376, gated on `surface_signature` change. The interrupted/known_failure clearing could be added (a) in the same `surface_signature` block, (b) as a separate guard in `_process_signals_locked()`, or (c) as a new `_invalidate_stale_terminal_locked()` helper. The choice matters for test structure and for avoiding interference with the settle timer. The design should state explicitly which signals constitute "authoritative newer-turn evidence" in code-level terms (e.g., `editing_input == "yes"` when `last_turn_result != "none"`, or `active_evidence == True` after a terminal state).

> **DECISION: Accept.**
> Rationale: The semantic trigger set is present, but the artifact set does not yet map those triggers onto the actual session code paths. The revision should define a shared stale-terminal invalidation helper and say explicitly that explicit-input authority and snapshot-driven newer-turn authority both use it, instead of burying this inside success-specific settle logic.

### Should Fix / Important

3. **Task 2.2 scope is under-described relative to the actual SGR parsing code.** The design says prompt-marker color and `39`/`49` resets should be neutral, but `_apply_sgr_codes()` (`prompt_behavior.py`, lines 231-265) already treats codes 39 and 49 as neutral (they're handled explicitly and don't set `active_other_style`). The real issue is that _other_ foreground/background color codes (e.g., `38;5;N` palette sequences used by Claude's prompt marker `❯`) fall into the catch-all `else` branch and set `active_other_style = True`, which makes `payload_plain_flags` entries `False` for real draft characters that appear after those codes. The task should explicitly identify which additional SGR code families (256-color palette `38;5;N`, RGB `38;2;R;G;B`, basic `30-37`/`40-47`/`90-97`/`100-107`) should be treated as neutral for prompt-payload classification, versus which should preserve the `active_other_style` flag.

> **DECISION: Refine.**
> Rationale: The reviewer is right about the substance of the bug, but the artifact change should stay one level above a verbatim SGR table. The revision should say that foreground/background color-setting families and their resets are neutral for Claude prompt payload classification, while dim, inverse, and still-unrecognized non-color styles remain meaningful and are verified in code/tests.

4. **The complex fixture's 8-step lifecycle includes "active-draft hold while the tool is still running" (steps 3 and 6), but no task explicitly covers authoring guidance for inducing overlapping active-draft visibility.** Codex and Claude differ in how they display the prompt area while a turn is active. Claude's current `build_prompt_area_snapshot()` uses a `_PROMPT_REGION_RADIUS = 2` (`prompt_behavior.py`, line 15) to extract region context. If the active-turn output scrolls the prompt marker off-screen, the "active-draft" span may not be observable. Task 4.1 mentions documenting hold durations but not the spatial/scroll constraint for active-draft visibility.

> **DECISION: Accept.**
> Rationale: This is a real authoring reproducibility gap. The complex fixture recipe needs an explicit “keep the prompt region visible while overlapping active draft is sampled” note in the maintainer guidance, not just hold durations, because the tracker’s prompt classifiers only work when the prompt anchor remains on screen.

5. **The ordered sweep contract `ready_success -> active -> ready_interrupted -> active -> ready_interrupted -> ready_success` does not include the `ready_draft` phases that the spec explicitly requires to be labeled.** The `shared-tui-tracking-recorded-validation` delta spec (lines 28-33) requires judging `last_turn.result=none` during "both ready-draft spans" and `surface.editing_input=yes` during "both active-draft spans." The sweep contract as stated in the design (line 99) collapses those into `active`, which would not catch a regression where the draft span is missing but the active turn begins correctly. Consider whether the sweep contract should be `ready_success -> ready_draft -> active -> ready_interrupted -> ready_draft -> active -> ready_interrupted -> ready_success` to match the labeled lifecycle.

> **DECISION: Reject.**
> Rationale: The ordered sweep is intentionally a coarse cadence-robustness gate, not the primary oracle for every labeled span. `ready_draft` and `active_draft` correctness are already judged by the strict per-sample ground-truth labels, and expanding [config.py](/data1/huangzhe/code/houmao/src/houmao/demo/shared_tui_tracking_demo_pack/config.py) with new sweep labels would widen this change beyond its intended scope.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should the Claude detector derive the "latest-turn region" boundary from pane content?

- Decision: `Blocking`

#### Why this matters

- The entire Task 2.1 (Claude status scoping) depends on this decision. Without it, the implementer must invent the algorithm, which risks over- or under-scoping.

#### Examples

- A pane showing two turns: first turn's `⎿ Interrupted` line at row 5, second turn's prompt `❯` marker at row 15 with active thinking at rows 16-20. The "latest-turn region" should include rows 15-20 and exclude row 5. But how is row 15 identified as the turn boundary?
- A pane where the latest prompt marker has scrolled off-screen and only the turn's output remains visible. There is no prompt marker to anchor the region.

#### Why this question arises in code

- `_BaseClaudeCodeSignalDetector.detect()` (`profile.py`, lines 76-248) currently scans all lines for `INTERRUPTED_RE` without region filtering.
- `build_prompt_area_snapshot()` (`prompt_behavior.py`, lines 120-154) already locates the prompt marker line and extracts a ±2-line region, but this is used only for prompt classification, not for status scoping.
- The detector's `detect()` method receives a full `TuiRawSnapshot` with all visible lines. There's no existing turn-boundary index.

#### Options

**Option A: Use the last prompt marker (`❯`) position as the turn region boundary.**

- Code impact:
  - Add a helper to find the last `❯` line index in the pane text.
  - Filter `INTERRUPTED_RE` and `known_failure` matching to lines at or below that index.
  ```python
  def _latest_turn_start_line(lines: list[str]) -> int | None:
      for i in range(len(lines) - 1, -1, -1):
          if "❯" in lines[i]:
              return i
      return None
  ```
- User-facing difference:
  - Stale `Interrupted` lines above the latest prompt stop affecting current turn state.
- Pros:
  - Simple, Claude-specific, uses existing UI element.
  - `❯` is already tracked by `build_prompt_area_snapshot()` so there's precedent.
- Cons:
  - If the prompt marker scrolls off-screen, the boundary is unknown and the detector must fall back to full-pane scanning or `unknown`.
  - Multiple `❯` characters in tool output could confuse the scan (though Claude's output style makes this unlikely).

**Option B: Track the last transition from non-prompt to prompt-visible state across snapshots.**

- Code impact:
  - Add lightweight inter-snapshot state to the detector (last known prompt line, or last known turn-start sample).
  - Requires `detect()` to carry forward a `_last_turn_boundary` field.
  ```python
  class _BaseClaudeCodeSignalDetector:
      def __init__(self, ...):
          self._last_turn_start_line: int | None = None
      def detect(self, snapshot: TuiRawSnapshot) -> DetectedTurnSignals:
          prompt_line = self._find_prompt_line(snapshot)
          if prompt_line is not None:
              self._last_turn_start_line = prompt_line
          region_start = self._last_turn_start_line or 0
          # scope INTERRUPTED_RE to lines[region_start:]
  ```
- User-facing difference:
  - Same as Option A for visible prompts; additionally handles the case where the prompt scrolls off-screen by remembering the last boundary.
- Pros:
  - Handles prompt-off-screen edge case.
  - More robust for long-running turns with extensive output.
- Cons:
  - Breaks the current stateless-per-snapshot pattern of `detect()`.
  - Adds complexity to the detector lifecycle (must be reset across sessions).
  - Inter-snapshot state is a new pattern for this module.

> **PROPOSED: Option A (prompt-marker position as region boundary) with a fallback to full-pane scanning when no prompt marker is visible.**
> Rationale: The existing `build_prompt_area_snapshot()` in `prompt_behavior.py` (lines 120-154) already locates the `❯` prompt marker as a spatial anchor. Reusing this approach for status scoping keeps the detector stateless per-snapshot, which matches every other detector method in both `claude_code/profile.py` and `codex_tui/profile.py`. The prompt-off-screen edge case is rare in the target interaction pattern (the complex fixture's interrupted turns happen quickly, before output scrolls the prompt away), and a full-pane fallback is no worse than the current behavior. Adding inter-snapshot state (Option B) introduces a new lifecycle concern without a clear test case motivating it.
>
> **DECISION: Refine PROPOSED. Use the last visible prompt marker as the stateless current-turn anchor, and when no prompt marker is visible, degrade conservatively instead of falling back to full-pane status scanning.**
> Rationale: [SurfaceView.last_prompt_index()](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/surface.py#L35) and [build_prompt_area_snapshot()](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/apps/claude_code/signals/prompt_behavior.py#L107) already use the prompt line as the repo-native spatial anchor, so the status scoping rule should reuse that pattern and stay per-snapshot/stateless. A full-pane fallback would reintroduce the exact stale `⎿ Interrupted` false positive this change is trying to remove; if no anchor is visible, the safer v1 posture is to avoid asserting terminal status from ambiguous transcript history.

### Q2) Should the `last_turn` stale-clear logic live alongside the existing success-invalidation block or as a separate guard?

- Decision: `Deferrable`

#### Why this matters

- The placement affects test isolation and the risk of interfering with the settle timer or re-entrant state emission.

#### Examples

- Scenario: `last_turn.result=interrupted`, operator types draft text → `editing_input=yes`. The stale-clear should fire and reset `last_turn` to `none`. Where in `_process_signals_locked()` does this check run relative to the interrupted-signal handler (line 293) and the success-invalidation block (line 364)?

#### Why this question arises in code

- `_process_signals_locked()` in `session.py` processes signals in a specific order: interrupted (line 293) → known_failure (line 312) → active evidence (line 331) → success invalidation (line 364) → ready posture (line 385) → success candidate (line 460). The stale-clear logic must fire before the signal handlers that could re-set `last_turn` in the same snapshot.

#### Options

**Option A: Extend the existing success-invalidation block (line 364) to also clear interrupted/known_failure.**

- Code impact:
  - Widen the `if` condition at line 364 to include `last_turn_result in ("interrupted", "known_failure")` and check for newer-turn evidence signals instead of only `surface_signature` change.
- User-facing difference:
  - None; same observable behavior.
- Pros:
  - Minimal code change; keeps all terminal-state clearing in one location.
- Cons:
  - The success-invalidation guard uses `surface_signature` comparison, which may not be the right trigger for interrupted/known_failure clearing (those should clear on `editing_input=yes` or `active_evidence=True`, not signature change alone).

**Option B: Add a new `_invalidate_stale_terminal_locked()` helper called early in `_process_signals_locked()`.**

- Code impact:
  - New method that checks `last_turn_result != "none"` and `editing_input == "yes" or active_evidence == True`, emits `last_turn_result="none"`.
  - Called before the interrupted/known_failure signal handlers.
- User-facing difference:
  - None; same observable behavior.
- Pros:
  - Clean separation of concerns; testable in isolation.
  - Clear about which signals trigger clearing (editing evidence, active evidence) versus which signals set terminal state (interrupted, known_failure, success).
- Cons:
  - Slightly more code; new helper to maintain.

> **PROPOSED: Option B (new `_invalidate_stale_terminal_locked()` helper called early in `_process_signals_locked()`).**
> Rationale: The success-invalidation block at `session.py` line 364 is gated on `surface_signature` comparison, which is a success-specific heuristic (the signature captures the "settled screen shape" of a successful turn). Interrupted/known_failure clearing should trigger on different evidence (draft visibility, active-turn signals), so mixing the two into one `if` block conflates distinct invariants. A separate helper called before the main signal handlers (before line 293) also avoids the risk of a snapshot both clearing stale state _and_ re-setting it from the same interrupted signal in one pass, which would be confusing to debug.
>
> **DECISION: Refine PROPOSED. Use a dedicated stale-terminal invalidation helper, but invoke it from both `_handle_input_event()` and the early snapshot-processing path rather than only from `_process_signals_locked()`.**
> Rationale: The separation-of-concerns point is correct, but explicit input submission is already one of the chosen authoritative newer-turn signals and it currently enters through [_handle_input_event()](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/session.py#L196), not the snapshot branch. The clean v1 shape is one shared helper that is reused by explicit-input and snapshot-driven newer-turn authority so the semantics stay aligned and the tests can cover both entry paths.

### Q3) Which SGR code families should be treated as neutral for Claude prompt-payload classification?

- Decision: `Deferrable`

#### Why this matters

- Task 2.2 says prompt-marker color and reset codes should not downgrade real draft text to `unknown`, but `_apply_sgr_codes()` has a catch-all `else` branch that sets `active_other_style = True` for any unrecognized code. The fix needs to decide which codes are "marker styling" (neutral) versus which are "meaningful styling" (should affect classification).

#### Examples

- Claude prompt marker `❯` is rendered with a foreground color (e.g., `\x1b[38;5;33m`). If the user types `hello` after the marker and the terminal emits `\x1b[39m` (reset foreground) before the user text, the `39` is already neutral but the `38;5;33` from the marker set `active_other_style = True`. Characters following inherit that flag until a code-0 reset.
- A hypothetical future Claude version uses bold (`\x1b[1m`) for the prompt marker. Should bold be neutral too?

#### Why this question arises in code

- `_apply_sgr_codes()` (`prompt_behavior.py`, lines 231-265) handles codes 0, 2, 7, 22, 27, 39, 49 explicitly. Everything else sets `active_other_style = True`. The foreground color codes `30-37`, `38;5;N`, `38;2;R;G;B`, and `90-97` (plus background equivalents) all fall into the catch-all.

#### Options

**Option A: Treat all foreground/background color codes as neutral (add explicit handling for `30-37`, `38`, `40-47`, `48`, `90-97`, `100-107`).**

- Code impact:
  - Expand `_apply_sgr_codes()` to recognize color-setting codes and ignore them (don't set `active_other_style`).
- Pros:
  - Comprehensive; handles marker colors, user-configured terminal themes, and future color changes.
- Cons:
  - Broadens neutrality — if a future Claude version uses color to distinguish placeholder from draft, this would miss it.

**Option B: Reset `active_other_style` on each new non-whitespace payload character, so only the style _at_ each character matters.**

- Code impact:
  - Change the iteration in `_prompt_payload_render_state()` to reset `active_other_style` per-character based on the SGR state at that character's position, rather than carrying forward from earlier codes.
- Pros:
  - Naturally scopes marker styling to the marker character and draft styling to draft characters.
- Cons:
  - Requires understanding which codes are "positional" versus "inherited" — most SGR codes are inherited until reset.

> **PROPOSED: Option A (treat all foreground/background color codes as neutral).**
> Rationale: The `_apply_sgr_codes()` function (`prompt_behavior.py`, lines 231-265) already explicitly neutralizes codes 39 and 49 (color resets). The meaningful styling distinction in Claude's prompt is dim (code 2) for placeholder vs. plain for draft. Foreground/background colors are terminal-theme and marker-styling artifacts, not semantic signals. Treating them as neutral is consistent with the existing dim/inverse/other classification model and avoids the complexity of per-character style scoping. The risk of a future Claude version using color as a semantic signal is low and can be addressed by a new profile version when it occurs.
>
> **DECISION: Accept PROPOSED. Treat foreground/background color-setting families and their resets as neutral for Claude prompt payload classification.**
> Rationale: The observed regression comes from Claude prompt-marker color sequences falling into the catch-all `active_other_style` path in [_apply_sgr_codes()](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/apps/claude_code/signals/prompt_behavior.py#L206), while the meaningful placeholder-vs-draft distinction remains dimness, not color. Neutralizing color-setting codes keeps the classifier aligned with the current UI without weakening the dim/inverse signals that already carry semantic weight.
