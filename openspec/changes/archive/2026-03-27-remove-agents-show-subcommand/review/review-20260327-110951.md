## Summary

- The change is well-scoped and the design rationale (remove CLI surface, preserve HTTP API and detail payload builders) is sound for a repository that explicitly welcomes breaking changes.
- The delta spec for `houmao-srv-ctrl-native-cli` is clean and correctly removes `show` from the minimum command list and adds a negative requirement against advertising it.
- **Blocking gap:** Two other main specs (`houmao-mgr-agents-join`, `houmao-mgr-registry-discovery`) explicitly reference `agents show`, and the current delta spec also leaves additional `agents show` references behind inside `houmao-srv-ctrl-native-cli`; implementing this change as written would leave the spec corpus contradictory.
- **Important gap:** The tasks do not mention updating OpenSpec main specs at all, and they also do not capture the repository-wide grep-sweep verification already called for by the design.
- The helper-removal blast radius is narrower than the review first suggested: for this change, `managed_agent_detail_payload` should remain in place for non-CLI callers such as the interactive full-pipeline demo.

## Findings

### Must Fix / Blocking

1. **Missing delta specs for `houmao-mgr-agents-join` and `houmao-mgr-registry-discovery`.**

   The main spec `openspec/specs/houmao-mgr-agents-join/spec.md` explicitly requires that `agents show` remains inspectable after a TUI join (lines 26 and 34–38), including a dedicated scenario "Joined TUI remains inspectable through later local state and detail commands" that uses `houmao-mgr agents show --agent-name tester`.

   The change only includes a delta spec for `houmao-srv-ctrl-native-cli`. Without companion delta specs for the other two capabilities, implementing this change would leave the main spec corpus internally contradictory: the native CLI spec says `show` is retired, but the join and registry-discovery specs still require it.

> **DECISION: _(accept / refine / reject / defer this finding)_**
> Rationale: Accept. `houmao-mgr-agents-join` and `houmao-mgr-registry-discovery` both still require `agents show`, so the change as written would leave the main spec corpus contradictory. ## Design Alignment With Existing System

- **Existing pattern observed:** The repository has previously retired CLI commands by removing them outright (e.g., `history` was excluded with a negative requirement at `openspec/specs/houmao-srv-ctrl-native-cli/spec.md:65`; the CAO command tree was retired and checked via `assert "cao" not in result.output` at `test_commands.py:171`). The proposed approach follows this established pattern: remove the command, add a negative requirement, update tests and docs.
- **Proposed design alignment:** The change aligns well with the existing retirement pattern. The one gap is that previous retirements appear to have been smaller in cross-spec footprint; the `show` command is referenced in three separate main specs, which makes the multi-spec update requirement novel for this repo. The delta-spec mechanism should handle this cleanly once the missing specs are added.
- **HTTP API preservation pattern:** Keeping the server detail endpoint while removing the CLI surface matches the existing architecture where CLI commands are thin wrappers over server or local helpers. The server route, client methods, and pair-client methods form a coherent API layer independent of the CLI, and preserving them is the right call.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should `managed_agent_detail_payload()` be preserved as a public helper or removed as CLI dead code?

- Decision: `Blocking`

#### Why this matters

The function is the bridge between the CLI `show` command and the local detail-building logic. Removing it as dead code is the clean approach, but one demo pack (`houmao_server_interactive_full_pipeline_demo`) imports it directly, and the function also serves as a convenient local-path equivalent of the server's `get_managed_agent_state_detail()`.

#### Examples

- If removed: `houmao_server_interactive_full_pipeline_demo/commands.py:675` breaks at import time. The demo must be migrated to use the server client or a replacement local helper.
- If preserved: the function stays as an orphaned public API with no CLI caller, which may confuse future readers unless documented as "for programmatic use only."

#### Why this question arises in code

- `src/houmao/srv_ctrl/commands/managed_agents.py:317` defines `managed_agent_detail_payload()`
- `src/houmao/srv_ctrl/commands/agents/core.py:348` is the only CLI caller
- `src/houmao/demo/houmao_server_interactive_full_pipeline_demo/commands.py:54,675` is a non-CLI caller
- The two private helpers `_local_tui_detail_response_from_state` and `_local_headless_detail_response` are only reachable through this function

#### Options

**Option A: Remove `managed_agent_detail_payload` and migrate the demo**

- Code impact:
  - Delete `managed_agent_detail_payload`, `_local_tui_detail_response_from_state`, `_local_headless_detail_response` from `managed_agents.py`
  - Migrate `houmao_server_interactive_full_pipeline_demo` to use `HoumaoServerClient.get_managed_agent_state_detail()` (it already has a server client in scope)
  - Remove unit tests for `managed_agent_detail_payload` in `test_managed_agents.py`
- User-facing difference:
  - No observable difference; demo already runs against a server
- Pros:
  - Clean dead-code removal; no orphaned public APIs
- Cons:
  - Slightly larger change scope; demo migration is a small but real task

**Option B: Preserve `managed_agent_detail_payload` as a non-CLI public helper**

- Code impact:
  - Remove only `show_agent_command` and its import from `core.py`
  - Keep `managed_agent_detail_payload` and its helpers in `managed_agents.py`
  - Keep existing unit tests
- User-facing difference:
  - None
- Pros:
  - Smaller change; demo code untouched
- Cons:
  - Leaves a public function with no CLI caller, potential confusion about its purpose

> **PROPOSED: Option A — remove `managed_agent_detail_payload` and migrate the demo.**
> Rationale: The proposal explicitly says the server detail HTTP API is preserved, and the demo can use the server client method that maps to the same endpoint. Keeping a CLI-path-only helper with no CLI caller violates the existing pattern where `managed_agents.py` functions serve the CLI command layer (see `managed_agent_state_payload` → `state_agent_command`, `prompt_managed_agent` → `prompt_agent_command`). Option A keeps that 1:1 mapping clean. The demo migration is mechanical — `commands.py:675` already operates in a context where a server client is available.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Option B — preserve `managed_agent_detail_payload()` as a non-CLI helper for now. The user-requested scope is to remove the `agents show` subcommand, not to remove every detail-building helper. The current proposal and design already preserve the underlying detail API and internal detail payload helpers, and the interactive full-pipeline demo imports `managed_agent_detail_payload()` directly. Keeping the helper avoids widening this change into demo migration and unrelated unit-test churn while still removing the unwanted CLI surface.

### Q2) Should the registry-discovery spec scenarios be updated to use `agents state` or a generic placeholder like `agents <command>`?

- Decision: `Deferrable`

#### Why this matters

The `houmao-mgr-registry-discovery` spec uses `agents show` as the example command in five scenarios. These scenarios test registry resolution behavior, not the `show` command itself. Replacing `show` with `state` is the simplest fix, but those scenarios could also use a generic placeholder to avoid coupling registry-discovery spec maintenance to future command additions/removals.

#### Examples

- With `state`: `WHEN an operator runs houmao-mgr agents state <agent_ref>` — concrete and testable
- With generic: `WHEN an operator runs a post-launch houmao-mgr agents command for <agent_ref>` — more abstract, harder to validate

#### Why this question arises in code

- `openspec/specs/houmao-mgr-registry-discovery/spec.md` lines 17, 23, 57, 104, 118
- The scenarios are about registry discovery mechanics, not inspection semantics

#### Options

**Option A: Replace with `agents state`**

- Code impact:
  - Five line edits in `houmao-mgr-registry-discovery/spec.md`
- User-facing difference:
  - None; spec remains concrete and grep-verifiable
- Pros:
  - Simple, concrete, testable
- Cons:
  - If `state` is ever renamed/removed, the same update is needed again

**Option B: Use a generic command placeholder**

- Code impact:
  - Rewrite scenario WHEN clauses to use generic phrasing
- User-facing difference:
  - Spec is slightly more abstract
- Pros:
  - Future-proof against command renames
- Cons:
  - Harder to map to real CLI invocations; breaks the spec convention of using concrete commands in scenarios

> **PROPOSED: Option A — replace with `agents state`.**
> Rationale: Existing spec scenarios throughout the repo consistently use concrete CLI commands in WHEN clauses (e.g., `houmao-mgr agents state --agent-id abc123` at `houmao-srv-ctrl-native-cli/spec.md:68`, `houmao-mgr agents prompt` at line 82). Using a generic placeholder would break that convention. `state` is the surviving transport-neutral inspection command and the natural replacement. The five edits are mechanical and should be captured in the delta spec for this change.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Option A — replace the registry-discovery `agents show` scenarios with `agents state`. Repo specs consistently use concrete CLI commands in scenario WHEN clauses, and `state` is the surviving transport-neutral inspection command. That keeps the scenarios concrete, testable, and aligned with the post-removal CLI surface.
