## Summary

- **Solid framing:** The three-tier reliability model (lifecycle / operator-diagnostic / machine-critical) is the right architectural clarification and is well-motivated by the actual divergent consumer needs visible in `_TurnMonitor.observe_completion()`, the interactive demo inspect surface, and mailbox sentinel parsing.
- **Good existing alignment:** The shadow-first posture is already the runtime default in `launch_plan.py` (Claude and Codex both default to `"shadow_only"`), and `done.message` for shadow mode is already the fixed string `"prompt completed"` — so the change correctly targets the demo/doc/test layers that still scrape `done.message` as if it were reply text.
- **Projector abstraction is well-scoped:** Decomposing `_build_dialog_projection()` and `_project_line()` out of the monolithic parser classes into swappable processor instances is a natural and bounded refactor given the existing `ShadowParserStack` adapter layer.
- **Risk: Projector abstraction boundary is underspecified** — the specs and design leave several concrete interface questions open (Protocol vs ABC, what the projector owns, how `projector_id` composes with existing `ProjectionMetadata`), which could lead to implementation churn.
- **Risk: No spec or task covers a shared schema-extraction helper** — the design's open question about whether to build one is important but has no corresponding spec or task, leaving a gap if the answer is "yes."
- **Minor: Task ordering may benefit from leading with contract clarification** (Task 2) before the structural refactor (Task 1) to reduce rework risk.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What should the projector abstraction own — just line-level filtering, or the full projection pipeline including normalization and head/tail slicing?

- Decision: `Blocking`

#### Why this matters

- The projector's ownership boundary determines how much of the existing `_build_dialog_projection()` method moves into the projector vs stays in the parser class. This affects interface design, testability, and how much the parser class changes.

#### Examples

- If projectors own only line filtering: the parser still owns normalization, line iteration, and head/tail slicing. Projectors become simple `(line: str) -> str | None` callables. Easy to swap, but limited modularity — layout-level changes (e.g., multi-line block detection) can't be swapped.
- If projectors own the full pipeline: the parser only calls `projector.project(normalized_text) -> DialogProjection`. More modular, but the projector must replicate metadata construction and head/tail slicing, which are currently provider-independent.

#### Why this question arises in code

- `claude_code_shadow.py:392–470`: `_build_dialog_projection()` does normalization, line iteration with `_project_line()`, metadata construction, and head/tail slicing in one method. The line-filtering logic (`_project_line()` at line 435) is the provider-specific part; the rest is shared boilerplate.
- `codex_shadow.py:335–410`: Same structure, different `_project_line()` rules.
- Both parsers share `projection_head_tail()` from `shadow_parser_core.py` for head/tail slicing.

#### Options

**Option A: Projectors own only line-level filtering (`project_line(line) -> str | None`)**

- Code impact:
  - The parser class retains `_build_dialog_projection()` but delegates the per-line decision to `self._projector.project_line(line)`.
  - ```python
    class DialogProjector(Protocol):
        def project_line(self, line: str) -> str | None: ...
        @property
        def projector_id(self) -> str: ...
    ```
  - Minimal structural change; existing metadata/head-tail logic stays in the parser.
- User-facing difference:
  - None; same `DialogProjection` output.
- Pros:
  - Smallest interface, easiest to test, lowest risk of breaking existing pipeline.
  - Head/tail slicing and metadata construction remain shared and don't need duplication.
- Cons:
  - Multi-line or block-level projection rules (e.g., collapsing consecutive blank lines, detecting code blocks) can't be swapped without also changing the parser's iteration logic.

**Option B: Projectors own the full projection pipeline (`project(normalized_text) -> ProjectionResult`)**

- Code impact:
  - The parser delegates the entire `_build_dialog_projection()` body to the projector.
  - ```python
    class DialogProjector(Protocol):
        def project(self, normalized_text: str) -> ProjectionResult: ...
        @property
        def projector_id(self) -> str: ...
    ```
  - `ProjectionResult` would be a lightweight intermediate (dialog_text + evidence + line counts), and the parser wraps it into the final `DialogProjection` with head/tail.
- User-facing difference:
  - None; same `DialogProjection` output.
- Pros:
  - Full modularity: block-level and layout-level projection changes can be swapped.
  - Cleaner separation: parser only handles preset resolution, surface assessment, and final assembly.
- Cons:
  - Projectors must handle line iteration and evidence accumulation, increasing their complexity.
  - Risk of duplicating shared logic (head/tail slicing, metadata assembly) if not carefully factored.

> **PROPOSED: Option B (full pipeline), but with a shared base that handles iteration and head/tail slicing, so projectors only override the filtering/transformation rules.**
> Rationale: The existing `_build_dialog_projection()` in both `claude_code_shadow.py:392` and `codex_shadow.py:335` follows the same structure (normalize → iterate lines → filter → assemble metadata → slice head/tail). A shared base or helper can own that skeleton, and provider projectors override only the filtering rules. This gives full modularity without duplicating the shared boilerplate. The `VersionedPresetRegistry` pattern already shows the codebase's preference for version-aware selection with shared infrastructure, and this follows the same approach.
>
> **DECISION: Accept PROPOSED with refinement. Projectors should own the full projection stage for an already-normalized snapshot, not a `project_line()`-only hook, while parser/shared-core code keeps normalization, preset resolution, and common projection assembly centralized.**
> Rationale: A line-only callback is too weak for the TUI drift this change is trying to absorb because block-level and layout-level cleanup also need to be swappable. At the same time, ANSI stripping, newline normalization, preset/version resolution, and surface assessment already belong to the parser flow and should not migrate into projector implementations. Shared helpers should continue to cover common assembly such as fallback behavior, head/tail slicing, and projection metadata so provider projectors stay focused on projection rules rather than duplicating boilerplate.

### Q2) Should projector selection live in the provider parser, in `ShadowParserStack`, or at both levels?

- Decision: `Blocking`

#### Why this matters

- This determines who creates and configures projector instances, which affects test override ergonomics and the stack's role as a thin vs thick adapter.

#### Examples

- If selection lives in the provider parser: tests override projectors by configuring the parser directly (e.g., `ClaudeCodeShadowParser(projector_override=...)`). The stack stays thin.
- If selection lives in `ShadowParserStack`: the stack becomes a richer orchestrator. Tests configure the stack, which passes projectors to parsers. Two configuration surfaces.

#### Why this question arises in code

- `shadow_parser_stack.py:38–60`: The stack currently just selects the parser by tool name and delegates everything to it. Adding projector selection here would change its role.
- `claude_code_shadow.py:186+`: Preset resolution (which projector selection depends on) already lives inside the parser.
- The design's open question (design.md line 164): "Should the controlled projector override live at `ShadowParserStack` construction, provider parser construction, or both?"

#### Options

**Option A: Provider parser owns selection; stack passes optional override through**

- Code impact:
  - `ShadowParserStack.__init__` accepts an optional `projector_override` and forwards it to the selected parser's constructor.
  - The parser's `parse_snapshot()` checks for override, otherwise resolves from its preset registry.
  - ```python
    class ShadowParserStack:
        def __init__(self, tool: str, projector_override: DialogProjector | None = None): ...
    ```
- User-facing difference:
  - Single configuration surface at stack level for callers; parser-level for tests that construct parsers directly.
- Pros:
  - Consistent with existing architecture: the stack is a thin router, parsers own their internals.
  - Preset/version resolution stays co-located with projector selection in the parser.
- Cons:
  - Stack must add a pass-through parameter it doesn't use itself.

**Option B: Stack owns selection; parsers receive an injected projector**

- Code impact:
  - `ShadowParserStack` resolves projector based on tool + version hints, then injects it into the parser.
  - Parser's `_build_dialog_projection()` always uses the injected projector.
- User-facing difference:
  - Single configuration surface at stack level.
- Pros:
  - Centralizes all selection logic.
- Cons:
  - Breaks existing pattern where the stack is a thin router. Preset/version resolution would need to be partially duplicated or exposed from the parser to the stack.

> **PROPOSED: Option A — provider parser owns selection; stack passes optional override through.**
> Rationale: The existing `ShadowParserStack` (`shadow_parser_stack.py:35–90`) is a thin adapter that selects a parser by tool name and delegates all behavior. Preset and version resolution already live inside each parser class (`claude_code_shadow.py:186`, `codex_shadow.py` preset resolution). Keeping projector selection co-located with preset resolution in the parser preserves this layering. The stack only needs to accept and forward an optional override, which is the minimal change consistent with the existing pattern. This also matches the `_TurnEngine` protocol pattern in `cao_rest.py:124` where the engine delegates to internals rather than orchestrating them.
>
> **DECISION: Accept PROPOSED with refinement. Provider parsers own default projector selection, and both direct parser construction and `ShadowParserStack` should expose the same explicit override as a pass-through injection point.**
> Rationale: Preset/version resolution already lives in `ClaudeCodeShadowParser` and `CodexShadowParser`, so default selection should stay there. The stack is the normal repo integration surface and should allow callers to inject an override without instantiating provider-private parsers, but that override should only be forwarded, not re-resolved, so `ShadowParserStack` stays thin and does not become a second selector/orchestrator.

### Q3) Should a shared sentinel/schema-extraction helper be built as part of this change, or deferred?

- Decision: `Deferrable`

#### Why this matters

- The design recommends that machine-critical flows use schema-shaped extraction, and mailbox already implements this pattern in `mail_commands.py:127+`. If more consumers adopt the pattern (e.g., future structured-output flows), a shared helper avoids duplication. But building it now without a second consumer risks over-engineering.

#### Examples

- Mailbox uses `AGENTSYS_MAIL_RESULT_BEGIN/END` sentinels and parses JSON between them.
- A future schema-shaped consumer might use different sentinels but the same extraction logic (find delimiters in text surface, parse payload).

#### Why this question arises in code

- `mail_commands.py:127–175`: `parse_mail_result()` extracts sentinel-delimited JSON from event payloads. This is currently the only consumer of this pattern.
- `design.md:166`: The design explicitly asks this question but provides no spec or task for either answer.

#### Options

**Option A: Build a shared helper now**

- Code impact:
  - New module (e.g., `schema_extraction.py`) with a `extract_sentinel_payload(text, begin, end) -> str | None` helper.
  - `mail_commands.py` migrates to use it.
- User-facing difference:
  - None immediately.
- Pros:
  - Establishes the pattern early; future consumers have a clear integration point.
- Cons:
  - Only one consumer today; risks premature abstraction.

**Option B: Defer; let mailbox keep its own extractor**

- Code impact:
  - No new module. Mailbox extraction stays in `mail_commands.py`.
  - Document in design decisions that a shared helper is deferred until a second consumer appears.
- User-facing difference:
  - None.
- Pros:
  - Avoids over-engineering. Follows YAGNI.
- Cons:
  - If a second consumer appears soon, it will duplicate the pattern.

> **PROPOSED: Option B — defer the shared helper, but record the decision explicitly in the design.**
> Rationale: `mail_commands.py` is currently the only sentinel-extraction consumer, and its extraction logic (~50 lines) is tightly coupled to the mailbox response contract. Extracting a shared helper now would be premature abstraction with only one consumer. The design should record "deferred until a second sentinel-extraction consumer appears" so the decision is visible and doesn't need to be re-debated. This is consistent with the codebase's pattern of introducing abstractions when real pressure exists (e.g., `ShadowParserStack` was introduced when two providers needed the same interface, not preemptively).
>
> **DECISION: Accept PROPOSED. Defer a shared sentinel/schema-extraction helper in this change; keep mailbox extraction local until a second runtime-owned consumer exists, and record that deferral explicitly in the artifacts.**
> Rationale: `mail_commands.py` is the only current machine-critical consumer and its extraction path is still tightly coupled to mailbox-specific event surfaces and validation. Pulling that into a shared abstraction now would be speculative and would distract from the main work in this change, which is clarifying reliability boundaries and modularizing projector selection.

### Q5) Should the reliability tiers be formalized as a normative spec requirement, or remain as design-level guidance?

- Decision: `Deferrable`

#### Why this matters

- The three-tier model (lifecycle / operator-diagnostic / machine-critical) is the conceptual backbone of this change, but it appears only in `design.md` (lines 81–93). If it stays as design guidance only, future spec changes might inadvertently violate the tier boundaries because there's no requirement to check against.

#### Examples

- A future spec might add a machine-critical consumer that reads `dialog_text` directly without extraction. Without a normative tier requirement, there's no spec-level check that flags this as a violation.

#### Why this question arises in code

- `_TurnMonitor.observe_completion()` (`cao_rest.py:365`): lifecycle-tier consumer — compares `dialog_text` for change detection.
- `_best_effort_output_text_tail()` (`runtime.py:542`): operator-diagnostic tier — returns tail for human inspection.
- `parse_mail_result()` (`mail_commands.py:127`): machine-critical tier — uses sentinel extraction, not projection.
- These three consumers map cleanly to the three tiers, but the mapping is implicit.

#### Options

**Option A: Add a normative requirement to `shadow-dialog-projection/spec.md`**

- Code impact:
  - New requirement section defining the three tiers with example consumer classifications.
  - Scenarios that validate a new consumer declares its tier.
- User-facing difference:
  - Spec reviewers can check tier compliance.
- Pros:
  - Makes the tier model enforceable at spec level.
- Cons:
  - Adds spec surface that may feel heavyweight for a classification that's more about guidance than enforcement.

**Option B: Keep tiers as design guidance; reference from specs**

- Code impact:
  - Specs reference the design's tier model in non-normative notes.
- User-facing difference:
  - None.
- Pros:
  - Lighter weight; avoids over-formalizing what is essentially a usage guideline.
- Cons:
  - Future spec drift risk.

> **PROPOSED: Option B for now — keep tiers as design guidance, but add a non-normative note in `shadow-dialog-projection/spec.md` referencing the tier model.**
> Rationale: The tier model is a usage guideline, not a structural interface. The existing specs already encode tier-appropriate behavior through their scenarios (e.g., the "Machine-critical shadow consumer uses explicit extraction contract" scenario in `brain-launch-runtime/spec.md` lines 56–59 enforces machine-critical behavior without naming the tier). Adding a non-normative note that references the design's tier model provides discoverability without the overhead of normative tier-classification requirements. If the consumer population grows significantly, the tiers can be promoted to normative in a follow-up change.
>
> **DECISION: Reject PROPOSED as too weak. Add a lightweight normative spec requirement that names the three reliability tiers and maps the current repo-owned consumer classes to them, without introducing a heavy declaration mechanism.**
> Rationale: The tier model is the core contract clarification in this change, not just background commentary. The existing specs already encode pieces of the boundary, but leaving the tiers non-normative makes it easier for future specs to drift back toward treating projected dialog as authoritative reply text. A short requirement plus current examples is enough; we do not need a formal tier enum or mandatory annotations on every consumer.
