## Summary

- The core change is still valid: the registry `record.json` is packaged less completely than `launch_plan` and `session_manifest` today.
- The repository evolved after this change was drafted: the current registry record contract now includes optional `gateway` and `mailbox` groups and dedicated registry reference docs.
- The existing schema-alignment test pattern in `tests/unit/agents/realm_controller/test_schema_consistency.py` is a better concrete anchor than the current task wording.
- The artifacts should explicitly separate packaged-schema structure from model-only semantic invariants because the lightweight runtime validator cannot encode all `LiveAgentRegistryRecordV1` rules.

## Findings

### Must Fix / Blocking

- The change artifacts underspecify the current v1 registry record shape. `LiveAgentRegistryRecordV1` now includes nested `identity`, `runtime`, and `terminal` groups plus optional `gateway` and `mailbox` groups in `src/houmao/agents/realm_controller/registry_models.py:77-198`, while `docs/reference/registry/contracts/record-and-layout.md:48-156` already documents that fuller contract. The proposal/design/spec/tasks should explicitly say the packaged schema must cover the current v1 payload shape rather than an earlier narrower record surface.
- The current wording implies a stronger "schema matches model" promise than the runtime can actually enforce. `src/houmao/agents/realm_controller/schema_validation.py` only validates a lightweight subset (`type`, `required`, `enum`, `const`, `pattern`, array shape), while `LiveAgentRegistryRecordV1` and `RegistryGatewayV1` still rely on model validators for cross-field rules such as lease ordering, canonical-name/key derivation, `terminal.session_name == agent_name`, and gateway live-field grouping (`src/houmao/agents/realm_controller/registry_models.py:109-121`, `src/houmao/agents/realm_controller/registry_models.py:173-198`). The artifacts should say the packaged schema is the shipped structural disk contract and the Pydantic model remains authoritative for semantic invariants.

### Should Fix / Important

- The docs task is too generic for the current repo shape. The concrete follow-up docs are now `docs/reference/registry/contracts/record-and-layout.md` and `docs/reference/registry/internals/runtime-integration.md`, both of which already mention the missing packaged schema or publish-time behavior.
- The task for schema-consistency coverage should reference the existing packaged-schema alignment pattern in `tests/unit/agents/realm_controller/test_schema_consistency.py:15-151` so implementation work stays consistent with the rest of the runtime package.

### Nice To Have

- The proposal's impact section could name the registry reference docs explicitly so the eventual implementation scope is easier to scan.

## Design Alignment With Existing System

- Existing pattern(s) observed: packaged runtime schemas live under `src/houmao/agents/realm_controller/schemas/`, write-time schema validation goes through `src/houmao/agents/realm_controller/schema_validation.py`, and schema/model drift is checked by `tests/unit/agents/realm_controller/test_schema_consistency.py`.
- Proposed design alignment: the change aligns with those patterns if it reuses the existing schema directory, validator helper, and schema-consistency test style. It currently mismatches the evolved registry contract by not naming the now-present nested `gateway`/`mailbox` groups or the model-only invariants that the lightweight validator cannot express.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the change promise full parity with `LiveAgentRegistryRecordV1.model_json_schema()`, or explicit parity only for the validator-enforceable structural subset?

- Decision: `Blocking`

#### Why this matters

- This affects both the schema file shape and the test strategy.
- A "full parity" promise suggests the packaged schema alone can encode the whole contract, which is not true with the current lightweight validator.

#### Examples

- `gateway.host`, `gateway.port`, `gateway.state_path`, and `gateway.protocol_version` must appear together as a complete live-binding group.
- `lease_expires_at` must be later than `published_at`.
- `terminal.session_name` must equal `agent_name`.

#### Why this question arises in code

- The write-time validator is intentionally lightweight:

```markdown
validate_payload()
  -> load_schema()
  -> _validate_node()
       supports const / enum / type / required / additionalProperties / pattern / items
```

- The registry model still owns semantic cross-field checks:

```markdown
RegistryGatewayV1._validate_live_fields()
LiveAgentRegistryRecordV1._validate_schema_and_identity()
```

#### Options

**Option A: Promise full parity with the generated Pydantic JSON Schema.**

- Code impact:
  - Requires either expanding `schema_validation.py` substantially or introducing a more complete JSON Schema engine.
- User-facing difference:
  - The packaged schema becomes a stronger standalone contract, but implementation scope grows beyond this focused follow-up.
- Pros:
  - Clear single-source-of-truth story.
- Cons:
  - Broadens the change and does not match the current runtime validator design.

**Option B: Promise parity for the shipped structural subset, while keeping `LiveAgentRegistryRecordV1` authoritative for semantic invariants.**

- Code impact:
  - Lets the packaged schema stay within the current validator subset and reuse the existing schema-consistency pattern, while leaving model validators in place for cross-field rules.
- User-facing difference:
  - Operators still get a discoverable packaged schema for the persisted file, and runtime-managed writes still fail early on structural drift.
- Pros:
  - Fits the existing architecture and keeps the follow-up tightly scoped.
- Cons:
  - The contract remains intentionally split between the packaged schema and model-level invariants.

> **PROPOSED: Choose Option B and say that explicitly in the design, spec, and tasks.**
> Rationale: The current repository already uses a lightweight packaged-schema validator (`src/houmao/agents/realm_controller/schema_validation.py`) and a schema-alignment test helper that checks packaged schemas against Pydantic models structurally rather than by executing the packaged schema as a full semantics engine (`tests/unit/agents/realm_controller/test_schema_consistency.py`). That makes a structural packaged contract plus model-owned semantic invariants the most native fit for the current architecture, especially now that `LiveAgentRegistryRecordV1` contains several cross-field rules that the runtime validator cannot express cleanly.
>
> **DECISION: Choose Option B and treat the packaged registry schema as the validator-enforceable structural contract, with `LiveAgentRegistryRecordV1` remaining authoritative for semantic invariants.**
> Rationale: This keeps the change aligned with the current lightweight runtime schema validator and the existing packaged-schema consistency test pattern, while still shipping a concrete standalone schema for the persisted registry artifact.
