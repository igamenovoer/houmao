## Summary

- The proposal/design are directionally strong: they make the public compatibility boundary explicit and line up the HTTP and CLI stories around the same `cao` namespace.
- Tasks are mostly concrete and ordered by surface area, but one key contract is still underspecified: where `/cao` becomes authoritative for persisted runtime and gateway state.
- The repo has more first-party consumers of `HoumaoServerClient` than the change inventory acknowledges, especially demo tooling and its tests.
- OpenSpec currently reports artifact completion and apply-task progress through different status channels; that distinction does not require a change-artifact revision here.

## Findings

### Must Fix / Blocking

1. **The change does not yet define a single persisted/client contract for the `/cao` prefix, so runtime resume and gateway attach can drift into incompatible implementations.**

The design says pair-owned callers must stop assuming root CAO routes and should use an explicit compatibility seam, but it does not say whether manifests and attach contracts continue to store the server root URL or begin storing a `/cao`-qualified compatibility URL. That distinction matters because the current runtime and gateway layers persist a single `api_base_url` and then recreate plain `CaoRestClient` instances from it. `HoumaoServerRestSession` validates `houmao-server` first, then hands the unchanged base URL into `CaoRestSession`, which constructs `CaoRestClient(api_base_url)` for the actual session operations ([src/houmao/agents/realm_controller/backends/houmao_server_rest.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/backends/houmao_server_rest.py#L39), [src/houmao/agents/realm_controller/backends/cao_rest.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/backends/cao_rest.py#L446)). The same pattern exists in gateway attach metadata and manifest validation ([src/houmao/agents/realm_controller/gateway_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_models.py#L148), [src/houmao/agents/realm_controller/gateway_storage.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_storage.py#L329), [src/houmao/agents/realm_controller/gateway_service.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_service.py#L142), [src/houmao/agents/realm_controller/boundary_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/boundary_models.py#L193), [src/houmao/agents/realm_controller/runtime.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/runtime.py#L1222)).

> **DECISION: Accept.**
> Rationale: The review is correct that the current artifacts identify the need for a `/cao` client seam but do not yet make the persisted/runtime contract explicit enough. The developer decision is to keep persisted `api_base_url` rooted at the public `houmao-server` authority and make `/cao` an internal compatibility-routing concern handled by one shared client/factory used by `HoumaoServerClient`, `houmao_server_rest`, gateway attach, and other pair-owned callers. That keeps the existing manifest and attach-contract equality invariants intact and avoids leaking `/cao` into persisted runtime state.

### Should Fix / Important

2. **The change inventory misses first-party demo code and demo-backed flows that directly use root-shaped `HoumaoServerClient` CAO methods.**

The proposal impact list covers server/runtime/gateway/CLI paths, but repo-owned demo tooling also depends on `HoumaoServerClient` methods that today inherit root CAO paths. Examples include prompt submission in the tracking-debug flow and session query/delete flows in the dual-shadow watch demo ([src/houmao/demo/houmao_server_dual_shadow_watch/tracking_debug.py](/data1/huangzhe/code/houmao/src/houmao/demo/houmao_server_dual_shadow_watch/tracking_debug.py#L255), [src/houmao/demo/houmao_server_dual_shadow_watch/driver.py](/data1/huangzhe/code/houmao/src/houmao/demo/houmao_server_dual_shadow_watch/driver.py#L576), [src/houmao/demo/houmao_server_dual_shadow_watch/driver.py](/data1/huangzhe/code/houmao/src/houmao/demo/houmao_server_dual_shadow_watch/driver.py#L603), [src/houmao/demo/houmao_server_dual_shadow_watch/driver.py](/data1/huangzhe/code/houmao/src/houmao/demo/houmao_server_dual_shadow_watch/driver.py#L1398)). > **DECISION: Refine.**
> Rationale: The review is right that repo-owned demo flows and their tests are in scope because they instantiate `HoumaoServerClient` and call CAO-shaped session methods. The artifact gap is inventory scope, not architecture: I want the follow-up revision to explicitly include repo-owned demos and demo-backed tests in impact/tasks under the shared client seam and verification sweep, rather than introducing a separate demo-specific design track.

3. **Change status metadata is internally inconsistent, which makes implementation progress and review sign-off ambiguous.**

This is not a design flaw, but it is a review-process flaw: downstream reviewers cannot tell whether “complete” means artifact-ready, implementation-ready, or accidentally stale metadata.

> **DECISION: Reject.**

That distinction does not require proposal/design/spec/task changes for this boundary-reset change. If the semantics remain confusing, that should be handled separately in OpenSpec docs or tooling.

## Design Alignment With Existing System

- Existing pattern(s) observed: `HoumaoServerClient` currently mixes Houmao-owned routes with inherited CAO routes via subclassing `CaoRestClient` ([src/houmao/server/client.py](/data1/huangzhe/code/houmao/src/houmao/server/client.py#L41)); `houmao_server_rest` reuses CAO runtime machinery by subclassing `CaoRestSession` ([src/houmao/agents/realm_controller/backends/houmao_server_rest.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/backends/houmao_server_rest.py#L14)); gateway attach persists strict backend metadata and recreates clients from persisted `api_base_url` ([src/houmao/agents/realm_controller/gateway_storage.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_storage.py#L292), [src/houmao/agents/realm_controller/gateway_service.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_service.py#L121)); `houmao-srv-ctrl` currently mirrors upstream CAO at the top level and relies on simple passthrough wrappers ([src/houmao/srv_ctrl/commands/main.py](/data1/huangzhe/code/houmao/src/houmao/srv_ctrl/commands/main.py#L12), [src/houmao/srv_ctrl/commands/passthrough.py](/data1/huangzhe/code/houmao/src/houmao/srv_ctrl/commands/passthrough.py#L10)).
- Proposed design alignment: the namespace reset fits the repo’s existing “pair-owned vs compatibility” split, but it needs one more native step to really match the architecture: make the `/cao` boundary a shared client abstraction/factory decision instead of a per-caller string rewrite. That approach fits the existing strict manifest and attach-contract models better than scattering path-prefix knowledge across runtime, gateway, demos, and CLI helpers.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Where should the `/cao` prefix become authoritative for `houmao_server_rest` persisted state?

- Decision: `Blocking`

#### Why this matters

- Runtime resume, gateway attach, server query helpers, and demos all reconstruct clients from persisted or shared base-URL data.
- If some callers store `http://host:port` and others store `http://host:port/cao`, compatibility routing will become caller-specific and fragile.

#### Examples

- A runtime manifest could resume successfully if `HoumaoServerClient` injects `/cao`, while gateway attach against the same manifest still fails because it rebuilds plain `CaoRestClient(api_base_url)`.
- A future demo or tool could call `HoumaoServerClient.list_sessions()` and silently target the wrong surface if the storage contract is unclear.

#### Why this question arises in code

- `HoumaoServerRestSession` currently validates `houmao-server` and then delegates all session mechanics to `CaoRestSession`, which constructs a plain `CaoRestClient` from the stored base URL.
- Gateway attach metadata and manifest parsing both persist one `api_base_url` string and expect equality across sections.

```text
HoumaoServerRestSession(api_base_url=root)
  -> CaoRestSession(api_base_url=root)
     -> CaoRestClient(root)

GatewayAttachBackendMetadataHoumaoServerV1(api_base_url=root)
  -> GatewayAttachSessionControl
     -> CaoRestClient(root)
```

#### Options

**Option A: Keep persisted `api_base_url` at the server root and introduce one shared prefix-aware compatibility client/factory for `/cao` calls.**

- Code impact:
  - `HoumaoServerClient` stops inheriting root CAO assumptions and instead exposes explicit root/native methods plus a dedicated CAO-compat client or client factory.
  - `houmao_server_rest`, gateway attach, demos, and query helpers all consume the same factory rather than building `CaoRestClient` directly.
- User-facing difference:
  - Pair metadata stays rooted at the public `houmao-server` base URL.
  - `/cao` remains an internal compatibility-routing detail rather than leaking into persisted runtime contracts.
- Pros:
  - Fits current manifest/attach invariants, especially the equality checks around `api_base_url`.
  - Keeps `/health` and `/houmao/*` access on the same root authority without storing multiple base URLs.
  - Reduces the number of places that need to know how compatibility routing works.
- Cons:
  - Requires touching shared client abstractions instead of only renaming server routes.

**Option B: Persist `/cao`-qualified compatibility base URLs for `houmao_server_rest` callers.**

- Code impact:
  - Manifests, gateway attach metadata, runtime validators, and demos would all need to treat `api_base_url` as a compatibility-surface URL rather than the server root.
  - Native `/health` and `/houmao/*` callers would need separate root URL handling.
- User-facing difference:
  - Persisted artifacts would embed the compatibility namespace directly.
- Pros:
  - Makes CAO-compatible HTTP targeting explicit in stored data.
- Cons:
  - Conflicts with current “one server authority” usage and complicates root/native calls.
  - Makes mixed root/native and compatibility access harder for paired clients that need both.

> **PROPOSED: Choose Option A and keep persisted `api_base_url` at the server root, with one shared prefix-aware compatibility client/factory for `/cao` access.**
> Rationale: The repository already treats `api_base_url` as the authority for both Houmao-owned and CAO-compatible behavior, and runtime validation currently enforces equality between `houmao_server.api_base_url` and `backend_state.api_base_url` ([src/houmao/agents/realm_controller/runtime.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/runtime.py#L1240)). Preserving that invariant avoids a contract split while still letting `HoumaoServerClient`, `houmao_server_rest`, gateway attach, and demos share one explicit compatibility seam instead of each inventing its own `/cao` rewrite.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Keep persisted `api_base_url` at the public server root and make `/cao` authoritative through one shared compatibility client/factory rather than by storing `/cao`-qualified base URLs in manifests or attach metadata. That preserves the current `houmao_server.api_base_url == backend_state.api_base_url` invariant, keeps root `/health` and `/houmao/*` on the same authority, and avoids caller-specific path-prefix drift across runtime resume, gateway attach, demos, and query helpers.

### Q2) How strict should `houmao-srv-ctrl cao launch/info/shutdown` be about preserving upstream CLI output behavior once they stop being blind passthrough commands?

- Decision: `Deferrable`

#### Why this matters

- The specs claim additive-only compatibility under `houmao-srv-ctrl cao ...`, but pair-aware wrappers can easily preserve command shape while still changing stdout, stderr, exit-code handling, or machine-readable output.
- The current implementation gets compatibility “for free” because the top-level commands are still passthrough-oriented.

#### Examples

- `houmao-srv-ctrl cao info` could preserve arguments but return Houmao-authored wording instead of upstream CAO text or JSON shape.
- `houmao-srv-ctrl cao launch` could register runtime artifacts correctly but omit upstream success text, progress output, or non-zero exit propagation that scripts rely on.

#### Why this question arises in code

- Today, top-level `info` and `shutdown` are just passthrough commands plus a pair check, and `launch` still relies on delegated `cao launch` plus Houmao follow-up work.

```text
cli.add_command(passthrough_command("info", require_pair=True))
cli.add_command(passthrough_command("shutdown", require_pair=True))
launch_command:
  require_supported_houmao_pair(...)
  run_passthrough("launch", ...)
  materialize/register Houmao artifacts
```

#### Options

**Option A: Preserve exit codes and compatibility-significant stdout/stderr behavior for `houmao-srv-ctrl cao ...`, especially where upstream already exposes machine-readable or script-consumed output.**

- Code impact:
  - Wrapper commands need output contracts and regression tests, not just argument-parsing tests.
  - Some wrappers may still proxy or mirror upstream output even when extra Houmao follow-up work happens behind the scenes.
- User-facing difference:
  - The `cao` namespace remains reliable for existing CAO-shaped scripts.
- Pros:
  - Best matches the additive-only compatibility promise in the spec.
  - Minimizes silent regressions when replacing passthrough with wrapper logic.
- Cons:
  - More implementation and test work.

**Option B: Preserve command shape and exit-code semantics, but allow wrapper-owned output wording as long as the command remains functionally equivalent.**

- Code impact:
  - Simpler wrappers and fewer output-focused tests.
- User-facing difference:
  - Human operators may see Houmao-authored messages under the compatibility namespace.
- Pros:
  - Faster to implement.
- Cons:
  - Weakens the practical meaning of “CAO-compatible” for scripts and docs.

> **PROPOSED: Choose Option A for the explicit `cao` namespace, and keep Houmao-authored messaging primarily at the top level.**
> Rationale: The current repo gets output compatibility from direct passthrough on `info`, `shutdown`, and much of `launch` ([src/houmao/srv_ctrl/commands/passthrough.py](/data1/huangzhe/code/houmao/src/houmao/srv_ctrl/commands/passthrough.py#L10), [src/houmao/srv_ctrl/commands/launch.py](/data1/huangzhe/code/houmao/src/houmao/srv_ctrl/commands/launch.py#L59)). Once those become repo-owned wrappers, preserving only argument parsing would be a weaker contract than what the repo already ships today, so the compatibility namespace should retain compatibility-significant output and exit behavior wherever practical.
>
> **DECISION: Accept PROPOSED, with one narrowing note.**
> Rationale: The explicit `cao` namespace should preserve exit codes and compatibility-significant stdout/stderr behavior wherever upstream CAO already exposes machine-readable or script-consumed output, while top-level Houmao-owned commands remain free to use Houmao-authored messaging. I do not want to promise byte-for-byte parity for every human-oriented line of prose, but the compatibility namespace should not regress practical script-facing behavior once commands stop being blind passthrough wrappers.
