## Summary

- **Solid:** The proposal correctly identifies a structural problem — recipes cannot own launch behavior — and the design introduces a clean layered override model with explicit precedence, provenance, and fail-closed semantics. The architectural direction is right.
- **Solid:** Fail-closed behavior for `cao_rest` and backend-reserved protocol arg protection are well-motivated by the real HTT-demo failure documented in `context/issues/known/issue-cao-rest-ignores-tool-adapter-launch-args.md`.
- **Solid:** Tasks are well-ordered in five phases with logical dependency flow: construction → registry → runtime → provider → tests/docs.
- **Risk:** The name `launch_surface` collides with the existing `LaunchSurface` literal type in `src/houmao/agents/launch_policy/models.py`, which is a backend-selection enum — a completely different concept. This will cause confusion across the codebase.
- **Risk:** Several merge/precedence edge cases between recipe and direct-build overrides lack specification, particularly around partial field overlap and `args.mode` composition.
- **Risk:** Manifest schema migration is implied but no task explicitly handles version bumping or legacy-manifest reading during the transition window.

## Findings

### Must Fix / Blocking

1. **Naming collision: `launch_surface` vs existing `LaunchSurface` backend-selection type.** The existing `LaunchSurface = Literal["raw_launch", "codex_headless", "codex_app_server", "claude_headless", "gemini_headless", "cao_rest", "houmao_server_rest"]` in `src/houmao/agents/launch_policy/models.py` (used by `engine.py` and `cli.py`) is a backend-selection enum. Having both concepts share the name `launch_surface` / `LaunchSurface` in the same codebase will create import confusion, ambiguous doc references, and will make grepping for either concept unreliable. The design should either rename the new concept (e.g., `launch_overrides`, `launch_config`, `launch_shape`) or rename the existing backend-selection type (e.g., `BackendSurface`, `LaunchTarget`).

> **DECISION: _(accept / refine / reject / defer this finding)_**
> Rationale: Accept. `LaunchSurface` already exists as the backend-selection enum in `launch_policy/models.py` and is used by `launch_policy/engine.py`. 2. **Task 1.3 and Task 1.4 appear partially contradictory.** Task 1.3 says "persist adapter-owned launch defaults, requested launch-surface overrides, and launch-surface provenance as separate structured fields." If the builder persists adapter defaults, requested overrides, *and* provenance (1.3), it is storing more than "only unresolved launch intent" (1.4). The relationship between these two tasks needs clarification — is 1.4 a constraint on 1.3 (i.e., 1.3 should persist defaults + overrides but NOT resolve backend applicability), or is it a separate concern about not pre-resolving effective args?

> **DECISION: _(accept / refine / reject / defer this finding)_**

## Design Alignment With Existing System

- **Existing pattern: tool-adapter as declarative YAML, builder as pure function, manifest as JSON snapshot.** The proposed design follows this pattern correctly — tool-adapter YAML gains optional launch metadata declarations, the builder resolves and persists snapshots, and the manifest stores enough state for reproducible launches. This aligns with how `brain_builder.py` already handles `ToolAdapter` loading and manifest generation.

- **Existing pattern: launch-plan composition reads manifest, validates per-backend, injects protocol args.** The current `build_launch_plan()` in `launch_plan.py` already does per-backend reserved-arg conflict detection (`_CLAUDE_HEADLESS_RESERVED_ARGS`, `_CODEX_HEADLESS_RESERVED_ARGS`) and backend-specific arg injection. The proposed refactor to add launch-surface resolution as an intermediate step fits naturally into this flow.

- **Existing pattern: `@dataclass(frozen=True)` for internal value objects.** The new `launch_surface` models should follow this convention rather than using Pydantic, consistent with `BuildRequest`, `BuildResult`, `ToolAdapter`, `LaunchPlan`, etc. The design doesn't specify, but the codebase convention is clear.

- **Mismatch: existing `LaunchSurface` type in `launch_policy/models.py`.** As noted in Finding #1, the proposed naming creates a collision with the existing backend-selection literal type. The existing `LaunchSurface` in `launch_policy` is used in `engine.py:_select_launch_surface()` and `cli.py:_parse_backend()`. The new concept and the existing type need different names.

- **Alignment concern: `headless_base.py` factoring.** The current `HeadlessInteractiveSession._build_command()` directly concatenates `self._plan.executable, *self._plan.args` and then appends protocol args. The proposed refactor would need the resolved launch-surface to already be folded into `LaunchPlan.args` (or a replacement field) before backends consume it. This is consistent with the design's intent but means `LaunchPlan` itself may need structural changes (not just the manifest).

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should the naming collision between the new `launch_surface` concept and the existing `LaunchSurface` backend-selection type be resolved?

- Decision: `Blocking`

#### Why this matters

- The existing `LaunchSurface = Literal["raw_launch", "codex_headless", ...]` in `launch_policy/models.py` selects which backend to use. The proposed `launch_surface` is a recipe-owned override contract for CLI args and tool params. These are orthogonal concepts, and sharing a name will make imports, grep results, documentation, and code review confusing.

#### Examples

- A developer searching for "launch_surface" to understand recipe overrides would also find all backend-selection logic in `launch_policy/`.
- Import statements like `from houmao.agents.launch_surface.models import LaunchSurface` vs `from houmao.agents.launch_policy.models import LaunchSurface` would be ambiguous and easy to confuse.

#### Why this question arises in code

- `src/houmao/agents/launch_policy/models.py` defines `LaunchSurface` as a `Literal` type alias for backend selection.
- `src/houmao/agents/launch_policy/engine.py` uses `_select_launch_surface()` to choose a backend.
- The proposed new package `src/houmao/agents/launch_surface/` would introduce a completely different `launch_surface` concept.

#### Options

**Option A: Rename the new concept to `launch_overrides` (package: `src/houmao/agents/launch_overrides/`)**

- Code impact:
  - Recipe YAML key becomes `launch_overrides:` instead of `launch_surface:`
  - `BuildRequest.launch_overrides` replaces `BuildRequest.launch_surface_override`
  - New package: `src/houmao/agents/launch_overrides/`
- User-facing difference:
  - Recipe YAML uses `launch_overrides:` which clearly conveys "this overrides defaults"
- Pros:
  - No collision; semantically precise (these ARE overrides)
  - Existing `LaunchSurface` type untouched
- Cons:
  - Slightly less general if the concept later expands beyond "overrides"

**Option B: Rename the existing `LaunchSurface` type to `LaunchTarget` or `BackendTarget`**

- Code impact:
  - Rename `LaunchSurface` → `LaunchTarget` in `launch_policy/models.py`, `engine.py`, `cli.py`
  - The new concept keeps `launch_surface` naming
- User-facing difference:
  - No change to recipe YAML
- Pros:
  - The new concept owns the `launch_surface` name which matches the design docs
- Cons:
  - Requires renaming an established type across `launch_policy/` (3+ files)

**Option C: Keep both names as-is and rely on package-path disambiguation**

- Code impact:
  - No renames; developers must use fully qualified imports to distinguish
- User-facing difference:
  - None
- Pros:
  - Zero churn
- Cons:
  - Ongoing confusion; grep for `launch_surface` always returns mixed results

> **PROPOSED: Option A — rename the new concept to `launch_overrides`.**
> Rationale: The new concept IS an override contract — it always layers on top of adapter defaults. `launch_overrides` is semantically precise, avoids all collision, and requires zero churn on the established `LaunchSurface` backend-selection type in `launch_policy/`. The package `src/houmao/agents/launch_overrides/` clearly conveys its role alongside `launch_policy/`.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Accept the proposed direction. The existing `LaunchSurface` backend-selection type is already established in `launch_policy/models.py` and `engine.py`, while the new concept is an overlay on adapter defaults. `launch_overrides` is more precise and avoids cross-package naming churn in the launch-policy stack.

### Q2) What is the partial-field merge behavior when a direct-build override only specifies a subset of the launch-surface model?

- Decision: `Blocking`

#### Why this matters

- The structured model has two top-level sections (`args` and `tool_params`). If a recipe declares both but a direct-build override only declares one, the behavior of the unmentioned section determines whether recipe intent is preserved or silently dropped. Getting this wrong would either discard recipe values the operator expected to survive, or silently keep recipe values the direct-build caller expected to replace.

#### Examples

- Recipe declares `tool_params: {include_partial_messages: true}` and `args: {mode: append, values: ["--verbose"]}`.
- Direct-build override declares only `args: {mode: replace, values: ["--quiet"]}` and says nothing about `tool_params`.
- Should `include_partial_messages: true` survive from the recipe?

#### Why this question arises in code

- `BuildRequest.launch_surface_override` will be a single structured value. The current `launch_args_override: list[str] | None` is all-or-nothing (either override all args or use adapter defaults). The new model has two independently meaningful sub-fields, so partial override semantics must be defined.
- Design Decision 6 says "direct build overrides win over recipe values when both are present" but does not define "present" at sub-field granularity.

#### Options

**Option A: Deep-merge per top-level section — only replace sections that are explicitly present in the direct-build override**

- Code impact:
  - Merge logic checks presence of `args` and `tool_params` independently
  - If direct-build supplies only `args`, recipe `tool_params` survive unchanged
  - If direct-build supplies `tool_params`, those keys replace matching recipe keys (not the entire section)
- User-facing difference:
  - Recipe intent is preserved for sections the direct-build doesn't mention
- Pros:
  - Least-surprise behavior; targeted overrides
  - Consistent with how config overlays typically work (e.g., OmegaConf merge)
- Cons:
  - More complex merge logic; harder to reason about "what's effective" without provenance

**Option B: Whole-model replace — if a direct-build override is present, it replaces the entire recipe launch surface**

- Code impact:
  - `BuildRequest.launch_surface_override` is all-or-nothing, similar to current `launch_args_override`
  - Recipe `launch_surface` is fully discarded when a direct-build override exists
- User-facing difference:
  - Direct-build callers must re-declare all desired launch-surface settings
- Pros:
  - Simple; predictable; easy to explain
- Cons:
  - Callers must know the full recipe launch surface to avoid accidentally dropping settings

> **PROPOSED: Option A — deep-merge per top-level section.**
> Rationale: The current codebase already uses structured multi-field models (e.g., `BrainRecipe` with independent `skills`, `config_profile`, `credential_profile` that can each be overridden). A per-section merge is consistent with this pattern and avoids forcing direct-build callers to duplicate recipe intent they don't want to change. Provenance tracking (Task 3.4) handles the "what's effective" concern.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Accept the reviewer direction with refinement. Merge by top-level section, but do not recursively compose everything the same way: unmentioned sections survive from the recipe, `tool_params` merges per key, and `args` is an atomic section. If a direct-build override provides `args`, that `args` section replaces the recipe `args` section and then applies its own `mode` against adapter defaults.

### Q3) Should manifest `schema_version` be bumped to 2, and should legacy v1 manifests be readable during a transition window?

- Decision: `Deferrable`

#### Why this matters

- The current manifest writes `"schema_version": 1`. The proposed change adds new fields and deprecates `runtime.launch_args` as the sole launch contract. Already-built brain homes on disk will have v1 manifests. If runtime code expects the new fields, old homes will break unless there's a migration reader or the version is bumped with a fallback path.

#### Examples

- A developer builds a brain home today, then upgrades the code. On next `start-session`, launch-plan composition tries to read `launch_surface_defaults` from the manifest and fails because the v1 manifest only has `runtime.launch_args`.

#### Why this question arises in code

- `brain_builder.py` line ~680 writes `"schema_version": 1` into manifests.
- `launch_plan.py` reads `runtime.launch_args` from the manifest and uses `_require_str_list()` which raises if the field is missing.
- The design's Migration Plan step 6 mentions a "migration window" but no task covers this.

#### Options

**Option A: Bump to schema_version 2, add a v1 compatibility reader that synthesizes the new fields from legacy data**

- Code impact:
  - New builder output writes `"schema_version": 2`
  - `build_launch_plan()` checks version and falls back to v1 layout for old manifests
- User-facing difference:
  - Old built homes continue to work during transition
- Pros:
  - Graceful migration
- Cons:
  - Extra code for a transition that may be short-lived

**Option B: Bump to schema_version 2 with no v1 compatibility — require rebuilding affected brain homes**

- Code impact:
  - New code rejects schema_version 1 manifests at launch time
- User-facing difference:
  - Operator must rebuild brain homes after code upgrade
- Pros:
  - Clean break; no legacy reader code
  - Consistent with project's "breaking changes allowed" stance
- Cons:
  - Operator friction for existing homes

> **PROPOSED: Option B — bump to schema_version 2, require rebuild, no legacy reader.**
> Rationale: The project status explicitly says "breaking changes are allowed" and "do not spend effort preserving legacy interfaces." Built brain homes are generated artifacts under `tmp/` (gitignored, safe to delete/rebuild per CLAUDE.md). A clean break avoids carrying compatibility debt. The design's own rollback strategy confirms "reverting the code and rebuilding the affected brain homes is sufficient."
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Accept the proposed direction. The current manifest reader path already hard-gates schema versions, built brain homes are generated artifacts, and the repo explicitly allows breaking changes. This change should bump to schema version 2 and require rebuilding affected homes rather than adding a temporary v1 compatibility reader.
