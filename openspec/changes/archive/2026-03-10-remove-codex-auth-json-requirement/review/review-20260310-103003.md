## Summary

- **Solid motivation:** The proposal is grounded in verified runtime evidence — Codex custom-provider flow works without `auth.json`, so the builder-enforced requirement is provably artificial.
- **Generic design approach is correct:** Extending the schema with per-mapping optionality (rather than hard-coding a Codex exception) aligns with the adapter-driven architecture and benefits future adapters.
- **Backward compatibility well-considered:** Required-by-default semantics ensure Claude and Gemini adapters are unaffected without any changes.
- **Pre-launch validation placement needs clarification:** Task 1.3 says "add Codex pre-launch validation" but does not specify where. The existing `codex_bootstrap.py:ensure_codex_home_bootstrap()` is the natural insertion point, already called from all four Codex execution contexts. The design should name this explicitly.
- **Schema flag naming is unresolved:** The design's open question (`required` vs `optional` vs another name) has downstream effects on YAML readability and Pydantic/dataclass defaults. This should be decided before implementation.
- **Yunwu change spec conflict not addressed:** The earlier `add-codex-yunwu-agent` change spec explicitly requires `files/auth.json` as a compatibility file. Implementing this change without reconciling that spec creates a formal contradiction in the spec tree.

## Findings

### Must Fix / Blocking

1. **Reconcile the `add-codex-yunwu-agent` spec before or during implementation.**
   The Yunwu change spec at `openspec/changes/add-codex-yunwu-agent/specs/codex-openai-compatible-brain-profile/spec.md` contains the requirement: _"The credential profile SHALL contain the adapter-required local compatibility file at `files/auth.json`."_ This directly contradicts the new spec scenario _"Env-backed Codex profile omits optional auth.json."_ The tasks do not include updating or superseding the Yunwu spec. Either archive/supersede that requirement as part of this change, or add an explicit task to update it.

2. **Task 1.3 must specify the implementation location for pre-launch auth validation.**
   The design says "Codex launch preparation" will validate auth, but the tasks and spec do not identify the module. The existing `codex_bootstrap.py:ensure_codex_home_bootstrap()` is called from all four Codex backend entry points (`headless_base.py` twice, `codex_app_server.py`, `cao_rest.py`). This is the correct single insertion point. If validation is placed elsewhere (e.g., in `launch_plan.py` or in the builder), it risks missing execution paths. The design or tasks should name `codex_bootstrap.py` explicitly.

### Should Fix / Important

3. **The delta spec should include a `SUPERSEDES` or `MODIFIES` note for the Yunwu spec's auth.json requirement.**
   Without this, a future reader will see two valid specs that contradict each other on whether `files/auth.json` is required for env-backed Codex profiles.

4. **Task 2.1 bundles four distinct test concerns into one task.**
   It combines: (a) missing required files still fail, (b) missing optional files are skipped, (c) optional files are projected when present, and (d) Codex launch rejection when both auth paths are absent. Items (a)–(c) are builder-level unit tests; item (d) is a launch-runtime test in a different module. Consider splitting (d) into its own task under section 1 or creating a new section, since it requires different test fixtures and targets different code.

5. **The spec scenario "Codex launch is rejected when no auth path exists" lives under the "Local-only credential profiles" requirement, but it describes launch-time behavior, not build-time behavior.**
   This is a layering mismatch. The credential profile requirement governs brain construction. Pre-launch auth validation is a runtime concern. Consider either: (a) placing this scenario under a new "Codex launch prerequisites" requirement in the spec, or (b) explicitly noting that the `component-agent-construction` spec covers both build and launch phases.

6. **Design should note that `codex_bootstrap.py` needs an `env` parameter to check `OPENAI_API_KEY`.**
   The current `ensure_codex_home_bootstrap(home: Path)` signature does not accept an environment mapping. The Claude bootstrap (`claude_bootstrap.py`) already takes `env: Mapping[str, str]` and validates `ANTHROPIC_API_KEY` from it — this is the proven pattern to follow. All four call sites already have the effective environment available and would pass it through.

### Nice To Have

7. **Consider adding a `schema_version: 2` bump (or a minor version) to the tool adapter schema when adding the `optional` field.**
   The current `codex.yaml` declares `schema_version: 1`. While backward compatibility is preserved (missing field defaults to required), a version bump signals the schema evolution to tooling and human readers. The adapter loader in `brain_builder.py` could accept both versions.

8. **The design's open question about flag naming (`required`, `optional`, or something else) could benefit from a concrete recommendation.**
   See Open Questions Q1 below.

9. **Task 3.2 (live smoke test) is inherently non-deterministic and depends on external service availability.**
   Mark it as a manual verification step rather than an automated test expectation.

## Design Alignment With Existing System

- **Existing pattern — env var allowlist (graceful skip):** `brain_builder.py` lines ~485–499 already implement a "present → project, absent → skip" pattern for credential env vars via `credential_env_allowlist`. The proposed optional file mapping extends this same philosophy to credential files. This is well-aligned.

- **Existing pattern — bootstrap validation:** `claude_bootstrap.py` validates `ANTHROPIC_API_KEY` from an `env` parameter before Claude launch. The proposed Codex pre-launch auth check mirrors this pattern exactly. Placing it in `codex_bootstrap.py:ensure_codex_home_bootstrap()` follows the established per-tool bootstrap convention.

- **Existing pattern — frozen dataclass with defaults:** `CredentialFileMapping` is a `@dataclass(frozen=True)`. Adding a boolean field with a default value (`required: bool = True` or `optional: bool = False`) is the standard pattern used throughout the codebase for backward-compatible dataclass evolution.

- **Existing pattern — YAML adapter loading with `_require_str`:** The `_load_tool_adapter()` function uses `_require_str()` for mandatory fields. Optional fields should use `.get()` with a default, consistent with how other optional YAML fields are parsed in the codebase.

- **Potential misalignment — launch validation in build spec:** The `component-agent-construction` spec currently governs the build phase. Adding a launch-time scenario (auth validation before Codex start) expands its scope. The codebase separates `brain_builder.py` (build) from `brain_launch_runtime/` (launch) at the module level. The spec should acknowledge this boundary or create a separate spec for launch prerequisites.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the per-mapping flag be named `required` (default `true`) or `optional` (default `false`)?

- Decision: `Deferrable`

#### Why this matters

The flag name affects YAML readability for adapter authors, Python dataclass field naming, and the conditional logic in the projection loop. The design explicitly calls this out as an open question.

#### Examples

- Adapter YAML with `required`:
  ```yaml
  file_mappings:
    - source: auth.json
      destination: auth.json
      mode: symlink
      required: false
  ```
- Adapter YAML with `optional`:
  ```yaml
  file_mappings:
    - source: auth.json
      destination: auth.json
      mode: symlink
      optional: true
  ```

#### Why this question arises in code

- `brain_builder.py` `CredentialFileMapping` dataclass (lines ~23–27) needs a new field.
- `_load_tool_adapter()` (lines ~166–188) needs to parse it from YAML.
- The projection loop (lines ~458–483) branches on it:
  ```python
  if not source.exists():
      if mapping.required:    # or: if not mapping.optional:
          raise BuildError(...)
      continue
  ```

#### Options

**Option A: `required: bool = True`**

- Code impact:
  - Field: `required: bool = True` on `CredentialFileMapping`
  - YAML: `required: false` to mark optional
  - Branch: `if not mapping.required: continue`
  ```python
  @dataclass(frozen=True)
  class CredentialFileMapping:
      source: str
      destination: str
      mode: str
      required: bool = True
  ```
- User-facing difference:
  - Adapter authors write `required: false` to opt out of mandatory check
- Pros:
  - Reads naturally for the common case — most mappings are required so the field is simply absent
  - `required` is a widely understood schema concept (JSON Schema, OpenAPI, etc.)
- Cons:
  - The "interesting" case (optional) is expressed as a double negative (`required: false`)

**Option B: `optional: bool = False`**

- Code impact:
  - Field: `optional: bool = False` on `CredentialFileMapping`
  - YAML: `optional: true` to mark optional
  - Branch: `if mapping.optional: continue`
  ```python
  @dataclass(frozen=True)
  class CredentialFileMapping:
      source: str
      destination: str
      mode: str
      optional: bool = False
  ```
- User-facing difference:
  - Adapter authors write `optional: true` to mark a file as skippable
- Pros:
  - The exceptional case reads as a positive assertion (`optional: true`) — no double negative
  - Slightly more intuitive when scanning YAML: "this file is optional" vs "this file is not required"
- Cons:
  - Less conventional than `required` in schema design tooling

> **PROPOSED: Use `required: bool = True` (Option A).**
> Rationale: The existing codebase uses `_require_str()` as its validation primitive in `brain_builder.py` (lines ~166–188), establishing `required` as the local vocabulary for mandatory-vs-optional semantics. The `schema_version: 1` adapter YAML has no precedent for either name, so alignment with the Python-side naming convention is the strongest signal. Additionally, `required` matches industry-standard schema conventions (JSON Schema, OpenAPI, Kubernetes CRDs), which reduces cognitive load for contributors familiar with those systems. The double-negative concern (`required: false`) is minor since only the Codex adapter currently needs it, and the YAML is read far less often than it is validated by code.
>

### Q2) Should the Codex pre-launch auth validation treat an empty `{}` auth.json as "no usable login state"?

- Decision: `Blocking`

#### Why this matters

The design says validation should check for "a valid `auth.json` login state." If an existing profile still carries the legacy `{}` placeholder, it contains no actual login tokens. The validation must decide whether `{}` counts as "valid auth.json present" (and thus satisfies the auth requirement alone) or "no usable login state" (requiring `OPENAI_API_KEY` as fallback).

#### Examples

- Profile A: has `auth.json` with real tokens, no `OPENAI_API_KEY` → should launch ✓
- Profile B: has `auth.json` containing `{}`, no `OPENAI_API_KEY` → should this launch? ✗ or ✓?
- Profile C: no `auth.json`, has `OPENAI_API_KEY` → should launch ✓
- Profile D: no `auth.json`, no `OPENAI_API_KEY` → should fail ✗

#### Why this question arises in code

- `codex_bootstrap.py:ensure_codex_home_bootstrap()` will need to check auth.json content, not just file existence.
- The current yunwu-openai fixture has `auth.json` containing `{}` AND `OPENAI_API_KEY` set — so both paths are satisfied today. But if someone removes the `{}` file (as this change enables) and also misconfigures env vars, the validation behavior on a lingering `{}` file matters.
  ```python
  # What counts as "valid login state"?
  auth_path = home / "auth.json"
  if auth_path.exists():
      content = json.loads(auth_path.read_text())
      has_login_state = bool(content)  # {} is falsy — treats empty as invalid
      # OR
      has_login_state = True           # file exists — treats any file as valid
  ```

#### Options

**Option A: File-existence check only (any `auth.json` present counts as valid)**

- Code impact:
  ```python
  has_auth = (home / "auth.json").exists()
  has_key = "OPENAI_API_KEY" in env and env["OPENAI_API_KEY"]
  if not has_auth and not has_key:
      raise BackendExecutionError(...)
  ```
- User-facing difference:
  - A `{}` placeholder satisfies the check — Codex itself decides whether it's usable at runtime
- Pros:
  - Simpler validation; does not couple the repo to Codex's internal auth.json schema
  - Avoids false negatives if Codex changes its auth.json format
- Cons:
  - A `{}` file provides no real auth — Codex will fail later with a less clear error

**Option B: Content-aware check (empty/minimal `auth.json` does not count as valid login state)**

- Code impact:
  ```python
  has_auth = False
  auth_path = home / "auth.json"
  if auth_path.exists():
      try:
          data = json.loads(auth_path.read_text())
          has_auth = isinstance(data, dict) and len(data) > 0
      except (json.JSONDecodeError, OSError):
          has_auth = False
  has_key = "OPENAI_API_KEY" in env and env["OPENAI_API_KEY"]
  if not has_auth and not has_key:
      raise BackendExecutionError(...)
  ```
- User-facing difference:
  - A `{}` placeholder is treated as "no auth" — validation catches the misconfiguration early
- Pros:
  - Provides earlier, clearer feedback when neither real auth path is configured
  - Matches the design intent: validation should catch "launch with no usable auth"
- Cons:
  - Couples the repo to assumptions about Codex's `auth.json` internal structure (what keys constitute a "valid" login state?)

> **PROPOSED: Use content-aware validation (Option B) with a minimal heuristic: treat `auth.json` as valid only if it parses as a non-empty dict.**
> Rationale: The design explicitly states the goal is to "fail fast before Codex launch when neither a valid `auth.json` login state nor `OPENAI_API_KEY` is available." The word "valid" implies content inspection, not mere file existence. The existing yunwu-openai fixture at `tests/fixtures/agents/brains/api-creds/codex/yunwu-openai/files/auth.json` contains exactly `{}`, while the real login-state fixture at `personal-a-default/files/auth.json` contains a populated dict with token fields. A simple `len(data) > 0` check distinguishes these cases without depending on specific Codex-internal field names. This is the same level of coupling as `claude_bootstrap.py` checking for `ANTHROPIC_API_KEY` — it validates a contract, not an internal implementation detail.
>

### Q3) Where in the task list should the `add-codex-yunwu-agent` spec reconciliation happen?

- Decision: `Blocking`

#### Why this matters

The earlier Yunwu change spec (`add-codex-yunwu-agent`) explicitly requires `files/auth.json` as a compatibility file. This change removes that requirement. Without reconciling the old spec, the spec tree contains a formal contradiction.

#### Examples

- A developer reads the Yunwu spec and creates a new Codex profile with `{}` auth.json because the spec says it's required.
- Another developer reads the new spec and omits auth.json because it's optional.
- Both are "correct" per their respective specs — confusion ensues.

#### Why this question arises in code

- `openspec/changes/add-codex-yunwu-agent/specs/codex-openai-compatible-brain-profile/spec.md` lines ~20–27 define the conflicting requirement.
- The current tasks (section 2.2) mention updating "Codex/Yunwu fixture docs" but do not mention the Yunwu spec artifact itself.

#### Options

**Option A: Add a task to this change to supersede/update the Yunwu spec's auth.json requirement**

- Code impact:
  - Add task ~2.3: "Update or annotate the `add-codex-yunwu-agent` spec's `files/auth.json` requirement to note it is superseded by optional credential file mapping support."
- User-facing difference:
  - Spec tree is internally consistent after this change lands
- Pros:
  - Clean spec history; no contradictions
- Cons:
  - Slightly expands scope of this change

**Option B: Defer to a separate follow-up change or archive of the Yunwu change**

- Code impact:
  - No change to current tasks; add a note in the design's "Risks" section
- User-facing difference:
  - Temporary spec contradiction until follow-up lands
- Pros:
  - Keeps this change tightly scoped
- Cons:
  - Risk of the follow-up being forgotten; spec contradiction persists

> **PROPOSED: Add a task to this change (Option A) — add task 2.3 to update the Yunwu spec's auth.json requirement.**
> Rationale: The design already acknowledges "Spec drift with the unarchived Yunwu change" as a risk (design.md, Risks section). Adding a single task to annotate or supersede the conflicting requirement in `openspec/changes/add-codex-yunwu-agent/specs/codex-openai-compatible-brain-profile/spec.md` is minimal scope expansion (one file edit) and eliminates the formal contradiction. The alternative — deferring to a separate change — risks the follow-up being forgotten, especially since the Yunwu change is already described as "unarchived." Resolving it here keeps the spec tree honest.
