# Discussion: CAO resume addressed by manifest only

- Feature/Change: `cao-resume-manifest-only`
- Timestamp: `2026-03-01T18:34:32Z`
- Context: This doc enumerates open decision points discovered while reviewing `proposal.md`, `design.md`, `specs/brain-launch-runtime/spec.md`, and `tasks.md` for this breaking change.

Constraints:
- Breaking change is intentional (no compatibility flags / migration shims).
- For CAO-backed sessions, the session manifest should be a complete address for resumed operations.

---

## Summary of recommended defaults

1) (Q1) Treat `session_manifest.cao.api_base_url` as the sole CAO endpoint; require equality with `backend_state.api_base_url` and fail fast on mismatch.
2) (Q2) Use `SessionManifestError` for manifest validation failures; reserve `BackendExecutionError` for CAO network/HTTP failures.
3) (Q3) Also remove `--cao-profile-store` from `send-prompt`/`stop-session` to keep “manifest pointer is enough” coherent (optional but recommended).
4) (Q4) Add hermetic unit tests that assert resume uses the manifest URL and that inconsistent manifests fail fast; avoid live CAO integration in unit CI.

Blocking decisions: Q1, Q2, Q4.
Deferrable decision: Q3.

---

## Q1) What is the canonical CAO endpoint source in the session manifest?

### Why this matters

- If the endpoint source is ambiguous, “resume by manifest” can target the wrong CAO host and become unsafe/unreliable.
- This affects runtime resume behavior, error messaging, and test assertions.

### Options

**Option A: Canonicalize on `cao.api_base_url` and enforce invariants**

- Pros:
  - Matches the principle “manifest is the complete address”.
  - Keeps the canonical endpoint in the backend-specific typed section (`cao`), not an untyped blob.
  - Enables clear invariant checks (`cao.api_base_url == backend_state.api_base_url`).
- Cons:
  - Requires deciding what to do if either field is missing (hard error vs fallback).

**Option B: Canonicalize on `backend_state.api_base_url`**

- Pros:
  - Uses the common “backend_state” payload that already exists for all backends.
- Cons:
  - Less explicit; weaker semantics (easy to treat as an opaque bag of fields).
  - Leaves the `cao` section under-utilized even though it is already persisted and validated.

**Option C: Allow fallback from one field to the other**

- Pros:
  - Potentially more permissive if a malformed manifest is encountered.
- Cons:
  - Encourages “partially valid” manifests and makes it harder to reason about correctness.
  - Risks hiding bugs in manifest persistence.

### Proposal (Recommended)

Choose **Option A**: treat `session_manifest.cao.api_base_url` as canonical. Require it to be non-empty for `backend=cao_rest`, and add a defensive invariant check that it matches `backend_state.api_base_url` (fail fast if not). This makes the manifest a single, trustworthy address and prevents silent retargeting.

### Pros / Cons (Proposal)

- Pros:
  - Single source of truth for resumed operations.
  - Catches corrupted or manually edited manifests early.
  - Aligns with the change goal (“no override; mismatch is meaningless”).
- Cons:
  - Strictness may surface errors that previously only appeared as CAO “not found”/connection errors.

> **DECISION: Accept Option A — canonicalize on `cao.api_base_url` and enforce equality with `backend_state.api_base_url`; fail fast on mismatch.**
> Rationale: `_resume_cao_state()` in `runtime.py:290` currently falls back via `cao.api_base_url or api_base_url`, which allows an externally-passed parameter to silently override the manifest — directly violating the “manifest is complete address” contract. Meanwhile, `CaoSectionV1` in `boundary_models.py:105` already validates `api_base_url` as non-blank via `_not_blank` (lines 112-123), and `manifest.py:115-123` populates `cao.api_base_url` from `backend_state[“api_base_url”]` — so both fields always carry the same value at creation time. Implementation: replace the fallback `or` on line 290 with a hard requirement that `cao.api_base_url` is non-empty, add an invariant check that `cao.api_base_url == payload.backend_state.get(“api_base_url”)`, and remove the `api_base_url` parameter from `_resume_cao_state()` entirely.

## Q2) What error semantics should we guarantee for invalid CAO resume manifests?

### Why this matters

- CLI users need predictable failures when a manifest cannot be used (wrong backend, missing CAO fields, inconsistent payload).
- Tests should assert stable error classes/messages for these local validation failures.

### Options

**Option A: Local validation failures raise `SessionManifestError`; remote CAO failures raise `BackendExecutionError`**

- Pros:
  - Clean separation: “manifest is unusable” vs “CAO request failed”.
  - Matches existing exception taxonomy in `brain_launch_runtime`.
- Cons:
  - Requires deciding which cases are considered “local validation” (e.g., malformed URL string).

**Option B: Always raise `BackendExecutionError` for resume issues**

- Pros:
  - Simpler classification.
- Cons:
  - Loses an important distinction (bad manifest vs CAO down).
  - Makes it harder to write crisp unit tests for contract violations.

**Option C: Implicit fallback to defaults (e.g., `http://localhost:9889`)**

- Pros:
  - Avoids breaking in some misconfigured cases.
- Cons:
  - Violates the core guarantee of this change; unsafe and confusing.

### Proposal (Recommended)

Choose **Option A**. Define “local validation failures” as: missing `cao` section for `backend=cao_rest`, blank `cao.api_base_url`, blank `terminal_id`, and invariant mismatch between `cao.api_base_url` and `backend_state.api_base_url`. These raise `SessionManifestError` with a message that names the required field. Network/HTTP issues while contacting CAO remain `BackendExecutionError`.

### Pros / Cons (Proposal)

- Pros:
  - Strong, testable contract.
  - Better operator ergonomics (“fix your manifest” vs “CAO is down”).
- Cons:
  - Slightly more up-front effort to keep messages clear and consistent.

> **DECISION: Accept Option A — local validation failures raise `SessionManifestError`; remote CAO failures raise `BackendExecutionError`.**
> Rationale: The error hierarchy in `errors.py` already cleanly separates `SessionManifestError` (line 20, "manifest is invalid or unusable") from `BackendExecutionError` (line 28, "backend command fails to execute"). This taxonomy is already used correctly: `manifest.py:160-187` raises `SessionManifestError` for file/JSON/Pydantic failures, and `cao_rest.py` raises `BackendExecutionError` for CAO API/HTTP failures. The resume path in `runtime.py` raises `SessionManifestError` for headless `session_id` violations (line 265). Extending this pattern to CAO-specific manifest checks (missing `cao` section, blank `api_base_url`, blank `terminal_id`, invariant mismatch per Q1) is consistent and requires no new error classes — just new raise sites in `_resume_cao_state()` with field-path messages matching the `format_pydantic_error()` style already used in `manifest.py`.

## Q3) Should we also remove `--cao-profile-store` from resumed operations?

### Why this matters

- Keeping override-looking flags on resume commands undermines the “manifest pointer is enough” mental model.
- Even if the flag is ignored in practice, its presence increases confusion and support load.

### Options

**Option A: Remove `--cao-profile-store` from `send-prompt`/`stop-session` (BREAKING)**

- Pros:
  - CLI becomes minimal and unambiguous for resumed operations.
  - Matches the “no override” stance.
- Cons:
  - Additional breaking change beyond base URL.

**Option B: Keep it but document that it is ignored for CAO resume**

- Pros:
  - Avoids additional breaking surface.
- Cons:
  - Contradicts the design principle; users will still try to “fix” resume by tweaking flags.

### Proposal (Recommended)

Choose **Option A**, but treat it as **deferrable** if scope needs to stay tight. If implemented now, keep `--cao-profile-store` only on `start-session` (where it has real meaning).

### Pros / Cons (Proposal)

- Pros:
  - Cleaner, more coherent UX for resumed operations.
- Cons:
  - Slightly larger change footprint (docs + any scripts that pass the flag).

> **DECISION: Defer `--cao-profile-store` removal to a follow-up; deferral is safe because the flag is already dead code on resume paths.**
> Rationale: In `cli.py:100-101,106-107`, `--cao-profile-store` is defined on `send-prompt` and `stop-session`, and passed through to `resume_runtime_session()`. However, the demo scripts (`cao-claude-session/run_demo.sh` lines 249-263) never pass `--cao-profile-store` to resume operations — only `--cao-base-url`. The flag flows into `CaoRestSession.__init__` as `profile_store_dir` (in `cao_rest.py`), where it only affects profile installation during `start-session`, not resume. Removing it now would be coherent but increases the change surface (CLI args, function signatures, and any external callers). Since it has no effect on resume behavior, deferring is safe and keeps the breaking change focused on Q1's endpoint canonicalization.

## Q4) What is the right test strategy for “manifest-only CAO resume”?

### Why this matters

- This is a contract change; without tests, resume-time overrides can creep back in.
- Unit tests should be hermetic and not require a running CAO server.

### Options

**Option A: Pure unit tests over manifest parsing + resume state construction**

- Pros:
  - Hermetic, fast, stable in CI.
  - Verifies the key contract: resume uses the persisted URL and rejects inconsistencies.
- Cons:
  - Does not exercise live HTTP request paths (by design).

**Option B: Unit tests with monkeypatching to assert CAO client construction uses manifest URL**

- Pros:
  - Still hermetic, but more directly checks “client uses manifest endpoint”.
- Cons:
  - Slightly more coupled to implementation details.

**Option C: Integration tests against a real CAO server**

- Pros:
  - End-to-end coverage.
- Cons:
  - Flaky/slow; requires environment setup; better suited for manual demos than unit CI.

### Proposal (Recommended)

Use **Option A** as the baseline and optionally add **Option B** for a stronger assertion. Do not add CAO integration tests to unit CI; rely on existing demo packs/manual runs for end-to-end validation.

### Pros / Cons (Proposal)

- Pros:
  - Stable tests that directly protect the new contract.
- Cons:
  - Requires discipline to keep demos up to date as the end-to-end smoke tests.

> **DECISION: Accept Option A (pure unit tests) as the baseline, plus Option B (monkeypatching) for the client-construction assertion. No live CAO integration tests in unit CI.**
> Rationale: `test_runtime_resume.py` already demonstrates the exact pattern needed: it builds a manifest payload via `build_session_manifest_payload()`, mutates fields to create an invalid state, writes it to disk, and asserts that `resume_runtime_session()` raises `SessionManifestError` with a specific field name. The new tests should follow this pattern for three cases: (1) manifest with `cao.api_base_url` set to a non-default URL verifies that `_resume_cao_state()` uses the manifest URL, not the external parameter; (2) manifest with `cao.api_base_url != backend_state["api_base_url"]` raises `SessionManifestError` with "api_base_url" in the message; (3) manifest with blank `cao.terminal_id` raises `SessionManifestError`. For client construction (Option B), `test_cao_client_and_profile.py` already monkeypatches `urllib.request.urlopen` and mocks `CaoRestClient` — reuse that pattern to assert the `CaoRestSession` is initialized with the manifest's URL. Keep live CAO tests in `scripts/demo/` where they already exist.
