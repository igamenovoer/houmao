## Summary

- **Solid foundations:** The proposal, design, and specs form a coherent narrative that clearly separates interactive TUI tracking from headless Codex backend concerns. The motivation (naming leakage + missing temporal inference) is well-grounded in archived source-validated signal notes.
- **Good structural coverage:** All four capabilities (1 new, 3 modified) have matching delta specs with concrete WHEN/THEN scenarios, and the 12 tasks follow a logical four-phase progression (boundary cleanup → engine extension → profile implementation → live adapter).
- **Architectural gap in temporal hint delivery:** The specs and design require profile-owned temporal inference from sliding recent-snapshot windows, but neither the specs nor tasks specify the mechanism by which a profile receives or accumulates frame history, given that the current detector protocol (`BaseTrackedTurnSignalDetector.detect()`) is a stateless single-snapshot call.
- **Missing module-layout guidance:** Task 2.1 calls for refactoring so "concrete tool logic can live in app/profile-owned modules," but no task or spec defines the target directory/module structure for `codex_tui` profile code.
- **Naming migration scope is ambiguous:** `codex_app_server` appears in 51 files across the repo. The tasks reference "tracker-facing" renaming but don't explicitly scope what is and isn't renamed, risking accidental backend-layer changes.
- **Design open questions are not carried into tasks:** Two open questions at the bottom of `design.md` (renderer variants, latest-turn-region signature field) have no corresponding task and will be silently dropped.

## Findings

### Must Fix / Blocking

1. **The detector protocol cannot deliver sliding-window history to profiles.** The current `BaseTrackedTurnSignalDetector.detect()` signature at `detectors.py:149-159` accepts only `output_text: str | None` and `parsed_surface: ParsedSurfaceContext | None` — a single snapshot. The design says "profiles contribute temporal hints" and the shared-tui-tracking-core spec says "the shared core accepts profile-owned temporal hints over sliding recent windows," but no artifact specifies HOW the profile accumulates or receives multi-frame history. Three plausible mechanisms exist (extend the detector protocol to accept recent frames; make profiles stateful with an internal buffer fed by the engine; add a separate temporal-hint callback alongside `detect()`), and the choice fundamentally shapes both the engine extension (task 2.2-2.3) and the Codex profile implementation (task 3.2). Without this decision, implementers must make an architectural call not captured in the plan.

> **DECISION: Accept.**
> Rationale: The finding identifies a real architectural gap. The current detector boundary in [detectors.py](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/detectors.py#L149) is single-snapshot only, while this change requires Codex TUI temporal inference over recent windows. We should resolve this by introducing a separate profile-owned temporal-hint callback alongside single-snapshot analysis rather than overloading `detect()` or silently making the current detector API stateful. This needs follow-up artifact revision in `design.md`, the shared-core delta spec, the profile-contract delta spec, and the task list.

2. **Success-candidate behavioral change needs explicit callout.** The current `CodexTrackedTurnSignalDetector` at `detectors.py:478-488` requires a `Worked for ...` marker for `success_candidate=True`. The codex-tui-state-tracking spec explicitly says the `Worked for ...` separator "SHALL NOT be required for successful completion." This is a concrete behavioral inversion that changes when success settles. The design documents this (Decision: Codex TUI success is a stable ready return), but no task explicitly says "remove the `Worked for ...` gate from success-candidate logic." Task 3.3 covers "success settlement rules" but phrases it as new implementation rather than a migration from existing behavior. An implementer unfamiliar with the current detector could miss the required behavioral change.

> **DECISION: Refine.**
> Rationale: The reviewer is correct that this behavioral inversion must not stay implicit. The requirements and design already state the intended behavior, and Task 3.3 is directionally aimed at it, so the gap is not missing scope but missing explicit migration wording. The next artifact revision should name the current `Worked for ...` gate in `CodexTrackedTurnSignalDetector` and state that Codex success settlement must stop requiring it.

### Should Fix / Important

3. **No migration path specified for the existing `CodexTrackedTurnSignalDetector`.** The current detector at `detectors.py:426-524` already implements substantial single-snapshot Codex logic (prompt detection, working-row matching, steer handoff, interruption, error cells, approval hints). Task 3.1 says "Create a codex_tui profile module structure" but doesn't say whether the existing detector is evolved in-place, moved to a new module, or replaced. The existing code is also the basis for the current `codex_app_server` registry entry at `registry.py:72-76`. Leaving this unspecified risks either duplicated logic or a broken migration.

> **DECISION: Refine.**
> Rationale: The migration ambiguity is real, but we do not need a long-lived parallel path. The intended direction is to treat the current [CodexTrackedTurnSignalDetector](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/detectors.py#L426) as seed logic to be extracted into `codex_tui` profile-owned modules, then replace the tracker-facing registry entry rather than duplicate Codex logic in two places. The design/tasks should say this explicitly in a later revision pass.

4. **Target module layout for profile-owned code is unspecified.** Task 2.1 says "shared modules define contracts/helpers and concrete tool logic can live in app/profile-owned modules," but all current detectors live in a single flat file (`detectors.py`). The change should specify the target structure — e.g., whether `codex_tui` logic goes into `shared_tui_tracking/profiles/codex_tui/` or remains consolidated in `detectors.py` with a class rename. This affects import paths, test organization, and the Claude Code detector's future evolution.

> **DECISION: Accept.**
> Rationale: The change is intentionally moving away from the current flat [detectors.py](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/detectors.py#L1) layout, and the target structure should be explicit. The intended layout is shared contracts/helpers in the core package plus app-owned modules under `shared_tui_tracking/apps/claude_code_tui/` and `shared_tui_tracking/apps/codex_tui/` (with Codex signal families split into focused modules). That needs to be captured in `design.md` and Task 2.1.

5. **Design open questions are not tracked in tasks.** `design.md` ends with two unresolved questions: (a) whether the first `codex_tui` profile should treat `tui` and `tui_app_server` renderer differences as one shared profile or explicit variants, and (b) whether the shared normalized-signal model needs an explicit latest-turn-region signature field. Neither question has a corresponding task or decision gate. If these are intentionally deferred, that deferral should be explicit; if they need resolution before implementation, they need tasks.

> **DECISION: Accept.**
> Rationale: The two open questions should not remain dangling. The first renderer-variant question is deferred for v1: start with one `codex_tui` profile and split renderer variants later only if real divergence appears. The second question is resolved now: do not add a shared normalized-signal field for latest-turn-region signatures in v1; keep that signature profile-private inside the Codex temporal-hint implementation unless a second TUI family demonstrates the same need. The design and tasks should be revised to record those decisions explicitly.
