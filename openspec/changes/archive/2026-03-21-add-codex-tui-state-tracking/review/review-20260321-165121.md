## Findings

### Must Fix / Blocking

1. **Engine-to-profile temporal data flow is still unspecified in the artifacts.** The prior review (Finding 1, decision: Accept) agreed that a "separate profile-owned temporal-hint callback alongside single-snapshot analysis" should be introduced. The current artifacts have not been revised to reflect this decision. Specifically:
   - The `shared-tui-tracking-core` delta spec says "the shared core accepts profile-owned temporal hints" but does not describe the delivery mechanism.
   - The `versioned-tui-signal-profiles` delta spec says "profiles may contribute temporal hint logic" but does not specify whether the profile receives frames from the engine, maintains its own buffer, or uses a new callback.
   - Task 2.2 says "extend the shared tracker/profile contract to allow profile-owned temporal hint streams" but does not reference the accepted decision about a separate callback.
   - Task 2.3 says "implement shared engine support for profile temporal hints" but does not connect to the accepted callback-based design.

   The session engine currently calls `self.m_resolved_profile.detector.detect(output_text=raw_text)` at `session.py:210` as a single-snapshot stateless call. The accepted design needs to appear in the specs and tasks before implementation can proceed coherently.

> **DECISION: Accept.**
> Rationale: The reviewer is correct. The prior accepted decision about a separate profile-owned temporal callback has not yet been propagated into the artifacts, and the current specs/tasks still leave the engine-to-profile temporal path implicit. The next revision should explicitly define a separate temporal-hint callback plus engine merge/gating step rather than leaving implementers to infer it from the current single-snapshot `detect()` flow.

2. **`success_candidate` gate logic in the engine relies on the detector signal, but the engine does not distinguish "armed turn authority" from "initial idle."** The current `_handle_snapshot_event` at `session.py:260-282` enters the `success_candidate` branch whenever `signals.success_candidate` is True. For Claude Code, the `Worked for ...` marker makes this safe because it only appears after real turns. For `codex_tui`, the design explicitly removes the `Worked for ...` requirement (design.md line 103-104) and replaces it with "stable ready return after prior turn authority." But the engine's success-candidate path has no notion of "prior turn authority" — it relies entirely on the detector signal. This means either:
   - (a) the `codex_tui` detector must internally track turn authority (making it stateful, contradicting the single-snapshot contract), or
   - (b) the engine must gain a "prior turn authority required for success-candidate" guard, or
   - (c) the `codex_tui` temporal-hint callback must be the mechanism that gates success candidacy.

   None of these options is specified in the current artifacts. Without this, a Codex TUI that starts in ready-composer posture would immediately become a success candidate and settle as `success` — a misclassification the design explicitly wants to avoid (design.md line 111: "initial idle surfaces would be misclassified").

> **DECISION: Accept.**
> Rationale: This is a real gap in the current artifacts. The design requires prior turn authority before Codex ready-return success can settle, but the current engine path only branches on `signals.success_candidate`. The follow-up revision should state that Codex success candidacy is gated by prior armed turn authority in the shared engine/temporal-merge path, with both `explicit_input` and `surface_inference` qualifying, rather than pushing tracker state into a stateful detector.

## Design Alignment With Existing System

- **Naming pattern alignment:** The existing system uses `app_id` as the tracker-facing identity (e.g., `"claude_code"`) distinct from `BackendKind` (e.g., `"claude_headless"`). The proposed `codex_tui` follows this same pattern — a tracker-facing surface identity rather than a backend label. This is well-aligned with the existing `claude_code` precedent in `registry.py:58-59`.

- **Profile/detector architecture:** The existing `BaseTrackedTurnSignalDetector` → `DetectorProfileRegistration` → `DetectorProfileRegistry` pipeline is preserved. The change extends this with temporal hints rather than replacing it. This is architecturally conservative and correct.

- **Session engine extension:** The proposed temporal-hint mechanism adds complexity to `TuiTrackerSession`, which currently has a clean linear flow: `on_snapshot` → `detect()` → signal-branch reduction → timer management. Adding a second callback path (temporal hints) needs careful integration to avoid making the snapshot-handling logic in `_handle_snapshot_event` (session.py:206-292) fragile. The design's choice to keep temporal logic profile-owned rather than engine-owned is the right call — it limits the engine's Codex-specific knowledge.

- **Test pattern alignment:** Existing tests use `TestScheduler` + `TuiTrackerSession.from_config()` for timer-dependent behavior (see `test_session.py:31-58`). The proposed Codex temporal tests should follow this same pattern, extending it with multi-snapshot sequences and scheduler advancement.

- **Flat-file → modular structure migration:** The current codebase has all detectors in one `detectors.py` (603 lines). The proposed move to app-owned modules (per prior review Finding 4, decision: Accept, target: `shared_tui_tracking/apps/codex_tui/`) is a reasonable step given that Codex temporal logic will add significant per-tool complexity. The Claude Code detector should follow the same migration path to keep the architecture symmetric.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the temporal-hint callback produce a modified `DetectedTurnSignals` or a separate hint type?

- Decision: `Blocking`

#### Why this matters

- The engine's snapshot-handling reduction logic (`session.py:206-292`) is entirely driven by `DetectedTurnSignals` fields (`interrupted`, `known_failure`, `active_evidence`, `success_candidate`). If temporal hints produce a separate type, the engine needs a new merge/override step. If they modify `DetectedTurnSignals`, the single-snapshot purity of that type is compromised.

#### Examples

- Codex TUI: latest snapshot shows ready posture with no visible running row, but the temporal window shows transcript growth in the last 2 seconds. The temporal callback should produce `active_evidence=True` to override the single-snapshot's `active_evidence=False`. Does this come as a modified `DetectedTurnSignals` or as a `TemporalHint(active_override=True)` that the engine merges?

#### Why this question arises in code

- `session.py:210`: `signals = self.m_resolved_profile.detector.detect(output_text=raw_text)` — this is the only signal source entering the reduction logic.
- `session.py:247-258`: The `active_evidence` branch consumes `signals.active_evidence` directly. A temporal hint would need to inject here.
- `models.py:174-198`: `DetectedTurnSignals` has no temporal fields today.

#### Options

**Option A: Temporal callback returns a modified `DetectedTurnSignals` that replaces the single-snapshot result before engine reduction.**

- Code impact:
  - Engine calls `detect()` for single-snapshot, then calls `temporal_hints()` which returns a new `DetectedTurnSignals` with overrides applied. Engine uses only the final result.
  ```python
  signals = profile.detect(output_text=raw_text)
  signals = profile.apply_temporal_hints(base=signals, recent_frames=window)
  # Engine reduction uses 'signals' as before
  ```
- User-facing difference:
  - None — the public `TrackedStateSnapshot` contract is unchanged.
- Pros:
  - Engine reduction logic stays untouched — it still consumes one `DetectedTurnSignals`.
  - Profile owns all signal-level decisions, including temporal overrides.
- Cons:
  - `DetectedTurnSignals` becomes semantically ambiguous — sometimes it's single-snapshot-derived, sometimes temporal.
  - Harder to trace in diagnostics whether a field came from snapshot or temporal logic.

**Option B: Temporal callback returns a separate `TemporalHintSignals` type; engine has an explicit merge step before reduction.**

- Code impact:
  - New dataclass `TemporalHintSignals` with optional override fields. Engine merges hints into signals before entering the reduction branches.
  ```python
  signals = profile.detect(output_text=raw_text)
  hints = profile.temporal_hints(recent_frames=window)
  merged = _merge_temporal_hints(signals, hints)
  # Engine reduction uses 'merged'
  ```
- User-facing difference:
  - None — same public contract.
- Pros:
  - Clear separation between single-snapshot facts and temporal inferences.
  - Diagnostics can trace exactly which fields were overridden by temporal logic.
  - `DetectedTurnSignals` retains its single-snapshot purity.
- Cons:
  - Adds a new type and a merge function to the engine.
  - Merge logic is a new surface area for bugs if override semantics aren't well-defined.

> **PROPOSED: Option B — separate `TemporalHintSignals` type with an explicit engine merge step.**
> Rationale: The existing `DetectedTurnSignals` at `models.py:174-198` is documented as "normalized tracked turn signals for one visible surface" — it's a single-snapshot contract by definition. Introducing temporal overrides into the same type silently changes its semantics. A separate `TemporalHintSignals` type preserves the single-snapshot purity, makes diagnostic tracing straightforward (the engine can log both the raw snapshot signals and the temporal hints separately, matching the existing `_trace("detector_signals", ...)` pattern at `session.py:212-218`), and keeps the merge semantics explicit rather than hidden inside the profile. The `_merge_temporal_hints` function is new surface area, but it's small, testable, and its rules are directly derived from the spec requirements (e.g., "temporal active evidence overrides missing running row").
>
> **DECISION: Accept PROPOSED (Option B).**
> Rationale: `DetectedTurnSignals` is currently the normalized output for one visible surface, and preserving that single-snapshot meaning is the cleaner architecture. A separate temporal-hint type plus explicit merge/gating step gives the engine one clear place to combine snapshot facts with recent-window inference, keeps tracing legible, and aligns with the prior accepted decision to add a separate temporal callback rather than silently changing `detect()`.

### Q2) Should `codex_tui` success candidacy require explicit `on_input_submitted` turn authority, or should surface-inferred turn authority (from prior `active_evidence`) also qualify?

- Decision: `Blocking`

#### Why this matters

- The design says success requires "prior turn authority" but doesn't define what counts as turn authority for Codex TUI. The engine currently has two turn-source paths: `explicit_input` (from `on_input_submitted`) and `surface_inference` (armed when `active_evidence` first appears, at `session.py:249-250`). If the live adapter doesn't call `on_input_submitted` for Codex sessions, only surface-inferred authority is available.

#### Examples

- A Codex TUI session starts, the user types a prompt via tmux send-keys, the live adapter does NOT call `on_input_submitted` (because it captures snapshots, not keystrokes), Codex works, returns to ready. Should this settle as success?
- Same scenario but the live adapter DOES call `on_input_submitted` when it detects a prompt submission event from the server layer. Should the behavior differ?

#### Why this question arises in code

- `session.py:184-204`: `_handle_input_event` sets `m_armed_turn_source = "explicit_input"`.
- `session.py:247-250`: Active evidence arms `m_armed_turn_source = "surface_inference"` if not already armed.
- `session.py:373-380`: `_terminal_turn_source_locked()` resolves the turn source for settled outcomes.
- The current engine does NOT require explicit input for success candidacy — it's purely detector-driven. The question is whether the `codex_tui` profile should change this.

#### Options

**Option A: Accept both explicit-input and surface-inferred turn authority for Codex TUI success.**

- Code impact:
  - No change to the engine's `m_armed_turn_source` logic. The `codex_tui` temporal callback gates success candidacy based on whether `m_armed_turn_source` is set (either source).
- User-facing difference:
  - Codex TUI success works even when the live adapter only captures snapshots without explicit input events.
- Pros:
  - More resilient to live-adapter implementation details.
  - Matches the design's "prior turn authority" language without restricting the authority source.
- Cons:
  - Slightly weaker correctness guarantee — surface-inferred authority could theoretically be wrong.

**Option B: Require `on_input_submitted` for Codex TUI success settlement.**

- Code impact:
  - The `codex_tui` temporal callback or a new engine guard checks that `m_armed_turn_source == "explicit_input"` before allowing success candidacy.
- User-facing difference:
  - Codex TUI success only settles when the live adapter explicitly reports prompt submission. Snapshot-only tracking cannot settle success.
- Pros:
  - Stronger correctness — explicit input events are more reliable than surface inference.
- Cons:
  - Requires the live adapter to reliably call `on_input_submitted` for Codex sessions, which may not be implemented yet.
  - Breaks snapshot-only replay/testing workflows.

> **PROPOSED: Option A — accept both explicit-input and surface-inferred turn authority.**
> Rationale: The existing engine at `session.py:247-250` already arms `m_armed_turn_source = "surface_inference"` when `active_evidence` first appears, and the success timer at `session.py:373-380` accepts either source. The design says "stable return to ready posture after prior turn authority" without restricting the authority source to explicit input. Requiring `on_input_submitted` would couple success settlement to live-adapter implementation details that may not exist yet for Codex sessions, and would break replay-based testing where only snapshots are available. The `codex_tui` temporal callback can additionally verify that prior active evidence existed in the recent window, providing a stronger guard than source type alone.
>
> **DECISION: Accept PROPOSED (Option A).**
> Rationale: Both explicit-input and surface-inferred authority should qualify. That matches the existing shared-core contract for explicit and inferred turn authority, preserves snapshot-only replay viability, and avoids coupling Codex success settlement to whether a live adapter happens to emit `on_input_submitted()`. The missing piece is not the source choice; it is making the authority guard explicit in the engine/temporal-merge path.
