## Context

This repo currently keeps agent profiles under `agents/<agent-name>/` with per-tool runtime “home” directories under `agents/<agent-name>/homes/{codex,claude,gemini}`. Those homes are generated by duplicated `bootstrap.sh` scripts that copy/symlink content from workspace-local `.codex/`, `.claude/`, and `.gemini/`.

This approach has three practical limitations:

1. Homes are “hand-assembled” and tightly coupled to each agent directory, making it hard to compose an agent from reusable parts (tool + skills + config + credentials).
2. There is no first-class concept of selecting among multiple credential profiles to balance usage across accounts.
3. Roles (system prompts) are intermingled with the “brain” runtime state, rather than being a reusable, tool-agnostic package.

We want to build agents in two phases:

1) **Construct the brain** (tool runtime): choose tool + skills + tool config + credential profile → build a fresh runtime CLI home directory.

2) **Apply the role** (behavior bias): start the CLI tool pointing at that home dir, then use a tool-agnostic system prompt (plus supporting files) to bias behavior.

Important nuance: some CLI tools accept credentials and routing configuration via **environment variables** (e.g., API keys), so a complete “brain” is not just a home directory. Brain construction must also define the **launch environment** (env var set) sourced from the selected credential profile.

Key constraints:
- Fresh at creation: constructed homes start empty (no pre-existing history/logs/sessions).
- Home lifecycle: homes are created by script; deletion is manual (developer decides whether to preserve history by keeping the directory).
- Secrets must never be committed; credential profiles are local-only.
- Roles must be brain-agnostic (tool-independent).

## Goals / Non-Goals

**Goals:**
- Define a clear, component-based on-disk layout for:
  - brain components (skills, tool configs, credentials, recipes)
  - role packages (system prompt + referenced files)
  - generated runtime homes and resolved manifests
- Provide a single “brain builder” entrypoint that constructs a fresh tool home dir from selected components.
- Make multi-credential selection explicit and safe-by-default (local-only creds, no accidental commits).
- Reduce duplicated per-agent bootstrap logic by making brain construction declarative.

**Non-Goals:**
- Implement a full multi-agent orchestration runtime (CAO integration can consume the outputs, but is not the primary deliverable here).
- Define one universal credential format across tools (each tool’s adapter defines what files are considered credentials).
- Automatic GC/deletion of old homes (deletion remains manual).

## Decisions

### 1) Separate “brain components” from “roles”

We adopt a split under `agents/`:

- `agents/brains/…` is the source-of-truth store for constructing tool homes.
- `agents/roles/…` stores role packages that are tool-independent.
- A generated runtime root (configurable; often under `tmp/` or outside the repo) stores constructed homes and manifests.

Rationale: this makes composition explicit and reduces cross-contamination between runtime state (brain) and reusable behavioral intent (role).

### 1.1) Directory layout (source-of-truth vs generated runtime)

The design separates **committed component repositories** from **generated runtime state**:

```text
agents/
  brains/                              # committed: brain component repos
    skills/                            # tool-agnostic skills (Agent Skills format)
      <skill-name>/SKILL.md
      packs/<pack>.yaml                # optional: named skill sets
    cli-configs/                       # committed: tool-specific config profiles (no secrets)
      codex/<profile>/config.toml
      claude/<profile>/...
      gemini/<profile>/...
    api-creds/                         # local-only (gitignored): credential profiles
      codex/<cred-profile>/            # may include both files and env vars
        files/...
        env/vars.env
      claude/<cred-profile>/
        files/...
        env/vars.env
      gemini/<cred-profile>/
        files/...
        env/vars.env
    tool-adapters/                     # committed: per-tool home layout + projection rules
      codex.yaml
      claude.yaml
      gemini.yaml
    brain-recipes/                     # committed: declarative brain presets (secret-free)
      codex/<recipe>.yaml
      claude/<recipe>.yaml
      gemini/<recipe>.yaml

  roles/                               # committed: brain-agnostic role packages
    <role-name>/
      system-prompt.md
      files/...

  blueprints/                          # optional, committed: {brain recipe + role} bindings
    <agent-name>.yaml
```

Generated runtime state is written under a configurable runtime root (examples: `tmp/agents-runtime/` inside the repo, or `~/.local/state/agent-system-dissect/agents/` outside the repo):

```text
<runtime_root>/                        # generated (NOT committed)
  homes/                               # constructed CLI home dirs (tool state lives here)
    <tool>/
      <home-id>/...
  manifests/                           # resolved, secret-free manifests per constructed home
    <tool>/
      <home-id>.yaml
  locks/                               # optional locks for shared credential profiles
    <tool>/
      <cred-profile>.lock
```

An example constructed runtime home for Codex (files beyond `config.toml` / `skills/` will appear only after the tool runs):

```text
<runtime_root>/
  homes/
    codex/
      brain-20260226-0823Z-abc123/
        config.toml
        skills/
          <skill-name>/...
        auth.json -> ../../../../brains/api-creds/codex/<cred-profile>/files/auth.json
        .env -> ../../../../brains/api-creds/codex/<cred-profile>/env/vars.env
        history.jsonl                  # created by tool after first run
        sessions/                      # created by tool after first run
        log/                           # created by tool after first run
  manifests/
    codex/
      brain-20260226-0823Z-abc123.yaml
  locks/
    codex/
      <cred-profile>.lock
```

### 2) Fresh-by-default runtime home construction

The brain builder always constructs a *fresh* tool home directory:
- no history/logs/sessions are populated at construction time
- any persisted state only appears after the tool runs

Rationale: avoids “time travel” bugs caused by stale state and makes constructed brains reproducible from their inputs.

Alternative considered: “always reuse a stable home path per agent”. Rejected because it hides state reuse and makes it harder to reason about which credentials/configs were used.

### 3) Declarative recipes + optional blueprints

We introduce declarative inputs:
- **Brain recipes**: select `{tool, skills, cli-config profile, credential profile}`.
- **Agent blueprints** (optional): bind `{brain recipe, role}` into a named definition for convenience.

Rationale: developers can keep reusable brain presets without coupling to a single role, and can share blueprints without leaking credentials (credential profile names only; actual secrets remain local).

### 4) Tool adapters define “home layout” and projection rules

Each tool is different (config path, skills directory, credential filenames, environment variables). Rather than hardcoding these differences inside the builder, we define per-tool “adapter” files under `agents/brains/tool-adapters/`.

An adapter defines:
- where config should be written within the tool home
- where skills should be installed/linked within the tool home
- which files count as credentials and where they live in the tool home
- which environment variables are sourced from credential profiles, and how they are applied at launch
- how the tool is pointed at the constructed home (env vars and/or CLI flags)

Rationale: keeps the builder generic and makes adding a new CLI tool mostly a data/config change.

Alternative considered: a single script with `if tool == ...` branches. Rejected because it becomes unmaintainable as tools evolve.

### 5) Local-only credential store + projection into tool homes

Credential profiles live under `agents/brains/api-creds/<tool>/<cred-profile>/` and are explicitly gitignored.

At build time, credential files are projected into the tool home as defined by the adapter. The default projection mode should be symlink-based where safe, so interactive login/refresh updates the underlying credential profile.

Credential profiles may also define environment variables (for tools that authenticate via env vars). A standard location (e.g., `env/vars.env`) stores key/value pairs for the credential profile; tool adapters define an allowlist of supported env keys and whether the tool can read a home-local dotenv file (e.g., Codex reads `$CODEX_HOME/.env`) or requires the launcher to export env vars in-process.

Rationale: enables selecting among multiple accounts without copying secrets into committed locations and supports “login once, reuse many brains” safely.

### 6) Brain launch environment is part of the brain contract

To make “brain construction” complete for env-auth tools, the brain builder must emit a launch contract that includes:
- the runtime home directory path
- the tool home selector (env var / CLI flag) needed to point the tool at that home
- the set of env var **names** to apply at launch (values remain only in the local credential profile)
- the path to the credential env file (local-only) used to source those values

The resolved runtime manifest MUST NOT include secret values; it can include env var names and local paths.

## Risks / Trade-offs

- **Symlink portability**: some environments/tools may not handle symlinked credential files. Mitigation: adapter supports copy-mode fallback.
- **Concurrent use of one credential profile**: multiple brains sharing one cred profile can race on token refresh. Mitigation: optional per-cred lock file under `<runtime_root>/locks/…`, and documented guidance to use distinct cred profiles when balancing usage.
- **Directory churn**: fresh homes create many directories. Mitigation: manual deletion policy + optional naming conventions for homes.
- **Migration complexity**: existing workflows rely on `agents/<agent>/bootstrap.sh` and `agents/<agent>/homes/...`. Mitigation: provide a migration path and (temporarily) keep legacy scripts as wrappers around the new builder.
- **Env var leakage**: env-auth tools can leak secrets via shell history or process inspection if launched naively. Mitigation: use local-only env files, avoid printing values, and prefer per-process env injection (not global shell exports) where possible.

## Migration Plan

1. Add the new `agents/brains/` and `agents/roles/` structure alongside existing agent directories.
2. Create initial tool adapters for currently used tools (codex/claude/gemini) and a minimal brain builder CLI.
3. Move/duplicate existing prompts into `agents/roles/<role>/system-prompt.md` (keeping legacy locations temporarily).
4. Replace per-agent `bootstrap.sh` with a thin wrapper (or documentation) that invokes the brain builder to create a fresh home.
5. Update docs to point to the new composition workflow (brain build → tool launch with home → apply role).

## Open Questions

- Should `agents/brains/api-creds/` be inside the repo (gitignored) or recommended as a local symlink to an external secret directory? (Design assumes inside-repo gitignored by default.)
- For each tool, which files should be treated as “credential material” vs “session history” (adapter definitions must be precise to avoid copying history into creds or leaking creds into runtime logs)?
