## Summary

- **Solid foundation:** The proposal, design, and spec are well-structured, internally coherent, and correctly scoped. The two-layer split (TerminalSessionRegistry + WorkspaceController) is the right architectural move for decoupling session transport from visual layout.
- **Spec coverage is good:** All four spec requirements (reorder, session continuity, action routing, restore) have concrete WHEN/THEN scenarios that are testable.
- **Build tooling gap:** The change introduces a Dockview npm dependency but the existing Tailmux project is a zero-build, CDN-loaded static frontend. The design mentions "Bun-first frontend build wiring" but neither the design nor the tasks specify the bundler, build pipeline, or how Dockview assets will be served. This is the biggest implementation risk.
- **Dockview vanilla/core API compatibility is unverified:** Dockview's published vanilla JS integration (non-React) may have API surface constraints (e.g., no built-in `onWillDrop` interception for single-group enforcement). The design assumes blocking split-creating drops, but the exact API mechanism is not validated.
- **Persistence design is under-specified:** The design says "persist Dockview layout JSON together with restorable session descriptors" but does not specify the storage key structure, versioning, or migration strategy for localStorage schema.
- **Task granularity is appropriate** for execution but verification tasks (5.x) lack acceptance criteria.

## Findings

### Must Fix / Blocking

1. **Task 1.1 ("Add Dockview and minimal Bun-first frontend build wiring") is under-specified for a blocking prerequisite.**

   The current Tailmux frontend is fully CDN-loaded with no build step (`public/index.html` loads xterm and fit-addon from `cdn.jsdelivr.net`, `app.js` is a plain script). Dockview's `dockview-core` package is distributed as ESM and requires a bundler or import-map setup. Task 1.1 needs to specify the exact integration path: will Tailmux add a Bun-based bundler (e.g., `Bun.build` or esbuild via Bun), use an import map with a CDN-hosted ESM build of dockview-core, or adopt a different approach? Without this, later tasks (3.1, 3.2) that depend on Dockview APIs cannot proceed.

   The `package.json` currently has zero `devDependencies` and no build script. The design says "Bun-first" but does not define what that means concretely for the frontend asset pipeline.

> **DECISION: Accept.**
> Rationale: This is a real artifact gap. `extern/tracked/tailmux/package.json` still has no build script or frontend tooling, while `extern/tracked/tailmux/public/index.html` is a zero-build CDN page. We will revise the design/tasks to make the Bun-driven ESM bundle an explicit prerequisite rather than leaving Task 1.1 abstract.

2. **Single-group constraint mechanism (Task 3.3) needs a concrete Dockview API strategy.**

   Design Decision 4 says "Block or normalize drop targets that would create additional visible groups." Dockview's vanilla API exposes `onWillDragPanel` and `onWillDrop` events, but the exact mechanism to intercept and prevent group-creating drops differs between dockview versions and may require overriding the `DockviewGroupPanel` drop target behavior. The tasks should identify the specific Dockview API hooks that will be used (e.g., `createWatermarkComponent`, `showDndOverlay`, or `onWillShowOverlay`) and confirm they are available in the target dockview-core version.

   If Dockview does not provide a clean interception point, the fallback of "normalize after drop" (merge back to single group) should be specified as an explicit subtask.

> **DECISION: Refine.**
> Rationale: The review is correct that the single-group rule needs a concrete fallback path, but the OpenSpec layer does not need to lock itself to exact Dockview hook names yet. We will revise the artifacts to require one explicit enforcement strategy: use pre-drop blocking when the pinned Dockview version supports it, otherwise normalize immediately after drop back to one visible group and capture that validation in tasks.

### Should Fix / Important

3. **Persistence schema (Tasks 4.2, 4.3) lacks storage key design and versioning.**

   The design correctly identifies two kinds of persisted state (Dockview layout JSON + restorable session descriptors) but does not specify:
   - The `localStorage` key names and whether they replace or coexist with any existing stored state.
   - A schema version field to enable future migration (e.g., when split layouts are added later).
   - What happens when stored layout references panels whose session descriptors are missing or have changed mode.

   The spec scenario "tmux-backed tabs restore in their prior order" is testable only if the storage format is defined. Consider adding a subtask under 4.2 that defines the persistence schema explicitly.

> **DECISION: Accept.**
> Rationale: The persistence design needs a defined localStorage schema, including key names, a version field, and handling for stale panel/session references. That is required to make the restore requirement testable and to keep room for later layout evolution without ad hoc migration behavior.

4. **Terminal DOM reparenting vs. persistent renderer strategy needs clarification.**

   Design Decision 3 says "Dockview panels should use a persistent renderer mode so hidden or moved tabs keep DOM-backed terminal state intact." Dockview-core's vanilla API may or may not support a `renderer: 'always'` mode for non-React panels. The alternative is to keep terminal host elements outside the Dockview container and project them into panel content areas on activation (a "portal" pattern). The tasks should specify which approach will be used and verify it against the target dockview-core version's `IContentRenderer` contract.

   In the current codebase (`app.js:342-406`), each terminal wrapper is created as a child of `#terminal-container` and toggled via `.active` class. The Dockview migration must decide whether terminal wrappers remain in `#terminal-container` and are projected into panels, or are moved into Dockview's own DOM tree.

> **DECISION: Accept with refinement.**
> Rationale: We do need to decide DOM ownership now, but we do not need to overfit the plan to one Dockview API name before implementation. The artifact revision should state the real invariant: each session owns one stable xterm host/root, and Dockview panel content must attach that stable root without recreating the terminal; if persistent rendering is insufficient in the pinned version, we will keep the stable host and adapt the panel shell around it.

5. **Tab action buttons (tmux-new-window, tmux-rename) have hard DOM references that will break.**

   The current `index.html` embeds `#tmux-new-window-btn` and `#tmux-rename-btn` inside `#tab-bar > .tab-actions` (lines 49-66). Task 1.2 says "Replace the custom browser tab-strip markup with a Dockview workspace container while preserving the existing header, dashboard, and mobile control shells." The tab-action buttons live inside the tab bar that is being replaced. The tasks should explicitly state where these buttons will be relocated (e.g., into a custom Dockview header component, into the main `#header`, or into the Dockview tab renderer).

   `setupTmuxFastActions()` in `app.js:150-255` queries these buttons by ID at init time. If the buttons move or are lazily rendered, this setup code will silently fail.

> **DECISION: Accept.**
> Rationale: The review correctly identifies a concrete break in the current plan. Those buttons currently live inside the tab bar being removed, and the design already says phase 1 keeps an external action strip. We will revise the artifacts to relocate tmux-new-window and tmux-rename into a persistent header/workspace toolbar that targets the active session instead of tying them to Dockview tab chrome.

6. **`handleWindowResize()` and terminal fit logic are tightly coupled to `activeTabId`.**

   `handleWindowResize()` (`app.js:59-74`) only refits the active tab's terminal. After the migration, Dockview will manage panel sizing, and the fit logic should respond to Dockview's panel resize events rather than the window resize event. The current approach of `setTimeout(() => fitAddon.fit(), 0)` in `switchTab()` (`app.js:517-520`) will also need to be replaced with Dockview's panel lifecycle hooks. This coupling is not called out in any task.

> **DECISION: Accept.**
> Rationale: `activeTabId`-driven fit logic is one of the main couplings this migration is supposed to remove. The revised tasks/design should explicitly move terminal fit/focus behavior onto Dockview activation and panel-dimension events, with window-level resize retained only as a higher-level layout trigger.

### Nice To Have

7. **Verification tasks (5.x) lack measurable acceptance criteria.**

   Tasks 5.1–5.3 say "Verify drag-reorder, activate, and close behavior" but do not define success criteria. For a change that replaces the entire tab management system, consider adding specific pass/fail conditions:
   - 5.1: List exact session modes to test (new, tmux, attach) and expected outcomes per action.
   - 5.2: Define what "terminal continuity" means operationally (e.g., scrollback preserved, no WebSocket close/reopen, no xterm buffer reset).
   - 5.3: Define restore round-trip test: create N tabs of mixed modes, arrange, reload, verify order + active tab + skipped shell tabs.

> **DECISION: Accept.**
> Rationale: The verification section needs concrete pass/fail criteria, especially because this change replaces the whole tab-management path. We will revise the tasks so continuity, restore, and mixed-session behavior are expressed as observable acceptance checks rather than generic “verify” steps.

8. **The TODO.md in Tailmux mentions split 2-view and 4-view layouts as goals, while this change explicitly defers them.**

   `extern/tracked/tailmux/TODO.md` lists "dockable terminal groups", "split 2-view and 4-view layouts", and "panel attach/detach between groups" as goals. The proposal and design correctly scope this change to single-group tab reordering only. The TODO.md should be updated as part of this change to reflect the phased approach, so future contributors don't assume all TODO items are in scope.

> **DECISION: Refine.**
> Rationale: The scope mismatch is real, but it is not a planning blocker for the OpenSpec change itself because the proposal/design/specs already define the phase correctly. We should still update `extern/tracked/tailmux/TODO.md` during the implementation/docs pass so the submodule-facing roadmap reflects the phased rollout.

9. **Mobile drag-and-drop ergonomics are acknowledged as desktop-first but no fallback UX is specified.**

   Design risks mention "Mobile drag-and-drop ergonomics may be weaker than desktop." On mobile, Dockview's built-in drag handling may be poor or non-functional (touch DnD is notoriously inconsistent). Consider adding a task to verify touch drag behavior or provide a fallback tab-reorder mechanism (e.g., long-press context menu with "move left"/"move right") if drag fails on mobile.

> **DECISION: Refine.**
> Rationale: We should add explicit mobile verification and document desktop-first behavior, but a separate fallback reorder UX would expand v1 beyond the current goal. The artifact revision should require touch-behavior validation and a known limitation path, not a second tab-reordering mechanism.

10. **CDN-loaded xterm.js and fit-addon may conflict with a bundled Dockview setup.**

    Currently, `index.html` loads xterm and xterm-addon-fit via CDN script tags (lines 241-242), which expose `Terminal` and `FitAddon` as globals. If Task 1.1 introduces a bundler for Dockview, the project will have a mixed module strategy (CDN globals + bundled ESM). Consider whether xterm should also be moved into the bundled dependency graph to avoid version drift and global/module conflicts.

> **DECISION: Accept.**
> Rationale: Once we choose a bundled ESM frontend path for Dockview, mixing bundled Dockview with CDN-global xterm would create an unnecessary split dependency model. We will revise the plan so xterm and the fit addon move into the same Bun-managed frontend bundle as part of the foundational tooling change.

## Design Alignment With Existing System

- **Existing pattern observed:** Tailmux is a zero-build, CDN-loaded, vanilla JS frontend with a Node/Express + WebSocket backend. State is managed via a single `tabs` Map with `activeTabId` as the global cursor. All UI is imperative DOM manipulation. The backend (`server.js`) serves static files from `public/` and manages pty/tmux processes via WebSocket. No framework, no module system, no build step.

- **Proposed design alignment:**
  - The TerminalSessionRegistry / WorkspaceController split **aligns well** with the existing separation between transport logic (WebSocket, reconnect, pty) and UI logic (tab DOM, active class toggling). The current `tabs` Map already implicitly contains both concerns; the proposed split makes this explicit.
  - The "Bun-first frontend build" aspect **mismatches** the current architecture. Tailmux has no build step and the `package.json` has no build script. Introducing a bundler is a significant architectural change that should be treated as a foundational subtask with its own verification, not bundled into "add Dockview."
  - The proposal to keep header/dashboard/mobile controls external to Dockview **aligns** with the current layout where these are separate DOM regions (`#header`, `#dashboard`, `#virtual-keyboard`, `#tmux-panel`). The key integration point is replacing `activeTabId` references in these controls with queries to the WorkspaceController.
  - The `#terminal-container` element (line 234 in `index.html`) currently holds all `.terminal-wrapper` divs. The Dockview container will presumably replace or wrap this element. The design should clarify this DOM relationship since `styles.css` has fixed height calculations (`height: calc(100vh - 92px)`) for `#terminal-container` that assume the current header+tab-bar height.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What is the concrete frontend build strategy for loading Dockview?

- Decision: `Blocking`

#### Why this matters

- Dockview-core is distributed as ESM. The current Tailmux frontend has no module system — it loads scripts via CDN `<script>` tags and uses globals. The entire implementation plan depends on being able to `import` Dockview APIs.

#### Examples

- If using a bundler: `app.js` must be refactored from a global-scope script to an ESM entry point, and the build output must be served from `public/` or a `dist/` directory.
- If using import maps: a `<script type="importmap">` block must be added to `index.html`, and a CDN that serves dockview-core as ESM must be identified (e.g., esm.sh, skypack, or self-hosted).

#### Why this question arises in code

- `package.json` has no `build` script and zero `devDependencies`.
- `index.html` loads all JS via plain `<script>` tags (lines 241-243).
- `app.js` is a 1296-line global-scope script with no `import`/`export` statements.
- The design says "Bun-first" but Bun's build API (`Bun.build()`) requires an explicit entry point and output configuration.

#### Options

**Option A: Bun bundler with ESM entry point**

- Code impact:
  - Add `dockview-core` to `package.json` dependencies.
  - Add a `build` script using `Bun.build()` or `bun build` CLI.
  - Refactor `app.js` into an ESM module.
  - Serve bundled output from `public/` or `dist/`.
  - Update `server.js` static file serving if output path changes.
- User-facing difference:
  - Development requires a build step (`bun run build` or a watch mode).
  - Production deployment adds a build phase.
- Pros:
  - Full control over bundled output; tree-shaking removes unused Dockview code.
  - Consistent module system for all frontend code.
  - Aligns with "Bun-first" stated intent.
- Cons:
  - Adds build complexity to a previously zero-build project.
  - May require also bundling xterm to avoid CDN/bundle mismatch.

**Option B: Import maps with CDN-hosted dockview-core ESM**

- Code impact:
  - Add `<script type="importmap">` to `index.html` mapping `dockview-core` to a CDN URL.
  - Convert `app.js` to `<script type="module">` with `import` statements.
  - No bundler required.
- User-facing difference:
  - No build step; development remains edit-and-reload.
  - Depends on CDN availability at runtime.
- Pros:
  - Minimal tooling change; preserves zero-build simplicity.
  - Faster to implement initially.
- Cons:
  - Import map browser support requires modern browsers (Chrome 89+, Firefox 108+, Safari 16.4+).
  - CDN dependency at runtime; no offline support.
  - Cannot tree-shake unused Dockview modules.
  - Mixing CDN globals (xterm) with ESM imports may cause ergonomic friction.

> **PROPOSED: Option A — Bun bundler with ESM entry point.**
> Rationale: The design document already commits to "Bun-first frontend build wiring" (Design Decision 1, Migration Plan step 1). Tailmux's `package.json` already specifies `node >= 20` and is part of a Bun-capable project. A bundler gives full control over the Dockview integration surface, enables tree-shaking the unused React/Angular layers from dockview, and positions the project for future dependency management (e.g., bundling xterm too). The cost is a build step, but this is a one-time setup that aligns with the stated design intent. Task 1.1 should be split into a sub-task for the build pipeline setup and a sub-task for adding the Dockview dependency.
>
> **DECISION: Accept Option A with one refinement: move xterm and the fit addon into the same Bun-managed frontend bundle.**
> Rationale: We already shifted Tailmux workflow toward Bun, and the current zero-build page in `extern/tracked/tailmux/public/index.html` is the exact constraint this migration must make explicit. A Bun-driven ESM bundle is the cleanest way to introduce Dockview without leaving the frontend in a mixed CDN-global / package-import state.

### Q2) Should the "Reset workspace layout" action be included in this phase?

- Decision: `Deferrable`

#### Why this matters

- The proposal's open questions explicitly ask this. A corrupted or stale layout in localStorage could leave users unable to recover their workspace without manually clearing storage or using browser devtools.

#### Examples

- User drags tabs into a state that feels broken (e.g., all tabs ended up in an unexpected order after a glitchy drag). Without a reset action, they must open devtools → Application → Local Storage → delete the key.
- Stored layout references sessions that no longer exist and the restore logic shows multiple "skipped" toasts on every reload with no way to clear the stale layout.

#### Why this question arises in code

- The persistence tasks (4.2, 4.3) introduce stored layout state that will survive across reloads.
- There is no existing "reset" mechanism in the current codebase.
- The dashboard (`index.html:108-145`) has an actions section that could host a reset button.

#### Options

**Option A: Include a minimal "Reset layout" button in the dashboard**

- Code impact:
  - Add a button to `dashboard-actions` div.
  - On click: clear layout + session descriptor keys from localStorage, reload.
- User-facing difference:
  - Users have a visible escape hatch for layout issues.
- Pros:
  - Low implementation cost (a button + `localStorage.removeItem()` + `location.reload()`).
  - Prevents support requests from users stuck with broken layouts.
- Cons:
  - Minor scope expansion.

**Option B: Defer to a later layout-management pass**

- Code impact:
  - None in this phase.
- User-facing difference:
  - Users must use devtools to clear layout if it becomes corrupted.
- Pros:
  - Keeps scope minimal.
- Cons:
  - Risk of user frustration if early layout persistence has bugs.

> **PROPOSED: Option A — Include a minimal "Reset layout" button in the dashboard.**
> Rationale: The implementation cost is trivial (a button in `dashboard-actions` at `index.html:136-139` that calls `localStorage.removeItem()` on the layout keys and reloads). This is a safety valve for a feature (layout persistence) that is new and unproven. Users who encounter layout corruption in this first phase will have no other recourse, and the dashboard already has an actions section designed for this kind of control. Deferring this creates unnecessary support burden for a < 10-line addition.
>
> **DECISION: Accept Option A.**
> Rationale: This is a small addition with disproportionate recovery value for a new persistence feature. The dashboard already has an actions region, so the reset control fits the current UI structure without forcing a broader redesign.

### Q3) How should non-restorable shell tabs be surfaced after reload?

- Decision: `Deferrable`

#### Why this matters

- The proposal's open questions ask whether a dashboard status or a toast is sufficient. The spec says "surfaces that the shell tab was not restored" but does not prescribe the UX mechanism.

#### Examples

- User has 3 tmux tabs and 1 shell tab, reloads. The shell tab is gone. How do they know?
- User has only shell tabs, reloads. All tabs are gone. The session selector appears. Is the "skipped" notification useful or just noise?

#### Why this question arises in code

- The toast system (`app.js:19-57`) supports `info`, `warning`, `error`, and `success` variants with configurable duration.
- The dashboard (`index.html:108-145`) could show a "skipped sessions" section.
- The session selector (`index.html:70-105`) appears when no tabs remain.

#### Options

**Option A: Toast notification per skipped tab**

- Code impact:
  - During restore, call `showToast()` for each non-restorable tab.
- User-facing difference:
  - Brief, non-blocking notification. May stack if many shell tabs were open.
- Pros:
  - Simple, uses existing infrastructure.
  - Non-intrusive.
- Cons:
  - Multiple toasts for multiple shell tabs could be noisy.

**Option B: Single summary toast**

- Code impact:
  - Collect skipped tab names, show one toast: "N shell tab(s) were not restored: shell, shell-2".
- User-facing difference:
  - One concise notification regardless of count.
- Pros:
  - Cleaner UX for the common multi-shell-tab case.
- Cons:
  - Slightly more logic than per-tab toasts.

> **PROPOSED: Option B — Single summary toast.**
> Rationale: The existing toast system (`showToast()` at `app.js:35-57`) already supports variable-duration messages. A single summary toast avoids toast stacking when multiple shell tabs were open (a common scenario for power users). The implementation is a trivial `filter` + `map` + `join` on the skipped descriptors before calling `showToast()`. Per-tab toasts would clutter the notification area and add no informational value over a summary.
>
> **DECISION: Accept Option B.**
> Rationale: The summary-toast approach fits the existing notification system and keeps reload recovery readable when several shell tabs were open. If every tab is non-restorable and the session selector reappears, one summary toast is still enough to explain what happened without creating notification spam.
