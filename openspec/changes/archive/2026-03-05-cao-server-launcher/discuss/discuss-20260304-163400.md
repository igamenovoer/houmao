# Discussion: CAO Server Launcher Open Questions

- Feature/Change: `cao-server-launcher`
- Timestamp: `2026-03-04T16:34:00Z`
- Context: This doc captures unresolved design/contract questions discovered while reviewing the OpenSpec change artifacts (`proposal.md`, `design.md`, `specs/**`, `tasks.md`) plus existing runtime/docs and demo scripts.

Key constraints:

- Do not patch upstream CAO code under `extern/`.
- Do not run CAO server from vendored sources (`extern/orphan/...`) via `PYTHONPATH`; only use the official `uv` installation (`cao-server` on `PATH`).
- Default behavior should remain safe and non-surprising; “trust workdir” must be explicit/configured.
- Proxy behavior must be deterministic for localhost CAO calls.

---

## Summary of recommended defaults

1) Take a clean separation: manage CAO server via a standalone launcher CLI (no `brain_launch_runtime start-session` ensure-server integration). (Q1)
2) No auto-start: CAO server is started as a separate step using its own config (not implicitly by `start-session`, and not opportunistically by a base-URL “ensure”). (Q2)
3) Client: no proxy-bypass logic for localhost CAO HTTP calls; if proxy env breaks CAO connectivity, fix the environment instead. Server launch: support proxy-policy `clear|inherit` (default `clear`) and preserve/merge `NO_PROXY/no_proxy`. (Q3, Q4)
4) Model CAO “trusted home” as CAO server `HOME`: workdirs must be within `HOME`, and CAO writes state under `HOME/.aws/cli-agent-orchestrator/`. Choose a CAO home directory that contains your working repos as subdirectories (copy/move if you want to keep an original repo clean). Document this clearly. (Q5)
5) Store pid/log artifacts under `runtime_root/cao-server/<host>-<port>/` to avoid collisions and make cleanup discoverable. (Q6)
6) Use only the official `uv` installation of CAO server (`cao-server` on `PATH`); do not use vendored CAO sources. (Q8)

## Q1) What is the CLI surface for “ensure CAO server” integration?

### Why this matters

- Decision: `Blocking`
- Affects: CAO launcher CLI UX, docs, scripts (`scripts/explore/*`), and test plans.
- If ambiguous, we risk churn across multiple entry points (runtime CLI vs standalone launcher CLI).

### Options

**Option A: Extend `start-session` with opt-in flags**

- Pros:
  - Single command for users: build brain then start session (and it ensures CAO when requested).
  - Keeps policy close to the only place CAO sessions are started in this repo.
- Cons:
  - Adds more flags to `start-session`.

**Option B: Add a new `brain_launch_runtime ensure-cao-server` subcommand**

- Pros:
  - Keeps `start-session` smaller; makes server lifecycle a distinct concern.
- Cons:
  - Scripts must call two commands (ensure then start), or re-implement piping.

**Option C: Only provide a standalone launcher CLI (outside brain_launch_runtime)**

- Pros:
  - Clean separation from session runtime.
- Cons:
  - Harder for `start-session` to guarantee reliability; scripts will diverge again.

### Proposal (Recommended)

Adopt **Option C**:

- Provide a standalone CAO server launcher CLI (outside `brain_launch_runtime`) for managing CAO server lifecycle (see Q2 for whether we support auto-start or only explicit `start`).
- Keep `brain_launch_runtime` unchanged: it continues to require that CAO is already running at `--cao-base-url`.
- Scripts and docs should use a two-step flow:
  1. start CAO server separately using the CAO server launcher CLI (or `cao-server`), with explicit config
  2. `pixi run python -m agent_system_dissect.agents.brain_launch_runtime start-session --backend cao_rest ...`

### Pros / Cons (Proposal)

- Pros:
  - Clean separation: session runtime stays focused on sessions, while server lifecycle is explicit/operational.
  - Matches existing patterns in this repo (demos already start `cao-server` out-of-band).
- Cons:
  - Requires an explicit “start CAO server” step before `start-session`.

> **DECISION: Take Option C; manage CAO server separately via a standalone launcher CLI (no `start-session` ensure-server integration).**
> Rationale: `brain_launch_runtime` currently assumes CAO is already running and only performs a health check in the CAO backend start path (no server process management). Existing demos already manage `cao-server` separately (for example `scripts/demo/*/run_demo.sh`), so keeping CAO server lifecycle in a dedicated launcher preserves separation and reduces coupling.

## Q2) Do we support auto-start at all, or require explicit CAO server start with its own config?

### Why this matters

- Decision: `Blocking`
- Affects: whether users can rely on “ensure and start if missing” behavior, or must always start CAO in a separate step.
- Affects: how proxy policy / trusted-home behavior is supplied (flags vs config file).

### Options

**Option A: Auto-start via `ensure` when CAO is unhealthy**

- Pros:
  - Convenient: one command can “do the right thing” if CAO is not running.
  - Matches existing demo-style “auto-start and cleanup” patterns.
- Cons:
  - Easy to accidentally start a background process when the user only wanted a health check.
  - Requires careful gating to avoid trying to start remote CAO servers.

**Option B: Explicit start only (no auto-start)**

- Pros:
  - Clean and unsurprising: starting CAO is always an explicit step.
  - Configuration can be owned by CAO server start (a dedicated config file), not inferred from a client base URL.
- Cons:
  - Slightly more steps for users/scripts.

**Option C: No CAO server start tooling in this repo**

- Pros:
  - Avoids any risk of killing/starting the wrong process.
  - Users rely purely on upstream CAO docs and tooling.
- Cons:
  - Loses repo-owned defaults for proxy policy / trusted-home / pid/log locations.

### Proposal (Recommended)

Adopt **Option B**:

- No “auto-start” behavior (no “start if unhealthy”).
- CAO server is started explicitly as a separate step, using its own config (for example a small repo-owned config file that records trusted-home and proxy policy).

### Pros / Cons (Proposal)

- Pros:
  - No surprises: the act of starting CAO is always intentional.
  - Keeps configuration owned by the server start command/config, rather than a client URL.
- Cons:
  - Adds an explicit step to any workflow that previously relied on “ensure”.

> **DECISION: Take Option B; no auto-start. CAO server must be started separately using its own config.**
> Rationale: Auto-start (“start if missing”) creates surprising background processes and conflates client connectivity with server lifecycle. For our workflows we want CAO to be an explicitly managed service with its own configuration, while `brain_launch_runtime` remains a pure client of `--cao-base-url`.

## Q3) How should we guarantee localhost CAO HTTP calls are not proxied?

### Why this matters

- Decision: `Blocking`
- Affects: launcher health checks and all runtime CAO REST calls (terminal creation, prompt sending, stop) when CAO is addressed via loopback (`localhost` / `127.0.0.1`).
- Relying purely on user-provided `NO_PROXY` is fragile.

### Options

**Option A: Environment-only (no proxy bypass; respect caller env)**

- Pros:
  - No custom HTTP plumbing or hidden behavior.
  - Failures are transparent and attributable to the environment.
- Cons:
  - Can fail in proxy-heavy shells where `localhost` is routed through a proxy.

**Option B: Code-level no-proxy HTTP client for loopback URLs**

Implement CAO REST requests with a `urllib` opener using `ProxyHandler({})` when host is loopback.

- Pros:
  - Deterministic; does not depend on caller env.
- Cons:
  - Requires slight refactor of `CaoRestClient` request path.

**Option C: Both env enforcement and code-level bypass**

- Pros:
  - Most robust.
  - Still sets sensible env for the launched server process and any subprocesses.
- Cons:
  - More implementation work.

### Proposal (Recommended)

Adopt **Option A**:

- Do not add proxy-bypass logic for localhost CAO HTTP calls.
- If proxy settings cause CAO connectivity failures, treat it as an explicit environment issue to fix (for example by configuring `NO_PROXY`) rather than masking it in code.
- When launching `cao-server`, unset proxy env vars (`HTTP_PROXY`, `HTTPS_PROXY`, `ALL_PROXY`, plus lowercase variants), but do not unset `NO_PROXY/no_proxy`.

### Pros / Cons (Proposal)

- Pros:
  - Keeps behavior unsurprising and avoids implicit networking overrides.
- Cons:
  - Makes localhost CAO less “it just works” in proxy-heavy environments.

> **DECISION: Take Option A; no proxy bypass for localhost CAO HTTP calls; launch `cao-server` with proxy env vars unset (keep `NO_PROXY/no_proxy`).**
> Rationale: If proxy settings break localhost CAO connectivity, we want the failure to be visible and attributable to the environment; we will not hide proxy behavior in client code. Separately, when we do launch `cao-server` we want its process environment to be deterministic and not implicitly coupled to user proxy env vars (while still allowing `NO_PROXY/no_proxy` to be used explicitly).

## Q4) What are the exact proxy env var semantics (clear/inherit), and how do we merge `NO_PROXY`?

### Why this matters

- Decision: `Blocking`
- Affects: the launched `cao-server` process env (and therefore the default env inherited by CAO-managed tmux sessions/windows, unless overridden).

### Options

**Option A: Minimal set, overwrite `NO_PROXY`**

- Pros:
  - Simple.
- Cons:
  - Clobbers user `NO_PROXY` values; can break other tooling.

**Option B: Clear/inherit proxy vars, append to existing `NO_PROXY`**

- Pros:
  - Respects user `NO_PROXY` while ensuring loopback is covered.
  - Predictable and non-destructive.
- Cons:
  - Requires parsing/deduping comma-separated lists.

**Option C: Manage only uppercase env vars**

- Pros:
  - Smaller surface.
- Cons:
  - Many tools look at lowercase `http_proxy`/`no_proxy`.

### Proposal (Recommended)

Adopt **Option B** and manage both upper/lowercase:

- Controlled proxy vars:
  - `HTTP_PROXY`, `HTTPS_PROXY`, `ALL_PROXY`
  - `http_proxy`, `https_proxy`, `all_proxy`
- `--cao-server-proxy-policy=clear`:
  - remove all controlled proxy vars from the target environment
- `--cao-server-proxy-policy=inherit`:
  - preserve controlled proxy vars as-is
- In both cases:
  - ensure `NO_PROXY` and `no_proxy` contain at least: `localhost,127.0.0.1,::1`
  - merge by appending missing entries, deduping, and preserving existing order

### Pros / Cons (Proposal)

- Pros:
  - Predictable and non-destructive.
  - Works across tooling ecosystems.
- Cons:
  - Slightly more code.

> **DECISION: Take Option B; implement proxy-policy `clear|inherit` for launched `cao-server` and merge `NO_PROXY/no_proxy` non-destructively.**
> Rationale: We want deterministic server launch behavior by default (`clear`), but still allow explicit proxy inheritance when needed. Merging `NO_PROXY/no_proxy` by appending missing loopback entries avoids clobbering user configuration while ensuring the bypass list is complete.

## Q5) What is the trusted-home default and how do we expose it safely?

### Background: CAO home/workdir contract (upstream behavior)

CAO does not have an independent “trust list”. Instead it enforces a simple
contract based on the CAO server process `HOME`:

- **CAO state location:** CAO writes its own state under the CAO server process
  home directory (for example `HOME/.aws/cli-agent-orchestrator/...`).
- **Allowed working directories:** When CAO creates tmux sessions/windows, it
  validates that each requested `working_directory` resolves to a path inside
  the CAO server process home directory tree (and rejects blocked system paths).

Implications:

- To use CAO with repos under `/data/...`, the CAO server process `HOME` must be
  set to a directory whose tree includes those repos.
- If you set CAO `HOME` to a repo root, CAO will write its state under that repo
  (for example `<repo>/.aws/cli-agent-orchestrator/...`).
- If you want to keep a repo clean, set CAO `HOME` to a separate “CAO home”
  directory and place/copy working repos as subdirectories under it; CAO state
  stays at the CAO home root, not inside each repo.
- Read-only repos can still be used as `working_directory` as long as CAO `HOME`
  itself is writable (CAO must be able to write its DB/logs under `HOME/.aws/...`).

### Why this matters

- Decision: `Blocking`
- CAO uses `HOME` to validate allowed working directories and to place state under `$HOME/.aws/cli-agent-orchestrator/`.

### Options

**Option A: Default `trusted_home` unset (no override), require explicit opt-in**

- Pros:
  - Safe and unsurprising; preserves existing CAO behavior.
  - Avoids relocating CAO DB/logs unexpectedly.
- Cons:
  - `/data/...` workflows require the user to set one extra flag.

**Option B: Automatically derive trusted-home from `--workdir` when `--workdir` is outside real `~`**

- Pros:
  - “Just works” for `/data/...` repos.
- Cons:
  - Surprising side effects (CAO state relocation).
  - Can accidentally broaden trust scope.

**Option C: Always set `trusted_home` to the session `--workdir`**

- Pros:
  - Simplest mental model: “CAO home == trust root”; CAO state and repos live under one directory tree.
  - Works naturally for `/data/...` layouts when users organize a dedicated CAO home directory.
- Cons:
  - CAO writes state under the configured CAO home (by design); users must organize directories if they want a “clean repo” elsewhere.

### Proposal (Recommended)

Adopt **Option C**:

- Treat “trusted home” as the CAO server process `HOME` (the trust root).
- Allow CAO to write its state under `HOME/.aws/cli-agent-orchestrator/`.
- Document the limitation and recommended directory layouts in `docs/`:
  - If you want a clean upstream repo, create a dedicated CAO home directory and copy/mirror the repo into a subdirectory under it for CAO-backed workflows.
  - Ensure CAO `HOME` is writable even if the repo subdirectory is read-only.

### Pros / Cons (Proposal)

- Pros:
  - Simple and consistent: one tree is both the trust root and CAO state root.
  - Users can reason about failures by looking at directory layout, not hidden allowlists.
- Cons:
  - Requires users to consciously structure their directories if they want to keep a separate “clean repo” pristine.

> **DECISION: Take Option C; define CAO “trusted home” as the CAO server process `HOME` and allow CAO to write its state there.**
> Rationale: This matches upstream CAO behavior (workdir must be under `HOME`, and CAO state is under `HOME/.aws/...`) and keeps the mental model simple. Users who care about repo cleanliness can control it by directory structure (dedicated CAO home + repos as subdirs) rather than expecting invisible trust mechanisms.

## Q6) How should we structure pid/log artifact paths under the runtime root?

### Why this matters

- Decision: `Blocking`
- Affects: avoiding collisions between multiple CAO base URLs and parallel runs.

### Options

**Option A: Single global paths**

- Example: `tmp/agents-runtime/cao-server/cao-server.pid` and `.../cao-server.log`
- Pros:
  - Simple.
- Cons:
  - Collides if multiple CAO servers are used.
  - Harder to reason about which base URL the pid belongs to.

**Option B: Partition by `<host>-<port>`**

- Example: `tmp/agents-runtime/cao-server/localhost-9889/cao-server.pid`
- Pros:
  - Supports multiple base URLs without collisions.
  - Human-readable.
- Cons:
  - Requires URL parsing and sanitization.

**Option C: Hash the base URL**

- Pros:
  - Collision-resistant.
- Cons:
  - Less discoverable for humans.

### Proposal (Recommended)

Adopt **Option B**:

- `runtime_root/cao-server/<host>-<port>/`
  - `cao-server.pid`
  - `cao-server.log`
  - (optional) `launcher_result.json` for diagnostics

### Pros / Cons (Proposal)

- Pros:
  - Easy to debug and supports multi-server setups.
- Cons:
  - Slightly more code than a single global path.

> **DECISION: Take Option B; partition runtime artifacts under `runtime_root/cao-server/<host>-<port>/`.**
> Rationale: A single global pid/log path is collision-prone when running multiple CAO servers (different base URLs for parallel tests or demos). Using `<host>-<port>` keeps artifacts human-discoverable and makes cleanup/debugging unambiguous.

## Q7) What are the stop semantics and how do we avoid killing the wrong process?

### Why this matters

- Decision: `Deferrable`
- A pidfile alone is not enough to guarantee the pid is still a CAO server (pid reuse is possible).

### Options

**Option A: Stop only if this process started the server**

- Pros:
  - Safest.
- Cons:
  - Hard to use across processes/shells; doesn’t fit a standalone `stop` command.

**Option B: Stop by pidfile, but verify process identity**

- Pros:
  - Practical for scripts and manual ops.
  - Safer than blind kill.
- Cons:
  - Verification is best-effort and OS-dependent.

**Option C: No stop command**

- Pros:
  - Avoids risk entirely.
- Cons:
  - Leaves more orphan servers; worse UX.

### Proposal (Recommended)

Adopt **Option B**:

- `stop` reads pidfile and performs best-effort verification before killing:
  - verify process exists,
  - verify command line contains `cao-server` (or the known console-script path),
  - if verification fails, refuse to kill and print actionable diagnostics.
- Use defaults aligned with existing demo scripts:
  - SIGTERM, wait up to `10s`, then SIGKILL.

### Pros / Cons (Proposal)

- Pros:
  - Good UX without being reckless.
- Cons:
  - Some systems may not expose commandline info; verification may be weaker there.

> **DECISION: Take Option B; provide `stop` by pidfile with best-effort process identity verification before killing.**
> Rationale: A standalone launcher needs a practical stop path across shells/processes, but we must avoid blind kills (pid reuse). Verifying identity (best-effort) before sending signals balances safety with operational usability.

## Q8) Should we prefer vendored CAO (`extern/orphan/...`) or the system-installed CAO?

### Why this matters

- Decision: `Blocking`
- The repo’s CAO REST client contract and fixtures are written against the vendored CAO API implementation.

### Options

**Option A: Always run `cao-server` as found on `PATH` (no PYTHONPATH injection)**

- Pros:
  - Minimal magic.
- Cons:
  - Version drift is likely; can break contracts silently.

**Option B: Prefer vendored CAO when present by setting `PYTHONPATH`**

- Pros:
  - Matches the API contract we vendor and reference.
  - Aligns with existing `scripts/demo/*` behavior.
- Cons:
  - Requires careful `PYTHONPATH` composition.

**Option C: Add a flag to select mode**

- Pros:
  - Explicit.
- Cons:
  - More config surface.

### Proposal (Recommended)

Adopt **Option A**:

- Always launch `cao-server` from `PATH` (the official `uv` installation).
- Do not inject `PYTHONPATH` to point at vendored CAO sources.

### Pros / Cons (Proposal)

- Pros:
  - Avoids coupling this repo to a vendored CAO snapshot.
  - Matches the “official installation” operational story (`uv tool install ...`).
- Cons:
  - Increases risk of drift between our expectations and upstream CAO server versions; requires tighter compatibility discipline in our client/launcher code and docs.

> **DECISION: Take Option A; only use the official `uv` installation of CAO (`cao-server` on `PATH`), and do not use vendored CAO sources.**
> Rationale: We will not modify or depend on the `extern/orphan` CAO code at runtime. Keeping CAO server as an externally installed tool simplifies ownership boundaries: this repo provides a launcher/client and documented operational defaults, while CAO itself is managed via `uv`.
