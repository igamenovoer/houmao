## Summary

- **Solid problem statement and motivation.** The split between shared runtime roots (`~/.houmao/runtime`), working-directory jobs (`<cwd>/.houmao/jobs`), and project-local overlay state is a real pain point. Unifying under the active overlay is the right direction.
- **Good precedence design.** The `CLI > env > discovered overlay > bootstrap` chain is consistent across all 9 delta specs and matches the existing pattern in `owned_paths.py`.
- **Registry isolation is correct.** Keeping registry global at `~/.houmao/registry` preserves cross-project discovery semantics without overcomplicating the change.
- **Risk: the central resolver's relationship to `owned_paths.py` is architecturally ambiguous.** The design says "introduce one shared project-aware local-root resolver" but doesn't specify whether it wraps, replaces, or extends the existing per-root resolvers in `owned_paths.py`. This is the single most impactful design decision for implementation complexity.
- **Risk: `start_runtime_session()` API change for jobs_root is implicit.** The current function derives job dirs from `working_directory`, not a configurable `jobs_root` parameter. The spec requires overlay-derived job dirs but doesn't call out the API signature change.
- **Risk: Git worktree boundary detection adds a subprocess dependency to every project-aware command's hot path.** Fallback behavior when `git` is unavailable is unspecified.

## Findings

### Must Fix / Blocking

1. **The `brain-launch-runtime` spec requires overlay-derived job dirs but the current `start_runtime_session()` API has no `jobs_root` parameter.** The function at `src/houmao/agents/realm_controller/runtime.py:923` derives job dirs from `working_directory` via `resolve_session_job_dir(session_id=..., working_directory=...)`. The spec scenario "Project-context session derives its job dir under the overlay" requires `<active-overlay>/jobs/<session-id>/`, but there is no mechanism to inject a custom jobs base into the runtime session API. Either `start_runtime_session()` needs a `jobs_root` parameter, or `resolve_session_job_dir()` in `owned_paths.py` needs an override path, or the caller needs to pre-create and pass the resolved job dir. Task 2.2 mentions updating `runtime.py` but the spec should explicitly define the API change contract rather than leaving it implicit.

> **DECISION: Refine.**
> Rationale: The reviewer is right that the current `start_runtime_session()` boundary cannot produce overlay-local job dirs with its present `working_directory`-derived logic. We should revise the artifacts so session startup receives an explicit jobs-root or equivalent resolved job-dir input, but we should not freeze one exact Python signature in the behavioral spec.

2. **The design does not specify the integration contract between the new central project-aware resolver (task 1.2) and the existing per-root resolvers in `owned_paths.py`.** The design decision "Introduce one central project-aware local-root resolver" rejects teaching existing shared-root helpers to inspect `HOUMAO_PROJECT_OVERLAY_DIR` directly. However, it does not clarify the resulting architecture:
   - **Option A:** The new resolver is a higher-level function that returns all overlay-local roots as a bundle, and callers use that bundle instead of calling `resolve_runtime_root()` etc. directly. This means two parallel resolution paths coexist.
   - **Option B:** The new resolver wraps existing `owned_paths.py` functions by computing overlay-local defaults and passing them as `explicit_root` to the existing resolvers. This preserves the existing env-var precedence.
   - **Option C:** `owned_paths.py` functions gain an optional `project_overlay_root` parameter that slots into the precedence chain between env-var and default.

   The chosen approach affects how many call sites change, whether env-var overrides still work uniformly, and whether the existing test suite for `owned_paths.py` needs restructuring. Without this decision, implementers of tasks 1.2 through 3.3 will make inconsistent choices.

> **DECISION: Accept.**
> Rationale: This is the main architectural gap in the current artifacts. The change should define one central project-aware roots bundle above `owned_paths.py`, while `owned_paths.py` remains the low-level explicit/env/default authority; overlay-derived runtime/mailbox/jobs paths must enter as the default tier rather than as `explicit_root` so env overrides still win.

### Should Fix / Important

3. **Git worktree boundary detection has no specified fallback when `git` is unavailable.** The `project-aware-local-roots` spec requires "nearest ancestor discovered project overlay within the current Git worktree boundary" (scenario: "Discovery does not cross the current Git worktree boundary"). The current `_discover_nearest_project_overlay()` at `src/houmao/project/overlay.py:374` walks all ancestors unconditionally. Adding Git boundary detection means calling `git rev-parse --show-toplevel` (or reading `.git` markers), which introduces a subprocess dependency. The spec should define behavior when:
   - `git` is not installed on the system.
   - The working directory is not inside a Git repository.
   - The `git` command fails for another reason (e.g., corrupted `.git`).
   A reasonable fallback would be "walk all ancestors" (current behavior), but this should be explicit.

> **DECISION: Refine.**
> Rationale: The boundary behavior does need to be explicit, but we do not want a `git` subprocess dependency in this hot path. The artifact revision should standardize on `.git` marker detection during the ancestor walk, which matches the spec scenarios and removes the whole "git unavailable" fallback problem.

4. **`HoumaoProjectOverlay` already exposes `mailbox_root` and `easy_root` as properties (overlay.py lines ~55-65), but the proposal frames these as newly needed.** The overlay dataclass at `src/houmao/project/overlay.py:31` already derives `mailbox_root → <overlay>/mailbox` and `easy_root → <overlay>/easy`. Only `runtime_root` and `jobs_root` are genuinely new. The proposal and task 1.1 should acknowledge the existing properties and focus the overlay extension on `runtime` and `jobs` only, plus the new ensure-or-bootstrap path. This avoids re-implementing what already exists.

> **DECISION: Refine.**
> Rationale: `HoumaoProjectOverlay` already exposes `mailbox_root` and `easy_root`, so task 1.1 should not imply those properties are newly introduced. The proposal's ownership statement is still correct; the later artifact revision should narrow the implementation wording to add `runtime_root`, `jobs_root`, and the ensure-or-bootstrap path while reusing the existing mailbox/easy properties.

5. **Auto-bootstrap at `<cwd>/.houmao` could create unexpected overlay directories.** When an operator runs a stateful command from an arbitrary subdirectory (e.g., `~/Downloads`), the spec says the command will bootstrap `~/Downloads/.houmao`. This is correct per the precedence chain, but potentially surprising. The design acknowledges this risk ("Limit auto-bootstrap to commands that genuinely require local state materialization") but no spec scenario defines user-facing feedback (e.g., a warning message or confirmation). Consider whether auto-bootstrap should emit a visible notice when creating a new overlay outside a Git repository, to help operators catch accidental bootstraps early.

> **DECISION: Refine.**
> Rationale: Auto-bootstrap is an intentional part of the contract and should remain automatic for stateful flows, so we are not adding a confirmation gate. We should, however, carry this into later artifact revision as an operator-facing notice requirement when a command implicitly creates a new overlay, especially from an uninitialized location.

6. **Cleanup scope for operators with mixed old/new session locations is unclear.** The `houmao-mgr-cleanup-cli` spec says project-context cleanup defaults to `<active-overlay>/runtime`, but operators upgrading from the shared-root model will have sessions under both `~/.houmao/runtime` and `<overlay>/runtime`. The spec doesn't describe whether cleanup commands can target both locations in one invocation or whether the operator must run cleanup twice (once with `--runtime-root ~/.houmao/runtime`, once with project defaults). Since this is a breaking change with no auto-migration, this gap could cause abandoned artifacts under the old shared root.

> **DECISION: Refine.**
> Rationale: The mixed old/new root concern is real, but the first cut should stay single-root-per-invocation rather than inventing multi-root cleanup semantics. Later artifact revision should state explicitly that operators clean old shared-root artifacts by passing `--runtime-root ~/.houmao/runtime`, while project-context cleanup defaults to the overlay-local runtime root.

### Nice To Have

7. **Task 4.1 ("Audit maintained `houmao-mgr` local command families") is underspecified.** It doesn't list which command families are in scope or what the audit output looks like. Breaking this into enumerated sub-tasks (one per command group) would reduce the risk of incomplete coverage. For reference, the maintained local command groups include at least: `brains`, `agents`, `project`, `mailbox`, `admin`, `server`, and runtime artifact surfaces.

> **DECISION: Accept.**

8. **No migration tooling for consolidating shared-root sessions into overlay-local roots.** The design correctly says "leave existing runtime artifacts in place" and notes that registry manifest paths are absolute. However, operators who want a clean project-local state will need to manually move or re-create sessions. A future convenience command (e.g., `houmao-mgr admin migrate-runtime --from ~/.houmao/runtime --to <overlay>/runtime`) could smooth adoption. This is not blocking but worth tracking.

> **DECISION: Reject.**
> Rationale: Migration tooling is outside the stated scope of this change; the design explicitly says we are not auto-migrating or rewriting older runtime artifacts in place. Existing sessions remain addressable through absolute manifest and registry pointers, and explicit cleanup/override flows are sufficient for this first cut.

If this is intentional, the design should state it explicitly. If per-project path customization is desired later, the config schema should be extended with optional `[paths]` entries.

> **DECISION: Accept.**
> Rationale: The intended v1 contract is convention-derived overlay subpaths with env-var overrides for exceptions, not per-project path-schema expansion. Later artifact revision should say that explicitly so the fixed-subpath choice reads as deliberate rather than accidental.

## Design Alignment With Existing System

- **Existing pattern: `owned_paths.py` per-root resolvers** (`src/houmao/owned_paths.py`): Each root has its own `resolve_*_root(explicit_root, env, base)` function following `explicit > env > default` precedence. The proposed change introduces a new layer above this, which is architecturally sound only if the integration contract is defined (see finding #2).

- **Existing pattern: `HoumaoProjectOverlay` as read-only discovery** (`src/houmao/project/overlay.py:31-75`): The overlay dataclass is currently a frozen discovery result. The change extends it to be the default authority for runtime state placement. This is a role expansion — from "where are agent definitions?" to "where does all project-local Houmao state live?" The frozen dataclass pattern supports this cleanly, but the ensure-or-bootstrap path will need a mutable entry point (likely a factory function) that returns the immutable overlay after creation.

- **Existing pattern: `BuildRequest.runtime_root` as optional** (`src/houmao/agents/brain_builder.py`): Brain build already accepts an optional `runtime_root`. The change threads project-aware defaults into this existing parameter, which is clean and non-disruptive.

- **Existing pattern: `start_runtime_session()` uses `working_directory` for jobs** (`src/houmao/agents/realm_controller/runtime.py:923`): This is the main friction point. The function uses `resolve_session_job_dir(session_id, working_directory)` where `working_directory` doubles as the jobs base anchor. Decoupling jobs root from working directory is necessary and represents a genuine API change.

- **Existing pattern: Mailbox commands use `resolve_mailbox_root(explicit_root=mailbox_root)`** (`src/houmao/srv_ctrl/commands/mailbox.py`): The change proposes injecting project-aware defaults via the same explicit_root parameter, which aligns cleanly with the existing precedence chain.

- **Existing pattern: Server config uses `Field(default_factory=resolve_runtime_root)`** (`src/houmao/server/config.py:25`): The change needs this default_factory to become project-aware. Since `HoumaoServerConfig` is a Pydantic model instantiated at server startup, the factory can be wrapped to check for an active overlay first. This is straightforward.

- **Overall alignment: Good.** The proposal works with the existing architecture rather than against it. The main tension is in the jobs-root derivation (currently coupled to working_directory) and the unspecified resolver integration contract.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should the central project-aware resolver integrate with the existing per-root resolvers in `owned_paths.py`?

- Decision: `Blocking`

#### Why this matters

- The resolver integration pattern determines how many call sites change, whether env-var overrides continue to work uniformly, and whether existing tests for `owned_paths.py` need restructuring. It is the single highest-impact architectural decision in this change.

#### Examples

- When `houmao-mgr agents launch` resolves runtime root: should it call `project_aware_roots.runtime_root` (new resolver) and then pass that as `explicit_root` to `resolve_runtime_root()` (existing resolver)? Or should it call a unified function?
- When `HOUMAO_GLOBAL_RUNTIME_DIR` is set and an active overlay exists: does the env var win over the overlay default, or does the overlay default win? The design says "existing env-var overrides as stronger precedence than project-aware defaults", which implies env > overlay.

#### Why this question arises in code

- `src/houmao/owned_paths.py` defines `_resolve_root()` with precedence `explicit > env > default` (lines 136-158).
- The new resolver needs to inject overlay-derived defaults somewhere in this chain.
- If overlay defaults replace the hardcoded `default_root`, the existing function signature works unchanged.
- If overlay defaults are a separate precedence tier, the function signature changes.

```python
# Current:
def _resolve_root(*, explicit_root, env_key, default_root, base) -> Path

# Option B would use it as:
overlay_root = active_overlay.runtime_root  # <overlay>/runtime
resolved = _resolve_root(explicit_root=overlay_root, env_key=..., default_root=..., base=...)
# But this makes overlay win over env, violating the stated precedence.

# Option C would be:
def _resolve_root(*, explicit_root, env_key, project_root, default_root, base) -> Path
# explicit > env > project_root > default_root
```

#### Options

**Option A: Separate bundle resolver, callers pass results as explicit_root**

The new resolver returns a `ProjectAwareRoots` dataclass with all overlay-derived paths. Each command extracts the relevant root and passes it as `explicit_root` to existing `resolve_*_root()` functions. The existing functions are unchanged.

- Code impact:
  - New module (e.g., `src/houmao/project/roots.py`) with `resolve_project_aware_roots(overlay) -> ProjectAwareRoots`.
  - Every project-aware command call site adds 2-3 lines to extract and pass roots.
  - `owned_paths.py` unchanged.
- User-facing difference:
  - Env-var overrides would be **ignored** for project-aware commands because `explicit_root` wins over env in `_resolve_root()`. This violates the stated design goal "Preserve explicit CLI overrides and existing env-var overrides as stronger precedence."
- Pros:
  - Minimal changes to `owned_paths.py`.
  - Clear separation of concerns.
- Cons:
  - Breaks env-var override semantics unless callers add their own env-check before passing explicit_root.
  - Duplicates precedence logic across callers.

**Option B: Wrapper resolver that computes overlay defaults and feeds them as the `default_root` fallback**

The new resolver checks for an active overlay, computes overlay-derived roots, then calls existing `resolve_*_root()` with the overlay-derived path as `default_root` instead of the hardcoded shared default. Env-var overrides continue to win because they sit above `default_root` in the existing precedence.

- Code impact:
  - New wrapper functions (e.g., `resolve_project_aware_runtime_root(*, overlay, explicit_root)`) that call `resolve_runtime_root(explicit_root=explicit_root, default_root=overlay.runtime_root)`.
  - Requires `_resolve_root()` to accept an optional `default_root` override (minor signature change).
  - Call sites switch from `resolve_runtime_root()` to `resolve_project_aware_runtime_root()`.
- User-facing difference:
  - Precedence becomes `explicit CLI > env var > overlay default > shared default`, which matches the stated design goal.
- Pros:
  - Preserves env-var semantics correctly.
  - Minimal changes to existing `_resolve_root()` internals.
  - Test suite for `owned_paths.py` needs only additive tests.
- Cons:
  - Introduces a second set of resolver functions that must stay in sync with the originals.

**Option C: Extend `_resolve_root()` with a `project_root` tier**

Add an optional `project_root` parameter to `_resolve_root()` that slots between env-var and default in the precedence chain: `explicit > env > project_root > default`.

- Code impact:
  - `_resolve_root()` gains one optional parameter.
  - All public `resolve_*_root()` functions gain a `project_root` parameter.
  - Callers that are project-aware pass `project_root=overlay.runtime_root`.
  - Callers that are not project-aware pass nothing (backward compatible).
- User-facing difference:
  - Same as Option B: `explicit > env > overlay > shared default`.
- Pros:
  - Single resolution path; no parallel resolver family.
  - Existing tests extend naturally with new parameter.
  - No wrapper indirection.
- Cons:
  - Modifies the core `_resolve_root()` function that all callers depend on.
  - Every `resolve_*_root()` signature changes (though the new parameter is optional).

> **PROPOSED: Option B — wrapper resolver that feeds overlay-derived paths as `default_root` overrides.**
> Rationale: Option B preserves the stated precedence (`explicit > env > overlay > shared default`) without modifying `_resolve_root()`'s core semantics. The existing `resolve_runtime_root(explicit_root=..., default_root=...)` signature at `src/houmao/owned_paths.py` already accepts a `base` parameter for relative path resolution; adding `default_root` override is a minimal extension. Option A breaks env-var semantics (explicit_root wins over env). Option C is cleaner long-term but touches the core resolution function that 20+ call sites depend on, adding risk for a change that is intentionally scoped to "maintained local project-aware flows." Wrappers keep the blast radius small while preserving correct precedence.
>
> **DECISION: Accept with refinement — keep one central project-aware roots bundle above `owned_paths.py`.**
> Rationale: Accept with refinement. We should keep one central project-aware roots bundle, and preserve `owned_paths.py` as the low-level explicit/env/default layer by feeding overlay-derived paths into the default tier rather than the explicit tier. Task 1.2 already calls for one shared project-aware resolver, and `resolve_project_aware_agent_def_dir()` in `project/overlay.py` shows the repo's existing pattern: project-aware logic lives in one shared helper, not in ad hoc caller code. The later artifact revision should define an optional default-root hook or equivalent internal seam in `owned_paths.py` while avoiding project-aware branches in every generic helper.

### Q2) What should happen when Git is unavailable for worktree boundary detection?

- Decision: `Deferrable`

#### Why this matters

- The `project-aware-local-roots` spec requires bounding ancestor discovery at the Git worktree root. If `git` is not installed or the directory is outside a Git repo, the boundary is undefined. Without a fallback, the implementation could either crash, walk all ancestors (current behavior), or refuse to discover.

#### Examples

- An operator runs `houmao-mgr agents launch` in a Docker container without `git` installed. The current directory has a `.houmao/houmao-config.toml` in a parent directory. Should discovery find it or refuse?
- An operator runs from a directory that is not inside any Git repository (e.g., `/tmp/scratch`). The ancestor walk has no Git boundary to respect.

#### Why this question arises in code

- `src/houmao/project/overlay.py:374` — `_discover_nearest_project_overlay()` currently walks all ancestors unconditionally.
- The new spec adds a Git boundary constraint, but the current function has no Git dependency.
- Adding `subprocess.run(["git", "rev-parse", ...])` introduces a fallible external dependency.

#### Options

**Option A: Walk all ancestors (degrade gracefully)**

When `git` is unavailable or the directory is outside a Git repo, fall back to unbounded ancestor walk (current behavior). Log a debug-level notice.

- Code impact:
  - Wrap `git rev-parse --show-toplevel` in a try/except; on failure, set boundary to filesystem root.
- User-facing difference:
  - No visible change when Git is available. Without Git, overlay discovery behaves as it does today.
- Pros:
  - Zero-breakage fallback.
  - Works in minimal environments (Docker, CI without Git).
- Cons:
  - Can cross repo boundaries silently when Git is missing — the exact case the spec wants to prevent.

**Option B: Fall back to `.git` marker detection (no subprocess)**

Instead of calling `git rev-parse`, walk ancestors looking for a `.git` file or directory. Use the nearest `.git` marker as the boundary. This avoids the subprocess dependency entirely.

- Code impact:
  - Pure Python ancestor walk checking `(ancestor / ".git").exists()`.
  - No subprocess needed.
- User-facing difference:
  - Slightly less accurate than `git rev-parse --show-toplevel` for unusual Git configurations (e.g., `GIT_DIR` overrides, bare repos), but correct for standard clones and worktrees.
- Pros:
  - No external dependency.
  - Fast (filesystem stat only).
  - Handles the common case (nested `.git` directories) correctly.
- Cons:
  - Won't detect `GIT_DIR` env overrides or linked worktrees created with `git worktree add`.

> **PROPOSED: Option B — use `.git` marker detection instead of subprocess.**
> Rationale: The overlay discovery path at `src/houmao/project/overlay.py:374` is a synchronous hot path called by every project-aware command. Adding a subprocess call (`git rev-parse`) would add latency and a dependency on `git` being installed. The `.git` marker approach (checking `(parent / ".git").exists()` during the ancestor walk) handles the dominant case — nested Git repositories — without any external dependency. The spec scenarios all reference `.git` existence as the boundary indicator (e.g., "AND WHEN `/parent/nested-repo/.git` exists"). For `git worktree add` linked worktrees, `.git` is a file (not a directory) pointing to the main repo's worktree metadata, so `exists()` still returns True and the boundary is still detected correctly. Only truly exotic Git configurations (bare repos with `GIT_DIR` overrides) would be missed, and those are outside normal operator workflows.
>
> **DECISION: Accept PROPOSED — use `.git` marker detection instead of subprocess.**
> Rationale: Accept. The current `_discover_nearest_project_overlay()` path is a synchronous ancestor walk, and the spec scenarios already express the boundary in terms of `.git` existence. Checking for a `.git` file or directory keeps discovery fast, works for standard clones and linked worktrees, and avoids turning every project-aware command into a `git` subprocess client.

### Q3) Should `project status` trigger overlay bootstrap, or remain purely non-creating?

- Decision: `Deferrable`

#### Why this matters

- The design decision says "Commands that need a local Houmao-owned root to mutate or materialize state will ensure the overlay exists. Pure informational surfaces may report the selected overlay root without forcing creation." The proposal's open questions section explicitly asks: "Should `project status` remain non-creating, or should it also bootstrap the overlay to make the 'selected root' always concrete?"

#### Examples

- An operator runs `houmao-mgr project status` in a directory with no `.houmao/`. Currently this reports "no overlay found." Under the new model, should it create `.houmao/` to show concrete roots, or report hypothetical roots?

#### Why this question arises in code

- `src/houmao/srv_ctrl/commands/project.py:111-135` — `project_status_command()` currently calls `resolve_project_overlay()` and reports what it finds. It does not call `bootstrap_project_overlay()`.
- The spec says "Commands that only need to report selection state MAY report the selected overlay root without forcing bootstrap" — the MAY makes this a design choice, not a requirement.

#### Options

**Option A: Keep non-creating (current behavior)**

`project status` reports the discovered/hypothetical overlay root without bootstrapping. If no overlay exists, it reports "would bootstrap at `<cwd>/.houmao`" without actually creating it.

- Code impact:
  - No change to `project_status_command()`.
- User-facing difference:
  - Operators can inspect what would happen without side effects.
- Pros:
  - No surprise filesystem mutations from read-only commands.
  - Consistent with the "MAY report without forcing bootstrap" allowance.
- Cons:
  - Reported paths may not exist yet, which could confuse operators checking before a launch.

**Option B: Bootstrap on status**

`project status` ensures the overlay exists before reporting, so all reported roots are concrete directories.

- Code impact:
  - Add `ensure_overlay()` call before reporting.
- User-facing difference:
  - Running `project status` in a new directory creates `.houmao/`.
- Pros:
  - Reported roots always exist — no ambiguity.
- Cons:
  - Side effect from a "status" command is surprising.

> **PROPOSED: Option A — keep `project status` non-creating.**
> Rationale: The existing `project_status_command()` at `src/houmao/srv_ctrl/commands/project.py:111` follows a read-only contract. The design decision explicitly allows "pure informational surfaces MAY report the selected overlay root without forcing bootstrap." Adding creation side effects to a status command would violate least-surprise. Operators who want the overlay created can run any stateful command (e.g., `agents launch`, `mailbox init`) or the explicit `project init`.
>
> **DECISION: Accept PROPOSED — keep `project status` non-creating.**
> Rationale: Accept. `project_status_command()` currently only resolves and reports selection state, and the design explicitly separates read-only inspection from stateful bootstrap. Preserving that contract avoids surprise filesystem mutation from a status command; stateful commands and `project init` remain the creation surfaces.

### Q4) Which generic command families are in scope for the first cut: only maintained local CLI families, or also deprecated compatibility entrypoints?

- Decision: `Deferrable`

#### Why this matters

- The proposal's open questions section asks this directly. The answer affects the scope of tasks 4.1-4.3 and how much code needs to change in the first implementation pass.

#### Examples

- `houmao-cli` (`src/houmao/cli.py`) is a deprecated compatibility entrypoint. Should it gain project-aware root defaults, or only `houmao-mgr`?
- `houmao-cao-server` (`src/houmao/cao_cli.py`) is a deprecated launcher. Should it resolve runtime root project-aware?

#### Why this question arises in code

- `src/houmao/cli.py` and `src/houmao/cao_cli.py` are marked as deprecated compatibility entrypoints in the repo instructions.
- `src/houmao/srv_ctrl/cli.py` (`houmao-mgr`) is the maintained surface.
- Touching deprecated entrypoints adds implementation cost with no benefit if they are being phased out.

#### Options

**Option A: Maintained surfaces only (`houmao-mgr`, `houmao-server`)**

Only wire project-aware defaults into the maintained CLI surfaces. Deprecated entrypoints keep their current behavior.

- Code impact:
