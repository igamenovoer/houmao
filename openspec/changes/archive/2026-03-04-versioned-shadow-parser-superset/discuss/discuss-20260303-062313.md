- Timestamp: `2026-03-03T06:23:13Z`
- Context: We are about to implement a version-aware, runtime-owned shadow parser stack. This doc enumerates design choices affecting contracts, error semantics, and rollout defaults; scope is now Codex/Claude for this change, with Gemini deferred. Inputs reviewed: the change proposal/design/specs, current runtime shadow parsers, upstream CAO provider patterns/fixtures, and upstream `gemini-cli` source/docs for headless output contracts.

This change is now scoped to Codex/Claude CAO shadow parsing only.

## Summary of recommended defaults

1) Treat unsupported/unknown output as an explicit parse error (code: `unsupported_output_format`) and fail fast (Q1).
2) For Codex TUI variant detection, anchor on footer/status tokens like `context left` / `? for shortcuts` plus prompt glyphs, not on semver banners (Q2).
3) Defer Gemini parser architecture to a dedicated follow-up (keep current headless behavior unchanged for now) (Q3).
4) Flip Codex CAO default to `shadow_only` only after parity tests/fixtures pass; keep an explicit operator override to force `cao_only` during rollout (Q4).
5) Copy a curated subset of upstream CAO fixtures into repo-owned test fixtures (don’t depend on `extern/` at test time) (Q5).

## Q1) Should "unknown/unsupported format" be a status or an error?

### Why this matters

- If this is a pollable status, the runtime could accidentally loop until timeout if it is not treated specially everywhere.
- If this is an error, we get clear fail-fast behavior, but lose the ability to represent it in the same status channel as `idle/processing/completed`.
- This choice affects the CAO shadow-only polling loops and the error surface of the CLI (`brain_launch_runtime` prints exceptions as plain `error: ...`).

### Options

**Option A: Distinct status `unknown_format`**

- Pros:
  - Uniform "status" surface (everything is a status).
  - Potentially easier for telemetry: "we saw unknown_format N times".
- Cons:
  - Easy to mishandle and accidentally treat as `processing` (hangs/timeouts).
  - Would require auditing all wait loops and canonical status mapping to ensure it always triggers failure.

**Option B: Always raise an explicit parse error (`unsupported_output_format`)**

- Pros:
  - Aligns with existing Codex shadow behavior today (Codex shadow already raises `unsupported_output_format`).
  - Simpler invariants: unknown means fail, never poll.
  - Naturally satisfies "explicit detection" from the feature request.
- Cons:
  - If we want structured telemetry later, the error is currently only a string at the CLI layer.

### Proposal (Recommended)

Pick Option B: represent unknown/unsupported output as a structured parse error (error code `unsupported_output_format`) and fail fast. If we want status-like telemetry, attach an anomaly code and include it in the error message and/or in `parser_metadata` for non-failing diagnostics paths.

### Pros / Cons (Proposal)

- Pros:
  - Removes a whole class of timeout-as-bug behavior.
  - Keeps behavior consistent with current Codex shadow parser semantics.
- Cons:
  - Telemetry requires extra plumbing later if we want more than stderr strings.

Notes / Follow-ups:

- Deferrable: add a structured error JSON envelope in the CLI (not required for initial implementation).

> **DECISION: Unsupported format is an explicit parse error (`unsupported_output_format`), fail fast.**
> Rationale: `CodexShadowParser._require_supported_format` in `codex_shadow.py` already raises `CodexShadowParseError` with `error_code="unsupported_output_format"` (line 246-252), and `cao_rest.py` catches it in `_classify_shadow_status` (line 649-657), converting it to a `BackendExecutionError` that surfaces as an `error: …` string on stderr via `cli.py` (line 50-51). Treating unknown format as a pollable status would require auditing every `_wait_for_shadow_*` loop in `cao_rest.py` (lines 533-612) to special-case it, whereas the current exception path already short-circuits polling. The CLI error surface is currently a plain string; adding a structured JSON envelope is deferred but not blocking. Implementation: keep the existing `CodexShadowParseError` raise path; no spec change needed.

## Q2) What is the minimal, stable set of Codex markers to probe supported TUI variants?

### Why this matters

- Codex output varies by version/channel; probes that are too strict will produce false `unsupported_output_format` failures.
- Probes that are too permissive can "match" unrelated output and extract garbage (or misclassify waiting prompts).
- We need a probe that works both:
  - at session readiness time (no assistant responses yet), and
  - mid-session (long scrollback where early banners may have scrolled away).

### Options

**Option A: Require a Codex banner/version line**

- Pros:
  - Low false positives.
- Cons:
  - Real TUI output may not keep a stable banner in `mode=full` scrollback.
  - Current strict banner detection is already known to be insufficient (misses "Welcome to Codex", boxed titles, etc.).

**Option B: Anchor on footer/status bar tokens + prompt glyphs (recommended)**

- Pros:
  - Matches what we see in Codex TUI snapshots and upstream CAO provider logic: footer tokens like `? for shortcuts` and `context left`, and prompt glyphs like `›/❯`.
  - Robust to scrollback truncation because footer is continuously re-rendered near the bottom.
- Cons:
  - Requires careful tail-window parsing so we don’t misread footer content as user/assistant content.

**Option C: Anchor only on prompt glyphs (`›/❯/codex>`)**

- Pros:
  - Simple.
- Cons:
  - Higher false positives (many CLIs use `>`-like prompts).
  - Prompt glyphs can appear inside transcripts.

### Proposal (Recommended)

Use Option B:

- "Supported Codex TUI variant" probe succeeds when, within the tail window, we see:
  - at least one footer token: `context left` OR `? for shortcuts`, and
  - at least one prompt glyph line starting with `›` or `❯` (ghost text after the glyph is allowed).
- Do not require a semver banner.
- For extraction stop boundaries, treat the footer region and prompt/ghost lines as chrome to exclude.

### Pros / Cons (Proposal)

- Pros:
  - Grounded in upstream Codex TUI rendering and CAO provider heuristics.
  - Works for both readiness and post-turn completion snapshots.
- Cons:
  - If Codex TUI changes footer wording, we’ll need to add a new preset/variant quickly (but drift will be explicit).

Notes / Follow-ups:

- Blocking: define the exact footer token set and tail window length for probing/status classification.

> **DECISION: Anchor Codex TUI variant detection on footer tokens (`context left`, `? for shortcuts`) plus `›` prompt glyph, not on semver banners.**
> Rationale: The Codex TUI `.snap` files in `extern/tracked/codex/codex-rs/tui/src/bottom_pane/snapshots/` and `chatwidget/snapshots/` consistently render `? for shortcuts` and `context left` in the footer bar across dozens of test snapshots, and use `›` as the primary prompt glyph (e.g., `› Ask Codex to do anything`). In contrast, the current `codex_shadow.py` `detect_output_format` (line 84-96) requires `_BANNER_RE` (a `codex cli vN` banner), which is fragile because banners scroll out of the tail window in long sessions. Replacing the banner gate with footer-token + `›`-glyph detection gives a stable probe grounded in upstream rendering. Implementation: update `detect_output_format` to check for footer tokens in `_tail_lines` and replace `_BANNER_RE` requirement; keep `_DEFAULT_STATUS_TAIL_LINES = 100` as the tail window.

## Q3) Should Gemini headless JSON parsing be part of the same "shadow parser stack" contract?

### Why this matters

- The feature request names Gemini alongside Codex/Claude, but the runtime today uses Gemini headless JSON output (not CAO/TUI scrollback parsing), and upstream `gemini-cli` explicitly supports structured `--output-format stream-json`.
- If we unify everything under one parser stack contract, we risk ballooning scope and conflating two different parsing problems (JSON event parsing vs TUI scraping).
- If we keep them separate, we still need consistent metadata/anomaly reporting and stream-contract drift detection for diagnostics and future extension.

### Options

**Option A: Unify Gemini headless parsing under the same stack contract**

- Pros:
  - One conceptual abstraction: "provider output in -> status/answer/diagnostics out".
  - Consistent anomaly reporting across all tools.
- Cons:
  - Forces the stack contract to accept multiple transport types (tmux scrollback vs JSON event streams).
  - More refactor risk across working headless paths.

**Option B: Keep Gemini headless parsing separate, but share a metadata/anomaly envelope (recommended)**

- Pros:
  - Preserves the currently robust approach: parse machine-readable JSON output, don’t scrape TUI.
  - Limits scope for the initial shadow parser stack (CAO `mode=full` parsing).
  - Still enables consistent "diagnostic shape" across backends.
- Cons:
  - Two parsing subsystems remain (headless runner vs shadow parser stack).

### Proposal (Recommended)

Pick Option B:

- Treat the "shadow parser stack" as the CAO/TUI parsing system for `mode=full`.
- Keep Gemini headless parsing in the headless runner.
- Parse Gemini as a versioned structured stream profile (event schema + completion invariants), not terminal-chrome heuristics.
- Introduce a shared "parser metadata + anomalies" envelope so Gemini and CAO shadow parsing can both report:
  - provider/tool id,
  - version (when known),
  - anomaly codes (when relevant),
  - excerpts/diagnostics pointers.

### Pros / Cons (Proposal)

- Pros:
  - Keeps working Gemini automation stable.
  - Leaves room to add a Gemini TUI preset family later if/when we introduce a CAO/TUI transport.
- Cons:
  - Requires a small amount of deliberate design to avoid two diverging metadata shapes.

Notes / Follow-ups:

- Deferrable: unify under one stack only if a real Gemini TUI parsing need emerges.

> **DECISION: Keep Gemini headless parsing separate from CAO/TUI shadow parsing; model Gemini as a versioned `stream-json` contract plus shared metadata/anomaly envelope.**
> Rationale: upstream `gemini-cli` defines non-interactive output formats `text|json|stream-json` and stream event types `init`, `message`, `tool_use`, `tool_result`, `error`, `result` (`packages/core/src/output/types.ts`, `docs/cli/headless.md`, `packages/cli/src/nonInteractiveCli.ts`). This machine-readable transport is fundamentally different from tmux scrollback scraped by `ClaudeCodeShadowParser` and `CodexShadowParser`. Forcing Gemini into the same immediate parser type would couple unrelated transports and increase risk. A shared metadata envelope (provider id, version, anomaly codes) keeps diagnostics coherent while preserving low-risk Gemini headless behavior. Implementation: keep Gemini on headless runner, add stream-schema drift fixtures, and use a common metadata/anomaly shape emitted by both headless and shadow paths.

## Q4) When should we flip Codex CAO default to `shadow_only`?

### Why this matters

- This is a behavioral default change that could surprise operators if parity is incomplete.
- The change request wants shadow parsing preferred by default, but our migration plan suggests a phased rollout.
- Codex currently defaults to `cao_only` in the runtime; changing it means shadow parser robustness must be high.

### Options

**Option A: Flip default immediately as part of this change**

- Pros:
  - Forces parity work to be complete before merge.
  - Aligns the implementation with the desired end state.
- Cons:
  - Higher risk if we missed a Codex variant in the field.
  - Fail-fast unknown-format behavior could cause incident churn early.

**Option B: Keep default `cao_only` until parity tests pass, then flip as the last step (recommended)**

- Pros:
  - Lets us build the stack, fixtures, and diagnostics before changing defaults.
  - Lower operational risk during development.
- Cons:
  - Requires discipline to actually flip at the end (or in a follow-up change).

### Proposal (Recommended)

Pick Option B:

- Implement the new stack and demonstrate parity in unit fixtures and at least one end-to-end CAO Codex run under `shadow_only`.
- Only then flip the default mapping to `tool=codex -> shadow_only`.
- Keep an explicit, documented operator override to force `cao_only` for fast rollback.

### Pros / Cons (Proposal)

- Pros:
  - Reduces the chance that a single missed Codex UI change breaks default behavior.
- Cons:
  - Slightly delays achieving the final "preferred by default" state.

Notes / Follow-ups:

- Blocking: decide whether the spec should encode the "end state" default or the "phased rollout" default.

> **DECISION: Keep Codex default at `cao_only` during development; flip to `shadow_only` only after parity fixtures pass; expose an env/CLI override for rollback.**
> Rationale: `launch_plan.py` line 18-21 defines `_CAO_PARSING_MODE_DEFAULT_BY_TOOL = {"claude": "shadow_only", "codex": "cao_only"}`, and the CLI already accepts `--cao-parsing-mode` (cli.py line 98-101) which flows through `resolve_cao_parsing_mode` (launch_plan.py line 214-239), providing a ready operator override. Flipping the default prematurely would affect every `_wait_for_shadow_*` polling path in `cao_rest.py` before `CodexShadowParser.detect_output_format` is updated (per Q2). The spec should encode the end-state default (`shadow_only`) but gate it behind a parity milestone. Implementation: add a `AGENTSYS_CAO_CODEX_PARSING_MODE` env override (parallel to the CLI flag) and flip `_CAO_PARSING_MODE_DEFAULT_BY_TOOL["codex"]` only in the final commit of this change.

## Q5) Should unit tests depend on upstream fixtures in `extern/`, or should we copy fixtures into repo-owned test data?

### Why this matters

- Depending on `extern/` content makes tests sensitive to submodule state and upstream changes.
- Copying fixtures increases duplication but gives us stable, curated coverage aligned to the runtime contract.

### Options

**Option A: Reference fixtures directly from `extern/`**

- Pros:
  - No duplication; always testing against upstream artifacts.
- Cons:
  - Tests become brittle if submodules are missing/uninitialized.
  - Upstream fixture changes can cause surprising test changes unrelated to runtime behavior.

**Option B: Copy a curated subset into `tests/fixtures/...` (recommended)**

- Pros:
  - Hermetic unit tests with stable data.
  - We can add drifted fixtures intentionally to validate explicit failure behavior.
- Cons:
  - Requires manual curation and occasional updates.

### Proposal (Recommended)

Pick Option B: copy a curated subset of representative fixtures into repo-owned test fixtures and treat upstream `extern/` as reference material only.

### Pros / Cons (Proposal)

- Pros:
  - Stable tests and faster iteration on parser behavior.
- Cons:
  - More maintenance when upstream formats evolve (but this is the core problem we are solving anyway).

> **DECISION: Copy a curated subset of upstream CAO fixtures into repo-owned `tests/fixtures/`; do not depend on `extern/` at test time.**
> Rationale: No `tests/fixtures/` directory currently exists (glob found zero matches), so we start fresh. The upstream fixtures live in `extern/orphan/cli-agent-orchestrator/` (which is git-ignored and may be absent) and `extern/tracked/codex/codex-rs/tui/src/*/snapshots/*.snap` (submodule-dependent). Depending on either makes `pytest` fail when submodules are missing or upstream format changes land. A curated fixture set lets us add intentionally-drifted snapshots to verify `unsupported_output_format` fail-fast behavior. Implementation: create `tests/fixtures/shadow_parser/{codex,claude}/` and copy representative `.snap` excerpts; add a periodic maintenance note to refresh from upstream.

## Q6) What should we do when the per-turn baseline is invalidated (scrollback reset/truncation)?

### Why this matters

- Baseline gating is the main mechanism preventing "extract an answer from a previous turn".
- In CAO `mode=full`, scrollback can shrink (tmux history limits, session resets, or other truncation), making the saved baseline offset larger than the current output length.
- If we handle this incorrectly, we can silently return stale answers, which is worse than a loud failure.

### Options

**Option A: Best-effort fallback (treat baseline as 0)**

- Pros:
  - Maximizes resilience; extraction keeps working through scrollback resets.
  - Simple implementation.
- Cons:
  - Higher risk of extracting earlier-turn answers if the output does not contain a clear stop boundary.

**Option B: Fail the turn explicitly when baseline is invalidated**

- Pros:
  - Strong safety: avoids stale extraction risk.
- Cons:
  - More operational friction; benign scrollback truncation would break turns.

**Option C: Conditional best-effort (recommended)**

- Pros:
  - Keeps the system resilient while still protecting against stale extraction.
  - Mirrors existing Claude behavior: allow baseline invalidation but require a stop boundary to prove the extracted segment is bounded.
- Cons:
  - Slightly more complexity; requires defining "stop boundary" per provider/preset.

### Proposal (Recommended)

Pick Option C:

- If baseline is invalidated, proceed with extraction only when the selected preset can confirm a safe stop boundary after the last assistant marker (for example an idle prompt line, separator token, or a known footer chrome region).
- Otherwise, fail with an explicit extraction error and include diagnostics excerpt.
- Always attach anomaly `baseline_invalidated=true` when this happens.

### Pros / Cons (Proposal)

- Pros:
  - Stronger safety guarantees than naive best-effort fallback.
  - Avoids unnecessary failures when output still contains clear boundaries.
- Cons:
  - Requires careful definition of stop boundaries for Codex TUI vs label-style variants.

> **DECISION: Use conditional best-effort extraction on baseline invalidation: require a stop boundary, otherwise fail explicitly.**
> Rationale: `claude_code_shadow.py` already implements exactly this pattern in `classify_shadow_status` (lines 242-261): when `baseline_invalidated` is true, it calls `_has_stop_boundary_after` and only returns `completed` if a stop boundary (idle prompt or separator token `────────`) is found after the response marker. `extract_last_answer` (lines 324-327) likewise raises `ClaudeCodeShadowParseError` when `baseline_invalidated and not stop_boundary_found`. The Codex shadow parser in `codex_shadow.py` does not yet enforce this—it silently resets `effective_baseline = 0` (line 120) without requiring a stop boundary. Implementation: port the Claude conditional-boundary pattern to `CodexShadowParser`, using idle prompt (`_IDLE_PROMPT_RE`) and box-drawing separators as Codex stop boundaries; always attach `baseline_invalidated=true` in `parser_metadata`.

## Q7) How should operators pin/override parser presets (naming + config surface)?

### Why this matters

- When upstream TUIs drift, operators need an escape hatch:
  - pin to an older preset that still matches their installed tool version, or
  - force a specific preset during rollout testing.
- Today, Claude has an env override (`AGENTSYS_CAO_CLAUDE_CODE_VERSION`), but Codex has no equivalent.
- The override surface must be stable and easy to document, and should not require code edits during incidents.

### Options

**Option A: Provider-specific env vars only**

- Pros:
  - Simple and explicit: one override per provider (Claude already has this).
  - Low risk of ambiguous parsing when multiple providers exist.
- Cons:
  - Requires inventing and documenting multiple env keys (Codex, Claude, Gemini).

**Option B: One unified env var that selects a fully-qualified preset id**

- Pros:
  - One knob to document; can encode provider + preset in one value.
- Cons:
  - Harder UX: users must know exact preset ids.
  - More room for misconfiguration (typos lead to confusing behavior unless validated well).

### Proposal (Recommended)

Start with Option A:

- Keep the existing Claude env override as-is for compatibility.
- Add equivalent provider-specific overrides for Codex (and Gemini only if needed), with strict validation:
  - unknown override values should fail fast with an explicit error.
- Optionally add Option B later if a unified selector becomes valuable, but only after preset ids are stable and documented.

### Pros / Cons (Proposal)

- Pros:
  - Minimal change to existing operational workflows.
  - Easier to roll out incrementally provider-by-provider.
- Cons:
  - Slightly more documentation surface area.

> **DECISION: Use provider-specific env var overrides for parser preset pinning, starting with `AGENTSYS_CAO_CODEX_VERSION` alongside the existing `AGENTSYS_CAO_CLAUDE_CODE_VERSION`.**
> Rationale: `claude_code_shadow.py` already defines `_ENV_PRESET_OVERRIDE = "AGENTSYS_CAO_CLAUDE_CODE_VERSION"` (line 24) and uses it in `_requested_version` (lines 382-385) with highest priority, falling back to banner detection and then latest-preset. This pattern is proven and operator-friendly. Codex has no equivalent today—`codex_shadow.py` hardcodes `_PARSER_PRESET_ID = "codex_shadow_v1"` (line 29) with no override mechanism. Adding `AGENTSYS_CAO_CODEX_VERSION` with strict validation (fail-fast on unknown values, mirroring `resolve_cao_parsing_mode` in `launch_plan.py` lines 222-231) is minimal and consistent. A unified preset selector can be added later once preset ids are stable across both providers. Implementation: add `_ENV_PRESET_OVERRIDE` to `codex_shadow.py` and a `resolve_preset_version` method mirroring Claude's; validate unknown values with an explicit error.
