## MODIFIED Requirements

### Requirement: `houmao-mgr internals config-drafts` exposes concise config draft generators
`houmao-mgr` SHALL expose an internal command family shaped as:

```text
houmao-mgr internals config-drafts <verb>
```

At minimum, `internals config-drafts` SHALL expose:

- `list`
- `generate`

The initial draft registry SHALL include stable draft ids for:

- `project.specialist`
- `project.profile`
- `internals.native-agent.launch-dossier`

`list` SHALL return a compact inventory containing draft ids, short descriptions, config kinds, and required intent keys. It SHALL NOT return full CLI option catalogs, executable-command schema metadata, omitted-field prose, or target argv.

`generate --id <draft-id>` SHALL accept a structured JSON intent with a `fields` mapping and SHALL emit a YAML config draft for the selected draft id without mutating project or native-agent state.

#### Scenario: Agent lists supported config drafts
- **WHEN** an agent runs `houmao-mgr --print-json internals config-drafts list`
- **THEN** the output includes `project.specialist`, `project.profile`, and `internals.native-agent.launch-dossier`
- **AND THEN** each listed item contains concise draft metadata rather than executable command fields or target argv

#### Scenario: Agent generates a project profile YAML draft
- **WHEN** an agent runs `houmao-mgr internals config-drafts generate --id project.profile --intent '{"fields":{"name":"reviewer-fast","specialist":"reviewer","credential":"reviewer-creds"}}'`
- **THEN** the command emits a YAML document for a specialist-backed project profile
- **AND THEN** the command does not create or update a project profile only because draft generation ran

#### Scenario: Missing required input fails clearly
- **WHEN** an agent generates `project.profile` without a profile name
- **THEN** the command fails clearly with a missing-input diagnostic
- **AND THEN** it does not emit a misleading partial profile draft as though it were ready to use

### Requirement: Config drafts are generated from concrete domain model builders
Config drafts SHALL be generated by Houmao-owned typed Python code and domain model builders rather than by a general dynamic template language.

Draft generators SHALL encode fixed values for the selected domain shape. For example, `project.profile` SHALL encode `profile_lane: easy_profile` and `source.kind: specialist`; callers SHALL NOT need to supply those values.

Generated YAML SHALL include only values meaningful to the selected config draft. It SHALL NOT include unsupported fields, omitted optional fields, clear flags, conflict descriptions, generic omit semantics, executable-command metadata, or renderer bookkeeping.

Config drafts SHALL NOT use Jinja2, JSON Schema, placeholder expressions, loops, filters, or a YAML mini-language as the source of truth for draft generation.

#### Scenario: Project profile draft fixes lane and source kind
- **WHEN** an agent generates `project.profile` with fields `name=reviewer-fast`, `specialist=reviewer`, and `credential=reviewer-creds`
- **THEN** the generated YAML records the profile lane as `easy_profile`
- **AND THEN** the generated YAML records the source kind as `specialist` without requiring those values in the intent

#### Scenario: Generated YAML omits executable command schema metadata
- **WHEN** an agent generates any supported config draft
- **THEN** the YAML output does not include `target_argv`, `fields`, `omitted_fields`, `applied_fields`, or `conflicts`
- **AND THEN** the output is shaped as a concrete config document rather than a CLI option schema

## ADDED Requirements

### Requirement: Config drafts remain YAML-only
`houmao-mgr internals config-drafts` SHALL remain scoped to generating configuration documents.

Config drafts SHALL NOT render executable `houmao-mgr` argv, expose command inventories, validate action-command option conflicts, or replace ordinary command help and skill-owned shell snippets.

#### Scenario: Config drafts do not render action commands
- **WHEN** an agent needs to run an action such as launching an agent, adding a credential, sending gateway input, or reading mailbox state
- **THEN** the agent uses the maintained `houmao-mgr` command directly
- **AND THEN** it does not ask `config-drafts` to produce executable argv
