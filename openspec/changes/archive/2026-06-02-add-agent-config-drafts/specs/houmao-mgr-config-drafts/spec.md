## ADDED Requirements

### Requirement: `houmao-mgr internals config-drafts` exposes concise config draft generators
`houmao-mgr` SHALL expose an internal command family shaped as:

```text
houmao-mgr internals config-drafts <verb>
```

At minimum, `internals config-drafts` SHALL expose:

- `list`
- `generate`

The initial draft registry SHALL include stable draft ids for:

- `project.easy.specialist`
- `project.easy.profile`
- `project.agents.launch-profile`

`list` SHALL return a compact inventory containing draft ids, short descriptions, config kinds, and required intent keys. It SHALL NOT return full CLI option catalogs, command-template field metadata, omitted-field prose, or target argv.

`generate --id <draft-id>` SHALL accept a structured JSON intent with a `fields` mapping and SHALL emit a YAML config draft for the selected draft id without mutating project state.

#### Scenario: Agent lists supported config drafts
- **WHEN** an agent runs `houmao-mgr --print-json internals config-drafts list`
- **THEN** the output includes `project.easy.specialist`, `project.easy.profile`, and `project.agents.launch-profile`
- **AND THEN** each listed item contains concise draft metadata rather than command-template fields or target argv

#### Scenario: Agent generates an easy profile YAML draft
- **WHEN** an agent runs `houmao-mgr internals config-drafts generate --id project.easy.profile --intent '{"fields":{"name":"reviewer-fast","specialist":"reviewer"}}'`
- **THEN** the command emits a YAML document for a specialist-backed easy profile
- **AND THEN** the command does not create or update a project profile only because draft generation ran

#### Scenario: Missing required input fails clearly
- **WHEN** an agent generates `project.easy.profile` without a profile name
- **THEN** the command fails clearly with a missing-input diagnostic
- **AND THEN** it does not emit a misleading partial profile draft as though it were ready to use

### Requirement: Config drafts are generated from concrete domain model builders
Config drafts SHALL be generated by Houmao-owned typed Python code and domain model builders rather than by a general dynamic template language.

Draft generators SHALL encode fixed values for the selected domain shape. For example, `project.easy.profile` SHALL encode `profile_lane: easy_profile` and `source.kind: specialist`; callers SHALL NOT need to supply those values.

Generated YAML SHALL include only values meaningful to the selected config draft. It SHALL NOT include unsupported fields, omitted optional fields, clear flags, conflict descriptions, generic omit semantics, or command-template renderer bookkeeping.

Config drafts SHALL NOT use Jinja2, JSON Schema, placeholder expressions, loops, filters, or a YAML mini-language as the source of truth for draft generation.

#### Scenario: Easy profile draft fixes lane and source kind
- **WHEN** an agent generates `project.easy.profile` with fields `name=reviewer-fast` and `specialist=reviewer`
- **THEN** the generated YAML records the profile lane as `easy_profile`
- **AND THEN** the generated YAML records the source kind as `specialist` without requiring those values in the intent

#### Scenario: Generated YAML omits command-template schema metadata
- **WHEN** an agent generates any supported config draft
- **THEN** the YAML output does not include `target_argv`, `fields`, `omitted_fields`, `applied_fields`, or `conflicts`
- **AND THEN** the output is shaped as a concrete config document rather than a CLI option schema

### Requirement: Initial config drafts cover pre-launch agent definition shapes
The initial config-draft registry SHALL cover the pre-launch project configuration shapes that agents commonly author before launching managed agents.

`project.easy.specialist` SHALL generate a high-level specialist draft that captures the specialist name, tool lane, credential/setup references or explicit credential material when supplied, launch defaults, skills, mailbox posture, and prompt-related references supplied in the intent.

`project.easy.profile` SHALL generate a specialist-backed easy profile draft with the source specialist and any explicit launch defaults supplied in the intent.

`project.agents.launch-profile` SHALL generate a recipe-backed raw launch-profile draft with the source recipe and any explicit launch defaults supplied in the intent.

These drafts SHALL remain authoring aids and SHALL NOT make generated YAML files the canonical storage source for the project catalog.

#### Scenario: Specialist draft uses high-level specialist semantics
- **WHEN** an agent generates `project.easy.specialist` for name `reviewer` and tool `codex`
- **THEN** the generated YAML identifies the specialist and selected tool lane
- **AND THEN** it does not pretend that specialist state is stored as one existing compatibility projection file when the persisted specialist spans catalog rows and managed content references

#### Scenario: Raw launch-profile draft uses recipe-backed source
- **WHEN** an agent generates `project.agents.launch-profile` for name `alice` and recipe `reviewer-codex`
- **THEN** the generated YAML records the source kind as `recipe`
- **AND THEN** the generated YAML records only the explicit launch defaults supplied in the intent

#### Scenario: Draft generation does not replace project mutation commands
- **WHEN** an agent generates a profile or specialist config draft
- **THEN** the command returns an authoring document
- **AND THEN** persisted project catalog state remains unchanged until a maintained mutation command is run
