## Summary

- The change is directionally strong: it removes the biggest source of ambiguity by making full mailbox address the routing identity and pushing join/leave into explicit managed lifecycle operations.
- The runtime tightening (no `--instruction`, explicit body inputs, full-form recipient validation) aligns well with the existing “runtime-owned mailbox command” design in `src/gig_agents/agents/brain_launch_runtime/mail_commands.py`.
- Assumption: this is pre-adoption and intentionally breaking; it is acceptable to require deleting and re-bootstrapping any existing mailbox roots.
- The main risk is not “what” but “how”: the artifacts don’t yet pin down lock keys and the exact SQLite constraints/foreign-key strategy needed to make `safe`/`force`/`stash`/`deactivate`/`purge` deterministic and race-resistant.

## Findings

### Must Fix / Blocking

- Specify the concrete SQLite DDL strategy (tables, keys, constraints, and deletion semantics) needed to enforce “at most one active registration per address” and to make `purge` possible without breaking canonical-message retention (current schema is principal-keyed and uses FKs; see `src/gig_agents/mailbox/filesystem.py:20`).
- Specify the locking model for delivery vs register/deregister so `force`/`stash` and `purge` cannot race with delivery or state mutation (current locking is principal-id based under `locks/principals/`; see `src/gig_agents/mailbox/managed.py:547` and `src/gig_agents/mailbox/managed.py:886`).
- Define the managed script payload/result contract for `register_mailbox.py` and `deregister_mailbox.py` (follow the existing `--payload-file` JSON pattern in `src/gig_agents/mailbox/assets/rules/scripts/deliver_message.py`).

### Should Fix / Important

- Be explicit in code/docs/tests that this change is a hard reset (old mailbox roots are unsupported; delete and re-bootstrap instead of migrating).
- Call out the runtime env-binding impact explicitly: today the runtime binds `AGENTSYS_MAILBOX_FS_INBOX_DIR` as `mailboxes/<principal_id>/inbox` in `src/gig_agents/agents/mailbox_runtime_support.py:271`; an address-keyed mailbox directory layout implies this binding (and tests/docs) will change.
- Tighten/standardize address validation if addresses become filesystem directory names and lock-file names; current validation only checks “email-like” and “no whitespace” in `src/gig_agents/mailbox/protocol.py:60`.
- Expand task granularity with concrete file targets for the breaking CLI contract changes (tests already encode the old behavior, for example `tests/unit/agents/brain_launch_runtime/test_mail_commands.py:229` and `tests/integration/agents/brain_launch_runtime/test_mailbox_runtime_contract.py:272`).

### Nice To Have

- Add a lifecycle transition table (or Mermaid diagram) for `active`/`inactive`/`stashed` and replacement relationships so the “one active” invariant is easy to audit.
- Include example JSON payloads and example outputs for `register_mailbox.py` / `deregister_mailbox.py` that cover `safe|force|stash` and `deactivate|purge`.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should we bump the mailbox protocol version (and/or rename “v1” artifacts) as part of this breaking refactor?

- Decision: `Deferrable`

#### Why this matters

- This change is intentionally breaking and pre-adoption, so compatibility with existing mailbox roots is not a goal.
- Even so, choosing whether to bump `protocol-version.txt` (and/or rename v1-labeled protocol notes) affects how much time developers spend when they accidentally reuse a stale mailbox root during iteration.

#### Examples

- A developer has a stale mailbox root with the old principal-keyed registry schema (created by code in `src/gig_agents/mailbox/filesystem.py:20`) and then sees confusing failures when the new managed layer expects address-scoped registrations.

#### Why this question arises in code

- `protocol-version.txt` is validated against `MAILBOX_PROTOCOL_VERSION` in `src/gig_agents/mailbox/filesystem.py:205` and `src/gig_agents/mailbox/managed.py:563`.
- The same `MAILBOX_PROTOCOL_VERSION` constant is also used as the canonical message envelope version in `src/gig_agents/mailbox/protocol.py:125`, so bumping it cascades into message parsing/serialization and tests.
- Several protocol artifacts are labeled as v1 today (`src/gig_agents/mailbox/assets/rules/protocols/filesystem-mailbox-v1.md`, `tests/unit/mailbox/test_filesystem.py:14`).

#### Options

**Option A: Keep `protocol-version.txt` at `1` and treat this as the new v1 contract; require devs to delete/re-bootstrap old mailbox roots.**

- Code impact:
  - Update schema/scripts/docs/tests in place.
  - Optionally add a targeted “schema mismatch; delete and re-bootstrap” error when detecting legacy tables (for example, a legacy `principals` registry).
- User-facing difference:
  - Stale roots are not supported; the workflow is to delete and recreate them.
- Pros:
  - Avoids version churn and avoids cascading changes to the canonical message envelope version.
- Cons:
  - Easier to accidentally point at a stale root and lose time debugging before realizing it must be rebuilt.

**Option B: Bump the on-disk mailbox protocol version (and optionally rename v1 docs to v2) so stale roots fail fast.**

- Code impact:
  - Update version checks (`src/gig_agents/mailbox/filesystem.py:205`, `src/gig_agents/mailbox/managed.py:563`) and update protocol notes/tests that assume v1 naming.
  - Decide whether to also bump the message envelope version (tied to `MAILBOX_PROTOCOL_VERSION` in `src/gig_agents/mailbox/protocol.py:125`) or to split “root protocol version” from “message envelope version”.
- User-facing difference:
  - Stale roots fail immediately with a clear “unsupported mailbox protocol version” error and can be recreated.
- Pros:
  - Clear “hard reset” signal; less confusion during iterative development.
- Cons:
  - More churn/renames; may require decoupling version constants if you do not want to bump the message envelope format simultaneously.

> **PROPOSED: Option A (keep version `1`, no compatibility), plus an explicit “delete and re-bootstrap mailbox root” note and a targeted legacy-schema detection error if it’s cheap.**
> Rationale: Because `MAILBOX_PROTOCOL_VERSION` is currently shared between `protocol-version.txt` checks (`src/gig_agents/mailbox/filesystem.py:205`, `src/gig_agents/mailbox/managed.py:563`) and the canonical message envelope (`src/gig_agents/mailbox/protocol.py:125`), bumping it has a wider blast radius than a pure registration refactor. Since we explicitly do not care about compatibility, the simplest approach is to treat old roots as unsupported and make the “delete/re-bootstrap” workflow obvious (optionally with a crisp error when the legacy principal registry is detected).
>
> **DECISION: Accept Option A, refined as “keep version `1` and treat this as the intended v1 contract; old mailbox roots are unsupported and should be deleted and re-bootstrapped.”**
> Rationale: The current change already says there is no migration plan because this is a pre-adoption refactor, and that is the right boundary to keep. Bumping `MAILBOX_PROTOCOL_VERSION` would also touch the canonical envelope version in `src/gig_agents/mailbox/protocol.py`, which is not what this review is trying to solve. We should keep the version at `1`, keep v1 naming, and make stale principal-keyed roots fail with a clear “delete and re-bootstrap” message when legacy schema detection is straightforward.

### Q2) What is the lock key strategy for delivery vs register/deregister in an address-routed world?

- Decision: `Blocking`

#### Why this matters

- `force`/`stash` join and `purge` deregistration can mutate registrations and mailbox paths while delivery is writing projections and mutating SQLite state; incorrect lock scoping can cause lost updates or inconsistent projections.

#### Examples

- A delivery to `AGENTSYS-bob@agents.localhost` resolves “active registration”, while a concurrent `stash` join renames or replaces that mailbox entry; without a shared lock key, delivery can project into the wrong mailbox path.

#### Why this question arises in code

- Today the managed layer locks per principal id via `locks/principals/<principal>.lock` and always takes an `index.lock` (`src/gig_agents/mailbox/managed.py:547`, `src/gig_agents/mailbox/managed.py:886`).
- The proposed design routes and deconflicts by mailbox address, which may be transferred/replaced across owners during `force`/`stash`.

#### Options

**Option A: Introduce `locks/addresses/<address>.lock` and use address locks (plus `index.lock`) for delivery and lifecycle ops.**

- Code impact:
  - Update managed lock acquisition to accept addresses (or registration ids) as lock keys instead of principal ids (`src/gig_agents/mailbox/managed.py:886`).
  - Update filesystem-layout docs (`src/gig_agents/agents/brain_launch_runtime/assets/system_skills/mailbox/email-via-filesystem/references/filesystem-layout.md`).
- User-facing difference:
  - More predictable serialization around address replacement, fewer surprising races.
- Pros:
  - Aligns lock keys with routing identity.
- Cons:
  - Address strings become filesystem lock-path segments; requires clear validation/encoding rules.

**Option B: Keep principal-id locks only and ensure lifecycle ops lock all involved principals.**

- Code impact:
  - Minimal filesystem layout changes; keep `locks/principals/`.
  - Requires lifecycle ops to know which principals to lock (including “conflicting owner” during `force`/`stash`).
- User-facing difference:
  - Higher contention; address replacement can still be racy if ownership changes and delivery uses different principal resolution.
- Pros:
  - Less churn in filesystem layout.
- Cons:
  - Does not naturally serialize “address replacement” unless every operation reliably identifies and locks the same principal set.

> **PROPOSED: Add address-scoped locks (and keep `index.lock`) so register/deregister/deliver serialize on the routing identity.**
> Rationale: The managed layer currently assumes the lock key is the same thing it uses to find mailbox paths (`principal_id` in `src/gig_agents/mailbox/managed.py:591` and locks in `src/gig_agents/mailbox/managed.py:886`). The change makes the routing identity the mailbox address, so the simplest way to prevent delivery/lifecycle races is to lock by address as well.
>
> **DECISION: Accept Option A, refined as “use address-scoped locks plus `index.lock`, and stop treating principal-id locks as the authoritative serialization key.”**
> Rationale: Delivery, register, deregister, and mailbox-state updates should serialize on the routing identity they are mutating, which is now the full mailbox address rather than `principal_id`. The simplest coherent rule is: acquire all affected address locks in sorted order, then acquire `index.lock`. The implementation should use one shared address-to-path-segment helper for both `mailboxes/<address>` and `locks/addresses/<address>.lock` so path safety is defined once.

### Q3) What concrete SQLite model enforces “one active registration per address” and supports `stash`/`purge` cleanly?

- Decision: `Blocking`

#### Why this matters

- The current SQLite schema is principal-id keyed and uses foreign keys that will interact with `purge` semantics; the refactor needs a clear, enforceable model for active/inactive/stashed registrations and “replacement linkage”.

#### Examples

- Two concurrent joins attempt to create `active` registrations for the same address. If the uniqueness rule is only enforced in Python, it is easier to accidentally regress; if it is enforced in SQLite, it becomes testable and race-resistant.

#### Why this question arises in code

- The current schema has a `principals` table with `principal_id` primary key (`src/gig_agents/mailbox/filesystem.py:22`) and ties `mailbox_state` / `mailbox_projections` to `principal_id` (`src/gig_agents/mailbox/filesystem.py:78`).
- Managed delivery loads registrations by `principal_id` (`src/gig_agents/mailbox/managed.py:591`) and projects into `mailbox_path/inbox|sent` based on that registration (`src/gig_agents/mailbox/managed.py:644`).

#### Options

**Option A: Introduce a `mailbox_registrations` table with stable `registration_id` plus SQLite-enforced constraints (including a partial unique index for active address).**

- Code impact:
  - Replace the principal-keyed “registry” schema (`src/gig_agents/mailbox/filesystem.py:20`) with registration tables + indexes.
  - Re-key mutable state/projections to `registration_id` instead of `principal_id`.
  - Update managed delivery/state/repair paths to resolve recipients by address to an active registration.
- User-facing difference:
  - `stash` produces a historical `stashed` registration that remains inspectable; `purge` can delete only registration-scoped mutable state while preserving canonical messages.
- Pros:
  - Directly matches the goals in the design and the new lifecycle requirements.
- Cons:
  - Larger refactor; needs careful choices around foreign keys and repair/reindex behavior.

**Option B: Keep principal-keyed state/projections and treat address routing as a lookup layer only.**

- Code impact:
  - Smaller schema change; keep `mailbox_state` keyed by principal id.
  - Add an address-to-principal mapping and enforce “one active address per principal”.
- User-facing difference:
  - Harder to represent `stash` as “historical mailbox artifacts” without reintroducing principal-address coupling.
- Pros:
  - Lower implementation churn.
- Cons:
  - Conflicts with the change goals: it preserves principal as the effective mailbox identity for state/projections.

> **PROPOSED: Use an explicit `mailbox_registrations` model with SQLite-enforced invariants (including “one active registration per address”) and attach mutable state/projections to the concrete registration.**
> Rationale: The refactor explicitly states that “registration-scoped mutable data such as mailbox projections and mailbox-state rows will attach to the concrete registration” (`openspec/changes/refactor-mailbox-address-routing-and-lifecycle/design.md:87`). The current schema and managed code are principal-keyed (`src/gig_agents/mailbox/filesystem.py:78`, `src/gig_agents/mailbox/managed.py:591`), so an explicit registration table with DB-level constraints is the most coherent way to implement `stash` history and `purge` cleanup without reintroducing principal-based identity coupling.
>
> **DECISION: Accept Option A, and pin it down further: introduce `mailbox_registrations` keyed by `registration_id`, enforce “one active registration per address” in SQLite, keep canonical recipient history decoupled from mutable registration rows, and key mutable mailbox state/projections by `registration_id`.**
> Rationale: This is the cleanest way to satisfy the refactor’s actual goals. Concretely, the change should add a `mailbox_registrations` table with at least `registration_id`, `address`, `owner_principal_id`, `status`, `mailbox_kind`, `mailbox_path`, `created_at_utc`, and replacement linkage, plus a SQLite partial unique index on `address` for rows where `status = 'active'`. `mailbox_projections` and `mailbox_state` should reference `registration_id` so `purge` can delete registration-scoped mutable state cleanly, while canonical message-recipient history should remain address-snapshot-oriented instead of depending on a live registration row.

### Q4) Should `mailboxes/<address>/` use literal addresses or an encoded directory name?

- Decision: `Deferrable`

#### Why this matters

- The design chooses literal full addresses for directory names, but the current address validation is permissive; once addresses become filesystem path segments and lock file names, validation/encoding becomes part of the safety contract.

#### Examples

- An address containing a path separator could lead to unexpected directory layout or lock-file paths unless explicitly rejected or encoded.

#### Why this question arises in code

- The current address validator in `src/gig_agents/mailbox/protocol.py:60` checks only `@` placement and whitespace.
- Current runtime bindings and docs assume principal-keyed directories (`src/gig_agents/agents/mailbox_runtime_support.py:271`, `src/gig_agents/agents/brain_launch_runtime/assets/system_skills/mailbox/email-via-filesystem/references/filesystem-layout.md`).

#### Options

**Option A: Use the literal full address as the directory name and tighten validation to ensure it is a safe path segment.**

- Code impact:
  - Add a shared “address is a safe path segment” validator/helper and use it anywhere the address becomes a path.
- User-facing difference:
  - Directory names are inspectable and map 1:1 to routing identity.
- Pros:
  - Matches the design rationale (“operator inspection honest”) and is easy to reason about.
- Cons:
  - Requires precise validation rules and portability expectations.

**Option B: Encode addresses for directory names (while keeping the canonical address in SQLite/message headers).**

- Code impact:
  - Introduce encoding/decoding helpers and update docs accordingly.
- User-facing difference:
  - Mailbox directories are less human-readable.
- Pros:
  - Avoids filesystem portability concerns.
- Cons:
  - Adds indirection and more moving parts for operators.

> **PROPOSED: Use literal full addresses for `mailboxes/<address>/` but tighten validation and centralize the “address-to-path-segment” helper.**
> Rationale: The change already commits to literal address directory names (examples in `openspec/changes/refactor-mailbox-address-routing-and-lifecycle/design.md:159`). Tightening validation is cheaper than introducing an encoding layer, and it directly mitigates the fact that address validation is currently minimal (`src/gig_agents/mailbox/protocol.py:60`).
>
> **DECISION: Accept Option A, with no separate encoding layer in v1.**
> Rationale: Literal full addresses keep operator inspection and docs honest, and they align with the design examples already chosen. The right refinement is to strengthen address validation and route all filesystem usage through one shared helper that rejects unsafe path-segment cases before an address is used for a mailbox directory or lock filename. That keeps the model readable without introducing a second encoded naming scheme.
