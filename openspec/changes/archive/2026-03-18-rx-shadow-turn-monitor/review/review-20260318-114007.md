## Summary

- The proposal targets real reliability problems in `_TurnMonitor` and is directionally aligned with the repo's push toward deterministic shadow-mode testing.
- The current D2/D3 operator placement is semantically wrong for Rx: filtered `debounce()` and filtered `timeout()` branches do not reset or cancel on non-matching observations, so the spec's transient-idle and stall-recovery scenarios fail as written.
- D7/D8 and tasks 4.2/4.3 conflict with the current single-thread polling loops in `cao_rest.py`; `observable.run()` cannot block the same thread that is expected to keep polling and feeding the `Subject`.
- The artifact set under-specifies repo-native touchpoints for `completion_stability_seconds` and for replacing the existing `_TurnMonitor`-focused unit suite.

## Findings

### Must Fix / Blocking

1. **The readiness stall design is incorrect as written: `ops.timeout()` on an unknown-only sub-stream will still fire after the runtime has already recovered to a known state.**

   `openspec/changes/rx-shadow-turn-monitor/design.md:53-58` and `openspec/changes/rx-shadow-turn-monitor/tasks.md:10-12` describe the stall timer as `ops.timeout()` over observations filtered to "unknown-for-stall". But the current implementation resets unknown/stalled timing on any known observation via `_recover_if_stalled()` in `src/houmao/agents/realm_controller/backends/cao_rest.py:381-386` and `src/houmao/agents/realm_controller/backends/cao_rest.py:512-536`, and the spec requires known business state and stall recovery to avoid `stalled` in `openspec/changes/rx-shadow-turn-monitor/specs/brain-launch-runtime/spec.md:14-38`. A filtered unknown branch does not see the recovery event, so it cannot cancel the pending timeout. I verified this locally with the installed `reactivex` package using `TestScheduler`: `unknown@10 -> known@30` piped through `filter(unknown) |> timeout(50)` still produced `Timeout@60`.

> **DECISION: Accept.**
> Rationale: The filtered sub-stream approach is structurally broken for Rx. The current imperative code resets unknown/stalled timing on any known observation via `_recover_if_stalled()` (`cao_rest.py:381-386`, `512-536`), and the spec requires stall recovery on known business state (`spec.md:14-38`). A `filter(unknown) |> timeout()` branch is blind to recovery events, so the timeout fires even after the surface has returned to a known state. D3 and task 2.2 must be rewritten to operate on the full classified-state stream, where any non-unknown emission cancels or resets the pending stall timeout.

2. **The completion debounce design is also incorrect as written: `ops.debounce()` on a submit-ready-only sub-stream will complete even if the surface returns to `working` during the stability window.**

   `openspec/changes/rx-shadow-turn-monitor/design.md:45-52` and `openspec/changes/rx-shadow-turn-monitor/tasks.md:16-20` say to debounce the "submit-ready-with-evidence" sub-stream, but `openspec/changes/rx-shadow-turn-monitor/specs/brain-launch-runtime/spec.md:74-97` requires any state change, including `business_state` transitions back to `working`, to reset the stability timer. The current monitor explicitly treats `business_state == working` as in-progress before considering completion in `src/houmao/agents/realm_controller/backends/cao_rest.py:436-447`. A filtered submit-ready branch hides the `working` recovery, so the debounce timer is not reset. Local `TestScheduler` validation in this workspace showed `submit_ready@10 -> working@30` through `filter(submit_ready) |> debounce(50)` still emitted the earlier submit-ready event at `60`.

> **DECISION: Accept.**
> Rationale: Same structural flaw as Finding 1. The spec requires any state change — including `business_state` transitions back to `working` — to reset the stability timer (`spec.md:79`). The current imperative code handles this correctly: `business_state == "working"` sets `m_state = "in_progress"` before completion is considered (`cao_rest.py:436-437`). A `filter(submit_ready) |> debounce()` branch is blind to `working` recoveries, so the debounce timer runs to completion even when the surface has returned to active work. D2 and task 3.4 must be rewritten to operate on the full classified-state stream, where any non-idle emission resets the debounce window.

3. **The proposed execution model is internally inconsistent: `observable.run()` cannot share a thread with the polling loop that feeds the `Subject`.**

   `openspec/changes/rx-shadow-turn-monitor/proposal.md:13`, `openspec/changes/rx-shadow-turn-monitor/design.md:82-94`, and `openspec/changes/rx-shadow-turn-monitor/tasks.md:24-26` say `_wait_for_shadow_ready_status()` / `_wait_for_shadow_completion()` will create a `ReplaySubject`, keep pushing observations from the polling loop, and block on `observable.run()`. The current producer is the loop itself in `src/houmao/agents/realm_controller/backends/cao_rest.py:1175-1231` and `src/houmao/agents/realm_controller/backends/cao_rest.py:1270-1350`; once `run()` blocks that same thread, no further polls happen. This needs a concrete topology decision: either a background producer thread owns polling, or the existing current-thread loop subscribes non-blockingly and exits when the pipeline yields a terminal result.

> **DECISION: Accept.**
> Rationale: `observable.run()` blocks the calling thread, but the calling thread is the same one that owns the `while time.monotonic() < deadline:` polling loop (`cao_rest.py:1175-1231`, `1270-1350`). Once `run()` blocks, no further `_get_terminal_output_full()` calls happen, so the `Subject` starves. This is resolved by Q2's decision: keep polling on the current thread (Option A), drop `observable.run()`, and let the pipeline deliver terminal results via a callback or result-holder. D7 and tasks 4.2/4.3 must be revised accordingly.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Which snapshot surface should the completion pipeline treat as change evidence?

- Decision: `Resolved` — Option A (`DialogProjection.normalized_text`)

#### Why this matters

- This choice determines whether D4 really decouples lifecycle behavior from projector drop-rule changes or merely renames the same dependency.
- It also decides how baseline invalidation and future parser-preset upgrades affect completion semantics.

#### Examples

- If a parser preset update changes which banner or spinner lines are dropped, should completion behavior stay stable across that preset bump?
- If scrollback shrinks and `baseline_invalidated` is set, should the lifecycle diff key come from the closer-to-source snapshot or from the best-effort projected dialog?

#### Why this question arises in code

- `src/houmao/agents/realm_controller/backends/shadow_parser_core.py:90-104` documents `normalized_text` as the closer-to-source surface and `dialog_text` as a best-effort projection.
- The current completion monitor compares `dialog_projection.dialog_text` to a stored baseline in `src/houmao/agents/realm_controller/backends/cao_rest.py:411-418`.

```python
if dialog_projection.dialog_text != baseline_projection_text:
    self.m_saw_projection_change_after_submit = True
```

#### Options

**Option A: Key change detection on `DialogProjection.normalized_text`, with any extra stripping rules defined explicitly in the pipeline module.**

- Code impact:
  - `ShadowObservation` or the pipeline helpers should read from the closer-to-source snapshot surface instead of projector-derived dialog text.
  - Any extra normalization becomes a tested lifecycle helper rather than implicit projector behavior.
- User-facing difference:
  - Completion semantics stay stable across projector/preset churn unless the lifecycle normalization rules are intentionally changed.
- Pros:
  - Best matches the existing parser contracts.
  - Actually fulfills the proposal's decoupling goal.
- Cons:
  - Requires defining and testing a small normalization contract in the pipeline layer.

**Option B: Keep using projector-derived `dialog_text`, optionally wrapped by `normalize_projection_text()`.**

- Code impact:
  - Migration is mechanically simpler because it mirrors today's baseline comparison.
  - The new helper still depends on projector output semantics.
- User-facing difference:
  - Completion behavior can still drift when projector drop rules change across presets.
- Pros:
  - Lowest short-term implementation cost.
- Cons:
  - Undercuts one of the main reasons for the change.

> **PROPOSED: Use `DialogProjection.normalized_text` (Option A) and keep any additional lifecycle-only stripping rules in the new pipeline module.**
> Rationale: `src/houmao/agents/realm_controller/backends/shadow_parser_core.py:93-95` explicitly separates closer-to-source `normalized_text` from best-effort `dialog_text`, so the former is the only surface that actually matches D4's stated goal of decoupling lifecycle decisions from projector heuristics. If a few transient lines still need ignoring, define those rules in one pipeline-owned helper and cover them with unit tests rather than relying on parser preset behavior.
>
> **DECISION: Accept PROPOSED (Option A).**
> Rationale: `normalized_text` is the closer-to-source surface and the only field whose semantics are stable across projector preset changes. The parser contract at `shadow_parser_core.py:90-104` already documents this separation intentionally. Keying lifecycle change detection on `dialog_text` would preserve the exact coupling D4 exists to remove — preset bumps that change which banners or spinners are dropped would silently alter completion behavior. Any additional lifecycle-only stripping (e.g., ignoring cursor blink artifacts) belongs in a tested `normalize_projection_text()` helper owned by the pipeline module, not inherited from projector drop rules. The revision pass should update D4 and task 1.4 to explicitly name `DialogProjection.normalized_text` as the input surface.

### Q2) What producer/subscriber topology should the synchronous wait functions use?

- Decision: `Resolved` — Option A (current-thread polling, callback-based result capture)

#### Why this matters

- This choice determines whether the Rx rewrite can preserve today's synchronous API without deadlocking the wait functions.
- It also controls how much new concurrency complexity the backend introduces just to support `debounce()` and `timeout()`.

#### Examples

- `_wait_for_shadow_ready_status()` must keep polling CAO every `poll_interval_seconds` while the readiness pipeline watches for ready, blocked, failed, or stalled terminal outcomes.
- `_wait_for_shadow_completion()` must let mailbox observer bypasses return immediately without making thread shutdown or error propagation ambiguous.

#### Why this question arises in code

- The current wait functions own both the polling loop and the error translation in `src/houmao/agents/realm_controller/backends/cao_rest.py:1164-1366`.
- D7/D8 currently propose both same-function polling and a blocking `observable.run()` call.

```python
while time.monotonic() < deadline:
    output = self._get_terminal_output_full()
    subject.on_next(observation)
    ...
# this cannot coexist with same-thread observable.run()
```

#### Options

**Option A: Keep polling on the current thread, subscribe once to the pipeline, and let callbacks store a terminal result while the loop continues pushing observations until that result is set.**

- Code impact:
  - No dedicated producer thread is needed.
  - The Rx scheduler handles only timed operators; the existing wait functions still own polling, cancellation, and error mapping.
- User-facing difference:
  - External behavior stays synchronous, with less new concurrency under the hood.
- Pros:
  - Best matches the current `cao_rest.py` architecture.
  - Keeps shutdown/error handling close to the existing code paths.
- Cons:
  - Needs a small result-holder or coordination object instead of `observable.run()`.

**Option B: Move polling into a dedicated producer thread and block the caller on `observable.run()`.**

- Code impact:
  - Requires thread startup/shutdown, cross-thread exception propagation, and explicit producer lifecycle management.
  - Makes `ReplaySubject` or another cross-thread bridge part of the design.
- User-facing difference:
  - External API stays synchronous, but the backend becomes more concurrent internally.
- Pros:
  - Fits the current wording around `observable.run()`.
- Cons:
  - Adds concurrency complexity that the current backend does not otherwise need.

> **PROPOSED: Keep the producer on the current thread (Option A) and drop the `observable.run()` requirement.**
> Rationale: `src/houmao/agents/realm_controller/backends/cao_rest.py:1164-1366` is already structured around current-thread polling plus immediate error translation, and nothing else in this backend introduces a per-turn producer thread. Letting the poll loop remain the producer while the Rx scheduler only handles timed operators is the smallest architectural change that still enables deterministic `TestScheduler` unit tests.
>
> **DECISION: Accept PROPOSED (Option A).**
> Rationale: The current wait functions own both the polling loop and error translation on a single thread (`cao_rest.py:1164-1366`). Finding 3 already established that `observable.run()` deadlocks this architecture. Option B would introduce thread startup/shutdown, cross-thread exception propagation, and producer lifecycle management — concurrency complexity that nothing else in this backend requires and that makes error handling harder to reason about. Option A preserves the existing loop structure: the poll loop pushes `ShadowObservation` into a `Subject`, the Rx pipeline handles temporal operators (`debounce`, `timeout`) on a scheduler, and a callback or result-holder captures the terminal outcome so the loop knows when to stop. The revision pass should update D7 to drop `NewThreadScheduler` + `observable.run()` in favor of current-thread polling with callback-based result capture, and update tasks 4.2/4.3 accordingly.
