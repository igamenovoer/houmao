## Summary

- **Strong structural alignment:** The change correctly mirrors the existing CAO interactive demo's operator shape (start/inspect/send-turn/interrupt/verify/stop) while cleanly swapping the post-launch control boundary to direct houmao-server HTTP routes. All referenced server routes (`/houmao/agents/…`, `/houmao/terminals/…`, `/cao/sessions/…`) exist and behave as described.
- **Correct backend and route validation:** `houmao_server_rest` is a valid `BackendKind` in `src/houmao/agents/realm_controller/models.py` and its backend implementation (`backends/houmao_server_rest.py`) validates real houmao-server connectivity. The `POST /houmao/agents/{agent_ref}/stop` route is confirmed headless-only per `service.py:879-920`, justifying the design's v1 use of `DELETE /cao/sessions/{session_name}` for TUI teardown.
- **Registration parameter format mismatch risk:** The server's `POST /houmao/launches/register` endpoint currently takes individual parameters (query/form), not a JSON body. Neither the design nor the spec clarifies this, creating implementation ambiguity.
- **`tracked_agent_id` discovery gap:** After registration, the `HoumaoRegisterLaunchResponse` returns `session_name` and `terminal_id` but not `tracked_agent_id`. The design says to persist it "once discoverable" without specifying the discovery mechanism.
- **Shared code reuse strategy is unspecified:** The CAO demo's `brain_recipes.py` module is well-tested and directly reusable, but neither the design nor the tasks specify whether to share it, copy it, or rewrite it.
- **Demo-owned houmao-server lifecycle is the highest-risk area:** It requires launching a houmao-server (which itself wraps a child CAO process), selecting a free port, and managing teardown — significantly more complex than the CAO demo's `cao_server.py` equivalent.

## Findings

### Must Fix / Blocking

1. **Spec minimum file list is incomplete relative to task scope.** The spec's first requirement lists exactly five files for the demo-pack directory: `README.md`, `run_demo.sh`, `launch_alice.sh`, `send_prompt.sh`, `stop_demo.sh`. However, Task 1.3 specifies CLI wiring for six commands: `start`, `inspect`, `send-turn`, `interrupt`, `verify`, and `stop`. The existing CAO demo at `scripts/demo/cao-interactive-full-pipeline-demo/` includes wrapper scripts for all its commands (`run_demo.sh`, `launch_alice.sh`, `send_prompt.sh`, `send_keys.sh`, `stop_demo.sh`). The spec either needs to list all expected wrapper scripts or explicitly state the five-file list is a minimum set. Without this, an implementer who follows the spec exactly will not produce the wrappers needed by the tasks.

> **DECISION: Reject.**
> Rationale: The current spec requirement already behaves as a minimum shell surface rather than an exhaustive wrapper inventory. `run_demo.sh` can own the full `start / inspect / send-turn / interrupt / verify / stop` command surface while `launch_alice.sh`, `send_prompt.sh`, and `stop_demo.sh` remain selective convenience wrappers, which is consistent with the older CAO demo pattern.

2. **Registration endpoint parameter format is unspecified and could cause implementation errors.** The actual `POST /houmao/launches/register` route in `src/houmao/server/app.py:252-277` accepts individual parameters (FastAPI dependency injection from query/form), not a JSON request body. The design lists the registration fields but does not specify the parameter passing format. If an implementer constructs a JSON body (a common default assumption for POST endpoints), the request will fail silently or with a 422. The spec should state the expected submission format, or the demo implementation should use the existing `HoumaoServerClient` which abstracts this detail.

> **DECISION: Refine.**
> Rationale: This is a real implementation risk, but the right fix is to steer implementation toward the existing `HoumaoServerClient.register_launch()` helper or an equivalent parameterized call shape in design and tasks, not to hard-code low-level POST parameter mechanics into the behavioral spec.

### Should Fix / Important

3. **No explicit strategy for reusing the CAO demo's `brain_recipes.py` module.** The Migration Plan (design.md step 2) says "Reuse the tracked brain-recipe selection model from the CAO interactive demo", but the tasks don't operationalize this. `src/houmao/demo/cao_interactive_demo/brain_recipes.py` contains `resolve_demo_brain_recipe()` and `ResolvedDemoBrainRecipe` — a well-tested 195-line module with selector normalization, basename ambiguity detection, and `default_agent_name` validation. Without explicit guidance, an implementer may duplicate this logic in the new package or create a fragile cross-package import. The tasks should specify which approach to take: extract to a shared module under `src/houmao/demo/`, import from the CAO demo package directly, or copy with attribution.

> **DECISION: Accept.**
> Rationale: The current artifacts say the new demo should reuse the CAO demo's recipe-selection behavior, but they do not yet choose a reuse mechanism. That ambiguity should be removed before implementation so the code does not drift into either a fragile cross-package import or duplicated selector logic.

4. **`tracked_agent_id` discovery after registration is unresolved.** Design Decision 3 says: "The demo state will then persist both runtime-owned identifiers and server-owned lookup coordinates, including `tracked_agent_id` once it is discoverable." However, `HoumaoRegisterLaunchResponse` (in `src/houmao/server/models.py:400-405`) only returns `session_name` and `terminal_id`, not `tracked_agent_id`. The managed-agent resolution logic in the server (`_resolve_managed_agent_ref`) accepts `session_name` as a valid `agent_ref`, so the demo can use `session_name` directly for all `{agent_ref}` path parameters. But the design implies a separate `tracked_agent_id` exists and must be discovered. The tasks (2.4) should clarify whether to use `session_name` as the stable `agent_ref` or to query `GET /houmao/agents` to discover a server-assigned `tracked_agent_id`.

> **DECISION: Refine.**
> Rationale: The reviewer correctly identified an unnecessary and underspecified discovery step. The v1 decision is to persist `session_name` as the stable `agent_ref` immediately after registration and treat `tracked_agent_id` as optional later enrichment rather than a startup requirement.

5. **Demo-owned houmao-server lifecycle management lacks implementation detail.** Task 2.1 bundles three complex responsibilities: run-root provisioning, houmao-server lifecycle, and working-tree setup. The houmao-server lifecycle alone is significantly more complex than the CAO demo's `cao_server.py` (which targets a single-process CAO server at a fixed port). A houmao-server must:
   - Launch its own child CAO process
   - Select a free loopback port (per Design Decision 2)
   - Configure its server root with registration directories
   - Wait for health check readiness before proceeding

   The existing `houmao_server_dual_shadow_watch` demo manages its own server via `houmao-srv-ctrl` CLI. The design's Non-Goals say "Require `houmao-srv-ctrl` as the interactive operator surface" — but that refers to operator interaction, not server lifecycle. Task 2.1 should specify whether to use `houmao-srv-ctrl` for lifecycle management (as the shadow-watch demo does) or build a custom launcher module. The `config/` directory has launcher configs and `src/houmao/server/launcher.py` exists for this purpose.

> **DECISION: Refine.**
> Rationale: The lifecycle area is the highest-risk part of this demo and needs more explicit implementation guidance, but the follow-up should point to the actual repo precedent: a dedicated helper module that launches `houmao-server` as a subprocess, captures logs, and waits for health readiness. The review's reference to `src/houmao/server/launcher.py` does not match the current tree.

6. **Task 2.3 underspecifies the `start-session` command shape.** The task says "Implement `houmao-cli start-session --backend houmao_server_rest` startup" but doesn't enumerate the required flags. Looking at the CLI definition in `src/houmao/agents/realm_controller/cli.py:130-216`, `start-session` requires `--brain-manifest`, `--role`, `--workdir` at minimum, plus optional `--agent-name`, `--mailbox-mode`, etc. For `houmao_server_rest` specifically, the backend needs `api_base_url` (from the demo-owned server) passed through launch configuration. The session manifest's `HoumaoServerSectionV1` (in `boundary_models.py:334+`) captures `api_base_url, session_name, terminal_id, parsing_mode` — these are the primary sources for registration and follow-up commands. The task should reference these fields as the data contract between startup and registration.

> **DECISION: Refine.**
> Rationale: Task 2.3 does not need to spell out every CLI flag exhaustively, but it should name the required launch inputs and the persisted manifest contract that the next step depends on. The later artifact revision should anchor this task around `--brain-manifest`, `--role`, `--workdir`, `--backend houmao_server_rest`, the server base URL input, and the persisted `houmao_server` manifest section.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the new demo import `brain_recipes.py` from the CAO demo package, extract it to a shared module, or copy it?

- Decision: `Deferrable`

#### Why this matters

- The CAO demo's `resolve_demo_brain_recipe()` and `ResolvedDemoBrainRecipe` are 195 lines of well-tested recipe resolution logic with selector normalization, basename ambiguity detection, and `default_agent_name` validation. The new demo needs identical logic. The choice affects maintenance burden and cross-package coupling.

#### Examples

- If the new demo imports from `cao_interactive_demo.brain_recipes`, any refactoring of the CAO demo could break the new demo.
- If the logic is copied, bug fixes must be applied in two places.
- If extracted to a shared module (e.g., `src/houmao/demo/_shared/brain_recipes.py`), both demos import from a stable location.

#### Why this question arises in code

- `src/houmao/demo/cao_interactive_demo/brain_recipes.py` defines the full resolution pipeline
- The migration plan step 2 says "Reuse the tracked brain-recipe selection model" without specifying the reuse mechanism
- No shared utility module currently exists under `src/houmao/demo/`

#### Options

**Option A: Extract to a shared module under `src/houmao/demo/`**

- Code impact:
  - Create `src/houmao/demo/_shared/brain_recipes.py` (or `src/houmao/demo/common/brain_recipes.py`)
  - Update `cao_interactive_demo` to import from the shared module
  - New demo imports from the shared module
- User-facing difference:
  - None; operator workflow is unchanged
- Pros:
  - Single source of truth for recipe resolution
  - Both demos stay in sync on bug fixes and feature additions
- Cons:
  - Requires a small refactoring PR for the CAO demo
  - Introduces a shared module dependency in a directory that currently has no shared utilities

**Option B: Import directly from `cao_interactive_demo.brain_recipes`**

- Code impact:
  - New demo adds `from houmao.demo.cao_interactive_demo.brain_recipes import resolve_demo_brain_recipe, ResolvedDemoBrainRecipe`
- User-facing difference:
  - None
- Pros:
  - Zero code duplication, no refactoring needed
  - Fast to implement
- Cons:
  - Creates a runtime dependency from the new demo to the CAO demo package
  - Refactoring or removing the CAO demo could break the new demo

**Option C: Copy with attribution**

- Code impact:
  - Copy `brain_recipes.py` into the new demo package
- User-facing difference:
  - None
- Pros:
  - Full isolation between demo packages
  - No cross-package coupling
- Cons:
  - Two copies to maintain; divergence risk over time

> **PROPOSED: Option A — extract to a shared module under `src/houmao/demo/`.**
> Rationale: The CAO demo and the new demo have identical recipe resolution requirements (selector normalization, ambiguity detection, `default_agent_name` validation). A shared module eliminates duplication while avoiding fragile cross-package imports. The project is under active development where both demos will likely evolve together, making a single source of truth more valuable than isolation. The refactoring cost is low (move + re-import) and the existing test coverage in `tests/unit/demo/test_cao_interactive_demo.py` can be redirected to the shared module.
>
> **DECISION: Accept PROPOSED with refinement.**
> Rationale: The correct direction is a shared helper under `src/houmao/demo/` so both demos consume the same selector normalization and ambiguity logic without creating a permanent dependency from the new demo onto the CAO demo package.

### Q2) How should the demo obtain a stable `agent_ref` for managed-agent routes after registration?

- Decision: `Blocking`

#### Why this matters

- Every post-launch command (`inspect`, `send-turn`, `interrupt`) needs an `{agent_ref}` path parameter. The design says to persist `tracked_agent_id` "once it is discoverable", but registration doesn't return it. Getting this wrong means follow-up commands fail.

#### Examples

- If the demo uses `session_name` as `agent_ref`, all routes work immediately because server-side `_resolve_managed_agent_ref()` accepts it.
- If the demo queries `GET /houmao/agents` to find `tracked_agent_id`, there's a timing window where the agent may not appear immediately after registration.

#### Why this question arises in code

- `src/houmao/server/models.py:400-405`: `HoumaoRegisterLaunchResponse` returns only `session_name` and `terminal_id`
- `src/houmao/server/service.py`: `_resolve_managed_agent_ref()` accepts multiple ref formats including `session_name`
- Design Decision 3 says "persist … `tracked_agent_id` once it is discoverable" without specifying discovery

#### Options

**Option A: Use `session_name` directly as `agent_ref` for all managed-agent routes**

- Code impact:
  - Demo state stores `agent_ref = session_name` immediately after registration
  - No additional discovery query needed
  - All `{agent_ref}` path params use this value
- User-facing difference:
  - Simpler state model, no discovery delay
- Pros:
  - Works immediately after registration
  - `_resolve_managed_agent_ref()` in `service.py` already supports session_name lookup
  - Matches what the registration response actually returns
- Cons:
  - If the server ever changes resolution semantics, this could break
  - `session_name` may be less stable than a server-assigned ID

**Option B: Query `GET /houmao/agents` after registration to discover `tracked_agent_id`**

- Code impact:
  - Add a post-registration discovery step with retry/backoff
  - Filter the agent list by `session_name` to find `tracked_agent_id`
  - Persist `tracked_agent_id` in state
- User-facing difference:
  - Slight startup delay for discovery
- Pros:
  - Uses a server-assigned stable identifier
  - More future-proof if resolution semantics change
- Cons:
  - Adds timing sensitivity and retry logic
  - Increases startup complexity
  - The registration response should ideally return this ID if it's needed

> **PROPOSED: Option A — use `session_name` directly as `agent_ref`.**
> Rationale: The server's `_resolve_managed_agent_ref()` in `service.py` already accepts `session_name` as a valid agent reference and resolves it through the managed-agent registry. The `HoumaoRegisterLaunchResponse` returns `session_name` as a first-class identifier. Adding a post-registration discovery step introduces timing sensitivity and retry complexity for no immediate benefit. The design can note that `session_name` serves as the stable `agent_ref` in v1, and a dedicated `tracked_agent_id` can be adopted later if the server adds it to the registration response.
>
> **DECISION: Accept PROPOSED.**
> Rationale: `session_name` is already returned by registration and already resolves through the managed-agent alias logic, so it gives the demo an immediate stable v1 reference without adding a discovery race. `tracked_agent_id` can stay optional future enrichment.

### Q3) How should the demo launch and manage its own houmao-server instance?

- Decision: `Blocking`

#### Why this matters

- The demo-owned server is the foundation for all post-launch interaction. Getting the lifecycle wrong (port selection, child-process management, health readiness, teardown) will make the entire demo unreliable.

#### Examples

- The `houmao_server_dual_shadow_watch` demo manages its own server via `houmao-srv-ctrl` subprocess calls.
- The CAO demo manages a CAO server via `cao_server.py` with a fixed port and simple health check.
- `src/houmao/server/launcher.py` provides programmatic server launch utilities.

#### Why this question arises in code

- Task 2.1 says "demo-owned `houmao-server` lifecycle" without specifying the mechanism
- `src/houmao/server/launcher.py` exists but its API and suitability for demo use is unspecified in the design
- `houmao-srv-ctrl` CLI wraps server lifecycle management and is used by the shadow-watch demo
- The CAO demo's `cao_server.py` pattern (dedicated module, health check, fixed port) is the closest template

#### Options

**Option A: Use `houmao-srv-ctrl` CLI via subprocess (like shadow-watch demo)**

- Code impact:
  - Call `houmao-srv-ctrl start --port <free-port>` as a subprocess
  - Poll health endpoint until ready
  - Call `houmao-srv-ctrl stop` during teardown
- User-facing difference:
  - Requires `houmao-srv-ctrl` to be installed and on PATH
- Pros:
  - Proven pattern from the shadow-watch demo
  - Handles child-CAO process management internally
  - CLI manages PID tracking and graceful shutdown
- Cons:
  - Adds `houmao-srv-ctrl` as a runtime dependency for the demo
  - Less control over server configuration details
  - Design Non-Goals say not to "require `houmao-srv-ctrl` as the interactive operator surface" (though this refers to operator interaction, not lifecycle)

**Option B: Use `src/houmao/server/launcher.py` programmatically**

- Code impact:
  - Import launcher utilities and call them from a demo `houmao_server.py` module
  - Configure port, server root, and child-CAO settings programmatically
  - Manage the server process lifecycle within the demo
- User-facing difference:
  - No external CLI dependency; self-contained demo
- Pros:
  - Full control over configuration and lifecycle
  - Self-contained demo package
  - Can be tested with mocks like the CAO demo's `cao_server.py`
- Cons:
  - More code to write and maintain
  - Must handle child-CAO process management directly
  - May duplicate launcher logic

**Option C: Create a dedicated `houmao_server.py` module mirroring `cao_server.py` pattern, using subprocess `houmao-server` launch**

- Code impact:
  - New module `src/houmao/demo/houmao_server_interactive_full_pipeline_demo/houmao_server.py`
  - Launch `houmao-server` as a subprocess with dynamic port
  - Health-check loop before returning
  - Teardown via process termination
- User-facing difference:
  - Requires `houmao-server` entrypoint on PATH
- Pros:
  - Mirrors the established `cao_server.py` demo pattern closely
  - Clean separation between demo logic and server internals
  - Easy to test with mock subprocess runner
- Cons:
  - Must select free port and configure via CLI flags or env vars
  - Must handle child-CAO lifecycle indirectly through houmao-server's own startup

> **PROPOSED: Option C — create a dedicated `houmao_server.py` module using subprocess launch, mirroring the `cao_server.py` pattern.**
> Rationale: The CAO demo's `cao_server.py` is the most natural structural template: a dedicated demo module that launches the server as a subprocess, waits for health readiness, and provides a teardown function. This keeps the new demo self-contained without importing server internals. Using the `houmao-server` entrypoint (defined in `pyproject.toml` as `houmao-server = "houmao.server.app:cli"` or similar) as a subprocess is consistent with how the demo already uses `houmao-cli` as a subprocess for brain build and session launch. Port selection can use `socket.bind(("127.0.0.1", 0))` to find a free port, matching the design's "loopback api_base_url selected and persisted for that run" requirement.
>
> **DECISION: Refine PROPOSED.**
> Rationale: The structural choice is correct, but the concrete repo precedent is the dual shadow-watch demo's `_start_server_process()` helper, which launches `sys.executable -m houmao.server serve`, captures logs, and polls health. There is no `src/houmao/server/launcher.py` in the current tree, so the artifact revision should pin the lifecycle strategy to a dedicated demo helper that follows that existing subprocess pattern.

### Q4) Should `inspect` surface parsed dialog tail by default or behind an explicit flag?

- Decision: `Deferrable`

#### Why this matters

- This is listed as an open question in the design and affects the default `inspect` output for operators.

#### Examples

- The CAO demo's `inspect_demo()` takes `output_text_tail_chars: int | None = None` — parsed dialog tail is opt-in, not default.
- An operator running `inspect` after a prompt submission may expect to see the assistant's latest response text in the output.

#### Why this question arises in code

- Design open question 1 asks this directly
- The CAO demo precedent is opt-in via `--output-text-tail-chars`
- The spec says inspect "SHALL read managed-agent state" and "When the demo exposes optional parser-derived text… it SHALL read that information through `GET /houmao/terminals/{terminal_id}/state`"

#### Options

**Option A: Opt-in via explicit flag (match CAO demo pattern)**

- Code impact:
  - `inspect` CLI accepts `--output-text-tail-chars N` or `--show-dialog-tail`
  - Default output shows managed-agent state without parsed text
- User-facing difference:
  - Cleaner default output; operator must opt in for text
- Pros:
  - Consistent with CAO demo
  - Avoids parser-dependency in default path
- Cons:
  - New operators may miss that dialog text is available

**Option B: Show parsed dialog tail by default**

- Code impact:
  - `inspect` always queries terminal state and includes parsed text
  - Add `--no-dialog-tail` to suppress
- User-facing difference:
  - Richer default output but potentially noisy
- Pros:
  - More informative for operators
- Cons:
  - Breaks pattern with CAO demo
  - Adds terminal-state dependency to every inspect call

> **PROPOSED: Option A — opt-in via explicit flag, matching the CAO demo's `output_text_tail_chars` pattern.**
> Rationale: The spec uses the word "optional" for parser-derived text output, and the CAO demo's `inspect_demo()` in `commands.py` takes `output_text_tail_chars: int | None = None` as an opt-in parameter. Following the established pattern keeps the default inspect output clean and consistent across demo packs. Operators who want dialog text can pass the flag.
>
> **DECISION: Accept PROPOSED.**
> Rationale: The current artifacts should resolve this now instead of carrying it as an open question because the CAO demo already establishes the intended operator pattern: default inspect output stays focused on state, while parsed dialog text is available only when explicitly requested.
