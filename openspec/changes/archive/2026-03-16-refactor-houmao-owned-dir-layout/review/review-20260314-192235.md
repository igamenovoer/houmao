## Summary

- **Well-structured four-zone model**: The separation of registry / runtime / mailbox / job-dir responsibilities is clean, well-motivated, and addresses real operational pain from the current layout.
- **Thorough scenario coverage**: Every spec requirement includes concrete WHEN/THEN scenarios with specific paths and env-var combinations, making testability clear.
- **SHA-256 → MD5 identity hash conflict is a blocking concern**: The existing codebase and main `agent-discovery-registry` spec use SHA-256 for `agent_key`, but this change introduces `agent_id` derived via MD5. The relationship between these two identity surfaces is undefined, creating a migration gap and potential dual-hash confusion.
- **CAO launcher path renaming (`cao-server/` → `cao_servers/`) is under-specified**: The naming convention change is present in the design layout but not called out as a deliberate breaking rename, and the task list does not include a migration or compatibility note for existing launcher artifacts.
- **Task overlap between 1.2 and 2.1 risks implementation confusion**: Both tasks touch runtime-owned metadata, `agent_id` derivation, and session-root consumers with unclear division of responsibility.
- **`.houmao/` in working directories needs `.gitignore` guidance**: The design acknowledges the risk but defers any convention, which may cause friction in practice.

## Findings

### Must Fix / Blocking

#### F1. `agent_id` (MD5) vs `agent_key` (SHA-256) identity hash conflict

The existing main spec `openspec/specs/agent-discovery-registry/spec.md` and production code (`src/houmao/agents/realm_controller/registry_models.py:207-211`) define `agent_key` as `sha256(canonical_name).hexdigest()`. The `LiveAgentRegistryRecordV1` model validates this invariant at line 184-186:

```python
expected_key = derive_agent_key(self.agent_name)
if self.agent_key != expected_key:
    raise ValueError("agent_key must equal sha256(canonical agent_name).hexdigest()")
```

This change introduces a new `agent_id` concept derived as `md5(canonical_agent_name).hexdigest()` and requires that Houmao-owned directories and registry publication key by `agent_id` instead of `agent_key`.

**Problems:**
1. The delta spec for `agent-discovery-registry` says registry directories are keyed by `agent_id` (MD5), but the main spec and code enforce `agent_key` (SHA-256). The transition from one to the other is not addressed.
2. Two different hash functions for the same conceptual purpose (stable agent identifier derivation) introduces unnecessary complexity. The design does not motivate why MD5 was chosen over reusing SHA-256.
3. The `LiveAgentRegistryRecordV1` schema (`src/houmao/agents/realm_controller/schemas/live_agent_registry_record.v1.schema.json`, required field `agent_key`) would need to either evolve to a v2 schema with `agent_id` or carry both fields during transition. Neither path is specified.
4. The `registry_storage.py` functions `record_path_for_agent_name()` and `derive_agent_key()` (lines 87, 207-211) currently use SHA-256 for path derivation. Switching the directory key to MD5-derived `agent_id` will orphan existing registry records unless a migration or dual-lookup is defined.

**Recommendation:** Either (a) use SHA-256 consistently for `agent_id` (renaming the concept but reusing the hash), or (b) explicitly specify the registry schema version bump, migration path for existing records, and the coexistence strategy for `agent_key` vs `agent_id`.

> **DECISION: Accept this finding as valid against the current implementation, but not the reviewer's preferred remedy.**
> Rationale: The current code and current main spec still unquestionably implement SHA-256 `agent_key` as the live registry identity surface, so this remains a real blocking gap between implementation and the planned change. However, after revising the change artifacts, the chosen direction is no longer "reuse SHA-256 under a new name." The planned change now explicitly retires registry-specific `agent_key`, adopts cross-module `agent_id`, keeps `md5(canonical agent name)` only as the initial bootstrap when no explicit or persisted `agent_id` exists, and requires persisted `agent_id` to win afterward. So the finding stands, but the selected fix is: schema and code move fully to `agent_id`, not a SHA-256 rename and not a dual-hash coexistence model.

#### F2. Missing registry schema migration plan

The delta spec `specs/agent-discovery-registry/spec.md` replaces `<agent-key>` directory naming with `<agent-id>` in `live_agents/<agent-id>/record.json`, but:
- No schema version bump for `LiveAgentRegistryRecordV1` is mentioned.
- No new field `agent_id` is added to the record schema.
- No dual-lookup (check both old SHA-256 key and new MD5 key) is specified for backward compatibility.
- The existing validation at `registry_models.py:184-186` would reject records keyed by an MD5-derived path.

The design's own Decision 7 ("Treat this as a default-path migration, not as an in-place data migration") should extend to registry records: specify whether old `agent_key`-keyed records are abandoned, coexist, or are re-published under the new key on next session start.

> **DECISION: Accept this finding as valid against the current implementation, and choose a schema bump with no backward-compatibility path for legacy registry directories.**
> Rationale: The current implementation still only understands `LiveAgentRegistryRecordV1`, still requires `agent_key`, and still derives registry paths from the old SHA-256 key, so this is a genuine blocking implementation gap. The revised change now resolves the design ambiguity: the registry contract must bump to the next schema version, replace `agent_key` with `agent_id`, and update readers and writers accordingly. Old `agent_key`-keyed directories are not part of a migration or coexistence strategy in this repository; they become useless legacy on-disk state after cutover and users may clean them up manually. So the "must fix" part is the explicit schema/versioned contract update, not any compatibility bridge.

## Design Alignment With Existing System

### Existing patterns observed

- **Registry root resolution** (`registry_storage.py:50-63`): Already uses `platformdirs`-aware `~/.houmao/registry` with `AGENTSYS_GLOBAL_REGISTRY_DIR` env-var override. This is the reference pattern the design extends to runtime and mailbox roots.
- **Agent identity** (`registry_models.py:207-211`, `agent_identity.py`): SHA-256 `agent_key` derived from canonical `AGENTSYS-<name>`. Used for registry directory keying and record validation.
- **Build-path bucketing** (`brain_builder.py:471-473`): `runtime_root / "homes" / request.tool` and `runtime_root / "manifests" / request.tool`. Design removes the tool intermediate directory.
- **Mailbox root default** (`mailbox_runtime_support.py:196-197`): `runtime_root.resolve() / "mailbox"`. Design moves to `~/.houmao/mailbox`.
- **CAO artifact paths** (`server_launcher.py:388`): `runtime_root / "cao-server" / f"{host}-{port}"` flat structure. Design restructures to `cao_servers/<host>-<port>/launcher/` + `/home/`.
- **Session root derivation** (`manifest.py:57-60`): `runtime_root / "sessions" / backend / session_id`. Design preserves this pattern under the new runtime root.
- **Env-var conventions**: Consistently prefixed with `AGENTSYS_`. The design follows this convention for new env vars.

### Proposed design alignment assessment

**Aligns well:**
- The four-zone model naturally extends the existing registry-root pattern to runtime and mailbox. The override precedence (explicit > env-var > default) matches the established `AGENTSYS_GLOBAL_REGISTRY_DIR` pattern.
- Flattening build-state paths (removing tool bucketing) is consistent with the design's goal of metadata-based association, and the existing `home_id` format already embeds the tool name.
- The per-agent job-dir pattern (`<working-directory>/.houmao/jobs/<session-id>/`) is new but follows the project's convention of session-id-keyed directories.
- The `AGENTSYS_JOB_DIR` env-var binding follows the established pattern of runtime-published session bindings like `AGENTSYS_MANIFEST_PATH` and `AGENTSYS_MAILBOX_FS_ROOT`.

**Mismatches requiring resolution:**
- **Hash function divergence**: The design introduces MD5 for `agent_id` while the existing system uses SHA-256 for `agent_key`. Both are derived from the same canonical agent name. This is the most significant alignment gap (see F1).
- **CAO path structure change**: The design's `cao_servers/<host>-<port>/launcher/` + `home/` split is a structural departure from the current flat `cao-server/<host>-<port>/` layout. The existing `CaoServerRuntimeArtifacts` dataclass (`server_launcher.py:365-395`) would need significant restructuring.
- **Missing `agent_name`/`agent_id` in session manifest**: The current `SessionManifestPayloadV2` does not carry agent identity fields at all. Adding them touches the manifest schema, which is validated against a JSON Schema file (`src/houmao/agents/realm_controller/schemas/`). This is a non-trivial schema evolution (see F7).

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should `agent_id` reuse SHA-256 (like existing `agent_key`) or introduce MD5 as a separate hash?

- Decision: `Blocking`

#### Why this matters

- The entire registry keying, directory naming, and identity association model pivots on this choice. A wrong answer creates a long-lived dual-hash surface that every consumer must handle.

#### Examples

- Agent `AGENTSYS-gpu` today → registry directory `live_agents/<sha256-hex>/record.json` with `agent_key` field.
- Under the proposal → registry directory `live_agents/<md5-hex>/record.json` with `agent_id` field.
- If both coexist: two different directory names for the same canonical agent, requiring dual-lookup or migration.

#### Why this question arises in code

- `registry_models.py:207-211` defines `derive_agent_key()` using SHA-256; the Pydantic validator at line 184-186 enforces `agent_key == sha256(name)`.
- The delta `agent-discovery-registry` spec says `agent_id` uses MD5 and replaces `agent_key` as the directory key.
- No bridge or schema bump is specified.

```python
# Current: registry_models.py:207-211
def derive_agent_key(agent_name: str) -> str:
    canonical_name = canonicalize_registry_agent_name(agent_name)
    return hashlib.sha256(canonical_name.encode("utf-8")).hexdigest()

# Proposed: md5 variant (not yet in code)
def derive_agent_id(agent_name: str) -> str:
    canonical_name = canonicalize_registry_agent_name(agent_name)
    return hashlib.md5(canonical_name.encode("utf-8")).hexdigest()
```

#### Options

**Option A: Reuse SHA-256 for `agent_id`, rename `agent_key` → `agent_id` across the board**

- Code impact:
  - Rename `derive_agent_key()` → `derive_agent_id()` in `registry_models.py`.
  - Update `LiveAgentRegistryRecordV1` field `agent_key` → `agent_id` (schema version bump to v2).
  - Registry directory paths change only in name (`<agent-key>` → `<agent-id>`) but not in hash value, so existing on-disk records remain valid.
  - All consumers of `agent_key` across `registry_storage.py`, `runtime.py`, `cli.py` rename to `agent_id`.
- User-facing difference:
  - Existing registry records can be re-published with the new field name without changing directory paths.
  - No dual-hash confusion.
- Pros:
  - Zero hash migration; existing directory names stay valid.
  - One hash function across the system.
  - SHA-256 is 64 chars (longer paths) but already accepted everywhere.
- Cons:
  - Schema rename still requires a version bump.
  - MD5 would produce shorter (32-char) directory names, which may be preferred for readability.

**Option B: Use MD5 for `agent_id` as specified, keep `agent_key` (SHA-256) as a legacy field**

- Code impact:
  - Add `derive_agent_id()` using MD5 alongside existing `derive_agent_key()`.
  - Add `agent_id` field to `LiveAgentRegistryRecordV1` (or v2 schema); keep `agent_key` for validation compatibility.
  - Registry directories change from `<sha256-hex>` to `<md5-hex>`, orphaning existing records.
  - Dual-hash functions in the identity layer.
- User-facing difference:
  - Existing registry records become orphaned (different directory names).
  - Operators see shorter directory names (32-char MD5 vs 64-char SHA-256).
- Pros:
  - Shorter, more readable directory names.
  - Clean separation between old `agent_key` (registry v1 era) and new `agent_id` (four-zone era).
- Cons:
  - Two hash functions for the same input, increasing cognitive load.
  - Orphans existing registry state; needs explicit "old records are abandoned" statement or migration.
  - MD5 is cryptographically broken (not security-relevant here, but invites questions).

> **PROPOSED: Option A — reuse SHA-256 for `agent_id`, rename `agent_key` → `agent_id`.**
> Rationale: The existing codebase already uses SHA-256 consistently for agent identity derivation (`registry_models.py:211`, validated at `registry_models.py:184-186`). Reusing the same hash under a new name preserves existing on-disk registry directories without migration, avoids introducing a second hash function, and keeps the one-hash-one-purpose invariant clean. The 64-char hex string is already accepted in all path consumers. The schema rename (`agent_key` → `agent_id`) can ride the same registry schema v2 bump that adding `agent_id` to the record would require anyway.
>
> **DECISION: Reject PROPOSED Option A; use MD5 only as the initial bootstrap for `agent_id`, and retire `agent_key` rather than carrying both surfaces.**
> Rationale: The revised change no longer treats `agent_id` as "the same hash under a new label." `agent_id` is now the cross-module, persisted agent identity that must survive later build/start/resume flows and may survive rename-like changes. The initial bootstrap remains `md5(canonical agent name)` only when no explicit or previously persisted `agent_id` exists; after that, persisted `agent_id` wins over recomputation. That means the old SHA-256 `agent_key` is being replaced semantically as well as nominally, not merely renamed. We do not want a long-lived dual-hash model, so the chosen path is: `agent_key` is retired, `agent_id` becomes authoritative, and registry / manifest schemas are bumped accordingly. The migration cost is accepted as part of this breaking refactor.

### Q2) How should existing `agent_key`-keyed registry directories coexist with new `agent_id`-keyed directories?

- Decision: `Blocking`

#### Why this matters

- The shared registry is a cross-session, cross-runtime discovery surface. If the directory key changes, old published records become invisible to new lookups unless a transition strategy is defined.

#### Examples

- Agent `AGENTSYS-gpu` published before upgrade: `~/.houmao/registry/live_agents/<sha256>/record.json`.
- Same agent published after upgrade: `~/.houmao/registry/live_agents/<new-key>/record.json`.
- A `list-agents` query after upgrade might miss the old record or return duplicates.

#### Why this question arises in code

- `registry_storage.py:87` derives the directory path from `derive_agent_key(canonical_name)`.
- `registry_storage.py` scan/prune functions iterate `live_agents/` children by directory name.
- If the hash changes, the old directory is invisible to the new derivation function.

#### Options

**Option A: Treat old records as abandoned (no migration)**

- Code impact:
  - Old `<sha256>` directories remain on disk but are ignored by new code.
  - `prune_expired_agents()` can optionally be taught to clean up directories that don't match the new key derivation.
- User-facing difference:
  - After upgrade, previously published agents appear "gone" until re-published.
  - Stale directories accumulate until manually cleaned.
- Pros:
  - Simplest implementation; no dual-lookup.
- Cons:
  - Brief window of "phantom missing agents" in multi-agent environments during rolling upgrades.

**Option B: Re-publish on next session start/resume (lazy migration)**

- Code impact:
  - `start-session` and `resume` paths re-derive the new key and publish a fresh record.
  - Old records expire naturally via 24-hour lease TTL.
- User-facing difference:
  - Agents reappear in registry within one session start cycle.
  - 24-hour window where both old and new records may coexist.
- Pros:
  - No explicit migration command needed; natural lifecycle handles it.
  - Fits the design's "default-path migration, not in-place data migration" philosophy.
- Cons:
  - Agents that are not restarted within 24 hours disappear from discovery.

> **PROPOSED: Option B — lazy re-publication on next session start; old records expire via lease TTL.**
> Rationale: The design already declares this a default-path migration (design.md Decision 7). The existing 24-hour lease TTL in `registry_models.py` naturally expires stale records. Session start already calls the publication path (`runtime.py:1476`), so re-deriving under a new key is a minimal code change. This avoids an explicit migration command while keeping the registry eventually consistent. Agents that are never restarted are already expected to expire.
>
> **DECISION: Reject PROPOSED Option B; choose Option A with no backward-compat handling.**
> Rationale: We are not treating old registry directories as a migration concern in this repository. This change is a breaking identity cutover from registry-scoped `agent_key` to cross-module `agent_id`, and old `agent_key`-keyed registry directories simply become useless legacy state after upgrade. New sessions will publish under the new `agent_id`-keyed layout as part of normal runtime behavior, but that is not considered a compatibility bridge or migration strategy. We are not adding dual-lookup, TTL-based coexistence handling, or special cleanup logic here; users can remove old registry directories manually.

### Q3) Should the session manifest schema bump to v3 to carry `agent_name` and `agent_id`?

- Decision: `Deferrable`

#### Why this matters

- The `brain-launch-runtime` delta spec requires persisted runtime metadata to carry both canonical agent name and `agent_id`, but the current v2 manifest has neither as a top-level field.

#### Examples

- Current v2 manifest payload (`manifest.py:97-108`): top-level fields are `schema_version`, `backend`, `tool`, `role_name`, `working_directory`, etc. No `agent_name` or `agent_id`.
- After change: something like `{"schema_version": 3, "agent_name": "AGENTSYS-gpu", "agent_id": "<hash>", ...}`.

#### Why this question arises in code

- `manifest.py:24` defines `SESSION_MANIFEST_SCHEMA_VERSION = 2`.
- `manifest.py:97-108` builds the payload without agent identity fields.
- The JSON schema at `schemas/live_agent_registry_record.v1.schema.json` would need a companion session manifest schema update.

#### Options

**Option A: Bump to v3 with required `agent_name` + `agent_id` fields**

- Code impact:
  - Increment `SESSION_MANIFEST_SCHEMA_VERSION` to 3.
  - Add `agent_name` and `agent_id` to manifest payload builder and JSON schema.
  - Add a v2→v3 forward-compat check in `load_session_manifest()` for old manifests.
- User-facing difference:
  - Resume of old v2 manifests must either fail gracefully or fill defaults.
- Pros:
  - Clean, explicit schema evolution.
  - Machine-readable version for tooling.
- Cons:
  - Adds resume-compatibility burden.

**Option B: Store `agent_name` + `agent_id` inside `backend_state` without bumping schema version**

- Code impact:
  - Add fields to `backend_state` dict; no schema version change.
  - Reader code checks for presence but tolerates absence.
- User-facing difference:
  - Old manifests resume without issue; new manifests carry richer state.
- Pros:
  - No version bump; zero resume breakage.
- Cons:
  - Identity fields hidden in an untyped dict; harder to validate.

> **PROPOSED: Option A — bump to v3 with required `agent_name` and `agent_id`.**
> Rationale: The session manifest is the single durable record for a started session. Agent identity is fundamental enough to warrant first-class top-level fields rather than being buried in the untyped `backend_state` dict. The existing `load_session_manifest()` at `manifest.py:177-205` already validates `schema_version`, so adding a v2 forward-compatibility path (defaulting missing fields from the existing `role_name` or tmux session name) follows the established pattern. This also enables future manifest-based agent discovery without parsing backend-specific state.
>
> **DECISION: Accept Option A, but refine it to include the persisted tmux session name as a first-class manifest field as well.**
> Rationale: Under the revised assumptions, the manifest is now the authoritative durable metadata surface for recovering true agent identity. Because tmux session names are no longer assumed to equal canonical agent names, the manifest needs to persist at least canonical `agent_name`, authoritative `agent_id`, and the actual tmux session name used for that live session. Those belong as typed top-level schema fields, not buried inside `backend_state`. So this should be a manifest schema bump to v3 (or equivalent next version) with explicit forward handling for older v2 manifests during resume/control.

### Q4) Should `stop-session` auto-clean the per-agent job dir, or is cleanup always manual in this version?

- Decision: `Deferrable`

#### Why this matters

- Job dirs under `<working-directory>/.houmao/jobs/<session-id>/` accumulate with each session. Without any cleanup path, operators must manually identify and remove stale job dirs.

#### Examples

- After 20 sessions in `/repo/app`, the tree is `/repo/app/.houmao/jobs/session-...-1/`, `.../session-...-2/`, etc.
- An operator runs `stop-session` and expects the session's scratch space to be cleaned up, but it persists.

#### Why this question arises in code

- The `brain-launch-runtime` delta spec says the job dir "SHALL NOT replace the durable runtime-owned session root" but does not address its lifecycle beyond creation and resume.
- `stop-session` in `runtime.py` handles tmux session teardown but does not currently touch any filesystem cleanup outside the tmux process lifecycle.

#### Options

**Option A: Cleanup is always manual in this version (status quo)**

- Code impact:
  - No changes to `stop-session`.
  - Add a documentation note about manual cleanup.
- User-facing difference:
  - Job dirs accumulate; operators must prune manually.
- Pros:
  - Simplest; no risk of accidental data loss.
- Cons:
  - Disk usage grows silently.

**Option B: `stop-session --clean-job-dir` optional flag**

- Code impact:
  - Add an optional flag to `stop-session` CLI.
  - If set, `rmtree()` the job dir after session teardown.
- User-facing difference:
  - Opt-in cleanup; default behavior unchanged.
- Pros:
  - Gives operators a safe cleanup path without forcing it.
- Cons:
  - Slightly more implementation work; must handle edge cases (job dir in use, permissions).

> **PROPOSED: Option A — cleanup is manual in this version; document the expectation explicitly.**
> Rationale: The design's own open questions section asks this same question, indicating it was intentionally deferred. Adding an optional cleanup flag is straightforward future work. For this change, the priority is establishing correct directory creation and session binding. Documenting that job dirs are NOT auto-cleaned (and suggesting a manual `rm -rf .houmao/jobs/` pattern) is sufficient for operator clarity.
>
> **DECISION: Accept PROPOSED Option A.**
> Rationale: This change is already carrying a large identity and layout refactor. Auto-cleanup would add destructive behavior before we have a clear retention policy or a settled answer about which job-dir contents operators may still care about after stop. For this version, job-dir cleanup remains manual and the docs should state that explicitly.

### Q5) Should the design specify whether `.houmao/` gets an auto-generated `.gitignore`?

- Decision: `Deferrable`

#### Why this matters

- The job-dir is explicitly for destructive scratch (logs, tmp files, outputs). If accidentally committed, it pollutes repos with session-local garbage.

#### Examples

- Developer starts a session in `/repo/app`. Runtime creates `/repo/app/.houmao/jobs/session-.../`. Developer runs `git add -A` and accidentally stages scratch files.

#### Why this question arises in code

- No existing code writes `.gitignore` files during session creation.
- The `.houmao/` directory is new and has no established convention in the project's `.gitignore` patterns.

#### Options

**Option A: Runtime writes `.houmao/.gitignore` containing `*` on first job-dir creation**

- Code impact:
  - One `Path.write_text("*\n")` call in the job-dir creation path.
- User-facing difference:
  - `.houmao/` contents are automatically git-ignored.
- Pros:
  - Zero operator friction; works for any repo.
- Cons:
  - Runtime writes a file outside its "sanctioned" job-dir subdirectory.

**Option B: Document `.houmao/` in project `.gitignore` guidance only**

- Code impact:
  - None; documentation only.
- User-facing difference:
  - Operators must manually add `.houmao/` to `.gitignore`.
- Pros:
  - No surprise file creation.
- Cons:
  - Easy to forget.

> **PROPOSED: Option A — auto-write `.houmao/.gitignore` containing `*` on first creation.**
> Rationale: The runtime already creates the `.houmao/jobs/<session-id>/` directory tree, so writing a sibling `.gitignore` is a trivial addition. This follows the convention used by many tools (e.g., `node_modules/.gitignore`, `.pytest_cache/.gitignore`). The `.gitignore` file is idempotent and harmless if `.houmao/` is already in the project's root `.gitignore`.
>
> **DECISION: Reject PROPOSED Option A for this change; use documentation guidance only.**
> Rationale: Auto-writing `.houmao/.gitignore` with `*` is broader than the current contract and would silently hide any future non-job metadata that may later live under `.houmao/`. It also mutates workspace files in situations where `AGENTSYS_LOCAL_JOBS_DIR` may relocate jobs outside the workspace entirely, making that write unnecessary. For this change, the better scope is documentation guidance: recommend ignoring `.houmao/` or at minimum `.houmao/jobs/` when using the default workspace-local job-dir layout, without auto-creating repo files.

## Suggested Next Steps

1) **Keep the change artifacts aligned with the decisions above.** In particular: `agent_key` should be retired in favor of authoritative `agent_id`, no backward-compatibility path for old registry directories should be implied, and tmux session names should be specified as persisted live-session handles rather than inferred agent identities.

2) **Implement manifest schema evolution as a first-class part of the runtime work.** The runtime manifest needs explicit top-level identity fields for canonical `agent_name`, authoritative `agent_id`, and the actual tmux session name used for the live session, with legacy v2 handling for older manifests.

3) **Clarify task boundaries between Task 1.2 and Task 2.1** so it is obvious which task introduces `agent_id` bootstrap / persisted-reuse semantics and which task wires those fields into session manifests, tmux-backed runtime metadata, and registry publication.

4) **Keep the CAO path structure change explicitly breaking.** The proposal/design should continue to call out the `cao-server/` → `cao_servers/` path change and decide whether any launcher stop/status compatibility check for old artifact paths is still desired.

5) **Document manual job-dir cleanup and `.gitignore` guidance.** This version should document that job dirs are not auto-cleaned and should recommend ignoring `.houmao/` or `.houmao/jobs/` when using workspace-local job dirs.

6) **Keep the `AGENTSYS_LOCAL_JOBS_DIR` contract explicit.** The docs/specs should say clearly that this override is intended as a per-launch or per-agent override surface, and that using it relocates that session's job dir outside the default workspace-local `.houmao/jobs/` tree.
