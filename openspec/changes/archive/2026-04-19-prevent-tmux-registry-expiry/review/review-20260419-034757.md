## Summary

- The core direction is sound: current local list/selector paths already resolve through freshness-only registry helpers, and cleanup already owns tmux stale classification through `tmux_session_exists`.
- The plan is close to native implementation because `_build_shared_registry_record_for_controller()` already centralizes TTL selection for tmux-backed runtime-owned records.
- The main risk is semantic drift around "sentinel non-expiring": the existing joined-session sentinel is only 30 days, while the proposed requirement says tmux-backed records should not expire independently of the tmux session lifecycle.
- The artifacts need one clarification around passive server discovery, because that code path already performs eager tmux liveness filtering and therefore does not match a blanket "ordinary discovery is lease-only" statement.
- The non-tmux language is over-broad relative to the current registry model, which only supports `terminal.kind == "tmux"` and currently does not publish records for non-tmux controllers.

## Findings

### Must Fix / Blocking

1. **The artifacts leave the "non-expiring sentinel" under-specified and internally conflicted with the existing 30-day joined-session sentinel.** The design says tmux-backed live-agent records should use a "sentinel non-expiring lease window" and task 1.1 says to use a dedicated sentinel, but the design's open question still asks whether to reuse the joined-session sentinel. In the codebase, `JOINED_REGISTRY_SENTINEL_LEASE_TTL` is `timedelta(days=30)` in `src/houmao/agents/realm_controller/registry_storage.py:27`, and `_build_shared_registry_record_for_controller()` falls back to `DEFAULT_REGISTRY_LEASE_TTL` unless `session_origin == "joined_tmux"` in `src/houmao/agents/realm_controller/runtime.py:3362`. Reusing the 30-day joined sentinel would still make long-lived tmux sessions disappear on day 31, while using an absolute max timestamp without adjusting `_record_expired_beyond_grace()` risks timestamp overflow because cleanup currently adds `grace_period` to `lease_expires_at` in `src/houmao/agents/realm_controller/registry_storage.py:473`.

> **DECISION: Accept.**
> Rationale: The plan must not reuse `JOINED_REGISTRY_SENTINEL_LEASE_TTL` because 30 days is still an expiry boundary unrelated to tmux lifecycle. Revise the artifacts to require a dedicated tmux-backed sentinel TTL and to define it as a long finite operational sentinel, avoiding max-timestamp cleanup overflow concerns.

### Should Fix / Important

2. **The design claims cleanup owns tmux-backed stale classification, but passive discovery already performs eager tmux liveness filtering.** The design says ordinary discovery should remain lease-based and rejects widening discovery paths to inspect local tmux state. That is true for local list/selector helpers such as `_list_registry_records()` in `src/houmao/srv_ctrl/commands/managed_agents.py:2447`, which delegates to `resolve_live_agent_record_by_agent_id()`. It is not true for `RegistryDiscoveryService`, whose `_run_scan()` loads live tmux sessions up front and excludes records whose `terminal.session_name` is absent in `src/houmao/passive_server/discovery.py:176`. Because task 2.1 explicitly names passive discovery tests, the change should say whether passive discovery is an existing exception or whether it should be moved to the lease-only contract.

> **DECISION: Accept.**
> Rationale: Passive discovery is an existing probe-backed index and should be called out as an exception to the no-eager-probe statement. The ordinary lookup/list paths should remain lease-based, while passive discovery should continue to require both lease freshness and live tmux presence.

3. **The non-tmux publication requirement is broader than the current registry contract can exercise.** The delta spec says "Published records for ordinary non-tmux live agents SHALL use the default 24-hour lease TTL in v1," and task 1.2 requires preserving non-tmux publication paths. Current `RegistryTerminalV1.kind` is `Literal["tmux"]` in `src/houmao/agents/realm_controller/registry_models.py:63`, and `_build_shared_registry_record_for_controller()` immediately returns `None` for non-tmux-backed controllers in `src/houmao/agents/realm_controller/runtime.py:3346`. That makes the requirement look mandatory even though there is no representable non-tmux record path today.

> **DECISION: Accept.**
> Rationale: The current registry schema only represents tmux terminal records, so the spec should not introduce mandatory non-tmux publication behavior in this change. Revise the artifacts to scope the change to current tmux-backed publication and to state that existing non-tmux/no-publish behavior remains unchanged.

### Nice To Have

4. **Task 2.3 is useful but a bit too vague to verify without naming the concrete command/helper coverage.** The plan says to verify local managed-agent list and selector paths continue to work without eager tmux probes, and the likely target is `_list_registry_records()` plus `_resolve_local_managed_agent_record_with_miss_context()` in `src/houmao/srv_ctrl/commands/managed_agents.py:390` and `src/houmao/srv_ctrl/commands/managed_agents.py:2447`. Naming those functions or their CLI-facing tests would make completion evidence less subjective.

> **DECISION: Accept.**
> Rationale: The task should name the concrete local list and selector helpers or CLI-facing tests so implementation evidence is objective. This is a task clarity fix, not a design change.

## Design Alignment With Existing System

- Existing pattern(s) observed: `registry_storage.py` owns lease constants, freshness checks, atomic publication, generation conflict checks, and stale cleanup. `runtime.py` owns runtime-controller record construction and already selects between the default TTL and the joined-session sentinel at publication time. `managed_agents.py` local list/selector flows consume registry freshness helpers and do not probe tmux directly. `passive_server/discovery.py` is a separate polling index that currently treats both lease freshness and tmux liveness as discovery requirements.
- Proposed design alignment: Moving the tmux-backed TTL decision into publication-time record construction aligns well with the current layering. A dedicated tmux-specific lease constant in `registry_storage.py`, imported by `runtime.py`, would match the existing `DEFAULT_REGISTRY_LEASE_TTL` and `JOINED_REGISTRY_SENTINEL_LEASE_TTL` pattern while preventing accidental reuse of the joined 30-day value.
- Proposed design mismatch: The artifacts currently overstate lease-only discovery unless passive discovery is explicitly carved out. They also describe non-tmux published records even though the current strict registry schema only models tmux terminals and the runtime publication helper only builds records for `_TMUX_BACKED_BACKENDS`.
- Testability: The requested tests are feasible with existing unit fixtures. Add focused assertions in runtime registry tests for TTL selection, registry storage tests for freshness/cleanup behavior beyond the old 24-hour boundary, passive discovery tests with a live tmux-session set, and local command tests that monkeypatch tmux probes to fail if list/selector accidentally starts probing.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What exact sentinel representation should tmux-backed records use?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- A 30-day sentinel does not satisfy the stated lifecycle contract for long-lived tmux sessions.
- A far-future absolute timestamp can be made to work, but cleanup must avoid arithmetic overflow if the value approaches `datetime.max`.
- Tests need a stable, named invariant rather than an implicit "larger than 24 hours" guess.

#### Examples

- A tmux-backed session published on `2026-04-19` is still alive on `2026-05-20`. If the implementation reuses the current 30-day joined-session TTL, ordinary local list/resolve will hide it despite the proposal's goal.
- A sentinel `9999-12-31T23:59:59Z` record enters cleanup. The current cleanup helper evaluates `lease_expires_at + grace_period < now`, so a max-like timestamp can overflow before tmux liveness is checked.

#### Why this question arises in code

- `src/houmao/agents/realm_controller/registry_storage.py:26` defines the 24-hour default.
- `src/houmao/agents/realm_controller/registry_storage.py:27` defines the joined sentinel as only 30 days.
- `src/houmao/agents/realm_controller/runtime.py:3362` currently chooses between the joined sentinel and default TTL.
- `src/houmao/agents/realm_controller/registry_storage.py:473` performs lease-plus-grace cleanup arithmetic.

```python
lease_ttl = (
    JOINED_REGISTRY_SENTINEL_LEASE_TTL
    if controller.agent_launch_authority is not None
    and controller.agent_launch_authority.session_origin == _JOINED_SESSION_ORIGIN
    else DEFAULT_REGISTRY_LEASE_TTL
)
```

#### Options

**Option A: Add a dedicated finite tmux-backed sentinel TTL constant and define "non-expiring" as effectively non-expiring within supported operator lifetimes.**

- Code impact:
  - Add `TMUX_BACKED_REGISTRY_SENTINEL_LEASE_TTL` in `registry_storage.py`.
  - Change runtime publication to use it for all runtime-owned tmux-backed records that currently receive the default TTL.
  - Keep cleanup arithmetic unchanged as long as the selected TTL is safely below `datetime.max`.
- User-facing difference:
  - Tmux-backed records no longer disappear at 24 hours or 30 days when the chosen sentinel is long enough for supported operations.
- Pros:
  - Minimal code change and aligns with existing TTL-based storage model.
  - Avoids schema changes.
- Cons:
  - The record technically still expires someday, so the spec should avoid implying mathematical infinity.

**Option B: Store a max-like absolute sentinel expiration timestamp and update cleanup arithmetic to avoid overflow.**

- Code impact:
  - Add a helper that writes a sentinel `lease_expires_at` for tmux-backed records.
  - Change cleanup from `lease_expires_at + grace_period < now` to an overflow-safe comparison such as `lease_expires_at < now - grace_period`.
  - Add tests around sentinel cleanup and freshness.
- User-facing difference:
  - Tmux-backed records are effectively non-expiring in persisted state until explicit teardown or tmux-probe cleanup.
- Pros:
  - Best matches the literal "non-expiring lease window" wording.
- Cons:
  - Requires more careful timestamp arithmetic and more edge-case coverage.

> **PROPOSED: Use Option A with a dedicated tmux-backed sentinel TTL constant, and explicitly define it as an effectively non-expiring operational sentinel rather than reusing the joined 30-day value.**
> Rationale: The current code is TTL-oriented (`DEFAULT_REGISTRY_LEASE_TTL` and `JOINED_REGISTRY_SENTINEL_LEASE_TTL` in `registry_storage.py`) and runtime publication already selects a TTL in one place (`runtime.py:3362`). A dedicated constant keeps the change small and native while preventing accidental reuse of the 30-day joined-session value that would recreate the disappearance bug after a longer interval. The implementation should add tests that prove freshness beyond the former 24-hour boundary and at least beyond the current joined 30-day boundary if the chosen sentinel is meant to supersede it.
>
> **DECISION: Accept Option A with refinement: use a dedicated long finite tmux-backed sentinel TTL for all tmux-backed records, including joined sessions.**
> Rationale: Use a new tmux-specific constant, not the existing 30-day joined sentinel and not a max timestamp. A finite operational sentinel such as 100 years keeps existing timestamp arithmetic safe, makes records effectively non-expiring for supported operator lifetimes, and avoids a second cleanup-arithmetic change. Artifact revision should make this exact scope explicit and remove the open question about reusing the joined sentinel.

### Q2) Is passive server discovery an exception to the lease-only ordinary discovery rule?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- The proposal says ordinary discovery remains lease-based and does not add eager tmux probes.
- The implementation already has one discovery surface that eagerly probes tmux every scan.
- Task 2.1 asks for passive discovery tests, so implementers need to know whether to preserve or remove that existing behavior.

#### Examples

- A sentinel-fresh record exists and the tmux session is live. Passive discovery should include it after 24 hours if the task's passive-discovery requirement is retained.
- A sentinel-fresh record exists and tmux is unavailable. Current passive discovery empties the index, while local `houmao-mgr agents list` can still list fresh records without tmux probing.

#### Why this question arises in code

- `src/houmao/passive_server/discovery.py:179` calls `_get_live_tmux_session_names()`.
- `src/houmao/passive_server/discovery.py:201` checks lease freshness.
- `src/houmao/passive_server/discovery.py:203` skips records whose tmux session is not live.
- `src/houmao/srv_ctrl/commands/managed_agents.py:2457` local list delegates to `resolve_live_agent_record_by_agent_id()` and therefore remains lease-based.

#### Options

**Option A: Keep passive server discovery as an existing probe-backed exception and scope "ordinary discovery" to registry storage helpers plus local managed-agent list/selector paths.**

- Code impact:
  - No passive discovery behavior change beyond ensuring sentinel-fresh records pass when the tmux session is live.
  - Add a passive discovery test with `live_sessions` containing the old record's session name.
  - Add local list/selector tests that fail if a tmux probe is introduced.
- User-facing difference:
  - Passive server keeps filtering out dead tmux sessions, while local list/resolve stays registry-first and lease-only.
- Pros:
  - Preserves existing passive-server safety behavior and matches current tests.
- Cons:
  - Requires clear wording so "ordinary discovery" is not read as every registry consumer.

**Option B: Make passive server discovery lease-only too.**

- Code impact:
  - Remove or bypass `_get_live_tmux_session_names()` in `RegistryDiscoveryService._run_scan()`.
  - Update tests that currently assert dead tmux sessions and unavailable tmux produce an empty passive index.
- User-facing difference:
  - Passive server may expose dead tmux-backed records until explicit cleanup runs.
- Pros:
  - A single simpler discovery rule across local and passive registry consumers.
- Cons:
  - Reverses an existing passive-server design and broadens stale-record exposure.

> **PROPOSED: Use Option A and explicitly document passive server discovery as a probe-backed index, not as the ordinary registry lookup path covered by the no-eager-probe rule.**
> Rationale: `RegistryDiscoveryService` already has tmux-live semantics in `passive_server/discovery.py:176`, and tests such as `tests/unit/passive_server/test_discovery.py:226` assert dead tmux sessions are excluded. Local managed-agent list and selector paths already use freshness-only registry helpers, so preserving that boundary matches the stated goal without forcing a passive-server redesign into this change. The spec/tasks should require passive discovery to include sentinel-fresh records only when its existing tmux-live check passes.
>
> **DECISION: Accept Option A.**
> Rationale: Passive discovery should remain a probe-backed index. The "ordinary discovery remains lease-based" rule applies to registry storage helpers and local managed-agent list/selector flows, while passive discovery keeps its existing tmux-liveness filter and should only include sentinel-fresh records when the tmux session is live.

### Q3) Should this change define non-tmux registry behavior now or scope it as a future invariant?

<!-- Classification: mark every question -->
- Decision: `Deferrable`

#### Why this matters

- The spec currently uses SHALL language for non-tmux published records.
- The current strict model cannot represent a non-tmux terminal, and runtime construction returns no registry record for non-tmux controllers.
- Leaving impossible mandatory language makes implementation completion harder to judge.

#### Examples

- `codex_app_server` is a `BackendKind` but not in `_TMUX_BACKED_BACKENDS`; the current runtime record builder returns `None` before constructing a registry record.
- A future non-tmux publisher may need a different `terminal.kind` discriminator or no terminal authority at all, which is outside this change's intended scope.

#### Why this question arises in code

- `src/houmao/agents/realm_controller/registry_models.py:66` fixes `RegistryTerminalV1.kind` to `Literal["tmux"]`.
- `src/houmao/agents/realm_controller/runtime.py:3346` skips non-tmux-backed controllers.
- `src/houmao/agents/realm_controller/runtime.py:240` lists the tmux-backed backend allowlist, and `codex_app_server` is the notable backend outside it.

#### Options

**Option A: Keep this change scoped to current tmux-backed publication and phrase non-tmux behavior as "unchanged if introduced elsewhere."**

- Code impact:
  - No schema expansion.
  - Revise the delta spec wording to avoid mandatory non-tmux publication behavior in v1.
  - Keep task 1.2 focused on ensuring all existing non-tmux/no-publish paths remain unchanged.
- User-facing difference:
  - No new non-tmux registry behavior is promised.
- Pros:
  - Matches current code and the change's stated goal.
- Cons:
  - Future non-tmux registry behavior remains unspecified until a later change.

**Option B: Expand the registry schema and runtime publication model to support non-tmux records now.**

- Code impact:
  - Add terminal kinds or a terminal-optional schema branch.
  - Add publication paths and tests for non-tmux records.
  - Revisit cleanup because tmux liveness probing would not apply.
- User-facing difference:
  - Non-tmux live agents could become discoverable through the shared registry.
- Pros:
  - Makes the non-tmux SHALL language directly implementable.
- Cons:
  - Much larger than the issue-driven tmux expiry fix.

> **PROPOSED: Use Option A and revise the spec/tasks so non-tmux behavior is an unchanged/future invariant, not a new mandatory publication contract.**
> Rationale: The current registry contract is strictly tmux-terminal-shaped (`RegistryTerminalV1.kind: Literal["tmux"]`), and the runtime helper intentionally returns `None` for non-tmux-backed controllers. This change can still protect ordinary bounded lease behavior by not altering any future or external non-tmux publisher assumptions, but adding actual non-tmux records would require schema and cleanup design that is outside the tmux expiry bug. Task 1.2 should therefore verify "no current non-tmux/no-publish path changes" rather than imply testable non-tmux records exist.
>
> **DECISION: Accept Option A.**
> Rationale: Do not define new non-tmux registry behavior in this change. Revise the spec and tasks so they say non-tmux/no-publish behavior is unchanged, rather than requiring a non-tmux record path the current schema cannot represent.
