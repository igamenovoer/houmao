## Summary

- **Solid motivation**: The proposal correctly identifies the root cause — launching agents in a full main-repo worktree with heavyweight GPU roles causes unnecessary repo crawl, slow turns, and diagnostic difficulty. The fix-the-fixture-shape-first approach is architecturally sound.
- **Clean separation of concerns**: The deterministic-automatic vs opt-in-real-agent split is well-reasoned and honest about what each lane validates. This addresses a real credibility gap in the current spec wording.
- **Good reuse of existing patterns**: The inspect surface mirrors the existing `cao_interactive_demo inspect` contract; the fixture family follows the established `tests/fixtures/agents/` structure; the `run_demo.sh` command routing extends the existing wrapper pattern.
- **Path mismatch in design artifacts**: The design references `tests/fixtures/agents/brain-recipes/` and `tests/fixtures/agents/blueprints/` at the top level, but the actual fixture structure nests recipes under `tests/fixtures/agents/brains/brain-recipes/`. This needs reconciliation before implementation.
- **Missing `inspect` command in current wrapper**: The current `run_demo.sh` supports `auto`, `start`, `roundtrip`, `verify`, `stop` but not `inspect`. The specs add `inspect` without explicitly noting this as a new command addition to the wrapper routing logic.
- **Open questions on `.git` initialization and smoke lane placement** are acknowledged but unresolved — these need decisions before implementation can proceed cleanly.

   Blueprints are at `tests/fixtures/agents/blueprints/` (top level), so that path is correct. But brain recipes are under `brains/`. The design, specs, and tasks must use consistent paths that match the actual repo layout to avoid confusion during implementation.

   **Affected artifacts**: `design.md` (Decision 1), `tasks.md` (task 1.2), `specs/runtime-agent-dummy-project-fixtures/spec.md`.

2. **Roles path inconsistency.**
   The design says `tests/fixtures/agents/roles/` but existing roles live at `tests/fixtures/agents/roles/gpu-kernel-coder/system-prompt.md` — this path is correct. However, the `runtime-agent-dummy-project-fixtures` spec says "lightweight mailbox-demo agent definitions separate from heavyweight engineering roles" under `tests/fixtures/agents/roles/` without specifying the naming convention for the new role directory. The implementation should follow the existing `<role-name>/system-prompt.md` pattern (e.g., `roles/mailbox-demo-worker/system-prompt.md`).

### Should Fix / Important

3. **`run_demo.sh` does not currently route an `inspect` command.**
   The current `run_demo.sh` (156 lines) routes `auto`, `start`, `roundtrip`, `verify`, and `stop`. The `mailbox-roundtrip-demo-automation` spec adds `inspect` to the supported command list, and the `mailbox-roundtrip-tutorial-pack` spec references `run_demo.sh inspect --agent <sender|receiver>`. Task 2.2 covers this, but no spec explicitly calls out that the wrapper's command-routing case statement needs extension. This is implicit but should be noted to avoid surprises.

4. **The `tutorial_pack_helpers.py` module is 2,628 lines and growing.**
   Adding inspect logic, dummy-project provisioning, and fixture-selection code to this module risks making it unwieldy. Consider whether the inspect surface should be factored into a separate helper or whether the tutorial pack should delegate to the existing `cao_interactive_demo` inspect infrastructure (which already implements the same contract in `src/houmao/demo/cao_interactive_demo/commands.py:319-380`).

5. **The `demo_parameters.json` schema needs a new field for the dummy-project fixture selector.**
   The current schema (`scripts/demo/mailbox-roundtrip-tutorial-pack/inputs/demo_parameters.json`) defines blueprint paths, agent identities, mailbox addresses, and message files. The design says the pack should "copy a tracked dummy project fixture" — but there is no parameter for which dummy project to copy. A `dummy_project_fixture` field (or equivalent) should be added to the demo parameters schema so the fixture selection is declarative and overridable, matching the existing parameter pattern.

6. **The `test_mailbox_roundtrip_tutorial_pack_live.py` integration test (47.8 KB) dynamically loads helpers.**
   The test uses `importlib.util.spec_from_file_location` to load `tutorial_pack_helpers.py` as a module. When the helper module grows to include dummy-project provisioning and inspect logic, this dynamic loading pattern becomes more fragile. The tasks should note that integration tests need updating to account for the new helper capabilities and any new parameters in `demo_parameters.json`.

7. **Spec scenario coverage gap for `inspect` with `--with-output-text`.**
   The `mailbox-roundtrip-demo-automation` spec defines `inspect` with `--with-output-text <num-tail-chars>` but the corresponding spec does not include a scenario for the `--with-output-text` flag succeeding. There is a scenario for unavailable live state but none for the happy path of output text retrieval. Add a scenario to cover this.

### Nice To Have

8. **Consider a shared inspect protocol between `cao_interactive_demo` and tutorial-pack inspect.**
   The `cao_interactive_demo` inspect surface (`src/houmao/demo/cao_interactive_demo/commands.py:319-380`, `rendering.py:22-63`) already produces the exact fields the tutorial-pack inspect requires (`agent_identity`, `session_name`, `tool`, `tool_state`, `tmux_target`, `tmux_attach_command`, `terminal_id`, `terminal_log_path`, etc.). Rather than reimplementing this in the tutorial pack, the pack could delegate to the same underlying inspection logic. This would reduce duplication and keep the two surfaces in sync.

9. **The fixture guidance document (task 1.3) could include a decision tree diagram.**
   The design distinguishes dummy-project/lightweight-role from repo-worktree/heavyweight-role. A visual decision tree in the fixture guidance would make this easier for new contributors to internalize quickly.

10. **The `expected_report/report.json` snapshot will change.**
    Switching from GPU roles to mailbox-demo blueprints will change the sanitized report contract. Task 2.3 covers updating verification artifacts, but the implementation should run `--snapshot-report` early to establish the new baseline rather than debugging snapshot mismatches late in the process.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the copied dummy project include a pre-built `.git` directory from the fixture, or should the pack initialize a fresh git repository after copying?

- Decision: `Blocking`

#### Why this matters

Codex resolves workspace trust from the nearest `.git` root above the working directory (noted in the design document). If the dummy project lacks a `.git` directory at launch time, Codex will either refuse to operate or fall back to scanning upward for a parent `.git`, which could re-introduce the repo-crawl problem the change is designed to solve.

#### Examples

- If `.git` is pre-built in the fixture and copied, the dummy project is immediately Codex-trusted but the fixture contains binary git objects that may be fragile across git versions.
- If `.git` is initialized fresh after copy, the dummy project gets a clean git history but requires an extra `git init && git add . && git commit` step in the provisioning path, adding latency and complexity.

#### Why this question arises in code

- The tutorial pack provisioning path in `tutorial_pack_helpers.py` creates `<demo-output-dir>/project` and the design says this must be "a standalone git-backed workdir."
- The Codex tool adapter (`tests/fixtures/agents/brains/tool-adapters/codex.yaml`) depends on `.git` presence for workspace trust resolution.
- The current worktree provisioning uses `git worktree add` which inherits `.git` from the parent — the dummy-project path has no such inheritance.

#### Options

**Option A: Fresh `git init` after copy**

- Code impact:
  - Provisioning helper adds `git init`, `git add .`, `git commit -m "init"` after `shutil.copytree()`.
  ```python
  shutil.copytree(fixture_path, project_path)
  subprocess.run(["git", "init"], cwd=project_path)
  subprocess.run(["git", "add", "."], cwd=project_path)
  subprocess.run(["git", "commit", "-m", "init"], cwd=project_path)
  ```
- User-facing difference:
  - Clean git history with one commit, no stale object fragility.
  - Slightly slower provisioning (~100ms for small repos).
- Pros:
  - Portable across git versions.
  - No binary git objects in tracked fixtures.
  - Clean single-commit history is easy to reason about.
- Cons:
  - Adds subprocess calls to provisioning path.
  - Requires git to be available in the test environment (already a given).

**Option B: Pre-built `.git` in tracked fixture**

- Code impact:
  - Fixture directory includes `.git/` with pre-committed objects.
  - Provisioning helper does only `shutil.copytree(fixture_path, project_path)`.
- User-facing difference:
  - Instant git-backed workdir with no extra steps.
  - Fixture may break if git internals change format.
- Pros:
  - Simpler provisioning code.
  - Faster provisioning (no subprocess calls).
- Cons:
  - Binary git objects in tracked fixtures are fragile.
  - `.git` directories in tracked fixtures are unusual and can confuse tooling.
  - Harder to maintain if fixture content changes.

> **PROPOSED: Option A — fresh `git init` after copy.**
> Rationale: The existing test provisioning patterns in `test_mailbox_roundtrip_tutorial_pack_runner.py` already use subprocess calls for setup. Fresh init avoids fragile binary git objects in tracked fixtures and is portable across git versions. The ~100ms overhead is negligible for a demo/test provisioning step. Additionally, tracking `.git/` directories inside `tests/fixtures/` would conflict with the repo's own git and confuse tools like `git status`.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._

### Q2) Should the real-agent smoke entrypoint live only under `tests/manual/`, or also as an env-gated pytest target?

- Decision: `Deferrable`

#### Why this matters

The placement affects discoverability and CI integration. A `tests/manual/` script is explicit but invisible to pytest collection. An env-gated pytest target is discoverable by `pytest --collect-only` but requires careful skip logic to avoid accidental execution in CI.

#### Examples

- A maintainer running `pytest tests/` accidentally triggers a real-agent smoke test that takes 5 minutes and consumes API credits.
- A maintainer looking for smoke test entrypoints doesn't find them because they're in `tests/manual/` which isn't on the standard test discovery path.

#### Why this question arises in code

- The existing `tests/manual/` directory contains `manual_*.py` files that are explicitly excluded from standard pytest runs.
- The existing `tests/integration/demo/` tests use subprocess-based execution of `run_demo.sh`, which is a pattern the smoke lane could follow.
- The task list (3.2) says "opt-in real-agent smoke entrypoint" without specifying the mechanism.

#### Options

**Option A: `tests/manual/` script only**

- Code impact:
  - New file: `tests/manual/manual_mailbox_roundtrip_real_agent_smoke.py`
  - No pytest collection or skip logic needed.
- User-facing difference:
  - Must know to look in `tests/manual/` and run explicitly.
  - Cannot accidentally trigger in CI.
- Pros:
  - Zero risk of accidental execution.
  - Follows existing `tests/manual/manual_*.py` convention.
- Cons:
  - Less discoverable.
  - Cannot be run via `pytest` with a flag.

**Option B: Env-gated pytest target under `tests/integration/demo/`**

- Code impact:
  - New test function with `@pytest.mark.skipunless(os.environ.get("HOUMAO_REAL_AGENT_SMOKE"))` or equivalent.
  - Needs careful skip-by-default to avoid CI accidents.
- User-facing difference:
  - Discoverable via `pytest --collect-only`.
  - Can be triggered by setting an env var: `HOUMAO_REAL_AGENT_SMOKE=1 pytest tests/integration/demo/test_real_agent_smoke.py`.
- Pros:
  - Discoverable and composable with pytest tooling.
  - Consistent with other env-gated test patterns.
- Cons:
  - Risk of accidental execution if skip logic is misconfigured.
  - Adds skip-annotation noise to integration test suite.

**Option C: Both — `tests/manual/` script delegates to a shared helper, and an env-gated pytest target reuses the same helper**

- Code impact:
  - Shared helper function for the smoke workflow.
  - `tests/manual/manual_*.py` calls helper directly.
  - `tests/integration/demo/test_*_smoke.py` calls same helper with env-gate.
- User-facing difference:
  - Two entrypoints, same behavior.
- Pros:
  - Maximum discoverability and safety.
  - No logic duplication.
- Cons:
  - Two entrypoints to maintain (though they share a helper).

> **PROPOSED: Option C — both entrypoints sharing a common helper.**
> Rationale: The existing repo already has both `tests/manual/manual_*.py` scripts and env-gated patterns in integration tests. Providing both entrypoints maximizes discoverability while maintaining the `tests/manual/` convention for explicit invocation. The shared helper avoids logic duplication. The `test_mailbox_roundtrip_tutorial_pack_runner.py` already demonstrates the subprocess-based test pattern that both entrypoints can reuse.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._

### Q3) Should mailbox-demo blueprints include bounded CAO shadow-stall overrides from the start, or defer that to a follow-up?

- Decision: `Deferrable`

#### Why this matters

The design acknowledges that even with smaller fixtures, real-agent smoke may still suffer from provider stall. Shadow-stall overrides in blueprints would provide per-session timeout tuning. Including them now means the smoke lane has tuning knobs from day one; deferring means the initial smoke lane uses default timeouts and stall behavior.

#### Examples

- A real-agent smoke run with Claude stalls for 90 seconds on a mailbox turn. Without per-blueprint stall overrides, the smoke lane uses the global default, which may be too short or too long for narrow mailbox turns.
- With per-blueprint overrides, the mailbox-demo blueprint sets `shadow_stall_timeout_s: 120` and the smoke lane waits appropriately.

#### Why this question arises in code

- The design (Decision 5) says "mailbox-demo blueprints may carry bounded CAO shadow-stall overrides" but frames it as secondary.
- The blueprint schema in `tests/fixtures/agents/blueprints/` does not currently include shadow-stall fields — this would be a schema extension.
- The shadow parser and stall detection logic is in `src/houmao/agents/realm_controller/` and would need to consume blueprint-level overrides.

#### Options

**Option A: Defer to follow-up**

- Code impact:
  - No blueprint schema changes in this change.
  - Smoke lane uses default stall behavior.
- User-facing difference:
  - Simpler initial implementation.
  - May need adjustment after first real-agent smoke runs.
- Pros:
  - Keeps scope bounded.
  - Can tune based on observed real-agent behavior.
- Cons:
  - First smoke runs may hit stall-related failures that require manual investigation.

**Option B: Include bounded overrides now**

- Code impact:
  - Blueprint schema extension for optional `shadow_stall_overrides` field.
  - Shadow parser / stall detection must read blueprint-level overrides.
  - New mailbox-demo blueprints include bounded stall timeout values.
- User-facing difference:
  - Smoke lane has tuned stall behavior from day one.
- Pros:
  - Smoke lane is more likely to succeed on first use.
  - Demonstrates the override pattern for other blueprints.
- Cons:
  - Adds scope to an already sizable change.
  - Chosen stall values may need adjustment anyway.

> **PROPOSED: Option A — defer to follow-up.**
> Rationale: The primary goal of this change is fixing the fixture shape (workdir + roles), not stall tuning. The design explicitly frames stall overrides as secondary (design.md Decision 5: "That tuning is secondary"). Including them now would add blueprint schema changes, shadow parser consumption logic, and chosen timeout values that may need immediate re-tuning. Better to land the fixture shape change, observe real-agent smoke behavior, and then add targeted stall overrides with empirical data.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._

### Q4) Should the tutorial-pack `inspect` surface delegate to `cao_interactive_demo` inspect logic or reimplement independently?

- Decision: `Blocking`

#### Why this matters

The `cao_interactive_demo` module (`src/houmao/demo/cao_interactive_demo/commands.py:319-380`) already implements the exact inspect contract: session metadata, tmux target, terminal log path, tool state with graceful degradation, optional output text tail. Reimplementing this in the tutorial pack's shell/helper layer risks divergence and duplication. Delegating keeps the two surfaces in sync but introduces a dependency from the tutorial pack on the demo module.

#### Examples

- A maintainer adds a new field to `cao_interactive_demo inspect` (e.g., `memory_usage`). If the tutorial pack reimplements inspect independently, the field is missing from `run_demo.sh inspect` until someone remembers to add it there too.
- If the tutorial pack delegates to the demo module's inspect logic, both surfaces stay in sync automatically.

#### Why this question arises in code

- `src/houmao/demo/cao_interactive_demo/commands.py` `inspect_demo()` function (lines 319-380) produces exactly the fields listed in the spec: `agent_identity`, `session_name`, `tool`, `tool_state`, `tmux_target`, `tmux_attach_command`, `terminal_id`, `terminal_log_path`, etc.
- `src/houmao/demo/cao_interactive_demo/rendering.py` (lines 22-63) renders the human-readable version.
- The tutorial pack is shell-based (`run_demo.sh`) delegating to `tutorial_pack_helpers.py`, while the demo module is a Python CLI. The integration path isn't obvious.

#### Options

**Option A: Delegate — tutorial pack inspect calls the demo module's inspect logic**

- Code impact:
  - `tutorial_pack_helpers.py` imports or invokes `cao_interactive_demo` inspect functions.
  - Tutorial pack maps its session metadata (from `control/` artifacts) into the demo module's expected input shape.
  ```python
  from houmao.demo.cao_interactive_demo.commands import inspect_demo
  result = inspect_demo(session_name=..., cao_base_url=..., ...)
  ```
- User-facing difference:
  - Identical output format between `cao-interactive-demo inspect` and `run_demo.sh inspect`.
- Pros:
  - Single source of truth for inspect logic.
  - Automatic field parity.
- Cons:
  - Couples tutorial pack to demo module internals.
  - May need adapter layer for session metadata mapping.

**Option B: Reimplement — tutorial pack builds its own inspect from persisted session state**

- Code impact:
  - New inspect functions in `tutorial_pack_helpers.py` that read `control/` artifacts and query CAO.
  - Independent rendering logic.
- User-facing difference:
  - Potentially different output format from `cao-interactive-demo inspect`.
- Pros:
  - Tutorial pack remains self-contained.
  - No coupling to demo module.
- Cons:
  - Duplicates inspect logic.
  - Risk of field drift between two inspect surfaces.

> **PROPOSED: Option A — delegate to `cao_interactive_demo` inspect logic.**
> Rationale: The `cao_interactive_demo` inspect surface (`commands.py:319-380`, `rendering.py:22-63`) already produces the exact fields required by the spec. The tutorial pack's `tutorial_pack_helpers.py` already imports from `houmao` packages for CAO client operations, so adding a dependency on the demo module's inspect function is architecturally consistent. The adapter layer to map tutorial-pack session metadata to the demo module's input shape is small compared to reimplementing the full inspect + graceful-degradation + rendering logic. Field parity is maintained automatically.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
