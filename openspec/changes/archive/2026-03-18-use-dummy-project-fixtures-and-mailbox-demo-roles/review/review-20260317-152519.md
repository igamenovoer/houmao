## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the copied dummy project include prebuilt `.git` metadata or get a fresh `git init`?

- Decision: `Blocking`

#### Why this matters

- Codex resolves workspace trust from the nearest `.git` root above the working directory. The `.git` shape determines whether the tool treats the directory as a trusted workspace.
- Prebuilt `.git` is faster but may carry fixture-specific commit SHAs into test assertions. Fresh `git init` is cleaner but slower and requires deterministic author/date/message settings for reproducible snapshots.

#### Examples

- If the fixture ships with `.git/`, `shutil.copytree()` copies it and the workspace is immediately git-backed. But commit hashes will differ from any `git init`-based approach, affecting `report.json` sanitization.
- If the fixture ships without `.git/`, the copy helper must run `git init && git add . && git commit -m "init"` with controlled `GIT_AUTHOR_*`/`GIT_COMMITTER_*` env vars to produce stable hashes.

#### Why this question arises in code

- The current `ensure_project_worktree()` at `tutorial_pack_helpers.py:791-833` creates a detached worktree (`git worktree add --detach`). The replacement must produce a git-backed directory without worktree semantics.
- The sanitization rules in `sanitize_report.py:50-77` mask commit hashes and paths. Whether the `.git` is prebuilt or fresh affects which values appear in the raw report and whether existing placeholders suffice.

#### Options

**Option A: Ship prebuilt `.git/` in the tracked fixture**

- Code impact:
  - Copy helper is a simple `shutil.copytree(fixture_path, target_path)`.
  - The `.git/` directory must be committed to the repo (unusual but valid for test fixtures).
- User-facing difference:
  - Faster provisioning (no subprocess calls).
  - Fixture directory is larger on disk due to `.git/` objects.
- Pros:
  - Simplest implementation.
  - No dependency on `git` binary during test provisioning.
- Cons:
  - Committing `.git/` subdirectories is non-standard and may confuse tooling (nested `.git`).
  - Fixture updates require manually rewriting the `.git` state.
  - Risk of `.git/config` carrying host-specific settings.

**Option B: Ship tree files only; `git init` + commit after copy**

- Code impact:
  - Copy helper copies the tree, then runs:
    ```python
    subprocess.run(["git", "init"], cwd=target)
    subprocess.run(["git", "add", "."], cwd=target)
    subprocess.run(["git", "commit", "-m", "init"], cwd=target,
                   env={**os.environ, "GIT_AUTHOR_NAME": "fixture", ...})
    ```
  - Needs controlled env vars for deterministic commit hashes.
- User-facing difference:
  - Slightly slower provisioning (~100ms for small tree).
  - Clean `.git` with no host-specific state.
- Pros:
  - Standard fixture pattern — no nested `.git` in repo.
  - Clean, reproducible `.git` state per run.
  - No risk of host-specific `.git/config` leakage.
- Cons:
  - Depends on `git` binary being available during test/demo provisioning.
  - Commit hashes still vary across git versions unless `GIT_COMMITTER_DATE` and `GIT_AUTHOR_DATE` are pinned.

**Option C: Ship tree files plus a `.git-init.sh` script in the fixture**

- Code impact:
  - Similar to Option B but the init logic is co-located with the fixture.
  - Copy helper calls the fixture-owned init script after copy.
- User-facing difference:
  - Same as Option B but the init contract is visible in the fixture directory.
- Pros:
  - Self-documenting fixture.
  - Init logic can be fixture-specific (e.g., create branches, tags).
- Cons:
  - Extra indirection.
  - Still depends on `git` binary.

> **PROPOSED: Option B — ship tree files only and `git init` after copy.**
> Rationale: The existing codebase already depends on `git` being available (the current `ensure_project_worktree()` calls `git worktree add`). Option B avoids the non-standard pattern of committing `.git/` subdirectories, produces a clean workspace per run, and aligns with the `tests/fixtures/` convention of storing only source content. The slight provisioning cost (~100ms) is negligible compared to agent turn latency. Pin `GIT_AUTHOR_DATE` / `GIT_COMMITTER_DATE` to a fixed epoch value to ensure hash stability across environments.
>
> **DECISION: Accept Option B.**
> Rationale: This matches the repo's current assumption that `git` is available for tutorial-pack provisioning, avoids committing nested `.git/` directories into tracked fixtures, and gives the copied project its own clean trust boundary. The later artifact revision should make the determinism requirements explicit: initialize after copy, add/commit the copied tree, and pin author/committer identity plus timestamps.

  - No pytest infrastructure needed.

> **DECISION: Accept Option A.**
> Rationale: The first real-agent mailbox smoke entrypoint should live under `tests/manual/` as a standalone script. ### Q4) Should mailbox-demo blueprints carry bounded CAO shadow-stall overrides immediately?

- Decision: `Deferrable`

#### Why this matters

- Design open question 3 asks whether mailbox-demo blueprints need stall overrides now or as follow-up tuning.
- The design correctly identifies the fixture shape as the primary fix. But if real-agent smoke turns still stall after the fixture switch, having pre-configured stall tuning avoids a second change cycle.

#### Examples

- A mailbox `mail send` turn with a real Claude session may take 30-60 seconds even with a dummy project if the provider is slow. Without stall tuning, the smoke lane may timeout on the CAO shadow polling interval.
- The deterministic direct-path lane uses fake CLIs and will not be affected by stall tuning.

#### Why this question arises in code

- Stall overrides would be expressed as blueprint-level or recipe-level CAO shadow configuration. The exact mechanism depends on how `cao_parsing_mode` and shadow polling are configured in `start-session` (cli.py:464).
- The current tutorial pack hard-codes `--cao-parsing-mode shadow_only`.

#### Options

**Option A: Include bounded stall overrides in mailbox-demo blueprints from the start**

- Code impact:
  - Add stall-related fields to blueprint or recipe YAML (e.g., `shadow_stall_timeout_seconds: 120`).
  - May require extending the blueprint schema if such fields are not yet supported.
- User-facing difference:
  - Real-agent smoke is more tolerant of slow providers out of the box.
- Pros:
  - Preemptive fix for a known issue.
  - Avoids a follow-up change for stall tuning.
- Cons:
  - May require blueprint/recipe schema changes.
  - Tuning values are guesses until validated with real-agent runs.

**Option B: Defer stall tuning to a follow-up change**

- Code impact:
  - No stall-related changes in this change.
  - Follow-up change adds tuning after real-agent smoke validates the new fixture shape.
- User-facing difference:
  - Real-agent smoke may timeout on slow providers until follow-up lands.
- Pros:
  - Keeps this change focused on fixture shape.
  - Tuning values can be calibrated from real-agent smoke observations.
- Cons:
  - Real-agent smoke may not be fully usable until follow-up lands.

> **PROPOSED: Option B — defer stall tuning to a follow-up change.**
> Rationale: The design explicitly says "That tuning is secondary and must not relax the sentinel-delimited correctness boundary." The fixture shape change is the primary intervention. Stall tuning values should be calibrated from actual real-agent smoke observations rather than guessed. The smoke lane is opt-in and can tolerate initial timeouts while tuning is developed. Adding stall overrides now would also risk expanding the blueprint schema without validated requirements.
>
> **DECISION: Accept Option B.**
> Rationale: Keep this change focused on fixture shape, inspectability, and honest test-lane separation. The runtime already has a place for CAO shadow stall-policy metadata, so a later tuning change remains available once real-agent smoke produces actual observations. That keeps the sentinel-delimited correctness boundary strict in this change.
