## Summary

- **Solid architectural direction**: The proposal correctly identifies the core problem — tracker semantics are tangled with host concerns in `server/tui/tracking.py` and replay-shaped assumptions in `shared_tui_tracking/reducer.py`. The standalone session boundary is a well-motivated refactoring target.
- **Strong alignment with existing Rx patterns**: The proposed Rx session with injectable schedulers directly mirrors the established `abc.SchedulerBase` injection pattern already used in `shared_tui_tracking/reducer.py` (line 55), `lifecycle/rx_lifecycle_kernel.py`, and `cao_rx_monitor.py`.
- **Plugin/profile contract fills a real gap**: The current detector selection in `detectors.py` (`select_tracked_turn_signal_detector()`, lines 639+) is a hardcoded factory with `if tool == "claude"` / `if tool == "codex"` branching. The proposed registry-based profile resolution is a meaningful improvement.
- **Risks around API surface size and migration scope**: The change touches at least five modules (`shared_tui_tracking/`, `server/tui/`, `terminal_record/`, `explore/claude_code_state_tracking/`, and related tests). The tasks list is appropriately staged but the specs underspecify the compatibility adapter shapes and the boundary between "tracker state" and "lifecycle kernel state".
- **Open design questions from the proposal remain unanswered in the specs**: The two explicit open questions in `design.md` (lines 199–201) about debug-report publicity and stability-metadata ownership are not resolved in any spec.
- **Version profile resolution overlaps with an existing pattern**: `VersionedPresetRegistry` in `shadow_parser_core.py` (lines 258–389) already implements closest-compatible version resolution with semver floor matching. The versioned-tui-signal-profiles spec does not reference or reuse this existing mechanism.

## Findings

### Must Fix / Blocking

1. **The specs do not define the standalone tracker session's concrete Python API surface.** The design (section 2) describes `on_snapshot(raw_text)`, `on_input_submitted(...)`, `current_state()`, and `drain_events()` but no spec formalizes parameter types, return types, error semantics, or thread-safety guarantees. The existing `StreamStateReducer` in `shared_tui_tracking/reducer.py` has a well-defined API (`process_observation(observation: RecordedObservation) -> TrackedTimelineState`, `process_input_event(event: RecordedInputEvent) -> None`, `drain_events() -> list[ReplayEvent]`). The new session API needs at least equivalent precision — especially since the proposal explicitly changes the input from `RecordedObservation` (timestamped, with runtime diagnostics) to raw `str` snapshots. Without a concrete contract, Task 1.1 ("Define the standalone tracked-TUI session contracts") is unimplementable from the spec alone.

> **DECISION: Accept.**
> Rationale: The review is correct that Task 1.1 is not implementable from the current spec text alone. The artifact revision should define the concrete session methods, input/output models, and concurrency/error semantics instead of leaving them only in design prose.

2. **The boundary between the standalone tracker's state and the lifecycle kernel's completion/readiness pipelines is undefined.** The existing `LiveSessionTracker` in `server/tui/tracking.py` (lines 118–125) creates `Subject[LifecycleObservation]` and subscribes to `build_readiness_pipeline()` and `build_anchored_completion_pipeline()` from `lifecycle/rx_lifecycle_kernel.py`. The proposal says the standalone tracker "owns Rx subjects for incoming snapshot and input events" and "emitted transition events" (design.md, section 2), but does not clarify whether the lifecycle kernel's readiness/completion observables are inside or outside the standalone tracker boundary. If inside, the tracker must own `CompletionState` and `ReadinessState` transitions (currently server-owned via `rx_lifecycle_kernel.py`). If outside, the server adapter must still orchestrate two reactive pipelines plus the tracker, which may not simplify the current design.

> **DECISION: Accept.**
> Rationale: This ownership edge has to be resolved before implementation starts. The intended answer is that readiness/completion lifecycle pipelines stay host-owned outside the standalone tracker, and the artifacts should state that explicitly.

3. **The `shared-tui-tracking-core` spec's "raw TUI snapshot strings" input requirement conflicts with the existing detector interface.** The current `BaseTrackedTurnSignalDetector.detect()` in `detectors.py` (lines 153–163) accepts both `output_text: str | None` and `parsed_surface: ParsedSurfaceContext | None`. The `ClaudeCodeSignalDetectorV2_1_X` (lines 184–380) heavily uses `parsed_surface` fields (`business_state`, `input_mode`) for signal detection. If the standalone tracker only accepts "raw TUI snapshot strings", then either: (a) the tracker must own parsing internally (contradicting design decision 1, which excludes parser selection), or (b) the detector contract must change to work from raw text only (breaking existing detectors that depend on parsed surface), or (c) "raw TUI snapshot" must be redefined to include optional parsed context (which dilutes the "raw" boundary claim). This tension is not addressed.

> **DECISION: Accept.**
> Rationale: The reviewer correctly found a real conflict between the current detector contract and the intended raw-snapshot tracker boundary. The resolution is to keep the tracker raw-text-first and make detector/profile refactoring explicit in the artifacts rather than softening the public tracker input contract.

### Should Fix / Important

4. **The versioned-tui-signal-profiles spec does not reference or reuse `VersionedPresetRegistry` from `shadow_parser_core.py` (lines 258–389).** That class already implements the exact pattern described: version-sorted presets, semver floor matching, env-override fallback, and a `.resolve()` method. The proposed "closest-compatible profile resolution" (versioned-tui-signal-profiles spec, requirement 1) should either reuse this proven mechanism or explicitly justify why a separate resolution strategy is needed. Duplicating version-resolution logic creates a maintenance divergence risk.

> **DECISION: Refine.**
> Rationale: The real gap is that the desired resolution semantics are underspecified, not that the tracker must directly reuse `VersionedPresetRegistry`. The artifact revision should specify closest-compatible semver-floor behavior while keeping tracker profile resolution independent from parser/backend registry machinery.

5. **The compatibility adapter strategy (design decision 7) lacks specificity about which adapters are needed.** The design says "wrap old replay helpers around the new tracker session" and "adapt server `record_cycle()` to feed the new tracker", but does not describe the adapter shapes. The current integration points are:
   - `replay_timeline()` in `reducer.py` (lines 428–462) which creates a `TestScheduler` + `StreamStateReducer` and schedules observations via `scheduler.schedule_absolute()`.
   - `StreamStateReducer` subclass in `explore/claude_code_state_tracking/state_reducer.py` (lines 15–34) which hardcodes `tool="claude"`.
   - `LiveSessionTracker.record_cycle()` in `server/tui/tracking.py` (lines 378–615) which calls `select_tracked_turn_signal_detector()` and builds the full state response.
   
   Each of these has different adapter requirements. Without specifying adapter interfaces, Tasks 4.1 and 4.2 risk becoming design-on-the-fly work.

> **DECISION: Refine.**
> Rationale: The gap is real and should be corrected in the design/tasks so migration work is not left to interpretation during implementation. The revision should explicitly name the replay helper, server live adapter, explore wrapper, and recorder/replay consumers as concrete adapter seams.

6. **The proposal marks this as BREAKING but no spec captures the migration path for the `explore/claude_code_state_tracking/` module.** The `StreamStateReducer` subclass in `state_reducer.py` (lines 15–34) and the `interactive_watch.py` module (600+ lines) directly depend on the current `StreamStateReducer` API. Neither the `shared-tui-tracking-core` spec nor the tasks mention this module explicitly despite it being listed in the proposal's impact section.

> **DECISION: Refine.**
> Rationale: The concern is valid, but it is best handled as an explicit migration/task update rather than as a new top-level spec requirement. The revision should name `explore/claude_code_state_tracking` directly in the migration plan and tasks, including the compatibility wrapper impact.

7. **The "app plugin contract" in the versioned-tui-signal-profiles spec uses vague terms without mapping to existing abstractions.** It says each app registers "an app identifier, closest-compatible profile resolution from observed version metadata, and a detection entrypoint". The existing detector hierarchy already has `BaseTrackedTurnSignalDetector` (abstract), `BaseVersionedClaudeDetector` (abstract with `compatibility_score()`), and concrete per-version detectors. The spec should clarify whether the "plugin contract" replaces, wraps, or extends this hierarchy. The current `compatibility_score()` approach in `BaseVersionedClaudeDetector` (lines 166–181) is per-detector, not per-profile — the spec proposes per-profile but doesn't show how profile-level resolution relates to detector-level scoring.

> **DECISION: Accept.**
> Rationale: The plugin/profile layer needs a clearer relationship to the current detector seam so the migration target is not ambiguous. The intended direction is that the new shared boundary replaces the current tool-string selector, while existing detector classes may be reused internally as temporary scaffolding during migration.

8. **Thread-safety guarantees are not specified for the new standalone session.** The current `StreamStateReducer` uses `threading.Lock()` (`m_lock`) and the `LiveSessionTracker` uses `threading.RLock()`. The design proposes a new session object that "owns Rx subjects" and "current tracked state cache" but does not specify whether it must be thread-safe. Since the server adapter will feed it from polling threads (see `SessionWatchWorker` in `supervisor.py`), thread safety is an implementation-critical contract that should appear in the spec.

> **DECISION: Accept.**
> Rationale: Concurrency expectations cannot remain implicit because server polling already runs on background threads. The shared-core spec should state thread-safety expectations for the public session methods and preserve the current lock-based implementation style.

### Proposed Design Alignment Assessment

**Aligns well:**

- The Rx session with injectable schedulers is a natural evolution of the existing `StreamStateReducer` + `HistoricalScheduler` pattern. The codebase already treats schedulers as an injection seam; making it a first-class session parameter is consistent.
- Separating host observation from tracker semantics mirrors the existing clean dependency direction: `shared_tui_tracking` → independent, `server/tui` → depends on shared. The proposal preserves and strengthens this layering.
- The staged migration with compatibility adapters (design decision 7) is appropriate given the subsystem's criticality and the number of callers.

**Potential mismatches:**

- The proposal describes a "plugin registry" but the codebase's closest existing patterns are either hardcoded factories (`select_tracked_turn_signal_detector()`) or constructor-injected tuples (`VersionedPresetRegistry(presets=(...))` in `shadow_parser_core.py:268`). The codebase does not use module-level registries with decorator-based registration or entry-point discovery. A new plugin registry should stay close to the constructor-injection-of-tuples pattern rather than introducing a global mutable registry.
- The "raw TUI snapshot strings" input boundary is stricter than what the existing detector interface requires. `BaseTrackedTurnSignalDetector.detect()` takes `parsed_surface: ParsedSurfaceContext | None` alongside `output_text`. If the standalone tracker truly only accepts raw strings, it would need to either own parsing (contradicting design goal) or lose parsed-surface signal quality. The codebase pattern is to pass both raw and parsed data where available.
- The `LiveSessionTracker` in `server/tui/tracking.py` currently owns lifecycle-kernel integration (readiness + completion pipelines via `rx_lifecycle_kernel.py`). The proposal does not clearly assign this to either the standalone tracker or the server adapter. In the existing architecture, these pipelines are deeply coupled to the server's `HistoricalScheduler` and `Subject[LifecycleObservation]` — moving them requires rethinking the observation model that feeds the kernel.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the standalone tracker accept only raw text, or raw text plus optional parsed-surface context?

- Decision: `Blocking`

#### Why this matters

- The existing detectors (`ClaudeCodeSignalDetectorV2_1_X`, `CodexTrackedTurnSignalDetector`) rely on `ParsedSurfaceContext` fields (`business_state`, `input_mode`) for critical signal decisions. If the standalone tracker only accepts raw strings, either the tracker must own parsing or the detectors must be rewritten to work from raw text alone — both are significant scope changes not captured in the current tasks.

#### Examples

- `ClaudeCodeSignalDetectorV2_1_X.detect()` (detectors.py, lines 210–380) checks `parsed_surface.business_state` for values like `"waiting"`, `"processing"`, `"interrupted"` to determine turn signals. Without parsed context, the detector would need to re-derive this from raw ANSI text.
- `CodexTrackedTurnSignalDetector.detect()` (detectors.py, lines 442–559) similarly uses `parsed_surface.input_mode` to distinguish ready vs active states.

#### Why this question arises in code

- The design says the tracker's "host-facing inputs are limited to raw snapshot events and optional explicit input-authority events" (design.md, decision 1).
- But the existing `BaseTrackedTurnSignalDetector.detect()` signature (detectors.py, lines 153–163) is:
  ```python
  def detect(
      self,
      *,
      output_text: str | None,
      parsed_surface: ParsedSurfaceContext | None = None,
  ) -> DetectedTurnSignals:
  ```
- The detector interface already supports optional parsed context. The question is whether the standalone tracker should pass it through or suppress it.

#### Options

**Option A: Accept raw text only — tracker does not pass parsed context to detectors**

- Code impact:
  - All detectors must be rewritten to derive signals from raw text only
  - `SurfaceView` class in `detectors.py` (lines 67–128) would become the primary parsing surface
  - `ParsedSurfaceContext` removed from detector contract
- User-facing difference:
  - Simpler tracker input API
  - Potential signal quality regression during migration
- Pros:
  - Cleaner "raw-only" boundary
  - Truly independent of any parser
- Cons:
  - Major detector rewrite scope not in current tasks
  - Signal quality regression risk (parsed context provides structured data that raw text regex cannot reliably match)

**Option B: Accept raw text plus optional parsed-surface context — pass both to detectors**

- Code impact:
  - Tracker input becomes `on_snapshot(raw_text: str, parsed_surface: ParsedSurfaceContext | None = None)`
  - Detectors keep existing interface unchanged
  - Hosts that have parsed context pass it; hosts that don't omit it
  ```python
  def on_snapshot(
      self,
      raw_text: str,
      parsed_surface: ParsedSurfaceContext | None = None,
  ) -> None:
      signals = self.m_detector.detect(
          output_text=raw_text,
          parsed_surface=parsed_surface,
      )
  ```
- User-facing difference:
  - Richer signal detection when parsed context available
  - No regression for current callers
- Pros:
  - Preserves existing detector quality
  - Backward-compatible with current detector interface
  - Hosts that lack parsing can still use the tracker
- Cons:
  - "Optional parsed surface" weakens the "raw-only" boundary claim
  - Introduces an implicit parser-shaped dependency even if optional

> **PROPOSED: Option B — accept raw text plus optional parsed-surface context.**
> Rationale: The existing `BaseTrackedTurnSignalDetector.detect()` interface already supports optional `parsed_surface` (detectors.py:153–163). Both `ClaudeCodeSignalDetectorV2_1_X` and `CodexTrackedTurnSignalDetector` use it for critical signal decisions. Stripping it forces a detector rewrite not scoped in the current tasks and risks signal regression. The "raw-only" boundary claim can be softened to "raw text is required, parsed context is an optional enrichment" — this preserves the host-neutrality goal (non-parsing hosts simply pass `None`) while keeping signal quality for hosts that do parse. The existing `RecordedObservation` already includes optional `ParsedSurfaceContext` (models.py:93–102), showing this pattern is native to the codebase.
>
> **DECISION: Reject PROPOSED; keep the standalone tracker raw-text-first and do not pass parsed-surface context through its public API.**
> Rationale: The original design goal for this change is to make the tracker consume raw TUI strings and derive signals independently of server/parser ownership. Accepting optional `ParsedSurfaceContext` at the public tracker boundary would preserve exactly the parser-shaped coupling this refactor is trying to remove. The artifact revision should instead make the detector/profile rewrite scope explicit and require supported app profiles to derive their own signals from raw text.

### Q2) Should the lifecycle kernel's readiness/completion pipelines live inside or outside the standalone tracker?

- Decision: `Blocking`

#### Why this matters

- This determines the scope of the standalone tracker and the complexity of the server adapter. If lifecycle pipelines are inside, the tracker becomes a more complete state machine but also heavier. If outside, the server adapter must still orchestrate multiple reactive pipelines, which is close to the current design's complexity.

#### Examples

- The current `LiveSessionTracker` (tracking.py:118–125) owns a `Subject[LifecycleObservation]` and subscribes to `build_readiness_pipeline()` and `build_anchored_completion_pipeline()` from `rx_lifecycle_kernel.py`.
- These pipelines drive `ReadinessState` and `CompletionState` transitions, which feed into the published `HoumaoTerminalStateResponse`.

#### Why this question arises in code

- The design says the tracker "owns Rx subjects for incoming snapshot and input events" and "emitted transition events" (design.md, decision 2). This suggests lifecycle pipelines are inside.
- But design decision 1 says "host adapters remain responsible for... process and transport diagnostics" and the tracker only owns "TUI turn-state semantics". Readiness and completion involve transport/process state (e.g., `ReadinessState` can be `"blocked"` due to process death), which is host-owned.
- `rx_lifecycle_kernel.py` defines `ReadinessLifecycleStatus` and `AnchoredCompletionStatus` as separate type aliases from `TurnPhase` — suggesting they are a different layer.

#### Options

**Option A: Lifecycle pipelines inside the standalone tracker**

- Code impact:
  - Tracker owns `build_readiness_pipeline()` and `build_anchored_completion_pipeline()` subscriptions
  - Tracker output includes `ReadinessState` and `CompletionState`
  - Server adapter becomes thin: feed observations + consume state
  ```python
  class StandaloneTrackerSession:
      def on_snapshot(self, raw_text: str, ...) -> None:
          # feeds lifecycle kernel internally
      def current_state(self) -> TrackerState:
          # includes readiness + completion + turn
  ```
- User-facing difference:
  - Richer standalone tracker output
  - Server adapter significantly simplified
- Pros:
  - One place for all state-machine logic
  - Simpler server adapter
- Cons:
  - Tracker must accept transport/process state as input (weakens "raw TUI" boundary)
  - Heavier module with more responsibilities

**Option B: Lifecycle pipelines outside — owned by server adapter**

- Code impact:
  - Tracker only owns turn-phase transitions (ready/active/unknown) and turn result
  - Server adapter keeps lifecycle kernel integration (current design)
  - Server combines tracker output + lifecycle pipeline output into published state
- User-facing difference:
  - Standalone tracker is leaner and easier to test
  - Server adapter retains current complexity
- Pros:
  - Cleaner separation — tracker only does what it uniquely needs to
  - Lifecycle kernel stays a server concern
- Cons:
  - Server adapter is not much simpler than current `LiveSessionTracker`
  - Two state sources to merge (tracker + lifecycle pipelines)

> **PROPOSED: Option B — lifecycle pipelines stay outside the standalone tracker, owned by the server adapter.**
> Rationale: The lifecycle kernel (`rx_lifecycle_kernel.py`) operates on `LifecycleObservation` events that include transport state, process state, and parsed-surface readiness — all of which are host-owned concerns per design decision 1. Moving these pipelines into the standalone tracker would require the tracker to accept host-diagnostic inputs, contradicting its "raw TUI + input events" boundary. The existing `build_readiness_pipeline()` and `build_anchored_completion_pipeline()` are already well-factored as composable Rx functions that the server adapter can independently subscribe to. The tracker should output turn-phase and turn-result events; the server adapter merges those with its own lifecycle pipeline snapshots, exactly as `LiveSessionTracker.record_cycle()` currently assembles `HoumaoTerminalStateResponse` from multiple sources (tracking.py:535–610). This keeps the standalone tracker testable without simulating transport/process state.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Readiness/completion monitoring remains outside the standalone tracker and stays host-owned. That keeps the tracker bounded to turn-state semantics over raw snapshots and explicit input events, while the server adapter continues to own `LifecycleObservation`-driven pipelines and merges both layers into the published live contract.

### Q3) Should the versioned profile resolution reuse `VersionedPresetRegistry` or introduce a new mechanism?

- Decision: `Deferrable`

#### Why this matters

- Code duplication of version-resolution logic is a maintenance burden. The codebase already has a battle-tested version-resolution class that handles semver sorting, floor matching, and env overrides.

#### Examples

- `VersionedPresetRegistry` in `shadow_parser_core.py` (lines 258–389) resolves parser presets by version with exact-match → floor-match → latest fallback.
- The proposed profile resolution needs the same semantics: closest-compatible version from a set of registered profiles.

#### Why this question arises in code

- The `versioned-tui-signal-profiles` spec says "the plugin resolves the closest-compatible profile for that version" without specifying the resolution algorithm.
- `VersionedPresetRegistry.resolve()` (shadow_parser_core.py:293–389) already implements: env override check → version detection → exact match → semver floor match → latest fallback.
- The detector hierarchy already has `compatibility_score()` in `BaseVersionedClaudeDetector` (detectors.py:166–181), which is a different resolution approach (scoring) than `VersionedPresetRegistry` (floor matching).

#### Options

**Option A: Reuse `VersionedPresetRegistry` as the profile resolution mechanism**

- Code impact:
  - Signal profiles become `VersionedParserPreset`-like objects with a `version` field
  - Resolution uses the existing floor-match + env-override logic
  - Requires extracting `VersionedPresetRegistry` from `shadow_parser_core.py` into a shared utility
- User-facing difference:
  - Consistent version resolution across parser presets and signal profiles
- Pros:
  - Proven implementation
  - No duplicate version-resolution logic
- Cons:
  - May need generalization (currently parser-specific naming)
  - Extracts from a backend module into shared space

**Option B: New profile resolution mechanism specific to signal profiles**

- Code impact:
  - New registry class in `shared_tui_tracking/` with profile-specific resolution
  - May use scoring (like existing `compatibility_score()`) instead of floor matching
- User-facing difference:
  - Profile resolution may behave differently from parser preset resolution
- Pros:
  - Can be tailored to signal-profile needs
  - No coupling to parser-specific code
- Cons:
  - Duplicates version-resolution concepts
  - Two different resolution strategies to maintain

> **PROPOSED: Extract the version-resolution algorithm from `VersionedPresetRegistry` into a shared utility, then reuse it for signal-profile resolution.**
> Rationale: `VersionedPresetRegistry` (shadow_parser_core.py:258–389) already implements exactly the resolution semantics described in the versioned-tui-signal-profiles spec — semver-sorted candidates, closest-floor matching, and fallback. Extracting the resolution logic (not the parser-specific preset model) into a shared utility (e.g., `src/houmao/shared_tui_tracking/version_resolution.py` or a more general location) avoids duplicating a subtle algorithm. The existing `compatibility_score()` approach in `BaseVersionedClaudeDetector` can coexist as an intra-profile ranking mechanism, while the registry-level resolution uses floor matching to select the profile itself.
>
> **DECISION: Reject PROPOSED; do not reuse `VersionedPresetRegistry` directly, but do make the tracker's closest-compatible resolution semantics explicit and semver-based.**
> Rationale: The tracker should not take on a direct dependency on parser/backend registry machinery or its env-override behavior just to resolve signal profiles. The revision should specify a small tracker-local profile resolver with explicit floor-match semantics, and it may reuse only generic semver comparison helpers if that reduces duplication cleanly.

### Q4) Should the standalone tracker expose profile-owned matched-signal reports as a first-class public debugging API?

- Decision: `Deferrable`

#### Why this matters

- This is one of the two explicit open questions from `design.md` (line 200). The answer affects API surface size and whether debugging tools can depend on signal-report structure.

#### Examples

- `DetectedTurnSignals` in `models.py` (lines 105–128) has 17 fields including `accepting_input`, `active_evidence`, `success_candidate`, `interrupted`, `known_failure`, etc. These are the normalized signals.
- Profile-owned detail would be app-specific evidence like "matched spinner pattern X on line Y" or "detected Claude v2.1 prompt at position Z".

#### Why this question arises in code

- The current `StreamStateReducer` exposes `latest_signals` as a property (reducer.py), which returns `DetectedTurnSignals` — these are the normalized signals, not the profile-owned detection evidence.
- The `TrackingDebugSink` protocol in `server/tui/tracking.py` receives detailed cycle events for diagnostics. This is a server-owned debugging surface, not a tracker API.

#### Options

**Option A: Keep matched-signal reports internal/testing-only initially**

- Code impact:
  - Profile detectors return normalized `DetectedTurnSignals` as today
  - Profile-owned evidence available only in tests or via explicit debug accessor
- User-facing difference:
  - Public API stays small and stable
- Pros:
  - Smaller API surface
  - Evidence format can evolve without breaking consumers
- Cons:
  - Debugging tools can't inspect profile-specific evidence without test-only APIs

**Option B: Expose matched-signal reports as a public debugging API**

- Code impact:
  - `DetectedTurnSignals` (or a wrapper) includes profile-owned evidence
  - API surface grows; evidence format becomes part of public contract
- User-facing difference:
  - Debugging and diagnostic tools get richer signal reports
- Pros:
  - Useful for interactive-watch and TrackingDebugSink
  - Transparency into detection decisions
- Cons:
  - Larger API to maintain
  - Profile-specific evidence format becomes a stability concern

> **PROPOSED: Keep matched-signal reports as an internal/testing surface initially, with a documented path to promotion.**
> Rationale: The current `TrackingDebugSink` in `server/tui/tracking.py` is already a separate debugging channel, not part of the tracked-state public API. Following that precedent, profile-owned evidence should be accessible via a debug/testing accessor (e.g., `_debug_last_evidence()` or through the trace-writer pattern used by `StreamStateReducer.m_trace_writer`) but not part of the first-class `current_state()` return type. This keeps the public API stable while still allowing diagnostic tools to inspect evidence when needed. Once the profile structure stabilizes, promotion to public API can be a follow-up change.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Profile-owned matched-signal evidence should remain an internal or debug/testing surface in v1, not a stable first-class API on `current_state()`. The public tracker contract should stay small and app-neutral while leaving room for a later explicit debugging surface if it proves necessary.

### Q5) Should stability metadata remain partly server-owned where diagnostics participate in the public signature?

- Decision: `Deferrable`

#### Why this matters

- This is the second explicit open question from `design.md` (line 201). The current stability tracking in `LiveSessionTracker` (tracking.py) considers transport/process state changes as instability events, not just turn-state changes. If the standalone tracker owns stability, it only sees turn-state changes. If the server keeps stability, the tracker output is less self-contained.

#### Why this question arises in code

- `LiveSessionTracker` tracks `m_stable_since_monotonic` and `m_stable_since_utc` (tracking.py), resetting them on any state change including diagnostic changes.
- The published `HoumaoStabilityMetadata` includes `stable_since_utc` and `stable_duration_seconds` in the public API response.
- Transport/process state flaps (e.g., tmux disconnect/reconnect) currently reset stability — this is a host concern.

#### Options

**Option A: Standalone tracker owns turn-state stability; server adds diagnostic stability separately**

- Code impact:
  - Tracker reports `turn_stable_since` based on its own state changes
  - Server merges with host-diagnostic stability for the public response
- User-facing difference:
  - Two stability signals available; server computes the "effective" one
- Pros:
  - Tracker stays host-independent
  - Server can still account for diagnostic flaps
- Cons:
  - Merge logic required in server adapter

**Option B: Server owns all stability metadata**

- Code impact:
  - Tracker does not track stability
  - Server computes stability from tracker transitions + diagnostic transitions
- User-facing difference:
  - Stability is purely a server concern
- Pros:
  - Simpler tracker
  - Stability definition stays in one place
- Cons:
  - Non-server hosts can't get stability from the tracker alone

> **PROPOSED: Option A — standalone tracker owns turn-state stability; server adds diagnostic stability.**
> Rationale: The tracker already knows when its own state changes (turn phase, turn result, signals). Tracking `stable_since` from those transitions is lightweight and useful for non-server hosts (replay, interactive-watch). The server can then compute an "effective stability" that also accounts for transport/process flaps, using `min(tracker_stable_since, diagnostic_stable_since)`. This mirrors the design principle of keeping the tracker general while letting the server add host-specific enrichment. The `explore/claude_code_state_tracking/interactive_watch.py` module would directly benefit from tracker-owned stability without needing server infrastructure.
>
> **DECISION: Accept PROPOSED with refinement.**
> Rationale: The standalone tracker should own stability over its own emitted tracker state, because replay and non-server hosts benefit from that directly. The server should separately own effective visible stability when diagnostics and other host-owned fields participate in the published signature, so the artifact revision needs to describe both layers explicitly.
