# Discussion: Shadow Provider Open Questions

Date: 2026-02-28
Change: `fix-cao-claude-output-mode-last`

This note collects open questions introduced by the revised "runtime shadow provider" design (see [design.md](../design.md)). The core constraint is that we do not modify upstream CAO; therefore we treat CAO as a transport and parse `mode=full` output locally for Claude Code.

## Q1) What bounded tail window should shadow parsing use?

Problem:
- Shadow status and extraction should avoid stale scrollback false positives.
- CAO only exposes `output?mode=full` reliably today, so we still receive full scrollback; the tail window mainly affects parsing correctness (and compute), not network cost.

Options:

1. **Fixed tail window (lines)**: Always parse the last `N` lines (e.g. `N=200`).
   - Pros:
     - Simple and deterministic.
     - Easy to test with fixtures.
   - Cons:
     - If a response is very long, the marker or early response lines could fall outside the window.
     - Requires picking a conservative `N`, which increases parsing work.

2. **Adaptive tail window (lines)**: Start with `N_small` (e.g. 200); if no marker/stop boundary is found, expand up to `N_max` (e.g. 2000).
   - Pros:
     - Fast in the common case, robust in edge cases.
     - Naturally handles unusually long answers.
   - Cons:
     - More complex state machine and more branches to test.
     - Still cannot avoid full payload fetch; only reduces parsing work.

3. **Fixed tail window (chars)**: Parse last `K` characters (e.g. 50k).
   - Pros:
     - Bound is closer to actual cost (string scanning).
   - Cons:
     - Can split lines mid-way; harder to reason about prompt/marker-at-line-start rules.
     - Less "human readable" to tune and explain.

Recommendation:
- Use adaptive lines (Option 2): status detection can use a smaller window, while extraction can expand if needed.

> **Answer:** Use two separate fixed windows rather than a single adaptive one.
>
> - **Shadow status classification:** fixed tail of **100 lines** from the end of the full
>   scrollback. This is enough to contain recent spinners, idle prompts, and separator
>   lines. Named constant `STATUS_TAIL_LINES = 100` so it is easy to tune.
>
> - **Answer extraction:** **all lines after the baseline offset** (no additional tail
>   limit). The baseline already provides the forward bound; a further tail limit would
>   risk truncating long responses. The adaptive expansion from Option 2 is not needed
>   because the baseline makes the search space naturally bounded.
>
> The adaptive approach adds branching that complicates testing without meaningful
> benefit once the two-window model is in place.

## Q2) How should we associate an extracted marker with the current turn (baseline/cursor)?

Problem:
- The scrollback includes previous turns. We must not re-extract an old response.
- We need a robust "baseline" captured immediately before (or after) prompt submission.

Options:

1. **Baseline = last marker position**: Before sending input, parse `mode=full` and record the byte/char index of the last response marker match. After sending, only accept markers with `start > baseline_pos`.
   - Pros:
     - Directly matches the extraction semantics.
     - Simple and stable across prompt text changes.
   - Cons:
     - If scrollback is truncated/reset and the output string shrinks, the baseline index may become invalid (must handle gracefully).

2. **Baseline = output length**: Before sending input, record `len(mode_full_output)` and only accept markers after that index.
   - Pros:
     - Very cheap to compute.
   - Cons:
     - More fragile if the output capture reflows or includes non-deterministic UI redraws (length changes that do not correspond to new turn content).

3. **Baseline = prompt boundary**: Record the last idle prompt occurrence (position/index) before sending input; require the extracted marker to occur after that prompt.
   - Pros:
     - Tracks the UI boundary, not just markers.
   - Cons:
     - Prompts can appear during processing; prompt boundary alone is not sufficient, so this must be combined with marker checks.

Recommendation:
- Use Option 1 (last marker position) plus a guard: if baseline is out of range (scrollback truncated), reset baseline to "no baseline" and require both marker + stop boundary to appear after prompt submission time.

> **Answer:** Use Option 1 — baseline is the **character offset of the end of the last
> response marker match** in `mode=full` output captured immediately before prompt
> submission.
>
> Concretely:
> - Before sending input, scan the full `mode=full` output for all response marker
>   matches; record `baseline_pos = last_match.end()` (or `0` if no match found).
> - After sending, only accept response markers where `match.start() >= baseline_pos`.
>
> The guard case: if subsequent `mode=full` output is shorter than `baseline_pos`
> (scrollback was reset or truncated), treat baseline as invalidated and fall back to
> requiring both a response marker AND a stop boundary in the new output to confirm
> completion. Log a warning when this happens.
>
> Line-count offset (similar to Option 2) was considered but rejected: line count is
> less stable than a content-anchored position because ANSI wrapping and UI chrome can
> vary the line count for the same semantic content. Character offset of the last marker
> is semantically meaningful and directly serves the "marker appeared after this point"
> invariant.

## Q3) How should the runtime surface `waiting_user_answer` to callers?

Problem:
- Claude Code can present interactive selection UIs (arrow cursor + numbered options).
- The current `InteractiveSession` interface is oriented around "send one prompt, get one answer", not multi-step interactive selection.

Options:

1. **Treat as error**: Raise a `BackendExecutionError` with a clear message and include a short (ANSI-stripped) excerpt containing the options.
   - Pros:
     - Minimal interface changes.
     - Forces explicit handling; avoids silent wrong behavior.
   - Cons:
     - No built-in way to proceed programmatically.

2. **Treat as a non-terminal result event**: Return a `SessionEvent(kind="waiting_user_answer", ...)` and stop the turn without a `done` answer.
   - Pros:
     - Keeps the session alive without throwing.
     - Lets the CLI/UI show options in a first-class way.
   - Cons:
     - Callers must be updated to handle "turn not completed".
     - Still needs a follow-up mechanism to provide the choice.

3. **Extend the interface**: Add an explicit `answer()` / `send_choice()` method to `InteractiveSession` (or a Claude-only extension).
   - Pros:
     - Full programmatic support for interactive flows.
   - Cons:
     - Larger API surface and broader refactor.
     - Needs careful design to remain backend-agnostic.

Recommendation:
- Start with Option 1 for correctness (fail loud with actionable context).
- If we see this frequently, evolve to Option 2 (event) and only then consider Option 3.

> **Answer:** Option 1 — raise a `BackendExecutionError` (or equivalent) with a message
> that includes an ANSI-stripped excerpt showing the options Claude is presenting.
>
> Rationale: `waiting_user_answer` is rare in brain-launch-runtime flows today and we
> do not yet have a mechanism for the orchestrator to inject a choice back into a live
> session. Raising an error is honest about this limitation, fails loudly so it is
> noticed, and includes enough context (the options text) for a human operator or
> future handler to know what Claude was asking. Returning a structured result (Option 2)
> would require callers to be updated to handle a non-terminal turn and would be
> misleading until a follow-up `send_choice()` mechanism exists.
>
> If a send-choice mechanism is added later, this can evolve to Option 2 at that point
> without changing the semantics of the current callers.

## Q4) How do we prevent accidental reliance on CAO `status` / `mode=last` for Claude Code?

Problem:
- Generic CAO session code is tempting to reuse (poll `GET /terminals/{id}` and fetch `mode=last`).
- For Claude Code specifically, those are untrusted and can regress the fix if reintroduced.

Options:

1. **Dedicated Claude path in the backend**: Add explicit `if tool == "claude"` branches so readiness and output retrieval always use shadow parsing.
   - Pros:
     - Very clear and local to `cao_rest.py`.
   - Cons:
     - Conditional logic can grow if multiple providers need special casing.

2. **Split session classes**: Implement a separate backend class (e.g. `CaoRestClaudeSession`) that never calls `get_terminal()` and never uses `mode=last`.
   - Pros:
     - Stronger separation; harder to regress.
   - Cons:
     - More classes and more wiring in runtime selection.

3. **Make misuse fail fast**: Wrap/override the CAO client for Claude sessions so `get_terminal()` raises, and `get_terminal_output(mode="last")` is disallowed.
   - Pros:
     - Prevents accidental calls in tests and future edits.
   - Cons:
     - More intrusive; may be annoying if we still want terminal metadata for debugging.

Recommendation:
- Option 2 if we want maximum regression resistance; Option 1 if we want minimal moving parts.

> **Answer:** Option 1 — explicit `if tool == "claude"` branches in `cao_rest.py` for
> this change.
>
> Option 2 (separate session class) provides stronger regression resistance but expands
> the scope of this change significantly. The immediate goal is to fix the output
> extraction bug; introducing a new session class requires wiring changes in the runtime
> factory, additional tests for the new class hierarchy, and a broader review of what
> methods need to be overridden. That work is appropriate but belongs in a follow-up
> change once the shadow provider is stable.
>
> For now: gate every readiness wait and output retrieval in `cao_rest.py` behind a
> `if self._is_claude_code_session()` check (or equivalent), and add a comment
> documenting that these branches must not be merged back into the generic path.
> Task 4.1 (re-run demo) will confirm the branches are correct. Option 2 can be pursued
> as a follow-up hardening change once the behavior is verified.

## Q5) Should we opportunistically use CAO `mode=tail` if upstream starts supporting it?

Problem:
- CAO docs describe `mode=tail`, but current CAO server behavior may reject it.
- Fetching full scrollback on every poll is expensive, even if we parse only a small tail window.

Options:

1. **Ignore tail**: Always fetch `mode=full` and do client-side tail parsing.
   - Pros:
     - Works with the current server.
     - One code path to test.
   - Cons:
     - Network and CPU overhead remain higher than necessary.

2. **Try tail-first fallback**: Attempt `mode=tail` for polling; if CAO returns 4xx/422, fall back to `mode=full` and cache "tail unsupported" for that session.
   - Pros:
     - Future-proof; reduces payload if/when CAO supports tail.
   - Cons:
     - More complexity and additional error handling paths.

Recommendation:
- Option 2 if we expect upstream CAO to stabilize `mode=tail` soon; otherwise Option 1 for simplicity.

> **Answer:** Option 1 — always fetch `mode=full`, no tail fallback.
>
> `mode=tail` is explicitly listed in design.md non-goals ("Adding a new CAO API output
> mode in this change") and currently returns HTTP 422. Attempting it opportunistically
> would reintroduce the wasted round-trip that task 2.3 is specifically removing. If
> CAO upstream stabilizes `mode=tail` in the future, adding tail support is a clean,
> isolated change; there is no value in speculative handling now.
