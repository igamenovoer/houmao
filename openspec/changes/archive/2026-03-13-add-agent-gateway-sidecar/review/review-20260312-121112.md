## Findings

### Must Fix / Blocking

- **Resolve design/spec mismatch on default attachability publication.** `specs/brain-launch-runtime/spec.md` states the runtime SHALL publish attach metadata for runtime-owned tmux-backed sessions in v1, but `design.md` still lists “publish by default vs opt-in” as an open question. Pick one and align artifacts.
- **Resolve design/spec mismatch on `AGENTSYS_GATEWAY_ROOT`.** `design.md` implies `AGENTSYS_GATEWAY_ROOT` is optional (“optionally”), but `specs/agent-identity/spec.md` and `tasks.md` require it. Align on a single stable discovery contract.
- **Define the attach-contract schema and its relationship to `attach.json`.** `design.md` describes an attach contract JSON file, while the storage layout lists `attach.json`. The specs require a secret-free attach contract pointer (`AGENTSYS_GATEWAY_ATTACH_PATH`) but do not define required fields, versioning strategy, or validation rules.
- **Define who writes `state.json` when no gateway instance is attached.** The agent-gateway spec requires a stable local “current state artifact” aligned with `GET /v1/status`, but when the gateway is absent there is otherwise no obvious writer. Without a defined writer and initial schema, “state.json as stable local read contract” is underspecified.
- **Lock down the gateway-root identity derivation.** `design.md` references `<gateway-attach-identity>` without choosing a deterministic derivation that avoids collisions (especially when tmux session names are reused). This choice propagates into `AGENTSYS_GATEWAY_ROOT`, persistence, and test fixtures.
- **Document attach failure policy when auto-attach fails after the agent session starts.** The brain-launch-runtime spec calls out that the already-started agent session outcome should be a defined policy; right now it is flagged but not specified.

### Should Fix / Important

- **Clarify input override vs output binding for `AGENTSYS_AGENT_GATEWAY_HOST/PORT`.** The brain-launch-runtime spec uses these as caller env overrides, while agent-identity spec uses them as tmux-published live bindings. This dual use is consistent with `AGENTSYS_AGENT_DEF_DIR` but still needs explicit docs and careful CLI behavior to avoid surprise in read-only commands.
- **Specify stale live-binding validation.** `design.md` states tmux live bindings are hints and should be validated via health checks or run-state files. The concrete rule for clients (runtime-owned gateway clients) should be spelled out so every caller doesn’t invent its own heuristic.
- **Blueprint validation is currently too permissive.** `load_blueprint()` (currently) ignores unknown fields; the component-agent-construction spec requires explicit rejection of invalid gateway host/port. Ensure tasks include strict parsing and clear errors.
- **Port selection + “no silent reselect” needs an explicit restart story.** If the runtime selects a free port when none is provided, decide whether that selection becomes the desired persisted listener (and thus must be reused on restart, failing if conflicting) or whether it can change across instances (which makes stale clients more likely).

### Nice To Have

- Add a small spec appendix or separate spec file for the attach-contract JSON schema to keep “gateway discovery” concrete and testable.
- Add a sequence diagram for attach-later and detach showing env vars + file writes/clears (this will help implementers align on lifecycle edges).
- Consider batching tmux env reads for gateway discovery (the current runtime reads one variable per `tmux show-environment` call); this is an optimization, not a correctness requirement.

## Design Alignment With Existing System

- Existing pattern(s) observed:
  - Name-based resolution reads tmux env pointers with strict absolute-path validation (`_read_tmux_session_env_var()` in `src/gig_agents/agents/brain_launch_runtime/runtime.py:813` and `_resolve_manifest_path_from_tmux_session()` in `src/gig_agents/agents/brain_launch_runtime/runtime.py:868`).
  - Tmux-backed backends re-publish env pointers on resume (`CaoRestSession._publish_tmux_session_environment()` in `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:729`, invoked during resume at `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:644`; headless `_publish_tmux_session_environment()` in `src/gig_agents/agents/brain_launch_runtime/backends/headless_base.py:297`, invoked during resume at `src/gig_agents/agents/brain_launch_runtime/backends/headless_base.py:262`).
  - Session manifests are strict and version-locked (`_StrictBoundaryModel` config in `src/gig_agents/agents/brain_launch_runtime/boundary_models.py:24` and `SessionManifestPayloadV2` in `src/gig_agents/agents/brain_launch_runtime/boundary_models.py:162`; schema version pinned in `src/gig_agents/agents/brain_launch_runtime/manifest.py:24`), so adding new top-level manifest fields requires a schema/version bump rather than opportunistic extension.
  - Blueprints are parsed loosely today (`load_blueprint()` in `src/gig_agents/agents/brain_launch_runtime/loaders.py:85`), which conflicts with the new “reject invalid gateway fields” requirement.
  - Mailbox support demonstrates a good “resolved config -> env bindings -> persisted state” pattern (`resolve_effective_mailbox_config()` in `src/gig_agents/agents/mailbox_runtime_support.py:154` and `mailbox_env_bindings()` in `src/gig_agents/agents/mailbox_runtime_support.py:264`) that the gateway could mirror for attach metadata and live bindings.

- Proposed design alignment:
  - Keep gateway discovery rooted in tmux env pointers, implemented by extending existing tmux env publication sites (CAO + headless) rather than inventing a parallel discovery channel.
  - Avoid session-manifest schema churn by either (a) persisting gateway metadata in gateway-root files (`protocol-version.txt`, `attach.json`, `desired-config.json`, `state.json`) or (b) storing stable pointers inside `backend_state` (which is already a flexible JSON object) if you need resume-time recovery without deriving paths.
  - Make blueprint parsing strict enough to validate `gateway.host` and `gateway.port`, and ensure invalid values are rejected as specified.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should runtime-owned tmux-backed sessions publish gateway attach metadata by default?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- This choice determines whether attach-later can work for “normal” sessions with no gateway flags, and whether the gateway-capable contract becomes ubiquitous like `AGENTSYS_MANIFEST_PATH`.
- It also determines whether gateway-root artifacts get created for every tmux-backed session, impacting disk layout and upgrade expectations.

#### Examples

- A developer runs `start-session` today and later wants to attach a gateway without restarting the agent.
- A developer upgrades to a new version and tries gateway attach on an older running session; is attach metadata expected to exist?

#### Why this question arises in code

- The canonical “publish pointers into tmux env” mechanisms live in `src/gig_agents/agents/brain_launch_runtime/backends/headless_base.py` and `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py`.
- Name-based resolution already depends on stable tmux env pointers for non-gateway control (`src/gig_agents/agents/brain_launch_runtime/runtime.py`), suggesting gateway discovery should follow the same always-present pattern if attach-later is a core goal.

#### Options

**Option A: Publish attachability by default for all runtime-owned tmux-backed sessions.**

- Code impact:
  - Extend tmux env publication to always include `AGENTSYS_GATEWAY_ATTACH_PATH` + `AGENTSYS_GATEWAY_ROOT` alongside existing pointers.
  - Add gateway-root bootstrap (at least `protocol-version.txt` + `attach.json` + seeded `state.json`) during start/resume.
- User-facing difference:
  - Any runtime-owned tmux session is immediately gateway-capable, even if no gateway is running.

**Option B: Publish attachability only when explicitly requested (opt-in).**

- Code impact:
  - Add a CLI flag (or blueprint flag) that triggers attachability publication; thread that decision through start/resume code paths.
  - Attach-later is impossible for sessions started without the opt-in flag.
- User-facing difference:
  - Gateway attach becomes an “enabled feature” rather than a universal capability; attach-later requires forethought.

> **PROPOSED: Option A (publish attachability by default for runtime-owned tmux-backed sessions).**
> Rationale: The existing system already treats tmux env pointers as the durable discovery contract for name-based operations (strict env-var reads in `src/gig_agents/agents/brain_launch_runtime/runtime.py:813` and manifest resolution in `src/gig_agents/agents/brain_launch_runtime/runtime.py:868`). The two current tmux-backed env publication sites (`src/gig_agents/agents/brain_launch_runtime/backends/headless_base.py:297` and `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:700`) are the natural insertion points for gateway attach pointers. The delta spec for brain-launch-runtime already states that in v1 the runtime SHALL publish attach metadata for runtime-owned tmux-backed sessions, so leaving this as opt-in would force a spec change and weaken “attach-later without restart” as a first-class goal.
>
> **DECISION: Accept Option A for new runtime-owned tmux-backed sessions in v1, while keeping legacy sessions with no attach metadata valid.**
> Rationale: The current proposal, specs, and tasks already lean on default attachability publication to make attach-later a first-class flow rather than something a caller had to predict at session start. This still keeps the gateway optional because publishing stable attach metadata is not the same thing as starting a live gateway instance, and older or manual sessions that publish no gateway pointers remain valid non-gateway sessions.

### Q2) Is `AGENTSYS_GATEWAY_ROOT` required, or should it be derived from `AGENTSYS_GATEWAY_ATTACH_PATH`?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- The discovery contract needs to be stable and unambiguous for both tools and human operators.
- The current design doc implies `AGENTSYS_GATEWAY_ROOT` may be optional, but the agent-identity spec requires it, and implementation patterns for strict env reads will treat missing vars as hard errors.

#### Examples

- A tool wants to read `state.json` from disk without relying on a running gateway process.
- A session publishes `AGENTSYS_GATEWAY_ATTACH_PATH` but not `AGENTSYS_GATEWAY_ROOT`; does that session count as gateway-capable?

#### Why this question arises in code

- The runtime’s tmux env read helper fails explicitly on missing variables (`src/gig_agents/agents/brain_launch_runtime/runtime.py`), so a “sometimes present” `AGENTSYS_GATEWAY_ROOT` will need a deliberate compatibility strategy.

#### Options

**Option A: Require `AGENTSYS_GATEWAY_ATTACH_PATH` and `AGENTSYS_GATEWAY_ROOT` together for gateway-capable sessions.**

- Code impact:
  - Treat both as the stable contract; validate both as absolute paths, and validate root as directory.
- User-facing difference:
  - Discovery is explicit; tools do not need to infer path relationships.

**Option B: Only require `AGENTSYS_GATEWAY_ATTACH_PATH`; derive `AGENTSYS_GATEWAY_ROOT` as `dirname(attach_path)`.**

- Code impact:
  - Tooling must assume attach contract lives inside the root directory.
  - Changing attach-contract location becomes a breaking change.
- User-facing difference:
  - Fewer env vars, but implicit coupling between file path and root.

> **PROPOSED: Option A (require both `AGENTSYS_GATEWAY_ATTACH_PATH` and `AGENTSYS_GATEWAY_ROOT`).**
> Rationale: The existing discovery approach favors explicit pointers (manifest path and agent-def-dir are separate explicit bindings) and strict validation (`src/gig_agents/agents/brain_launch_runtime/runtime.py:813`). Requiring the root pointer removes an implicit dependency on where the attach contract file happens to live and matches the delta agent-identity spec contract.
>
> **DECISION: Accept Option A and require both `AGENTSYS_GATEWAY_ATTACH_PATH` and `AGENTSYS_GATEWAY_ROOT` as the stable gateway-capability contract.**
> Rationale: The repo’s tmux discovery pattern already prefers explicit, separately validated pointers over implicit derivation, and the current agent-identity and runtime specs plus tasks are written around both values being present. Keeping the root explicit avoids freezing `dirname(attach_path)` into the protocol and leaves room to move or version the attach file layout later without breaking discovery.

### Q3) What is the deterministic gateway root identity (`<gateway-attach-identity>`)?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- The gateway root must not collide across sessions, and it must remain stable across resume.
- A poor identity choice risks state leakage between sessions or hard-to-debug “resumed session attaches to the wrong gateway root” failures.

#### Examples

- A developer starts `AGENTSYS-gpu`, stops it, and later starts a new `AGENTSYS-gpu`; those must not share gateway queue/state.
- A gateway restarts and must recover queued work for the same session.

#### Why this question arises in code

- New sessions already have a unique `session_id` that is used as the durable key for session-manifest storage (`generate_session_id(prefix=selected_backend)` in `src/gig_agents/agents/brain_launch_runtime/runtime.py` and `default_manifest_path()` in `src/gig_agents/agents/brain_launch_runtime/manifest.py`).

#### Options

**Option A: Use the runtime-generated session id (manifest filename stem) as the gateway attach identity.**

- Code impact:
  - Gateway root becomes `<runtime_root>/gateways/<session_id>/`.
  - Attach contract can include `session_id` and back-reference the manifest path.
- User-facing difference:
  - Stable, collision-resistant identity, even when tmux session names are reused.

**Option B: Use canonical tmux session name as the gateway attach identity.**

- Code impact:
  - Gateway root becomes `<runtime_root>/gateways/<canonical_tmux_name>/`.
  - Must handle collisions when names are reused over time.
- User-facing difference:
  - Root paths are more human-readable but can be unsafe without extra disambiguation.

> **PROPOSED: Option A (use the runtime-generated session id as the gateway attach identity).**
> Rationale: The repo already uses a backend-prefixed session id as the stable storage key for session manifests (`session_id = generate_session_id(prefix=selected_backend)` in `src/gig_agents/agents/brain_launch_runtime/runtime.py:296`, stored at `runtime_root/sessions/<backend>/<session_id>.json` via `default_manifest_path()` in `src/gig_agents/agents/brain_launch_runtime/manifest.py:56`). That id is designed to be unique over time, so reusing it for gateway roots avoids collisions when a human reuses a tmux session name and aligns with existing storage layout conventions under `runtime_root/`.
>
> **DECISION: Refine Option A: use the runtime-generated `session_id` as the gateway-root identity for runtime-owned sessions in v1, and defer the manual-session identity rule to the future attach-contract schema.**
> Rationale: For the current v1 scope, `session_id` is the only collision-resistant durable key the runtime already generates and persists, so it is the safest basis for `<gateway-attach-identity>`. The change should not overgeneralize that rule to future manual sessions yet; those sessions should publish or derive their own stable attach identity as part of the later manual attach-contract definition.

### Q4) Who writes `state.json` when the gateway is not currently attached?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- The agent-gateway spec requires a stable local “current state artifact” and says readers should not need to reconstruct state by replaying the event log.
- If the gateway is optional and can be absent, a “gateway-owned” artifact still needs an initial writer or a defined absence contract.

#### Examples

- Immediately after `start-session` (no gateway running), an operator wants to inspect `state.json` under `AGENTSYS_GATEWAY_ROOT`.
- A gateway was detached gracefully; should `state.json` reflect `offline`/“not attached”?

#### Why this question arises in code

- The runtime already eagerly persists session manifests at session start (`controller.persist_manifest()` in `src/gig_agents/agents/brain_launch_runtime/runtime.py`), but there is currently no gateway artifact creation.
- Backends already have a “republish pointers on resume” pattern; adding a “seed state.json” step fits the same lifecycle hook points.

#### Options

**Option A: Runtime seeds `state.json` at capability publication (start/resume) and updates it on detach.**

- Code impact:
  - Add a small writer in runtime/backends that creates `<gateway_root>/state.json` with `gateway_health=offline` (or equivalent) when no live instance exists.
  - On detach, clear live tmux env vars and write an offline snapshot.
- User-facing difference:
  - `state.json` always exists for gateway-capable sessions, even before first attach.

**Option B: Only the gateway process writes `state.json`; absence means “not attached”.**

- Code impact:
  - Tools must treat missing state.json as offline/unattached.
  - Specs must be clarified to allow the file to not exist prior to first attach.
- User-facing difference:
  - Fewer runtime writes, but weaker “stable local read contract” guarantees.

> **PROPOSED: Option A (runtime seeds `state.json` when publishing attachability, gateway updates it when running).**
> Rationale: The change explicitly wants `state.json` to be a stable local read contract that mirrors `GET /v1/status`, and the gateway is optional/attach-later by design. Seeding an offline snapshot at session start aligns with the existing “persist durable artifacts at start” approach (session manifest is written immediately at `src/gig_agents/agents/brain_launch_runtime/runtime.py:360`) and avoids forcing every consumer to interpret “missing file” as a special case.
>
> **DECISION: Accept Option A: runtime-owned lifecycle code seeds `state.json` when attachability is published, and gateway attach or detach updates it thereafter.**
> Rationale: `state.json` is already positioned as a stable local read contract aligned with `GET /v1/status`, so its existence should not depend on whether a gateway process is currently alive. Seeding an offline snapshot at capability publication and refreshing it on graceful detach keeps the contract simple for readers while still letting the running gateway own live status updates.

### Q5) When the system selects a free port, is that selection persisted as the desired listener for later restarts?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- The specs require failing on bind conflicts and forbidding silent port reselection, but also allow “system-selected free port” when no explicit port is provided.
- Without a persistence rule, restart behavior can become surprising or inconsistent across implementations.

#### Examples

- First attach chooses port `43123`; the gateway crashes; on restart, should it attempt `43123` and fail if now in use?
- A caller reads tmux env live bindings and caches them; a restart on a different port makes caches stale.

#### Why this question arises in code

- Existing runtime persistence already favors stable identity and “resume picks up where you left off” semantics via the session manifest (`src/gig_agents/agents/brain_launch_runtime/manifest.py`).
- The design introduces `desired-config.json` and a split between stable metadata and live bindings, but does not yet state whether a system-selected port becomes part of desired stable config.

#### Options

**Option A: Persist the first resolved host/port (even if system-selected) to `desired-config.json` and reuse it on restart.**

- Code impact:
  - Implement `desired-config.json` as the source of desired listener; on bind failure, fail attach and require explicit override or manual intervention.
- User-facing difference:
  - Listener config is stable across restarts unless explicitly changed; restarts may fail if the port is taken.

**Option B: Treat system-selected ports as ephemeral and reselect on every new gateway instance when no explicit port is supplied.**

- Code impact:
  - Port selection happens each time; live bindings change more often.
  - Must heavily emphasize live-binding invalidation and health checks.
- User-facing difference:
  - More robust restarts in the presence of port conflicts, but higher churn and more stale-pointer scenarios.

> **PROPOSED: Option A (persist the resolved host/port as the desired listener and reuse it on restart).**
> Rationale: The “no silent port reselection” rule is most meaningful when it applies consistently across restarts, not only within a single process lifetime. Persisting the selected port in `desired-config.json` makes the desired listener explicit and discoverable, and matches the repo’s preference for durable, resume-friendly state (resume is driven by persisted manifests in `src/gig_agents/agents/brain_launch_runtime/runtime.py:412`, and manifests already preserve backend identifiers and tmux bindings).
>
> **DECISION: Refine Option A: once a gateway instance starts successfully, persist its resolved host and port in `desired-config.json` and reuse them on later restarts unless the caller explicitly overrides them.**
> Rationale: This preserves the “no silent reselection” rule across restarts without turning host or port into immutable session identity, because live bindings remain instance-level facts that still require validation. Persisting the first successful listener also gives restart and re-attach deterministic behavior instead of surprising callers with a different auto-selected port every time.

### Q6) Should gateway-aware control commands auto-attach a gateway when none is running?

<!-- Classification: mark every question -->
- Decision: `Deferrable`

#### Why this matters

- Auto-attaching implies starting a network listener (even if loopback) as a side effect of a “control” command; that can surprise operators and complicate troubleshooting.
- Failing explicitly keeps lifecycle control separate and makes early versions easier to test and reason about.

#### Examples

- `send-prompt` is run against a gateway-capable session with no live gateway: does it error, or does it start a gateway automatically and then enqueue?
- A script expects a prompt submission to be “pure” and not spawn extra processes.

#### Why this question arises in code

- The current CLI routes `send-prompt` and `send-keys` directly through resumed backend sessions (`src/gig_agents/agents/brain_launch_runtime/cli.py` + `runtime.py`) with no lifecycle side effects beyond the control action itself.
- The design explicitly separates lifecycle control from the HTTP request API (“ensure attached, then use HTTP”).

#### Options

**Option A: Do not auto-attach by default; fail with explicit “no live gateway attached” and provide a separate attach command.**

- Code impact:
  - Implement `gateway attach` / `gateway detach` lifecycle commands first; keep control paths simple.
- User-facing difference:
  - More explicit steps, but fewer surprising side effects.

**Option B: Auto-attach by default when session is gateway-capable but gateway is not running.**

- Code impact:
  - Control commands must incorporate listener resolution, port selection, and failure policy; increases complexity and test matrix.
- User-facing difference:
  - More “just works” UX, but higher implicit behavior.

> **PROPOSED: Option A (fail explicitly by default; add auto-attach later as an explicit mode if desired).**
> Rationale: The current CLI and runtime favor explicit control surfaces and explicit errors when prerequisites are missing. Keeping “attach” as an explicit lifecycle step matches the design’s separation of concerns and reduces the risk of accidental port exposure or hard-to-debug background processes in early iterations.
>
> **DECISION: Accept Option A for v1: gateway-aware control commands fail explicitly when no live gateway is attached, and attach remains a separate lifecycle action.**
> Rationale: The revised design intentionally separates gateway lifecycle from request submission, and the current runtime still preserves direct control behavior for sessions with no live gateway instance. Avoiding implicit auto-attach keeps control commands from silently starting background processes or listeners, while leaving room for a future explicit `ensure-attached` mode if that proves worth adding.
