# Discussion: Agent Gateway Remaining Open Questions

- Feature/Change: `add-agent-gateway-sidecar`
- Timestamp: `2026-03-12T15:20:03Z`
- Context: This doc captures the remaining unresolved design and implementation questions after the review-driven revision of the `add-agent-gateway-sidecar` OpenSpec change. It is intended to turn the leftover ambiguities into explicit decisions before runtime code work starts.
- Inputs reviewed: `proposal.md`, `design.md`, `tasks.md`, `specs/agent-gateway/spec.md`, `specs/agent-identity/spec.md`, `specs/brain-launch-runtime/spec.md`, `specs/component-agent-construction/spec.md`, `review/review-20260312-121112.md`, `src/gig_agents/agents/brain_launch_runtime/cli.py`, `src/gig_agents/agents/brain_launch_runtime/runtime.py`, `src/gig_agents/agents/brain_launch_runtime/loaders.py`, `src/gig_agents/agents/brain_launch_runtime/manifest.py`, `src/gig_agents/agents/brain_launch_runtime/boundary_models.py`, and `src/gig_agents/agents/mailbox_runtime_support.py`.

Key constraints:

- Newly started runtime-owned tmux-backed sessions are gateway-capable by default in v1.
- Stable attachability and live gateway state are separate concerns.
- `AGENTSYS_GATEWAY_ATTACH_PATH` and `AGENTSYS_GATEWAY_ROOT` are both required stable pointers for gateway-capable runtime-owned sessions.
- Runtime-owned session roots use the runtime-generated `session_id` in v1, and the gateway root is the nested `gateway/` subdirectory under that session root.
- `state.json` is a stable local read contract aligned with `GET /v1/status` and is seeded before the first live attach.
- Gateway-aware control commands do not auto-attach a gateway implicitly in v1.
- Manual-session gateway support is intentionally left for a follow-up, but the current attach contract should not make that follow-up needlessly hard.

---

## Summary of recommended defaults

1) Define one small versioned `attach.json` schema now, with a required core that runtime-owned sessions publish in v1 and future manual sessions can also satisfy. (Q1)
2) Treat runtime-owned `stop-session` as authoritative teardown: if a live gateway is attached, stop it too and write the gateway root back to offline or not-attached state. (Q2)
3) If launch-time auto-attach fails after the agent starts, keep the agent session running, return a structured partial-start failure, and let operators decide whether to retry attach or stop the session. (Q3)
4) Make blueprint parsing strict in v1, using an explicit typed schema that rejects invalid gateway fields and unknown top-level fields instead of silently ignoring them. (Q4)
5) Standardize runtime-owned live-binding validation on a health-first rule: tmux env bindings are hints, `GET /health` is the authoritative liveness check, and attach-capable sessions with failed health checks are treated as having no live gateway attached. (Q5)

## Q1) What exact v1 `attach.json` schema should be required, and how much manual-session compatibility should it reserve now?

- Decision: `Blocking`

### Why this matters

- The current change now requires both `AGENTSYS_GATEWAY_ATTACH_PATH` and `AGENTSYS_GATEWAY_ROOT`, but the actual `attach.json` field set is still only described informally in `openspec/changes/add-agent-gateway-sidecar/design.md`.
- `tasks.md` already commits the implementation to `attach.json`, so leaving the schema implicit will force implementers to invent incompatible payloads.
- The design still wants future manual sessions to publish “the same attach contract later,” which means today’s schema should either reserve that path intentionally or explicitly reject it for now.

### Examples

- A runtime-owned CAO session publishes `AGENTSYS_GATEWAY_ATTACH_PATH=/tmp/agents-runtime/sessions/cao_rest/cao_rest-20260312-120000Z-abcd1234/gateway/attach.json`; a lifecycle command must know exactly which fields are guaranteed there before it can attach.
- A future manually launched tmux session may have no gig-agents session manifest at all, but still needs to communicate enough information for the gateway to observe the session and choose the right backend adapter.

### Why this question arises in code

- Strict boundary payloads already exist for persisted runtime state in `SessionManifestPayloadV2` at `src/gig_agents/agents/brain_launch_runtime/boundary_models.py:162`, but there is no equivalent attach-contract model yet.
- Runtime name-based discovery already relies on strict tmux env pointer validation in `_read_tmux_session_env_var()` and `_resolve_manifest_path_from_tmux_session()` at `src/gig_agents/agents/brain_launch_runtime/runtime.py:813` and `src/gig_agents/agents/brain_launch_runtime/runtime.py:868`.
- The mailbox path shows the repo’s preferred pattern for “resolved config -> env bindings -> persisted state” via `resolve_effective_mailbox_config()` and `mailbox_env_bindings()` in `src/gig_agents/agents/mailbox_runtime_support.py:154` and `src/gig_agents/agents/mailbox_runtime_support.py:264`.

### Options

**Option A: Define one small unified v1 attach-contract schema with a required core plus optional runtime-owned fields**

- Code impact:

```python
class GatewayAttachContractV1(_StrictBoundaryModel):
    schema_version: int
    attach_identity: str
    backend: Literal["cao_rest", "codex_headless", "claude_headless", "gemini_headless"]
    tmux_session_name: str
    working_directory: str
    manifest_path: str | None = None
    agent_def_dir: str | None = None
    runtime_session_id: str | None = None
    desired_host: str | None = None
    desired_port: int | None = None
    backend_metadata: JsonObject
```

- Code impact:
  - Runtime-owned sessions write this schema now.
  - Future manual sessions must satisfy the same core fields but can omit runtime-only pointers such as `manifest_path`.
- User-facing difference:
  - One documented contract exists for both runtime-owned and future manual sessions.
- Pros:
  - Keeps attachability discoverable and versionable from the start.
  - Avoids a second incompatible attach format later.
  - Matches the repo’s strict boundary-model style.
- Cons:
  - Needs careful field-tiering now so the runtime does not over-promise manual-session support in v1.

**Option B: Define a runtime-owned-only schema now and leave manual-session compatibility for a future contract**

- Code impact:

```python
class RuntimeOwnedGatewayAttachContractV1(_StrictBoundaryModel):
    schema_version: int
    runtime_session_id: str
    manifest_path: str
    agent_def_dir: str | None = None
    backend: Literal["cao_rest", "codex_headless", "claude_headless", "gemini_headless"]
    backend_metadata: JsonObject
```

- Code impact:
  - Simpler v1 parser and fewer optional fields.
  - Future manual sessions need a separate schema or a schema bump.
- User-facing difference:
  - Clear v1 scope, but no credible “manual sessions can adopt this later” story yet.
- Pros:
  - Smaller implementation surface for the first version.
  - Tightly aligned to the runtime-owned use case.
- Cons:
  - Likely creates schema churn when manual support is added.
  - Weakens the design claim that manual sessions can adopt the same contract later.

**Option C: Define separate schemas now for runtime-owned sessions and manual sessions**

- Code impact:

```python
if attach_contract.kind == "runtime_owned":
    parse_runtime_owned_contract(payload)
elif attach_contract.kind == "manual":
    parse_manual_contract(payload)
else:
    raise GatewayAttachContractError("unsupported attach contract kind")
```

- Code impact:
  - More explicit branches and potentially cleaner manual-session support later.
  - More spec, validation, and test overhead immediately.
- User-facing difference:
  - More explicit future extensibility, but more moving parts in v1.
- Pros:
  - Avoids overloading one shape with too many optional fields.
  - Makes future manual support an explicit branch instead of an implicit omission.
- Cons:
  - Premature complexity for a v1 that only implements runtime-owned live attach.
  - Increases the chance of drift between the two schemas.

### Proposal (Recommended)

Take **Option A**. Define one small versioned `attach.json` schema now, with a required core that is meaningful for both runtime-owned and future manual sessions, and an optional runtime-owned section for manifest-path and other runtime-only data. That keeps v1 implementable without pretending manual sessions are already supported, while still letting the schema grow by extension rather than replacement.

### Pros / Cons (Proposal)

- Pros:
  - Preserves one attachability contract across runtime-owned and future manual sessions.
  - Fits the repo’s preference for strict, versioned boundary payloads.
  - Makes validation and operator docs concrete.
- Cons:
  - Requires deciding field ownership now instead of letting implementation infer it ad hoc.

> **DECISION: Accept Option A with one versioned `attach.json` schema that has a required shared core and optional runtime-owned fields.**
> Rationale: The runtime already treats persisted boundaries as strict contracts through `_StrictBoundaryModel` and `SessionManifestPayloadV2` in [src/gig_agents/agents/brain_launch_runtime/boundary_models.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/boundary_models.py#L24) and [src/gig_agents/agents/brain_launch_runtime/boundary_models.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/boundary_models.py#L162), while tmux discovery already rejects missing or malformed env pointers in [_read_tmux_session_env_var()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/runtime.py#L813) and [_resolve_manifest_path_from_tmux_session()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/runtime.py#L868). The mailbox flow also shows the repo’s preferred pattern of “resolve config, publish env bindings, then persist stable state” in [resolve_effective_mailbox_config()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/mailbox_runtime_support.py#L154) and [mailbox_env_bindings()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/mailbox_runtime_support.py#L264). This decision requires adding a strict attach-contract model now, with required fields that manual sessions can satisfy later and runtime-owned extras such as `manifest_path` and `runtime_session_id` kept optional instead of splitting the contract.

### Notes / Follow-ups

- The follow-up revision should decide whether `backend_metadata` is itself backend-tagged typed payloads or a looser opaque object in v1, and should keep the runtime-owned path examples aligned with the nested session-root layout.
- If this option is accepted, `design.md`, `specs/brain-launch-runtime/spec.md`, `specs/agent-gateway/spec.md`, and `tasks.md` should all pin the same required fields and validation rules.

## Q2) When `stop-session` tears down a runtime-owned agent with a live attached gateway, should it stop the gateway too?

- Decision: `Blocking`

### Why this matters

- The change now supports independent `attach` and `detach`, but runtime-owned teardown still has no decided relationship to the attached gateway process.
- If the runtime leaves the gateway alive after `stop-session`, it will need a clear orphaned-gateway status contract, cleanup behavior, and operator story.
- If the runtime stops the gateway automatically, that becomes a lifecycle guarantee that should be reflected in status transitions, logs, and tests.

### Examples

- An operator runs `start-session --backend cao_rest`, later attaches a gateway, then runs `stop-session`. Should the resulting state be “agent stopped, gateway also stopped” or “agent stopped, gateway still alive but reporting offline”?
- A script calls `stop-session --force-cleanup` for a headless tmux session. If the gateway survives, that script also has to know whether it must call a second lifecycle command to clean up the listener.

### Why this question arises in code

- `_cmd_stop_session()` simply resumes the session and calls `controller.stop(force_cleanup=...)` in `src/gig_agents/agents/brain_launch_runtime/cli.py:389`.
- `RuntimeSessionController.stop()` delegates to `backend_session.terminate()` and persists the manifest, but it has no gateway lifecycle branch yet at `src/gig_agents/agents/brain_launch_runtime/runtime.py:172`.
- The current design open question in `openspec/changes/add-agent-gateway-sidecar/design.md` already acknowledges this policy gap.

### Options

**Option A: Runtime-owned `stop-session` also stops any live attached gateway**

- Code impact:

```python
def stop_runtime_owned_session(...):
    if gateway_is_attached(session):
        stop_gateway(session, reason="agent_stop")
        write_offline_gateway_state(session)
        clear_live_gateway_bindings(session)
    terminate_backend_session(session)
```

- User-facing difference:
  - `stop-session` becomes the single authoritative teardown operation for runtime-owned sessions.
- Pros:
  - Prevents orphaned listeners and stale live bindings.
  - Aligns with the current meaning of `stop-session` as “terminate backend resources.”
  - Simplifies operator expectations and cleanup scripts.
- Cons:
  - Reduces lifecycle independence during agent teardown.
  - Means “keep gateway alive to inspect offline state” is not the default behavior.

**Option B: Leave the gateway running and let it report offline or degraded state**

- Code impact:

```python
def stop_runtime_owned_session(...):
    terminate_backend_session(session)
    # gateway keeps running and detects agent loss
    gateway_updates_state_to_offline(session)
```

- User-facing difference:
  - After `stop-session`, the gateway may still answer `GET /health` and `GET /v1/status`.
- Pros:
  - Maximizes gateway independence.
  - Preserves a live process that can surface last-known gateway state without restart.
- Cons:
  - Leaves extra processes and listeners around after session teardown.
  - Makes cleanup, operator docs, and test flows more surprising.

**Option C: Make teardown behavior configurable**

- Code impact:

```python
stop.add_argument("--keep-gateway", action="store_true")
...
if keep_gateway:
    terminate_backend_only()
else:
    terminate_backend_and_gateway()
```

- User-facing difference:
  - More flexible, but every caller must understand the new flag.
- Pros:
  - Lets power users choose between strict cleanup and post-stop inspection.
- Cons:
  - Adds CLI and lifecycle complexity before v1 behavior is proven.
  - Creates more test branches immediately.

### Proposal (Recommended)

Take **Option A** for v1. The runtime already owns `stop-session`, the managed manifest, and the tmux-backed session lifecycle, so it should also own cleaning up any attached gateway for that same runtime-owned session. If later we want a more flexible operator mode, it is easier to add an explicit “keep gateway alive” variant than to retroactively tighten a surprising default.

### Pros / Cons (Proposal)

- Pros:
  - Produces one predictable teardown path for runtime-owned sessions.
  - Avoids orphan processes and stale live env pointers.
  - Keeps detach as the explicit operation for “stop gateway while agent stays alive.”
- Cons:
  - Sacrifices some lifecycle independence during full session teardown.

> **DECISION: Accept Option A: runtime-owned `stop-session` must also stop any live attached gateway and write gateway state back to offline or not-attached.**
> Rationale: The current CLI exposes `stop-session` as one authoritative operator command in [_cmd_stop_session()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/cli.py#L389), and `RuntimeSessionController.stop()` currently owns backend termination without any second cleanup path in [src/gig_agents/agents/brain_launch_runtime/runtime.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/runtime.py#L172). Leaving a live gateway behind would create an orphaned listener outside the current runtime lifecycle contract and would force every stop caller to learn an extra teardown path. This decision implies `stop()` needs a gateway lifecycle branch that clears live bindings, updates `state.json`, and then terminates backend resources as part of the same runtime-owned shutdown flow.

### Notes / Follow-ups

- If this proposal is accepted, the specs should say that runtime-owned teardown writes `state.json` back to offline or not-attached state and clears live gateway bindings.
- Manual-session teardown can remain a separate question later because gig-agents does not own those processes the same way.

## Q3) If launch-time auto-attach fails after the agent session has already started, should the runtime keep the agent running or tear it down?

- Decision: `Blocking`

### Why this matters

- The current specs already say this outcome must be a defined runtime policy, but no policy has been chosen yet.
- Because the revised design decouples gateway lifecycle from agent lifecycle, destroying the agent after a gateway-only failure is no longer an obviously correct default.
- The choice affects the `start-session` CLI contract, exit codes, operator troubleshooting, and what data the CLI returns when startup only partially succeeds.

### Examples

- `start-session` launches a CAO session successfully, but the gateway attach step fails because the resolved port is already in use.
- A headless tmux-backed session starts and publishes attachability metadata, but the gateway process crashes during its own startup probe.

### Why this question arises in code

- `start-session` currently has a simple “start backend, then print JSON payload” success path in `_cmd_start_session()` inside `src/gig_agents/agents/brain_launch_runtime/cli.py`.
- `start_runtime_session()` in `src/gig_agents/agents/brain_launch_runtime/runtime.py:242` creates the backend session and persists the manifest before any future gateway lifecycle code will run.
- The new design intentionally treats auto-attach as a convenience path layered on top of session startup, which makes “tear down the already-started agent” a policy choice instead of a structural requirement.

### Options

**Option A: Keep the agent session running and return a structured partial-start failure**

- Code impact:

```python
session = start_backend_session(...)
publish_gateway_capability(session)
try:
    attach_gateway(session)
except GatewayAttachError as exc:
    return PartialStartResult(
        exit_code=2,
        session_started=True,
        session_manifest=session.manifest_path,
        agent_identity=session.agent_identity,
        gateway_error=str(exc),
    )
```

- User-facing difference:
  - `start-session` reports failure, but still tells the caller where the live agent session is so they can retry attach or stop the session.
- Pros:
  - Respects the design’s optional-gateway model.
  - Avoids destroying a successfully started interactive session because of a listener issue.
  - Produces better operator recovery options.
- Cons:
  - Makes startup failure semantics more nuanced.
  - Requires a partial-success output shape.

**Option B: Tear down the already-started agent session automatically**

- Code impact:

```python
session = start_backend_session(...)
publish_gateway_capability(session)
try:
    attach_gateway(session)
except GatewayAttachError:
    stop_backend_session(session)
    raise
```

- User-facing difference:
  - `start-session` behaves as all-or-nothing when auto-attach is requested.
- Pros:
  - Simpler success/failure story for callers.
  - Avoids leaving unexpected live sessions behind.
- Cons:
  - Conflicts with the “gateway is optional” design direction.
  - Risks tearing down interactive state the operator might want to keep.

**Option C: Add a configurable policy such as `--gateway-fail-open` vs `--gateway-fail-closed`**

- Code impact:

```python
if gateway_auto_attach_requested:
    if gateway_failure_policy == "fail_closed":
        stop_backend_session(session)
    else:
        keep_backend_session_running(session)
```

- User-facing difference:
  - Flexible but more complex startup UX.
- Pros:
  - Lets operators choose the policy per invocation.
- Cons:
  - Introduces extra CLI and documentation complexity before v1 behavior is proven.
  - Multiplies the integration test matrix immediately.

### Proposal (Recommended)

Take **Option A** for v1. The revised change deliberately treats the gateway as optional and attachable after startup, so a gateway-only failure should not implicitly destroy a successfully started agent session. The CLI should instead return a structured partial-start failure that includes the live session identity and gateway error so operators can retry attach, inspect the running session, or stop it explicitly.

### Pros / Cons (Proposal)

- Pros:
  - Best matches the decoupled lifecycle model now documented for the change.
  - Preserves operator control and reduces destructive surprise.
  - Gives scripts enough information to recover deliberately.
- Cons:
  - Requires the CLI output contract to model partial success explicitly.

> **DECISION: Accept Option A: if launch-time auto-attach fails after the agent has started, keep the agent running and return a structured partial-start failure.**
> Rationale: `start_runtime_session()` currently creates the backend session, resolves the runtime-owned `session_id`, and persists the manifest before returning in [src/gig_agents/agents/brain_launch_runtime/runtime.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/runtime.py#L242), so any future gateway auto-attach step is layered on top of an already-live session rather than being a prerequisite for backend creation. The CLI’s current start path in [src/gig_agents/agents/brain_launch_runtime/cli.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/cli.py#L328) only models full success today, but tearing down a successfully started interactive session because an optional gateway listener failed would directly conflict with the new attach-later design. This decision requires a new partial-start result shape and exit code that still returns the live session manifest or identity so operators can retry attach or stop explicitly.

### Notes / Follow-ups

- If accepted, the proposal/specs/tasks should decide the exact `start-session` output shape and exit code for partial-start failures.
- This question is independent from whether future gateway-aware control commands may gain an explicit `ensure-attached` mode later.

## Q4) How strict should blueprint parsing and validation become for gateway configuration in v1?

- Decision: `Blocking`

### Why this matters

- The current spec requires invalid `gateway.host` and `gateway.port` values to be rejected, but the current loader still only checks `name`, `role`, and `brain_recipe`.
- Without a decision on parser strictness, v1 could end up half-strict: some bad gateway fields rejected, other unknown fields silently ignored.
- The answer also affects whether blueprint validation follows the repo’s broader boundary-model style or remains a special case.

### Examples

- A blueprint includes `gateway.port: "43123"` as a string. Should that be accepted with coercion, rejected, or ignored?
- A blueprint includes an unexpected top-level field such as `gateway_mode: auto`. If the loader ignores it silently, an operator may think they configured behavior that does not exist.

### Why this question arises in code

- `load_blueprint()` currently reads a raw mapping and only validates `schema_version`, `name`, `role`, and `brain_recipe` in `src/gig_agents/agents/brain_launch_runtime/loaders.py:85`.
- The repo already has strict boundary parsing through `_StrictBoundaryModel` in `src/gig_agents/agents/brain_launch_runtime/boundary_models.py:24`.
- The project guidance explicitly allows breaking changes in pursuit of a clearer design, so preserving today’s permissive blueprint behavior is not required.

### Options

**Option A: Introduce a strict typed blueprint schema and reject unknown top-level or nested fields**

- Code impact:

```python
class BlueprintGatewayV1(_StrictBoundaryModel):
    host: Literal["127.0.0.1", "0.0.0.0"] | None = None
    port: int | None = None

class BlueprintBindingV1(_StrictBoundaryModel):
    schema_version: Literal[1]
    name: str
    role: str
    brain_recipe: str
    gateway: BlueprintGatewayV1 | None = None
```

- Code impact:
  - `load_blueprint()` becomes model-based rather than ad hoc.
  - Unknown fields are explicit errors instead of silently ignored data.
- User-facing difference:
  - Blueprint mistakes are caught early and clearly.
- Pros:
  - Aligns with the repo’s strict boundary-model pattern.
  - Matches the spec’s “invalid values are rejected” requirement.
  - Avoids silent operator misconfiguration.
- Cons:
  - Can reject previously ignored fields in existing local blueprints.

**Option B: Validate only the gateway subtree and continue ignoring other unknown fields**

- Code impact:

```python
payload = _load_mapping_file(path)
validate_known_gateway_fields(payload.get("gateway"))
return BlueprintBinding(...)
```

- Code impact:
  - Smaller code change and lower immediate migration surface.
  - Keeps overall blueprint parsing loose.
- User-facing difference:
  - Gateway mistakes are caught, but other unknown fields are still silently ignored.
- Pros:
  - Smaller change to existing loader behavior.
  - Lower immediate breakage risk for local experimental fields.
- Cons:
  - Keeps blueprints inconsistent with the repo’s strict persisted-boundary style.
  - Leaves room for silent non-gateway misconfiguration.

**Option C: Keep blueprints loose and validate only when gateway attach is attempted**

- Code impact:

```python
blueprint = load_blueprint_loose(path)
...
if gateway_attach_requested:
    validate_gateway_defaults(blueprint.gateway)
```

- Code impact:
  - Fewest upfront parser changes.
  - More attach-time error branches and weaker validation locality.
- User-facing difference:
  - Errors surface later and only in some control paths.
- Pros:
  - Minimal parser churn.
- Cons:
  - Conflicts with the current spec.
  - Makes debugging configuration errors harder.

### Proposal (Recommended)

Take **Option A**. Gateway defaults are becoming part of the change’s declared operator-facing contract, so blueprints should move onto an explicit strict schema rather than preserving today’s permissive loader behavior. That is the cleanest fit with `_StrictBoundaryModel`, and the repo’s current “breaking changes are allowed” guidance makes this a good time to tighten the contract instead of adding another special-case parser.

### Pros / Cons (Proposal)

- Pros:
  - Produces deterministic and early configuration errors.
  - Aligns blueprint handling with the repo’s general strict-boundary approach.
  - Makes specs, implementation, and tests much easier to keep in sync.
- Cons:
  - May reject previously tolerated experimental blueprint keys.

> **DECISION: Accept Option A: move blueprint loading to a strict typed schema that rejects unknown top-level and nested gateway fields.**
> Rationale: `load_blueprint()` currently performs ad hoc field checks and silently ignores everything except `schema_version`, `name`, `role`, and `brain_recipe` in [src/gig_agents/agents/brain_launch_runtime/loaders.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/loaders.py#L85), which is too loose for gateway defaults that are becoming part of the operator-facing contract. The rest of the runtime already uses strict boundary parsing via `_StrictBoundaryModel` in [src/gig_agents/agents/brain_launch_runtime/boundary_models.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/boundary_models.py#L24), and the repository guidance explicitly allows breaking changes to get to a clearer design. This decision implies adding a typed blueprint payload model and rejecting invalid or unknown gateway keys during load rather than deferring those errors to attach time.

### Notes / Follow-ups

- If accepted, the artifact revision should say whether `gateway.host` and `gateway.port` are the only supported gateway keys in v1.
- The implementation should also decide whether blueprint validation gets a dedicated command or remains part of load-time failure behavior.

## Q5) What exact rule should runtime-owned gateway clients use to validate live bindings before trusting them?

- Decision: `Blocking`

### Why this matters

- The design already says tmux-published live bindings are hints, not ground truth, but it does not yet pin a single validation rule.
- Without one rule, different control paths may treat stale bindings differently, which is especially risky once `send-prompt`, `interrupt`, and status reads begin using the gateway.
- The chosen rule will determine how “no live gateway attached” is detected and when callers should surface retry or attach guidance.

### Examples

- `AGENTSYS_AGENT_GATEWAY_HOST`, `AGENTSYS_AGENT_GATEWAY_PORT`, and `AGENTSYS_GATEWAY_STATE_PATH` still exist in tmux env after the gateway crashed.
- A session has valid stable attach pointers, but the live bindings point to an old port while `desired-config.json` and `run/current-instance.json` show newer information.

### Why this question arises in code

- Name-based resolution today already treats tmux env pointers as explicit inputs and validates them strictly in `src/gig_agents/agents/brain_launch_runtime/runtime.py:813`.
- The gateway spec now guarantees `GET /health` as a structured health endpoint and `GET /v1/status` as the stable status surface.
- The current CLI prompt and control commands (`_cmd_send_prompt()` and `_cmd_send_keys()` in `src/gig_agents/agents/brain_launch_runtime/cli.py:345` and `src/gig_agents/agents/brain_launch_runtime/cli.py:366`) do not yet have a gateway-client layer, so the validation rule has not been baked into code yet.

### Options

**Option A: Health-first validation with file checks as supporting evidence**

- Code impact:

```python
live = read_live_gateway_bindings(session_name)
stable = read_stable_attach_metadata(session_name)

if not live:
    raise NoLiveGatewayError(...)

validate_absolute_paths(live)
validate_protocol_version(live.protocol_version, stable.protocol_version)

if not http_get_health(live.host, live.port).ok:
    raise NoLiveGatewayError(...)
```

- Code impact:
  - All gateway-aware control paths share one preflight validator.
  - `run/current-instance.json` or `state.json` can still help produce better diagnostics after health failure.
- User-facing difference:
  - Callers get a consistent “no live gateway attached” outcome when the process is gone, instead of seeing arbitrary connection errors.
- Pros:
  - Uses the health endpoint the gateway contract already defines.
  - Produces one clear rule for all clients.
  - Keeps tmux env as discovery hints rather than trusted truth.
- Cons:
  - Adds one HTTP preflight round trip before control or status operations.

**Option B: File-first validation with no mandatory health probe**

- Code impact:

```python
live = read_live_gateway_bindings(session_name)
current_instance = load_run_state(...)
if current_instance.pid_is_alive():
    trust_live_bindings()
else:
    raise NoLiveGatewayError(...)
```

- Code impact:
  - Depends more heavily on pid and run-state correctness.
  - May vary across process supervisors and crash modes.
- User-facing difference:
  - Faster local checks, but more ambiguous failure modes when files are stale.
- Pros:
  - Avoids an extra HTTP round trip.
  - Works even if health endpoint is temporarily slow.
- Cons:
  - Less authoritative than the listener’s own health response.
  - More room for divergent heuristics across callers.

**Option C: Optimistically trust live bindings and let each request fail naturally**

- Code impact:

```python
client = GatewayClient.from_tmux_env(...)
return client.post_request(...)
```

- Code impact:
  - Simplest code path initially.
  - Pushes stale-binding behavior into many call sites.
- User-facing difference:
  - Users may see raw connection failures instead of a consistent “no live gateway” result.
- Pros:
  - Minimal upfront implementation work.
- Cons:
  - Conflicts with the design’s “bindings are hints” principle.
  - Makes operator UX and retry behavior inconsistent.

### Proposal (Recommended)

Take **Option A**. Runtime-owned gateway clients should use a single validation helper that reads stable attachability metadata, validates the live binding pointers structurally, checks protocol-version compatibility, and then treats `GET /health` as the authoritative “is there a live gateway instance here?” answer. That keeps the control plane explicit and makes stale tmux env bindings degrade cleanly into the already-documented “no live gateway attached” state.

### Pros / Cons (Proposal)

- Pros:
  - Produces one stable validation rule across prompt, interrupt, and status clients.
  - Best aligns with the explicit health endpoint already in the v1 contract.
  - Gives the clearest operator-facing error story.
- Cons:
  - Slightly increases per-operation validation cost.

> **DECISION: Accept Option A: runtime-owned gateway clients must validate live bindings structurally and then treat `GET /health` as the authoritative liveness check.**
> Rationale: The runtime already treats tmux-published pointers as inputs that must be validated before use, not as trusted truth, in [_read_tmux_session_env_var()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/runtime.py#L813) and [_resolve_manifest_path_from_tmux_session()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/runtime.py#L868). The gateway-aware control paths do not exist yet, but the current direct-control entrypoints in [_cmd_send_prompt()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/cli.py#L345), [_cmd_send_keys()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/cli.py#L366), and the underlying controller methods [send_prompt()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/runtime.py#L120) and [interrupt()](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/runtime.py#L127) show why a single shared validator is needed before gateway routing is added. This decision implies one preflight helper that verifies stable pointers and protocol compatibility, uses `/health` to decide whether a live gateway is really present, and collapses stale bindings into a consistent “no live gateway attached” outcome for v1.

### Notes / Follow-ups

- If accepted, the artifacts should also say when clients may skip the health preflight, if ever, for pure status reads against a known-local gateway.
- The implementation should decide whether validation failures emit separate error codes for “stale bindings,” “protocol mismatch,” and “listener unreachable,” or collapse them into one no-live-gateway surface in v1.
