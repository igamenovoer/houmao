## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What provider hooks (if any) does the Gemini unattended strategy require?

- Decision: `Blocking`

#### Why this matters

- Claude requires 2 hooks (`ensure_api_key_approval`, `ensure_project_trust`) and Codex requires 4 hooks. These hooks handle config-file mutations and state seeding that CLI args alone cannot cover. If Gemini needs similar state preparation (e.g., onboarding bypass, trust seeding, config initialization), omitting hooks would cause the strategy's actions to be incomplete and unattended launches to fail on fresh homes.

#### Examples

- If Gemini CLI checks for an onboarding/first-run state file, the strategy must seed it (like Claude's `hasCompletedOnboarding` in `.claude.json`).
- If Gemini CLI has a trust or permissions model, the strategy must pre-approve the working directory.
- If Gemini requires no such state and works with just CLI args + env vars, the strategy can use zero hooks.

#### Why this question arises in code

- `provider_hooks.py` currently handles `claude.*` (lines 29-300) and `codex.*` (lines 300-587) hooks. A `provider_hook.call` action in the strategy YAML triggers `run_provider_hook()`, which dispatches by prefix.
- The Gemini headless backend (`gemini_headless.py`) does not show any provider-side config mutations, suggesting Gemini may not need hooks. But the backend only covers runtime session behavior, not fresh-home initialization.
- The design says "owned startup surfaces or provider hooks" (decision 2) without resolving which applies.

#### Options

**Option A: Gemini strategy uses zero provider hooks (CLI args + file-set actions only)**

- Code impact:
  - No changes to `provider_hooks.py`
  - Strategy YAML uses only `cli_arg.ensure_present`, `json.set`/`toml.set`, and `validate.reject_conflicting_launch_args` action kinds
- User-facing difference:
  - Simplest implementation; fewer failure modes during strategy application
- Pros:
  - Lower complexity; faster to implement and test
  - If Gemini CLI truly needs no config mutations, this is correct by construction
- Cons:
  - If Gemini CLI does need fresh-home state, this will be discovered as a runtime failure during testing, not at design time

**Option B: Add Gemini-specific hooks for credential readiness and any discovered config state**

- Code impact:
  - Add `gemini.*` hook implementations in `provider_hooks.py`
  - Strategy YAML includes `provider_hook.call` actions with `gemini.*` hook IDs
  - Hook dispatch in `run_provider_hook()` gains a `gemini.*` branch
- User-facing difference:
  - More robust fresh-home support; parity with Claude/Codex strategies
- Pros:
  - Handles any discovered Gemini startup requirements
  - Maintains architectural consistency across all three tool strategies
- Cons:
  - Requires discovery work to identify what hooks are needed
  - Adds code that may be trivial if Gemini needs no config mutations

> **PROPOSED: Start with Option A and promote to Option B if live testing reveals fresh-home failures.**
> Rationale: The existing `GeminiHeadlessSession` backend (`backends/gemini_headless.py`) shows no config-file mutations, suggesting Gemini CLI may not require provider-side state initialization. The reserved args (`-p`, `--output-format`, `--resume`) are simpler than Claude's or Codex's, and Gemini auth is purely env-based (no config-file credential storage like Codex's `auth.json`). Starting with zero hooks keeps the strategy minimal and lets live testing expose any gaps. If fresh-home testing fails, adding a `gemini.validate_credential_readiness` hook follows the same pattern as Codex.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Accept PROPOSED. The current Gemini runtime path is already mostly env-driven and the codebase shows no known Gemini-owned startup state analogous to Claude `.claude.json` or Codex `config.toml`. The minimal maintained contract here is to prove whether zero hooks is sufficient on a fresh home, then add only the smallest `gemini.*` hook set if live testing demonstrates a real gap.

### Q2) What Gemini CLI version range should the registry strategy declare?

- Decision: `Blocking`

#### Why this matters

- The launch-policy engine requires `supported_versions` as a PEP-440 specifier to match against `gemini --version` output. Without a known-good lower bound, the strategy cannot be written.

#### Examples

- If Gemini CLI v1.0.0+ supports `-p` (programmatic/headless) mode reliably, declare `>=1.0.0`.
- If only specific recent versions work, the range may need an upper bound (e.g., `>=0.5.0,<2.0`).

#### Why this question arises in code

- `engine.py:detect_tool_version()` runs `gemini --version` and extracts a semantic version via regex `(\d+\.\d+\.\d+)`.
- `engine.py:resolve_strategy()` checks `detected_version in strategy.supported_versions`.
- The design says "version-scoped" but never names a version.

#### Options

**Option A: Use a broad lower bound based on live testing (e.g., `>=0.1.0`)**

- Code impact:
  - Single strategy entry covers all known versions
- User-facing difference:
  - Widest compatibility; any installed Gemini CLI that matches will be accepted
- Pros:
  - Simple; avoids version-gating prematurely
- Cons:
  - May accept Gemini versions that don't actually support unattended mode reliably

**Option B: Pin to the version known during implementation, with evidence**

- Code impact:
  - Strategy declares tight lower bound based on tested version (e.g., `>=X.Y.Z`)
  - Evidence section documents the specific version tested
- User-facing difference:
  - Tighter; rejects untested older versions
- Pros:
  - Matches the evidence-backed pattern used by Claude (`>=2.1.81`) and Codex (`>=0.116.0`)
  - Fail-closed is correct for untested versions
- Cons:
  - Requires knowing the installed Gemini CLI version at implementation time

> **PROPOSED: Option B - pin to the version available at implementation time and document it as live_probe evidence.**
> Rationale: Both existing strategies (Claude `>=2.1.81`, Codex `>=0.116.0`) use specific tested lower bounds, not broad ranges. The registry's value is in declaring what has actually been verified. Pinning matches the established pattern. The implementer should run `gemini --version` on the development machine, test unattended startup at that version, and use it as the lower bound with a `live_probe` evidence entry.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Accept PROPOSED. This matches the existing Claude and Codex registry pattern and preserves fail-closed behavior for untested Gemini versions. The change artifacts should not invent a lower bound before implementation has actually validated one.

### Q3) Which Gemini auth fixture should be the canonical demo lane?

- Decision: `Deferrable`

#### Why this matters

- Design decision 4 says the canonical demo lane should use OAuth (`oauth_creds.json`), but the fixture directory contains both `api-key-a-default/` and `personal-a-default/` under `tests/fixtures/agents/tools/gemini/auth/`. The demo_parameters.json must reference one fixture by name.

#### Why this question arises in code

- `demo_parameters.json` requires an `auth_fixture_dir` path per tool.
- `runtime.py:import_project_auth_from_fixture()` reads `env/vars.env` and auth files from that fixture directory.
- The fixture choice determines which auth family the canonical demo exercises.

#### Options

**Option A: Use `personal-a-default` (OAuth fixture) as the canonical demo lane**

- Code impact:
  - `demo_parameters.json` references `tests/fixtures/agents/tools/gemini/auth/personal-a-default`
  - Auth import code must handle `oauth_creds.json`
- User-facing difference:
  - Demo exercises the OAuth path per design decision 4
- Pros:
  - Aligns with design decision 4's stated preference for OAuth-backed validation
  - Exercises the richer auth pathway
- Cons:
  - OAuth credentials may be harder to set up for new operators

**Option B: Use `api-key-a-default` (API-key fixture) as the canonical demo lane**

- Code impact:
  - `demo_parameters.json` references `tests/fixtures/agents/tools/gemini/auth/api-key-a-default`
  - Auth import code handles env-only credentials
- User-facing difference:
  - Demo exercises the simpler API-key path
- Pros:
  - Lower friction for operators setting up the demo
  - API-key auth is more commonly available
- Cons:
  - Under-exercises the OAuth unattended path that the design explicitly wants to validate

> **PROPOSED: Use `personal-a-default` (OAuth) as the canonical demo lane, matching design decision 4.**
> Rationale: Design decision 4 explicitly says "the canonical supported demo lane should use the existing OAuth-backed fixture" and rejects API-key-only because "it would under-exercise the newly maintained OAuth-backed unattended path." The `personal-a-default` fixture appears to be the OAuth lane based on naming convention. The runtime importer should still accept both auth families for manual variation, as the design states.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Accept PROPOSED. The current fixture contents confirm that `personal-a-default` is the OAuth lane (`oauth_creds.json` plus `GOOGLE_GENAI_USE_GCA=true`), while `api-key-a-default` is the API-key lane. The supported demo should validate the richer unattended path, while importer logic can still accept the API-key fixture for manual variation.
