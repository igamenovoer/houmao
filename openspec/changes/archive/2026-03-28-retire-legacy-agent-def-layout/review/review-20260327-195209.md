## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What happens to `tests/fixtures/agents/brains/api-creds.tar.gz.gpg`?

- Decision: `Blocking`

#### Why this matters

The encrypted credential archive at `tests/fixtures/agents/brains/api-creds.tar.gz.gpg` is a tracked file under the legacy `brains/` tree. If Phase 3 (task 3.1) removes or restructures `brains/`, this archive either needs to be relocated, re-encrypted at a canonical path, or explicitly exempted.

#### Examples

- If `brains/` is deleted entirely, the archive disappears from the repo and any restore/provisioning scripts that source it will break.
- If `brains/` is kept as a compatibility-only stub with only the archive, the "no legacy mirror directories" requirement is technically violated.

#### Why this question arises in code

- `tests/fixtures/agents/brains/api-creds.tar.gz.gpg` is a tracked git blob in the legacy tree.
- The `.env` file selected by the user references `AGENT_CREDENTIAL_COMPRESS_PASSWORD`, suggesting an active decrypt/provision workflow that depends on this archive.
- No task, spec, or design paragraph addresses encrypted archives specifically.

#### Options

**Option A: Relocate the encrypted archive to `tests/fixtures/agents/tools/` (or a new `tests/fixtures/agents/credentials/` directory) and update any provision/decrypt scripts.**

- Code impact:
  - Move the `.tar.gz.gpg` file and update any scripts that reference it (likely `setup-envs.sh` or similar provisioning helpers).
- User-facing difference:
  - Provisioning workflow uses a new path; documented in migration notes.
- Pros:
  - Fully removes the legacy `brains/` tree.
  - Aligns the encrypted archive with the canonical `tools/<tool>/auth/` target layout.
- Cons:
  - Requires re-encrypting or at least relocating+updating provision scripts in the same change.

**Option B: Keep `brains/api-creds.tar.gz.gpg` in place and document it as a provisioning-only artifact exempt from the layout migration.**

- Code impact:
  - No file move; add an explicit exemption note in the fixture README or MIGRATION.md.
- User-facing difference:
  - `brains/` directory continues to exist but only for the encrypted archive.
- Pros:
  - No risk of breaking the credential provisioning workflow.
  - Smallest implementation delta.
- Cons:
  - Legacy `brains/` directory survives in the fixture tree, weakening the "no legacy mirrors" contract.

> **PROPOSED: Option A — relocate the archive to a canonical path.**
> Rationale: The `component-agent-construction` delta spec explicitly says tracked trees "SHALL NOT require or ship legacy `brains/` … directories as parallel source-of-truth mirrors." Leaving the encrypted archive under `brains/` violates that contract. Relocation is a single file move plus a grep-and-update of any provision scripts, which is low risk and keeps the migration clean.
>
> **DECISION: Refine Option A — retire the legacy `tests/fixtures/agents/brains/api-creds.tar.gz.gpg` archive and standardize on the existing canonical `tests/fixtures/agents/tools.tar.gz.enc` snapshot.**
> Rationale: The repo already documents `tests/fixtures/agents/tools.tar.gz.enc` as the encrypted snapshot of the canonical `tools/` tree, including `tools/<tool>/auth/**`. Keeping `brains/api-creds.tar.gz.gpg` would preserve a legacy-only subtree solely for credential provisioning, which conflicts with this change's no-legacy-mirror goal.

### Q3) Should the `cao-rest-client-contract` delta spec also update the main spec's credential-profile scenarios beyond just the demo-script requirement?

- Decision: `Deferrable`

#### Why this matters

The main spec at `openspec/specs/cao-rest-client-contract/spec.md` has multiple credential-profile scenarios referencing `agents/brains/api-creds/` as the authoritative path (lines ~123, 128, 133, 138, 148). The delta spec in this change only adds a modified requirement about demo scripts and uses `agents/tools/<tool>/auth/` there. After applying this change, the main spec would contain both `brains/api-creds/` references (in older scenarios) and `tools/<tool>/auth/` references (in the newly modified demo scenario), creating an internal inconsistency.

#### Examples

- Main spec scenario: "WHEN a developer runs the Codex CAO demo script with valid Codex/OpenAI credentials present under `agents/brains/api-creds/`" — this would remain after the delta is applied.
- Delta spec scenario: "WHEN a developer runs the Codex CAO demo script with valid Codex/OpenAI credentials present under `agents/tools/codex/auth/`" — this is the updated version.

#### Why this question arises in code

- The delta spec mechanism replaces/adds specific requirements but the main spec's other credential scenarios still say `brains/api-creds/`.
- The design says "Align shipped docs and OpenSpec requirements with the canonical layout" which implies all references should be updated, not just the demo-script requirement.

#### Options

**Option A: Expand the `cao-rest-client-contract` delta spec to also modify the credential-profile scenarios, replacing `agents/brains/api-creds/` with `agents/tools/<tool>/auth/` throughout.**

- Code impact:
  - Larger delta spec; modifies 4-5 additional requirement scenarios.
- User-facing difference:
  - Main spec becomes internally consistent after sync.
- Pros:
  - Eliminates the mixed-path inconsistency.
  - Matches the design's stated goal of aligning specs with canonical layout.
- Cons:
  - Larger spec change; more scenarios to verify.

**Option B: Leave the other credential-profile scenarios as-is and defer their update to a follow-up change.**

- Code impact:
  - No additional delta spec work.
- User-facing difference:
  - Main spec has mixed old/new path references until a follow-up lands.
- Pros:
  - Keeps this change focused on the demo/fixture/consumer migration.
- Cons:
  - Inconsistency survives in the main spec.

> **PROPOSED: Option A — expand the delta spec to cover all credential-profile scenarios.**
> Rationale: The design's stated goal (design.md §Goals) is to "Align shipped docs and OpenSpec requirements with the canonical layout." Leaving 4-5 credential scenarios referencing `brains/api-creds/` while updating only the demo-script scenario creates the same "inconsistent repository contract" the proposal identifies as the motivating problem. The extra delta lines are straightforward path substitutions.
>
> **DECISION: Reject Option A as additional delta expansion is not needed.**
> Rationale: The current `cao-rest-client-contract` delta already replaces the full `Live demo scripts prove end-to-end prompt processing with real providers` requirement, including the credential-path scenarios the review cites. The needed follow-up is to make sync and verification explicit so that updated requirement lands in the main spec, not to broaden the delta beyond that requirement.
