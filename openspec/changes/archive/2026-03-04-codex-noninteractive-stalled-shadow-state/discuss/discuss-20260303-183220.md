# Discussion: Codex Non-Interactive Bootstrap + Shadow Unknown/Stalled

- Feature/Change: `codex-noninteractive-stalled-shadow-state`
- Timestamp: `2026-03-03T18:32:20Z`
- Context: Capture remaining open questions from `proposal.md`, `design.md`, `tasks.md`, and `specs/*/spec.md` plus a quick scan of existing runtime code (`cao_rest.py`, shadow parsers, and Claude bootstrap) before implementing.

## Summary of recommended defaults

1) Emit dedicated anomaly code(s) for stalled entry/recovery (Q1).
2) Use a single `runtime.cao.shadow.unknown_to_stalled_timeout_seconds` threshold for both readiness and completion phases (Q2).
3) Keep Codex bootstrap policy-driven: always seed trust and required notice state; only apply approval/sandbox defaults if explicitly present in the Codex config profile (Q3).
4) Default `runtime.cao.shadow.unknown_to_stalled_timeout_seconds` to 30 seconds (Q4).

## Q1) Should recovery from stalled emit a dedicated anomaly code?

Blocking: Yes (impacts what diagnostics/operators can rely on, and what tests/fixtures must assert).

### Why this matters

- A "stalled then recovered" event is operationally different from a brief unknown blip or baseline drift; we want it to be searchable and reportable.
- The current runtime already serializes parser anomalies (`done.payload.parser_metadata.anomalies`), so we need to decide whether stalled lifecycle lives there or only in free-form diagnostics.

### Options

**Option A: Add dedicated anomaly codes for stalled lifecycle transitions**

- Pros:
  - Easy to grep/aggregate (`stalled_entered`, `stalled_recovered`, etc.).
  - Fits the existing anomaly plumbing (`ShadowParserAnomaly`) and serialization path.
- Cons:
  - More surface area (new codes + tests) and potential bikeshedding over naming.

**Option B: Reuse generic unknown-state anomalies only (no dedicated recovery code)**

- Pros:
  - Smaller contract surface area.
  - Fewer fixtures and doc updates.
- Cons:
  - Harder to distinguish "persistently unknown then recovered" from "momentary unknown".

### Proposal (Recommended)

Add dedicated anomaly code(s) for stalled lifecycle transitions, at minimum:

- `stalled_entered` with details: `phase={readiness|completion}`, `elapsed_unknown_seconds`, `parser_family`
- `stalled_recovered` with details: `phase={readiness|completion}`, `elapsed_stalled_seconds`, `parser_family`, `recovered_to={processing|idle|completed|waiting_user_answer}`

### Pros / Cons (Proposal)

- Pros:
  - Makes postmortems and live debugging simpler without parsing free-form text.
  - Keeps the "what happened" signal stable even if message phrasing changes.
- Cons:
  - Requires choosing stable code names and maintaining them long-term.

> **DECISION: Accept proposal — add dedicated `stalled_entered` and `stalled_recovered` anomaly codes.**
> Rationale: The existing `ShadowParserAnomaly` dataclass (`shadow_parser_core.py:21-27`) already carries `code`, `message`, and `details` fields that match the proposed structure exactly. Three anomaly codes are already defined as module-level constants (`shadow_parser_core.py:12-14`) and flow through a well-tested merge (`cao_rest.py:507-523`) and serialization (`cao_rest.py:491-504`) pipeline. Adding two more constants is low-effort and consistent with the established pattern. The `details` dict naturally accommodates `phase`, `elapsed_*_seconds`, `parser_family`, and `recovered_to` without schema changes. Dedicated codes are far more useful for automated aggregation than parsing free-form diagnostics.

## Q2) Do we want separate unknown-to-stalled thresholds for readiness vs completion wait?

Blocking: Yes (determines config surface area and how CAO polling state machines will be wired).

### Why this matters

- Readiness wait and completion wait can have different expected durations; using the same threshold may be too aggressive for one phase or too lax for the other.
- Adding multiple thresholds increases config complexity and validation surface area.

### Options

**Option A: Single threshold (current design) applied in both phases**

- Pros:
  - Minimal config surface area: one key to document and validate.
  - Matches existing pattern of a single `timeout_seconds` used for both readiness and completion loops.
- Cons:
  - Less tunable if real-world readiness and completion unknown behavior differ materially.

**Option B: Two explicit thresholds**

- Pros:
  - Lets operators tune readiness vs completion independently.
- Cons:
  - Expands config surface area (naming, defaults, schema validation, docs).
  - Raises questions about how to apply thresholds to other phases that may be added later.

### Proposal (Recommended)

Keep a single `runtime.cao.shadow.unknown_to_stalled_timeout_seconds` threshold for both readiness and completion phases. If we later find phase-specific needs, add phase-specific keys in a follow-up change.

### Pros / Cons (Proposal)

- Pros:
  - Faster to implement and reason about; fewer config pitfalls.
- Cons:
  - Could require a follow-up if phase-specific tuning becomes necessary.

> **DECISION: Accept proposal — use a single `unknown_to_stalled_timeout_seconds` threshold for both phases.**
> Rationale: The existing `CaoRestSession` already uses one `timeout_seconds` parameter (default 120s, `cao_rest.py:360`) as the hard cap for both `_wait_for_shadow_ready_status()` (`cao_rest.py:567-598`) and `_wait_for_shadow_completion()` (`cao_rest.py:600-665`). Both loops share the same `poll_interval_seconds` (`cao_rest.py:359`). Adding the stalled threshold as a single value mirrors this established single-timeout pattern. There is no empirical evidence yet that readiness and completion phases exhibit materially different unknown-state durations, so splitting prematurely would add config surface area without known benefit. Phase-specific keys can be added in a follow-up if real-world data warrants it.

## Q3) Should Codex bootstrap set approval/sandbox defaults, or only seed trust/notice state?

Blocking: Yes (changes runtime-owned bootstrap responsibilities and affects how non-interactive guarantees are achieved).

### Why this matters

- Codex config contains multiple knobs that can trigger interactive prompts (trust, notices, approvals/sandbox warnings). Bootstrap needs to be explicit about which are runtime-owned.
- Seeding trust can also implicitly influence default sandbox/approval behavior, so "trust-only" is not always behaviorally minimal.

### Options

**Option A: Minimal bootstrap: seed trust + required notice state only**

- Pros:
  - Smallest scope and lowest risk of overreach into user policy.
  - Keeps "what is runtime-owned" tight and auditable.
- Cons:
  - Non-interactive guarantees may still depend on the chosen Codex config profile and defaults.

**Option B: Policy-driven bootstrap: apply trust + notice + approval/sandbox defaults sourced from config profile**

- Pros:
  - Makes non-interactive behavior explicit and repeatable across different runtime homes.
  - Keeps policy in config profiles instead of hardcoding in runtime logic.
- Cons:
  - Requires deciding exactly which config keys are in-scope for "runtime-enforced" edits.

**Option C: Hardcode non-interactive approval/sandbox defaults in runtime**

- Pros:
  - Most deterministic; no per-profile drift.
- Cons:
  - Highest risk of surprising behavior and coupling runtime logic to policy decisions.

### Proposal (Recommended)

Adopt policy-driven bootstrap:

- Always seed per-project trust for the resolved launch workspace target.
- Seed only the notice state needed to avoid onboarding/warning prompts for the chosen policy.
- Apply `approval_policy` / `sandbox_mode` defaults only if explicitly present in the Codex config profile (do not hardcode new defaults in runtime).

### Pros / Cons (Proposal)

- Pros:
  - Keeps runtime edits minimal while still making non-interactive policy configurable and explicit.
- Cons:
  - Still requires defining what "needed notice state" means and how to validate it.

> **DECISION: Accept proposal — adopt policy-driven bootstrap (Option B) for Codex.**
> Rationale: The Claude bootstrap at `claude_bootstrap.py:102-124` already implements a policy-driven pattern: `_overlay_runtime_enforced_keys()` seeds `hasCompletedOnboarding`, `numStartups`, and `customApiKeyResponses` — all derived from environment/config rather than hardcoded behavioral policy. The Codex side has a parallel structure: trust is seeded per-project via `config.toml` `[projects."<path>"].trust_level` (Codex `config/mod.rs:726-805`), and notice flags like `hide_full_access_warning` live in `[notice]` (Codex `config/types.rs:554-574`). The Codex shadow parser already detects trust prompts (`codex_shadow.py:54-56`) and approval prompts (`codex_shadow.py:51-52`), confirming that un-seeded trust/approval state causes interactive prompts that break non-interactive orchestration. Sourcing `approval_policy` and `sandbox_mode` from the config profile (rather than hardcoding) keeps the runtime bootstrap auditable and avoids coupling to specific Codex policy semantics.

## Q4) What should the default `runtime.cao.shadow.unknown_to_stalled_timeout_seconds` be?

Blocking: Yes (even with configurability, a default is part of the behavior contract and influences operator expectations).

### Why this matters

- Too low: frequent false "stalled" classification during transient UI states.
- Too high: operators see long periods of ambiguous "unknown" that look like hangs, defeating the purpose of the stalled lifecycle.

### Options

**Option A: 10 seconds**

- Pros:
  - Fast signal that something is wrong.
- Cons:
  - Likely to over-trigger on transient redraw/reflow states.

**Option B: 30 seconds**

- Pros:
  - Balances false positives vs time-to-signal in typical polling loops.
- Cons:
  - Still somewhat arbitrary; may need operator tuning for slow environments.

**Option C: 60 seconds**

- Pros:
  - Very conservative; unlikely to false-trigger.
- Cons:
  - Too slow to be useful for most "looks hung" operator experiences.

### Proposal (Recommended)

Default to 30 seconds.

### Pros / Cons (Proposal)

- Pros:
  - Provides a meaningful diagnostic transition well before the existing 120s polling timeout.
- Cons:
  - Could require retuning if real-world unknown periods are longer for certain Codex/Claude versions.

> **DECISION: Accept proposal — default `unknown_to_stalled_timeout_seconds` to 30 seconds.**
> Rationale: The existing hard timeout is 120 seconds (`cao_rest.py:360`), and both wait loops poll at 0.4-second intervals (`cao_rest.py:359`). A 30-second stalled threshold fires at 25% of the hard cap, giving operators roughly 90 seconds of visible "stalled" diagnostic state before the session times out entirely. At the 0.4s poll interval, approximately 75 poll cycles will have completed by the 30-second mark — more than sufficient to distinguish a genuinely stuck shadow parser from a transient unknown blip. The value is also consistent with the single-threshold decision (Q2): since both `_wait_for_shadow_ready_status()` (`cao_rest.py:567-598`) and `_wait_for_shadow_completion()` (`cao_rest.py:600-665`) share the same deadline and poll cadence, a single 30s default applies cleanly to both phases without phase-specific tuning.
