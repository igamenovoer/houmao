## Summary

- **Well-scoped and internally coherent.** The four artifacts (proposal → design → spec delta → tasks) tell a consistent story: reframe the demo as a server-state observation surface, not a second tracker.
- **Architecture is already aligned.** The source code (`monitor.py`, `driver.py`, `models.py`) already implements the pure server-consumer pattern. `ShadowWatchMonitor` polls `HoumaoServerClient` and renders `HoumaoTerminalStateResponse` directly — no demo-local parser, reducer, or state machine exists. This change is catching documentation and labels up to match already-correct code.
- **Unit tests encode the right contract.** `test_houmao_server_dual_shadow_watch_monitor.py` asserts server-consumer semantics (authority labels, compact status cards from server payloads). No demo-local tracking assertions exist.
- **Profile wording is the clearest remaining misalignment.** `projection-demo.md` line 8 ("live shadow parser and lifecycle validation") implies demo-level ownership of parser validation — task 2.3 targets this, but the spec delta doesn't prescribe replacement wording.
- **Dashboard label specifics are under-specified.** Decision 2 mandates distinguishing visible-state stability from completion debounce, but neither the spec delta nor the tasks prescribe what the new label text should be. The current header shows `stable=1.0s` which is exactly the conflation the design identifies as misleading.
- **Risk is low.** This is a presentation/documentation-only change with no server payload or tracking logic modifications. Rollback is trivial.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the dashboard header timing knobs be relabeled to make server-ownership explicit?

- Decision: `Option A`

#### Why this matters

The current dashboard header shows `poll=0.5s  stable=1.0s  unknown->stalled=30.0s`. These values are passed as CLI args to the demo runner but ultimately configure the server's behavior for that run. Decision 3 in the design says "docs and inspect output will treat them as the server posture configured for that run." The dashboard header is a prominent surface where this reframing could land.

#### Examples

- Current: `poll=0.5s  stable=1.0s  unknown->stalled=30.0s`
- Option A: `server: completion_debounce=1.0s  unknown→stalled=30.0s  |  monitor: poll=0.5s`
- Option B: Keep current compact format but add a label prefix like `(server posture)` above the line

#### Why this question arises in code

The header is built in `monitor.py` during dashboard layout construction. The timing values come from `DemoTimingConfig` in `models.py` and are persisted in the demo state. The `poll_interval_seconds` is genuinely demo-owned (how often the monitor refreshes), while `completion_stability_seconds` and `unknown_to_stalled_timeout_seconds` are server configuration — but the current header presents all three identically.

#### Options

**Option A: Split the header into server-posture and monitor-local sections**

- Code impact:
  - Modify the header panel in `monitor.py` to group timing values by ownership (server config vs. monitor-local poll interval)
- User-facing difference:
  - Operator can immediately see which timing values are server-owned configuration vs. demo-local refresh rate
- Pros:
  - Directly implements Decision 3's intent
  - Prevents the mental model error the change is designed to fix
- Cons:
  - Slightly wider header; may require layout adjustment
  - Adds implementation scope to task 1.2

**Option B: Keep the compact header format, address ownership only in docs**

- Code impact:
  - No dashboard code changes for this specific aspect
  - README and inspect output describe timing ownership
- User-facing difference:
  - Dashboard looks the same; operator learns ownership from documentation
- Pros:
  - Minimal implementation work
  - Dashboard remains compact
- Cons:
  - The dashboard is the primary operator surface — relying on docs alone may not fix the mental model for operators who skip the README

> **PROPOSED: Option A — split the header into server-posture and monitor-local sections.**
> Rationale: The dashboard is the primary operator surface during a live run. Decision 3 explicitly says the demo should "treat them as the server posture configured for that run" in visible output, not just docs. Since `poll_interval_seconds` is the only genuinely demo-local timing value, the split is simple: one line for server posture (`completion_debounce`, `unknown→stalled`), one for monitor config (`poll`). This is a small code change in `monitor.py` header construction and directly serves the change's core goal.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Choose Option A. The dashboard is the operator's primary live surface, so ownership and timing semantics need to be visible there, not only in docs. I will split the header so `poll_interval_seconds` remains clearly monitor-local while completion debounce and unknown-to-stalled timing are presented as server posture for the run.
