## Q1) Should we introduce `session_manifest.v3` now or keep `v2` with additive `backend_state` fields?

Blocking: Yes.

### Why this matters

- Session-manifest compatibility and resume semantics are central to this change.
- A schema bump touches validation, parsing, docs, and migration behavior in one shot.

### Options

**Option A: Keep `session_manifest.v2` and add headless tmux identity/resume-id fields under `backend_state`**

- Pros:
  - Minimal migration blast radius; existing loaders and validators continue working.
  - Faster implementation path while still enabling name-based identity checks.
- Cons:
  - Weaker typing guarantees for new headless fields compared to first-class sections.

**Option B: Introduce `session_manifest.v3` with explicit typed tmux/headless identity section**

- Pros:
  - Stronger schema-level guarantees and cleaner long-term model.
  - Easier to reason about invariants across backends.
- Cons:
  - Requires coordinated schema/version migration now.
  - Higher near-term delivery risk for this refactor.

### Proposal (Recommended)

Use Option A now: keep `session_manifest.v2`, store `tmux_session_name` and resume-id fields in `backend_state`, and enforce strict runtime validation on those keys for tmux-backed headless identity resolution. Revisit `v3` once new behavior is stable and tested.

### Pros / Cons (Proposal)

- Pros:
  - Preserves current manifest pipeline while allowing immediate progress.
  - Keeps migration work out of the critical path of backend unification.
- Cons:
  - Typed guarantees are split between pydantic structure and runtime validation code.

> **DECISION: Accept Option A — keep `session_manifest.v2` and persist tmux/resume identity in `backend_state` now; defer `session_manifest.v3`.**
>
> Decision explanation:
> - Delivery risk is currently dominated by runtime lifecycle unification (tmux identity, turn execution, resume behavior), so combining it with an immediate schema migration would make failures harder to isolate.
> - `backend_state` is sufficient to carry required invariants now (`tmux_session_name` plus tool resume id such as `session_id`/`thread_id`) while enforcing strict fail-fast validation on read and resolution paths.
> - Deferring `v3` is safe because `session_manifest.v2` already supports additive backend state and this repository allows intentional forward-breaking cleanup once semantics stabilize.
> - A dedicated follow-up should introduce `v3` only after tmux-backed headless behavior is stable across Codex, Claude, and Gemini and field semantics stop changing.

## Q2) For Codex headless, how should role prompt injection be implemented?

Blocking: Yes.

### Why this matters

- Codex headless behavior must preserve role intent currently carried by `codex_app_server` developer instructions.
- Wrong injection strategy can pollute user prompts or break resume consistency.

### Options

**Option A: Pass role prompt via Codex config override (`-c developer_instructions=...`)**

- Pros:
  - Preserves separation between role instructions and user turn text.
  - Aligns with Codex config model (`developer_instructions`) and avoids prompt rewriting.
- Cons:
  - Requires careful escaping/quoting for CLI overrides.

**Option B: Inject role as first-turn bootstrap text in the user prompt**

- Pros:
  - Simple to implement with existing headless bootstrap-message pattern.
- Cons:
  - Mixes system/developer semantics into user content.
  - Increases prompt noise and may shift model behavior subtly.

### Proposal (Recommended)

Use Option A: for Codex headless, inject role prompt through config override (`developer_instructions`) so user prompts remain clean and the role channel semantics match existing Codex session behavior.

### Pros / Cons (Proposal)

- Pros:
  - Closest semantic match to existing codex app-server `thread/start` developer instructions.
  - Avoids mutating first-turn prompt payload.
- Cons:
  - Needs robust argument encoding and tests for edge-case prompt content.

> **DECISION: Accept Option A — inject Codex role prompts through `-c developer_instructions=...` rather than bootstrap prompt rewriting.**
>
> Decision explanation:
> - Role instructions are developer/system semantics, so they should remain outside user-turn payloads; config override preserves that channel boundary.
> - This aligns with current Codex app-server semantics and avoids mutating first-turn content, improving comparability across retries and resume turns.
> - Implementation must include robust quoting/escaping for multiline and shell-sensitive role content so argv construction remains deterministic.
> - Tests should explicitly cover empty role text, multiline text, special-character content, and parity between new-session and resume-session command construction.
> - First-turn bootstrap rewriting was rejected because it conflates channels, increases prompt noise, and introduces subtle behavior drift risk.

## Q3) What should be the default `stop-session` behavior for tmux-backed headless sessions?

Blocking: Yes.

### Why this matters

- This affects debuggability, resource cleanup, and user expectations.
- The change objective explicitly values interactive tmux inspection.

### Options

**Option A: Preserve tmux session on `stop-session` by default; add explicit force-kill mode**

- Pros:
  - Supports postmortem inspection and interactive handoff.
  - Matches inspectability goals of the new unified contract.
- Cons:
  - Leaves stale sessions unless users/operator workflows clean them up.

**Option B: Kill tmux session by default on `stop-session`**

- Pros:
  - Cleaner runtime resource hygiene.
  - Simpler mental model for automation.
- Cons:
  - Loses inspectability right at stop time.
  - Conflicts with user-visible “headless shares tmux contract with TUI” intent.

### Proposal (Recommended)

Use Option A: default to preserving tmux sessions for headless stop, and provide an explicit cleanup/force mode for automation and CI pipelines.

### Pros / Cons (Proposal)

- Pros:
  - Optimizes for debugging and continuity during rollout.
  - Allows automation hygiene through explicit force behavior.
- Cons:
  - Requires clear docs to avoid confusion about stopped vs deleted sessions.

> **DECISION: Accept Option A — default `stop-session` preserves tmux for inspectability; provide explicit force-cleanup for automation.**
>
> Decision explanation:
> - This directly supports the unification goal that headless sessions share the inspectable tmux container contract used by TUI/CAO paths.
> - Default stop behavior should end runtime control while intentionally retaining tmux artifacts for post-stop debugging and handoff.
> - Automation and CI should use an explicit force-cleanup path so hygiene is intentional and not conflated with interactive debugging behavior.
> - Documentation and CLI messaging must clearly distinguish "session stopped" from "tmux container deleted" and point operators to cleanup commands.
> - Cleanup behavior should stay idempotent and diagnostic-friendly so stale-session handling is safe and observable.

## Q4) How much tmux logic should be shared between CAO and headless backends?

Blocking: Yes.

### Why this matters

- Over-sharing can couple unrelated policies (especially env handling).
- Under-sharing duplicates session naming/creation/error logic.

### Options

**Option A: Extract full tmux stack (session lifecycle + env composition + naming) into one shared module**

- Pros:
  - Maximum deduplication.
  - Single path for tmux interactions.
- Cons:
  - Harder to keep CAO-specific env precedence and headless-specific policy separate.
  - Increases coupling risk for future backend-specific behavior.

**Option B: Extract tmux primitives + identity allocation only; keep env composition backend-specific**

- Pros:
  - Shares the right low-level mechanics while preserving policy boundaries.
  - Reduces risk of accidental secret/env behavior regressions.
- Cons:
  - Some duplicated orchestration glue remains.

### Proposal (Recommended)

Use Option B: share tmux primitives and identity/naming helpers; keep launch-env composition policy in each backend.

### Pros / Cons (Proposal)

- Pros:
  - Maintains clear ownership for security-sensitive env policy.
  - Still removes the high-value duplication around tmux commands and identity rules.
- Cons:
  - Requires disciplined boundaries in module design.

> **DECISION: Accept Option B — share tmux primitives and identity/naming helpers; keep env composition policy backend-specific.**
>
> Decision explanation:
> - tmux lifecycle commands and naming invariants are low-level mechanics with strong reuse value and should be centralized.
> - Environment composition is security-sensitive policy and should remain backend-owned to preserve clear trust boundaries and precedence rules.
> - The shared module scope should cover session ensure/list/create/kill/check, env publication primitives, and identity allocation/canonicalization helpers.
> - Backend modules should continue to own credential allowlists, tool-home selection, backend-specific env overlays, and launch-policy decisions.
> - Small orchestration-glue duplication is acceptable to avoid over-coupling CAO and headless policy behavior.

## Q5) How should we deprecate `codex_app_server` while switching default Codex backend?

Blocking: Yes.

### Why this matters

- Hard removal can break existing scripts and docs abruptly.
- Keeping fallback too long prolongs dual-maintenance overhead.

### Options

**Option A: Hard cutover now (remove `codex_app_server` immediately)**

- Pros:
  - Fastest path to one canonical architecture.
- Cons:
  - High compatibility risk for existing users and automation.

**Option B: One deprecation window: switch default to `codex_headless`, keep explicit `codex_app_server` override temporarily**

- Pros:
  - Controlled migration with lower operational risk.
  - Easier rollback while validating headless rollout.
- Cons:
  - Temporary backend complexity persists.

### Proposal (Recommended)

Use Option B: set Codex default to headless immediately, keep `codex_app_server` as explicit override for one deprecation window, then remove after stability criteria are met.

### Pros / Cons (Proposal)

- Pros:
  - Balances delivery momentum and migration safety.
  - Provides clear runway for docs/tests and downstream scripts.
- Cons:
  - Requires explicit sunset criteria and cleanup follow-through.

> **DECISION: Accept Option B — switch default Codex backend to headless now; keep `codex_app_server` as explicit override for one deprecation window.**
>
> Decision explanation:
> - This gives immediate architectural convergence on the target model while preserving a deliberate rollback path during stabilization.
> - Existing scripts can transition safely by explicitly pinning `codex_app_server` during the deprecation window instead of relying on hidden behavior.
> - The deprecation window must be bounded by explicit sunset criteria (stability, test coverage, docs parity) and communicated in docs/help text.
> - Override usage should remain explicit opt-in only; no silent automatic fallback from the new default path.
> - Once criteria are met, remove `codex_app_server` to avoid indefinite dual-maintenance drift.

## Q6) How should name-based `--agent-identity` manifest validation generalize beyond CAO?

Blocking: Yes.

### Why this matters

- Current validation is CAO-only and blocks tmux-backed headless parity.
- Weak validation can target the wrong session manifest.

### Options

**Option A: Keep CAO-only validation for name-based resolution**

- Pros:
  - No changes to existing invariant checks.
- Cons:
  - Violates this change's new headless tmux identity requirements.

**Option B: Validate tmux-backed backend set + persisted canonical session-name invariant per backend**

- Pros:
  - Enables headless tmux name resolution while retaining strong safety checks.
  - Keeps session-addressing behavior consistent across tmux-backed backends.
- Cons:
  - Requires backend-specific accessor logic for persisted session-name fields.

### Proposal (Recommended)

Use Option B: define a tmux-backed backend set and require a backend-appropriate persisted canonical session name (CAO: `cao.session_name`; headless: `backend_state.tmux_session_name`) to match addressed tmux session.

### Pros / Cons (Proposal)

- Pros:
  - Preserves robust mismatch detection.
  - Delivers the intended parity between CAO/TUI and headless identity workflows.
- Cons:
  - Adds small complexity to resolution validation logic.

> **DECISION: Accept Option B — generalize name-based `--agent-identity` validation to tmux-backed backends with persisted canonical session-name invariants.**
>
> Decision explanation:
> - CAO-only validation is incompatible with the new tmux-backed headless contract and would leave cross-backend identity behavior inconsistent.
> - Name-based addressing must continue to fail fast on mismatches to prevent accidental attachment to the wrong manifest/session pair.
> - Validation should resolve by tmux session identity and then require backend-specific persisted canonical name equality.
> - Canonical fields are `cao.session_name` for CAO and `backend_state.tmux_session_name` for tmux-backed headless backends.
> - Implementation should gate this logic through an explicit tmux-backed backend set and emit actionable mismatch errors.

## Q7) What turn completion mechanism should the tmux headless runner use?

Blocking: Yes.

### Why this matters

- Completion detection determines correctness of event parsing, timeout behavior, and interrupt semantics.
- A brittle mechanism can cause hung turns or partial outputs.

### Options

**Option A: Primary `tmux wait-for` signaling with marker-file fallback**

- Pros:
  - Deterministic synchronization path with explicit completion signal.
  - Fallback path can recover from edge-case signaling failures.
- Cons:
  - Slightly more implementation complexity.

**Option B: Poll marker/status files only**

- Pros:
  - Simpler first implementation.
- Cons:
  - More race-prone and less responsive under load.

**Option C: Parse pane output and infer completion from prompt state**

- Pros:
  - No additional signaling plumbing.
- Cons:
  - Brittle due to ANSI/prompt/output format drift.

### Proposal (Recommended)

Use Option A: implement `tmux wait-for` as the primary completion contract and retain marker-file polling as fallback.

### Pros / Cons (Proposal)

- Pros:
  - Best reliability/latency tradeoff for turn-level orchestration.
  - Avoids fragile prompt-shape coupling.
- Cons:
  - Requires careful interrupt/timeout cleanup handling.

> **DECISION: Accept Option A — use `tmux wait-for` as primary turn completion signaling, with marker/status polling fallback.**
>
> Decision explanation:
> - `tmux wait-for` provides explicit synchronization semantics and a cleaner turn boundary than pure polling.
> - Marker/status polling remains as resilience fallback for edge cases where signaling fails, avoiding total turn-path fragility.
> - Timeout and interrupt cleanup must be correct for both completion paths so no dangling runner state remains.
> - Diagnostics should report whether completion came from wait-for or fallback polling to aid troubleshooting.
> - Poll-only and prompt-shape inference were rejected due to race risk and brittle coupling to terminal output formatting.
