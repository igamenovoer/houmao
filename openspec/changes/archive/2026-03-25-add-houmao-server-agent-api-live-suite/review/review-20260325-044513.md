## Response Format (Proposal / Decision Contract)

Open questions use paired `PROPOSED` and `DECISION` blockquotes immediately after each question's `Options` subsection. Actionable findings have a developer-owned `DECISION` blockquote directly after the finding.

## Summary

- The change is well-motivated: the existing demo-based verification is provably broken (timeout, worktree leak, tmux leak per `automatic-test-20260325.md`), and a dedicated API-centric live suite is a clear improvement.
- Artifact coherence is strong across proposal → design → spec → tasks. Route references, transport split, and lifecycle decisions are consistent.
- The primary risk is that the TUI provisioning path inherits the same 15-second `CaoRestClient` timeout that already blocks the demo; the design acknowledges this but no artifact specifies the concrete timeout value or client construction pattern to use.
- Fixture creation (Task 1.1) is under-specified: no content definition for the `server-api-smoke` fixture family (roles, recipes, dummy-project selection).
- The `materialize_delegated_launch()` function in `srv_ctrl/commands/runtime_artifacts.py` is the natural helper for TUI artifact staging, but no artifact references it explicitly, creating a gap between design intent and implementation guidance.

## Findings

### Must Fix / Blocking

**F1: TUI session creation will hit the same 15-second timeout unless the suite explicitly overrides the client timeout.**

The design's Risk section acknowledges "real tool launch latency exceeds default CAO client expectations" and prescribes "suite-controlled direct clients with larger timeout budgets." However, no artifact specifies the concrete timeout value, and the default `HoumaoServerClient.__init__` uses `timeout_seconds=15.0` (`src/houmao/server/client.py:60`), which is exactly the value that caused Finding 1 in `automatic-test-20260325.md`. The spec scenarios for TUI provisioning (spec lines 37-44) do not mention timeout requirements.

Without an explicit requirement or task sub-item for timeout override, the implementer may reproduce the exact demo failure. The spec or tasks should state the minimum required timeout for CAO session creation calls.

> **DECISION: Refine.**
> Rationale: Refine. Task 2.1 already requires an explicit timeout budget, so the change does not leave implementation on the implicit 15-second client default. The follow-up artifact pass should pin the initial default and override shape, but that belongs in design/tasks rather than turning the spec into a hard-coded timeout table.

**F2: Task 2.1 TUI provisioning path omits the concrete helper function reference and the registration request payload shape.**

Task 2.1 says "materialize delegated runtime artifacts" but does not reference `materialize_delegated_launch()` in `src/houmao/srv_ctrl/commands/runtime_artifacts.py:64`, which is the existing canonical function for this exact purpose. That function requires `runtime_root`, `api_base_url`, `session_name`, `terminal_id`, `tmux_window_name`, `provider`, `agent_profile`, and `working_directory`. The registration payload (`HoumaoRegisterLaunchRequest` in `src/houmao/server/models.py`) also requires specific fields including `session_name` and `tool`.

Without referencing these concrete interfaces, the task is ambiguous about whether the suite should call `materialize_delegated_launch()` directly, reimplement its logic, or use a different path. This affects both implementation time and correctness.

> **DECISION: Refine.**
> Rationale: Refine. The direction is right: the implementation plan should anchor to the existing `materialize_delegated_launch()` helper and the current registration request model instead of leaving the path implicit. That is implementation-guidance detail for design/tasks, not something the normative spec needs to spell out field-by-field.

### Should Fix / Important

**F3: The `server-api-smoke` fixture family is referenced but never defined.**

Task 1.1 says "Add a lightweight tracked `server-api-smoke` fixture family" but no artifact defines what this fixture contains: which roles, which brain recipes, which cli-config profiles, which dummy-project to copy. The existing fixture README (`tests/fixtures/agents/README.md`) documents `mailbox-demo` and `skill-invocation-demo` as the lightweight options. The suite needs at minimum one role, two recipes (Claude and Codex), matching cli-config profiles, and a dummy-project workdir.

The design should specify whether the suite reuses or extends existing lightweight fixtures, or creates entirely new ones, and what content each fixture file contains.

> **DECISION: Refine.**
> Rationale: Refine. We do need to define the fixture composition before implementation starts, but the change does not need exhaustive prose for every tracked file. The revision should make the intended split explicit: a dedicated `server-api-smoke` role/fixture family for the suite, with reuse of an existing lightweight copied dummy-project tree where that keeps scope smaller.

**F4: Headless provisioning path (Task 2.2) does not specify how `brain_manifest_path` is obtained.**

Task 2.2 says "build the runtime home and manifest from tracked fixtures" and then call `POST /houmao/agents/headless/launches`. The headless launch route (`HoumaoHeadlessLaunchRequest`) requires `brain_manifest_path` as a server-readable filesystem path. This means the suite must either:

(a) call `BrainBuilder.build_brain_home()` (`src/houmao/agents/brain_builder.py`) to produce a real manifest, or
(b) call `materialize_headless_launch_request()` (`src/houmao/srv_ctrl/commands/runtime_artifacts.py:286`) which resolves convenience inputs into a launch request.

Neither option is referenced in the tasks. The implementer needs to know which build path to use and where the manifest will be written relative to the suite's run root.

> **DECISION: Refine.**
> Rationale: Refine. The current task already commits us to building a real runtime home and manifest, so the gap is helper selection rather than missing behavior. The artifact pass should choose one canonical helper, and the preferred path is `materialize_headless_launch_request()` because it already resolves the recipe and calls `build_brain_home()` internally.

**F5: Spec asserts `state.detail.transport` but the actual field path in the response model is `detail.transport`.**

The spec (lines 57-58) says the suite SHALL verify `state.detail.transport = "tui"` and `state.detail.transport = "headless"`. The actual response model `HoumaoManagedAgentDetailResponse` (`src/houmao/server/models.py:696-702`) nests the transport discriminator at `detail.transport` (where `detail` is a `ManagedAgentDetailPayload` discriminated union at line 690-692). The `state` route (`GET .../state`) returns `HoumaoManagedAgentStateResponse`, which does not have a `detail` field — `detail` is only on `GET .../state/detail`.

The spec text should clarify that the transport assertion targets the `/state/detail` response's `detail.transport` field, not a nested path on the `/state` response. The scenarios (lines 65-68) correctly reference the `/state/detail` route, so this is a wording inconsistency rather than a logic error.

> **DECISION: Accept.**
> Rationale: Accept. This is a real wording bug in the current spec: the transport assertion belongs on the `/state/detail` response's `detail.transport` field, not on a nonexistent `state.detail` path from `/state`.

**F6: No explicit task for server health-check verification before lane provisioning.**

The design mentions starting `houmao-server` as a subprocess, but neither the spec nor the tasks include an explicit step for verifying server health (`GET /health`) and readiness before the suite proceeds to provision lanes. The existing `HoumaoServerClient` has `health_extended()` (`src/houmao/server/client.py:65-68`). Without a health gate, the suite may attempt lane provisioning against a server that hasn't finished startup, causing flaky failures.

> **DECISION: Accept.**
> Rationale: Accept. A live suite needs an explicit server-readiness gate, and the existing client already exposes `GET /health`. The startup helper should make that sequencing visible in the artifacts instead of leaving it as an implementation assumption.

**F7: Codex "API-key mode" dependency is stated in the proposal but not reflected in fixture or task details.**

The proposal (line 28) says "Codex exercised in API-key mode." This implies specific credential/env-var setup for Codex lanes. Neither the fixture definition (Task 1.1) nor the Codex provisioning tasks (2.1, 2.2) specify what API-key mode means concretely: which env vars need to be set, which credential profile to use, or whether preflight checks (Task 1.2) should validate this specific mode.

> **DECISION: Accept.**
> Rationale: Accept. The user explicitly scoped Codex coverage to API-key mode, and the current artifacts only mention that at proposal level. The revision should make the preflight and fixture/documentation expectation explicit without hardcoding secret values or provider-specific credential files into the spec text.

**F8: Cleanup of `houmao-server` subprocess itself is not covered in Task 2.3.**

Task 2.3 covers lane teardown through `POST /houmao/agents/{agent_ref}/stop` and best-effort tmux cleanup. Task 1.3 mentions "startup and shutdown helpers." However, the teardown sequence should specify that `houmao-server` process shutdown happens _after_ all lane stops complete, and that partial-failure cleanup should still attempt to kill the server subprocess. The spec's partial-failure scenario (spec lines 97-100) mentions best-effort stop but does not explicitly address server process cleanup order.

> **DECISION: Refine.**
> Rationale: Refine. Server-process cleanup is already owned by Task 1.3's startup/shutdown helpers, so the missing point is teardown sequencing rather than who owns the work. The artifact pass should clarify that lane stop/cleanup completes before process teardown and that partial-failure paths still terminate the server.

### Nice To Have

**F9: The two open questions in the design could be resolved before implementation starts.**

The design ends with two open questions: (1) whether to add a pytest wrapper later, and (2) whether v1 should support per-lane selection. Resolving these upfront would avoid design churn during implementation. Per-lane selection in particular affects Task 1.2 (entrypoint shape) and Task 4.1 (operator documentation).

> **DECISION: Accept.**
> Rationale: Accept. We are deciding the open-question surface now instead of carrying it into implementation. For v1, the suite remains manual-first and per-lane selection is accepted, so the follow-up artifact pass should remove the remaining ambiguity from design/tasks.

**F10: No task for validating tmux availability during preflight.**

Task 1.2 mentions "prerequisite checks" for executables and credential material. Both TUI and headless backends depend on tmux (`tmux_runtime` is the shared session/window primitive per `CLAUDE.md`). Adding a tmux-availability check to the preflight would prevent a confusing failure mode.

> **DECISION: Refine.**
> Rationale: Refine. Task 1.2 already covers prerequisite executable checks, so `tmux` is implicitly in scope today. The artifact revision should still name `tmux` explicitly in preflight/docs because both transports rely on it and the failure mode is otherwise opaque.

**F11: Run-root location convention is not specified.**

The design says "suite-owned run root" and "isolated per-run root" but doesn't specify where it should be created. The repo convention uses `tmp/` for generated runtime homes (gitignored). The tasks should state whether the run root goes under `tmp/`, a system temp dir, or is operator-specified.

> **DECISION: Accept.**
> Rationale: Accept. The suite needs a predictable default run-root location plus an override path so operators can find artifacts without guessing. The artifact pass should set a repo-local `tmp/` default and document an explicit override knob.

## Design Alignment With Existing System

**Route coverage:** All routes referenced in the spec and design exist in the current codebase. `GET /houmao/agents`, `GET /houmao/agents/{agent_ref}/state`, `GET /houmao/agents/{agent_ref}/state/detail`, `POST /houmao/agents/{agent_ref}/requests`, `POST /houmao/agents/{agent_ref}/stop`, `POST /houmao/agents/headless/launches`, `POST /houmao/launches/register`, and `POST /cao/sessions` are all registered in `src/houmao/server/app.py:95-338`.

**Client availability:** `HoumaoServerClient` (`src/houmao/server/client.py:57`) already exposes typed methods for every route the suite needs: `list_managed_agents()`, `get_managed_agent_state()`, `get_managed_agent_state_detail()`, `submit_managed_agent_request()`, `stop_managed_agent()`, `launch_headless_agent()`, and `get_headless_turn_status()`. It also inherits `create_session()` from `CaoRestClient`. The design's decision to use this client directly is well-aligned.

**Transport discriminator:** The spec's transport assertions (`transport = "tui"` / `transport = "headless"`) match the actual Pydantic discriminated union in `models.py:690-692`, with `HoumaoManagedAgentTuiDetailView.transport = Literal["tui"]` (line 621) and `HoumaoManagedAgentHeadlessDetailView.transport = Literal["headless"]` (line 649).

**Test placement:** Placing the suite under `tests/manual/` is consistent with the two existing manual tests (`manual_headless_tmux_smoke.py`, `manual_mailbox_roundtrip_real_agent_smoke.py`). The split between `tests/manual/` for credentialed live-tool tests and `tests/integration/` for mock-able integration tests follows the established convention.

**Fixture model:** The design's fixture approach (lightweight dedicated fixtures + copied dummy projects) aligns with the fixture README's guidance: use `tests/fixtures/dummy-projects/` for narrow, deterministic scenarios. The existing `mailbox-demo-python` and `projection-demo-python` dummy projects are the current lightweight options.

**Registration path:** The TUI provisioning flow (CAO session creation → artifact materialization → `POST /houmao/launches/register`) matches the documented registration and discovery architecture in `docs/developer/houmao-server/internals/registration_and_discovery.md`. The registration payload shape (`HoumaoRegisterLaunchRequest`) and the admission path (`_ensure_tracker_for_registered_launch()`) are well-documented and stable.

**Naming:** The suite name `houmao-server-agent-api-live-suite` follows the project's kebab-case naming convention for capabilities and change identifiers.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1: Should the suite call `materialize_delegated_launch()` directly for TUI artifact staging, or reimplement the staging logic?

**Options:**

- (A) Call `materialize_delegated_launch()` from `src/houmao/srv_ctrl/commands/runtime_artifacts.py:64` directly.
- (B) Reimplement the staging logic in suite-owned helpers.
- (C) Call `houmao-mgr` CLI commands that wrap `materialize_delegated_launch()`.

> **PROPOSED: (A) Call `materialize_delegated_launch()` directly.**
> Rationale: This function already handles manifest creation, session-root layout, registry record persistence, and gateway setup for delegated CAO launches. It is the canonical implementation used by `houmao-mgr cao launch` (the pair CLI). Reimplementing would duplicate ~200 lines of non-trivial logic and create a maintenance burden. The suite should import and call this function with its own runtime root, avoiding wrapper CLI dependencies per the design decision.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Accept PROPOSED. The suite should call `materialize_delegated_launch()` directly from suite-owned helpers instead of reimplementing the artifact path or routing back through the CLI wrapper. That keeps the test API-centric while still following the current manifest/session-root/registry materialization path used by registered TUI launches.

### Q2: What timeout should the suite use for CAO session creation (`POST /cao/sessions`)?

**Options:**

- (A) 60 seconds
- (B) 90 seconds
- (C) 120 seconds
- (D) Per-lane configurable with a high default

> **PROPOSED: (B) 90 seconds default with per-lane override capability.**
> Rationale: The current 15-second default in `CaoRestClient` (`src/houmao/cao/rest_client.py:68`) is proven insufficient — both Claude and Codex TUI launches timed out at 15s in the demo harness (`automatic-test-20260325.md`, Finding 1). The server-side `create_session()` in `core.py:121-190` performs synchronous tmux session creation and terminal initialization before responding. Real-world tool startup (especially Claude Code) can take 30-60 seconds. A 90-second default provides margin while remaining bounded. `HoumaoServerClient.__init__` already accepts `timeout_seconds` as a constructor parameter.
>
> **DECISION: Refine PROPOSED to Option D with a 90-second default.**
> Rationale: The suite should expose a configurable TUI creation timeout with an initial default of 90 seconds. That preserves the reviewer's core point about escaping the current 15-second client default while matching Task 4.2's plan to tune budgets from live runs instead of freezing one hard-coded number into the spec.

### Q3: Which dummy-project fixture should the suite copy for lane workdirs?

**Options:**

- (A) Reuse existing `mailbox-demo-python` from `tests/fixtures/dummy-projects/`.
- (B) Create a new minimal `server-api-smoke-python` dummy project.
- (C) Reuse `projection-demo-python`.

> **PROPOSED: (A) Reuse existing `mailbox-demo-python`.**
> Rationale: The suite's purpose is API contract verification, not complex agent behavior. `mailbox-demo-python` is already tracked, lightweight, and designed for narrow runtime-contract scenarios per the fixture README's decision tree. Creating a new dummy project adds fixture maintenance without clear benefit for an API-focused suite.
>
> **DECISION: Refine PROPOSED to Option A only for the dummy-project workdir.**
> Rationale: Reusing `mailbox-demo-python` as the copied lightweight workdir is acceptable and avoids unnecessary fixture churn. The suite should still keep a dedicated `server-api-smoke` role/recipe or blueprint family so the agent behavior stays API-smoke-specific rather than inheriting mailbox semantics.

### Q4: Should headless lanes use `BrainBuilder.build_brain_home()` or `materialize_headless_launch_request()` to produce the manifest?

**Options:**

- (A) `BrainBuilder.build_brain_home()` — full build-phase execution producing a real runtime home and manifest.
- (B) `materialize_headless_launch_request()` — convenience wrapper that resolves pair-style inputs into a `HoumaoHeadlessLaunchRequest`.
- (C) Pre-bake static manifests as test fixtures.

> **PROPOSED: (A) `BrainBuilder.build_brain_home()` for headless lanes.**
> Rationale: The suite is meant to exercise the real end-to-end lifecycle. `build_brain_home()` (`src/houmao/agents/brain_builder.py`) produces a real runtime home with projected configs, skills, and credentials — the same path production launches use. The headless launch route requires `brain_manifest_path` pointing to a real readable manifest on the server filesystem. `materialize_headless_launch_request()` is a pair-CLI convenience that still needs a recipe and agent-def-dir, and ultimately produces the same launch request shape. Pre-baked static manifests would be fragile across format changes. Using the real builder ensures the suite validates the full pipeline.
>
> **DECISION: Reject PROPOSED; choose Option B.**
> Rationale: `materialize_headless_launch_request()` is the better suite helper. It already resolves the recipe from the pair-style inputs, calls `build_brain_home()`, and returns the exact `HoumaoHeadlessLaunchRequest` payload the server expects, so using it avoids duplicating builder and request-construction logic in the suite.

### Q5: Should v1 support per-lane selection, or only an all-lanes aggregate mode?

**Options:**

- (A) All-lanes only in v1.
- (B) Both per-lane selection and all-lanes from day one.

> **PROPOSED: (B) Both per-lane selection and all-lanes from day one.**
> Rationale: Per-lane selection is essential for debugging and iterating on individual lanes during the initial stabilization phase (Task 4.2). The all-lanes mode is needed for full verification. The implementation cost is low: the suite already needs per-lane provisioning functions; the entrypoint just needs a lane filter argument. Without per-lane selection, the operator must wait for all four lanes to complete even when debugging one specific transport/tool combination.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Accept PROPOSED. V1 should support both per-lane selection and an all-lanes aggregate mode. That keeps debugging fast during live stabilization and does not materially complicate the harness because provisioning and verification are already lane-scoped.
