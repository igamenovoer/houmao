## Summary

- The change picks the right architectural target: tracked-state reduction is duplicated today across `src/houmao/server/tui/tracking.py`, `src/houmao/terminal_record/service.py`, and `src/houmao/explore/claude_code_state_tracking/state_reducer.py`.
- Reusing `houmao.lifecycle` as the lower-level timing kernel is aligned with existing repo structure and avoids pushing route-specific semantics into that package.
- The current design does not yet resolve detector ownership cleanly enough to guarantee that the new shared core stays below `server` and `explore`.
- Replay output migration is well-motivated, but the operator-facing label surface is still specified too loosely relative to the current `terminal_record add-label` interface.
- The explore-harness spec still contains wording that can be read as preserving an independent mirrored reducer, which conflicts with the stated design intent.

## Findings

### Must Fix / Blocking

1. **The plan does not define who owns tool-specific turn-signal detectors after extraction, so the proposed shared core can still violate its own dependency boundary.**

The shared-core spec says the new layer SHALL remain independent of `houmao.server`, `houmao.demo`, and `houmao.terminal_record` adapters, but the current turn-signal path is split across those boundaries. `src/houmao/server/tui/tracking.py` still selects detectors from `src/houmao/server/tui/turn_signals.py`, and that module already imports the Claude detector from `src/houmao/explore/claude_code_state_tracking/detectors` (`src/houmao/server/tui/turn_signals.py:10`). Task `1.2` says to move turn-signal interpretation out of `houmao.server.tui`, but the design never states whether detector families move into the shared core or whether the shared core accepts adapter-supplied normalized signals. Without that decision, this change can "extract" the reducer while leaving the dependency inversion intact.

## Design Alignment With Existing System

- Existing pattern(s) observed: `src/houmao/lifecycle/rx_lifecycle_kernel.py` already provides a reusable lower-level timing kernel; `src/houmao/server/tui/tracking.py` layers server-owned observation, debug emission, route-model construction, and recent-transition/stability handling on top of it; `src/houmao/terminal_record/service.py` currently performs replay analysis by importing demo-owned tracking code; `src/houmao/server/tui/turn_signals.py` already bridges across package boundaries by importing explore-owned Claude detectors.
- Proposed design alignment: the shared-core direction is native to the repo if it stays as a reducer/evidence layer above `houmao.lifecycle` and below adapter code. The best fit is: shared core owns neutral evidence inputs, reduction, and optional tool-detector interfaces; server remains responsible for tmux/process/probe observation, route shaping, and debug/history publication; recorder and explore paths become adapters that feed normalized observations into the same core instead of owning separate reducers.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Which recorded input events should count as authoritative explicit-input evidence during replay?

- Decision: `Blocking`

#### Why this matters

- The shared-core spec requires replay to preserve `last_turn.source=explicit_input` when authoritative input evidence exists.
- The current recorder artifact set stores two different kinds of input evidence with different trust levels, and the design does not yet say how replay should interpret them.

#### Examples

- A run captured via active mode contains `managed_send_keys` events written by the runtime bridge and later also contains raw `asciinema_input` keystrokes for the same prompt.
- A tainted run degrades capture level away from `authoritative_managed`, but still has some input events on disk.

#### Why this question arises in code

- `src/houmao/terminal_record/runtime_bridge.py:48-82` writes `managed_send_keys` events into `input_events.ndjson` during active recording.
- `src/houmao/terminal_record/service.py:669-703` later parses raw `asciinema` client input as `asciinema_input`.
- `src/houmao/terminal_record/models.py:207-216` shows that both event types currently share one generic schema with only `source`, `sequence`, and timing metadata.

```text
managed_send_keys  -> runtime-owned control path
asciinema_input    -> client-observed raw keystrokes
```

#### Options

**Option A: Treat only runtime-managed input events as authoritative explicit-input evidence.**

- Code impact:
  - Replay filters `input_events.ndjson` to sources that came from the runtime bridge and only arms explicit-input authority from those events.
  - Raw `asciinema_input` remains available for debugging, but replay falls back to `surface_inference` or `none` when managed events are absent.
- User-facing difference:
  - `last_turn.source=explicit_input` appears only when the recorder captured high-confidence runtime-owned submit evidence.
- Pros:
  - Matches the "authoritative" wording in the spec.
  - Avoids over-crediting noisy or partial raw client keystrokes as prompt submission events.
- Cons:
  - Some active recordings may lose explicit-input attribution if only raw client input survived.

**Option B: Allow both runtime-managed input events and raw `asciinema_input` to arm explicit-input authority.**

- Code impact:
  - Replay must infer submit semantics from raw keystroke sequences and reconcile them with managed events when both exist.
  - The adapter needs extra de-duplication and confidence rules for overlapping input artifacts.
- User-facing difference:
  - More recordings may report `last_turn.source=explicit_input`, but confidence in that attribution is lower.
- Pros:
  - Maximizes the amount of replay data treated as explicit input.
- Cons:
  - Makes replay provenance less defensible and more tool-specific.
  - Increases the chance of false `explicit_input` attribution from ordinary editing keystrokes.

> **PROPOSED: Treat only runtime-managed recorder events as authoritative explicit-input evidence, and treat raw `asciinema_input` as secondary debug data.**
> Rationale: The current code already distinguishes runtime-owned `managed_send_keys` events from raw `asciinema_input` capture (`src/houmao/terminal_record/runtime_bridge.py:48-82`, `src/houmao/terminal_record/service.py:669-703`). That distinction maps naturally to the spec's "authoritative input events" language and avoids inventing submit certainty from raw keystrokes. If no managed event is available, replay should degrade to `surface_inference` or `none` rather than overstate provenance.
>

### Q2) Should server code keep `Houmao*` tracked-state names as the route boundary, or switch internal server assembly directly to neutral core model names?

- Decision: `Deferrable`

#### Why this matters

- The change explicitly moves tracked-state semantic ownership below `houmao.server`, but the current server API and tests still import `Houmao*` route models directly.
- A clean answer here affects how disruptive the migration will be for server code and tests.

#### Examples

- `tests/unit/server/test_client.py` still requests `HoumaoTerminalStateResponse` as the public model.
- `tests/unit/server/test_app_contracts.py:205-259` constructs `HoumaoTerminalStateResponse`, `HoumaoTrackedSurface`, and `HoumaoTrackedLastTurn` directly in contract tests.

#### Why this question arises in code

- `src/houmao/server/models.py:274-329` currently owns both the route-facing models and the tracked-state semantic types.
- The design says that `houmao.server.models` should become a thin adapter or re-export boundary, but it does not choose between re-exporting neutral models and rewriting server internals to use only neutral names.

```text
shared core -> neutral tracked-state models
server      -> route response assembly and API contract
```

#### Options

**Option A: Keep `Houmao*` names as the server route boundary and adapt/re-export neutral core models there.**

- Code impact:
  - Shared core defines neutral internal models.
  - `houmao.server.models` continues to define or re-export the route-facing `Houmao*` shapes, with explicit conversion at the boundary.
- User-facing difference:
  - Server API contracts and most existing server tests keep their current names.
- Pros:
  - Preserves a clear server-owned API boundary.
  - Minimizes churn in server contract tests and client-facing code.
- Cons:
  - Maintains two naming layers in the codebase.

**Option B: Move server internals and route assembly directly to neutral model names.**

- Code impact:
  - Server code, tests, and route models are renamed or deeply refactored to use the neutral shared-core types directly.
- User-facing difference:
  - Internal consistency improves, but the migration is broader and touches more server-owned files at once.
- Pros:
  - Reduces duplicate naming.
  - Makes shared-core ownership maximally explicit.
- Cons:
  - Creates wider refactor churn without changing the public API behavior.

> **PROPOSED: Keep `Houmao*` names as the explicit server route boundary and adapt neutral shared-core models into that boundary.**
> Rationale: The repository already treats `src/houmao/server/models.py` as the route-contract module, and server tests still construct `HoumaoTerminalStateResponse`, `HoumaoTrackedSurface`, and `HoumaoTrackedLastTurn` directly (`tests/unit/server/test_app_contracts.py:205-259`; `tests/unit/server/test_client.py:8-68`). Preserving those names at the server boundary keeps the API contract stable inside the repo while still moving semantic ownership down into the shared core. The tradeoff is one extra adapter layer, but that is cleaner than renaming route contracts during the same extraction.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._
