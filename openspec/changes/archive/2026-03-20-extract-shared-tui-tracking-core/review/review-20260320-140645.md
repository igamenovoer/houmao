## Summary

- The revised docs materially improved the plan: they now clearly preserve the dual-shadow-watch demo as an independent reference implementation and explicitly state that recorder replay must not depend on the demo tracker.
- The revised shared-core and replay specs also clarify the explicit-input path by naming authoritative managed input events rather than leaving raw recorder input authority ambiguous.
- One blocking architectural gap remains: detector ownership is still not specified tightly enough to guarantee that the extracted shared core stays below `server` and `explore`.
- The explore-harness spec still contains wording that can be read as preserving an independent mirrored reducer, which conflicts with the revised design.
- The replay label contract is much clearer than before, but it still does not say how the existing `terminal_record add-label` surface should evolve to satisfy the new vocabulary.

## Findings

### Must Fix / Blocking

1. **Detector ownership is still underspecified, so the shared core can still inherit the current dependency inversion instead of removing it.**

The design says the shared core will own turn-signal interpretation (`openspec/changes/extract-shared-tui-tracking-core/design.md:35-41`) and the migration plan says step 1 will extract "detector dependencies" from the current server tracker (`openspec/changes/extract-shared-tui-tracking-core/design.md:196-197`). But it still does not choose a concrete ownership model for those detectors. Today the server detector layer imports the Claude detector from the explore package (`src/houmao/server/tui/turn_signals.py:10`) and then wraps it in server-owned logic (`src/houmao/server/tui/turn_signals.py:68-111`). Without an explicit decision here, the implementation can move code into a new package while still depending upward into `houmao.explore` or retaining server-shaped detector APIs.

> **DECISION: Accept.**
> Rationale: The finding identifies a real remaining gap. Today the official/runtime detector path still crosses upward into `houmao.explore`, and the current artifacts do not yet make it impossible for that dependency inversion to survive extraction. The intended fix is narrower than a general plugin system: the official/runtime detector layer should move under the shared core or another package below `server` and `explore`, while the explore harness may keep separate groundtruth detectors as its independent reference path.

### Should Fix / Important

2. **The explore-harness spec still contradicts the revised design by describing replay as an "independent ReactiveX tracker."**

The design now clearly says the harness replay path will stop mirroring tracked-state semantics and will use the shared core directly (`openspec/changes/extract-shared-tui-tracking-core/design.md:153-166`). The spec, however, still starts with "Explore harness SHALL replay recorded observations through an independent ReactiveX tracker" (`openspec/changes/extract-shared-tui-tracking-core/specs/claude-code-state-tracking-explore-harness/spec.md:3-8`). That wording still leaves room for a harness-owned reducer wrapper that preserves the old split in practice. The requirement should say "harness-owned replay adapter over the shared core" or equivalent, so the acceptance criteria cannot be satisfied by keeping a parallel reducer architecture.

> **DECISION: Accept.**
> Rationale: The review is correct. The design rejects a mirrored harness reducer, but the current requirement title and opening sentence still preserve that reading. The spec should describe a harness-owned replay adapter over the shared core, with independence reserved for the content-first groundtruth path and for separation from the live server adapter.

3. **The replay label contract is still not concrete enough at the current operator interface boundary.**

The replay spec is much better than before: it now requires official tracked-state label fields and makes legacy readiness/completion assertions secondary (`openspec/changes/extract-shared-tui-tracking-core/specs/terminal-record-replay/spec.md:29-41`). But the current operator-facing label surface still only exposes `business_state`, `input_mode`, `ui_context`, `readiness_state`, and `completion_state` through `terminal_record add-label` (`src/houmao/terminal_record/service.py:734-807`). Task `3.2` says to update "outputs, labels, and tests", but the docs still do not say whether the intended migration is CLI expansion, direct JSON editing, or both. That leaves a real gap between the current native workflow and the revised contract.

> **DECISION: Accept.**
> Rationale: This is a real artifact gap. The revised replay contract now exceeds the current `terminal_record add-label` CLI surface, so leaving the migration implicit would allow implementation to satisfy replay rows while missing the operator-facing labeling workflow. The intended direction is to extend `terminal_record add-label` and keep the CLI as the primary authoring surface, with JSON artifacts remaining the persistence layer rather than the primary workflow.

## Design Alignment With Existing System

- Existing pattern(s) observed: `src/houmao/lifecycle/rx_lifecycle_kernel.py` is already the reusable lower-level timing kernel; `src/houmao/server/tui/tracking.py` layers observation, authority handling, route-model shaping, debug emission, and stability/history publication above it; `src/houmao/terminal_record/service.py` owns the current replay and labeling CLI surface; `src/houmao/server/tui/turn_signals.py` currently bridges into explore-owned Claude detectors.
- Proposed design alignment: the revised plan is well aligned if the shared core becomes a neutral evidence/reduction layer above `houmao.lifecycle`, while server, recorder, and explore stay as adapters. The important remaining native-design step is to make detector ownership explicit so the shared core can accept either neutral detector plugins or adapter-supplied normalized signals without depending on higher-level packages.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should `houmao.server.models` remain the explicit route boundary while neutral shared-core models stay internal?

- Decision: `Deferrable`

#### Why this matters

- The revised design still leaves this open, and the answer determines how broad the extraction refactor becomes inside server-owned code.
- Choosing the boundary explicitly will reduce churn in contract tests and make task `2.2` more concrete.

#### Examples

- `tests/unit/server/test_client.py` still uses `HoumaoTerminalStateResponse` as the server contract model.
- `tests/unit/server/test_app_contracts.py` still constructs `HoumaoTrackedSurface` and `HoumaoTrackedLastTurn` directly when validating server routes.

#### Why this question arises in code

- `src/houmao/server/models.py` currently owns both the public route types and the tracked-state semantic types.
- The design says `houmao.server.models` becomes a thin adapter or re-export boundary, but it still leaves two implementation directions open.

```text
shared core -> neutral tracked-state models
server      -> route-contract adapter boundary
```

#### Options

**Option A: Keep `Houmao*` names as the explicit server route boundary and adapt neutral core models into them.**

- Code impact:
  - Shared core introduces neutral models for generic tracking semantics.
  - Server code converts or re-exports those shapes at the route layer.
- User-facing difference:
  - Current server contract naming remains stable inside the repo.
- Pros:
  - Matches the current code and tests.
  - Limits churn while still moving semantic ownership below `houmao.server`.
- Cons:
  - Preserves two naming layers.

**Option B: Move server internals and route assembly directly onto neutral model names.**

- Code impact:
  - Route assembly, tests, and server-owned state types all get renamed or restructured around the neutral shared-core vocabulary.
- User-facing difference:
  - Internal consistency improves, but the extraction change becomes much broader.
- Pros:
  - Reduces duplicated naming.
  - Makes shared-core ownership more explicit.
- Cons:
  - Increases refactor surface without changing external behavior.

> **PROPOSED: Keep `Houmao*` names as the explicit server route boundary and adapt neutral shared-core models into that boundary.**
> Rationale: The current server contract and tests are already centered on `src/houmao/server/models.py`, including `HoumaoTerminalStateResponse`, `HoumaoTrackedSurface`, and `HoumaoTrackedLastTurn`. Preserving that boundary keeps the extraction focused on semantic ownership rather than renaming the public server layer in the same change. The tradeoff is an adapter boundary, but that boundary is consistent with the repo's current layering.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Keep `Houmao*` names as the explicit server route boundary and adapt or re-export neutral shared-core models into that boundary. That keeps this change focused on semantic ownership, detector/reducer extraction, and dependency repair instead of broad route-model renaming. The extra adapter layer is acceptable and consistent with the current server contract tests.

### Q2) Is the intended label migration path to extend `terminal_record add-label`, or to treat the persisted JSON artifact as the primary authoring surface?

- Decision: `Blocking`

#### Why this matters

- The revised replay spec now defines the official tracked-state vocabulary for labels, but implementation still needs a concrete operator workflow that matches existing repository patterns.
- If this stays implicit, the change can satisfy the artifact schema on paper while leaving the actual labeling workflow behind.

#### Examples

- A developer today can only pass `--business-state`, `--input-mode`, `--ui-context`, `--readiness-state`, and `--completion-state` into `terminal_record add-label`.
- The revised contract expects fields like diagnostics posture, `surface` observables, `turn.phase`, `last_turn.result`, and `last_turn.source`.

#### Why this question arises in code

- `src/houmao/terminal_record/service.py:734-807` shows the current CLI surface and expectation payload builder.
- `openspec/changes/extract-shared-tui-tracking-core/specs/terminal-record-replay/spec.md:29-41` expands the expected label vocabulary but does not tie it to a specific authoring interface.

```text
current CLI -> legacy-style expectation fields only
revised spec -> official tracked-state vocabulary
```

#### Options

**Option A: Extend `terminal_record add-label` to support the official tracked-state fields and keep the CLI as the primary labeling surface.**

- Code impact:
  - Add new CLI flags and expectation serialization for diagnostics, `surface`, `turn`, and `last_turn` fields.
  - Tests shift from legacy-only label assertions to the new vocabulary.
- User-facing difference:
  - The existing labeling workflow stays intact, but with a revised field set.
- Pros:
  - Fits the current repo workflow.
  - Keeps operator labeling discoverable and consistent.
- Cons:
  - Requires a larger CLI update in the same change.

**Option B: Keep the CLI minimal and treat direct JSON editing or exported fixtures as the primary authoring surface for the new fields.**

- Code impact:
  - Fewer CLI changes, but more burden on artifact conventions and docs.
- User-facing difference:
  - Advanced labels require manual file editing or fixture generation tooling instead of the current CLI.
- Pros:
  - Smaller immediate implementation.
- Cons:
  - Regresses the current operator workflow.
  - Is less native to the existing `terminal_record` tool surface.

> **PROPOSED: Extend `terminal_record add-label` to support the official tracked-state fields and keep the CLI as the primary labeling surface.**
> Rationale: The repository already has a clear operator-facing labeling entry point in `src/houmao/terminal_record/service.py:734-807`, so the most native way to land this contract change is to evolve that surface rather than route around it. That keeps replay-grade labeling aligned with the existing tool workflow and prevents the contract shift from becoming "JSON-only" in practice. The tradeoff is a larger CLI patch, but that is preferable to leaving the user workflow underspecified.
>
> **DECISION: Accept PROPOSED.**
> Rationale: The label migration should extend `terminal_record add-label` and keep the CLI as the primary authoring surface. That is the most native fit with the existing operator workflow, and it prevents the replay contract shift from becoming artifact-schema-only. Direct JSON editing may still exist as a low-level escape hatch, but it is not the primary path this change should rely on.

## Suggested Next Steps

1. Revise the design or specs to make detector ownership explicit and keep the official/runtime detector layer below `server` and `explore`.
2. Tighten the explore-harness requirement so it describes a harness-owned replay adapter over the shared core rather than an "independent ReactiveX tracker."
3. Revise the replay spec and task `3.2` to reflect the decision that `terminal_record add-label` remains the primary label-authoring surface.
4. Clarify in the design that `Houmao*` names remain the server route boundary while neutral tracked-state models live below that boundary.
