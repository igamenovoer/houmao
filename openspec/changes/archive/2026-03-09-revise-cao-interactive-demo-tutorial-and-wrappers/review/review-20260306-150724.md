## Summary

- All required spec-driven artifacts are present, and the change intent is clear: keep the Python lifecycle engine, add a friendlier operator surface, and move `verify` out of the main walkthrough.
- The proposed layering mostly aligns with the existing system: `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py` already owns state, guardrails, and report generation, while demo packs typically expose shell entrypoints plus a README and verifier script.
- The main blocking issue is that the artifacts do not clearly say whether the new wrappers should call `run_demo.sh` or call the Python module directly, even though `run_demo.sh` is currently the single source of truth for workspace and environment defaults.
- The validation story should stay manual and tutorial-oriented rather than growing new Pixi gating, but the artifacts should say that explicitly so reviewers do not mistake these demos for release certification.
- The open-ended tutorial design should explicitly describe what `verify` still means after arbitrary manual prompting, since the current verifier checks a minimum two-turn contract rather than every recorded turn.

## Findings

### Must Fix / Blocking

- Clarify the wrapper delegation boundary. The spec says the new shell entrypoints delegate to the underlying interactive demo CLI (`specs/cao-interactive-demo-operator-workflow/spec.md:12-28`), while the design's drift mitigation says wrappers should delegate all real behavior to `run_demo.sh` (`design.md:53-58`). Those are materially different implementation shapes because `run_demo.sh` currently centralizes `DEMO_WORKSPACE_ROOT`, `AGENT_DEF_DIR`, `CAO_LAUNCHER_HOME_DIR`, and `DEMO_ROLE_NAME`, and also owns `verify --snapshot-report` post-processing (`scripts/demo/cao-interactive-full-pipeline-demo/run_demo.sh:4-9`, `scripts/demo/cao-interactive-full-pipeline-demo/run_demo.sh:23-51`). Without an explicit decision, an implementation can satisfy the spec while still duplicating exactly the defaults the design is trying to keep centralized.

### Should Fix / Important

- Clarify that tasks `3.1` through `3.3` are satisfied by manual developer review of the demo flow, with any syntax checks framed as lightweight author aids rather than Pixi-managed or release-gating automation. The current task wording (`tasks.md:14-18`) can be read as certification-style validation, but the repo-level Pixi tasks are aimed at format, lint, typecheck, and unit/runtime test suites (`pyproject.toml:105-112`), not at treating tutorial demos as CI gates.
- Document `verify` as a minimum regression check rather than a full transcript check. The design correctly moves verification into a secondary maintainer path (`design.md:47-51`), but the current verifier only requires at least two turns and snapshots the first two sanitized turns (`scripts/demo/cao-interactive-full-pipeline-demo/scripts/verify_report.py:67-95`, `scripts/demo/cao-interactive-full-pipeline-demo/scripts/verify_report.py:119-133`). Because the new tutorial is intentionally open-ended (`specs/cao-interactive-demo-operator-workflow/spec.md:30-36`), the README appendix should explicitly explain that extra turns are allowed and a passing `verify` only proves the minimum regression contract still holds.

### Nice To Have

- Add one appendix note that operator input `alice` will appear in persisted state and inspection output as the canonicalized identity `AGENTSYS-alice`, because the engine normalizes agent names before starting the session (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:378-383`, `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:781-786`). That will save confusion when readers compare the wrapper command with `state.json` or `run_demo.sh inspect`.
- Cite a concrete in-repo tutorial exemplar when saying "API usage tutorial format". `scripts/demo/cao-server-launcher/README.md` already shows the local pattern of goal framing, checklist, implementation idea, walkthrough, troubleshooting, and appendix (`scripts/demo/cao-server-launcher/README.md:5-29`, `scripts/demo/cao-server-launcher/README.md:117-238`). Naming that reference would make review expectations less subjective.

## Design Alignment With Existing System

- Existing pattern(s) observed:
  - Stateful demo behavior lives in Python modules, not in shell scripts. `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py` owns `start`, `send-turn`, `inspect`, `verify`, and `stop`, plus persisted state and report generation (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:307-596`).
  - Demo packs usually expose a small shell surface rooted in `run_demo.sh`, with README instructions and a verifier script alongside it (`scripts/demo/cao-interactive-full-pipeline-demo/run_demo.sh`, `scripts/demo/cao-claude-session/README.md`, `scripts/demo/cao-server-launcher/README.md`).
  - Integration tests validate these flows via Python `pytest` harnesses that stub external binaries instead of requiring real CAO services (`tests/integration/demo/test_cao_interactive_full_pipeline_demo_cli.py:21-270`).
- Proposed design alignment:
  - Keeping `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py` as the workflow engine matches the current layering and avoids duplicating JSON/state logic in bash.
  - Reframing the README toward an operator workflow is consistent with the stronger tutorial style already visible in `scripts/demo/cao-server-launcher/README.md`.
- Proposed design mismatches:
  - The artifacts do not yet explicitly preserve `run_demo.sh` or a shared sourced helper as the source of truth for wrapper defaults, which risks creating parallel shell implementations instead of thin facades.
- The testing section should explicitly distinguish optional developer validation from release-gating automation so the demo remains a learning surface rather than a certification artifact.

## Open Questions (With Proposed Defaults)

### Q1) What should the new wrapper scripts delegate to?

- Decision: `Blocking`

#### Why this matters

- This determines whether workspace and environment defaults remain centralized or get duplicated across multiple shell files.
- It also decides whether future changes to launcher home, agent fixtures, or role defaults require editing one file or several.

#### Examples

- A launch wrapper that directly invokes `pixi run python -m gig_agents.demo.cao_interactive_full_pipeline_demo start --agent-name alice`.
- A launch wrapper that simply forwards to `run_demo.sh start --agent-name alice`.

#### Why this question arises in code

- `run_demo.sh` already centralizes the default workspace and environment wiring:

```markdown
WORKSPACE_ROOT="${DEMO_WORKSPACE_ROOT:-$REPO_ROOT/tmp/cao_interactive_full_pipeline_demo}"
AGENT_DEF_DIR="${AGENT_DEF_DIR:-$REPO_ROOT/tests/fixtures/agents}"
LAUNCHER_HOME_DIR="${CAO_LAUNCHER_HOME_DIR:-$WORKSPACE_ROOT}"
ROLE_NAME="${DEMO_ROLE_NAME:-gpu-kernel-coder}"
```

- The spec talks about delegating to the underlying CLI, while the design's explicit drift mitigation points at `run_demo.sh`.

#### Options

**Option A: Call the Python module directly from each wrapper**

- Code impact:
  - Each wrapper reconstructs the `pixi run python -m gig_agents.demo.cao_interactive_full_pipeline_demo ...` invocation and repeats the default flags.

```markdown
launch.sh -> pixi run python -m gig_agents.demo.cao_interactive_full_pipeline_demo ... start --agent-name alice
prompt.sh -> pixi run python -m gig_agents.demo.cao_interactive_full_pipeline_demo ... send-turn --prompt "$PROMPT"
stop.sh   -> pixi run python -m gig_agents.demo.cao_interactive_full_pipeline_demo ... stop
```

- User-facing difference:
  - No visible difference if implemented correctly, but drift can appear later when defaults change.
- Pros:
  - Direct and explicit.
  - Keeps wrappers independent from `run_demo.sh`.
- Cons:
  - Duplicates configuration logic already present in `run_demo.sh`.
  - Makes task `1.4` harder to enforce over time.

**Option B: Treat `run_demo.sh` or a helper extracted from it as the shared wrapper backend**

- Code impact:
  - New wrappers forward subcommands into the existing shell entrypoint, or all shell entrypoints source a common helper that contains the current defaults.

```markdown
launch.sh -> run_demo.sh start --agent-name alice
prompt.sh -> run_demo.sh send-turn --prompt "$PROMPT"
stop.sh   -> run_demo.sh stop
```

- User-facing difference:
  - Users get memorable commands while the underlying workspace and environment semantics stay identical to today's demo.
- Pros:
  - Preserves a single source of truth for shell-level defaults.
  - Matches the design's stated anti-drift strategy.
  - Minimizes the amount of new behavior that needs separate testing.
- Cons:
  - Adds one extra layer of shell indirection.
  - If `run_demo.sh` remains complex, wrappers inherit that complexity.

#### Proposal (Recommended)

Use Option B. Keep `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py` as the true engine, but make `run_demo.sh` or a factored helper the shell-level source of truth that every new wrapper reuses.

#### Pros / Cons (Proposal)

- Pros:
  - Aligns with the repo's current demo-pack layering.
  - Directly satisfies task `1.4` instead of relying on convention.
  - Reduces future drift when shell defaults change.
- Cons:
  - Requires the artifacts to be more explicit about the chosen indirection.

> **DECISION: Treat `run_demo.sh` or a helper extracted from it as the authoritative shell backend for the new wrappers.**
> Rationale: `scripts/demo/cao-interactive-full-pipeline-demo/run_demo.sh:4-9` already owns the default workspace and environment plumbing, and `design.md:55-58` explicitly cites delegation to `run_demo.sh` as the mitigation for drift. Reusing that layer is more native to this repo than copying the same flags into three new scripts.

### Q2) After the tutorial becomes open-ended, what should `verify` mean?

- Decision: `Deferrable`

#### Why this matters

- The README will encourage repeated manual prompting, but maintainers still need a crisp explanation of what a later `verify` run proves.
- If this remains vague, a passing verification can be mistaken for "all manual turns matched expectations" instead of "the minimum regression contract still holds."

#### Examples

- A user launches `alice`, sends five prompts, then runs `run_demo.sh verify`.
- A maintainer snapshots a report after an open-ended manual session and expects every turn to be compared.

#### Why this question arises in code

- The current verifier only enforces a minimum-two-turn contract and sanitizes the first two turns:

```markdown
if len(turns) < 2:
    raise ValueError("report.turns must contain at least two entries")

verified_turns = [
    _sanitize_turn(normalized_turns[0], expected_index=1),
    _sanitize_turn(normalized_turns[1], expected_index=2),
]
```

- That behavior is compatible with an open-ended tutorial, but only if the docs say so clearly.

#### Options

**Option A: Keep the current minimum-two-turn verification contract and document it clearly**

- Code impact:
  - No report-schema or verifier changes are required.
  - The maintainer appendix explains that `verify` confirms the existing regression contract after any session with at least two good turns.

- User-facing difference:
  - The tutorial remains open-ended, and `verify` stays optional maintainer tooling.
- Pros:
  - Keeps this change focused on tutorial surfaces and wrappers.
  - Avoids changing `expected_report/report.json` semantics.
- Cons:
  - Verification remains intentionally narrower than the full manual session history.

**Option B: Change verification to validate all turns or a different subset**

- Code impact:
  - Requires changes to `verify_report.py`, the expected snapshot, and likely the unit or integration tests.
  - Expands the change beyond tutorial reshaping into report-contract redesign.

- User-facing difference:
  - `verify` becomes a stronger claim about the whole session.
- Pros:
  - Reduces ambiguity about what was validated.
- Cons:
  - Increases scope and test churn.
  - Conflicts with the design's "surfaces, not architecture" framing.

#### Proposal (Recommended)

Use Option A for this change. Keep the verifier semantics as-is and document them explicitly in the README maintainer section or appendix.

#### Pros / Cons (Proposal)

- Pros:
  - Fits the change scope described in `design.md:3-5`.
  - Keeps snapshot churn and verification semantics stable.
  - Still supports open-ended manual usage.
- Cons:
  - Requires careful wording so maintainers do not over-interpret `verify`.

> **DECISION: Keep `verify` as a minimum two-turn regression check and document that scope explicitly.**
> Rationale: `scripts/demo/cao-interactive-full-pipeline-demo/scripts/verify_report.py:67-95` and `scripts/demo/cao-interactive-full-pipeline-demo/scripts/verify_report.py:119-133` already codify a minimum-two-turn contract, and changing that would expand the work from tutorial and wrapper UX into report-contract redesign. Documenting the current behavior is the smaller, codebase-aligned move.

### Q3) What validation should count as "implementation review ready" for tasks `3.1` through `3.3`?

- Decision: `Deferrable`

#### Why this matters

- Review expectations should match the purpose of these demo packs.
- If the artifacts sound too much like CI or release gating, maintainers may over-engineer the validation story for something that is meant to be a learning walkthrough.

#### Examples

- A reviewer assumes the wrappers need new Pixi tasks or automated certification because the task wording sounds test-heavy.
- A maintainer wants to sanity-check the demo manually and is unsure whether that is considered sufficient evidence.

#### Why this question arises in code

- The repo has no Pixi task for shell-wrapper integration or syntax checking:

```markdown
[tool.pixi.tasks]
format = ...
lint = ...
typecheck = ...
test = ...
test-runtime = ...
```

- The current task wording in `tasks.md:14-18` does not say whether this review is meant to be manual demo validation or automated regression coverage.

#### Options

**Option A: Treat these tasks as manual demo-review work, not new Pixi automation**

- Code impact:
  - The tasks and README can clarify that developers review the wrapper flow manually.
  - Optional `bash -n` checks can still be mentioned as author sanity checks without introducing new repository tasks.

- User-facing difference:
  - The demo remains positioned as learning material and live documentation.
- Pros:
  - Matches the stated purpose of the demo pack as an operator tutorial.
  - Avoids implying that these wrappers are release-certification gates.
- Cons:
  - Review evidence is less standardized across maintainers.
  - Some regressions may only be caught during manual walkthroughs.

**Option B: Add explicit automated commands or new Pixi tasks for wrapper validation**

- Code impact:
  - Update tasks to prescribe concrete automated commands and potentially add new task wiring for them.

- User-facing difference:
  - The demo starts to read more like a certification surface than a tutorial pack.
- Pros:
  - Produces more standardized review evidence.
- Cons:
  - Pushes the change toward automation that the demo may not need.
  - Risks mis-positioning the demo as release gating.

#### Proposal (Recommended)

Use Option A. Clarify in the tasks and supporting prose that this validation is manual developer review of the demo flow, with any shell syntax checks treated as optional author aids rather than Pixi tasks or release-certification evidence.

#### Pros / Cons (Proposal)

- Pros:
  - Keeps the review bar aligned with the demo's purpose as learning material.
  - Avoids creating extra automation surface that the repo does not rely on for demo packs.
  - Reduces the chance that maintainers treat the demo as a release gate.
- Cons:
  - Validation remains more judgment-based than automated.
  - Future contributors may still choose different manual review depth unless the README/tasks are explicit.

> **DECISION: Do not create new Pixi tasks for Q3; treat tasks `3.1` through `3.3` as manual developer review of the demo workflow, with any shell syntax checks remaining lightweight and non-gating.**
> Rationale: `pyproject.toml:105-112` shows Pixi tasks are reserved for core repo workflows such as format, lint, typecheck, and unit/runtime tests, while this change is explicitly about improving a tutorial-style demo surface rather than adding release certification. Keeping validation manual preserves the demo's role as a learning source and live documentation artifact.
