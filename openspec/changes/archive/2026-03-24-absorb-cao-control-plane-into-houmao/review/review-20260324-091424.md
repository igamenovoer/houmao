## Summary

- **Solid architectural direction.** The proposal, design, and specs are internally coherent: the supported pair boundary stays stable, CAO drops from runtime dependency to oracle, and the new `control_core` seam is well-scoped. The decomposition (`service`, `registry`, `tmux_controller`, `provider_adapters`, `profile_store`, `compat/`) maps cleanly to the narrow CAO slice that is actually used.
- **Pair seam preservation is explicit and well-reasoned.** Decision 2 (land underneath existing `houmao_server_rest`) and Decision 3 (keep watch/mailbox/gateway separate) correctly protect the recently stabilized gateway-attach contract from the archived `reserve-window-zero-and-allow-auxiliary-agent-windows` change.
- **`HoumaoServerRestSession` inheritance from `CaoRestSession` is the hardest migration surface.** The current class hierarchy has `HoumaoServerRestSession` inheriting from `CaoRestSession`, which embeds deep CAO-specific lifecycle logic (profile rendering, tmux bootstrap, terminal start/stop via `CaoRestClient`). The design proposes replacing the server-side CAO authority but does not specify how this inheritance seam will be rewired.
- **The `proxy()` → child-CAO data flow is straightforward to replace**, but the install flow (`ChildCaoManager.install_agent_profile` shelling out to `cao install`) and profile-store behavior need Houmao-native reimplementation with careful parity.
- **Risk of scope creep in provider-adapter absorption.** The design proposes `control_core/provider_adapters/*` but the current CAO provider quirks live inside the pinned CAO source, not in Houmao code. Extracting and absorbing those quirks is potentially the most labor-intensive and under-specified part of the change.
- **The delta specs are comprehensive and well-structured**, with clear ADDED/MODIFIED/REMOVED sections. The removal list for `cao-server-launcher` is particularly thorough.

## Findings

### Must Fix / Blocking

1. **`HoumaoServerRestSession` inherits from `CaoRestSession` — the absorption design does not address this class hierarchy.** Today `HoumaoServerRestSession` (`src/houmao/agents/realm_controller/backends/houmao_server_rest.py`) is a thin subclass of `CaoRestSession` (`src/houmao/agents/realm_controller/backends/cao_rest.py`). It overrides only `backend` and the constructor, then delegates all session lifecycle (profile rendering, tmux session creation, terminal start/stop, prompt delivery, output capture) to `CaoRestSession`, which in turn calls `CaoRestClient` methods. After absorption, the server-side `/cao/*` routes will be served by the Houmao control core instead of a child CAO — but the *client-side* `HoumaoServerRestSession` still uses the same `CaoRestClient` with `path_prefix="/cao"` to call those routes. The design and specs are silent on whether the client-side class hierarchy changes, stays identical, or is gradually decoupled. This needs explicit treatment because: (a) `CaoRestSession._start_terminal()` calls `_ensure_required_executable(executable="cao-server", ...)` which will fail if `cao-server` is not on PATH — does the pair path still hit this code?; (b) the profile rendering and install logic inside `CaoRestSession` currently depends on shelling out to `cao install` — if the server now owns profile install natively, does the client-side session still try to install locally first?; and (c) `CaoRestSession` has 2100+ lines of deeply intertwined CAO-specific logic that the `houmao_server_rest` path may or may not actually exercise post-absorption.

> **DECISION: Refine.**
> Rationale: The class-hierarchy gap is real and needs explicit artifact coverage, but the finding overstates one mechanism. `HoumaoServerRestSession` currently inherits `CaoRestSession`, and `_start_terminal()` in `cao_rest.py` does hard-require `cao-server` on `PATH`, so the pair path needs a v1 override or extraction plan. But `CaoRestSession` does not shell out to `cao install`; it renders and writes a compatibility profile locally via `install_cao_profile()`. The artifact revision should target the real pair blockers: the `_start_terminal()` executable guard and the local CAO-profile-store ownership assumptions.

2. **Task list does not address `CaoRestSession` client-side refactoring.** The tasks (1.1–4.2) cover the server-side control core, `/cao/*` route rewiring, `houmao-srv-ctrl` CLI rework, and verification. None of them mention refactoring `CaoRestSession` or `HoumaoServerRestSession` to remove hidden dependencies on `cao-server` on PATH or local `cao install` calls. If these hidden dependencies survive, the pair path will still fail when `cao` is not installed, which contradicts the core spec requirement "The supported pair SHALL NOT require a supervised child `cao-server` process or an installed `cao` executable."

> **DECISION: Accept.**
> Rationale: The task list needs one explicit client-side migration item for `HoumaoServerRestSession` and `CaoRestSession`. The current tasks cover server and CLI rewiring, but they do not name the pair-facing runtime seam that still carries the `cao-server` startup assumption.

### Should Fix / Important

3. **Profile install absorption needs more specificity.** The current install flow chains: `houmao-srv-ctrl install` → `HoumaoServerClient.install_agent_profile()` → `POST /houmao/agent-profiles/install` → `HoumaoServerService.install_agent_profile()` → `ChildCaoManager.install_agent_profile()` → subprocess call to `cao install <agent_source> --provider <provider>`. The design says profile install will move to a "Houmao-owned profile store," and the `houmao-cao-control-core` spec requires it, but neither the design nor the tasks specify what "absorbing `cao install` behavior" means concretely. The pinned CAO `install` command performs: agent-source resolution (git clone or local path), YAML frontmatter validation, provider-specific profile copying, and profile metadata indexing. The new `control_core/profile_store` needs to reimplement at least the subset of this behavior that the pair actually uses. Task 1.2 mentions this but only as "absorb the used CAO profile-store and install behavior" — this should be broken down further.

> **DECISION: Accept.**
> Rationale: The current server install path is concretely `HoumaoServerService.install_agent_profile()` -> `ChildCaoManager.install_agent_profile()` -> subprocess `cao install`, so "absorb install behavior" is too vague. The next artifact revision should spell out the minimum owned behaviors in `control_core/profile_store` rather than leaving parity to implication.

4. **Health response `service` field compatibility needs explicit treatment.** Today `HoumaoHealthResponse.health_response()` in `service.py` returns `service="cli-agent-orchestrator"` (the CAO service identity string) in the root `/health` response, while `houmao_service="houmao-server"` is the Houmao extension. The `HoumaoServerRestSession` constructor checks `health.houmao_service != "houmao-server"` to validate the pair. After absorption, the server will no longer have a child CAO, but `/health` still needs to be compatible for existing clients. The delta spec for `houmao-server` says `child_cao` metadata disappears from health, but does not explicitly say whether `service="cli-agent-orchestrator"` stays (for CAO client compat) or changes to a Houmao-owned value. Given that `GET /health` is under the root namespace (not `/cao/*`), there is ambiguity about whether this remains CAO-compatible or becomes purely Houmao-owned.

> **DECISION: Accept.**
> Rationale: The artifacts should state the v1 health compatibility answer explicitly. The safest contract is to keep `service="cli-agent-orchestrator"` and `houmao_service="houmao-server"` on root `/health` while removing `child_cao`; `HoumaoHealthResponse` already inherits `CaoHealthResponse`, and `HoumaoServerRestSession` validates the Houmao extension field.

5. **Provider-adapter scope and source material are under-specified.** The design proposes `control_core/provider_adapters/*` for "provider-specific bootstrap, readiness, input, exit, and cleanup quirks." Currently these quirks live entirely inside the pinned CAO source (not in Houmao code). The design acknowledges the risk ("Provider bootstrap quirks are easy to accidentally duplicate") but neither the design nor the tasks specify which providers need adapters in v1 or what the adapter interface looks like. At minimum, `codex`, `claude_code`, and `gemini` need adapters since these are the three providers actually supported in the current pair. Task 1.1 bundles this into a single task alongside service interfaces, registries, tmux controller, and compat models — it should be broken out or at least enumerated.

> **DECISION: Accept.**
> Rationale: The under-specification is real, but the scope should be derived from the preserved pair compatibility surface, not from `BackendKind` alone. Today `houmao-srv-ctrl launch` and `houmao-srv-ctrl cao launch` accept `kiro_cli`, `claude_code`, `codex`, `gemini_cli`, `kimi_cli`, and `q_cli`; the artifacts need to enumerate what stays in v1 or explicitly retire anything narrower.

6. **`UrlLibProxyTransport` and `proxy()` method need rerouting or removal.** Today the `proxy()` method in `HoumaoServerService` forwards all `/cao/*` requests to `self.m_config.child_api_base_url` (the child CAO server at port+1). After absorption, those requests must instead dispatch to the local control core. The design says this clearly (Decision 4: "dispatch directly into the local control core"), but the current `ProxyTransport` protocol abstraction assumes an external HTTP target. The implementation will need either: (a) a new in-process dispatch adapter that satisfies the same `ProxyResponse` contract, or (b) direct route handler rewriting that bypasses `proxy()` entirely. Neither path is mentioned in the tasks.

> **DECISION: Accept.**
> Rationale: The implementation needs an explicit v1 dispatch pattern. Given the current `app.py` shape and side-effect hooks, the artifact revision should choose either an in-process `ProxyTransport` adapter or a full route rewrite instead of leaving the seam implicit.

7. **`houmao-cli` deprecation behavior for `cao_rest` backend is under-specified.** The `brain-launch-runtime` delta spec says standalone `cao_rest` sessions should fail with migration guidance. In `runtime.py`, the `cao_rest` backend path (around line 1033) creates a `CaoRestSession` directly when `launch_plan.backend == "cao_rest"`. The spec says "public runtime-management CLI entrypoints that would create or control standalone CAO-backed sessions SHALL fail fast," but the delta spec does not specify at which layer the guard should be placed: in the CLI argument parser, in `start_runtime_session()`, in the backend constructor, or elsewhere. Task 3.3 says "add explicit migration failures" but does not specify the insertion point. This matters because internal code may still need `CaoRestSession` for testing or parity verification.

> **DECISION: Accept.**
> Rationale: The public deprecation point is not specified tightly enough. The change should state where `houmao-cli` rejects `backend=cao_rest` while preserving internal `CaoRestSession` usage for parity tests and transitional pair machinery.

### Nice To Have

8. **Consider a transitional `startup_child=False` mode before full absorption.** The existing `HoumaoServerConfig.startup_child` flag already supports running `houmao-server` without starting the child CAO. This could serve as a useful intermediate validation step: run the pair with `startup_child=False` and the new control core active, compare behavior, and only then remove the child-CAO code paths entirely. The design does not mention this phased approach, but it could reduce risk during the migration.

> **DECISION: Refine.**
> Rationale: `startup_child=False` is a useful bring-up tactic because the flag already exists in `HoumaoServerConfig`, but it should stay an implementation strategy, not become part of the design contract. The end-state artifacts do not need to promise a phased mode to justify the absorption.

9. **`CaoApiError` and `CaoRestClient` remain in the pair client import chain after absorption.** `HoumaoServerClient` (`src/houmao/server/client.py`) extends `CaoRestClient` and uses `CaoApiError` for error handling. `agent_gateway.py` imports `CaoApiError` directly. After absorption, these CAO-named types remain in the public pair client chain. While this is not a functional issue, it creates naming confusion — the "CAO" prefix suggests dependency on CAO when the pair no longer depends on it. Consider whether a renaming or wrapping pass is warranted as a follow-up.

> **DECISION: Defer.**
> Rationale: The naming cleanup is real but non-blocking. We should first remove CAO as a runtime dependency while keeping the existing client and error types stable, then decide in a smaller follow-up whether to rename or wrap the remaining CAO-named types.

10. **The `houmao-cao-control-core` main spec does not exist yet.** The delta spec (`specs/houmao-cao-control-core/spec.md`) adds new requirements for the control core, but there is no corresponding main spec at `openspec/specs/houmao-cao-control-core/spec.md`. This is expected since the control core is new, but the task list should explicitly include creating the main spec as part of spec sync. Task 4.2 mentions updating docs but not main spec creation.

> **DECISION: Reject.**
> Rationale: In the current OpenSpec delta workflow, a new capability's main spec is created during spec sync or archive, not as an implementation task inside the change. The delta spec is the correct artifact here; no task-list change is required to make the current change coherent.

11. **Inbox queue compatibility scope is defensively correct but possibly over-engineered for v1.** The `houmao-cao-control-core` spec devotes significant space to the compatibility inbox queue separation from Houmao mailbox. While the separation principle is sound, the inbox route family (`/cao/terminals/{terminal_id}/inbox/messages`) may not be actively used by the pair today. If it is not exercised in the supported pair path, implementing a full compatibility inbox queue in the control core adds work with no immediate user value. Consider deferring inbox implementation to a later pass and returning 501 for inbox routes in v1.

> **DECISION: Refine.**
> Rationale: Auditing real usage is worthwhile, and I did not find a current pair caller beyond the exposed routes, client methods, and contract tests. But returning `501` would intentionally narrow a currently advertised `/cao/*` compatibility surface. The right v1 move is "keep inbox separate from mailbox and implement the smallest compatibility queue or stub that preserves the route contract," unless a later artifact revision explicitly retires that route family.

## Design Alignment With Existing System

### Existing Patterns Observed

- **Proxy-and-decorate pattern:** `houmao-server` currently proxies `/cao/*` requests to a child CAO server via `UrlLibProxyTransport` and decorates some responses with side effects (`sync_created_terminal`, `handle_deleted_session`, `note_prompt_submission`). See `src/houmao/server/app.py` lines 83–251.
- **Thin subclass delegation:** `HoumaoServerRestSession` inherits `CaoRestSession` and only overrides `backend` and the constructor, delegating all lifecycle to the parent. See `src/houmao/agents/realm_controller/backends/houmao_server_rest.py`.
- **Config-derived child identity:** `HoumaoServerConfig` derives child CAO port, root, and launcher config path from the parent config. See `src/houmao/server/config.py`.
- **Capability publication seam:** Pair-managed sessions publish gateway attachability through `gateway/attach.json`, `gateway/state.json`, and tmux session env vars (`AGENTSYS_GATEWAY_ATTACH_PATH`, `AGENTSYS_GATEWAY_ROOT`). See `src/houmao/agents/realm_controller/gateway_storage.py`.
- **Pydantic-based response models:** All HTTP responses use Pydantic v2 models. See `src/houmao/server/models.py`.
- **Stateful service with `m_` prefixed members:** `HoumaoServerService` follows the `m_` convention for instance state. See `src/houmao/server/service.py`.

### Proposed Design Alignment

- **Strongly aligned** on preserving the `/cao/*` route namespace, pair CLI surface, and gateway capability publication artifacts. The design explicitly protects the recently stabilized `houmao_server_rest` + gateway-attach seam.
- **Aligned** on the internal decomposition. The proposed `control_core/` package with `service`, `registry`, `tmux_controller`, `provider_adapters`, `profile_store`, and `compat/` layers follows the project's pattern of separating concerns into focused modules.
- **Partially misaligned** on the route handler rewriting strategy. The current `app.py` routes all call `resolved_service.proxy(...)` and then apply side effects. Post-absorption, each `/cao/*` route handler will need to call the control core directly and project a `Response`. The design says "dispatch directly into the local control core" but does not specify whether the `ProxyResponse` abstraction survives (adapted to wrap control-core results) or is replaced by direct Pydantic model returns. Given that `app.py` already uses direct Pydantic model returns for all `/houmao/*` routes, the more native pattern would be direct model returns for `/cao/*` too — but this means rewriting every `/cao/*` route handler, which is a larger change than the tasks suggest.
- **Partially misaligned** on the client-side hierarchy. `HoumaoServerClient` extending `CaoRestClient` is a reasonable transitional pattern, but `CaoRestClient` has methods like `health()` that return `CaoHealthResponse` (the raw CAO model), while `HoumaoServerClient` adds `health_extended()` returning `HoumaoHealthResponse`. After absorption, the `CaoRestClient` base class still works because the Houmao control core will project CAO-compatible responses — but the naming and inheritance direction may confuse future maintainers who expect "CAO" to mean "external CAO dependency."

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should the `HoumaoServerRestSession ← CaoRestSession` inheritance be handled during absorption?

- Decision: `Blocking`

#### Why this matters

- `HoumaoServerRestSession` is the pair-facing runtime backend. Its entire lifecycle (profile install, tmux session creation, terminal creation, prompt delivery, output capture, termination) is inherited from `CaoRestSession`. Post-absorption, the *server-side* authority changes but the *client-side* session class still calls `CaoRestClient` methods against `/cao/*` routes. If `CaoRestSession._start_terminal()` still calls `_ensure_required_executable(executable="cao-server")`, the pair path breaks without `cao-server` on PATH.

#### Examples

- Operator runs `houmao-srv-ctrl launch` → `HoumaoServerRestSession.__init__` → `CaoRestSession.__init__` → `_start_terminal()` → `_ensure_required_executable(executable="cao-server")` → **fails** because `cao-server` is not installed.
- Operator installs profile → `CaoRestSession` calls local `cao install` subprocess → **fails** because `cao` is not on PATH.

#### Why this question arises in code

- `src/houmao/agents/realm_controller/backends/cao_rest.py` line ~1655: `_ensure_required_executable(executable="cao-server", ...)` is called during `_start_terminal()`.
- `src/houmao/agents/realm_controller/backends/houmao_server_rest.py` line 15: `class HoumaoServerRestSession(CaoRestSession)` — full inheritance, no override of `_start_terminal`.
- The pair path goes: `HoumaoServerRestSession.__init__` → `super().__init__` → `CaoRestSession.__init__` → `self._start_terminal()`.

#### Options

**Option A: Keep inheritance, override `_start_terminal()` and profile install in `HoumaoServerRestSession` to skip local `cao-server` and `cao install` checks.**

- Code impact:
  - Override `_start_terminal()` in `HoumaoServerRestSession` to skip `_ensure_required_executable("cao-server")` and rely on the server-side control core for terminal creation.
  - Override profile rendering/install to route through `houmao-server` install API instead of local `cao install`.
  - Keep the rest of the inherited lifecycle (prompt, output, terminate) since those just call `CaoRestClient` methods against `/cao/*` routes that the control core now serves.
- User-facing difference:
  - None — pair behavior stays identical, but `cao-server` and `cao` are no longer required on PATH.
- Pros:
  - Minimal structural change; leverages existing ~2100 lines of tested lifecycle code.
  - Preserves the `CaoRestClient`-based request path, which still works because the control core projects CAO-compatible responses.
- Cons:
  - Deepens the fragile inheritance chain; future changes to `CaoRestSession` can break `HoumaoServerRestSession` in subtle ways.
  - Hidden CAO assumptions in inherited private methods remain landmines.

**Option B: Extract a shared base class or protocol for the common lifecycle, then make `HoumaoServerRestSession` and `CaoRestSession` siblings rather than parent-child.**

- Code impact:
  - Factor out prompt delivery, output capture, turn management, and tmux interaction into a shared base or protocol.
  - `HoumaoServerRestSession` gets its own `_start_terminal()` and profile install that use the Houmao control core.
  - `CaoRestSession` keeps its own `_start_terminal()` for the deprecated standalone path.
- User-facing difference:
  - None externally, but internal architecture is cleaner.
- Pros:
  - Removes the fragile inheritance chain.
  - Makes the pair path independent of `CaoRestSession` evolution.
- Cons:
  - Significant refactoring of a 2100-line class.
  - Risk of introducing regressions in the shared lifecycle extraction.

> **PROPOSED: Option A for v1, with Option B as a documented follow-up.**
> Rationale: The current `HoumaoServerRestSession` works by calling `CaoRestClient` against `/cao/*` routes, and those routes will continue to project CAO-compatible responses from the new control core. The primary blockers are the `_ensure_required_executable("cao-server")` guard and the local `cao install` subprocess call. Overriding those two code paths in `HoumaoServerRestSession` is surgical and low-risk. A full sibling refactoring is valuable but can be done as a separate change after absorption is stable.
>
> **DECISION: Refine Option A for v1.**
> Rationale: Keep `HoumaoServerRestSession` on the current `CaoRestSession` plus `CaoRestClient(path_prefix=\"/cao\")` seam for the initial absorption so the pair-facing runtime contract stays stable under the new server authority. But the artifact revision should name the exact override or extraction points precisely: `_start_terminal()` must stop requiring `cao-server`, and the pair path must stop depending on CAO-owned local profile-store assumptions. I am not accepting the review's "local `cao install` subprocess in `CaoRestSession`" premise, because that class currently writes the generated profile directly.

### Q2) Should the `/cao/*` route handlers in `app.py` keep the `ProxyResponse` abstraction or switch to direct Pydantic model returns?

- Decision: `Deferrable`

#### Why this matters

- Today every `/cao/*` route in `app.py` calls `resolved_service.proxy(method, path, params)` which returns a `ProxyResponse` (status_code + raw JSON bytes), then converts to a FastAPI `Response`. This works because the proxy target is an external HTTP server. After absorption, the "proxy target" is an in-process control core. The route handlers need a new dispatch mechanism.

#### Examples

- `POST /cao/sessions/{session_name}/terminals` currently proxies to child CAO and intercepts the response for `sync_created_terminal`. Post-absorption, it would call `control_core.create_terminal(...)` and project the result into a CAO-compatible response shape.

#### Why this question arises in code

- `src/houmao/server/app.py` lines 83–251: All `/cao/*` handlers use `resolved_service.proxy(...)`.
- `src/houmao/server/service.py` lines 339–349: `proxy()` uses `UrlLibProxyTransport` targeting an external URL.
- `/houmao/*` routes in `app.py` (lines 252–422) already use direct Pydantic model returns — the pattern is established.

#### Options

**Option A: Replace `proxy()` with an in-process adapter that still returns `ProxyResponse`, preserving the existing route handler structure.**

- Code impact:
  - Create an in-process `ControlCoreTransport` that satisfies the `ProxyTransport` protocol.
  - `proxy()` dispatches to the control core instead of HTTP.
  - Route handlers and side-effect hooks (`sync_created_terminal`, etc.) remain unchanged.
- User-facing difference:
  - None.
- Pros:
  - Minimal route handler changes; all side-effect hooks stay in place.
  - Preserves the `ProxyResponse` abstraction for potential future HTTP-backed engines.
- Cons:
  - Adds a serialization/deserialization round-trip (control core → JSON → `ProxyResponse` → FastAPI `Response`) that is unnecessary for an in-process call.
  - The `ProxyResponse` abstraction becomes misleading since nothing is being proxied.

**Option B: Rewrite `/cao/*` route handlers to call the control core directly and return Pydantic models, matching the `/houmao/*` route pattern.**

- Code impact:
  - Each `/cao/*` route handler calls `control_core.service.create_terminal(...)` and projects the result through `control_core.compat.cao_models`.
  - Side-effect hooks move inline or into a service method.
  - Remove `ProxyTransport` and `proxy()`.
- User-facing difference:
  - None externally; eliminates one unnecessary serialization layer.
- Pros:
  - Consistent with `/houmao/*` route pattern.
  - No phantom "proxy" abstraction.
- Cons:
  - Rewrites all ~15 `/cao/*` route handlers.
  - More implementation work upfront.

> **PROPOSED: Option A for the initial absorption, with Option B as a follow-up optimization.**
> Rationale: The `ProxyTransport` protocol in `src/houmao/server/service.py` is a clean abstraction boundary. Replacing `UrlLibProxyTransport` with a `ControlCoreTransport` that serializes the control-core result into a `ProxyResponse` preserves all existing route handlers and side-effect hooks. This minimizes the blast radius of the absorption change. The unnecessary serialization round-trip is negligible for a local server and can be eliminated in a follow-up that rewrites the `/cao/*` handlers to match the direct-return pattern used by `/houmao/*` routes.
>
> **DECISION: Accept PROPOSED for v1.**
> Rationale: The existing `/cao/*` handlers in `app.py` are already structured around `proxy()` plus Houmao-owned side effects. An in-process compatibility transport preserves that stable seam and keeps the initial absorption focused on replacing runtime authority, not rewriting route plumbing. Direct model returns can be a later cleanup once parity is stable.

### Q3) What is the minimum viable set of provider adapters for v1?

- Decision: `Blocking`

#### Why this matters

- The control core's `provider_adapters/*` layer is responsible for provider-specific terminal bootstrap, readiness detection, input delivery quirks, exit behavior, and cleanup. These quirks currently live inside the pinned CAO source and are not replicated in Houmao code. The design mentions "provider-specific bootstrap and shutdown quirks" but does not enumerate which providers or which quirks.

#### Examples

- Codex: requires `codex` on PATH, uses `codex exec` for headless or `codex` for TUI, has specific SIGTERM/SIGKILL exit behavior.
- Claude Code: requires `claude` on PATH, uses `claude -p` for headless, has different readiness detection (waits for specific prompt patterns).
- Gemini: requires `gemini` on PATH, different startup timing.

#### Why this question arises in code

- `src/houmao/agents/realm_controller/backends/cao_rest.py`: The `CaoRestSession` delegates provider-specific behavior to the CAO server, which internally selects provider adapters. After absorption, Houmao must own those adapters.
- No existing `provider_adapters/` directory in `src/houmao/`.

#### Options

**Option A: Absorb only `codex` and `claude_code` provider adapters for v1, since those are the actively used providers in the pair.**

- Code impact:
  - Two provider adapter modules plus a base protocol/class.
  - Gemini adapter deferred.
- User-facing difference:
  - Gemini sessions through the pair would fail with an explicit "unsupported provider" error until the adapter is added.
- Pros:
  - Reduces v1 scope.
  - Codex and Claude Code are the primary pair users.
- Cons:
  - Gemini is listed as a supported backend in `BackendKind` and may have existing users.

**Option B: Absorb all three (`codex`, `claude_code`, `gemini`) provider adapters for v1.**

- Code impact:
  - Three provider adapter modules plus a base protocol/class.
- User-facing difference:
  - All currently supported providers remain available.
- Pros:
  - No functional regression for any existing provider.
  - Matches the current CAO capability.
- Cons:
  - More implementation work.
  - Gemini adapter may be under-tested if there are few Gemini users.

> **PROPOSED: Option B — absorb all three provider adapters for v1.**
> Rationale: `BackendKind` in `src/houmao/agents/realm_controller/models.py` lists `gemini_headless` as a supported backend alongside `codex_headless` and `claude_headless`. The `cao-rest-client-contract` main spec at `openspec/specs/cao-rest-client-contract/spec.md` documents `codex` → `codex` and `claude` → `claude_code` provider mappings explicitly, and the pinned CAO source supports gemini. Dropping gemini support silently would violate the non-regression intent of the absorption. The gemini adapter quirks are typically simpler than codex or claude, so the marginal implementation cost is low.
>
> **DECISION: Reject the A/B framing.**
> Rationale: The minimum v1 adapter set must be derived from the compatibility surface we are preserving, not just from the three native headless tools. Today the supported pair CLI accepts six provider identifiers (`kiro_cli`, `claude_code`, `codex`, `gemini_cli`, `kimi_cli`, `q_cli`) in both `src/houmao/srv_ctrl/commands/launch.py` and `src/houmao/srv_ctrl/commands/cao.py`. The next artifact revision should enumerate the exact v1 provider set and any intentional retirements explicitly; we should not silently narrow it to two or three providers.

### Q4) Where should the `cao_rest` deprecation guard be placed?

- Decision: `Deferrable`

#### Why this matters

- The `brain-launch-runtime` delta spec requires that public CLI entrypoints creating standalone `cao_rest` sessions fail fast with migration guidance. But `CaoRestSession` may still be needed internally for parity verification, testing, or as the base class for `HoumaoServerRestSession`. The guard placement determines whether internal usage is also blocked.

#### Examples

- `houmao-cli start-session --backend cao_rest` should fail with migration guidance.
- Internal test code creating a `CaoRestSession` for parity verification should still work.
- `HoumaoServerRestSession(...)` creating a session through the pair should still work (it inherits from `CaoRestSession` but uses `backend = "houmao_server_rest"`).

#### Why this question arises in code

- `src/houmao/agents/realm_controller/runtime.py` line ~1033: `if launch_plan.backend == "cao_rest":` — this is where the backend is instantiated.
- `src/houmao/agents/realm_controller/backends/cao_rest.py`: The `CaoRestSession` class itself.
- `src/houmao/cli.py` → `src/houmao/agents/realm_controller/cli.py`: The CLI entrypoint.

#### Options

**Option A: Guard at the CLI layer — reject `--backend cao_rest` in the CLI argument parser before reaching `start_runtime_session()`.**

- Code impact:
  - Add a check in the CLI entrypoint that rejects `cao_rest` as a backend choice.
  - Internal code and tests can still instantiate `CaoRestSession` directly.
- User-facing difference:
  - CLI users get a clear migration error.
  - Programmatic internal usage is unaffected.
- Pros:
  - Clean separation between public deprecation and internal capability.
  - Tests and parity verification continue to work.
- Cons:
  - Does not prevent accidental public usage through programmatic API.

**Option B: Guard at the `start_runtime_session()` function — reject `backend="cao_rest"` unless an explicit internal-use flag is passed.**

- Code impact:
  - Add a `_allow_deprecated_cao_rest: bool = False` parameter.
  - CLI passes `False` (default), tests and parity code pass `True`.
- User-facing difference:
  - Same as Option A for CLI users.
- Pros:
  - Catches both CLI and programmatic public usage.
- Cons:
  - Adds a flag parameter that is arguably unnecessary complexity.

> **PROPOSED: Option A — guard at the CLI layer.**
> Rationale: The CLI entrypoint at `src/houmao/cli.py` → `src/houmao/agents/realm_controller/cli.py` is the only public operator-facing surface for starting sessions. Guarding there is the simplest change, preserves internal `CaoRestSession` usage for testing and parity, and matches the delta spec's language about "public runtime-management CLI entrypoints." The `start_runtime_session()` function is an internal API used by both the CLI and `houmao-server`; adding a deprecation flag there conflates public surface policy with internal capability.
>
> **DECISION: Accept PROPOSED.**
> Rationale: The delta spec deprecates public CLI entrypoints, not the internal runtime constructor. Guarding in `src/houmao/agents/realm_controller/cli.py` keeps the migration error where operators see it while preserving `CaoRestSession` for parity tests, debugging, and the transitional `HoumaoServerRestSession` inheritance path.
