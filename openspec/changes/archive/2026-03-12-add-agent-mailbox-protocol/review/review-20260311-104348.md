## Summary

- The change is exceptionally thorough in scope and articulation. The proposal, design, five delta specs, and two skill packages are internally coherent and well-layered. The canonical message model, principal-based addressing, and filesystem transport design are sound.
- The prior review (review-20260311-102408) identified five critical and actionable open questions (skill injection path, mailbox enablement mechanism, sentinel parsing, manifest persistence, and error handling). Those findings remain valid and are not repeated here.
- This review focuses on areas the prior review did not cover: the skill artifacts themselves, the `rules/scripts/` materialization lifecycle, principal registration bootstrapping, staging area crash semantics, message-ID collision risk, and several spec-to-skill alignment gaps.
- The `rules/scripts/` design is the most architecturally novel element and the one with the highest integration risk. The specs and skills assume those scripts exist but no artifact defines who creates them, when they are materialized, or how they are versioned relative to the protocol version.
- The skill artifacts (`email-via-filesystem`, `email-via-mail-system`) are well-structured and operationally detailed, but several guardrails in the skills go beyond what the specs formally require, creating a soft specification gap.

## Findings

### Must Fix / Blocking

1. **`rules/scripts/` materialization lifecycle is undefined.** The design (`design.md:182-189`), the fs-transport spec (requirement "Filesystem mailbox root publishes shared mailbox rules"), and the `email-via-filesystem` skill all assume that scripts like `deliver_message.py`, `update_mailbox_state.py`, `insert_standard_headers.py`, and `repair_index.py` exist under `rules/scripts/`. The filesystem-layout reference (`skills/email-via-filesystem/references/filesystem-layout.md`) even names four specific scripts. However, no artifact answers:
   - Who writes these scripts (are they platform-owned templates materialized by the runtime, or authored per-mailbox by a human operator)?
   - When are they created (at mailbox initialization time, at brain-build time, or manually before first use)?
   - How are they versioned (tied to `protocol-version.txt`, shipped inside the Python package, or part of the agent-def-dir)?
   - What happens when an agent invokes a missing script (the spec says the agent should use them but has no failure scenario for script absence)?

   This is blocking because the entire write-path correctness model depends on agents invoking these scripts rather than improvising SQLite/lock-file mutations. If the scripts do not exist at the expected paths when an agent first attempts delivery, the soft-enforcement model collapses.

2. **No spec or design coverage for mailbox initialization / bootstrap.** The fs-transport spec has a scenario "Creating a filesystem mailbox initializes required layout" but no artifact specifies:
   - Which runtime operation triggers mailbox initialization (is it `build_brain_home`, `start_runtime_session`, the first `mail send`, or a standalone `init-mailbox` command)?
   - Whether the principal's own mailbox directory (`mailboxes/<principal>/inbox|sent|archive|drafts`) is created at initialization or lazily at first delivery.
   - Whether `index.sqlite` schema creation and `rules/` population are atomic.

   Without this, the tasks.md artifact (already flagged as missing in the prior review) cannot specify the initialization implementation.

3. **Principal registration bootstrapping has a circular dependency.** The design says a participant "dynamically joins the mail group by creating that symlink entry under `mailboxes/`" (`design.md:176`). The fs-transport spec says "Missing or dangling principal mailbox symlinks are treated as delivery-time registration failures" (`design.md:282`). But:
   - Who creates the initial `mailboxes/<principal>/` directory for a local (non-symlinked) principal?
   - The SQLite principal registry should record each principal's registration, but who does the initial insert?
   - If the runtime creates the principal directory during session start, that is a mailbox mutation that by the spec's own rules should go through `rules/scripts/`. But the scripts may not exist yet if the mailbox has not been initialized.

   This chicken-and-egg needs an explicit bootstrap path that is allowed to operate without the shared scripts.

### Should Fix / Important

4. **Skill references filesystem-layout.md names concrete scripts that the design does not commit to.** The `email-via-filesystem/references/filesystem-layout.md` names four specific scripts: `deliver_message.py`, `insert_standard_headers.py`, `update_mailbox_state.py`, `repair_index.py`. The design only describes script categories (delivery commit, mailbox-state mutation, lock orchestration, reindex) without committing to specific filenames. If the skill template is platform-owned and ships as a fixed reference, these filenames become de-facto API contracts. Either the design should commit to these names, or the skill reference should use category descriptions rather than filenames.

5. **Staging area crash semantics are unspecified.** The design mentions `staging/` (`design.md:173`) and the write path starts with "Compose and validate the message in `staging/`" (`design.md:228`). But:
   - What happens if the sender crashes after writing to staging but before acquiring locks?
   - Are staged messages garbage-collected? By whom? On what trigger?
   - Can a reindex/repair operation detect orphaned staging artifacts?

   The fs-transport spec has scenarios for crash recovery of the SQLite index but none for staging cleanup.

6. **Message-ID format and collision risk.** The design shows the message-ID pattern `msg-20260311T041500Z-a1b2c3d4` (`design.md:82`). The trailing segment appears to be 8 hex chars (32 bits). For a local-only filesystem transport this is likely sufficient, but:
   - The spec does not formalize the ID generation algorithm.
   - 32-bit random suffixes have a ~50% collision probability at ~77k messages (birthday bound). For long-lived mailboxes with many agents, this may be too low.
   - The spec says `message_id` must be "globally unique immutable" (`design.md:62`) but the format shown does not guarantee global uniqueness.

   Consider either specifying a longer random component (e.g., UUID4) or explicitly scoping uniqueness to "unique within one mailbox root."

7. **Thread ancestry chain (`references`) can grow without bound.** The design says replies extend `references` (`design.md:309`) but sets no limit. Email RFCs recommend trimming `References` headers. For long-lived agent threads (hundreds of turns), the `references` list in YAML front matter could become very large. The spec should either set a cap or explicitly document that unbounded growth is acceptable for the filesystem transport.

8. **The `email-via-filesystem` skill's guardrails exceed spec requirements, creating an implicit spec.** The skill SKILL.md contains guardrails like "Do not copy delivered canonical message bodies into `inbox/` or `sent/`" and "Do not treat mailbox filenames alone as unread or read markers" that encode design decisions not captured as formal spec requirements. If the skill template is the authoritative runtime-facing document and the specs are the authoritative design-facing document, these should be in sync. Currently, a spec-compliant implementation that ignores the skill text could violate these guardrails.

9. **No spec scenario for multi-transport coexistence or transport migration.** The design reserves `AGENTSYS_MAILBOX_EMAIL_*` env vars and the `email-via-mail-system` skill explicitly documents future compatibility. But the specs do not address:
   - Whether a mailbox root can be served by multiple transports simultaneously.
   - Whether migrating from `filesystem` to `email` transport requires re-initializing the mailbox or can reuse the existing message corpus.
   - Whether `AGENTSYS_MAILBOX_TRANSPORT` can change mid-session (the refresh semantics are env-binding focused, not transport-switching focused).

   This is acceptable for v1, but a brief non-goal statement in the email-compatibility spec would prevent future implementors from assuming multi-transport was designed-for.

10. **Attachment digest validation is unspecified.** The canonical message includes `sha256` for attachment references (`design.md:101`). The spec requires "digest metadata when available" (`agent-mailbox-protocol/spec.md`). But no scenario covers:
    - When or by whom the digest is computed (sender at composition time? delivery script? recipient at read time?).
    - What happens when a `path_ref` attachment's content changes after delivery (digest mismatch on read).
    - Whether digest validation is recommended, required, or informational.

### Nice To Have

11. **Consider a `mailbox doctor` / `mailbox status` diagnostic command.** Given the complexity of the filesystem layout (symlinks, locks, SQLite, rules/scripts, staging), a diagnostic command that validates mailbox health (dangling symlinks, orphaned staging, lock staleness, missing scripts, SQLite integrity) would be valuable for operators and for automated testing.

12. **The `email-via-mail-system` skill could be deferred entirely.** It is explicitly non-operational and serves only as compatibility documentation. Shipping it as a skill template in the first implementation adds artifact surface without runtime value. Consider whether a design document or spec appendix would serve the same purpose with less maintenance burden.

13. **Attachment `kind` values could use a Literal type.** The design shows `path_ref` and `managed_copy` as the two kinds. A spec requirement constraining the valid `kind` values would prevent future drift and make validation easier.

14. **Consider documenting the expected `rules/README.md` content template.** If the runtime materializes the `rules/` directory, a standard README template would help human participants understand the mailbox root they are inspecting.

## Design Alignment With Existing System

### Existing pattern(s) observed

- **Build-time skill projection** (`brain_builder.py:288-293` validates skill existence, `brain_builder.py:470-476` projects skills via symlink/copy): Skills are sourced from `brains/skills/<name>/SKILL.md` and projected into the brain home under the tool adapter's `skills_destination`. The proposed `email-via-filesystem` skill follows this structure (SKILL.md + references/ + agents/) but the injection path is different since it is runtime-owned rather than recipe-selected.

- **Tool adapter contract** (`tests/fixtures/agents/brains/tool-adapters/codex.yaml`, `claude.yaml`): Tool adapters define `skills_projection.destination` and `skills_projection.mode` (symlink or copy). The mailbox skill needs to land at the same destination for the tool to discover it, regardless of injection path.

- **Session manifest as resumption contract** (`manifest.py`, `boundary_models.py`): `SessionManifestPayloadV2` with backend-specific sections (`headless`, `cao`) is the authoritative state for resume. Mailbox state needs a home here (as the prior review's Q4 proposed).

- **Env var injection surface** (`LaunchPlan.env`, `HeadlessInteractiveSession._plan.env`): The `LaunchPlan` carries env vars as a frozen `dict[str, str]` set at build time. Mailbox env vars need to enter through this surface for the initial session and through a new refresh surface for active sessions.

### Proposed design alignment assessment

**Aligns well:**

- The `email-via-filesystem` skill structure (SKILL.md + references/ + agents/openai.yaml) matches the existing skill directory convention under `brains/skills/`.
- The `AGENTSYS-...` principal-id convention reuses the existing agent identity namespace (`agent_identity.py`).
- The `mail` CLI subcommand following `--agent-identity` resolution is consistent with `send-prompt`, `send-keys`, and `stop-session` patterns in `cli.py`.
- The env-var binding contract (`AGENTSYS_MAILBOX_*`) follows the established `AGENTSYS_*` namespace for framework-level bindings.

**Mismatches requiring attention:**

- **Skill materialization timing mismatch.** The existing pattern is: recipe selects skills at declaration time -> `BrainBuilder` validates and projects at build time -> tool reads from home at launch time. The mailbox skill is described as "runtime-generated from platform-owned templates" but the skill files in this change (`skills/email-via-filesystem/`) are fully authored, not templates with substitution points. The env-var indirection means they do not need literal substitution, but the injection mechanism (when and how they appear in the brain home) is a new pattern. The prior review's Q1 addresses this; this review adds the observation that the skill content is already template-ready (no substitution needed), which favors build-time projection (Option A from the prior review).

- **The `rules/scripts/` pattern has no codebase precedent.** No existing component generates executable scripts into a runtime directory for agents to invoke. The closest pattern is `launch.sh` generation in `brain_builder.py:_build_launch_helper()`, but that is a launch-time artifact consumed by the runtime, not an agent-facing script. The `rules/scripts/` approach essentially creates a new contract surface where the runtime publishes executable code that agents are instructed to call. This is architecturally novel and needs careful treatment for: permissions (scripts must be executable), error reporting (script failures need to propagate clearly through agent output), and security (agents invoking arbitrary scripts in a shared directory).

- **No precedent for runtime-to-agent structured request/response contracts.** The existing `send_prompt()` is fire-and-forget text. The `mail` command introduces a structured request template (YAML-in-prompt with `AGENTSYS_MAIL_REQUEST`) and a structured response contract (sentinel-delimited JSON). This is a new interaction pattern for the runtime. While the prior review's Q3 addresses the parsing approach, the broader pattern of "runtime sends structured prompt, expects structured response, validates, and presents to operator" is a new runtime capability that may benefit from explicit naming and documentation as a reusable pattern.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Who creates and maintains the shared `rules/scripts/` helper scripts, and when are they materialized?

- Decision: `Blocking`

#### Why this matters

- The entire write-path correctness model depends on agents invoking `rules/scripts/` rather than hand-rolling SQLite and lock-file mutations. If these scripts do not exist when an agent first attempts delivery, the agent must either fail or fall back to direct manipulation, both of which violate design intent.

#### Examples

- First session starts against a freshly initialized mailbox root. Agent receives a `mail send` request. The `email-via-filesystem` skill instructs it to use `rules/scripts/deliver_message.py`. That script does not exist yet because nobody created it. The agent either fails the delivery or improvises direct SQLite writes.
- Mailbox protocol version is bumped from 1 to 2. The `deliver_message.py` script needs to be updated. Who triggers the update? Does the runtime detect a version mismatch and regenerate?

#### Why this question arises in code

- `brain_builder.py:_build_launch_helper()` generates `launch.sh` from Python code at build time, establishing a precedent for runtime-generated scripts. But `launch.sh` targets are well-defined (one script per brain home).
- `rules/scripts/` targets are shared across all participants in a mailbox root, not per-brain-home. The creation point must be mailbox-root-scoped, not brain-home-scoped.
- The `email-via-filesystem/references/filesystem-layout.md` names four specific scripts but no artifact defines their source or generation logic.

#### Options

**Option A: Runtime materializes `rules/` from package-internal templates during mailbox initialization**

- Code impact:
  - A new `mailbox_init.py` module ships `rules/` templates as package data or inline string templates.
  - Mailbox initialization (triggered by `start_runtime_session` or a new `init-mailbox` command) copies/generates the `rules/` tree from those templates.
  - `protocol-version.txt` and script versions are tied together.
  ```python
  # mailbox_init.py
  def initialize_mailbox_root(mailbox_root: Path) -> None:
      if not mailbox_root.exists():
          materialize_rules_from_templates(mailbox_root / "rules")
          create_sqlite_schema(mailbox_root / "index.sqlite")
          (mailbox_root / "protocol-version.txt").write_text("1\n")
  ```
- User-facing difference:
  - Scripts are auto-created when the mailbox root is first initialized. Operators can inspect and customize them post-creation.
- Pros:
  - Single authoritative source for script content (Python package).
  - Version-tied to the protocol.
  - No manual step before first use.
- Cons:
  - Operator customizations can be overwritten on upgrade unless a merge/conflict strategy is defined.
  - Shipped scripts become part of the package's public API surface.

**Option B: Scripts are manually authored and maintained per-mailbox by operators**

- Code impact:
  - Runtime validates script presence during mailbox operations and fails with a clear error if missing.
  - Documentation provides reference implementations.
- User-facing difference:
  - Operators must set up `rules/scripts/` before first use.
- Pros:
  - Maximum flexibility per-mailbox.
  - No package-internal template maintenance.
- Cons:
  - High friction for first use. Easy to forget.
  - Script quality and correctness vary per installation.
  - Contradicts the "runtime-owned" philosophy of the design.

**Option C: Hybrid: runtime materializes defaults, operators can override**

- Code impact:
  - Same as Option A for initial creation.
  - A `.managed` marker or version file in `rules/scripts/` tracks which scripts are runtime-managed vs. operator-customized.
  - Runtime upgrades can refresh managed scripts but skip operator-customized ones.
- User-facing difference:
  - Zero-friction setup with customization escape hatch.
- Pros:
  - Best of both: auto-setup + operator flexibility.
  - Clean upgrade path.
- Cons:
  - More complex materialization logic.
  - Needs a merge/conflict UX for upgrades.

> **PROPOSED: Option C (hybrid: runtime materializes defaults, operators can override).**
> Rationale: The design describes `rules/scripts/` as the shared mailbox-local coordination surface (`design.md:180-189`) and the skill instructs agents to use these scripts for all sensitive operations. This requires zero-friction availability. The `launch.sh` generation pattern in `brain_builder.py:_build_launch_helper()` demonstrates that the runtime already generates executable scripts from templates. Extending that to mailbox scripts follows the same pattern but at mailbox-root scope. The hybrid approach preserves the operator-customization path mentioned in the design ("mailbox maintainers should treat `rules/` as part of the operational mailbox surface") while ensuring scripts exist on first use. A `.managed` marker keeps upgrades safe.
>
> **DECISION: Option A (runtime materializes and maintains a fixed managed `rules/scripts/` set during mailbox initialization).**
> Rationale: I agree with the review that zero-friction availability is mandatory, but I do not want to take on operator-override semantics in v1. The change already leans on runtime-owned artifacts elsewhere: mailbox skills are platform-owned, bindings are runtime-managed env vars, and the filesystem mailbox has explicit initialization scenarios. The most coherent contract is therefore that the runtime/package owns creation and upgrade of the sensitive-operation scripts at mailbox-init time, with `protocol-version.txt` tying the script set to the on-disk protocol. Mailbox-local customization can still happen in `rules/README.md`, `rules/protocols/`, and additional optional helpers, but the core managed scripts should not be overrideable until a later change explicitly defines merge/conflict behavior.

### Q2) Should the filesystem-layout reference in the skill name concrete script filenames, or should it use abstract categories?

- Decision: `Deferrable`

#### Why this matters

- The `email-via-filesystem/references/filesystem-layout.md` names four concrete scripts (`deliver_message.py`, `insert_standard_headers.py`, `update_mailbox_state.py`, `repair_index.py`). If these are shipped as part of the skill template and injected into every agent session, the filenames become part of the agent-facing API. Renaming or restructuring the scripts later would require updating the skill template, rebuilding all brain homes, and potentially confusing agents that cached the old names.

#### Examples

- Developer decides `deliver_message.py` should be split into `stage_message.py` and `commit_delivery.py`. Every agent session using the old skill template would reference a non-existent script.
- A new transport-specific script is added. Agents using the old skill template would not know about it.

#### Why this question arises in code

- The `email-via-filesystem` SKILL.md uses abstract instructions ("use the shared helper script from `rules/scripts/` for sensitive steps") rather than naming scripts directly. But the filesystem-layout reference names them explicitly. Agents reading the reference as part of the skill would see concrete filenames.
- The `design.md` describes script categories but does not commit to filenames.

#### Options

**Option A: Name concrete scripts in the reference (current approach)**

- Code impact:
  - No change. Filenames in `filesystem-layout.md` are treated as stable API.
- User-facing difference:
  - Agents and operators see exact filenames. Less ambiguity.
- Pros:
  - Unambiguous for agents. Easy to validate ("does `deliver_message.py` exist?").
  - Concrete references reduce LLM hallucination risk.
- Cons:
  - Locks in filenames. Harder to restructure scripts later.

**Option B: Use abstract categories in the reference, let `rules/README.md` map categories to scripts**

- Code impact:
  - `filesystem-layout.md` describes `scripts/` as "contains helper scripts for delivery, state mutation, header normalization, and index repair."
  - Each mailbox root's `rules/README.md` lists the actual script filenames and their categories.
  - Agents are instructed to read `rules/README.md` first to discover available scripts.
- User-facing difference:
  - Agents must do one extra read (`rules/README.md`) before knowing script names.
- Pros:
  - Decouples skill template from script naming. Easier to evolve.
  - Supports per-mailbox customization of script names.
- Cons:
  - One more file read per mailbox interaction.
  - LLMs may struggle with indirect discovery vs. concrete names.

> **PROPOSED: Option A (keep concrete script names in the reference).**
> Rationale: The design already uses env-var indirection for all transport-specific paths, so the skill template is stable across path changes. Script filenames within `rules/scripts/` are a smaller, more stable namespace. Concrete names reduce the chance of LLM agents hallucinating script filenames or failing to discover them. The four named scripts (`deliver_message.py`, `insert_standard_headers.py`, `update_mailbox_state.py`, `repair_index.py`) map cleanly to the design's four script categories. If renaming becomes necessary, a `protocol-version.txt` bump with migration guidance is the appropriate mechanism.
>
> **DECISION: Option A (keep concrete script names in the reference and treat them as part of the v1 mailbox contract).**
> Rationale: This follows from the Q1 decision. If the runtime owns the managed script set, the names should be stable and discoverable without an extra level of indirection. The current design, skill, and layout reference are already converging on these four names, so I would rather formalize them than require agents to discover a second mapping layer in `rules/README.md`. If the script set changes later, that should happen through an intentional protocol-versioned migration rather than by leaving filenames abstract now.

### Q3) Should `message_id` use a longer random component to reduce collision risk?

- Decision: `Deferrable`

#### Why this matters

- The design example shows `msg-20260311T041500Z-a1b2c3d4` with 8 hex chars (32 bits of randomness). The spec says `message_id` must be "globally unique immutable." For a single-host filesystem mailbox, collision probability at 77k messages is ~50% (birthday bound). Multi-agent workflows generating many messages per day could approach this.

#### Examples

- A research team runs 10 agents sending 100 messages/day for 6 months = ~180k messages. The birthday-bound collision probability for 32-bit IDs at 180k messages is effectively certain (>99%).
- An attachment references a message by `message_id`. A collision would cause the wrong message to be associated.

#### Why this question arises in code

- The design uses `message_id` as the filesystem filename (`messages/<YYYY-MM-DD>/<message-id>.md`) and as the canonical key in SQLite. A collision would cause a file overwrite or a SQLite constraint violation.
- The design example format is `msg-{timestamp}-{8hex}`. Since the timestamp includes seconds, two messages in the same second share the same prefix and rely entirely on the 8-hex suffix for uniqueness.

#### Options

**Option A: Keep 8-hex suffix (current design)**

- Code impact:
  - No change.
- User-facing difference:
  - Shorter, more readable message IDs.
- Pros:
  - Compact. Easy to type and reference.
- Cons:
  - 32-bit birthday bound. Not safe for high-volume mailboxes.

**Option B: Extend to UUID4 suffix (128 bits)**

- Code impact:
  - Message-ID format becomes `msg-{timestamp}-{uuid4}` (e.g., `msg-20260311T041500Z-a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6`).
- User-facing difference:
  - Longer filenames but practically zero collision risk.
- Pros:
  - Standard UUID4 collision probability is astronomically low.
  - Aligns with RFC 5322 `Message-ID` uniqueness expectations for email compatibility.
- Cons:
  - Longer filenames. Less human-friendly.

**Option C: Use 16-hex suffix (64 bits) as a middle ground**

- Code impact:
  - `msg-{timestamp}-{16hex}`.
- User-facing difference:
  - Moderately longer IDs. Birthday bound at ~4 billion messages.
- Pros:
  - ~4B-message birthday bound is safe for any local-only deployment.
  - More compact than UUID4.
- Cons:
  - Non-standard length. Neither UUID4 nor compact.

> **PROPOSED: Option B (UUID4 suffix) for `message_id`, with the design documenting the format as `msg-{YYYYMMDDTHHMMSSZ}-{uuid4-no-dashes}`.**
> Rationale: The spec says `message_id` must be "globally unique immutable" and the design's email-compatibility mapping sends it as `Message-ID` in RFC 5322 headers. RFC 5322 Message-IDs are expected to be globally unique. UUID4 (128 bits) makes collisions negligible for any realistic deployment scale. The timestamp prefix provides human-readable sorting and date-partitioned filesystem storage. The extra length is acceptable because message IDs are primarily machine-consumed (SQLite keys, symlink targets, protocol references) rather than human-typed.
>
> **DECISION: Option B (use a UUID4 suffix, formatted as `msg-{YYYYMMDDTHHMMSSZ}-{uuid4-no-dashes}`).**
> Rationale: The current 8-hex example is too weak for a field the change describes as globally unique and then reuses in filenames, SQLite keys, message threading, and future email header mapping. I do not want uniqueness to depend on mailbox volume assumptions or a future collision-retry rule. The timestamp prefix keeps the IDs readable and date-oriented, while the UUID4 suffix makes the uniqueness claim credible for both filesystem transport and future `Message-ID` compatibility.

### Q4) Should the spec explicitly require a bootstrap/initialization path that operates without `rules/scripts/`?

- Decision: `Blocking`

#### Why this matters

- The design says all sensitive mailbox mutations (SQLite, locks) should go through shared scripts in `rules/scripts/`. But mailbox initialization itself is a sensitive mutation (creating SQLite schema, populating `rules/`, creating lock directories). This creates a circular dependency: you need the scripts to do mailbox mutations, but you need to do a mailbox mutation to create the scripts.

#### Examples

- Runtime starts the first-ever session for a new mailbox root. The mailbox root does not exist yet. The runtime needs to create the directory tree, initialize SQLite, and populate `rules/scripts/`. These are exactly the kinds of operations the design says should go through shared scripts.
- A `repair_index.py` script needs to recreate the SQLite schema. But if the script itself is missing (filesystem corruption), the repair cannot proceed.

#### Why this question arises in code

- The `_build_launch_helper()` in `brain_builder.py` shows that the runtime can generate scripts at build time without needing pre-existing scripts. Mailbox initialization could follow the same model: a Python module in the package performs initialization directly, then places the scripts for subsequent agent use.
- The design's "agent compliance is soft-enforced" note (`design.md:191`) suggests the design already accepts that some operations bypass the shared scripts. Initialization should be explicitly one of those.

#### Options

**Option A: Spec explicitly allows runtime-direct initialization without shared scripts**

- Code impact:
  - Add a spec scenario: "When the runtime initializes a new filesystem mailbox root, the runtime performs initialization directly through package-internal code without requiring pre-existing `rules/scripts/` helpers."
  - Initialization code lives in the Python package, not in `rules/scripts/`.
- User-facing difference:
  - Mailbox initialization is always reliable, even from a clean slate.
- Pros:
  - Breaks the circular dependency cleanly.
  - Makes the initialization path testable without filesystem fixtures.
- Cons:
  - Creates a second code path for SQLite schema creation (package code + `repair_index.py`).

**Option B: `rules/scripts/` are pre-seeded from a package template as part of directory creation, before any SQLite or lock operations**

- Code impact:
  - Initialization is a two-step sequence: (1) create directories and copy scripts from templates, (2) invoke the newly-created scripts for SQLite and lock setup.
- User-facing difference:
  - Same outcome, but scripts are used from the start.
- Pros:
  - Single code path for SQLite operations.
  - Scripts are exercised from the very first operation.
- Cons:
  - More complex initialization sequence.
  - If script materialization fails, the entire initialization fails.

> **PROPOSED: Option A (spec explicitly allows runtime-direct initialization).**
> Rationale: The initialization path is a one-time operation that creates the substrate on which `rules/scripts/` operates. The runtime is the trusted component that owns the protocol version and schema. Having the runtime use package-internal initialization code (like `brain_builder.py:_build_launch_helper()` uses package-internal templates) is the natural pattern. The shared scripts in `rules/scripts/` are for agent-facing operations after the mailbox is initialized. Adding a spec scenario that explicitly allows runtime-direct initialization breaks the circular dependency without weakening the agent-facing script requirement.
>
> **DECISION: Option A (spec explicitly allows a runtime-direct bootstrap path with no pre-existing `rules/scripts/`).**
> Rationale: This is the clean way to break the chicken-and-egg identified in the review. Initialization is substrate creation, not an ordinary mailbox interaction, so it should be a trusted package-internal runtime operation that creates the mailbox root, SQLite schema, locks area, `rules/`, and the managed scripts themselves. After bootstrap, the agent-facing contract becomes stricter: sensitive operations should go through the published scripts. This keeps bootstrap simple and aligns with existing repo patterns where package code materializes runtime artifacts before agents consume them.

### Q5) Should `archive/` and `drafts/` folder semantics be specified or explicitly deferred?

- Decision: `Deferrable`

#### Why this matters

- The design layout (`design.md:169`) and filesystem-layout reference both show `archive/` and `drafts/` as mailbox subfolders. The fs-transport spec requires symlink-registered mailboxes to expose these folders. But no spec scenario covers archive or draft workflows, and no skill instruction addresses them.

#### Examples

- An agent is told to "archive" a message. The `email-via-filesystem` skill has no archive instructions. The agent guesses: should it move the inbox symlink to `archive/`? Update SQLite? Both?
- An operator inspects a principal's `drafts/` folder expecting to find in-progress messages. Nothing has been written there because no workflow uses it.

#### Why this question arises in code

- The `email-via-filesystem` SKILL.md mentions `inbox` and `sent` projections in its send/reply workflow but never mentions `archive` or `drafts`.
- The fs-transport spec scenarios only cover `inbox` and `sent` projections.
- The principal registration requirement says symlink targets must expose `inbox/`, `sent/`, `archive/`, and `drafts/`, but only two of these have defined semantics.

#### Options

**Option A: Explicitly defer archive and drafts to a follow-up change**

- Code impact:
  - Add a non-goal: "Archive and draft mailbox workflows are deferred to a follow-up change."
  - Still create the directories at initialization (for forward compatibility) but do not define semantics.
- User-facing difference:
  - Clear expectations. No undefined behavior.
- Pros:
  - Reduces v1 scope. Clearer spec.
- Cons:
  - Directories exist but are unused, which may confuse operators.

**Option B: Add minimal spec scenarios for archive and drafts**

- Code impact:
  - Add spec requirements: "Moving a message to archive removes its inbox projection and creates an archive projection." "Drafts are staged messages not yet delivered."
  - Update the `email-via-filesystem` skill with archive/drafts instructions.
- User-facing difference:
  - Full mailbox folder semantics from day one.
- Pros:
  - Complete mailbox UX.
- Cons:
  - Increases v1 scope. More implementation and testing.

> **PROPOSED: Option A (explicitly defer archive and drafts to a follow-up change).**
> Rationale: The design's migration plan (`design.md:565-573`) already has seven steps. Adding archive/draft semantics increases implementation scope without supporting the core async messaging use case. The `email-via-filesystem` skill only needs `inbox` and `sent` for the v1 `check/send/reply` operations. Creating the directories at initialization preserves forward compatibility. A brief non-goal statement prevents ambiguity.
>
> **DECISION: Option A (explicitly defer archive and drafts semantics to a follow-up change).**
> Rationale: The current change is about durable async delivery, not full mailbox UX. `check`, `send`, and `reply` do not require archive or draft workflows, and the injected filesystem skill does not define them. I want the next spec revision to say clearly that `archive/` and `drafts/` may be provisioned as reserved placeholder directories for forward compatibility, but v1 behavior must not rely on any archive/draft semantics beyond that. That keeps the filesystem layout future-friendly without pretending this change already specifies those workflows.
