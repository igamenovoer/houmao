# Discussion: Managed Mailbox Recovery Parser Contract

- Feature/Change: `add-agent-mailbox-protocol`
- Timestamp: `2026-03-12T02:36:18Z`
- Context: This doc captures a blocking implementation question discovered while applying task `3.3` in `tasks.md`. Inputs reviewed: `openspec/changes/add-agent-mailbox-protocol/tasks.md`, `openspec/changes/add-agent-mailbox-protocol/specs/agent-mailbox-fs-transport/spec.md`, `src/gig_agents/mailbox/protocol.py`, `src/gig_agents/mailbox/managed.py`, and `src/gig_agents/mailbox/assets/rules/scripts/repair_index.py`.

The key constraint is that v1 promises mailbox-local managed scripts under `rules/scripts/` while also treating canonical Markdown message files as the durable recovery source of truth. Our current canonical message parser lives in package code that depends on `pydantic` and `PyYAML`, so task `3.3` needs an explicit contract decision before implementation continues.

---

## Summary of recommended defaults

1) Keep `rules/scripts/*.py` stdlib-only for v1, including `repair_index.py`.
2) Add a narrow recovery parser in the managed mailbox layer that extracts only the fields needed to rebuild structural mailbox state from canonical message documents.
3) Add parity tests so canonical serialization and recovery parsing do not drift on the recovery-critical fields.

## Q1) How should `repair_index.py` parse canonical message documents while preserving the managed-script contract?

- Decision: `Blocking`

### Why this matters

- Task `3.3` cannot be completed until we decide whether `repair_index.py` is a portable mailbox-local tool or a thin wrapper around installed package internals.
- This choice affects operator expectations during recovery, the portability of the shared mailbox root, and whether the current fs-transport spec remains true as written.

### Examples

- A shared mailbox root is copied to a recovery host that has `python3.11` but does not have `gig-agents`, `pydantic`, or `PyYAML` installed. If `repair_index.py` imports the package parser directly, recovery fails before reindex begins.
- A follow-up change adds a new canonical front-matter field. If `repair_index.py` reuses the package parser, it picks up the new field and validation automatically. If `repair_index.py` uses a narrow stdlib parser, we must update that parser deliberately and verify parity on the recovery-critical fields.

### Why this question arises in code

- The fs-transport spec requires Python helpers under `rules/scripts/` to assume only the Python standard library and Python `>=3.11`: `openspec/changes/add-agent-mailbox-protocol/specs/agent-mailbox-fs-transport/spec.md:74-90`.
- The same spec requires the transport to rebuild structural mailbox indexes from canonical Markdown messages when the SQLite index is missing or unusable: `openspec/changes/add-agent-mailbox-protocol/specs/agent-mailbox-fs-transport/spec.md:228-246`.
- The existing canonical parser is `parse_message_document()` in `src/gig_agents/mailbox/protocol.py:308-344`, but that module imports `pydantic` and `yaml` at `src/gig_agents/mailbox/protocol.py:16-17`.
- The managed mailbox helper layer is intentionally stdlib-only, as stated in `src/gig_agents/mailbox/managed.py:1`.
- The remaining recovery entrypoint is still a stub in `src/gig_agents/mailbox/assets/rules/scripts/repair_index.py:1-28`, and task `3.3` is the first pending task in this area in `openspec/changes/add-agent-mailbox-protocol/tasks.md:13-18`.

```python
# Decision point inside repair_index.py
for message_path in iter_canonical_messages(mailbox_root):
    document = message_path.read_text(encoding="utf-8")
    if reuse_package_parser:
        message = parse_message_document(document)  # depends on pydantic + yaml
    else:
        message = parse_recovery_document(document)  # stdlib-only subset parser
    rebuild_index_rows(connection, message)
```

### Options

**Option A: Reuse the package parser from `gig_agents.mailbox.protocol`**

- Code impact:
  - `repair_index.py` imports `parse_message_document()` and uses `MailboxMessage` as the reindex record source.
  ```python
  from gig_agents.mailbox.protocol import parse_message_document

  for message_path in iter_canonical_messages(mailbox_root):
      message = parse_message_document(message_path.read_text(encoding="utf-8"))
      rebuild_index_rows(connection, message)
  ```
  - If we choose this option, we should update the spec and design to say that managed Python scripts may depend on the installed `gig-agents` package and its runtime dependencies.
- User-facing difference:
  - Recovery benefits from the same validation behavior as delivery-time parsing, but `repair_index.py` only works where package dependencies are installed.
- Pros:
  - Reuses the exact canonical parser and validation logic already implemented.
  - Minimizes drift risk between delivery-time parsing and recovery-time parsing.
  - Keeps reindex implementation smaller.
- Cons:
  - Conflicts with the current stdlib-only helper-script contract in the fs-transport spec.
  - Makes mailbox-local recovery less portable and more dependent on local environment setup.
  - Changes the operational meaning of `rules/scripts/` from "portable shared tools" to "wrappers around the installed application."

**Option B: Keep `repair_index.py` stdlib-only with a narrow recovery parser**

- Code impact:
  - Add a recovery-oriented parser in the managed mailbox layer that understands the exact v1 front-matter subset needed for reindex and default mailbox-state reconstruction.
  ```python
  for message_path in iter_canonical_messages(mailbox_root):
      document = message_path.read_text(encoding="utf-8")
      front_matter, body = split_markdown_document(document)
      payload = parse_v1_front_matter_subset(front_matter)
      record = build_recovery_record(payload, body)
      rebuild_index_rows(connection, record)
  ```
  - Add tests that serialize canonical messages through `serialize_message_document()` and verify that the recovery parser extracts the same recovery-critical fields.
- User-facing difference:
  - `repair_index.py` remains usable anywhere the mailbox root is present and Python `3.11` is available, but recovery validation becomes intentionally scoped to v1 reindex needs rather than full authoring-time canonical validation.
- Pros:
  - Preserves the spec's stdlib-only managed-script contract exactly as written.
  - Keeps `rules/scripts/` portable and mailbox-local for disaster recovery or ad hoc repair.
  - Aligns with the existing design of `src/gig_agents/mailbox/managed.py`.
- Cons:
  - Duplicates some parsing and validation logic that already exists in `protocol.py`.
  - Requires deliberate parity tests and maintenance when canonical message fields evolve.
  - May validate a narrower surface than the full canonical parser, which has to be documented and tested clearly.

### Proposal (Recommended)

Choose Option B for v1. The current artifacts already make a strong portability promise for managed Python scripts under `rules/scripts/`, and `managed.py` was written to preserve that contract. `repair_index.py` should therefore remain stdlib-only and add a narrow recovery parser that understands the exact v1 message document structure we serialize today, validates only the invariants needed for recovery, and rebuilds deterministic default mailbox state when prior mutable SQLite records are unavailable. If we later decide the managed scripts may depend on the installed package, that should be a follow-up spec revision rather than an implicit implementation shortcut.

### Pros / Cons (Proposal)

- Pros:
  - Keeps the v1 fs-transport contract true without artifact churn.
  - Preserves portability for mailbox-local repair and reindex flows.
  - Fits the existing managed helper architecture and task decomposition.
- Cons:
  - Adds implementation and test work now.
  - Creates a second parser boundary that must stay aligned with canonical serialization.

> **DECISION: Remove the stdlib-only restriction for managed Python mailbox scripts and publish a sibling `requirements.txt` so agents can install the declared dependencies before invoking those scripts.**
> Rationale: The current fs-transport spec requires Python helpers under `rules/scripts/` to rely only on the standard library at `openspec/changes/add-agent-mailbox-protocol/specs/agent-mailbox-fs-transport/spec.md:74-90`, but that restriction is now intentionally being relaxed. This decision allows `repair_index.py` and the other managed scripts to reuse package-level helpers such as `parse_message_document()` in `src/gig_agents/mailbox/protocol.py:308-344`, even though that module depends on `pydantic` and `yaml` via `src/gig_agents/mailbox/protocol.py:16-17`. Publishing `requirements.txt` next to the managed scripts preserves discoverability for agents and operators while avoiding duplicated recovery-specific parsing logic. The change also establishes a broader contract: mailbox-managed Python scripts may depend on PyPI packages as long as those dependencies are declared in the colocated `requirements.txt`.

### Notes / Follow-ups

- This decision supersedes the original Option B proposal above.
- The fs-transport spec should be revised to replace the stdlib-only script requirement with a dependency-disclosure requirement for a colocated `rules/scripts/requirements.txt`.
- Bootstrap/materialization should add that `requirements.txt` file alongside the managed scripts so the on-disk mailbox contract matches the decision.
- If implementation reuses package helpers for recovery, reindex should still be explicitly scoped to recovery-critical fields such as `message_id`, `thread_id`, `in_reply_to`, `references`, `created_at_utc`, participant identities, and attachment metadata required for structural index rows.
- Parity tests should use canonical documents produced by `serialize_message_document()` so recovery coverage tracks the real on-disk format rather than hand-written fixtures only.
