## Summary

- The high-level direction is coherent: making `houmao-server` the public authority, keeping CAO as an internal v1 adapter, and separating `houmao-srv-ctrl` from `houmao-cli` matches the repo's current ownership boundaries.
- The persistence-bucket thinking is strong and aligns with existing runtime/gateway/registry documentation that already distinguishes durable artifacts from live control-plane state.
- The main gap is not intent but contract specificity: the artifacts do not yet define how a `houmao-server` session fits the repo's strict backend, manifest, gateway, and registry schemas.
- A second blocking gap is that `houmao-srv-ctrl launch` registration has no explicit authority model for runtime-owned manifests, registry publication, or gateway/mailbox reuse.
- The compatibility promise is also under-specified because the change says "supported/pinned" `cao` and `cao-server` versions without actually naming the pinned upstream build that parity tests will target.

## Findings

### Must Fix / Blocking

1. **The change does not define the concrete runtime/backend schema needed to represent a `houmao-server` session without leaking CAO details.** The change artifacts require the runtime to persist `houmao-server` base URL plus session/terminal identity while hiding upstream-adapter details ([design.md](/data1/huangzhe/code/houmao/openspec/changes/add-houmao-server/design.md#L112), [specs/brain-launch-runtime/spec.md](/data1/huangzhe/code/houmao/openspec/changes/add-houmao-server/specs/brain-launch-runtime/spec.md#L3), [specs/houmao-server/spec.md](/data1/huangzhe/code/houmao/openspec/changes/add-houmao-server/specs/houmao-server/spec.md#L190)). But the current repo hard-validates a closed backend set in [models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/models.py#L16), hard-validates backend-specific manifest sections in [boundary_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/boundary_models.py#L239) and [manifest.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/manifest.py#L113), and only allows gateway-capable CAO metadata for `backend='cao_rest'` in [gateway_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_models.py#L138), [gateway_service.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_service.py#L122), and [runtime.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/runtime.py#L1957). As written, implementation can only proceed by either breaking those strict contracts or smuggling `houmao-server` state through CAO-shaped fields, which contradicts the stated goal that CAO stays an internal adapter.

> **DECISION: Accept.**
> Rationale: The current change still does not name a first-class persisted backend identity, manifest section, or gateway attach metadata for `houmao-server`. Given the strict backend-discriminated contracts in [models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/models.py#L16), [boundary_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/boundary_models.py#L239), [manifest.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/manifest.py#L113), and [gateway_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_models.py#L138), this gap is real and still needs artifact revision.

2. **`houmao-srv-ctrl launch` registration is missing an authority model for runtime-owned manifests, registry pointers, and gateway/mailbox compatibility.** The artifacts say delegated CLI launches must register live agents with `houmao-server` so the server can manage watch and state features ([proposal.md](/data1/huangzhe/code/houmao/openspec/changes/add-houmao-server/proposal.md#L14), [design.md](/data1/huangzhe/code/houmao/openspec/changes/add-houmao-server/design.md#L165), [specs/houmao-srv-ctrl-cao-compat/spec.md](/data1/huangzhe/code/houmao/openspec/changes/add-houmao-server/specs/houmao-srv-ctrl-cao-compat/spec.md#L65)). But current discovery and follow-up control paths depend on runtime-owned manifest pointers and session roots: shared-registry records require `runtime.manifest_path` in [registry_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/registry_models.py#L41) and are documented as pointers back to authoritative runtime state in [record-and-layout.md](/data1/huangzhe/code/houmao/docs/reference/registry/contracts/record-and-layout.md#L107); name-based resolution falls back to that manifest pointer in [runtime.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/runtime.py#L763); and gateway attach plus mail notifier support reload the runtime-owned manifest in [gateway_service.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_service.py#L136) and [gateway_service.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_service.py#L430). The change never says whether a delegated `houmao-srv-ctrl launch` also creates a Houmao session root and manifest, whether registry publication still points to those artifacts, or whether CLI-launched sessions are intentionally outside gateway/mailbox/runtime-resume support in v1.

> **DECISION: Accept, with refinement.**
> Rationale: The missing authority model is a real gap. The refinement is that the fix should define Houmao-owned authoritative session artifacts first, then describe how transitional registry publication and gateway/mailbox compatibility point back to those artifacts while those subsystems still depend on runtime-owned manifests. That keeps the long-term architecture Houmao-owned without pretending the current pointer-oriented subsystems disappear in v1.

### Should Fix / Important

3. **The compatibility target is described as "supported/pinned" but never actually pinned to an upstream CAO build.** The design and specs repeatedly promise parity with the supported `cao-server` / `cao` version ([design.md](/data1/huangzhe/code/houmao/openspec/changes/add-houmao-server/design.md#L58), [specs/houmao-server/spec.md](/data1/huangzhe/code/houmao/openspec/changes/add-houmao-server/specs/houmao-server/spec.md#L3), [specs/houmao-srv-ctrl-cao-compat/spec.md](/data1/huangzhe/code/houmao/openspec/changes/add-houmao-server/specs/houmao-srv-ctrl-cao-compat/spec.md#L3)), but no artifact records the exact version, tag, or commit that defines that contract. The current repo docs only show an example install from the `hz-release` branch of the supported fork in [cao_server_launcher.md](/data1/huangzhe/code/houmao/docs/reference/cao_server_launcher.md#L7), which is not a stable enough parity oracle for "full HTTP API" and "full command family" claims. The change should name one concrete upstream source of truth and bind parity tests to it.

> **DECISION: Accept.**
> Rationale: The change now says "supported" and "pinned" more clearly than it did when this review was written, but it still does not name one exact upstream `cao` / `cao-server` source of truth. Full parity claims need a concrete pinned build, tag, commit, or equivalent reference in the artifacts and verification plan.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should a `houmao-server`-backed runtime session be represented in persisted backend identity?

- Decision: `Blocking`

#### Why this matters

- The current runtime, manifest, gateway, and registry layers all branch on an explicit backend identity and strict backend-specific payload sections.
- If this is left implicit, implementation will either leak child-CAO details into Houmao-owned contracts or break existing schema validation.

#### Examples

- A session started through the new runtime mode persists `houmao-server` base URL and terminal identity, then later gateway attach tries to load backend-specific metadata.
- A resume flow needs to know whether to instantiate a CAO client directly or a `houmao-server` client without guessing from incidental fields.

#### Why this question arises in code

- The backend enum is closed in [models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/models.py#L16).
- Manifest validation is backend-specific in [boundary_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/boundary_models.py#L280) and materialization is backend-specific in [manifest.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/manifest.py#L130).
- Gateway attach contracts and attach-time behavior branch on `backend='cao_rest'` in [gateway_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_models.py#L182), [gateway_service.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_service.py#L122), and [runtime.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/runtime.py#L1957).

```text
start_runtime_session()
  -> LaunchPlan.backend
  -> SessionManifestPayloadV3.backend + backend-specific section
  -> GatewayAttachContractV1.backend + backend_metadata
  -> resume/attach branch on backend
```

#### Options

**Option A: Introduce a new explicit runtime backend kind and dedicated persisted sections for `houmao-server`.**

- Code impact:
  - Extend `BackendKind`, manifest section models, gateway attach metadata, runtime factory branches, and registry identity to represent `houmao-server` explicitly.
  - Keep child-CAO details in server-owned internal state instead of the runtime-owned manifest contract.
- User-facing difference:
  - Persisted session artifacts clearly say "this is a `houmao-server` session," which matches the public architecture.
- Pros:
  - Matches the design goal that CAO becomes an internal adapter.
  - Preserves a clean migration seam for future non-CAO backends.
  - Avoids overloading CAO-shaped fields with non-CAO meaning.
- Cons:
  - Requires broader schema and test updates across runtime, gateway, and registry contracts.

**Option B: Keep `backend='cao_rest'` and reinterpret existing CAO fields as the `houmao-server` transport contract.**

- Code impact:
  - Reuse existing manifest and gateway schemas with minimal structural changes.
  - Encode `houmao-server` base URL and related state inside the current CAO-shaped payloads.
- User-facing difference:
  - Persisted runtime artifacts continue to look CAO-native even when the public server is `houmao-server`.
- Pros:
  - Smaller immediate implementation delta.
  - Reuses more of the current runtime/gateway stack unchanged.
- Cons:
  - Conflicts with the stated goal that upstream-adapter details stay out of the public backend identity.
  - Makes future non-CAO replacement harder because the durable contract still says `cao_rest`.
  - Leaves registry and attach metadata semantically misleading.

> **PROPOSED: Pick Option A and make `houmao-server` a first-class persisted backend identity with its own manifest and gateway sections.**
> Rationale: The current repo is built around strict backend-discriminated contracts in [models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/models.py#L16), [boundary_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/boundary_models.py#L280), and [gateway_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_models.py#L182). Reusing `cao_rest` would keep the durable truth CAO-shaped even after the design says CAO is only an internal v1 adapter, which weakens the future replacement seam and makes persisted artifacts misleading. The wider schema churn is real, but it is the repo-native way to express a new public authority boundary.
>
> **DECISION: Accept Option A.**
> Rationale: `houmao-server` is the public authority and CAO is an internal adapter, so the persisted backend identity should say `houmao-server`, not `cao_rest`. The repo already uses explicit backend-discriminated manifest and gateway schemas, so adding a first-class backend kind plus dedicated manifest and gateway sections is the repo-native way to express this boundary cleanly.

### Q2) What authoritative artifact set should `houmao-srv-ctrl launch` produce for delegated CAO launches?

- Decision: `Blocking`

#### Why this matters

- Today, follow-up control, gateway attach, registry discovery, and mailbox notifier support all assume there is an authoritative runtime-owned manifest path.
- The new pairwise replacement story is incomplete until delegated CLI launches can be located and reasoned about through those same authority boundaries, or until the change explicitly scopes them out.

#### Examples

- An operator runs `cao launch` via `houmao-srv-ctrl`; later another tool wants to resolve the live agent by `agent_name` and attach a gateway or inspect mailbox support.
- `houmao-server` learns about the launched agent, but the shared registry still needs a durable pointer if runtime-style control flows are meant to work.

#### Why this question arises in code

- Shared-registry records point back to runtime-owned manifest/session-root artifacts in [registry_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/registry_models.py#L41) and [record-and-layout.md](/data1/huangzhe/code/houmao/docs/reference/registry/contracts/record-and-layout.md#L107).
- Agent resolution falls back through registry manifest pointers in [runtime.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/runtime.py#L763).
- Gateway attach and mailbox notifier support both require a readable runtime-owned manifest in [gateway_service.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_service.py#L136) and [gateway_service.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_service.py#L430).

```text
houmao-srv-ctrl launch
  -> delegated `cao launch`
  -> ??? authoritative Houmao artifact
  -> registry.runtime.manifest_path
  -> resolve_agent_identity() / gateway attach / notifier
```

#### Options

**Option A: `houmao-srv-ctrl launch` must create or update a Houmao runtime session root, manifest, and registry record after delegated CAO success.**

- Code impact:
  - Add a post-launch materialization step that writes the authoritative Houmao session artifacts and publishes registry pointers to them.
  - Let `houmao-server` registration reference those same runtime-owned artifacts.
- User-facing difference:
  - CLI-launched sessions participate in the same discovery/control/gateway flows as runtime-launched sessions.
- Pros:
  - Fits the repo's existing pointer-oriented runtime and registry design.
  - Makes the pairwise replacement story operationally coherent.
  - Avoids splitting the system into two incompatible notions of a "live Houmao-managed session."
- Cons:
  - Requires more concrete post-launch reconciliation logic with delegated CAO outputs.

**Option B: Keep registration server-only in v1 and explicitly scope CLI-launched sessions out of runtime-owned manifest/gateway/mailbox compatibility.**

- Code impact:
  - `houmao-srv-ctrl` would only tell `houmao-server` about the live session and would not publish runtime-owned artifacts for it.
  - Specs/docs/tests would need to state that runtime/gateway/mailbox flows do not apply to delegated CLI launches in v1.
- User-facing difference:
  - Operators can launch through the compatibility CLI, but some existing Houmao follow-up flows would only work for runtime-started sessions.
- Pros:
  - Smaller v1 scope.
  - Avoids forcing a full session-root materialization path immediately.
- Cons:
  - Weakens the "paired replacement" claim substantially.
  - Creates two different authority models for live sessions inside the same product surface.
  - Makes registry-based resolution and gateway reuse inconsistent.

> **PROPOSED: Pick Option A and require delegated CLI launches to materialize the same authoritative Houmao runtime artifacts that discovery, gateway, and mailbox flows already use.**
> Rationale: The current system is explicitly pointer-oriented: registry records point at `manifest.json`, not at duplicated inline state, and gateway-side features reload that manifest as the durable contract in [registry_models.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/registry_models.py#L41), [record-and-layout.md](/data1/huangzhe/code/houmao/docs/reference/registry/contracts/record-and-layout.md#L107), and [gateway_service.py](/data1/huangzhe/code/houmao/src/houmao/agents/realm_controller/gateway_service.py#L430). Reusing that same authority path for CLI-launched sessions fits existing architecture much better than inventing a second server-only truth. If the team wants the smaller Option B scope, the artifacts need to narrow the promised capabilities much more explicitly.
>
> **DECISION: Accept Option A, refined.**
> Rationale: Delegated `houmao-srv-ctrl launch` should materialize Houmao-owned authoritative session artifacts after CAO succeeds, or the paired replacement story stays incomplete and downstream discovery/control semantics diverge. Refine the proposal slightly: the durable contract that must exist is a Houmao-owned session root and manifest, with any v1 transitional registry publication and gateway/mailbox pointers referencing those artifacts while those subsystems still use manifest-backed authority.
