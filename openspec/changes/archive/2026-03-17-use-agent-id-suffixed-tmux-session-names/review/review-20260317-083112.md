## Summary

- **Solid motivation and scope**: The change correctly identifies that the current `houmao-<session_id>` naming is opaque and that operator-readable tmux handles are valuable. The `<canonical-agent-name>-<agent-id[:6]>` scheme is a clean, deterministic improvement.
- **Existing resolution infrastructure already supports the new naming**: The tmux-local enumeration fallback in `runtime.py` (lines 1484–1532) already matches on `payload.agent_name` rather than requiring session name == canonical name, so suffixed handles will work with the current resolution pipeline.
- **Cross-cutting but well-mapped**: The design correctly identifies all major touch points — runtime start, manifest persistence, tmux-local resolution, demo state, demo cleanup — and the task list tracks them.
- **Spec delta quality is high**: Scenarios are concrete, edge cases (collision, ambiguity, stale pointers) are enumerated, and the distinction between canonical identity and tmux handle is consistently maintained.
- **Risk area — `generate_tmux_session_name()` is a second naming path**: The existing `tmux_runtime.py:generate_tmux_session_name()` (lines 262–283) generates `AGENTSYS-<tool>-<role>[-N]` names for legacy/helper paths. The design and tasks mention backend allocators should also use the shared helper, but the spec does not explicitly require retiring or reconciling this function.
- **Risk area — tmux character limits and separator ambiguity**: tmux session names have practical limits (~256 chars) and the `-` separator in `<agent-name>-<agent-id-prefix>` can be ambiguous when the canonical name itself contains trailing hex-like segments.

## Findings

### Must Fix / Blocking

1. **The delta-spec for `agent-identity` silently drops the existing `agent_name` validation requirement.**
   The main spec at `openspec/specs/agent-identity/spec.md` (lines 56–89) contains a detailed requirement ("User-provided agent names are validated and MUST NOT contain a standalone `AGENTSYS` token") with six scenarios. The delta-spec does not list this requirement under either ADDED or MODIFIED, yet the conceptual changes to how canonical agent names relate to tmux handles could interact with validation. This requirement must be explicitly preserved (carried forward unchanged) or, if intentionally omitted from the delta, a note should confirm it is unmodified. As written, merging the delta could be misread as replacing the entire spec.

2. **The delta-spec for `agent-identity` drops the `--agent-identity` path-vs-name dispatch requirement and the inexact-prefix warning requirement.**
   Similarly, the main spec's first two requirements (lines 7–27: path-like dispatch; lines 46–53: inexact prefix warning) are not referenced in the delta. The delta only shows ADDED and MODIFIED sections, but because the delta-spec completely restructures the name-based resolution requirement (which was the largest existing requirement), implementers need to be clear whether the delta is additive or replaces the full spec. Recommend adding a "## UNCHANGED Requirements" header listing carried-forward requirements, or a preamble confirming delta semantics.

3. **Task 1.1 does not specify where the shared helper should live.**
   The design says "introduce one shared tmux-session-name derivation helper," and Task 1.1 asks to add it, but neither names a target module. The natural home is `agent_identity.py` (which already owns `derive_auto_agent_name_base()` at line 125 and `derive_agent_id_from_name()`) or `tmux_runtime.py` (which owns `generate_tmux_session_name()` at line 262). The tasks should specify the module to avoid implementer ambiguity and ensure the helper doesn't end up duplicated.

### Should Fix / Important

4. **`generate_tmux_session_name()` in `tmux_runtime.py` needs an explicit retirement or reconciliation plan.**
   This function (lines 262–283) generates `AGENTSYS-<tool>-<role>[-N]` style names and is currently imported in at least the headless backends and the CAO backend allocation path. The design's migration plan step 2 mentions updating "headless and CAO backend allocation helpers," but the tasks don't explicitly call out deprecating or removing `generate_tmux_session_name()`. If it remains, there will be two competing naming paths: the new shared `<agent-name>-<agent-id[:6]>` helper and the old `AGENTSYS-<tool>-<role>` generator. Task 1.2 should explicitly address whether `generate_tmux_session_name()` is retired, wrapped, or kept for a specific legacy use case.

5. **Manifest fallback `agent_name = tmux_session_name` in `manifest.py` (line 322–323) is the exact anti-pattern the design calls out, but the spec doesn't add a negative scenario.**
   `_resolve_manifest_identity()` at `manifest.py:322–323` sets `agent_name = tmux_session_name` when `agent_name is None` for headless backends. The design (Decision 4, lines 109–129) correctly identifies this as incorrect under the new scheme. However, the delta-spec only states the positive requirement ("manifest construction should always receive explicit agent_name, agent_id, and tmux_session_name") without a scenario that would fail if the fallback fires with a suffixed handle. Adding a negative scenario (e.g., "WHEN manifest construction receives tmux_session_name=AGENTSYS-gpu-270b87 and agent_name is None, THEN the persisted agent_name SHALL NOT be AGENTSYS-gpu-270b87") would make the anti-pattern testable.

6. **Collision-extension algorithm lacks a specified step size.**
   The spec says "extend the agent_id prefix until the tmux session name becomes unique, up to the full authoritative agent_id." It does not specify whether extension is character-by-character (7, 8, 9, ...) or in larger steps (e.g., double). For a 32-character MD5 hex digest, character-by-character from 6 to 32 means up to 27 collision resolution attempts, each requiring a tmux session existence check. This is unlikely to matter in practice but should be specified for determinism and testability.

7. **The delta-spec for `agent-identity` adds extensive gateway environment variable requirements (lines 187–230) that appear to codify already-implemented behavior rather than change it.**
   `gateway_storage.py` (lines 42–54, 244–278) already publishes `AGENTSYS_GATEWAY_ATTACH_PATH`, `AGENTSYS_GATEWAY_ROOT`, and live gateway pointers. The delta-spec scenarios simply update the example tmux session name from `AGENTSYS-gpu` to `AGENTSYS-gpu-270b87`. If these gateway requirements are genuinely unchanged except for the example session name, they should be in the UNCHANGED section to reduce the review surface. If they are being freshly specified (previously unspecified), the ADDED header should be used and the motivation should note this is a first-time codification.

8. **Demo cleanup `_kill_tmux_session()` in `runtime.py` (lines 256–286) uses exact session names — it will miss suffixed sessions.**
   The current demo cleanup iterates `cleanup_session_names` which is populated with `requested_agent_identity` and `candidate_state.session_name` (`commands.py` lines 603–613). Under the new scheme, a stale session named `AGENTSYS-alice-270b87` would only be cleaned if it appeared in `candidate_state.session_name` from persisted state. If persisted state is absent or stale, the cleanup falls back to `requested_agent_identity` which is the bare canonical name `AGENTSYS-alice` — this won't match the suffixed tmux session. The startup-recovery delta-spec correctly identifies this case but doesn't prescribe the discovery mechanism (tmux enumeration + prefix matching? manifest scan?). Task 3.2 should specify the concrete approach.

### Nice To Have

9. **The design's "Non-Goals" section explicitly excludes user-configurable tmux-name prefix length, but the spec hardcodes `6` without a named constant.**
   Consider introducing a named constant (e.g., `DEFAULT_AGENT_ID_PREFIX_LENGTH = 6`) in the shared helper to make future configurability trivial.

10. **The tasks lack an explicit ordering dependency annotation.**
    Tasks 2.1 (resolution update) depends on 1.1 (shared helper) and 1.3 (manifest tightening). Tasks 3.1–3.2 (demo updates) depend on 2.2 (registry publication uses actual tmux handle). The tasks are sequenced in a reasonable order but don't annotate which are parallelizable vs. strictly sequential.

11. **The design mentions `codex_app_server` in `BackendKind` (models.py line 18) but doesn't discuss its naming interaction.**
    `codex_app_server` is a tmux-backed backend but isn't mentioned in the design or spec scenarios. If it shares the tmux naming path, a brief note confirming coverage would reduce implementer uncertainty.

## Design Alignment With Existing System

### Existing pattern(s) observed

- **Identity resolution already supports non-matching tmux names**: `_resolve_agent_identity_from_tmux_local()` at `runtime.py:1466–1532` already enumerates all tmux sessions and matches on `payload.agent_name` (line 1505), not session name. The suffixed naming change is naturally supported by this existing fallback path.
- **Manifest construction uses explicit identity fields**: `runtime.py:_resolve_start_session_identity()` (lines 1241–1294) already resolves `canonical_agent_name`, `agent_id`, and `tmux_session_name` as separate fields. The change only modifies the derivation of `tmux_session_name` at line 1292 from `f"houmao-{session_id}"` to the new `<agent_name>-<agent_id[:6]>` formula.
- **tmux env publication is already centralized**: Both `headless_base.py:_publish_tmux_session_environment()` (lines 295–329) and `cao_rest.py:_publish_tmux_session_environment()` (lines 737–745) call the shared `set_tmux_session_environment_shared()` from `tmux_runtime.py`. The new naming helper can plug into the existing `_resolve_start_session_identity()` without restructuring the publication flow.
- **Demo state model already separates `agent_identity` from `session_name` and `tmux_target`**: `DemoState` at `demo/cao_interactive_demo/models.py:52–75` has distinct fields. The demo inspect rendering at `rendering.py:22–63` already reads from `tmux_target`, not from `agent_identity`. The primary demo change is ensuring `tmux_target` receives the suffixed handle and that cleanup discovers suffixed sessions.

### Proposed design alignment: **Strongly aligns** with existing architecture

The design correctly leverages the existing conceptual split between canonical identity and tmux handle. Key alignment points:

1. **Single derivation point**: Replacing the `f"houmao-{session_id}"` assignment at `runtime.py:1292` with a call to the new shared helper is a clean, minimal change to the start flow.
2. **Manifest schema version unchanged**: No schema bump is needed because `tmux_session_name` is already a free-form string field in the manifest payload (`manifest.py:125`). The new value is just a different string.
3. **Resolution pipeline intact**: The enumeration-based resolution at `runtime.py:1491–1507` already handles the case where `session_name != canonical_agent_identity`. The direct-lookup fast path at `_resolve_agent_identity_from_addressed_tmux_session()` (lines 1535–1572) needs a graceful degradation to the enumeration path when the canonical name doesn't match any session — this is already the current behavior (the direct lookup raises, then the enumeration fallback runs).
4. **Potential misalignment**: The legacy `generate_tmux_session_name()` at `tmux_runtime.py:262–283` uses a different naming convention (`AGENTSYS-<tool>-<role>[-N]`). If not retired or reconciled, callers of this function will produce sessions that don't follow the new `<agent-name>-<agent-id[:6]>` contract. The design should be explicit about whether this function is deprecated.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the shared naming helper live in `agent_identity.py` or `tmux_runtime.py`?

- Decision: `Deferrable`

#### Why this matters

The helper combines agent-identity concepts (canonical name, agent_id) with tmux-specific concerns (session name uniqueness, occupied-session enumeration). Its placement determines import direction and module coupling.

#### Examples

- If in `agent_identity.py`: the identity module gains a tmux dependency (needs to list tmux sessions for collision checks), breaking its current transport-agnostic posture.
- If in `tmux_runtime.py`: aligns with existing `generate_tmux_session_name()` locality but means identity derivation logic lives in a transport module.

#### Why this question arises in code

- `agent_identity.py` currently owns `derive_auto_agent_name_base()` (line 125) and `derive_agent_id_from_name()` — both are pure functions with no tmux dependency.
- `tmux_runtime.py` owns `generate_tmux_session_name()` (line 262) which already calls `list_tmux_sessions()` for collision detection.
- The new helper needs both identity inputs (agent_name, agent_id) and tmux state (occupied sessions).

```python
# Proposed helper signature from design:
def derive_tmux_session_name(
    *,
    canonical_agent_name: str,
    agent_id: str,
    prefix_length: int = 6,
    occupied_sessions: set[str] | None = None,
) -> str: ...
```

#### Options

**Option A: Place in `tmux_runtime.py` alongside `generate_tmux_session_name()`**

- Code impact:
  - Natural extension of existing tmux naming code. `runtime.py:_resolve_start_session_identity()` already imports from `tmux_runtime`.
- User-facing difference:
  - None.
- Pros:
  - Keeps tmux-specific logic (collision detection, session enumeration) co-located.
  - Matches existing pattern where `generate_tmux_session_name()` lives.
- Cons:
  - Identity derivation logic (agent_name + agent_id → session name) is conceptually an identity concern.

**Option B: Place in `agent_identity.py` with an injected occupied-session set**

- Code impact:
  - The helper accepts `occupied_sessions: set[str] | None` as a parameter, avoiding a direct tmux import. Callers pass in the occupied set.
- User-facing difference:
  - None.
- Pros:
  - Keeps `agent_identity.py` as the single source for all identity derivation.
  - The helper itself remains a pure function (no I/O).
- Cons:
  - Callers must resolve the occupied-session set before calling, adding a coordination step.

> **PROPOSED: Place the helper in `agent_identity.py` as a pure function accepting an optional `occupied_sessions` set (Option B).**
> Rationale: `agent_identity.py` already owns `derive_auto_agent_name_base()` (line 125) and `derive_agent_id_from_name()` — both pure identity derivation functions. The new helper's core logic (string formatting + prefix extension) is also pure; only the collision check needs external state, which can be injected. This keeps `agent_identity.py` as the single source of truth for all identity-to-name derivation while preserving its transport-agnostic character. The occupied-session injection is a one-line addition at the call site in `runtime.py:_resolve_start_session_identity()` which already has access to tmux state.
>
> **DECISION: Accept PROPOSED with one refinement: the authoritative derivation helper should live in `agent_identity.py`, while `tmux_runtime.py` may keep only thin tmux-facing wrappers around it.**
> Rationale: `agent_identity.py` already owns canonical-name normalization and default `agent_id` derivation, and keeping the new algorithm there preserves the existing split between pure identity logic and tmux I/O. `tmux_runtime.py` should not become the source of truth for the naming contract, but a small delegating wrapper there is acceptable if it reduces backend call-site churn during implementation.

### Q2) What happens to `generate_tmux_session_name()` in `tmux_runtime.py`?

- Decision: `Blocking`

#### Why this matters

There are currently two tmux session naming paths: the runtime start flow (which produces `houmao-<session_id>`) and the backend allocation helper `generate_tmux_session_name()` (which produces `AGENTSYS-<tool>-<role>[-N]`). The design introduces a third formula (`<agent-name>-<agent-id[:6]>`). If `generate_tmux_session_name()` remains, there will be three naming conventions active in the codebase, creating confusion about which path applies where.

#### Examples

- A headless backend started via the main runtime flow gets `AGENTSYS-gpu-270b87`. But if a helper or test calls `generate_tmux_session_name(tool="claude", role_name="gpu")`, it gets `AGENTSYS-claude-gpu`. These two sessions for the same logical agent have incompatible naming.

#### Why this question arises in code

- `generate_tmux_session_name()` is defined at `tmux_runtime.py:262–283`.
- It is distinct from the runtime-owned start path at `runtime.py:1292`.
- The design's Task 1.2 says "Switch runtime-owned start flows and tmux-backed backend allocators to use the shared helper" but doesn't say whether `generate_tmux_session_name()` is removed, wrapped, or kept for non-runtime callers.

```python
# Current generate_tmux_session_name (tmux_runtime.py:262):
def generate_tmux_session_name(*, tool: str, role_name: str, ...) -> str:
    base = derive_auto_agent_name_base(tool=tool, role_name=role_name)
    primary = f"{AGENT_NAMESPACE_PREFIX}{base}"
    ...
```

#### Options

**Option A: Deprecate and remove `generate_tmux_session_name()`, migrate all callers to the new shared helper**

- Code impact:
  - All call sites must provide `canonical_agent_name` and `agent_id` instead of just `tool` and `role_name`. This may require threading identity parameters further into backend constructors.
- User-facing difference:
  - All tmux sessions, regardless of creation path, follow the `<agent-name>-<agent-id[:6]>` convention.
- Pros:
  - Single naming convention. No ambiguity.
- Cons:
  - Larger blast radius — backend constructors that currently only know tool+role must learn agent_name+agent_id.

**Option B: Keep `generate_tmux_session_name()` for legacy/test/helper paths, new shared helper for runtime-owned sessions only**

- Code impact:
  - Minimal changes to backend constructors. Runtime start path uses the new helper; legacy paths keep the old generator.
- User-facing difference:
  - Two naming conventions coexist. Sessions created by different paths look different in `tmux ls`.
- Pros:
  - Smaller change scope. Lower risk.
- Cons:
  - Operator confusion when `tmux ls` shows both `AGENTSYS-claude-gpu` and `AGENTSYS-gpu-270b87` style names.

> **PROPOSED: Deprecate `generate_tmux_session_name()` and migrate all callers to the new shared helper (Option A).**
> Rationale: The runtime-owned start path at `runtime.py:_resolve_start_session_identity()` (line 1241) already resolves `canonical_agent_name` and `agent_id` before generating the tmux session name. Backend allocation code that currently calls `generate_tmux_session_name()` runs downstream of identity resolution, so it can receive the resolved identity as parameters. Keeping two naming conventions undermines the change's goal of making tmux sessions "operator-readable" with a consistent scheme. The existing callers (headless backends, CAO allocation) are limited in number and already receive brain manifests that carry identity metadata. Marking `generate_tmux_session_name()` as deprecated with a `warnings.warn()` for one release cycle before removal would be a safe transition.
>
> **DECISION: Refine PROPOSED: retire the old `AGENTSYS-<tool>-<role>` formula in this change, but do not add a compatibility grace period; `generate_tmux_session_name()` may remain only as a delegating wrapper over the new helper or be removed outright.**
> Rationale: The repository is explicitly in unstable development, so a one-release deprecation cycle is unnecessary overhead. The important decision is that there must no longer be two naming algorithms. Keeping the function name as a thin adapter is acceptable if it lets existing backend/test call sites converge quickly, but the legacy formula itself should disappear as part of this change.

### Q3) How should demo startup cleanup discover suffixed tmux sessions when persisted state is absent?

- Decision: `Blocking`

#### Why this matters

The delta-spec for `cao-interactive-demo-startup-recovery` requires that "the reset flow SHALL use persisted or discovered tmux session metadata to clean that leftover tmux session instead of assuming the tmux handle is exactly `AGENTSYS-alice`." When persisted demo state is absent (fresh machine, manually deleted state file), the cleanup has no record of the actual suffixed tmux handle. It needs a discovery mechanism.

#### Examples

- A stale `AGENTSYS-alice-270b87` tmux session exists. No demo `state.json` exists. The user runs `start`. The cleanup must find and kill `AGENTSYS-alice-270b87` without knowing `270b87`.
- Two stale sessions exist: `AGENTSYS-alice-270b87` and `AGENTSYS-alice-f3c2a1` (from two previous runs). Both must be cleaned.

#### Why this question arises in code

- Current cleanup in `commands.py` (lines 603–633) iterates explicit session names from `requested_agent_identity` and `candidate_state.session_name`.
- `_kill_tmux_session()` in `runtime.py:256–286` calls `tmux has-session -t <exact-name>` — it cannot match by prefix.
- The runtime's `_resolve_agent_identity_from_tmux_local()` already enumerates sessions and matches on `payload.agent_name`, but the demo cleanup code doesn't use this path.

```python
# Current demo cleanup (commands.py:627-633):
for session_name in _dedup_preserve_order(cleanup_session_names):
    _kill_tmux_session(paths=paths, env=env, session_name=session_name, ...)
# cleanup_session_names = [requested_agent_identity, previous_state.session_name]
# If previous_state is None, only requested_agent_identity is used → misses suffixed sessions
```

#### Options

**Option A: Enumerate tmux sessions with prefix matching**

- Code impact:
  - Add a helper that calls `tmux ls` and filters sessions whose names start with the canonical agent identity prefix (e.g., `AGENTSYS-alice-`). Kill all matches.
  - ```python
    def _discover_tmux_sessions_for_identity(canonical_name: str) -> list[str]:
        all_sessions = list_tmux_sessions_shared()
        return [s for s in all_sessions if s == canonical_name or s.startswith(f"{canonical_name}-")]
    ```
- User-facing difference:
  - Cleanup is more aggressive — catches all suffixed stale sessions from any previous run.
- Pros:
  - Simple, no external dependency beyond `tmux ls`.
  - Handles the "no persisted state" case.
- Cons:
  - Prefix matching could over-match if another agent's canonical name is a prefix of a different agent's name (e.g., `AGENTSYS-alice` and `AGENTSYS-alice-extended`). However, the `AGENTSYS-<name>-<hex>` pattern makes false matches extremely unlikely in practice.

**Option B: Use the runtime's resolve-by-name machinery**

- Code impact:
  - Call into `_resolve_agent_identity_from_tmux_local()` or a similar enumeration+manifest-matching function to find sessions whose persisted `agent_name` matches the canonical identity.
- User-facing difference:
  - Cleanup is manifest-validated — only kills sessions whose manifests confirm they belong to the canonical identity.
- Pros:
  - More precise, avoids any false-positive risk.
- Cons:
  - Heavier — requires reading manifests from tmux env vars for each session.
  - May fail if the tmux session's `AGENTSYS_MANIFEST_PATH` is stale or missing (common for truly orphaned sessions).

> **PROPOSED: Use tmux enumeration with prefix matching (Option A), with an additional manifest validation when available.**
> Rationale: The demo cleanup path explicitly operates in a "best-effort, tolerate staleness" mode — `_stop_remote_session()` is called with `tolerate_stale=True` (commands.py line 623) and stale markers are silently accepted (models.py lines 30–39). Prefix matching on `AGENTSYS-alice-` is consistent with this tolerant posture. The `<canonical-name>-<hex>` format makes false positives negligible. A pure manifest-validation approach (Option B) would fail on truly orphaned sessions where the manifest path in the tmux env is stale — exactly the scenario the cleanup is designed to handle. For defense-in-depth, the cleanup could optionally validate the manifest's `agent_name` when the env pointer is readable, but should not require it.
>
> **DECISION: Reject PROPOSED as written; use metadata-backed tmux discovery as the authoritative path, with exact persisted-session and exact legacy-name cleanup as fallback, but do not kill arbitrary prefix matches without metadata.**
> Rationale: The current identity rules allow hyphenated canonical names, so a bare `startswith(f\"{canonical_name}-\")` check can over-match a different canonical identity such as `AGENTSYS-alice-extended`. The startup-recovery delta-spec already says cleanup should use persisted or discovered tmux session metadata, and the runtime already has tmux-local discovery machinery that matches on persisted `agent_name`. For truly orphaned sessions with no readable metadata, v1 cleanup should stay conservative rather than guessing from prefixes.

### Q5) Should the `-` separator between canonical name and agent-id prefix be specified as a distinct delimiter or just part of the format string?

- Decision: `Deferrable`

#### Why this matters

The canonical agent name itself can end with a hex-looking string (e.g., `AGENTSYS-claude-abc123`). When the agent-id prefix is appended (`AGENTSYS-claude-abc123-270b87`), it becomes ambiguous where the canonical name ends and the id prefix begins. While this ambiguity doesn't affect runtime behavior (the runtime never needs to parse the tmux name back into components), it could matter for future tooling, log analysis, or operator inspection.

#### Examples

- `AGENTSYS-test-feed-deadbe`: Is `deadbe` part of the canonical name or the agent-id prefix?
- For the runtime, this doesn't matter (it resolves via manifest, not by parsing the tmux name). But for operators reading `tmux ls`, it could be confusing.

#### Why this question arises in code

- `derive_auto_agent_name_base()` at `agent_identity.py:125–151` can produce bases containing hyphens and alphanumeric segments. The concatenation `f"{agent_name}-{agent_id[:6]}"` uses the same `-` delimiter as the name's internal separators.

#### Options

**Option A: Keep single `-` separator (current design)**

- Code impact:
  - `f"{agent_name}-{agent_id[:6]}"` — simple.
- User-facing difference:
  - Visually clean, but potentially ambiguous in edge cases.
- Pros:
  - Simple, clean.
  - The runtime never needs to parse the name back.
- Cons:
  - Minor visual ambiguity for operators.

**Option B: Use a distinct separator like `~` or `:`**

- Code impact:
  - `f"{agent_name}~{agent_id[:6]}"` — requires verifying tmux accepts the separator character.
- User-facing difference:
  - Visually unambiguous.
- Pros:
  - Clear boundary between canonical name and id prefix.
- Cons:
  - tmux session names have limited allowed characters; `~` is safe but unusual. `:` is already used by tmux for window targets.

> **PROPOSED: Keep single `-` separator (Option A).**
> Rationale: The runtime resolves sessions by manifest metadata (`agent_name` field), not by parsing the tmux session name string. The existing `generate_tmux_session_name()` at `tmux_runtime.py:276` already uses `-` for collision suffixes (e.g., `AGENTSYS-claude-gpu-2`). Using the same separator maintains visual consistency with existing tmux sessions. The ambiguity is purely cosmetic and doesn't affect any resolution or control path. Using an exotic separator would add a tmux-compatibility risk for no functional benefit.
>
> **DECISION: Accept PROPOSED, and make the non-goal explicit that tmux session names are transport handles, not a string contract that other code should reverse-parse.**
> Rationale: The current runtime already treats `agent_name`, `agent_id`, and `tmux_session_name` as separate persisted fields, and the new design continues that split. Keeping `-` preserves consistency with existing tmux naming, avoids introducing a new delimiter rule, and stays safe so long as the artifacts make it explicit that resolution must use persisted metadata rather than parsing the tmux handle.
