# Discussion: Agent Identity Open Questions

- Feature/Change: `agent-identity-tmux-manifest-path`
- Timestamp: `2026-03-02T04:48:55Z`
- Context: This doc lists open decisions before implementing `--agent-identity` name-based addressing for tmux-backed agents (including `AGENTSYS-` session naming and `AGENTSYS_MANIFEST_PATH` tmux env pointers). Inputs reviewed: change proposal/design/specs/tasks.

---

## Summary of recommended defaults

1) Q1: Explicit user-provided agent names fail fast on conflict; only auto-generated names auto-suffix.
2) Q2: Match `AGENTSYS-` prefix case-sensitively and exactly (no case conversion); treat any inexact look-alike as missing the prefix and emit a warning when the input contains substring `AGENTSYS` (case-insensitive).
3) Q3: Use `start-session --agent-identity <name>` (name-only for CAO) to keep the surface consistent; reject it for non-tmux backends for now.
4) Q4: Deterministic path-like parsing for `--agent-identity`: if it looks like a path, require the file exists (do not reinterpret as name).
5) Q5: Validate user-provided names strictly: restrict to tmux+filesystem-safe chars and reject names whose name portion contains standalone token `AGENTSYS` (token boundaries `[^0-9A-Za-z]` and string boundaries); do not sanitize or case-convert user input.
6) Q7: On CAO stop, best-effort clean up the tmux session to avoid stale `AGENTSYS-*` entries.

## Q1) Should explicit user-provided agent names fail on conflict or auto-suffix?

Blocking: Yes (affects UX contract and automation expectations).

### Why this matters

- If behavior is non-deterministic, operators can lose track of which agent a name refers to.
- It affects tmux session creation, error messages, and the “source of truth” for identity.

### Options

**Option A: Fail fast for explicit names (no auto-suffix)**

Pros:
- The user gets exactly the name they asked for, or a clear error.
- Scripts can rely on the chosen name being stable.

Cons:
- Users must choose another name manually when a session is already running or stale.

**Option B: Auto-suffix explicit names on conflict (`gpu` -> `gpu-2`)**

Pros:
- Fewer failed launches; lower friction in interactive use.

Cons:
- Surprising: the actual running agent identity differs from what the user typed.
- Scripts that assume the requested name exists can break unless they parse output.

### Proposal (Recommended)

Use Option A: explicit names fail fast on conflict. Keep auto-suffix behavior only for system-generated names (when the caller does not provide an agent name). Always print/return the canonical selected identity from `start-session`.

### Pros / Cons (Proposal)

Pros:
- Stable handle semantics: the caller controls the identity.
- Fewer “which one is gpu-2?” debugging loops.

Cons:
- Requires user action to resolve name conflicts (stop stale session or pick a new name).

> **DECISION: Accept Option A: explicit user-provided names fail fast on conflict; auto-suffix only for system-generated names.**
> Rationale: The current `generate_cao_session_name()` already auto-generates unique names for system-managed sessions (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:541`). In contrast, `_create_tmux_session()` fails immediately on duplicate tmux names and raises `BackendExecutionError`, which gives us a natural fail-fast path for explicit names (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:599`). This aligns with stable identity semantics needed for name-based resolution through `AGENTSYS_MANIFEST_PATH` in this change's spec.

## Q2) Should `AGENTSYS` reservation and `AGENTSYS-` prefix detection be case-sensitive?

Blocking: Mostly (small ambiguity, but easier to decide before coding).

### Why this matters

- tmux session names are case-sensitive; inconsistent casing can create hard-to-find sessions.
- It affects reserved-name bypass risk (`agentsys` vs `AGENTSYS`) and user confusion.

### Options

**Option A: Case-sensitive only (exact `AGENTSYS` reserved; exact `AGENTSYS-` prefix recognized)**

Pros:
- Simple to implement and reason about.

Cons:
- Users can accidentally create confusing near-duplicates (for example `AGENTSYS-gpu` vs `agentsys-gpu`).
- `agentsys` would not be reserved, defeating the “namespace root is reserved” intent.

**Option B: Case-insensitive handling for input; canonicalize output**

Pros:
- Users can type `agentsys-gpu` and still target `AGENTSYS-gpu`.
- Stronger reservation guarantee for the namespace root.

Cons:
- Requires explicit canonicalization rules (what is the canonical case?).

### Proposal (Recommended)

Use Option A: treat `AGENTSYS` / `AGENTSYS-` matching as case-sensitive and exact, and do not case-convert user-provided identities. If the input does not start with exact `AGENTSYS-` but contains substring `AGENTSYS` (case-insensitive), emit a warning that the prefix was not exact and will be treated as missing.

### Pros / Cons (Proposal)

Pros:
- Simple, predictable semantics (exact prefix match; no rewriting of user input).
- Preserves user intent for mixed-case names while still keeping a canonical `AGENTSYS-...` tmux namespace.

Cons:
- Users can still create confusing near-duplicates (for example `AGENTSYS-gpu` vs `agentsys-gpu`); warnings help but do not prevent this.
- The reservation is exact: `agentsys` is not reserved.

> **DECISION: Accept Option A: case-sensitive exact prefix matching with no case conversion.**
> Rationale: The change spec defines `AGENTSYS-` as a literal namespace prefix, and the user's decision is to keep it exact so callers can intentionally create mixed-case names when desired. To reduce user confusion when they type something that looks like the namespace (for example `agentsys-gpu`), we add a warning trigger based on case-insensitive substring match of `AGENTSYS`, without rewriting the identity.

## Q3) What should the `start-session` naming flag be?

Blocking: Yes (public CLI contract).

### Why this matters

- It determines whether the user can set the identity at creation time consistently.
- It affects doc/tutorial updates and demo scripts.

### Options

**Option A: Use `start-session --agent-identity <name>` (name-only in this command)**

Pros:
- One concept and one flag name across start/prompt/stop.

Cons:
- The same flag name has different allowed value shapes across commands (name-only vs name-or-path).

**Option B: Use `start-session --agent-name <name>` and keep `--agent-identity` for control**

Pros:
- Value shape matches the flag name; clearer separation.

Cons:
- More surface area; users must remember two related flags.

### Proposal (Recommended)

Use Option A for consistency, with a strict guard: in `start-session`, `--agent-identity` accepts only a name (reject path-like values). Also reject providing `--agent-identity` for non-tmux backends in this change (until a non-tmux resolution mechanism exists).

### Pros / Cons (Proposal)

Pros:
- Consistent story: “agent identity is the handle”.
- Keeps the CLI compact.

Cons:
- Needs careful help text to avoid confusion about value shapes.

> **DECISION: Accept Option A with guardrails: use `start-session --agent-identity <name>` (name-only) and reject it for non-tmux backends in this change.**
> Rationale: `start-session`, `send-prompt`, and `stop-session` are already a unified CLI surface in one parser module, so keeping one flag name reduces user-facing branching (`src/gig_agents/agents/brain_launch_runtime/cli.py:75`, `:94`, `:105`). The same parser also exposes multiple backend choices, including non-tmux backends, so we need an explicit backend guard when identity-by-name is introduced (`src/gig_agents/agents/brain_launch_runtime/cli.py:86`). This keeps the contract consistent while preserving current backend scope limits.

## Q4) What is the exact parsing rule for `--agent-identity` path vs name?

Blocking: Yes (error semantics and user trust).

### Why this matters

- A typo in a path must not silently turn into a different agent name.
- It impacts debugging: users need predictable resolution behavior.

### Options

**Option A: Heuristic path-like detection (recommended in spec)**

Definition:
- Treat as path if it contains `/` or `\\` or ends with `.json`.
- If path-like and missing, error; do not reinterpret as name.

Pros:
- Predictable and prevents “missing file becomes a name”.
- Names cannot accidentally include path separators.

Cons:
- Slightly “magic” heuristics; must be documented.

**Option B: Existence-based detection**

Definition:
- If it resolves to an existing file, treat as path; otherwise treat as name.

Pros:
- No heuristics beyond “file exists”.

Cons:
- Dangerous: a missing file path becomes a name (exactly what we want to avoid).

### Proposal (Recommended)

Use Option A (heuristic path-like detection with fail-fast if missing). Document it in CLI help and docs.

### Pros / Cons (Proposal)

Pros:
- Eliminates the worst UX failure mode (typo becomes new agent target).

Cons:
- Slight risk that users want a name containing `.json`; treat that as invalid by design.

> **DECISION: Accept Option A: deterministic path-like parsing (`/`, `\\`, or `.json`) with fail-fast when missing.**
> Rationale: `_resolve_path()` can resolve any string syntactically, so we need deterministic classification before resolution to avoid mis-targeting (`src/gig_agents/agents/brain_launch_runtime/cli.py:225`). When treated as a path, `load_session_manifest()` already provides an explicit missing-file error (`src/gig_agents/agents/brain_launch_runtime/manifest.py:160`). This matches the added requirement in this change's `agent-identity` spec and prevents typoed paths from silently becoming agent names.

## Q5) Should the system sanitize user-provided names or reject invalid characters?

Blocking: Yes (identity stability and tmux compatibility).

### Why this matters

- The agent name becomes a primary handle; silent rewriting makes it hard to target.
- tmux session names have practical constraints (no whitespace, avoid `:` target syntax issues).

### Options

**Option A: Reject invalid names with an explicit validation error**

Pros:
- The user always knows what identity they are creating.
- Avoids hidden canonicalization surprises.

Cons:
- Slightly more friction for casual usage (“why can’t I use spaces?”).

**Option B: Sanitize names (spaces -> `-`, strip chars) and accept**

Pros:
- Very forgiving input.

Cons:
- The effective identity differs from what the user typed.
- Sanitization rules become a hidden contract unless clearly printed/returned.

### Proposal (Recommended)

Use Option A for user-provided names: validate strictly using a tmux+filesystem-safe character set (ASCII letters/digits plus `_` and `-`), and reject names whose name portion contains standalone token `AGENTSYS` (token boundaries `[^0-9A-Za-z]` and string boundaries). Do not sanitize or case-convert user input. For auto-generated names, sanitize/truncate as needed since the system controls the output and returns the selected canonical identity.

### Pros / Cons (Proposal)

Pros:
- Keeps identities stable and human-targetable.

Cons:
- Requires clear docs and good error messages.

> **DECISION: Accept Option A: reject invalid user-provided names; reserve sanitization for auto-generated names.**
> Rationale: Agent identity is a primary handle; silent rewriting makes it hard to target reliably. The runtime already follows strict validation patterns at persistence/boundary layers, so strict identity validation aligns with the repo’s general approach. The additional standalone-token rule (`[^0-9A-Za-z]AGENTSYS[^0-9A-Za-z]` boundaries in the name portion) prevents confusing nested namespacing like `foo_AGENTSYS_bar` while allowing harmless substrings like `MYAGENTSYS`.

## Q6) What is the auto-generated name algorithm?

Deferrable: Partially (can be iterated), but pick a default before implementing.

### Why this matters

- Auto-names need to be short, discoverable, and not collide frequently.
- The algorithm affects operator ergonomics and demo scripts.

### Options

**Option A: Tool abbreviation + role/blueprint acronym (very short)**

Example: `AGENTSYS-cl-gkc`

Pros:
- Extremely short and easy to type.

Cons:
- Not always self-explanatory without context.

**Option B: Tool abbreviation + truncated slug (more readable)**

Example: `AGENTSYS-cl-gpu-kernel`

Pros:
- More human-readable in `tmux ls`.

Cons:
- Can get long; truncation rules matter.

### Proposal (Recommended)

Use Option B without introducing an additional length limit beyond tmux constraints: `AGENTSYS-<tool2>-<slug>` where `<slug>` is derived from blueprint name if available else role name, normalized to match the allowed character set. Add `-2`, `-3`, ... suffixes until `tmux has-session` reports uniqueness. Prefer short inputs (for example a short `<tool2>` and a reasonably short slug source) so the common case remains easy to type without relying on a hard cap.

### Pros / Cons (Proposal)

Pros:
- Better out-of-the-box clarity for humans inspecting tmux.

Cons:
- Slightly longer names; needs careful truncation logic.

> **DECISION: Accept Option B: auto-generate `AGENTSYS-<tool2>-<slug>` with numeric conflict suffixes (no extra length cap beyond tmux constraints).**
> Rationale: Existing generated CAO names already derive from role context and a uniqueness component, so readable slug-based generation is consistent with current behavior (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:541`). The launch plan already carries `tool` and role context used during startup, so deriving `<tool2>` and `<slug>` is available at session creation time (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:491`). For conflicts, we should deterministically probe and append `-2`, `-3`, etc., instead of relying on opaque randomness, while still using tmux conflict checks as the final guard.

## Q7) Should `stop-session` best-effort kill the tmux session?

Deferrable: No (cleanup policy affects operator experience immediately).

### Why this matters

- Stale `AGENTSYS-*` tmux sessions will mislead users and break name resolution.
- CAO deletion may not fully clean tmux resources in all failure modes.

### Options

**Option A: Rely on CAO session deletion only**

Pros:
- Simple; avoids direct tmux manipulation during stop.

Cons:
- If CAO fails or leaks tmux resources, stale sessions remain visible.

**Option B: Best-effort tmux cleanup on stop**

Pros:
- Stronger invariant: “if the agent is stopped, the `AGENTSYS-*` tmux session disappears”.

Cons:
- Needs careful error handling to avoid masking CAO errors.

### Proposal (Recommended)

Use Option B: after attempting CAO terminal/session deletion, best-effort `tmux kill-session -t <agent-session>` for CAO sessions. If tmux cleanup fails, surface it as an additional warning/error detail without hiding the primary stop outcome.

### Pros / Cons (Proposal)

Pros:
- Much cleaner operator experience.

Cons:
- Adds one more failure surface in stop logic (needs good tests/mocking).

> **DECISION: Accept Option B: best-effort `tmux kill-session` cleanup during stop.**
> Rationale: The helper `_cleanup_tmux_session()` already exists and is tolerant by design (`check=False`, ignore `OSError`), so the codebase already has a safe primitive for best-effort tmux cleanup (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:641`). Today, `terminate()` only deletes CAO terminal/session resources via API and does not call tmux cleanup, which can leave stale sessions (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:455`). Given Q1's fail-fast naming decision, stale tmux sessions would directly harm restart UX, so stop-path cleanup should be added and surfaced as supplemental warning/error detail.

## Q8) Should non-tmux backends support name-based `--agent-identity` in this change?

Deferrable: Yes (explicitly out-of-scope in current design, but worth calling out).

### Why this matters

- Users may expect `--agent-identity foo` to work uniformly across backends.
- Adding non-tmux resolution later could require new persistence/registry behavior.

### Options

**Option A: CAO/tmux-only for now (name mode); others require manifest path**

Pros:
- Matches the chosen tmux-env mapping approach.
- Keeps scope small and avoids global registries.

Cons:
- Inconsistent UX across backends.

**Option B: Add a non-tmux identity registry now**

Pros:
- Uniform UX across backends.

Cons:
- Conflicts with the stated constraint (not all agents share one parent root).
- Requires new persistence strategy (likely global state) and migration decisions.

### Proposal (Recommended)

Use Option A for this change. If we later want name-based identities for non-tmux backends, introduce a separate capability with an explicit storage location and lifecycle semantics.

### Pros / Cons (Proposal)

Pros:
- Avoids premature global-state decisions.

Cons:
- Users of headless backends still type manifest paths.

> **DECISION: Accept Option A: keep name-based `--agent-identity` CAO/tmux-only in this change; require manifest paths for non-tmux backends.**
> Rationale: Runtime backend creation shows non-tmux backends do not have tmux session context or a tmux env pointer mechanism; `codex_app_server` resume is explicitly unsupported, and headless backends resume by `session_id` state (`src/gig_agents/agents/brain_launch_runtime/runtime.py:201`, `:254`; `src/gig_agents/agents/brain_launch_runtime/backends/headless_base.py:16`). Introducing cross-backend name resolution now would require a new global registry and lifecycle semantics that do not exist in current runtime structures. Deferring non-tmux name-mode keeps scope aligned with the tmux-based architecture in this change.

## Overall Design Review (Claude Opus)

### Assessment
- The design is well-scoped: it layers name-based addressing on top of the existing manifest-as-source-of-truth architecture without modifying the manifest schema, which minimizes risk to persisted state and resume correctness.
- The tmux env pointer (`AGENTSYS_MANIFEST_PATH`) is a sound mechanism given the constraint that agents may not share a runtime root — it avoids introducing a global registry while still providing O(1) name resolution.
- The main structural risk is a **session-start ordering dependency**: the manifest path must be allocated *before* `CaoRestSession.__init__` creates the tmux session (`cao_rest.py:192-194`), but today `start_runtime_session` allocates `session_id`/`manifest_path` *after* the backend session is created (`runtime.py:132-133`). This reordering is acknowledged in `design.md` but is the single hardest implementation step and deserves explicit design attention.
- The error surface grows significantly (identity validation, tmux resolution, stale pointer, missing manifest) but `errors.py` currently has no identity-specific error class — all new failure modes would collapse into generic `BackendExecutionError` or `SessionManifestError` unless the hierarchy is extended first.
- The CLI breaking change (`--session-manifest` → `--agent-identity`) affects 13 callsites across 5 demo scripts plus the `interrupt_driver.py` helper (`scripts/demo/cao-claude-esc-interrupt/scripts/interrupt_driver.py:40`), and the public `__init__.py` re-export surface (`src/gig_agents/agents/__init__.py:30`).
- The decisions (Q1–Q8) are internally consistent and form a coherent contract: fail-fast naming + exact-prefix canonicalization (no case conversion) + strict validation + best-effort cleanup + CAO-only scope.

### Suggestions
- **[Errors]** Add `AgentIdentityError(BrainLaunchRuntimeError)` in `errors.py` *before* implementing any identity logic. Current error classes (`SessionManifestError`, `BackendExecutionError`) are semantically wrong for "reserved keyword", "invalid name characters", or "tmux session not found" failures. Subclass it into `AgentNameValidationError` and `AgentResolutionError` so callers can branch on whether the input was invalid vs. the resolution target was missing.
- **[Backend]** The manifest-path-before-tmux ordering problem in `runtime.py:124-133` requires `start_runtime_session` to compute `session_id` and `manifest_path` *before* calling `_create_backend_session`, then pass `manifest_path` into `CaoRestSession.__init__` so `_start_terminal` can call `_set_tmux_session_environment` with `AGENTSYS_MANIFEST_PATH`. Consider adding a `manifest_path: Path | None` parameter to `CaoRestSession.__init__` and having `_start_terminal` include it in the env dict alongside `launch_env`.
- **[Backend]** `CaoRestSession.__init__` currently calls `generate_cao_session_name(role_name)` unconditionally at line 192 when `existing_state is None`. This must become conditional: use the user-provided canonical name if present, otherwise invoke the new auto-generation algorithm (Q6). Add an `agent_name: str | None` parameter to `CaoRestSession.__init__` to carry the resolved canonical identity from the CLI layer.
- **[CLI]** The `start-session` output JSON at `cli.py:188-196` currently returns `session_manifest` as the primary key. When `--agent-identity` is used, also return `agent_identity` (the canonical `AGENTSYS-<name>`) in the output so scripts can capture it. Keep `session_manifest` in the output for backward-compatible tooling even though the flag is renamed.
- **[Validation]** Create a dedicated `agent_identity.py` module under `brain_launch_runtime/` for all identity parsing, normalization, and validation. This includes: `classify_identity(value) -> Literal["path", "name"]`, `validate_agent_name(name) -> str` (preserves original case), and `canonicalize_tmux_session_name(identity) -> str` (adds exact `AGENTSYS-` prefix when missing, with no case conversion). Putting this in a separate module keeps `cli.py` and `runtime.py` from accumulating parsing logic and makes unit testing straightforward.
- **[Validation]** Define the name validation regex explicitly as a module constant (e.g., `AGENT_NAME_RE = re.compile(r"^[0-9A-Za-z][0-9A-Za-z_-]*$")`) in the new `agent_identity.py` (no extra length cap beyond tmux constraints). Add a second check that rejects a standalone `AGENTSYS` token in the name portion using boundaries defined by `[^0-9A-Za-z]` and string boundaries.
- **[Backend]** In `terminate()` at `cao_rest.py:455-484`, insert a `_cleanup_tmux_session(self._session_name)` call after the `delete_session` block and before the final return. Append any tmux cleanup failure as a supplemental warning in the `errors` list rather than masking the CAO API outcome, matching the existing error-aggregation pattern.
- **[Migration]** The `interrupt_driver.py` helper at `scripts/demo/cao-claude-esc-interrupt/scripts/interrupt_driver.py:40` also uses `--session-manifest` and is a standalone argparse script outside the main CLI — it needs its own migration or should be refactored to import the identity resolution module.
- **[Tests]** Structure tests in `tests/unit/agents/brain_launch_runtime/test_agent_identity.py` covering: path-vs-name classification (including edge cases like `./foo`, `foo.json`, `foo/bar`), reserved keyword rejection (case variants), canonicalization, and name validation regex. Mock tmux subprocess calls using `monkeypatch` on `subprocess.run` following the pattern in `tests/unit/agents/brain_launch_runtime/test_cao_client_and_profile.py`.
- **[Docs]** Add a `context/hints/agent-identity-resolution.md` explaining the resolution flow (classify → validate → canonicalize → tmux lookup → manifest load) with a worked example, so future contributors can trace the data path without reading multiple source files.

### Suggested Ordering
1. **Add error classes first** — create `AgentIdentityError`, `AgentNameValidationError`, and `AgentResolutionError` in `errors.py` and re-export from `__init__.py`. Zero functional change, unblocks everything else.
2. **Create `agent_identity.py` module** with `classify_identity`, `validate_agent_name`, `canonicalize_tmux_session_name`, and the name regex constant. Add unit tests in `test_agent_identity.py`. This is pure logic with no integration dependencies.
3. **Reorder `start_runtime_session`** in `runtime.py` to allocate `session_id`/`manifest_path` before `_create_backend_session`, and thread `manifest_path` + `agent_name` into `CaoRestSession.__init__` / `_start_terminal`. Set `AGENTSYS_MANIFEST_PATH` in tmux env. This is the riskiest step — validate with an integration test.
4. **Add tmux cleanup to `terminate()`** in `cao_rest.py` and update auto-generated name logic in `generate_cao_session_name` (or its replacement). Add conflict-detection loop for auto-names.
5. **Replace CLI flags** — swap `--session-manifest` → `--agent-identity` in `cli.py`, add identity resolution dispatch (path vs name) in `_cmd_send_prompt` and `_cmd_stop_session`, add optional `--agent-identity` to `start-session` with CAO-only guard.
6. **Migrate demo scripts and `interrupt_driver.py`** — mechanical `--session-manifest` → `--agent-identity` replacements across all 5 demo `run_demo.sh` files and the interrupt driver.
7. **Add integration-level tests** for the full round-trip: start with `--agent-identity gpu` → verify `AGENTSYS-gpu` tmux session exists with `AGENTSYS_MANIFEST_PATH` set → resume by name → stop and verify tmux cleanup.
