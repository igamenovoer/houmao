## Summary

- The proposal is clear and solves a real gap: existing demos are mostly one-shot and not ideal for iterative tmux-inspected workflows.
- Design direction aligns with existing CAO demo mechanics (`build-brain` / `start-session` / `send-prompt` / `stop-session`) and launcher ownership safeguards.
- The current artifacts are missing key behavioral invariants (state lifecycle semantics, CAO ownership boundaries, and local-only constraints) that are required to avoid regressions and session leaks.
- Tasks are appropriately phased, but several acceptance details are under-specified and could lead to inconsistent implementations across shells/scripts.

## Findings

### Must Fix / Blocking

- Missing active-state lifecycle contract for repeated `start` and concurrent invocations.
  - Why blocking: The design introduces persisted state and multi-command lifecycle, but does not define behavior when a state file already marks a session active (reuse vs fail vs replace). Without this invariant, implementers can accidentally spawn orphan sessions or stomp active state.
  - Where: `design.md` (Decisions 2/3), `spec.md` (startup + turn-driving requirements), `tasks.md` (1.3, 2.1, 3.1).
  - Required addition: Add explicit requirement/scenario for `start` while state is active and for lock/serialization behavior across simultaneous command invocations.

- CAO launcher ownership and stop semantics are referenced but not specified as enforceable requirements.
  - Why blocking: Existing demos intentionally avoid unsafe teardown of externally owned servers (see launcher state checks and retry/fail/skip logic). The new change currently says “reuse guardrails” in tasks but does not encode this in spec-level SHALL behavior.
  - Existing pattern reference: `scripts/demo/cao-claude-esc-interrupt/run_demo.sh` (`parse_launcher_start_state`, ownership mismatch handling, `CAO_SERVER_STARTED` gating in cleanup).
  - Required addition: Add requirement that stop/cleanup only shuts down launcher-managed CAO processes started by the demo flow, and scenario for untracked healthy CAO reuse outcome.

### Should Fix / Important

- Local-only constraint is implicit in narrative but absent as a normative requirement.
  - Risk: Remote CAO endpoints may be used accidentally, conflicting with tmux-local inspection assumptions and guardrails in current demos.
  - Existing pattern reference: `scripts/demo/cao-claude-esc-interrupt/run_demo.sh` (`require_local_cao`), `scripts/demo/cao-claude-tmp-write/run_demo.sh` local-only skip behavior.
  - Suggestion: Add explicit requirement/scenario that unsupported CAO base URLs produce deterministic skip/fail behavior with actionable message.

- State artifact schema is underspecified for long-term maintainability.
  - Risk: Commands (`send-turn`, `inspect`, `stop`) may diverge on field names/paths, breaking backward compatibility and troubleshooting.
  - Current partial fields: `session_manifest`, `session_name`, `terminal_id`, `terminal_log_path` in `spec.md`; tasks also mention workspace metadata and tmux target.
  - Suggestion: Define a minimal versioned schema (e.g., `schema_version`, `active`, `session_manifest`, `session_name`, `tmux_target`, `terminal_id`, `terminal_log_path`, `runtime_root`, `updated_at`).

- Verification expectations do not yet define artifact location/shape for per-turn records.
  - Risk: Requirement asks for per-turn output artifacts and non-empty responses, but report format and storage location are unconstrained.
  - Suggestion: Specify deterministic file naming and report schema for turns to support reproducible verification and CI assertions.

### Nice To Have

- Clarify CLI surface naming convention and invocation UX in spec text.
  - Existing demos are `run_demo.sh` single entrypoints; this change introduces subcommands. A clear command contract would reduce docs drift.
- Include explicit cleanup guidance for stale state/session recovery (`force-stop` or `reset-state`) as operator ergonomics.

## Design Alignment With Existing System

- Existing pattern(s) observed: `scripts/demo/cao-claude-session/run_demo.sh`, `scripts/demo/cao-claude-esc-interrupt/run_demo.sh`, `scripts/demo/cao-claude-tmp-write/run_demo.sh` all share:
  - launcher-backed CAO startup/ownership handling,
  - tmux prerequisite/local constraints,
  - explicit `build-brain` + `start-session` + `send-prompt` + `stop-session` runtime flow,
  - machine-readable report + verifier scripts.
- Proposed design alignment:
  - Aligns well: dedicated new demo pack, explicit lifecycle commands, persisted metadata, multi-turn reuse goal.
  - Mismatches/gaps: does not yet codify ownership and local-only invariants at the spec layer; does not define state schema and active-state conflict behavior, which are necessary to match robustness of existing demos.
  - More native approach: model state/reporting after existing report+verify discipline (predictable JSON artifacts + verifier script), while preserving interactive command decomposition.

## Open Questions (With Proposed Defaults)

### Q1) What should `start` do when an active session state file already exists?

- Decision: `Blocking`

#### Why this matters

- This determines whether the workflow is safe for repeated use and automation without leaking sessions.

#### Examples

- Example 1: operator runs `start` twice without `stop`; second invocation must not silently orphan the first session.
- Example 2: CI retry on flaky network re-runs `start`; behavior must be deterministic and auditable.

#### Why this question arises in code

- `design.md` defines explicit lifecycle commands and persisted state, but does not define active-state conflict semantics.
- `tasks.md` includes startup persistence and stop handling, but no explicit lock/conflict rule.
- Existing demos are one-shot scripts and do not need cross-invocation state conflict handling.

```bash
# Decision point that needs a contract:
if state.active == true:
  ???  # fail | replace | reuse
else:
  start_session()
```

#### Options

**Option A: Hard fail with recovery guidance (Recommended)**

- Code impact:
  - Add a preflight guard in startup command that reads `state.json` and exits non-zero when `active=true`.
  - Include actionable remediation in output (`inspect`, `stop`, optional `reset-state --force`).
- User-facing difference:
  - Second `start` is rejected with clear next steps; no silent replacement.
- Pros:
  - Prevents accidental session leaks and hidden session replacement.
  - Keeps operator intent explicit.
- Cons:
  - Adds one extra explicit step (`stop` or `reset`) in some workflows.

**Option B: Implicitly replace existing active session**

- Code impact:
  - Startup path must stop/abandon existing session and rewrite state atomically.
  - Requires extra cleanup/error branches for partial replacement failure.
- User-facing difference:
  - Re-running `start` appears convenient but may lose context and hide orphan risk.
- Pros:
  - Fewer commands for users in a hurry.
- Cons:
  - High risk of orphaning server-side sessions and losing auditability.

#### Proposal (Recommended)

Make `start` fail if `active=true`, print current session identity + exact remediation (`inspect`, `stop`, optional `reset-state --force`).

#### Pros / Cons (Proposal)

- Pros:
  - Strong lifecycle safety.
  - Easier debugging and deterministic behavior.
- Cons:
  - Slightly less convenient for ad-hoc usage.

> **DECISION: For Q1, `start` force-closes any existing active session, then starts a new one.**
> Rationale: This workflow is a demo explicitly focused on operating one agent at a time; convenience and operator flow are prioritized over preserving prior interactive session continuity. The command should make replacement explicit in output and update state atomically so the single-session invariant remains clear.

### Q2) Should unsupported/non-local `CAO_BASE_URL` be skipped or failed?

- Decision: `Blocking`

#### Why this matters

- Existing demos differ between skip and fail semantics depending on scenario; this affects CI and local user expectations.

#### Examples

- Example 1: `CAO_BASE_URL=https://remote.example` in local dev should not attempt tmux-local interaction.
- Example 2: CI environment accidentally sets non-local CAO URL; outcome should be explicit and non-flaky.

#### Why this question arises in code

- The design states local-only assumptions, but the spec does not define normative behavior for unsupported base URLs.
- Existing scripts show local-only gate patterns that skip with explicit messaging.

```bash
case "$CAO_BASE_URL" in
  http://localhost:9889|http://127.0.0.1:9889) run_demo ;;
  *) skip_or_fail_with_message ;;
esac
```

#### Options

**Option A: Fail hard on non-local base URL**

- Code impact:
  - Validate `CAO_BASE_URL` during preflight and exit non-zero on unsupported URL.
- User-facing difference:
  - Misconfiguration is loud, but local-only scripts become less portable in mixed environments.
- Pros:
  - Strongly enforces intended local-only behavior.
- Cons:
  - Less flexible for advanced environments.

**Option B: Deterministic SKIP with explicit reason (Recommended)**

- Code impact:
  - Keep local-only gate, return success-with-skip status plus explicit diagnostic message.
  - Ensure report artifact preserves skipped reason for operator visibility.
- User-facing difference:
  - Users get clear "not runnable here" signal without noisy hard-fail in non-target environments.
- Pros:
  - Matches local-only demo ergonomics used by current tmux-focused demos.
  - Reduces false negatives in heterogeneous environments.
- Cons:
  - Can mask accidental misconfiguration if users miss skip logs.

#### Proposal (Recommended)

For tmux-inspection-focused interactive demo, use deterministic `SKIP` on non-local base URL with explicit accepted values and remediation guidance.

#### Pros / Cons (Proposal)

- Pros:
  - Consistent with existing local-only CAO/tmux demos.
  - Cleaner operator experience.
- Cons:
  - Requires clear reporting so skips are visible.

> **DECISION: For Q2, fix `CAO_BASE_URL` to `http://127.0.0.1:9889` for this demo.**
> Rationale: This demo is intentionally local and single-path; pinning the URL removes ambiguity and keeps operator behavior deterministic. The startup flow should ignore external base-url overrides and always use the fixed loopback target.

### Q3) What is the canonical on-disk schema for interactive state and per-turn artifacts?

- Decision: `Blocking`

#### Why this matters

- Multiple commands and tests will read/write the same files; schema drift is likely without a contract.

#### Examples

- Example 1: `send-turn` expects `session_manifest`, but startup wrote `manifest_path`; command fails unexpectedly.
- Example 2: `inspect` expects `tmux_target`, but state stores only `session_name`; user guidance becomes inconsistent.

#### Why this question arises in code

- `spec.md` lists only a subset of fields and does not define versioning or compatibility rules.
- `tasks.md` requires state sharing across startup/send/inspect/stop and per-turn reporting, which needs a stable schema.
- Existing demo verifiers assume deterministic report shape and fail on schema drift.

```json
{
  "schema_version": 1,
  "active": true,
  "session_manifest": "...",
  "session_name": "...",
  "tmux_target": "...",
  "terminal_id": "...",
  "terminal_log_path": "...",
  "runtime_root": "...",
  "updated_at": "2026-03-06T11:29:17Z"
}
```

#### Options

**Option A: Ad-hoc fields evolved per command**

- Code impact:
  - Minimal upfront schema work, but each command handles missing/renamed fields independently.
- User-facing difference:
  - Inconsistent failures and harder troubleshooting across commands/versions.
- Pros:
  - Fast initial iteration.
- Cons:
  - High compatibility and test fragility risk.

**Option B: Versioned schema with fixed required fields (Recommended)**

- Code impact:
  - Define required keys and validation at read/write boundaries for state and turn artifacts.
  - Add schema migration/version checks for future changes.
- User-facing difference:
  - Predictable command behavior and clearer error messages when artifacts are invalid.
- Pros:
  - Stable contract for tests and docs.
  - Easier migration if fields evolve.
- Cons:
  - Slight upfront design effort.

#### Proposal (Recommended)

Define `state.json` and `turns/*.json` schemas in design/spec, including required fields, timestamp format, and update rules (`active` transitions on start/stop).

#### Pros / Cons (Proposal)

- Pros:
  - Reliable behavior across lifecycle commands.
  - Better long-term maintainability.
- Cons:
  - More explicit specification work now.

> **DECISION: For Q3, use `--agent-identity` (name form) as the primary session handle, and support targeting sessions by name.**
> Rationale: Current runtime already supports name-based identity resolution for tmux-backed backends: `start-session` accepts name input and normalizes identity (`src/gig_agents/agents/brain_launch_runtime/runtime.py`, `src/gig_agents/agents/brain_launch_runtime/agent_identity.py`), and resume commands resolve `--agent-identity` names to the manifest path (`resolve_agent_identity` in `runtime.py`). The reference docs also state that `start-session --agent-identity <name>` is supported for tmux-backed backends (`docs/reference/brain_launch_runtime.md`). This demo should align with that existing capability and treat name-based identity as the intended operator-facing targeting model.
