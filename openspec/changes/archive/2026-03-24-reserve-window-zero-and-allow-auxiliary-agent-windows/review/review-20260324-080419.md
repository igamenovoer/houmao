## Summary

- The pair boundary is coherent: the change consistently moves the public attach surface to `houmao-srv-ctrl agent-gateway` and keeps `houmao-server` as the public authority behind `/houmao/agents/*`.
- The delta specs line up with existing runtime concepts such as stable tmux-published gateway pointers and offline gateway state, so the direction is native to the repository rather than a parallel model.
- Two implementation-critical contracts are still underspecified: how current-session attach turns tmux-published metadata into one managed-agent route target, and how same-session tmux-hosted gateways are represented as a durable execution handle for attach, detach, and cleanup.
- Launch-time attachability publication also needs a tighter sequencing story so the session is not advertised as attachable before both runtime artifacts and server-side managed-agent registration are actually ready.

## Findings

### Must Fix / Blocking

1. **The same-session gateway design does not define the new execution-handle contract that replaces the current PID-only detached-process model.** The existing live gateway state is centered on `GatewayCurrentInstanceV1`, which persists only `pid`, `host`, `port`, `managed_agent_instance_epoch`, and `managed_agent_instance_id` in `src/houmao/agents/realm_controller/gateway_models.py:301`. The current runtime attach and detach flows are equally PID-shaped: attach starts a detached `subprocess.Popen(...)` in `src/houmao/agents/realm_controller/runtime.py:2477`, waits on `process.poll()` in `src/houmao/agents/realm_controller/runtime.py:2507`, and detach reads `gateway.pid` plus `os.kill(...)` in `src/houmao/agents/realm_controller/runtime.py:2185`. The proposed design says same-session mode should treat an auxiliary tmux window and pane as the authoritative local execution surface, but the artifacts never specify where that pane/window identity is stored, how detach/crash cleanup resolves it, or how the state file evolves when the auxiliary window is recreated.

> **DECISION: Accept.**
> Rationale: The revised artifacts already commit to a tmux-owned same-session lifecycle, but they still do not define the persisted execution-handle contract that replaces the current PID-only live state. A later artifact revision should add one runtime-owned live gateway record under `<session-root>/gateway/run/` that carries execution mode plus tmux window/pane identity so attach, detach, crash cleanup, and auxiliary-window recreation do not rediscover non-contractual windows heuristically.

2. **Current-session `houmao-srv-ctrl agent-gateway attach` is missing a precise normalization rule from tmux-published attach metadata to the managed-agent route target.** The design correctly says current-session mode must validate `AGENTSYS_GATEWAY_ATTACH_PATH` and `AGENTSYS_GATEWAY_ROOT`, load a readable `houmao_server_rest` attach contract, and then call the same managed-agent attach route. But the current `houmao_server_rest` attach metadata only stores `api_base_url`, `session_name`, `terminal_id`, `parsing_mode`, and optional `tmux_window_name` in `src/houmao/agents/realm_controller/gateway_storage.py:367`, while the server alias resolver accepts several possible aliases in `src/houmao/server/service.py:1631`. The artifacts never state which field becomes `{agent_ref}`, whether the CLI must use the attach contract's persisted `api_base_url` or a user-supplied pair target, or how stale-pointer failure is distinguished from wrong-server or ambiguous-alias failure. That is core behavior for the new command surface, so it should be explicit in the design/tasks rather than left implicit.

> **DECISION: Accept.**
> Rationale: The revised design now requires tmux-env validation plus server-routed attach, but it still does not pin which persisted field becomes `{agent_ref}` or which server authority wins in no-`--agent` mode. The follow-up artifact revision should state that current-session attach uses the persisted `houmao_server_rest` attach contract as authoritative for both `api_base_url` and `session_name`, and should treat mismatch or failed resolution as invalid or stale current-session metadata rather than falling back to other aliases.

### Should Fix / Important

3. **The change should explicitly reuse the existing runtime-owned gateway-capability publisher instead of leaving room for a second pair-specific artifact writer.** The repository already has one authoritative seam for publishing `attach.json`, offline `state.json`, queue/bootstrap files, and stable tmux env pointers: `RuntimeSessionController.ensure_gateway_capability()` in `src/houmao/agents/realm_controller/runtime.py:425`. By contrast, delegated pair launch currently materializes only the manifest and a small tmux env set in `src/houmao/srv_ctrl/commands/runtime_artifacts.py:174` and `src/houmao/srv_ctrl/commands/runtime_artifacts.py:203`. The artifacts say pair launch should seed gateway capability early, but they do not say whether `houmao-srv-ctrl` should call the existing runtime seam or hand-roll another writer. The native approach here is to keep one gateway-capability publisher and have pair launch route through it.

> **DECISION: Refine.**
> Rationale: The reviewer is directionally right that pair launch should not invent a second gateway-capability writer. The artifact follow-up should require reuse of the existing runtime/gateway storage publication seam, but it does not need to hard-code a direct call to `RuntimeSessionController.ensure_gateway_capability()` if the implementation extracts and reuses the shared helper beneath that method instead of creating a pair-specific publisher.

4. **The launch-time readiness boundary for current-session attach is still too loose.** The change says delegated pair launch or launch registration should publish stable gateway pointers early enough that `houmao-srv-ctrl agent-gateway attach` can be run from inside the tmux session without `--agent`. In practice, that workflow depends on two different authorities being ready at the same time: the tmux session must already publish stable gateway env pointers, and `houmao-server` must already know the session under the managed-agent namespace so `/houmao/agents/{agent_ref}/gateway/attach` can resolve it. The current launch path does registration after artifact materialization in `src/houmao/srv_ctrl/commands/launch.py:145`, which is workable, but the change artifacts do not turn that ordering into an explicit contract. The design should state when current-session attach becomes valid, and tests should lock that boundary down.

> **DECISION: Accept.**
> Rationale: The revised docs require launch-time gateway capability publication, but they still leave the current-session readiness boundary implicit. The follow-up artifact revision should state that no-`--agent` attach becomes valid only after both the stable tmux gateway pointers are published and the managed-agent registration for that same `api_base_url` and session has succeeded, with tests covering the pre-registration failure case.

### Nice To Have

5. **Task 3.1 could call out the server-side TUI tracker by name so the work scope is easier to execute.** Pair-managed probing currently resolves a tracked tmux target through `TmuxTransportResolver.resolve_target(...)` in `src/houmao/server/service.py:1346`, using the stored `identity.tmux_window_name` rather than a pure active-pane heuristic. The current task wording correctly mentions resolvers, transport fallbacks, and repo-owned helpers, but naming the server tracker explicitly would make it clearer that this change is not only about demos or ad hoc scripts.

> **DECISION: Refine.**
> Rationale: Task 3.1 should make the server-side managed TUI tracker and resolver scope more explicit, but it should not freeze the work to one class name. The later artifact revision should call out the server-side managed-agent tmux tracking and resolution path alongside repo helpers so the implementation target is concrete without over-constraining the patch.

## Design Alignment With Existing System

- Existing pattern(s) observed:
  - `RuntimeSessionController.ensure_gateway_capability()` in `src/houmao/agents/realm_controller/runtime.py:425` is already the runtime-owned seam for stable gateway capability publication plus tmux env export.
  - Managed-agent gateway lifecycle already lives on `POST /houmao/agents/{agent_ref}/gateway/attach` in `src/houmao/server/app.py:383`, and alias resolution is centralized in `src/houmao/server/service.py:1631`.
  - The runtime already treats one file under `<session-root>/gateway/run/current-instance.json` as authoritative ephemeral live-gateway state via `src/houmao/agents/realm_controller/gateway_storage.py:505`.
- Proposed design alignment:
  - Moving the public attach verb to `houmao-srv-ctrl agent-gateway` aligns cleanly with the pair boundary already described in docs and server routes.
  - Requiring launch-time stable gateway pointers for delegated `houmao_server_rest` sessions aligns with the existing gateway-capability and offline-status model.
  - The design is least aligned where it leaves room for a second artifact writer and where it introduces tmux-hosted same-session gateway lifecycle without defining how that new local execution handle is persisted in the same way the detached PID model is persisted today.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Which identifier should current-session `houmao-srv-ctrl agent-gateway attach` send to `/houmao/agents/{agent_ref}/gateway/attach` after it loads the tmux-published `houmao_server_rest` attach contract?

- Decision: `Blocking`

#### Why this matters

- The change's main UX addition is no-`--agent` attach from inside the managed tmux session.
- That path only works if the CLI converts the validated attach contract into one exact managed-agent route target and one exact server authority.

#### Examples

- The attach contract points at `api_base_url=http://127.0.0.1:9889`, `session_name=cao-gpu`, and `terminal_id=abcd1234`.
- The server currently accepts several aliases for the same managed agent, including tracked session id, session name, terminal id, agent name, and agent id.

#### Why this question arises in code

- `src/houmao/agents/realm_controller/gateway_storage.py:367` stores both `session_name` and `terminal_id` for `houmao_server_rest`.
- `src/houmao/server/service.py:1650` accepts both `identity.session_name` and `identity.terminal_aliases` as valid managed-agent aliases.
- The design says current-session attach must fail closed on stale or ambiguous metadata, but different alias choices produce different failure surfaces.

#### Options

**Option A: Use the attach contract's `api_base_url` plus `backend_metadata.session_name` as the only managed-agent route target for current-session attach.**

- Code impact:
  - The CLI loads `attach.json`, verifies `backend == "houmao_server_rest"`, then calls `HoumaoServerClient(api_base_url).attach_managed_agent_gateway(session_name)`.
  - User-provided server targeting is either disallowed for this mode or must exactly match the persisted `api_base_url`.
- User-facing difference:
  - Current-session attach is tied to the session that published the attach contract, not to whichever pair target the operator happens to pass on the CLI.
- Pros:
  - Keeps raw `terminal_id` out of the public pair workflow.
  - Uses an alias that is already first-class in the managed-agent resolver.
  - Makes stale-pointer handling simpler because both authority and alias come from one persisted contract.
- Cons:
  - Requires the command surface to treat persisted server authority as canonical for no-`--agent` mode.

**Option B: Use the attach contract only as validation input, then resolve or call the server using `terminal_id` or another alias.**

- Code impact:
  - The CLI still loads `attach.json`, but the managed-agent route target becomes `terminal_id` or another alias instead of `session_name`.
  - The server target may come from the CLI or a different pair configuration source.
- User-facing difference:
  - The no-`--agent` path depends on more than the attach contract and exposes more of the raw CAO identity model behind the pair.
- Pros:
  - Reuses an alias that is also valid on the server today.
  - May be convenient if another pair command family already prefers `terminal_id`.
- Cons:
  - Weakens the pair abstraction.
  - Makes stale vs wrong-server vs ambiguous-reference behavior harder to explain.

> **PROPOSED: Use Option A and make the tmux-published `houmao_server_rest` attach contract authoritative for both the server base URL and the managed-agent alias, with `backend_metadata.session_name` as the route target.**
> Rationale: The pair change is explicitly about hiding raw `cao_rest` identities, and `session_name` is already a first-class alias in `src/houmao/server/service.py:1650` while `terminal_id` is the raw CAO-facing detail the change wants to keep out of the public model. Using the attach contract's `api_base_url` also keeps no-`--agent` attach tied to the session that actually published the stable pointers instead of a potentially unrelated CLI target.
>
> **DECISION: Accept PROPOSED. Current-session attach should use the persisted `api_base_url` plus `backend_metadata.session_name` as the only route target.**
> Rationale: That keeps the no-`--agent` flow bound to the session that published the stable tmux pointers, matches the server's existing first-class `session_name` alias, and keeps raw `terminal_id` out of the pair UX. If the CLI ever accepts an explicit pair target in this mode, it should have to match the persisted authority exactly rather than retargeting the command.

### Q2) Where should the same-session auxiliary-window execution handle be persisted so attach, detach, crash cleanup, and recreation all act on the right gateway surface?

- Decision: `Blocking`

#### Why this matters

- The design intentionally replaces detached subprocess ownership with tmux-owned auxiliary-window ownership for `houmao_server_rest`.
- That only works if the runtime has one authoritative place to remember which tmux surface currently owns the live gateway.

#### Examples

- Attach creates a new auxiliary window for the gateway and the process later crashes before binding its port.
- Detach runs after a gateway restart and must avoid killing the agent window `0` or clearing the wrong stale pane.

#### Why this question arises in code

- `src/houmao/agents/realm_controller/gateway_models.py:301` currently persists only PID-based current-instance metadata.
- `src/houmao/agents/realm_controller/runtime.py:2185` detach logic reads `gateway.pid` and kills that PID directly.
- The proposed same-session mode says liveness should come from tmux-owned pane state, but the artifact set does not yet define any pane/window handle.

#### Options

**Option A: Extend the runtime-owned live gateway state to include execution mode plus tmux pane/window identifiers.**

- Code impact:
  - Add fields such as `execution_mode`, `tmux_window_id`, `tmux_pane_id`, and optionally `tmux_window_name` to the current-instance payload or a sibling runtime-owned file.
  - Attach writes those fields when same-session mode is active, and detach/cleanup resolves the live gateway surface from them instead of from PID alone.
  ```json
  {
    "execution_mode": "tmux_aux_window",
    "tmux_window_id": "@12",
    "tmux_pane_id": "%34"
  }
  ```
- User-facing difference:
  - Detach and crash cleanup can be reliably targeted even after auxiliary-window recreation.
- Pros:
  - Follows the repository's existing pattern of one runtime-owned authoritative live-gateway state file.
  - Makes tests and recovery behavior explicit.
- Cons:
  - Requires a schema revision and migration handling for older live-state files.

**Option B: Keep the current PID-only live state and rediscover the auxiliary tmux surface ad hoc each time.**

- Code impact:
  - Same-session attach does not change the persisted schema.
  - Detach/cleanup must scan tmux windows and infer which pane is the gateway from logs, names, PIDs, or heuristics.
- User-facing difference:
  - Behavior depends more on runtime inference and less on explicit state.
- Pros:
  - Smaller state-schema change.
  - Lower immediate migration cost.
- Cons:
  - Conflicts with the design goal that non-zero window names, indices, and counts remain non-contractual.
  - Makes cleanup and recreation much more fragile.

> **PROPOSED: Use Option A and extend the runtime-owned live gateway state with an explicit same-session execution-handle shape for tmux-hosted mode.**
> Rationale: The current repository already treats `current-instance.json` as the authoritative live-gateway state for detached mode, so evolving that seam is more native than inventing runtime heuristics over non-contractual auxiliary windows. Persisted pane/window identifiers also let the change satisfy its own safety rules: detach, cleanup, and recreation can target the auxiliary gateway surface without guessing and without risking window `0`.
>
> **DECISION: Accept PROPOSED. Persist a runtime-owned same-session execution handle with explicit execution mode and tmux window/pane identifiers.**
> Rationale: Same-session lifecycle needs one authoritative execution handle just as detached mode already has one authoritative live-state file. The later artifact revision should extend `current-instance.json` or a sibling runtime-owned live-state file under `gateway/run/`, but it should not rely on ad hoc tmux rediscovery across attach, detach, cleanup, or auxiliary-window recreation.

## Suggested Next Steps

1. Revise the design and tasks to define the same-session live execution-handle/state contract, including where tmux pane/window identity is persisted and how detach/cleanup consume it.
2. Tighten current-session attach semantics so the artifacts explicitly state which attach-contract fields become the managed-agent route target and the authoritative server base URL.
3. Update the launch/publication tasks so they say pair launch reuses the existing runtime gateway-capability seam and define the exact point at which current-session attach becomes valid after registration.
