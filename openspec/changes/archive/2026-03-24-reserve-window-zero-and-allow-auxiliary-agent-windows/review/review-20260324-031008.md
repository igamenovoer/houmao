## Summary

- **Well-structured overall**: The proposal, design (7 decisions), delta specs, and task breakdown are internally consistent. The backend-specific scoping (headless + `cao_rest` get same-session; `houmao_server_rest` stays out-of-session) is clean and well-motivated.
- **Window-0 contract is sound and actionable**: Reserving window `0` for the agent surface while keeping non-zero windows non-contractual is a pragmatic split that avoids over-specification. The existing `HEADLESS_AGENT_WINDOW_INDEX = "0"` constant in `tmux_runtime.py:81` already establishes this for headless backends.
- **CAO window-0 normalization is under-specified for implementation**: The spec adds a "SHALL normalize ... so the CAO agent process occupies window `0`" requirement, but neither the design nor tasks specify the tmux primitive to achieve this (e.g., `move-window`, `swap-window`, `renumber-windows`). The current `_start_terminal` in `cao_rest.py:1644–1730` has no such operation, and the choice of tmux primitive has edge-case implications.
- **Gateway lifecycle monitoring gap**: The same-session tmux-window launcher replaces a `subprocess.Popen` handle with a tmux window. Existing gateway health-check and port-wait logic (runtime.py ~2496–2510) relies on `process.poll()` which won't work for a tmux-hosted process. The tasks don't address how to detect gateway process death in a tmux window.
- **`resolve_active_pane_id` usage is widespread**: At least 6 call sites across `demo/`, `explore/`, and `server/tui/` use session-wide active-pane resolution. Task 3.1 correctly scopes this but may be more work than it appears.
- **Relaunch-to-window-0 mechanics are unspecified**: The spec requires relaunched agent processes to be restored to window `0`, but the repo currently has no relaunch/recovery path in `runtime.py` (only a single `grep` match for "recover" in runtime.py relates to stdio, not tmux). The spec creates a new requirement without an implementation anchor.

## Findings

### Must Fix / Blocking

1. **CAO window-0 normalization lacks tmux-primitive specification.** The delta `brain-launch-runtime` spec adds: "the runtime SHALL normalize the tmux session so the CAO agent process occupies window `0`". The current `_start_terminal` in `cao_rest.py:1644–1730` performs `_select_tmux_window` and bootstrap-window pruning, but never issues a `move-window`, `swap-window`, or `renumber-windows` command. The design (Decision 5) says "normalize the session so the agent process occupies window `0`" without specifying the tmux operation. The choice matters:
   - `tmux move-window -s @<id> -t <session>:0` will fail if index 0 is already occupied by the bootstrap window (which it typically is, since `create_tmux_session` creates the initial window at index 0).
   - `tmux swap-window` requires two window targets and has different semantics.
   - `tmux move-window -r` (renumber) renumbers all windows starting from the base index but may affect future auxiliary windows.

   Without specifying the normalization primitive and its interaction with the bootstrap-window prune step (which may kill the window at index 0 first, freeing the slot), implementation risks silent failures or tmux errors.

> **DECISION: Accept.**
> Rationale: This is a real artifact gap and should be fixed before implementation starts. The revision pass should commit to a concrete v1 normalization sequence: prune the distinct bootstrap window first, then move the resolved CAO agent window into `:0`; if that cannot safely establish window `0`, same-session auxiliary-window topology remains unavailable for that session.

2. **Gateway process death detection needs a new mechanism for tmux-hosted mode.** The current `_start_gateway_process` in `runtime.py:2477–2494` launches the gateway via `subprocess.Popen` and uses `process.poll()` (~line 2507) plus PID-based health polling to detect process death. When the gateway runs in a tmux auxiliary window instead, there is no `Popen` handle. The design (Decision 3) describes the same-session launcher but neither the design nor the tasks address how the runtime will monitor gateway liveness (e.g., via `pane_dead` tmux state, `pane_pid`, or gateway HTTP health polling only). This is blocking because the gateway attach/detach lifecycle in `runtime.py` currently couples tightly to the `Popen` object.

> **DECISION: Accept.**
> Rationale: The reviewer correctly identifies a missing lifecycle contract. Same-session gateway launch cannot keep pretending the tmux-hosted gateway is just another `Popen`, so the artifacts need an explicit same-session handle and monitoring model for liveness, readiness, and teardown.

### Should Fix / Important

3. **`prepare_headless_agent_window` conflates rename and select into one call.** The function at `tmux_runtime.py:138–159` unconditionally performs both `rename-window` AND `select-window` for window `0`. Decision 4 says "stop assuming that window `0` must also be the currently selected window before each turn," but the function is called in two contexts:
   - Initial session creation (`headless_base.py:311` and `headless_base.py:274`) — here, selecting window `0` is appropriate.
   - Pre-turn preparation (`headless_runner.py:352`) — here, the select should be removed per the change.

   The tasks (2.1) say "Refactor headless tmux helpers so they keep window `0` named `agent` without forcing window `0` to be the selected foreground window," but should explicitly note that the initial-creation call path may need a separate select step (or accept that window `0` is not selected at creation either, since it's the only window at that point and is selected by default).

> **DECISION: Refine.**
> Rationale: The helper conflation is real, but task 2.1 already points at the right work. The later artifact revision should clarify that creation-time and turn-time call paths may use different selection behavior, rather than expanding scope into a separate new requirement.

4. **`TmuxTransportResolver` in `server/tui/transport.py` uses `_prefer_active_pane` as ultimate fallback.** When `window_name` is `None` (line 44–45), the resolver falls back to session-wide active-pane preference across all windows. With auxiliary windows, this could resolve to a gateway pane instead of the agent pane. The server TUI transport is used for `houmao_server_rest`-owned sessions (which the change excludes from same-session topology), so this may be safe today — but the fallback behavior is fragile if the transport is ever reused for headless sessions with auxiliary windows. Task 3.1 should consider adding a defensive check or explicit documentation that `TmuxTransportResolver` is only safe for server-managed sessions.

> **DECISION: Reject.**
> Rationale: The implementation caution is valid, but it is not a missing artifact gap in this change. Task 3.1 already includes transport fallbacks, and the same-session auxiliary-window topology explicitly excludes `houmao_server_rest`, which is where this resolver is used today.

5. **Relaunch-to-window-0 spec requirement has no existing implementation anchor.** The delta `brain-launch-runtime` spec adds a scenario "Relaunch restores the agent process to window 0" (lines 109–112), and the design (Decision 6) describes the recovery invariant. However, `runtime.py` has no relaunch/recovery code path for restoring a dead agent to window `0` — the single recovery-related match is about stdio channels (line 946). Task 1.3 says "Preserve and restore the canonical agent surface in window `0` across ... runtime relaunch flows," but this implies building new recovery code, not refactoring existing code. The task description should note that this is greenfield work requiring new primitives (detect agent death, reclaim window `0`, relaunch process there).

> **DECISION: Refine.**
> Rationale: The reviewer is right that task 1.3 is greenfield work, but lack of an existing implementation anchor is not itself a proposal defect. The later artifact revision should tighten the design/tasks so it is explicit that new recovery primitives are required rather than implying a refactor of existing recovery code.

6. **`headless_runner.py` targets pane via `headless_agent_pane_target` which uses index-based targeting, not `window_id`.** The runner at line 352 calls `prepare_headless_agent_window_shared` and then uses `headless_agent_pane_target` (which returns `session:0.0`). This index-based targeting is correct only as long as window `0` exists and contains the agent. If an auxiliary window is created before window `0` somehow (race condition, operator intervention), index-based targeting could be fragile. The existing code is fine for the current design, but the change should either validate that window `0` exists before each turn or switch to `window_id`-based targeting for robustness. This is a "should have" not a blocker, since the spec says window `0` is always reserved.

> **DECISION: Reject.**
> Rationale: The contract intentionally makes window `0` the canonical execution surface, so index-based targeting is aligned with the change rather than a mismatch. The implementation can add validation that window `0` still hosts the agent, but the artifacts should not pivot to `window_id` as the primary control contract.

### Nice To Have

7. **Consider naming the gateway auxiliary window for operator discoverability.** The design (Decision 2) says non-zero window names are non-contractual, but giving the gateway auxiliary window a consistent human-readable name (e.g., `gateway`) would improve the operator experience when `tmux list-windows` is used, without making it contractual. The durable contract remains window `0` only; the name would be a convenience.

> **DECISION: Reject.**
> Rationale: The user explicitly chose to keep non-zero windows non-contractual and not pinned to a preferred identity. An implementation may still choose readable names opportunistically, but the change artifacts should not steer or imply a stable auxiliary-window naming convention.

8. **Delta spec scenarios could include a negative test for tmux window count.** Adding a scenario that asserts "the runtime does not create more than one auxiliary window per gateway attach" would prevent future regressions where repeated attach/detach cycles leak tmux windows.

> **DECISION: Reject.**
> Rationale: Avoiding leaked auxiliary windows belongs in tests and implementation hygiene, not in a normative spec scenario about tmux window counts. This change intentionally leaves non-zero window counts non-contractual, so adding that scenario would over-specify v1 behavior.

## Design Alignment With Existing System

- **Existing pattern — headless window-0 ownership**: The `HEADLESS_AGENT_WINDOW_INDEX = "0"` and `HEADLESS_AGENT_WINDOW_NAME = "agent"` constants in `tmux_runtime.py:81–82` establish window `0` as the agent surface. The `headless_agent_window_target` and `headless_agent_pane_target` functions use these constants for stable targeting. The proposed change aligns with and strengthens this existing pattern.

- **Existing pattern — CAO bootstrap-window pruning**: `cao_rest.py:1644–1730` already records the bootstrap window, resolves the CAO terminal by name, selects it, and prunes the bootstrap window. The proposed window-0 normalization extends this sequence naturally. However, the existing code does NOT move or renumber windows — it only selects and prunes. The proposed normalization is a new operation inserted into an existing sequence, and its ordering relative to bootstrap pruning needs explicit specification (normalize first, then prune? or prune first to free index 0, then move?).

- **Existing pattern — gateway as detached subprocess**: `runtime.py:2477–2494` launches the gateway via `subprocess.Popen` with `start_new_session=True`, stdout redirected to `gateway.log`. The proposed same-session tmux-window launcher is a new pattern that diverges from the existing `Popen` model. The design correctly isolates this behind a launcher-mode split (Decision 3), keeping the detached launcher for `houmao_server_rest`.

- **Existing pattern — `resolve_active_pane_id` for session-wide pane lookup**: Used in `demo/shared_tui_tracking_demo_pack/tooling.py:138`, `demo/shared_tui_tracking_demo_pack/live_watch.py:165,508`, `demo/shared_tui_tracking_demo_pack/recorded.py:269`, `explore/claude_code_state_tracking/runtime_support.py:40`, `explore/claude_code_state_tracking/live.py:189`, and `explore/claude_code_state_tracking/interactive_watch.py:144,719`. All of these scan the entire session for the active pane, which breaks when an auxiliary window is selected. The proposed change (Decision 7, Task 3.1) correctly identifies this pattern as needing explicit agent-surface targeting.

- **Partial alignment — `TmuxTransportResolver`**: `server/tui/transport.py:22–59` already supports `window_name`-filtered pane resolution, which would work correctly when passed `"agent"`. However, it falls back to session-wide active-pane preference when `window_name` is `None`. For server-managed sessions (excluded from same-session topology), this is safe. The proposed change aligns with this existing resolver's design.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What tmux primitive should CAO window-0 normalization use, and in what order relative to bootstrap-window pruning?

- Decision: `Blocking`

#### Why this matters

- The spec adds a SHALL-level requirement for CAO agent-process placement in window `0`, but the implementation path depends on which tmux command is used and when it runs relative to the existing bootstrap-prune step.

#### Examples

- If the CAO terminal is created at index 1 (because index 0 is occupied by the bootstrap window), normalization must either (a) kill the bootstrap window first to free index 0, then move the CAO terminal there, or (b) swap windows 0 and 1, then kill the former bootstrap window.
- If the CAO terminal is created at index 0 (because the bootstrap window was already at index 0 and CAO replaced it), no normalization is needed.

#### Why this question arises in code

- `cao_rest.py:1659–1662` creates a tmux session (which creates window at index 0 by default) as the bootstrap window.
- `cao_rest.py:1673–1678` creates the CAO terminal, which may create a new window at a non-zero index.
- `cao_rest.py:1708` selects the CAO terminal window, and `cao_rest.py:1722` kills the bootstrap window if distinct.
- No `move-window` or `swap-window` call exists in the current flow.
- The normalization step must fit between these existing operations without breaking the bootstrap-prune logic.

#### Options

**Option A: Prune bootstrap window first, then move CAO terminal to index 0**

- Code impact:
  - After `_kill_tmux_window(bootstrap_window)` at line 1722, add `run_tmux_shared(["move-window", "-s", terminal_window.window_id, "-t", f"{session_name}:0"])`.
  - The bootstrap window at index 0 is killed first, freeing the slot.
  ```python
  # After existing bootstrap prune:
  _kill_tmux_window(window_id=bootstrap_window.window_id)
  # New normalization:
  run_tmux_shared(["move-window", "-s", terminal_window.window_id, "-t", f"{session_name}:0"])
  ```
- User-facing difference:
  - CAO terminal reliably ends up at window index 0 after startup.
- Pros:
  - Simple and avoids index collision since index 0 is freed before the move.
  - Leverages existing prune step ordering.
- Cons:
  - If bootstrap prune fails (warning-only today), the move will also fail due to index 0 being occupied.
  - Requires handling the edge case where bootstrap and terminal are the same window (no prune, no move needed).

**Option B: Use `swap-window` to exchange bootstrap and terminal indices, then prune**

- Code impact:
  - Before pruning, swap the bootstrap and terminal windows: `run_tmux_shared(["swap-window", "-s", terminal_window.window_id, "-t", bootstrap_window.window_id])`.
  - Then prune the bootstrap window (which now has the terminal's old index).
  ```python
  # Swap:
  run_tmux_shared(["swap-window", "-s", terminal_window.window_id, "-t", bootstrap_window.window_id])
  # Prune (bootstrap now at terminal's old index):
  _kill_tmux_window(window_id=bootstrap_window.window_id)
  ```
- User-facing difference:
  - Same end result: CAO terminal at index 0.
- Pros:
  - Atomic swap means there's no moment when window 0 is empty.
  - Works even if other windows exist.
- Cons:
  - `swap-window` swaps window contents, which may interact unexpectedly with tmux-internal state (e.g., `window_id` doesn't change, but index does).
  - More complex to reason about for the prune step since the bootstrap window is now at a different index.

> **PROPOSED: Use Option A (prune first, then move) since it follows the existing prune-then-select pattern in `_start_terminal` and avoids the semantic complexity of `swap-window`.**
> Rationale: The current `_start_terminal` already kills the bootstrap window before any post-startup adjustments (lines 1721–1728), so extending that flow with a `move-window` after the kill is the natural insertion point. The edge case where bootstrap and terminal are the same window is already handled at line 1718 (early return), which also means no normalization is needed in that case. The failure-path risk (bootstrap prune warning → move fails) can be mitigated by skipping the move when pruning fails, aligning with the spec's "refuse unsafe same-session auxiliary-window topology" clause.
>
> **DECISION: Accept with refinement: prune the bootstrap window first, then move the resolved CAO agent window to `:0`; if prune or move cannot safely establish window `0`, same-session auxiliary-window topology remains unavailable for that session.**
> Rationale: This follows the existing `_start_terminal(...)` sequence more naturally than a swap-based path and keeps the stronger invariant on window `0` without introducing a second tmux-shuffling mechanism. It also fits Decision 5's fail-closed rule by refusing unsafe normalization rather than guessing.

### Q2) How should the runtime monitor gateway liveness when the gateway runs in a tmux auxiliary window instead of a `Popen` subprocess?

- Decision: `Blocking`

#### Why this matters

- The existing gateway lifecycle in `runtime.py` depends on `subprocess.Popen` for startup wait (`process.poll()` at ~line 2507), port readiness detection, and death detection. The same-session tmux launcher removes the `Popen` handle entirely.

#### Examples

- After launching the gateway in a tmux window, `_wait_for_gateway_endpoint` needs to detect when the process has started and bound its port, without being able to call `process.poll()`.
- During `detach_gateway`, the runtime currently relies on `process.terminate()` or `process.kill()` — neither is available for a tmux-hosted process.

#### Why this question arises in code

- `runtime.py:2477` creates `subprocess.Popen` for the gateway.
- `runtime.py:2507` uses `process.poll()` in the startup wait loop.
- `runtime.py:2560` stores the process handle for later lifecycle management.
- None of these patterns work when the gateway is a foreground process in a tmux window.

#### Options

**Option A: Use gateway HTTP health endpoint as the sole liveness signal**

- Code impact:
  - Replace `process.poll()` with HTTP health polling against the gateway's advertised port.
  - For termination, send `C-c` or `kill` via tmux `send-keys` to the auxiliary window's pane.
  - Store the auxiliary window's `window_id` and `pane_pid` (from `list-panes`) instead of a `Popen` handle.
- User-facing difference:
  - Slightly slower death detection (HTTP timeout instead of instant `poll()`).
- Pros:
  - Transport-agnostic — works for both detached and tmux-hosted gateways.
  - Gateway already exposes `GET /v1/health`.
- Cons:
  - HTTP health polling adds latency to death detection.
  - Startup wait is trickier: must poll HTTP until the port opens, rather than checking that the process hasn't exited yet.

**Option B: Capture `pane_pid` from the tmux auxiliary window and use `os.kill(pid, 0)` for liveness checks**

- Code impact:
  - After creating the auxiliary tmux window, read `pane_pid` from `list-panes`.
  - Use `os.kill(pid, 0)` (signal 0 = existence check) instead of `process.poll()`.
  - For termination, use `os.kill(pid, signal.SIGTERM)`.
- User-facing difference:
  - Near-instant liveness detection, similar to current `poll()` behavior.
- Pros:
  - Direct PID-level liveness check, fast and reliable.
  - Keeps the existing polling-loop structure with minimal changes.
- Cons:
  - PID reuse risk (very unlikely but theoretically possible).
  - Requires reading `pane_pid` from tmux, adding a tmux dependency to the liveness path.

> **PROPOSED: Use Option B (capture `pane_pid` and use `os.kill(pid, 0)`) for liveness checks, combined with HTTP health for port-readiness.**
> Rationale: The existing `list_tmux_panes` in `tmux_runtime.py:279–316` already extracts `pane_pid` as an optional field on `TmuxPaneRecord`. Using PID-based liveness keeps the startup-wait loop structure nearly identical to the current `Popen`-based flow (replace `process.poll()` with `os.kill(pid, 0)` and `ProcessLookupError` catch). Port-readiness still requires HTTP polling, which `_wait_for_gateway_endpoint` already performs. For termination, `os.kill(pid, signal.SIGTERM)` is reliable and avoids the semantic indirection of tmux `send-keys C-c`. The PID-reuse risk is negligible for the short-lived startup-wait window.
>
> **DECISION: Refine the proposal into a hybrid: use tmux-owned auxiliary-window identity plus `pane_pid`/`pane_dead` for local liveness, keep HTTP health for readiness, and keep teardown window-scoped rather than raw-`Popen`-scoped.**
> Rationale: The review correctly identifies the missing lifecycle contract, but a pure `os.kill(pid, 0)` / `SIGTERM` replacement is too process-centric for a tmux-owned sidecar window. The repo already exposes `pane_pid` in `list_tmux_panes(...)`, and tmux-window teardown is the better ownership boundary for auxiliary processes that are intentionally hosted inside the session.

### Q3) Should `resolve_active_pane_id` helpers be refactored to a shared utility, or should each call site independently switch to explicit agent-surface targeting?

- Decision: `Deferrable`

#### Why this matters

- At least 6 call sites across `demo/`, `explore/`, and `server/tui/` implement the same `resolve_active_pane_id` pattern. Fixing them individually risks inconsistency and duplication.

#### Examples

- `demo/shared_tui_tracking_demo_pack/tooling.py:138` defines `resolve_active_pane_id` locally.
- `explore/claude_code_state_tracking/runtime_support.py:40` defines an identical copy.
- `server/tui/transport.py:53` uses `_prefer_active_pane` with similar semantics.

#### Why this question arises in code

- The three independent implementations have identical logic (iterate panes, return first active).
- Task 3.1 says "Update repo-owned tmux observers" but doesn't specify whether to consolidate or update each independently.

#### Options

**Option A: Add a `resolve_agent_surface_pane` function to `tmux_runtime.py` that targets window `0` / `agent` explicitly**

- Code impact:
  - New function in `tmux_runtime.py` that filters panes to window `0` or window name `agent`, then returns the pane.
  - All 6+ call sites import and use the shared function.
- User-facing difference:
  - None — behavior is the same, just correct for auxiliary-window scenarios.
- Pros:
  - Single source of truth for agent-surface resolution.
  - Easy to test in isolation.
- Cons:
  - Adds a cross-cutting dependency from demo/explore code to `tmux_runtime`.

**Option B: Update each call site independently to filter by window name**

- Code impact:
  - Each `resolve_active_pane_id` call site adds a `window_name="agent"` or `window_index="0"` filter.
- User-facing difference:
  - None.
- Pros:
  - No new shared function; each module stays self-contained.
- Cons:
  - Duplicated filtering logic; risk of inconsistent behavior across modules.

> **PROPOSED: Use Option A (shared `resolve_agent_surface_pane` in `tmux_runtime.py`) since the codebase already centralizes tmux primitives there.**
> Rationale: `tmux_runtime.py` already exports `headless_agent_window_target`, `headless_agent_pane_target`, `list_tmux_panes`, and the `HEADLESS_AGENT_WINDOW_INDEX`/`HEADLESS_AGENT_WINDOW_NAME` constants. Adding a `resolve_agent_surface_pane` function that uses these constants to filter `list_tmux_panes` output follows the existing centralization pattern. The demo and explore modules already import from `tmux_runtime` (e.g., `headless_runner.py:20` imports `HEADLESS_AGENT_WINDOW_NAME`), so the cross-cutting dependency is already established.
>
> **DECISION: Accept the shared-utility direction.**
> Rationale: The helper duplication is real across demo and explore flows, and `tmux_runtime.py` already owns the stable headless agent-surface constants and pane-listing primitives. The later artifact revision should add one explicit agent-surface resolver and move repo-owned observers onto it instead of re-implementing active-pane scanning per site.
