## Summary

- The proposal-to-design-to-specs-to-tasks chain is internally coherent and well-ordered; the migration plan is sensible (model first, CLI, resolution, fixtures/docs, verification).
- The preset schema, CLI surface, and launch selector specs are thorough with concrete scenarios covering the important paths including uniqueness rejection, overlay resolution modes, and role deletion guards.

## Findings

### Must Fix / Blocking

1. **No spec covers `brains build --preset <bare-name>` resolution semantics.**
   Design Decision 4 states that `brains build --preset <value>` accepts either an explicit preset path or a bare preset name. Task 3.1 implements this behavior. However, no spec defines the testable requirements: how a bare name maps to `agents/presets/<name>.yaml`, what error occurs when the name doesn't resolve, or whether resolution searches only the agent definition directory or also the project overlay. The `houmao-mgr-agents-launch/spec.md` covers only `agents launch` selectors, and `component-agent-construction/spec.md` covers the data model but not the `brains build` CLI surface. This leaves a spec-driven task (3.1) with no backing spec for its bare-name acceptance behavior.

> **DECISION: Accept.**
> Rationale: This is a real spec gap. The change artifacts explicitly intend new bare-name behavior for `brains build --preset`, but the current delta set does not add a testable requirement for that CLI surface. The follow-up should add an explicit `brains build` requirement and scenarios for bare-name resolution plus name-not-found failure.

### Should Fix / Important

2. **Preset name validation constraints are unspecified.**
   The `component-agent-construction` spec says "preset name SHALL be derived from the preset filename stem" but does not constrain valid preset names. Since preset names become first-class identifiers in CLI commands (`--name <preset>`), catalog keys, and display surfaces, undefined naming rules risk inconsistencies. Questions: Are spaces allowed? Uppercase? Unicode? Dots? What is the maximum length? The existing `roles` and `setup_profiles` in the catalog don't appear to enforce name constraints either, but presets are a new first-class resource — this is an opportunity to define the contract early.

> **DECISION: Refine.**
> Rationale: The reviewer is right that preset names become first-class identifiers, so some naming contract should be made explicit. However, this change does not need a full max-length, Unicode, and regex policy. A narrower v1 contract is enough: preset names should be non-empty, path-safe single-file identifiers with exact CLI matching and no path-separator ambiguity.

3. **Catalog schema migration strategy is missing.**
   The `presets` table at `src/houmao/project/catalog.py:996-1007` currently has no `name` column; it uses `UNIQUE(role_id, tool, setup_profile_id)` as its identity. Task 1.2 adds a first-class preset name to the catalog, which requires a schema change. Existing project overlays with populated catalog databases will break on the schema mismatch. The design rejects file-layout backward compatibility, which is fine, but doesn't state whether existing catalogs should be dropped-and-rebuilt or migrated. The `specialists` table also stores `preset_id` references. A schema version bump with a clear "drop and re-project" or "ALTER TABLE" strategy should be stated.

> **DECISION: Refine.**
> Rationale: The schema-migration gap is real, but the right artifact change is narrower than the review's full option set. This refactor already chooses a breaking direct rewrite posture, so the follow-up should state an explicit catalog schema bump plus fail-fast rebuild/re-project semantics for old overlays rather than implying a long-lived in-place `ALTER TABLE` migration promise.

4. **Task 2.2 description understates scope — `roles set` is an addition, not just a narrowing.**
   The current project CLI (at `src/houmao/srv_ctrl/commands/project.py`) exposes `roles list|get|init|scaffold|remove` — there is no `set` verb for roles today. Task 2.2 says "Narrow `houmao-mgr project agents roles` to prompt-only `list|get|init|set|remove` behavior," which reads as purely subtractive (remove scaffold, remove presets). In reality it is both subtractive (remove scaffold + nested presets) and additive (add `set` with `--system-prompt`, `--system-prompt-file`, `--clear-system-prompt`). The task description should acknowledge the addition to avoid implementer confusion about whether `set` is pre-existing.

> **DECISION: Accept.**
> Rationale: The current task wording reads as though task 2.2 only removes behavior. In fact it also introduces `roles set`, which is a meaningful implementation addition and should be called out explicitly to keep the task list accurate for apply work.

5. **`SpecialistCatalogEntry.resolved_preset_path()` update is under-scoped in task 3.3.**
   Task 3.3 says "Update manifest/reporting surfaces that expose preset paths." However, `SpecialistCatalogEntry.resolved_preset_path()` (in `catalog.py`) is not merely a reporting surface — it is used by `materialize_projection()` to write preset files to disk. This method constructs `roles/{role_name}/presets/{tool}/{setup_name}.yaml` paths. It belongs in the catalog/projection layer (task 1.2) rather than in a manifest/reporting task. If the implementer treats 3.3 as display-only, the projection path won't be updated until someone notices files landing in the old location.

> **DECISION: Accept.**
> Rationale: `resolved_preset_path()` is part of the catalog/projection seam, not just reporting. The reviewer correctly identified that leaving it under a display-oriented task risks missing the core projection rewrite from `roles/.../presets/...` to `presets/<name>.yaml`.

6. **`load_agent_catalog()` scan pattern not explicitly called out.**
   `load_agent_catalog()` in `definition_parser.py` currently scans `roles/*/presets/<tool>/*.yaml` recursively. After the refactor it must scan `presets/*.yaml` instead. This is implicitly covered by tasks 1.1 and 1.3 but is not stated as a concrete deliverable in any task. Since this function is the entry point for all catalog loading (used by both launch resolution and the `brains` command), an incorrect scan pattern would silently produce empty catalogs.

> **DECISION: Refine.**
> Rationale: `load_agent_catalog()` is absolutely a critical implementation hotspot, but it does not need its own standalone task. The better follow-up is to name it explicitly under the parser/model work in tasks 1.1 or 1.3 so the catalog scan-root rewrite is impossible to overlook.

## Design Alignment With Existing System

- **Overlay resolution pattern**: The `houmao-mgr-project-cli` spec correctly classifies new `presets add` and `presets set` as creating-path commands and `presets list`, `presets get`, `presets remove` as non-creating-path commands, matching the existing `_ensure_project_overlay` / `_resolve_existing_project_overlay` split in `src/houmao/srv_ctrl/commands/project.py`.
- **YAML rendering pipeline**: The existing `_render_preset_yaml()` in `catalog.py:1117-1134` renders `skills`, `auth`, `launch`, `mailbox`, and `extra` fields. The proposed preset schema adds `role`, `tool`, and `setup` as top-level required fields. This is a straightforward extension of the existing renderer; no structural change to the rendering pipeline is needed.
- **Uniqueness constraint**: The catalog already enforces `UNIQUE(role_id, tool, setup_profile_id)` on the `presets` table. The proposed design preserves this constraint and adds a preset name as a new identity axis. This is additive to the existing schema, not a redesign.
- **Provider-to-tool mapping**: The existing `tool_for_provider()` in `native_launch_resolver.py:11-15` maps `claude_code` → `claude`, `codex` → `codex`, `gemini_cli` → `gemini`. The proposed bare-role launch resolution depends on this same mapping. No change to the mapping is required.
- **Unknown-field rejection**: The existing parser already rejects unknown top-level preset fields. The proposed design adds `role`, `tool`, and `setup` to the accepted set, which is consistent with the existing validation pattern.
- **`AgentPreset` dataclass**: Currently has `path`, `role_name`, `tool`, `setup` (derived from path), plus content fields. The refactor changes the derivation source (content instead of path) and adds a `name` field. The dataclass shape is otherwise preserved, which minimizes downstream breakage.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should `brains build --preset <bare-name>` resolve the name to a file?

- Decision: `Blocking`

#### Why this matters

- `brains build --preset` is the build-phase entry point. If bare-name resolution works differently than launch-phase resolution, operators will get inconsistent behavior between `brains build --preset researcher-codex-default` and `agents launch --agents researcher --provider codex`.

#### Examples

- `pixi run houmao-mgr brains build --preset researcher-codex-default` — should this look in the agent definition directory only, or also the project overlay?
- `pixi run houmao-mgr brains build --preset agents/presets/researcher-codex-default.yaml` — explicit path, unambiguous.

#### Why this question arises in code

- `brains.py:97` currently calls `_resolve_path(preset, base=resolved_agent_def_dir)` which only resolves against the agent definition directory.
- Launch resolution in `native_launch_resolver.py:61-105` scans all preset files under the agent definition directory.
- The design doesn't specify whether `brains build` should search the project overlay or only the agent-def-dir when given a bare name.

#### Options

**Option A: Resolve bare name against agent-def-dir only — `<agent_def_dir>/presets/<name>.yaml`**

- Code impact:
  - `_resolve_path()` in `brains.py` adds a check: if `preset` has no path separators and no `.yaml` suffix, prepend `presets/` and append `.yaml`, then resolve against `agent_def_dir`.
  ```python
  if "/" not in preset and not preset.endswith((".yaml", ".yml")):
      preset = f"presets/{preset}.yaml"
  resolved = _resolve_path(preset, base=resolved_agent_def_dir)
  ```
- User-facing difference:
  - Consistent with current behavior where `brains build` always resolves against agent-def-dir.
  - Project-overlay-only presets are not reachable by bare name via `brains build`.
- Pros:
  - Simple, predictable resolution scope.
  - Matches the existing `brains build` resolution contract.
- Cons:
  - Project-overlay presets require explicit paths.

**Option B: Search both agent-def-dir and project overlay for the bare name**

- Code impact:
  - `brains build` needs project-overlay awareness, adding a dependency on overlay resolution that doesn't exist today.
  ```python
  candidates = [agent_def_dir / "presets" / f"{name}.yaml"]
  if overlay:
      candidates.append(overlay / "agents" / "presets" / f"{name}.yaml")
  ```
- User-facing difference:
  - Bare names work regardless of where the preset lives.
- Pros:
  - Uniform bare-name resolution across build and launch paths.
- Cons:
  - Adds overlay resolution dependency to `brains build`, which currently doesn't need it.
  - Introduces precedence ambiguity if the same name exists in both locations.

> **PROPOSED: Option A — resolve bare name against agent-def-dir only.**
> Rationale: `brains build` currently resolves all paths against `agent_def_dir` (see `brains.py:97`). Adding project-overlay awareness would change the command's dependency footprint for a marginal convenience gain. Operators working with project-overlay presets can pass explicit paths. This keeps `brains build` as a single-source-tree tool, consistent with its existing contract.
>
> **DECISION: Accept PROPOSED (Option A).**
> Rationale: `brains build` already resolves the effective agent-definition root through `resolve_materialized_project_aware_agent_def_dir()` in [`brains.py`](/data1/huangzhe/code/houmao/src/houmao/srv_ctrl/commands/brains.py), so resolving a bare preset name under `<agent_def_dir>/presets/<name>.yaml` still works for an active project overlay when that overlay is the selected agent-definition root. That keeps one resolution base, avoids a second overlay-search mechanism, and avoids precedence ambiguity. The artifact gap is missing spec text for this behavior, not a need for broader search scope.

### Q2) Should the catalog `presets` table store an explicit `name` column or derive the name from projection?

- Decision: `Deferrable`

#### Why this matters

- The design says "the catalog schema will treat preset name as a first-class identity" but doesn't specify whether `name` is a stored column or computed from `(role, tool, setup)` or from the ingestion source path. This affects migration, uniqueness enforcement, and specialist display.

#### Examples

- If `name` is stored: `SELECT name FROM presets WHERE role_id = ?` returns the user-chosen name.
- If `name` is derived: `SELECT role.name || '-' || tool || '-' || setup.name FROM presets ...` returns a synthetic name, which may not match the user's original filename.

#### Why this question arises in code

- The current `presets` table (`catalog.py:996-1007`) has no `name` column.
- `SpecialistCatalogEntry` already has a `name` field, but that's the specialist name, not the preset name.
- Projection currently writes to `roles/{role}/presets/{tool}/{setup}.yaml` — the filename is derived, not stored.

#### Options

**Option A: Add an explicit `name TEXT UNIQUE` column to the `presets` table**

- Code impact:
  - `_upsert_preset()` gains a `name` parameter.
  - Schema version bumps; existing DBs need migration or rebuild.
  ```sql
  CREATE TABLE presets (
      id INTEGER PRIMARY KEY,
      name TEXT NOT NULL UNIQUE,
      role_id INTEGER NOT NULL REFERENCES roles(id),
      ...
  )
  ```
- User-facing difference:
  - Preset names survive catalog round-trips faithfully.
- Pros:
  - Clean identity; no derivation drift.
  - Enables `presets get --name <name>` against catalog without scanning files.
- Cons:
  - Requires schema migration for existing catalogs.

**Option B: Derive name from `(role, tool, setup)` at display time**

- Code impact:
  - No schema change to `presets` table.
  - A helper function computes `f"{role}-{tool}-{setup}"` for display.
- User-facing difference:
  - Preset names in catalog views are synthetic; user-chosen filenames are not preserved through catalog operations.
- Pros:
  - No schema migration needed.
- Cons:
  - Synthetic names may not match user-chosen filenames.
  - Cannot distinguish presets with the same `(role, tool, setup)` by name (though this is prevented by uniqueness).

> **PROPOSED: Option A — add an explicit `name TEXT UNIQUE` column.**
> Rationale: The design treats preset name as a first-class identity. The catalog is the source of truth for project-local state, so the name must be stored there to survive round-trips. Derivation would lose the user's chosen name. The schema version can be bumped, and existing catalogs can be dropped-and-rebuilt since the project is under active development and explicitly allows breaking changes (per CLAUDE.md: "Breaking changes are allowed").
>
> **DECISION: Accept PROPOSED (Option A).**
> Rationale: The design says preset name becomes first-class identity, and the catalog is the project-local source of truth. Deriving a synthetic name from `(role, tool, setup)` would lose the user-chosen filename and weaken `project agents presets get --name <preset>` as a stable catalog-backed operation. A schema bump is acceptable here because this change already adopts a breaking rewrite posture.
