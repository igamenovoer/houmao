## Summary

- **Solid architecture:** The recipe-first startup model is a clean generalization that aligns with the existing `BrainRecipe` dataclass (`src/gig_agents/agents/brain_builder.py:48-54`), the `load_brain_recipe_from_path` loader (`src/gig_agents/agents/brain_launch_runtime/loaders.py`), and the `ShadowParserStack(tool=...)` adapter (`src/gig_agents/agents/brain_launch_runtime/backends/shadow_parser_stack.py`). The design correctly reuses all three rather than inventing parallel abstractions.
- **Well-justified breaking-change posture:** Opting for a clean recipe-first contract over compatibility shims is appropriate for this repo-owned operator surface, and the tasks explicitly include fixing all in-repo callers (task 3.4).
- **Risk: `default_agent_name` schema extension** touches the shared `BrainRecipe` dataclass and recipe-loading path used beyond the demo. The delta spec says "MAY declare" but the implementation path through `BrainRecipe` and its loader is not addressed in the tasks.
- **Risk: Recipe-to-`build-brain` invocation path is underspecified.** The design describes recipe-driven startup but does not commit to whether the demo loads the recipe and passes individual `--tool`/`--skill`/`--config-profile`/`--cred-profile` args or passes `--recipe <path>` directly to `brain_launch_runtime build-brain`.
- **Risk: Skill-list provenance is ambiguous.** The current demo hardcodes `DEFAULT_SKILLS = ("openspec-apply-change",)` while existing Codex recipes declare `skills: [openspec-apply-change, openspec-verify-change]`. The change does not specify whether recipes fully own the skill list or whether the demo retains override authority.

## Findings

### Must Fix / Blocking

1. **`BrainRecipe` dataclass and loader must accommodate `default_agent_name`.**
   The component-agent-construction delta spec adds an optional `default_agent_name` field to brain recipes. However, the existing `BrainRecipe` dataclass at `src/gig_agents/agents/brain_builder.py:48-54` is a frozen dataclass with exactly five fields (`name`, `tool`, `skills`, `config_profile`, `credential_profile`). The recipe loader at `src/gig_agents/agents/brain_launch_runtime/loaders.py:118-132` will fail on YAML files containing unrecognized keys unless the dataclass and loader are updated.

   **Impact:** The new Claude recipe and the updated Codex recipes (which must carry `default_agent_name` per spec) will be unloadable by the current loader. This is not covered by any task — task 1.1 mentions extending the "tracked demo brain-recipe schema and fixtures" but does not mention updating the shared `BrainRecipe` dataclass or `load_brain_recipe_from_path`.

   **Suggested fix:** Add a subtask under 1.1 (or a new task) that explicitly extends the `BrainRecipe` dataclass with `default_agent_name: str | None = None` and updates `load_brain_recipe_from_path` to accept and pass through the optional field without breaking existing callers that do not use it.

2. **Recipe invocation path through `build-brain` must be specified.**
   The current `_build_brain()` in `runtime.py:155-185` invokes `brain_launch_runtime build-brain` with individual `--tool`, `--skill`, `--config-profile`, `--cred-profile` arguments extracted from `DemoEnvironment` constants. The design says startup will "resolve through a brain recipe" but does not specify whether:
   - (A) the demo loads the recipe YAML, extracts fields, and passes them as individual CLI args (preserving current invocation shape), or
   - (B) the demo passes `--recipe <path>` to `build-brain` and lets the runtime handle recipe loading (if that flag exists).

   **Impact:** Option A keeps the demo as a recipe-aware consumer that manually extracts fields. Option B delegates recipe interpretation to the shared runtime. The choice affects error handling, layering, and whether `build-brain` needs a new `--recipe` flag. The design and specs must commit to one path so implementers don't make an ad-hoc choice.

### Should Fix / Important

3. **`DemoState` strict model will reject pre-existing state files.**
   `DemoState` at `models.py` uses `model_config = ConfigDict(extra="forbid")`. Adding `tool`, `variant_id`, and `brain_recipe` fields means any pre-existing `state.json` from a prior demo run will fail validation (missing required fields or, if made optional, the old files lack them). Since breaking changes are accepted, this is not blocking, but task 1.3 should note that stale state files from previous runs will become unreadable and the demo should handle this gracefully (e.g., clear stale state on startup).

4. **`variant_id` derivation rule needs to be explicit in the spec.**
   The design says `variant_id` is a "machine-friendly slug derived from the canonical recipe selector, for example `claude-gpu-kernel-coder-default`". The transformation appears to be `s|/|-|g` (slash to dash). The full-pipeline-demo spec (scenario: "Default startup uses the tracked Claude default recipe") says "a stable variant identifier derived from that canonical recipe selector" but does not define the derivation rule. This matters because:
   - Verification fixtures will key on `variant_id` (spec: "validates the report against the selected variant contract").
   - Test assertions need deterministic values.

   **Suggested fix:** Add a normalization rule to the full-pipeline-demo spec, e.g., "The `variant_id` SHALL be the canonical recipe selector with path separators replaced by hyphens."

5. **Skill-list provenance must be decided.**
   The current demo hardcodes `DEFAULT_SKILLS = ("openspec-apply-change",)` in `models.py:21`. The existing Codex recipes declare `skills: [openspec-apply-change, openspec-verify-change]`. The change does not specify:
   - Whether the recipe's `skills` list fully replaces `DEFAULT_SKILLS`, or
   - Whether the demo retains an override/merge mechanism.

   If recipes fully own the skill list, the Claude recipe must carry the correct skills and `DEFAULT_SKILLS` becomes dead code. If the demo overrides, the recipe's `skills` field is partly decorative for demo use. This should be specified in the full-pipeline-demo spec under the startup requirement.

6. **Inspect spec should specify the JSON field contract explicitly.**
   The inspect delta spec replaces `claude_code_state` with `tool_state` in human-readable output and mentions JSON output must match. However, it does not list the full JSON schema. The existing inspect JSON surface includes fields like `active`, `agent_identity`, `claude_code_state`, `tmux_target`, etc. The new spec adds `tool`, `variant_id`, and renames `claude_code_state` → `tool_state`, but the full expected JSON shape is not enumerated. Since downstream consumers (including `scripts/demo/cao-interactive-full-pipeline-demo/scripts/verify_report.py`) parse this JSON, the spec should enumerate the top-level JSON fields.

### Nice To Have

7. **README spec could reference the disambiguation error UX.**
   The operator-workflow spec says the README should document "when subdirectory context is required" for ambiguous basenames. It would be helpful to also specify what the error message should look like (e.g., "Multiple recipes match 'gpu-kernel-coder-default': claude/gpu-kernel-coder-default, codex/gpu-kernel-coder-default. Use a subdirectory selector to disambiguate."). This improves operator experience and makes the error testable.

8. **Consider documenting recipe schema version compatibility.**
   Adding `default_agent_name` to recipes while keeping `schema_version: 1` means existing and new recipes share the same version number despite different field sets. If the recipe schema is expected to evolve further, consider bumping to `schema_version: 2` or documenting that version 1 tolerates optional fields.

9. **Task ordering could make the dependency on `BrainRecipe` changes explicit.**
   Task 1.1 says "extend the tracked demo brain-recipe schema and fixtures" but the `BrainRecipe` dataclass lives in the shared `brain_builder.py` module, not the fixture layer. Making task 1.1 explicitly depend on (or include) updating the shared dataclass would prevent an implementer from only updating YAML fixtures and discovering the loader failure later.

## Design Alignment With Existing System

- **Existing pattern: `BrainRecipe` dataclass + YAML loader** (`src/gig_agents/agents/brain_builder.py:48-54`, `src/gig_agents/agents/brain_launch_runtime/loaders.py:118-132`): The proposed change correctly extends this pattern by adding a tracked Claude recipe and an optional `default_agent_name` field. The extension is additive and the "MAY declare" semantics align with the frozen-dataclass pattern (use a default of `None`).

- **Existing pattern: `ShadowParserStack(tool=...)` tool-aware adapter** (`src/gig_agents/agents/brain_launch_runtime/backends/shadow_parser_stack.py`): The proposed inspect generalization correctly delegates to this existing adapter instead of introducing demo-local parser branching. The current `_best_effort_output_text_tail` in `runtime.py` hardcodes `ClaudeCodeShadowParser`; the change correctly proposes routing through `ShadowParserStack(tool=persisted_tool)` instead.

- **Existing pattern: Fixed agent-definition directory layout** (`tests/fixtures/agents/brains/brain-recipes/{codex/,}`): The proposed recipe resolution path (`brains/brain-recipes/<selector>` under `env.agent_def_dir`) matches the existing fixture layout exactly. Adding `claude/gpu-kernel-coder-default.yaml` alongside the existing `codex/` subdirectory is the natural extension.

- **Existing pattern: `brain_launch_runtime` CLI as the build/launch interface** (`runtime.py:_runtime_cli_command`): The demo delegates to `pixi run python -m gig_agents.agents.brain_launch_runtime` for `build-brain`, `start-session`, and `stop-session`. The change preserves this delegation pattern rather than having the demo engine import brain-building internals directly.

- **Existing pattern: Strict Pydantic state models** (`models.py:DemoState` with `extra="forbid"`): Adding new fields to `DemoState` is consistent with the strict-model pattern. The change should ensure new fields are required (not optional) to maintain the "all fields present or fail" contract that the strict model enforces.

- **Potential misalignment: Recipe loading location.** The `BrainRecipe` dataclass and `load_brain_recipe_from_path` live in the shared `brain_builder`/`brain_launch_runtime` layer. The demo engine currently does not import from that layer for recipe loading (it only invokes the CLI). If the demo needs to load recipes to extract `default_agent_name` and `tool`, it will need to either (a) import `load_brain_recipe_from_path` directly, or (b) extend the `build-brain` CLI to accept `--recipe` and return the resolved metadata. The design should clarify which path to follow so the demo engine's dependency surface is intentional.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the demo load recipe YAML directly or pass `--recipe <path>` to `brain_launch_runtime build-brain`?

- Decision: `Blocking`

#### Why this matters

- The demo needs recipe metadata (at minimum `tool` and `default_agent_name`) at startup time, before `build-brain` returns. If recipes are loaded by the demo, the demo gains a direct Python dependency on the recipe loader. If recipes are loaded by `build-brain --recipe`, the demo stays CLI-only but needs the CLI to return recipe metadata in its output.

#### Examples

- Operator runs `run_demo.sh start --brain-recipe codex/gpu-kernel-coder-default`. The demo must know `tool=codex` and `default_agent_name=<value>` before invoking `build-brain` to emit the correct progress messages and persist the correct state.
- If `build-brain` is extended with `--recipe`, the demo passes one flag and gets back a manifest that includes tool and recipe metadata. If not, the demo must resolve the recipe path, load the YAML, extract fields, and pass them individually.

#### Why this question arises in code

- `runtime.py:_build_brain` currently passes `--tool`, `--skill`, `--config-profile`, `--cred-profile` individually:
  ```python
  command = _runtime_cli_command([
      "build-brain",
      "--agent-def-dir", str(env.agent_def_dir),
      "--runtime-root", str(paths.runtime_root),
      "--tool", DEFAULT_TOOL_NAME,  # "claude"
      *sum([["--skill", skill] for skill in env.skills], []),
      "--config-profile", env.config_profile,
      "--cred-profile", env.credential_profile,
  ])
  ```
- The demo engine currently has zero Python imports from `brain_builder` or `brain_launch_runtime.loaders`. Adding a direct import changes the dependency surface.
- The `brain_launch_runtime` CLI may or may not already support `--recipe` (design decision 2 references "the existing `build-brain --recipe` workflow" but the current code was not confirmed to have this flag).

#### Options

**Option A: Demo loads recipe YAML directly via `load_brain_recipe_from_path`**

- Code impact:
  - Demo gains `from gig_agents.agents.brain_launch_runtime.loaders import load_brain_recipe_from_path`
  - Demo resolves recipe path under `env.agent_def_dir / "brains" / "brain-recipes" / <selector>`
  - Demo extracts `tool`, `skills`, `config_profile`, `credential_profile`, `default_agent_name` from loaded recipe
  - Demo passes extracted fields as individual args to `build-brain` (same CLI shape as today)
  ```python
  recipe = load_brain_recipe_from_path(resolved_recipe_path)
  command = _runtime_cli_command([
      "build-brain",
      "--tool", recipe.tool,
      *sum([["--skill", s] for s in recipe.skills], []),
      "--config-profile", recipe.config_profile,
      "--cred-profile", recipe.credential_profile,
  ])
  ```
- User-facing difference:
  - No change to `build-brain` CLI contract
  - Demo owns recipe resolution and disambiguation logic
- Pros:
  - No changes needed to `brain_launch_runtime` CLI
  - Demo has full control over recipe resolution, disambiguation, and error messages
  - `default_agent_name` available immediately without CLI round-trip
- Cons:
  - Demo gains a Python-import dependency on the recipe loader
  - Recipe resolution logic (basename lookup, disambiguation) lives in the demo instead of the shared layer

**Option B: Extend `build-brain` with `--recipe <path>` and have it return recipe metadata**

- Code impact:
  - `brain_launch_runtime build-brain` gains `--recipe <path>` flag
  - CLI loads recipe, builds brain, and returns manifest including `tool`, `default_agent_name`
  - Demo passes `--recipe <resolved-path>` instead of individual fields
  ```python
  command = _runtime_cli_command([
      "build-brain",
      "--agent-def-dir", str(env.agent_def_dir),
      "--runtime-root", str(paths.runtime_root),
      "--recipe", str(resolved_recipe_path),
  ])
  ```
- User-facing difference:
  - `build-brain` becomes recipe-aware at the CLI level
  - Demo still needs to resolve the recipe path (for disambiguation) but delegates loading to the CLI
- Pros:
  - Demo stays CLI-only with no Python imports from brain_launch_runtime
  - `build-brain --recipe` is reusable beyond the demo
- Cons:
  - Requires extending `brain_launch_runtime` CLI and its output contract
  - Demo still must resolve the recipe path for disambiguation before passing it
  - `default_agent_name` only available after the `build-brain` round-trip unless the demo also pre-loads the YAML

> **PROPOSED: Option A — Demo loads recipe YAML directly via `load_brain_recipe_from_path`.**
> Rationale: The demo already depends on `gig_agents.agents.brain_launch_runtime` as a subprocess CLI target, so adding a Python import from the same package is a minor, well-scoped dependency increase. The demo needs `tool` and `default_agent_name` before invoking `build-brain` (to emit tool-specific progress messages at `commands.py:93-95` and to resolve the agent name for canonicalization). Pre-loading the recipe avoids a round-trip and keeps all resolution/disambiguation logic in one place (the demo engine). Extending `build-brain --recipe` is valuable but orthogonal and can be done later.
>
> **DECISION: Hybrid — the demo will resolve and load the selected recipe for metadata, and it will pass `--recipe <resolved-path>` to `build-brain` for the actual build.**
> Rationale: The review correctly identified that startup needs recipe metadata early, but its framing missed that `brain_launch_runtime build-brain` already supports `--recipe` today at [src/gig_agents/agents/brain_launch_runtime/cli.py](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/agents/brain_launch_runtime/cli.py). The most native path is therefore to keep recipe resolution, selector normalization, `tool`, and `default_agent_name` handling in the demo layer, while delegating the actual brain construction to the existing CLI recipe path instead of re-expanding recipe fields into `--tool`/`--skill`/`--config-profile`/`--cred-profile`. This keeps startup messages and agent-name selection deterministic without duplicating the shared build composition logic.

### Q2) Should recipes fully own the skill list, or should the demo retain an override/merge mechanism?

- Decision: `Deferrable`

#### Why this matters

- The current demo hardcodes `DEFAULT_SKILLS = ("openspec-apply-change",)` in `models.py:21`. The existing Codex recipes declare two skills (`openspec-apply-change`, `openspec-verify-change`). If recipes fully own the skill list, the new Claude recipe must carry the correct skills and `DEFAULT_SKILLS` becomes dead code. If the demo retains override authority, the recipe's skills field is partly ignored for demo launches.

#### Examples

- Developer creates a new recipe with `skills: [openspec-apply-change, openspec-verify-change, custom-skill]`. Does the demo honor all three skills, or does it override with its own `DEFAULT_SKILLS`?
- The existing Codex recipe `gpu-kernel-coder-default.yaml` has two skills. If the demo overrides, a Codex launch would unexpectedly lose `openspec-verify-change`.

#### Why this question arises in code

- `models.py:21`: `DEFAULT_SKILLS: tuple[str, ...] = ("openspec-apply-change",)`
- `runtime.py:_build_brain`: passes `env.skills` (from `DemoEnvironment`) to `--skill` args
- `DemoEnvironment.skills` defaults to `DEFAULT_SKILLS` with no recipe-aware override path
- Codex recipe `gpu-kernel-coder-default.yaml`: `skills: [openspec-apply-change, openspec-verify-change]`

#### Options

**Option A: Recipes fully own the skill list**

- Code impact:
  - Remove `DEFAULT_SKILLS` from `models.py`
  - `_build_brain` passes `recipe.skills` directly
  - Claude recipe must declare correct skill set
- User-facing difference:
  - Skill set visible in recipe YAML; no hidden demo overrides
- Pros:
  - Single source of truth for skills per recipe
  - Recipe is fully declarative
- Cons:
  - Changing demo skills requires editing recipe YAML

**Option B: Demo retains a merge/override mechanism**

- Code impact:
  - `_build_brain` merges `recipe.skills` with demo defaults or overrides
  - `DEFAULT_SKILLS` or a per-demo config survives
- User-facing difference:
  - Skills come from two sources; harder to audit
- Pros:
  - Demo can customize skills without recipe changes
- Cons:
  - Two-source skill composition is harder to reason about
  - Contradicts the recipe-as-single-source-of-truth design goal

> **PROPOSED: Option A — Recipes fully own the skill list.**
> Rationale: Design decision 2 in `design.md` states "recipes become the single source of truth for tool, skill, config-profile, and credential-profile composition across the demo." Retaining a demo-level skill override would violate this stated goal. The existing Codex recipes already carry skills, and the new Claude recipe should do the same. `DEFAULT_SKILLS` in `models.py:21` becomes dead code and should be removed as part of task 1.5.
>
> **DECISION: Accept Option A — recipes will fully own the skill list.**
> Rationale: I agree with the reviewer here. Once startup is recipe-first, keeping [DEFAULT_SKILLS](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/src/gig_agents/demo/cao_interactive_demo/models.py) in parallel would weaken the whole “recipes are the source of truth” move and create hidden behavior differences between Claude and Codex. We should remove the demo-owned default skill list, add the correct skills to the new Claude recipe, and make the change spec say that recipe skills are authoritative for this demo.

### Q3) What `default_agent_name` values should the tracked recipes carry?

- Decision: `Deferrable`

#### Why this matters

- The component-agent-construction delta spec requires each tracked interactive-demo recipe to declare `default_agent_name`. The design says the demo will use this as the identity default when `--agent-name` is omitted. The choice of default names affects the operator experience, test fixtures, and the `launch_alice.sh` wrapper narrative.

#### Examples

- If `claude/gpu-kernel-coder-default` uses `default_agent_name: cao-interactive-demo` (matching today's `DEFAULT_AGENT_NAME`), the no-arg Claude path is identity-identical to the current behavior.
- If `codex/gpu-kernel-coder-default` uses `default_agent_name: cao-codex-demo`, operators see a different canonical identity for Codex runs.
- `launch_alice.sh` overrides with `--agent-name alice` regardless, so wrapper users are unaffected.

#### Why this question arises in code

- `models.py:11`: `DEFAULT_AGENT_NAME = "cao-interactive-demo"` is the current hardcoded default
- `launch_alice.sh`: always passes `--agent-name alice`, overriding the default
- No existing recipe carries `default_agent_name`
- Test fixtures (e.g., verification reports) assert on `agent_identity` values derived from the default name

#### Options

**Option A: All recipes use the same default name (e.g., `cao-interactive-demo`)**

- Code impact:
  - Minimal test fixture changes; current identity semantics preserved
- User-facing difference:
  - Claude and Codex runs share the same default identity unless overridden
- Pros:
  - Backward-compatible identity for no-arg Claude path
  - Simple
- Cons:
  - Cannot distinguish Claude vs Codex sessions by identity alone in logs/state

**Option B: Recipes use tool-specific default names (e.g., `cao-claude-demo`, `cao-codex-demo`)**

- Code impact:
  - Test fixtures and verification snapshots need updates for new identity values
  - State files carry tool-distinguishable identities
- User-facing difference:
  - Different tools produce different default canonical identities
- Pros:
  - Operator can distinguish tool sessions in tmux, logs, CAO server
  - Aligns with `variant_id` providing tool-specific provenance
- Cons:
  - Breaks backward compatibility for the no-arg Claude path identity
  - More test fixture updates

> **PROPOSED: Option A — All recipes use `cao-interactive-demo` as `default_agent_name`.**
> Rationale: The change already provides `variant_id` and `tool` in persisted state and reports for tool-level disambiguation. Using the same default agent name preserves backward compatibility for the no-arg Claude path (matching `DEFAULT_AGENT_NAME = "cao-interactive-demo"` at `models.py:11`) and minimizes test fixture churn. Tool-specific naming can be revisited once there is operator demand for identity-level disambiguation beyond what `variant_id` provides.
>
> **DECISION: Reject Option A; use tool-specific default names.**
> Rationale: The review's compatibility argument is no longer a priority for this change, because we explicitly decided not to preserve legacy call shapes when the repo can be fixed in one pass. Tool-specific default names make direct `run_demo.sh start` sessions easier to identify in tmux, CAO state, and logs before a user reaches the persisted `variant_id` surface. The tracked recipes should therefore carry explicit tool-level defaults, with [launch_alice.sh](/data/ssd1/huangzhe/code/agent-system-dissect/extern/tracked/gig-agents/scripts/demo/cao-interactive-full-pipeline-demo/launch_alice.sh) remaining only a convenience override path.
