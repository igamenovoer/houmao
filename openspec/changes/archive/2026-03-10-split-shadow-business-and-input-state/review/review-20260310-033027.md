## Summary

- **Solid problem statement.** The proposal correctly identifies that the current `activity + accepts_input` model forces a lossy collapse — the code in both `claude_code_shadow.py` (lines 366–377) and `codex_shadow.py` (lines 305–320) literally computes `accepts_input = ui_context == "normal_prompt"`, which means `accepts_input=False` conflates "working", "modal prompt", "trust gate", and "closed surface" into one boolean. The motivation is genuine and well-grounded.
- **Clean orthogonal decomposition.** The two-axis `business_state × input_mode` model is a good factoring that naturally represents the real TUI surface space. The design correctly identifies representative combinations and shows they cover existing provider states.
- **Spec coverage is thorough.** All four impacted specs have delta updates, scenarios are concrete and testable, and the REMOVED requirements document migration paths.
- **The `_TurnMonitor` state machine redesign is underspecified.** The design and specs define derived predicates but do not address the `_TurnMonitor`'s internal state transitions with sufficient precision. This is the main risk area.
- **`interactive_busy` predicate is defined but has no consumer.** The design introduces it but no spec or task references what runtime should *do* with it.
- **The unknown→stalled pathway has a gap under the new model.** The current `_is_unknown()` checks `activity == "unknown"`, but the new model splits that into two axes — neither spec nor design says what happens when only one axis is unknown.

## Findings

### Must Fix / Blocking

- **MF-1: `_TurnMonitor` state machine transitions are not specified for the new model.** The current `observe_completion()` (cao_rest.py lines 355–407) uses a specific cascade: check `activity == "waiting_user_answer"` → `activity == "working"` → `accepts_input` → else. The new model replaces both `activity` and `accepts_input` with two independent axes, but neither the design nor the brain-launch-runtime delta spec defines the new cascade order or how `_TurnMonitor` internal states (`submitted_waiting_activity`, `in_progress`, `completed`) map to the new predicate checks. The spec says *what predicates exist* but not *in what priority order the monitor evaluates them*. Since the current monitor has a deliberate priority order (blocked > working > ready-with-evidence > waiting), the new monitor needs the same level of specification.

  **Concrete gap:** What should `observe_completion()` do when it sees `business_state=working, input_mode=modal`? The current code would hit the `activity == "working"` branch and set `in_progress`. Under the new model, should `input_mode=modal` during `working` change anything? The specs are silent.

- **MF-2: `_is_unknown()` semantics are ambiguous under two-axis unknown.** The current `_is_unknown()` (cao_rest.py line 486–493) returns true when `availability == "unknown"` OR `(availability == "supported" AND activity == "unknown")`. Under the new model, there are *three* ways to be partially unknown:
  1. `business_state=unknown, input_mode=known` — e.g., can't tell if tool is working, but prompt is clearly freeform
  2. `business_state=known, input_mode=unknown` — e.g., tool is idle, but can't tell if prompt is freeform or modal
  3. `business_state=unknown, input_mode=unknown` — fully ambiguous

  The shared spec says `input_mode=unknown` by itself "keeps the surface non-ready without automatically forcing a stalled transition" (design.md line 97), but case (1) — `business_state=unknown` with known `input_mode` — is not addressed. Should it still trigger the unknown→stalled timer? The stall policy in `brain-launch-runtime` spec still references "unknown" generically without clarifying which axis triggers it.

- **MF-3: The `submit_ready` predicate does not account for `awaiting_operator` as a blocking override.** The design defines `submit_ready := availability == supported AND business_state == idle AND input_mode == freeform`. But in the current `observe_readiness()`, there's an explicit early-return for `waiting_user_answer` *before* the `accepts_input` check (cao_rest.py lines 349–351). The new spec says `awaiting_operator` should fail the turn — but the `submit_ready` predicate alone doesn't express this. A surface with `business_state=idle, input_mode=freeform` might transiently appear submit-ready before a slower-to-detect operator gate becomes visible. The runtime spec needs to specify that `awaiting_operator` detection is checked *independently of and with higher priority than* `submit_ready`, not just that it "SHALL be treated as blocked." The priority ordering matters for correctness.

### Should Fix / Important

- **SF-1: `interactive_busy` predicate is dead on arrival.** The design (line 67) defines `interactive_busy := availability == supported AND business_state == working AND input_mode == freeform`. No delta spec references it. No task implements it. No runtime behavior is specified for it. If the point of the change is to represent `working + freeform`, the system needs to actually *do something* when it observes that state — otherwise the new model adds representational power that no one consumes. The brain-launch-runtime spec says "a surface with `business_state = working` and `input_mode = freeform` SHALL remain in progress" — but that's the same as current `activity=working` behavior; the `input_mode=freeform` part adds no new behavior. Either remove `interactive_busy` from the design or specify what new runtime behavior it enables.

- **SF-2: Completion evidence model doesn't exploit the new axes.** The current completion check (cao_rest.py lines 391–397) is: `accepts_input AND (saw_working OR saw_projection_change) → completed`. The new spec restates this as `submit_ready AND (post-submit working OR projection change)`. But `submit_ready` is strictly *more* constrained than `accepts_input` (it additionally requires `business_state=idle` and `input_mode=freeform`). This means the new model could *delay* completion detection in cases where the provider returns to `accepts_input`-equivalent but with `input_mode=modal` temporarily (e.g., a flash of slash-command UI during recovery). This is probably correct behavior, but it's a subtle semantic change from the baseline that is nowhere called out as intentional. The design should explicitly state that completion now requires full `submit_ready` (not just `accepts_input`) and acknowledge the behavioral delta.

- **SF-3: The Codex and Claude delta specs are structurally identical — missing provider-differentiated mapping tables.** Both provider specs define the same four `business_state` values and four `input_mode` values with nearly identical scenario text. But the *evidence-to-axis mapping* differs significantly between providers. Claude uses spinner lines, trust prompts, and `❯` idle markers; Codex uses processing indicators, approval prompts, and different prompt markers. Neither delta spec includes a concrete mapping table from provider evidence → `(business_state, input_mode)`. The current code (claude_code_shadow.py lines 352–377, codex_shadow.py lines 295–320) has explicit if/elif cascades for this mapping. The specs should include equivalent mapping tables or at minimum scenario coverage for each provider evidence type, to prevent implementers from guessing.

- **SF-4: `ui_context` relationship to `input_mode` is underspecified for conflict resolution.** The design says `ui_context` answers "which UI shape is active" while `input_mode` answers "what sort of input is allowed." But these can conflict: a `trust_prompt` `ui_context` implies `input_mode=modal`, and a `normal_prompt` `ui_context` implies `input_mode=freeform`. What happens if evidence is ambiguous — e.g., partial trust prompt text visible alongside an idle prompt marker? Which axis wins? The current code resolves this via if/elif priority in the parser. The spec should state whether `ui_context` is *derived from the same evidence as* `input_mode` (making conflicts impossible) or whether they're independently classified (requiring a conflict resolution rule).

- **SF-5: The REMOVED `waiting_user_answer` requirement lacks a migration bridge for `waiting_user_answer_excerpt`.** Both Claude and Codex delta specs remove the `waiting_user_answer` requirement and say to migrate to `business_state=awaiting_operator` with `ui_context` + `waiting_user_answer_excerpt` for detail. But `waiting_user_answer_excerpt` is currently a field on `SurfaceAssessment` itself (shadow_parser_core.py line 76). Under the new model, is that field kept, renamed, or moved to provider subclasses? The shared parser spec delta doesn't mention it. If the field stays on the shared base, its name references a concept (`waiting_user_answer`) that no longer exists in the shared vocabulary. If it moves, the migration path in the Claude/Codex specs is incomplete.

### Nice To Have

- **NTH-1: The design's "Alternative considered" sections are useful but could cite code.** For instance, Decision 1 rejects "keep `activity` and add more booleans." It would be stronger if it pointed at the specific code (e.g., `_TurnMonitor.observe_completion()` lines 387–400) where the boolean-collapse causes the current complexity.

- **NTH-2: Task 1.2 is vague.** "Add shared helper predicates or helpers for derived submit readiness without reintroducing the old one-dimensional contract as the primary API" — this should specify whether the helpers are module-level functions, `SurfaceAssessment` methods/properties, or standalone predicate functions. The repo convention (frozen dataclasses for value objects, separate functions for logic) suggests module-level functions, but the task should be explicit.

- **NTH-3: Tasks don't specify a migration order for downstream callers.** If `SurfaceAssessment.activity` and `accepts_input` are removed, every consumer breaks simultaneously. Task 1.1 replaces the fields, task 3.1–3.3 updates `cao_rest.py`, but there could be other consumers (tests, manual scripts, debugging utilities). A brief audit step would reduce surprise breakage.

## Design Alignment With Existing System

- **Existing pattern observed:** `SurfaceAssessment` is a `@dataclass(frozen=True)` in `shadow_parser_core.py` (line 66). Provider extensions (`ClaudeSurfaceAssessment`, `CodexSurfaceAssessment`) subclass it and add `evidence` tuples. Runtime logic in `_TurnMonitor` consumes shared fields only. Provider parsers produce provider-specific subclass instances.

- **Proposed design alignment:** The new `business_state` and `input_mode` fields naturally fit as `Literal` types on the frozen dataclass, matching the existing `availability` and `activity` pattern. The derived predicates (`submit_ready`, `operator_blocked`) align with the existing pattern of module-level helper functions like `_surface_status_label()` (cao_rest.py). The `ui_context` remaining provider-extensible matches the current subclass pattern. **Overall alignment is strong.**

- **Mismatch risk:** The current `_TurnMonitor` (cao_rest.py lines 301–493) is a mutable `@dataclass` with explicit state machine transitions. The proposal's "derived predicates" language implies stateless computation, but completion detection *requires* stateful tracking (`saw_working_after_submit`, `saw_projection_change_after_submit`). The design should be explicit that the monitor remains stateful and the derived predicates are *inputs to* the state machine, not replacements for it.

## Open Questions (With Proposed Defaults)

### Q1) How should `_is_unknown()` behave when only one axis is unknown?

- Decision: `Blocking`

#### Why this matters

The unknown→stalled timer is a critical safety mechanism that prevents the runtime from waiting forever on an unclassifiable surface. Under the current single-axis model, "unknown" is unambiguous. Under two axes, partial-unknown creates three distinct cases with different risk profiles.

#### Examples

- Claude shows a spinner (→ `business_state=working`) but the parser can't determine if the prompt region is freeform or modal (→ `input_mode=unknown`). Should this trigger the stall timer? Probably not — we know the tool is working.
- Output matches a supported family but the parser can't determine business state (→ `business_state=unknown`) while the prompt is clearly freeform (→ `input_mode=freeform`). Should this trigger the stall timer? Probably yes — we don't know if it's safe to submit.

#### Why this question arises in code

- `_TurnMonitor._is_unknown()` at cao_rest.py:486–493 currently checks one axis
- The delta spec for `brain-launch-runtime` references unknown handling generically
- Design line 97 handles `input_mode=unknown` specifically but not `business_state=unknown` with known `input_mode`

```python
# Current:
@staticmethod
def _is_unknown(surface_assessment: SurfaceAssessment) -> bool:
    return surface_assessment.availability == "unknown" or (
        surface_assessment.availability == "supported"
        and surface_assessment.activity == "unknown"
    )
# Under new model: what about business_state=unknown + input_mode=freeform?
```

#### Options

**Option A: Stall timer triggers only when `business_state=unknown` (regardless of `input_mode`)**

- Code impact:
  - `_is_unknown()` becomes: `availability == "unknown" or (availability == "supported" and business_state == "unknown")`
  - `input_mode=unknown` alone never triggers the timer
- User-facing difference:
  - Surfaces where the tool's work-state is ambiguous get timeout protection
  - Surfaces where only the prompt shape is ambiguous stay non-ready without timeout pressure
- Pros:
  - Matches the current semantic: the stall timer guards against unclassifiable *tool activity*, not unclassifiable *input region*
  - `input_mode=unknown` is safely handled by `submit_ready` already returning false
- Cons:
  - A surface stuck in `business_state=working, input_mode=unknown` forever would never stall (but it wouldn't be submit-ready either, so the turn-level timeout would catch it)

**Option B: Stall timer triggers when *either* axis is unknown**

- Code impact:
  - `_is_unknown()` becomes: `availability == "unknown" or (availability == "supported" and (business_state == "unknown" or input_mode == "unknown"))`
- User-facing difference:
  - Any ambiguity triggers stall protection
- Pros:
  - Maximum safety; no ambiguous surface escapes the stall timer
- Cons:
  - Overly aggressive; `business_state=working, input_mode=unknown` would stall even though we know the tool is busy
  - Contradicts design.md line 97 which says `input_mode=unknown` should not force stalled transition

#### Proposal (Recommended)

Option A. Stall timer keys off `business_state=unknown` only. `input_mode=unknown` keeps the surface non-ready (via `submit_ready` failing) without stall-timer pressure.

#### Pros / Cons (Proposal)

- Pros:
  - Preserves the current semantic intent of the stall timer (guard against unclassifiable tool activity)
  - Consistent with design.md line 97
  - The turn-level timeout already provides a safety net for any permanently non-ready surface
- Cons:
  - Requires documenting that `input_mode=unknown` is not covered by stall diagnostics (minor visibility gap)

> **DECISION: Stall timer triggers on `business_state=unknown` only; `input_mode=unknown` keeps surface non-ready without stall escalation.**
> Rationale: The existing `_is_unknown()` (cao_rest.py:486–493) targets unclassifiable *activity*, not unclassifiable *input shape*. The stall timer's purpose is to detect tools that have gone silent or unrecognizable — that maps to `business_state`, not `input_mode`. `input_mode=unknown` is safely non-ready via the `submit_ready` predicate, and the outer turn-level deadline provides the ultimate timeout safety net. This also aligns with the design's own statement (design.md line 97) that `input_mode=unknown` by itself should not force a stalled transition.

### Q3) Should `observe_completion()` gate completion on full `submit_ready` or only on `accepts_input`-equivalent?

- Decision: `Blocking`

#### Why this matters

This is a subtle but real behavioral change. Current completion: `accepts_input == True` + evidence. Proposed completion: `submit_ready` (= `idle + freeform + supported`) + evidence. The difference: under the new model, a surface that returns to `input_mode=freeform` while `business_state` is transiently still `working` (tool finishing up) won't trigger completion. Under the old model, this same surface *would* have had `accepts_input=True` and triggered completion.

#### Examples

- Claude finishes a response, the prompt reappears (→ old `accepts_input=True`), but the spinner hasn't fully cleared yet (→ new `business_state=working`). Old model: completed. New model: waits. This is probably *more correct* but could add latency.
- Codex returns to prompt after processing, briefly shows a slash-command flash (→ `input_mode=modal` for one poll). Old model: `accepts_input=False`, waits one more poll. New model: same — non-ready. No change here.

#### Why this question arises in code

- `observe_completion()` at cao_rest.py:391 checks `surface_assessment.accepts_input` for completion gate
- The brain-launch-runtime delta spec says completion requires "return to the submit-ready surface"
- `submit_ready` is strictly more constrained than `accepts_input`

```python
# Current (cao_rest.py:391-397):
elif surface_assessment.accepts_input:
    if self.m_saw_projection_change_after_submit or self.m_saw_working_after_submit:
        self.m_state = "completed"

# New model equivalent:
elif is_submit_ready(surface_assessment):  # idle + freeform + supported
    if self.m_saw_projection_change_after_submit or self.m_saw_working_after_submit:
        self.m_state = "completed"
```

#### Options

**Option A: Completion gates on full `submit_ready`**

- Code impact:
  - Replace `accepts_input` check with `submit_ready` predicate in `observe_completion()`
  - The monitor won't complete until the tool is fully idle + freeform
- User-facing difference:
  - Slightly more conservative completion — no false-early completions during transient states
  - Possible added latency if `business_state` lags behind prompt appearance
- Pros:
  - Consistent with the new model's philosophy
  - Safer: tool is provably idle before completion fires
- Cons:
  - Could add 1-2 poll cycles of latency in normal cases
  - Makes completion dependent on `business_state=idle` detection accuracy

**Option B: Completion gates on `input_mode=freeform` + `availability=supported` (ignoring `business_state`)**

- Code impact:
  - Replace `accepts_input` with `input_mode == freeform and availability == supported`
  - Preserves current latency characteristics
- User-facing difference:
  - Completion fires as soon as the prompt is typeable, even if spinner hasn't cleared
- Pros:
  - Minimal behavioral change from baseline
  - Avoids coupling completion latency to `business_state` detection
- Cons:
  - Partially undermines the point of separating the axes — if completion ignores `business_state`, the split adds less value
  - Could complete while tool is still technically working

#### Proposal (Recommended)

Option A. Gate on full `submit_ready`. The whole point of the change is to stop the system from treating "typeable" as "ready." Completion should respect that principle. The potential added latency is bounded by poll interval (typically < 1s) and is the correct trade-off for accuracy.

#### Pros / Cons (Proposal)

- Pros:
  - Philosophically consistent with the change's thesis
  - Prevents premature completion on transient freeform surfaces
  - `submit_ready` is already defined and reusable
- Cons:
  - Requires confident `business_state=idle` detection; if a provider is slow to transition from `working → idle`, completion stalls

> **DECISION: Completion gates on full `submit_ready`, same as submission readiness.**
> Rationale: The central thesis of this change (design.md Decision 1) is that "typeable ≠ ready." If completion still gates on input-mode alone, that thesis is contradicted at the most critical lifecycle transition. The current `observe_completion()` (cao_rest.py:391) already implicitly requires idle-like state because `accepts_input` is only true when `activity=ready_for_input` in the current parsers (claude_code_shadow.py:371–372: `accepts_input = ui_context == "normal_prompt"` only when `has_idle_prompt`). So in practice, the behavioral delta is small — the new model just makes the requirement *explicit* rather than *implicit through parser collapse*.

## Suggested Next Steps

1) **Add a `_TurnMonitor` transition table to the design or the brain-launch-runtime spec.** Specify the priority order of predicate checks in both `observe_readiness()` and `observe_completion()` — at minimum: availability-failed > operator-blocked > unknown/stalled > working > submit-ready-with-evidence > else. This is the most critical gap (MF-1).

2) **Resolve the `_is_unknown()` two-axis semantics** (MF-2) by specifying in the shared parser spec or brain-launch-runtime spec that the stall timer keys off `business_state=unknown` only.

3) **Add priority ordering for `awaiting_operator` detection** (MF-3) — make the brain-launch-runtime spec explicitly state that `awaiting_operator` is checked *before* `submit_ready` in both readiness and completion paths.

4) **Either specify a runtime behavior for `interactive_busy` or remove it** from the design (SF-1). If the intent is "future-proofing," say so explicitly and note it's not consumed in this change.

5) **Add evidence→axis mapping tables** to the Claude and Codex delta specs (SF-3), or at minimum add scenario coverage for each major evidence type (spinner, idle prompt, trust/approval, slash-command, error banner) showing the expected `(business_state, input_mode)` output.

6) **Clarify whether completion gates on `submit_ready` vs `accepts_input`-equivalent** (Q3) and document the behavioral delta from baseline explicitly in the design.

7) **Rename `waiting_user_answer_excerpt` → `operator_blocked_excerpt`** in task 1.3 scope (Q2).
