## Summary

- **Solid:** The artifact suite is internally coherent and well-structured. The proposal, design, spec, and tasks all trace to the same scope with no contradictions. The decision to use runtime-owned `mail` commands rather than direct managed-script invocation is architecturally correct and consistent with `docs/reference/mailbox/quickstart.md`.
- **Solid:** The choice to keep implementation tutorial-pack-local (no new `src/houmao/demo/` subpackage) is appropriate given there is only one mailbox tutorial use case.
- **Solid:** The CAO-server-launcher tutorial-pack pattern (`scripts/demo/cao-server-launcher/`) is well-defined and provides a clear structural template for the new pack.
- **Risk:** The design underspecifies how two concurrent sessions will coexist — the existing demo patterns (`cao-server-launcher`, `cao-interactive-full-pipeline-demo`) run a single session at a time. Two concurrent mailbox-enabled sessions with the same backend have not been demonstrated in any existing tutorial pack, and cleanup ordering needs care.
- **Risk:** The change assumes `mail send` returns a `message_id` in its structured result, but the current `mail_commands.py` result contract is session-driven (the session generates the JSON payload, not the runtime). The actual shape of the `mail send` result depends on the agent executing the mailbox skill, which means the tutorial must defensively extract and validate `message_id` from a semi-structured response.
- **Gap:** The spec and tasks do not address which brain recipe(s) and backend(s) the tutorial will use. Existing recipes (`claude/gpu-kernel-coder-default`, etc.) have no `mailbox` field. The tutorial must either create dedicated mailbox-enabled recipes or pass mailbox flags entirely through `start-session` CLI overrides.

## Findings

### Must Fix / Blocking

1. **No mailbox-enabled brain recipe exists; tracked inputs are underspecified.**
   The spec requires tracked `inputs/demo_parameters.json` containing two-agent launch configurations, but does not specify where the mailbox config lives. The three existing brain recipes under `tests/fixtures/agents/brains/brain-recipes/` have zero `mailbox` fields. The `BrainRecipe` dataclass (`src/houmao/agents/brain_builder.py:61-68`) supports an optional `mailbox: MailboxDeclarativeConfig`, but no tracked recipe uses it.

   The implementation must decide: create new mailbox-enabled recipe YAML files under `tests/fixtures/agents/brains/brain-recipes/`, or rely entirely on `start-session` CLI flags (`--mailbox-transport`, `--mailbox-root`, `--mailbox-principal-id`, `--mailbox-address`). The demo parameters JSON and README must make this explicit.

   See **Open Question Q1** below.

2. **Concurrent two-session lifecycle is not addressed in the runner design.**
   The `cao-server-launcher` `run_demo.sh` runs a single linear flow (status/start/status/stop/status). The `cao-interactive-full-pipeline-demo` maintains exactly one session at a time via `state.json`. The new mailbox tutorial requires two concurrent sessions with distinct agent identities, both active simultaneously during the `mail send` → `mail check` → `mail reply` → `mail check` sequence.

   The tasks (2.1, 2.2) mention "two-agent mailbox-enabled startup" and "two-session cleanup" but do not address:
   - How two concurrent session states are tracked (two state files? a single manifest?).
   - How `run_demo.sh` addresses each session distinctly in follow-up `mail` commands.
   - What the cleanup order is when one session fails during startup.

   This needs explicit task detail before implementation.

### Should Fix / Important

3. **`message_id` extraction from `mail send` result needs defensive handling.**
   Design Decision 4 states the tutorial will "record the parent `message_id` from the `mail send` result." The runtime-owned `mail_commands.py` (`run_mail_prompt`, `parse_mail_result`) parses the agent's sentinel-delimited JSON response but does not enforce a specific `message_id` field in the result schema — it validates `request_id`, `operation`, `transport`, and `principal_id` only (lines 227-247). The `message_id` field must come from the agent's skill execution, which is session-driven.

   The runner should validate that the parsed `mail send` result contains a non-empty `message_id` before proceeding to the `mail reply` step, and the sanitize/verify scripts should account for the non-deterministic `message_id` value.

4. **Sanitization scope is broader than the existing CAO-launcher pack.**
   The CAO-server-launcher `sanitize_report.py` handles a narrow set of fields (paths, boolean checks, mode strings). The mailbox tutorial's sanitization must handle:
   - `message_id` values (timestamp-derived, non-deterministic)
   - `thread_id` values
   - `request_id` values (contains UUID — `mailreq-{ts}-{uuid[:10]}`, see `mail_commands.py:168`)
   - `bindings_version` (ISO timestamp)
   - `filesystem_root` (absolute path)
   - `principal_id` and `address` (stable but environment-dependent if the demo derives them from hostnames)

   Task 2.3 mentions sanitization but does not enumerate the fields. The spec requirement "sanitize non-deterministic values such as absolute paths, timestamps, runtime-specific ids, session-manifest locations, and mailbox message identifiers" is good, but the tasks should list concrete field masks.

5. **Spec file inventory lists `scripts/sanitize_report.py` and `scripts/verify_report.py` under the tutorial pack, but tasks do not mention creating the `scripts/` subdirectory.**
   The CAO-server-launcher pack keeps its Python scripts under `scripts/demo/cao-server-launcher/scripts/`. Task 2.3 should explicitly note the subdirectory path `scripts/demo/mailbox-roundtrip-tutorial-pack/scripts/sanitize_report.py` and `verify_report.py`.

6. **Backend choice is implicit; the tutorial may need `claude_headless` or `codex_headless` for two concurrent sessions.**
   The existing `cao-interactive-full-pipeline-demo` uses CAO-backed sessions (`cao_rest` backend implied by the CAO server dependency). The mailbox quickstart examples use `--backend claude_headless`. The tutorial should declare which backend it uses and document the dependency. If it uses the headless backend, it avoids the CAO server dependency; if it uses `cao_rest`, it needs the CAO launcher flow.

   See **Open Question Q2** below.

The CAO-server-launcher pack is a single-session, single-flow demo with shell-driven steps, Python sanitize/verify helpers, and `--snapshot-report` support. The new pack should follow this shape exactly, which the spec correctly requires.

- **Existing interactive demo pattern:** `scripts/demo/cao-interactive-full-pipeline-demo/` uses a Python package (`src/houmao/demo/cao_interactive_demo/`) for stateful session management, recipe resolution, and multi-step lifecycle. The design explicitly rejects this pattern for the mailbox tutorial (Decision 2), which is correct — the mailbox roundtrip is linear and does not need persistent `state.json`-style session tracking.

- **Mailbox CLI surface alignment:** The `cli.py` mail subcommand registration (lines 340-366) matches the design's claimed runtime surfaces (`mail check`, `mail send`, `mail reply`). The `--agent-identity` flag used by all mail commands (`_add_mail_common_args`, line 820) means the runner must persist or compute the agent identity for each of the two sessions. This is a detail not yet addressed in the tasks.

- **Brain recipe + mailbox config alignment:** `BrainRecipe` (`brain_builder.py:61`) supports an optional `mailbox` field of type `MailboxDeclarativeConfig`. The `start-session` CLI also accepts `--mailbox-transport`, `--mailbox-root`, `--mailbox-principal-id`, `--mailbox-address` as runtime overrides. Both paths converge in the runtime startup flow. The tutorial has two valid options: declare mailbox config in tracked recipe YAML files, or pass all mailbox config as CLI flags. See Q1.

- **Agent identity naming:** Existing demos use canonicalized `AGENTSYS-` prefixed names (e.g., `AGENTSYS-alice`). The tutorial should pick two distinct, readable names (e.g., `AGENTSYS-alice` and `AGENTSYS-bob`) and document that the `--agent-identity` flag in `mail` commands uses these canonicalized names.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the tutorial create dedicated mailbox-enabled brain recipes or use CLI-only mailbox overrides?

- Decision: `Blocking`

#### Why this matters

The tutorial must enable mailbox support for two sessions. The `BrainRecipe.mailbox` field (`src/houmao/agents/brain_builder.py:68`) is supported but unused by any tracked recipe. Using CLI flags is more explicit in the README, but using recipe YAML files is more consistent with how the brain-first workflow is documented.

#### Examples

- If the tutorial uses CLI flags only, the `run_demo.sh` invocation for `start-session` would include `--mailbox-transport filesystem --mailbox-root "$SHARED_MAILBOX_ROOT" --mailbox-principal-id AGENTSYS-alice --mailbox-address AGENTSYS-alice@agents.localhost`. This is long but transparent.
- If the tutorial uses recipe YAML, the tracked inputs would include two recipe files with `mailbox:` sections, and the `start-session` invocations would be shorter but require the reader to cross-reference the recipe YAML.

#### Why this question arises in code

- `src/houmao/agents/brain_builder.py:68` — `BrainRecipe.mailbox` exists but no tracked recipe uses it.
- `src/houmao/agents/realm_controller/cli.py:452-455` — `start-session` accepts `--mailbox-*` CLI overrides.
- `tests/fixtures/agents/brains/brain-recipes/claude/gpu-kernel-coder-default.yaml` — example recipe without `mailbox`.
- The tutorial's `inputs/demo_parameters.json` will need to specify the approach.

#### Options

**Option A: Use CLI-only mailbox overrides (no new recipe files).**

- Code impact:
  - The `run_demo.sh` would pass `--mailbox-transport`, `--mailbox-root`, `--mailbox-principal-id`, and `--mailbox-address` as flags to `start-session` for each agent.
  - `inputs/demo_parameters.json` would contain the mailbox principal IDs, addresses, and shared root path.
  - No new recipe YAML files needed.
  ```bash
  pixi run python -m houmao.agents.realm_controller start-session \
    --brain-manifest "$MANIFEST_A" --role "$ROLE" --backend "$BACKEND" \
    --mailbox-transport filesystem \
    --mailbox-root "$SHARED_MAIL_ROOT" \
    --mailbox-principal-id AGENTSYS-alice \
    --mailbox-address AGENTSYS-alice@agents.localhost
  ```
- User-facing difference:
  - The README shows every mailbox parameter explicitly in each `start-session` command — more verbose but fully transparent.
- Pros:
  - Self-contained; reader sees every parameter without opening a recipe file.
  - Does not modify or add recipe files that may confuse users into thinking mailbox recipes are a standard pattern.
- Cons:
  - Long `start-session` commands in the README.
  - Does not demonstrate the `BrainRecipe.mailbox` declarative path, which is the intended production pattern.

**Option B: Create two new tracked mailbox-enabled brain recipes under `tests/fixtures/agents/brains/brain-recipes/`.**

- Code impact:
  - Two new recipe YAML files (e.g., `claude/mailbox-roundtrip-alice.yaml`, `claude/mailbox-roundtrip-bob.yaml`) with `mailbox:` section.
  - `run_demo.sh` references the recipes; only `--mailbox-root` needs a CLI override (shared root is environment-specific).
  ```yaml
  # brain-recipes/claude/mailbox-roundtrip-alice.yaml
  schema_version: 1
  name: mailbox-roundtrip-alice
  tool: claude
  default_agent_name: mailbox-alice
  skills: [openspec-apply-change]
  config_profile: default
  credential_profile: personal-a-default
  mailbox:
    transport: filesystem
    principal_id: AGENTSYS-alice
    address: AGENTSYS-alice@agents.localhost
  ```
- User-facing difference:
  - Shorter `start-session` commands. Reader must open the recipe YAML to see mailbox params.
- Pros:
  - Demonstrates the declarative mailbox-in-recipe pattern for the first time.
  - Cleaner shell commands.
- Cons:
  - Adds fixture recipe files that embed mailbox details, coupling the tutorial's identity/address choices to the fixture tree.
  - Two agents sharing a mailbox root but using different credential profiles may confuse readers about credential isolation.

> **PROPOSED: Use Option A (CLI-only mailbox overrides).**
> Rationale: The tutorial's primary goal is to teach the runtime `mail` workflow transparently. CLI flags make every parameter visible in the README command blocks without requiring cross-reference to recipe YAML. No tracked recipe in the repository currently uses the `mailbox` field (`tests/fixtures/agents/brains/brain-recipes/claude/gpu-kernel-coder-default.yaml`), so introducing it here would add a new pattern solely for the tutorial, not for an established production use case. If the declarative recipe pattern matures later, a second tutorial or a follow-up section can demonstrate it. Keeping the tutorial on CLI flags avoids coupling tracked fixture files to tutorial-specific identity choices.
>
> **DECISION: Accept PROPOSED, with one refinement: keep recipe usage for normal brain build inputs, but express mailbox enablement only through `start-session --mailbox-*` overrides documented in the tutorial.**
> Rationale: This keeps the tutorial aligned with the mailbox quickstart, which already treats `start-session` overrides as a first-class enablement path, while avoiding the introduction of tutorial-only mailbox recipe files that the rest of the repo does not currently use. It also keeps the README transparent, because the mailbox principal, address, and shared root stay visible in the command blocks and `inputs/demo_parameters.json` rather than being hidden inside new fixture recipes.

### Q2) Which backend should the tutorial use for the two concurrent sessions?

- Decision: `Blocking`

#### Why this matters

The mailbox quickstart (`docs/reference/mailbox/quickstart.md`) demonstrates `--backend claude_headless`. The interactive CAO demo (`scripts/demo/cao-interactive-full-pipeline-demo/`) uses a CAO-backed flow. The tutorial must choose a backend, and this affects prerequisites (does the reader need a running CAO server?), concurrency behavior (headless sessions are independent tmux processes; CAO sessions go through the orchestrator), and cleanup mechanics.

#### Examples

- If headless: two independent tmux-backed CLI processes. No CAO dependency. Cleanup is `stop-session` per agent.
- If CAO-backed: a running CAO server at `127.0.0.1:9889` is required. Sessions are managed through the orchestrator. The tutorial inherits the CAO-launcher prerequisite from the interactive demo.

#### Why this question arises in code

- `src/houmao/agents/realm_controller/cli.py:118` — `start-session` accepts `--backend`.
- `docs/reference/mailbox/quickstart.md:27` — quickstart example uses `--backend claude_headless`.
- `scripts/demo/cao-interactive-full-pipeline-demo/README.md` — the interactive demo requires a local CAO server.
- The tutorial's prerequisites checklist is directly shaped by this choice.

#### Options

**Option A: Use `claude_headless` (or the tool-appropriate headless backend).**

- Code impact:
  - `run_demo.sh` passes `--backend claude_headless` (or `codex_headless`, `gemini_headless` depending on recipe).
  - No CAO server dependency.
  - Each session is a direct tmux-backed CLI process.
- User-facing difference:
  - Simpler prerequisites: no CAO server install or launcher step.
  - Matches the mailbox quickstart documentation examples.
- Pros:
  - Fewer prerequisites, lower barrier to entry.
  - Consistent with the mailbox quickstart docs.
  - Independent sessions are easier to reason about for concurrent operation.
- Cons:
  - Does not demonstrate the CAO-backed path, which is the interactive demo's default.

**Option B: Use `cao_rest` (CAO-backed sessions).**

- Code impact:
  - `run_demo.sh` must include the CAO server launcher prerequisite flow (similar to `cao-interactive-full-pipeline-demo`).
  - Both sessions go through the orchestrator.
- User-facing difference:
  - Additional prerequisite: `cao-server` must be installed and healthy.
  - More complex startup flow.
- Pros:
  - Demonstrates the full CAO-backed session lifecycle.
- Cons:
  - Adds a heavyweight prerequisite for what should be a focused mailbox tutorial.
  - The tutorial's teaching goal is the mailbox workflow, not the CAO orchestrator.

> **PROPOSED: Use Option A (`claude_headless` or tool-appropriate headless backend).**
> Rationale: The tutorial's teaching goal is the runtime-owned mailbox roundtrip, not the CAO orchestrator. The mailbox quickstart (`docs/reference/mailbox/quickstart.md:27`) already demonstrates `--backend claude_headless`, so aligning with that keeps the tutorial consistent with the existing mailbox docs. Headless backends have no CAO server dependency, which reduces prerequisites and keeps the tutorial focused. The `run_demo.sh` can parameterize the backend in `inputs/demo_parameters.json` so future variants can swap backends without rewriting the runner.
>
> **DECISION: Reject PROPOSED. Use `cao_rest` and make CAO-backed, TUI-oriented agents the required v1 path for this tutorial pack.**
> Rationale: Current development focus is on TUI-based agents, so the demo needs to exercise the CAO-backed path rather than a simpler headless-only variant. That direction is already consistent with the repository's existing investment in CAO tutorial flows under `scripts/demo/cao-interactive-full-pipeline-demo/` and the corresponding docs reference in `docs/reference/cao_interactive_demo.md`. The artifact revision pass should therefore make the CAO prerequisite, launcher/runtime setup, and two-session CAO lifecycle explicit instead of treating them as optional follow-up scope.

### Q3) How should the runner track two concurrent session identities for follow-up `mail` commands?

- Decision: `Deferrable`

#### Why this matters

Existing demo packs track one session identity. The mailbox tutorial needs two. The `mail` subcommands require `--agent-identity` to target the correct session. The runner must know both identities after startup and use the correct one for each `mail` step.

#### Examples

- After starting agent A and agent B, the runner calls `mail send --agent-identity AGENTSYS-alice --to AGENTSYS-bob@agents.localhost ...`. Then `mail check --agent-identity AGENTSYS-bob ...`. The runner needs both identity strings available throughout the flow.

#### Why this question arises in code

- `src/houmao/agents/realm_controller/cli.py:820` — `_add_mail_common_args` adds `--agent-identity` to all mail subcommands.
- The `cao-interactive-full-pipeline-demo` persists one identity in `state.json` and recovers it for follow-up commands.
- The `cao-server-launcher` demo has no session identity at all (it's a launcher lifecycle demo).

#### Options

**Option A: Use shell variables in `run_demo.sh` — no persistent state file.**

- Code impact:
  - `run_demo.sh` captures `AGENT_IDENTITY_A` and `AGENT_IDENTITY_B` from `start-session` output or from `inputs/demo_parameters.json`.
  - All `mail` commands use these variables directly.
  - No `state.json` needed.
  ```bash
  AGENT_IDENTITY_A="AGENTSYS-alice"
  AGENT_IDENTITY_B="AGENTSYS-bob"
  ```
- User-facing difference:
  - README shows explicit variable assignments; no hidden state.
- Pros:
  - Simple, matches the `cao-server-launcher` pattern (shell-local state only).
  - Easy to follow in the README.
- Cons:
  - No resume or inspect capability (not needed for a linear tutorial).

**Option B: Write a `state.json` with both session entries.**

- Code impact:
  - `run_demo.sh` writes a JSON file after both sessions start, containing both identities, manifests, and mailbox bindings.
  - Follow-up steps read from this file.
- User-facing difference:
  - Adds a state-management artifact.
- Pros:
  - Enables potential future resume/inspect extensions.
- Cons:
  - Over-engineering for a linear tutorial flow that runs once.

> **PROPOSED: Use Option A (shell variables, no persistent state file).**
> Rationale: The `cao-server-launcher` tutorial-pack pattern (`scripts/demo/cao-server-launcher/run_demo.sh`) uses shell-local variables for all flow state and does not persist a `state.json`. The mailbox tutorial is also a linear, run-once flow. Shell variables for both identities keep the runner simple and the README transparent. If a future change adds interactive or multi-step support, `state.json` can be introduced then.
>
> **DECISION: Refine PROPOSED. Keep demo-specific `state.json` minimal or omit it entirely, and use the runtime's existing name-based tmux/manifest recovery path for anything the platform already publishes.**
> Rationale: For CAO-backed tmux sessions, the runtime already publishes `AGENTSYS_MANIFEST_PATH` and `AGENTSYS_AGENT_DEF_DIR` into session env and uses those pointers for name-addressed resumed operations such as `mail` and `stop-session` (`docs/reference/realm_controller.md:391-395`, `docs/reference/realm_controller.md:454-456`). So the demo should not duplicate that source of truth in a richer custom state file just to rediscover live sessions. The pack can still keep shell-local variables and captured `start-session` JSON artifacts for reporting, cleanup coordination, and readability, but any session identity or manifest lookup that the runtime can already recover natively should go through the existing mechanism instead.

### Q4) Should the tutorial use the same credential profile for both agents, or require two distinct credential profiles?

- Decision: `Deferrable`

#### Why this matters

The tutorial needs a low-friction default for two concurrent agents, but the credential choice should still follow the repo's brain/blueprint model instead of becoming an ad hoc shell concern. The corrected fixture docs now treat concurrent credential reuse as allowed when the provider/tool supports it, so the real decision is whether the tutorial should hardcode two profile lanes or let each blueprint's bound recipe determine the credential profile.

#### Examples

- Agent A launches with `--blueprint blueprints/mailbox-roundtrip-claude.yaml`; agent B launches with `--blueprint blueprints/mailbox-roundtrip-codex.yaml`. Each blueprint resolves a role plus a brain recipe, and each recipe selects its own `credential_profile`.
- For the quick-start path, both recipes can be API-key-backed and the user only needs to provide the two provider keys already expected by those tools. If an operator later wants separate rate-limit lanes, they can point either blueprint at a recipe with a different `credential_profile`.

#### Why this question arises in code

- `src/houmao/agents/brain_builder.py` — `BrainRecipe.credential_profile` determines which `api-creds/<tool>/<profile>/` directory is projected.
- `src/houmao/agents/realm_controller/cli.py:102-127` — both `build-brain` and `start-session` accept `--blueprint`.
- `src/houmao/agents/realm_controller/cli.py:376-386` and `src/houmao/agents/realm_controller/cli.py:432-456` — blueprint loading resolves the bound brain recipe, and the resulting recipe still controls tool/config/credential inputs while `start-session` sources the role from the blueprint.
- The corrected agent-fixture docs now explicitly say blueprints inherit recipe credential selection and that concurrent runs may reuse the same provider credentials when supported (`tests/fixtures/agents/README.md:110-111`, `tests/fixtures/agents/blueprints/README.md:10-12`, `tests/fixtures/agents/brains/README.md:36-38`).

#### Options

**Option A: Use the same credential profile for both agents.**

- Code impact:
  - Both `build-brain` invocations reference the same `credential_profile`.
  - Each gets an independent runtime home copy.
- User-facing difference:
  - One prerequisite: set up one credential profile.
- Pros:
  - Simpler prerequisites.
  - The builder copies credentials into separate runtime homes, so no filesystem conflict.
- Cons:
  - If the tool or API enforces per-key session limits, both sessions may compete.

**Option B: Require two distinct credential profiles.**

- Code impact:
  - `inputs/demo_parameters.json` references two profile names.
  - User must set up two credential directories.
- User-facing difference:
  - Higher setup cost.
- Pros:
  - No possible credential contention.
- Cons:
  - Doubles credential setup burden for a tutorial.

> **PROPOSED: Use Option A (same credential profile for both agents).**
> Rationale: The `BrainBuilder` (`src/houmao/agents/brain_builder.py`) copies credentials into independent runtime home directories during the build phase, so two concurrent builds from the same credential profile do not conflict at the filesystem level. The tutorial's prerequisite section should note that the same API key is used for both sessions. If a specific tool enforces per-key session limits, the tutorial's SKIP/troubleshooting section can document that edge case without requiring two profiles by default.
>
> **DECISION: Reject the same-vs-distinct framing as the tutorial's primary design lever. Launch agent A and agent B through different blueprints, let each blueprint's bound recipe determine the credential profile, and allow concurrent reuse of the same provider credentials when the provider/tool permits it.**
