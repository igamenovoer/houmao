## Summary

- The proposal and design correctly diagnose the root cause: `_reconcile_headless_active_turn()` (service.py:2460–2482) uses tmux pane presence as a liveness signal and can incorrectly downgrade active turns to `unknown`.
- The five design decisions are well-structured, internally consistent, and the target status mapping (design.md:83–91) is clean and testable.
- Two open questions in the design (retain `unknown`? persist runner PID or both PIDs?) were unresolved in the reviewed artifacts and directly block tasks 1.1 and 1.2; the developer decisions below resolve them, and the artifacts now need a revision pass to reflect those decisions before implementation begins.
- The specs cover the key reconciliation and restart scenarios but miss the finalization-time path and the restart-with-dead-process-and-no-exitcode case.
- Interrupt handling restructuring (task 2.2) is more involved than stated: the service layer's `_interrupt_active_turn_record()` currently bypasses in-memory and PID-based control entirely and goes directly to tmux `kill-window`.

## Findings

### Must Fix / Blocking

1. **The `unknown` status decision (design.md open question 1) must be resolved before implementation.** Tasks 1.2, 2.1, and 2.3 all depend on whether `unknown` remains in the `ManagedHeadlessTurnRecord.status` Literal type (`managed_agents.py:117`). If collapsed to `failed`, the Literal type changes, which cascades to `ManagedAgentTurnStatus` (`models.py:42`), `ManagedAgentLastTurnResult` (`models.py:41`), the `history_summary` dispatch dict (`service.py:2639–2645`), and the demo reporting layer. If retained for legacy/corrupt-evidence only, a compatibility path for stored turn records is needed. Either way, the decision shapes the data model and every downstream consumer.

> **DECISION: Accept.**
> Rationale: This is a real blocker because the status vocabulary is part of the persisted and public model, not just an internal implementation detail. The change should remove `unknown` as a normal managed-headless outcome and collapse the no-evidence path to `failed` with explicit diagnostics. That decision needs to be reflected consistently across the type aliases, stored turn records, summary mapping, and demo/reporting consumers before implementation starts.

2. **The PID granularity decision (design.md open question 2) must be resolved before task 1.1.** The design proposes `process.json` but leaves open whether to store the runner shell PID only or both the shell and child tool PID. For tmux-based execution, the runner launches a shell script inside a tmux pane (`headless_runner.py:265–276`) which then execs the CLI tool — these are separate processes. Liveness checks via `kill -0 <pid>` after restart need the correct PID. Moreover, capturing the tool child PID from inside the tmux script requires either modifying the script to write `$!` / `$$` or using `tmux list-panes -F '#{pane_pid}'` — the latter reintroduces tmux observation for metadata capture (distinct from using it for state truth, but still a coupling worth noting).

> **DECISION: Accept.**
> Rationale: The review is correct that durable process identity is a prerequisite for task 1.1. The only refinement is that the artifacts should decide the authority contract, not lock the implementation too early to a single capture primitive. The required decision is that headless turns persist durable process metadata for liveness checks and that this metadata is not derived from ongoing tmux watching.

### Should Fix / Important

3. **Service-layer interrupt bypasses in-memory and PID-based control paths.** The design (decision 4) specifies a three-tier interrupt fallback: (1) live in-memory execution handle, (2) persisted process identity, (3) tmux `kill-window`. But `_interrupt_active_turn_record()` (`service.py:2709–2747`) currently jumps directly to `tmux kill-window` (line 2736) without consulting the in-memory `HeadlessCliRunner._active_process` handle or any persisted PID. Task 2.2 says "update managed headless interrupt handling" but should explicitly note that the service-layer interrupt path needs restructuring to first try the runner's `interrupt()` method (`headless_runner.py:92–131`), which already checks the in-memory process handle before falling back to tmux. Currently the service layer and the runner have parallel but disconnected interrupt paths.

> **DECISION: Accept.**
> Rationale: The current service-layer interrupt path is materially out of sync with the design. `_interrupt_active_turn_record()` bypasses both the runner's in-memory handle and any future persisted PID path, so task 2.2 must explicitly restructure the service path rather than only adjusting the runner in isolation.

4. **`_finalize_headless_turn_record()` also produces `unknown` (service.py:2637) but specs don't cover this path.** The houmao-server-agent-api spec scenarios cover reconciliation-time decisions (active turn + no live evidence → failed). The finalization path is a separate code path reached from the worker thread (`service.py:2576`) and from reconciliation itself (`service.py:2453`). When the worker thread completes but no `exitcode` file exists and no interrupt was requested, finalization sets `unknown` (line 2637). This should become `failed` per the design's status mapping, but should also have an explicit spec scenario so it's testable independently from reconciliation.

> **DECISION: Accept.**
> Rationale: This is a separate code path and it needs explicit specification. The change is about controller-owned headless truth across the whole lifecycle, so worker finalization cannot keep a fallback `unknown` branch that the reconciliation path removes.

5. **No spec scenario for restart + execution dead + no exitcode.** The restart spec in `houmao-server-agent-api/spec.md` has scenarios for "execution still live" and "durable artifacts present." It does not cover the third case: server restarts, active turn recorded, execution is dead, no exitcode file exists, no interrupt requested. Per the design's status mapping this → `failed`, but adding an explicit spec scenario would close the gap and ensure test coverage (task 3.2) is anchored to a requirement.

> **DECISION: Accept.**
> Rationale: That restart branch is part of the core design intent and should not be left implicit. The artifacts should state it directly so the implementation and tests anchor to a concrete requirement instead of inferring it from adjacent scenarios.

6. **11 reconciliation call sites share identical behavior — the design should confirm uniform refresh semantics.** `_reconcile_headless_active_turn()` is called from 11 locations in `service.py` (lines 648, 668, 707, 736, 904, 951, 1032, 1047, 1087, 1113, 1603) spanning turn submission, status queries, event fetching, state/detail endpoints, and restart recovery. The new "execution-evidence refresh" logic will replace the reconciliation at all 11 sites. The design should confirm that the replacement behaves identically at every call site, or document any site-specific differences (e.g., a submission call site might need to block until refresh completes vs. a status query that can return stale state).

> **DECISION: Refine.**
> Rationale: The concern is valid, but the follow-up should be to make the refresh contract uniform by default rather than document eleven site-specific behaviors unless a real divergence is required. The artifact revision should say that the new execution-evidence refresh is shared across existing call sites, with any exceptions called out explicitly only if one actually exists.

### Nice To Have

7. **`process.json` write atomicity is unspecified.** The design mentions "make result/process markers atomic enough for readers" (design.md:139) but doesn't define the write mechanism. For a small JSON file containing PID + timestamps, practical corruption risk is low, but a temp-file-then-rename pattern would make the guarantee explicit and is easy to implement. The existing `ManagedHeadlessStore` write methods already use `json.dumps()` → `Path.write_text()` which is not atomic.

> **DECISION: Refine.**
> Rationale: Atomic-enough writes are worth capturing, but this is an implementation-detail refinement, not a new design branch. The revision should add brief guidance that `process.json` is written via a temp-file-and-rename pattern or equivalent atomic write discipline so readers do not observe partial JSON.

8. **Clarify whether `process.json` is written by the runner or the service layer.** The `HeadlessCliRunner` currently writes `stdout.jsonl`, `stderr.log`, and `exitcode` directly into the turn artifact directory (`headless_runner.py:258–260`). The service layer creates the turn record and active-turn record. `process.json` fits naturally alongside the runner's artifacts (same directory, same lifecycle), but the service layer has access to the `ManagedHeadlessActiveTurnRecord` model fields. Task 1.1 targets the runner; task 1.2 targets persistence models. Making explicit which component owns the `process.json` write avoids duplication or gaps.

> **DECISION: Accept.**
> Rationale: `process.json` belongs with the runner-owned execution artifacts. The runner already owns `stdout.jsonl`, `stderr.log`, and `exitcode` in the same turn directory, and `process.json` is part of the same execution-evidence set. The service layer should persist corresponding model fields, but it should not become the writer of the runner's artifact file.

9. **Demo reporting may need updates.** `src/houmao/demo/mail_ping_pong_gateway_demo_pack/reporting.py` has its own status mapping and display logic for managed headless turns. If `unknown` is removed or its semantics change, this reporting layer needs a corresponding update. Task 3.3 says "re-run managed-headless demo coverage" but doesn't mention updating demo-specific status handling code.

> **DECISION: Accept.**
> Rationale: Demo-facing reporting is a downstream consumer of the headless status vocabulary, so it needs to be included in the implementation scope. If `unknown` stops being a normal outcome, the demo layer should not silently preserve legacy assumptions.

10. **Consider renaming `_reconcile_headless_active_turn()` to reflect new semantics.** The design explicitly says "reconciliation becomes execution refresh, not watch-plane inference." Renaming the method (e.g., `_refresh_headless_active_turn()`) across its 11 call sites would reinforce the semantic shift and make it clear to future readers that tmux watching is no longer involved. This is a low-risk rename but improves long-term readability.

> **DECISION: Refine.**
> Rationale: The rename is sensible and aligned with the semantic shift, but it is readability work rather than a design blocker. The artifact revision should treat it as a recommended implementation cleanup, not as a requirement that delays the substantive lifecycle changes.

## Design Alignment With Existing System

- **Artifact directory model:** The proposed `process.json` fits the existing `{turn_artifacts_root}/{turn_id}/` layout alongside `stdout.jsonl`, `stderr.log`, and `exitcode` (`headless_runner.py:256–260`). The design is additive — no existing artifacts change format or location.
- **Persistence models:** `ManagedHeadlessActiveTurnRecord` (`managed_agents.py:69–109`) and `ManagedHeadlessTurnRecord` (`managed_agents.py:111–168`) currently store `tmux_session_name` and `tmux_window_name`. These fields should remain for attach/cleanup but become non-authoritative for state. New fields for process identity (PID, launch timestamp) need to be added to these models — this aligns with the existing Pydantic v2 model pattern and the additive-field approach already used in the codebase.
- **Interrupt layering:** The runner already has a two-tier interrupt model (`headless_runner.py:92–131`): in-memory process handle first, tmux `kill-window` as fallback. The design's three-tier model (in-memory → persisted PID → tmux) extends this pattern naturally. The gap is that the service layer's `_interrupt_active_turn_record()` (`service.py:2709–2747`) bypasses the runner entirely and goes straight to tmux — finding #3 above.
- **Status vocabulary:** The existing `ManagedAgentLastTurnResult` type (`models.py:41`) already distinguishes `"known_failure"` from `"unknown"`, which aligns with the design's goal of making `unknown` non-normal. The `ManagedAgentTurnStatus` type (`models.py:42`) parallels but uses different labels; both will need coordinated updates.
- **Test structure:** No dedicated managed-agent unit tests currently exist under `tests/unit/server/`. The existing test infrastructure in `tests/unit/agents/realm_controller/` covers the runner layer but not the service-layer reconciliation logic. Tasks 3.1–3.2 are critical and well-scoped.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the runner capture the CLI tool's PID via the tmux launch script or via `tmux list-panes`?

- Decision: `Blocking`

#### Why this matters

The design asks the runner to persist PID information in `process.json` for restart-time liveness checks. For tmux-based execution, the CLI tool runs inside a shell script inside a tmux pane. Getting the tool's PID is non-trivial: the runner doesn't directly spawn the tool process — tmux does.

#### Examples

- A `claude -p "..."` invocation is wrapped in a shell script (headless_runner.py:265–276). The PID visible to the runner is the tmux server's or nothing — the tool's PID lives inside the pane.
- After server restart, `kill -0 <pid>` needs the tool's PID, not the wrapper shell's PID (which may have already exited if `exec` was used, or may be alive while the tool is dead).

#### Why this question arises in code

- `headless_runner.py:265–276`: the launch script uses `set +e` and a shell pipeline, meaning `$!` would capture the shell's PID, not the tool's.
- The script could be modified to use `exec` for the tool command, making `#{pane_pid}` the tool PID itself.
- Alternatively, the script could write `$!` or `$$` to `process.json` before the tool starts.

#### Options

**Option A: Modify the tmux launch script to `exec` the CLI command and capture `#{pane_pid}` once after launch.**

- Code impact:
  - Change the tmux script to use `exec` for the tool command so the pane PID becomes the tool PID.
  - After `run_tmux_shared(["new-window", ...])`, immediately query `tmux list-panes -t <window> -F '#{pane_pid}'` to get the PID.
  - Write PID + timestamp to `process.json` in the turn artifact dir.
  - ```python
    # In _run_in_tmux, after launching:
    pane_pid = run_tmux_shared(["list-panes", "-t", window_id, "-F", "#{pane_pid}"])
    (turn_dir / "process.json").write_text(json.dumps({"pid": int(pane_pid), ...}))
    ```
- User-facing difference:
  - No UX change. Restart recovery can check `kill -0 <pid>` without tmux observation.
- Pros:
  - PID is the actual tool PID (most accurate for liveness checks).
  - One-time tmux query at launch is metadata capture, not ongoing observation — semantically distinct from tmux watching for state truth.
  - `exec` simplifies the process tree (one fewer shell layer).
- Cons:
  - Using `exec` changes the shell script structure; `exitcode` file must be written before `exec` or by a trap.
  - The tmux query is still a tmux dependency at launch time.

**Option B: Write PID from inside the launch script using `$$` or `$!` before the tool starts.**

- Code impact:
  - Add a line to the tmux script: `printf '%s\n' "$$" > process.json` (or a JSON wrapper).
  - The PID is the wrapper shell's PID, not the tool's PID (unless `exec` is also used).
  - ```shell
    # In the tmux launch script:
    printf '{"pid": %d, "launched_at_utc": "%s"}\n' "$$" "$(date -u +%FT%TZ)" > process.json
    exec <tool_command> > stdout.jsonl 2> stderr.log
    ```
- User-facing difference:
  - Same as Option A if `exec` is used. Without `exec`, liveness check targets the wrapper shell, which may survive after the tool exits.
- Pros:
  - No tmux query needed — fully self-contained.
  - Works even if tmux API changes.
- Cons:
  - Without `exec`, the captured PID is the wrapper shell, not the tool. With `exec`, the exitcode-writing logic needs to move to a trap handler.
  - JSON writing from shell is fragile compared to Python's `json.dumps()`.

> **PROPOSED: Option A — use `exec` in the tmux script and capture `#{pane_pid}` once after launch.**
> Rationale: This gives the most accurate PID (the actual CLI tool process) and the one-time tmux query at launch is metadata capture, not ongoing state observation. The `exec` approach also simplifies the process tree. The exitcode file can be written via a shell `trap` before `exec`, which is a well-understood pattern. The existing runner already queries tmux for launch confirmation (`headless_runner.py:278–295`), so one additional launch-time query is consistent.
>
> **DECISION: Refine PROPOSED.**
> Rationale: The review is right about the requirement but too specific about the capture primitive. The artifacts should decide that headless execution persists durable process identity owned by the runner, ideally including both wrapper-shell and child-tool identity when practical, and that this metadata is captured at launch rather than inferred later from tmux state. A one-time tmux query at launch is acceptable if needed, but the change should not hard-code `exec` plus `#{pane_pid}` as the only acceptable implementation because that unnecessarily constrains how `exitcode` and result markers are produced.

### Q2) Should `_reconcile_headless_active_turn()` check process liveness before concluding "execution dead"?

- Decision: `Blocking`

#### Why this matters

The current reconciliation checks (1) exitcode file, then (2) tmux pane presence. The design replaces (2) with "is the underlying execution still live?" But the design doesn't specify the exact mechanism for this liveness check in the non-restart case (when the server is still running and holds the worker thread).

#### Examples

- Normal operation: worker thread is running `send_prompt()`, which blocks inside the runner. A status query arrives on another thread, calls `_reconcile_headless_active_turn()`. The reconciliation sees no exitcode yet. Should it check `kill -0 <pid>` from `process.json`, or should it trust that the worker thread's existence means the turn is still active?
- After restart: no worker thread exists. Now `process.json` + `kill -0` is the only liveness signal.

#### Why this question arises in code

- `_reconcile_headless_active_turn()` (`service.py:2437–2500`) is called from 11 sites, some of which execute while the worker thread is still running.
- The `_ManagedHeadlessHandle` (in-memory) already tracks whether an active thread exists (`handle.set_active_thread()`). This could be the first-tier check.

#### Options

**Option A: Two-tier liveness — check in-memory handle first, then persisted PID.**

- Code impact:
  - If the in-memory handle shows an active worker thread, skip PID check and return (turn is active).
  - If no in-memory handle (restart or thread died), fall back to `kill -0 <pid>` from `process.json`.
  - If neither → `failed`.
- User-facing difference:
  - None visible. Internal reconciliation is faster in the common case (no disk/OS check needed).
- Pros:
  - Avoids unnecessary PID checks during normal operation.
  - Clean separation: in-memory for live server, persisted PID for restart recovery.
- Cons:
  - Requires `_reconcile_headless_active_turn()` to have access to the in-memory handle, which it currently doesn't receive as a parameter.

**Option B: Always check persisted PID, ignore in-memory handle.**

- Code impact:
  - Every reconciliation call reads `process.json` and runs `kill -0 <pid>`.
  - Simpler code path — one check regardless of context.
- User-facing difference:
  - Negligible; `kill -0` is fast.
- Pros:
  - Simpler logic, single code path for all scenarios.
  - Works identically pre- and post-restart.
- Cons:
  - Slightly more I/O on every reconciliation call (11 call sites, some on hot status-query paths).
  - PID reuse risk applies even during normal operation (unlikely but theoretically possible for very long turns).

> **PROPOSED: Option A — check in-memory handle first, persisted PID only after restart or thread death.**
> Rationale: The existing `_ManagedHeadlessHandle` already tracks active threads (service.py worker thread lifecycle). Using this as the first-tier check avoids redundant `kill -0` calls on the 11 reconciliation call sites during normal operation. The persisted-PID path activates only when the in-memory handle is absent (restart) or shows no active thread (unexpected thread death), which is exactly the restart-recovery scenario the design targets. This mirrors the interrupt design's layering (in-memory → persisted → tmux fallback).
>
> **DECISION: Accept PROPOSED.**
> Rationale: This matches the current architecture and keeps the common case cheap. The live server already has an in-memory execution handle, so that should be authoritative during normal operation; persisted PID liveness is the recovery path after restart or thread loss. That layering also aligns with the interrupt path the design is trying to establish.

### Q3) Should legacy turn records missing the new execution-evidence fields reconcile to `failed` or retain `unknown`?

- Decision: `Deferrable`

#### Why this matters

After deployment, existing persisted `ManagedHeadlessTurnRecord` entries may lack the new `process.json` fields. If a server restart encounters an active-turn record without process metadata and without an exitcode, it cannot perform a PID liveness check. The design mentions "support a compatibility path for pre-change records" (design.md:140) but doesn't specify the behavior.

#### Examples

- A turn was active before the code change was deployed. Server restarts with new code. Active-turn record has `tmux_session_name` and `tmux_window_name` but no PID metadata. No exitcode exists.
- Should this reconcile to `failed` (consistent with new semantics) or `unknown` (legacy bucket)?

#### Why this question arises in code

- `ManagedHeadlessActiveTurnRecord` (`managed_agents.py:69–109`) will gain optional PID fields. Old records will have `None` for these fields.
- The refresh logic needs a branch for "no PID metadata available."

#### Options

**Option A: Legacy records without PID metadata → `failed` with diagnostic explaining missing evidence.**

- Code impact:
  - Refresh logic treats missing PID metadata the same as "execution dead."
  - Add a diagnostic string like `"Pre-migration active turn without execution-evidence metadata."`
- User-facing difference:
  - Old active turns are immediately finalized as `failed` on first restart with new code.
- Pros:
  - Clean break — no `unknown` survives into the new regime.
  - Operators see a clear diagnostic.
- Cons:
  - A still-running legacy turn would be prematurely failed. (Unlikely: the upgrade window is typically short.)

**Option B: Legacy records without PID metadata → `unknown` with deprecation diagnostic.**

- Code impact:
  - Keep `unknown` in the Literal type.
  - Refresh logic uses `unknown` only for this specific legacy/missing-evidence case.
- User-facing difference:
  - Old active turns get `unknown` on restart, matching pre-change behavior.
- Pros:
  - No behavioral regression for existing deployments during transition.
  - `unknown` has a clearly bounded scope.
- Cons:
  - `unknown` remains in the type system and callers must still handle it.

> **PROPOSED: Option A — legacy records reconcile to `failed` with a clear diagnostic.**
> Rationale: The design explicitly says "`unknown` should no longer be a normal headless reconciliation outcome" (design.md:92). Keeping `unknown` for legacy records perpetuates the ambiguity the change is designed to remove. Since the system is under active and unstable development (per CLAUDE.md), backward compatibility with old stored records is not a priority. A clear `failed` + diagnostic message is more actionable for operators than `unknown`.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Legacy records without the new execution metadata should fail closed with an explicit diagnostic. Retaining `unknown` even for migration-only cases would keep the ambiguous status in the public model and undermine the purpose of this change. The repository guidelines explicitly do not prioritize backward compatibility here.
