## Summary

- **Solid motivation and clear intent.** The design correctly identifies that the current per-turn `tmux new-window` pattern (`headless_runner.py:349`) creates a topology mismatch with the operator mental model, and the serialized execution contract already makes per-turn windows unnecessary.
- **Good alignment with existing controller-owned lifecycle.** Keeping turn identity as disk artifacts (stdout, stderr, exitcode, process.json) while only changing the tmux surface is a clean separation of concerns.
- **The primary execution change is well-scoped.** Replacing `tmux new-window` with window-0 shell injection in `HeadlessCliRunner._run_in_tmux()` is a focused, testable refactoring.
- **Risk: The interrupt/terminate fallback in both `headless_runner.py` and `service.py` uses `kill-window` — reusing window 0 makes that destructive.** This is the single biggest implementation hazard and needs explicit task coverage.
- **Risk: The shell injection model introduces state leakage between turns that `new-window` avoids.** The design acknowledges this but the tasks don't include explicit cleanup/reset logic.
- **Missing: No concrete guidance on what the injected runner wrapper looks like or how the persistent shell is managed between turns.** The design says "inject one bounded runner command" but stops short of specifying the mechanism (tmux `send-keys`, `respawn-pane`, or a different approach).

## Findings

### Must Fix / Blocking

1. **`kill-window` fallback in `_interrupt_headless_tmux_target()` (service.py:2983) would destroy the stable agent surface under the new model.** Currently the server's last-resort interrupt path does `run_tmux(["kill-window", "-t", target])` where `target` is `{session}:{window_name}`. Under window-0 reuse, killing window 0 destroys the entire agent surface and effectively kills the tmux session. The design mentions this (design.md line 95: "last-resort interrupt should prefer live process identity first, then persisted process identity, then a control-input fallback aimed at window 0 rather than `kill-window`"), but task 2.1 is too vague — it says "update managed-headless active-turn metadata and fallback interrupt/control paths" without calling out that `kill-window` must be replaced with `send-keys C-c` or process-signal based interrupt. This is a correctness-critical change that should be its own explicit subtask.

> **DECISION: Accept.**
> Rationale: This is a real correctness hazard under the proposed stable-window model. The change artifacts need to call out, explicitly, that managed-headless fallback interrupt must stop using `kill-window` against the primary agent surface and instead fall back through live process identity, persisted process identity, and finally control-input aimed at window 0.

2. **`HeadlessCliRunner.interrupt()` and `terminate()` (headless_runner.py:108-200) also fall back to `kill-window` on the active window id.** Lines 136 and 184 both call `run_tmux(["kill-window", "-t", window_id])`. When window_id is window 0, this is equivalent to tearing down the session. The tasks must explicitly cover replacing these fallbacks with process-signal or `send-keys`-based alternatives that leave window 0 intact.

> **DECISION: Accept.**
> Rationale: The same destructive fallback exists in the runner layer, not just the server layer. The artifact revision should make this an explicit implementation requirement so both `HeadlessCliRunner.interrupt()` and `terminate()` preserve the stable window-0 surface.

3. **No specification of the shell injection mechanism.** The design says "inject one bounded runner command into that surface" (design.md line 67) but does not specify _how_. The current code uses `tmux new-window ... sh -lc <script>` (headless_runner.py:347-361) which creates a new shell that exits when the script finishes. For window-0 reuse, the runner must either: (a) use `tmux send-keys` to inject a command into the existing shell and then wait for completion, or (b) use `tmux respawn-pane` to restart the pane with a new command. Each approach has different signal, environment, and cleanup semantics. This decision point should be an explicit design decision or at minimum an explicit task with acceptance criteria.

> **DECISION: Accept.**
> Rationale: The change is under-specified until the execution primitive is chosen. The design should stop speaking generically about "injecting" commands into a persistent shell and instead name a concrete same-pane launch mechanism with acceptance criteria.

### Should Fix / Important

4. **`tmux_window_name` field in `ManagedHeadlessActiveTurnRecord` and `ManagedHeadlessTurnRecord` (managed_agents.py:78, 139) becomes meaningless or misleading.** Currently set to `f"turn-{backend_turn_index}"` (service.py:980). Under window-0 reuse, this field should either be removed, set to a constant like `"agent"`, or kept as `None`. The specs and tasks don't address how to migrate or repurpose this field. If it's kept for backward compatibility with persisted records, the migration strategy should be explicit.

> **DECISION: Refine.**
> Rationale: The underlying issue is real, but this change does not need a broad migration plan or field removal. For this change, `tmux_window_name` should be repurposed to a stable value such as `agent` for new managed-headless records and related inspectability payloads, while older persisted records are tolerated as legacy data.

5. **`TmuxTransportResolver.resolve_target()` (tui/transport.py:25-45) resolves panes by `window_name`.** The tracking cycle in service.py (line 1346) passes `identity.tmux_window_name` to the resolver. Under the new model, this will be `None` or `"agent"` for all turns, making the resolver fall back to `_prefer_active_pane(panes)` across all windows in the session. If auxiliary windows exist (e.g., gateway), the resolver might pick the wrong pane. Task 2.2 mentions "inspectability surfaces" but doesn't call out this resolver path specifically.

> **DECISION: Refine.**
> Rationale: The concern about auxiliary windows is valid, but the cited `TmuxTransportResolver.resolve_target()` path is part of the TUI tracking plane rather than the managed-headless execution contract this change is targeting. The artifact revision should instead make managed-headless attach/control surfaces explicit enough that they do not rely on generic active-pane heuristics across auxiliary windows.

6. **Window-0 shell state can leak between turns.** When `new-window` is used, each turn gets a clean shell environment. With window-0 reuse, environment variables, working directory changes, or shell state from a previous turn's wrapper script could leak into the next turn. The design acknowledges this risk (design.md line 111: "A persistent shell can remain in a surprising state after a failed turn wrapper") but the tasks don't include an explicit shell-reset or environment-sanitization step between turns.

> **DECISION: Refine.**
> Rationale: Shell-state leakage is a real risk, but the right artifact change is not a generic cleanup task layered on top of an unspecified injection model. The design should choose a fresh-shell same-pane primitive for each turn so clean per-turn execution follows from the launch model itself.

7. **`select-window` call after launch (headless_runner.py:384-393) needs rethinking.** Currently the runner selects the newly created turn window to make it the active window for operator visibility. Under window-0 reuse, window 0 is already the default selected window. This call should either be removed or redirected, but the tasks don't mention it.

> **DECISION: Accept.**
> Rationale: Any implementation that still selects a transient per-turn window would contradict the stable-surface goal. The tasks should explicitly cover removing or repointing this behavior to the stable primary agent window.

8. **CAO session manifest `tmux_window_name` resolution in runtime.py (lines 1180-1197) validates consistency between `cao.tmux_window_name` and `backend_state.tmux_window_name`.** Under the new model, this field's semantics change. The validation logic and the `CaoSessionState.tmux_window_name` field need updating to reflect that headless agents no longer have per-turn window names.

> **DECISION: Reject.**
> Rationale: The cited CAO manifest path is not part of the tmux-backed headless backend state being changed here. Headless manifests persist `tmux_session_name`, but not a headless `tmux_window_name` through the CAO section, so broadening this change into CAO-session schema alignment would be unrelated scope.

### Nice To Have

9. **Window 0 rename to `"agent"` is mentioned in design but not in tasks.** Design.md line 35 says "the runtime should rename window 0 to a stable name such as `agent`". This is a small operator UX improvement that should be a subtask of 1.1, or explicitly deferred.

> **DECISION: Accept.**
> Rationale: The design already states the intended operator-visible name, so the task list should carry that through explicitly. Renaming window 0 to `agent` is small, but it is part of the contract this change is trying to make stable.

10. **The `HeadlessSessionState` dataclass (headless_base.py:37-44) has no field for tracking the stable window.** While window 0 is an index convention, having an explicit `tmux_window_id` or `tmux_window_name` field in the state would make the contract self-documenting and allow the runner to reference it without hardcoding `:0`.

> **DECISION: Reject.**
> Rationale: The stable headless window is a fixed runtime convention, not mutable per-session state. Adding another field to `HeadlessSessionState` would duplicate the contract and create unnecessary drift risk without improving the design.

11. **Task 3.3 ("Re-run the headless demo or autotest harness") is vague.** It doesn't specify which demo or harness, making it hard to verify completion. Linking to a concrete script or test fixture would improve actionability.

> **DECISION: Accept.**
> Rationale: Verification should name a concrete harness so completion is testable. The existing `scripts/demo/mail-ping-pong-gateway-demo-pack/autotest/run-case.sh` path is the right concrete target for this change's demo-level verification.

## Design Alignment With Existing System

- **Existing pattern: per-turn tmux window allocation.** The current `HeadlessCliRunner._run_in_tmux()` (headless_runner.py:272-464) creates a new tmux window per turn with `tmux new-window -d -P -F #{window_id} -t <session> -n turn-<N> sh -lc <script>`. Turn completion is signaled via `tmux wait-for`, and per-turn artifacts (stdout.jsonl, stderr.log, exitcode, process.json) are written to a turn-specific directory under `<manifest>.turn-artifacts/turn-NNNN/`.

- **Proposed design alignment: The proposal correctly preserves the per-turn artifact contract** (durable stdout/stderr/exitcode/process.json files per turn). This is the right call — it keeps the controller-owned evidence model intact while only changing the tmux surface topology.

- **Mismatch: The proposal doesn't address how the runner wrapper script lifecycle works under window-0 reuse.** The current script (headless_runner.py:300-343) is a self-contained `sh -lc` invocation that sets up FIFOs, launches the command, captures output, writes exit status, and signals completion. Under window-0 reuse, this script needs to be re-executed in the existing shell — but the mechanism for doing so (and for cleaning up the previous turn's state) is not specified. This is the most significant gap between the design and the existing implementation pattern.

- **Mismatch: `_active_tmux_window_id` tracking in `HeadlessCliRunner` (headless_runner.py:61, 396, 461).** Currently the runner captures the window id from `tmux new-window -P` output and uses it for interrupt/terminate targeting. Under window-0, the runner would need to either resolve window 0's id at session creation time (from `_ensure_tmux_container`) or use a stable target like `{session}:0`. The runner's internal state tracking needs adjustment.

- **Alignment: `HeadlessInteractiveSession._ensure_tmux_container()` (headless_base.py:254-306) already creates the tmux session with `create_tmux_session()` which produces a default window 0.** The proposed change leverages this existing behavior — window 0 already exists after session creation, it's just currently left as an idle shell. This is a good starting point.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What tmux primitive should replace `new-window` for injecting turn commands into window 0?

- Decision: `Blocking`

#### Why this matters

- The injection mechanism determines signal handling, environment isolation, cleanup semantics, and whether the pane shell survives turn failures. Choosing wrong could make headless sessions unreliable.

#### Examples

- Turn 1 runs successfully, leaves the pane showing a bash prompt. Turn 2 must inject a new runner script without leftover environment or half-typed input.
- Turn 1 fails mid-execution (OOM kill, segfault). The pane may show an error or a dead shell. Turn 2 must still be injectable.

#### Why this question arises in code

- `headless_runner.py:347-361` currently uses `tmux new-window ... sh -lc <script>` which gives each turn a clean shell and a clean exit.
- Under window-0 reuse, the analogous operation must be one of: `send-keys` into an existing shell, `respawn-pane -t {session}:0`, or `send-keys` with a heredoc/wrapper approach.
- The choice affects how `_active_tmux_window_id` is resolved (line 396) and how completion signaling works.

#### Options

**Option A: `tmux send-keys` to inject the runner script into the existing window-0 shell**

- Code impact:
  - Replace `new-window` with `send-keys -t {session}:0 "sh -c '<script>'" Enter`
  - Window id is known statically as `{session}:0`
  - Completion still uses `tmux wait-for` (unchanged)
  - Need to add a shell prompt detection or readiness check before injection
- User-facing difference:
  - Operator sees command being "typed" into the pane
  - Pane scroll-back accumulates across turns (may be desirable for history)
- Pros:
  - Shell stays alive between turns; no pane respawn flicker
  - Scroll-back preserves multi-turn history in one pane
- Cons:
  - Sensitive to shell state; if previous turn left the shell in a bad state (e.g., backgrounded process, altered PS1), injection may fail
  - Requires quoting the entire runner script for `send-keys`, which is fragile

**Option B: `tmux respawn-pane -k -t {session}:0` with the runner script**

- Code impact:
  - Replace `new-window` with `respawn-pane -k -t {session}:0 sh -lc '<script>'`
  - Each turn gets a fresh process in the same pane (clean environment)
  - Window id remains stable; pane id changes but target `{session}:0` is stable
  - `_active_tmux_window_id` can use `{session}:0` directly
- User-facing difference:
  - Pane content resets between turns (clean terminal)
  - No scroll-back accumulation across turns
- Pros:
  - Clean environment per turn — no state leakage
  - Closest behavioral match to `new-window` but reuses the same window
  - Simpler quoting (same as current `new-window` invocation)
  - Pane pid is predictable via `respawn-pane -P -F #{pane_pid}`
- Cons:
  - Loses cross-turn scroll-back (mitigated by per-turn artifact files)
  - Kills any lingering process from a previous turn (usually desired but worth noting)

> **PROPOSED: Use `tmux respawn-pane -k -t {session}:0` (Option B).**
> Rationale: `respawn-pane` gives each turn a clean shell environment while keeping the pane in window 0, which is the closest behavioral equivalent to the current `new-window` pattern. It avoids the quoting fragility and shell-state sensitivity of `send-keys`, and it preserves the existing runner script structure (headless_runner.py:300-343) with minimal changes — only the tmux launch command changes from `new-window` to `respawn-pane`. The `-k` flag handles the case where a previous turn's shell is still running. Cross-turn scroll-back loss is acceptable because per-turn artifacts already capture all output durably.
>
> **DECISION: Refine PROPOSED.**
> Rationale: The change should use `tmux respawn-pane -k` or an equivalent same-pane fresh-process primitive for normal turn execution, not raw `send-keys` into a long-lived interactive shell. The artifact revision should also make explicit that the pane returns to an idle shell after the bounded runner completes, so window 0 stays usable as the stable agent surface between turns.

### Q2) Should the detailed-state payload expose an explicit stable attach target (e.g., `session:0`)?

- Decision: `Deferrable`

#### Why this matters

- The design's own open question (design.md line 131) asks this. If the payload exposes `tmux_attach_target: "{session}:0"`, demos and diagnostics can use it directly. If it remains an internal convention, callers must know to attach to window 0 by convention.

#### Examples

- A demo script wants to show the agent's live output. With an explicit field, it reads `tmux_attach_target` from the API. Without it, it hardcodes `:0`.
- A future TUI dashboard renders a "watch" button. An explicit target makes this self-describing.

#### Why this question arises in code

- `ManagedHeadlessActiveTurnRecord.tmux_window_name` (managed_agents.py:78) currently holds `"turn-N"` and is used by the transport resolver (service.py:1346) and interrupt fallback (service.py:2978). Under the new model this field must change.
- The question is whether to repurpose it (e.g., always `"agent"` or `"0"`), remove it, or add a new field.

#### Options

**Option A: Repurpose `tmux_window_name` to always be `"agent"` (the renamed window 0)**

- Code impact:
  - Set `tmux_window_name = "agent"` in service.py:980 instead of `f"turn-{backend_turn_index}"`
  - Transport resolver in tui/transport.py uses `"agent"` to filter panes
  - Interrupt fallback targets `{session}:agent` instead of `{session}:turn-N`
- User-facing difference:
  - API consumers see `tmux_window_name: "agent"` in active-turn records
- Pros:
  - Minimal schema change; field keeps the same name and type
  - Self-describing for callers
- Cons:
  - Slightly misleading — it's not really a "window name" in the per-turn sense anymore

**Option B: Add a new `tmux_attach_target` field and deprecate `tmux_window_name`**

- Code impact:
  - Add `tmux_attach_target: str` to active-turn records
  - Keep `tmux_window_name` as `None` for new turns
  - Transport resolver and interrupt fallback use `tmux_attach_target`
- User-facing difference:
  - Cleaner API semantics with a purpose-specific field
- Pros:
  - Clearer intent; no confusion with the old per-turn semantics
- Cons:
  - More fields to maintain; existing consumers must update

> **PROPOSED: Repurpose `tmux_window_name` to `"agent"` (Option A) for now.**
> Rationale: The field already exists in both `ManagedHeadlessActiveTurnRecord` and `ManagedHeadlessTurnRecord` (managed_agents.py:78, 139) and is consumed by the transport resolver (transport.py:38) and interrupt fallback (service.py:2978). Setting it to a constant `"agent"` is the smallest change that aligns with the new model. A dedicated `tmux_attach_target` field can be added later if the API needs to evolve beyond tmux-specific semantics. The development-status convention allows breaking changes, so repurposing the field is acceptable.
>
> **DECISION: Accept PROPOSED.**
> Rationale: For this change, a constant stable window name is the smallest coherent contract. Managed-headless records and inspectability surfaces can use `tmux_window_name=\"agent\"` together with the existing session name, and a dedicated attach-target field can remain out of scope until there is a broader API need.

### Q3) Should `HeadlessCliRunner` support both `new-window` and window-0 injection, controlled by a flag?

- Decision: `Deferrable`

#### Why this matters

- The rollback strategy (design.md line 123-126) says "if the stable-window execution path proves unreliable, keep the controller-owned artifact and process-evidence model but revert to the previous launch primitive temporarily." Having both paths behind a flag would make rollback trivial without reverting code.

#### Examples

- After deploying the window-0 change, a specific tool (e.g., gemini) has issues with `respawn-pane`. A flag allows reverting just that tool's execution model.

#### Why this question arises in code

- `HeadlessCliRunner._run_in_tmux()` (headless_runner.py:272) is the single execution path for all headless tools. Making the tmux primitive configurable would require either a constructor parameter or a method parameter.

#### Options

**Option A: Clean replacement — remove `new-window` path entirely**

- Code impact:
  - Replace `new-window` with the chosen window-0 primitive directly
  - Rollback requires git revert
- Pros:
  - Simpler code; no dead paths
  - Aligns with "breaking changes are allowed" development status
- Cons:
  - Rollback is a code revert

**Option B: Feature flag — keep both paths, default to window-0**

- Code impact:
  - Add a `use_stable_window: bool = True` parameter
  - Branch in `_run_in_tmux` between `new-window` and window-0 injection
- Pros:
  - Instant rollback via config change
  - Can be toggled per-tool if needed
- Cons:
  - Two code paths to maintain and test
  - Temporary complexity that should eventually be removed

> **PROPOSED: Clean replacement (Option A).**
> Rationale: The project's development-status convention explicitly allows breaking changes and discourages backward-compatibility shims (CLAUDE.md: "Breaking changes are allowed. Do not spend effort preserving legacy interfaces"). A feature flag adds test surface and maintenance burden for a rollback scenario that is better handled by git revert. The single execution path keeps `HeadlessCliRunner` simple and fully testable.
>
> **DECISION: Accept PROPOSED.**
> Rationale: This change should be a clean replacement. Keeping both per-turn-window and stable-window execution paths would dilute the design, expand the test matrix, and keep the old topology alive without a clear product reason.
