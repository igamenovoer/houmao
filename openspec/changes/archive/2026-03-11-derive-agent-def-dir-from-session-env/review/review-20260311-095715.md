## Summary

- **Solid problem framing and motivation.** The proposal clearly identifies the operational friction (cwd-dependence for name-addressed control) and the available leverage point (tmux session environment).
- **Design decisions are well-reasoned** with alternatives considered and rejected with justification; the scoping boundary (manifest-path unchanged) is appropriate.
- **Spec scenarios are concrete and testable**, covering the happy path, explicit override, and three distinct failure modes for invalid tmux pointers.
- **Publication data-flow gap**: the design specifies *what* to publish (`AGENTSYS_AGENT_DEF_DIR` in tmux env) and *when* to recover it, but does not specify *how* the `agent_def_dir` value reaches the backend publication point at session-start time — this is the primary implementation risk.
- **Resolution result type is undefined**: the design mentions centralizing "recovered manifest path and recovered agent-def-dir in one resolution result type" (Risks section) but neither names nor defines this type, leaving a structural ambiguity for tasks 2.1–2.2.
- **Interactive demo coverage is well-scoped** to the three control verbs (send-prompt, send-keys, stop-session), correctly keeping build/start on the existing explicit path.

## Findings

### Must Fix / Blocking

#### F1. Publication data-flow plumbing is unspecified

**What:** The design (Decision 1) states "Tmux-backed headless and CAO-backed launches will publish `AGENTSYS_AGENT_DEF_DIR`" and the migration plan (step 1) says "Extend tmux-backed session launch to publish `AGENTSYS_AGENT_DEF_DIR`." But the design does not specify which object carries the `agent_def_dir` value from the CLI resolution site to the backend publication point.

**Why it matters:** The current data flow is:

1. `cli.py` (lines ~372–380) resolves `agent_def_dir` from CLI arg / env / cwd default.
2. `cli.py` calls `build_brain(agent_def_dir, ...)` → `BrainManifest`.
3. `cli.py` calls `start_session(agent_def_dir, brain_manifest, ...)`.
4. `RuntimeSessionController` builds a `LaunchPlan`.
5. `LaunchPlan` is passed to the backend.
6. Backend starts tmux session and publishes env vars via `_publish_tmux_manifest_pointer()` (headless_base.py ~line 297) or `_set_tmux_session_environment()` (cao_rest.py ~line 1437).

Today, `agent_def_dir` is consumed in steps 2–3 but is not persisted into the `LaunchPlan`, `RuntimeSessionController`, `BrainManifest`, or session manifest. The design explicitly rejects storing it in the session manifest ("avoid manifest schema churn"), but does not specify an alternative carrier.

**Impact on tasks:** Task 1.1 ("Update tmux-backed launch paths to publish `AGENTSYS_AGENT_DEF_DIR`") cannot be implemented without a design decision on the carrier object. Different choices have different implications:

- Adding a field to `LaunchPlan`: minimal surface change, but `LaunchPlan` is currently a launch-time artifact, not a persisted metadata carrier.
- Threading it through `RuntimeSessionController`: requires constructor/method signature changes.
- Adding it to `BrainManifest` output: semantically arguable since the manifest describes the built brain, not the source tree.

**Recommendation:** The design should add a sub-decision under Decision 1 specifying which object carries `agent_def_dir` to the publication site and whether it is available on resume (needed for re-publication after session reconnect in headless_base.py line ~312 which also calls the publish function on resume).

#### F2. Resolution result type is unnamed and undefined

**What:** Decision 2 introduces a new resolution flow that returns both a manifest path and an `agent_def_dir`. The Risks section mentions "centralize recovered manifest path and recovered agent-def-dir in one resolution result type." However, neither the design nor the specs define this type, its fields, or where it lives in the module hierarchy.

**Why it matters:** The current `resolve_agent_identity()` in `runtime.py` (~line 274) returns a manifest path (via `_resolve_manifest_path_from_tmux_session()`). Tasks 2.1 and 2.2 require changing this return type. Without a named type:

- Implementers may ad-hoc the return as a tuple, a dict, or an inline dataclass, leading to inconsistency.
- The agent-identity spec (step 8: "proceed with resume/control operations using the resolved manifest path and resolved agent-definition root") implies a structured pair but doesn't name it.
- Test coverage (tasks 3.1–3.2) needs a concrete type to assert against.

**Recommendation:** Define a `@dataclass(frozen=True)` (consistent with the project's convention for internal value objects) in the design, e.g., `NameResolutionResult(manifest_path: Path, agent_def_dir: Path)`, and specify it lives in `agent_identity.py` or `runtime.py`.

### Should Fix / Important

#### F3. CLI handler resolution-order refactoring is underspecified

**What:** Decision 2 changes the resolution order for `send-prompt`, `send-keys`, and `stop-session` from "resolve `agent_def_dir` first, then identity" to "resolve identity first, then derive `agent_def_dir`." Meanwhile, `build-brain` and `start-session` keep the old order. The design and tasks acknowledge this but don't describe how the current shared resolution path in `cli.py` (~line 372) will be split.

**Why it matters:** Today, `cli.py` uses a common pattern to resolve `agent_def_dir` before dispatching any command. Changing the order for three commands while preserving it for two requires either:

- Splitting the shared resolution helper into two paths (build/start vs. control), or
- Making the shared helper mode-aware (e.g., accepting a "deferred resolution" flag), or
- Moving `agent_def_dir` resolution into each command handler individually.

Each approach has different complexity and testability implications. Without guidance, the implementer may choose an approach that complicates future commands.

**Recommendation:** Add a brief structural note to the design (or to task 2.2) describing the intended refactoring approach for the CLI handler split.

#### F4. brain-launch-runtime spec lacks a send-prompt success scenario

**What:** The brain-launch-runtime delta spec has success scenarios for `stop-session` and `send-keys` (name-based, omitting `--agent-def-dir`), but the only `send-prompt` scenario is a *failure* case (missing tmux pointer). There is no success scenario for `send-prompt` without explicit `--agent-def-dir`.

**Why it matters:** `send-prompt` is the most frequently invoked control verb. Omitting its happy-path scenario creates an asymmetry that could lead to incomplete test coverage. The existing `stop-session` and `send-keys` scenarios are not substitutes because `send-prompt` has additional semantics (prompt text, turn management).

**Recommendation:** Add a scenario like:

```
#### Scenario: Name-based send-prompt omits explicit agent-def-dir
- WHEN a developer runs `send-prompt --agent-identity chris --prompt "hello"`
- AND WHEN tmux session `AGENTSYS-chris` exists
- AND WHEN that tmux session publishes valid `AGENTSYS_MANIFEST_PATH` and `AGENTSYS_AGENT_DEF_DIR` values
- THEN the runtime resumes and delivers the prompt without requiring explicit `--agent-def-dir`
```

#### F5. Backward-compatibility scenario for legacy sessions is missing

**What:** The design's Risks section notes "[Older tmux-backed sessions lack `AGENTSYS_AGENT_DEF_DIR`] → Name-based control without explicit `--agent-def-dir` will fail clearly." However, the specs have no scenario covering the combined case: "legacy session + explicit `--agent-def-dir` provided" to confirm the override still works for pre-change sessions.

**Why it matters:** During migration, operators will have a mix of old sessions (without `AGENTSYS_AGENT_DEF_DIR`) and new sessions. The spec scenarios cover failure when the pointer is missing and override when it's present-but-different, but not the transition case where explicit `--agent-def-dir` rescues a legacy session. This is likely the most common real-world scenario immediately post-deployment.

**Recommendation:** Add a scenario to the agent-identity spec:

```
#### Scenario: Explicit agent-def-dir overrides missing tmux pointer for legacy sessions
- WHEN a caller provides `--agent-identity gpu --agent-def-dir /abs/agents`
- AND WHEN tmux session `AGENTSYS-gpu` exists but lacks `AGENTSYS_AGENT_DEF_DIR`
- THEN the runtime uses `/abs/agents` as the effective agent-definition root
- AND THEN the operation proceeds without error
```

### Nice To Have

#### F6. Task 1.2 validation helper could be named in the design

**What:** Task 1.2 says "Add shared validation/helpers for reading `AGENTSYS_AGENT_DEF_DIR` from tmux session environment." The design's Decision 3 prescribes strict validation (absolute, non-blank, existing directory) but doesn't name the helper or specify its module location.

**Why it matters:** The agent-identity spec already has detailed validation steps (step 5). Naming the helper (e.g., `validate_tmux_agent_def_dir()`) and placing it alongside the existing `_resolve_manifest_path_from_tmux_session()` in `runtime.py` would improve traceability from spec → design → code.

**Recommendation:** Consider adding the helper name and module location to the design or tasks.

#### F7. Consider whether `AGENTSYS_AGENT_DEF_DIR` needs re-publication on resume

**What:** The headless backend's `_publish_tmux_manifest_pointer()` is called not only at bootstrap but also at resume (headless_base.py ~line 312). On resume, the `AGENTSYS_MANIFEST_PATH` value is already available from the persisted session state. But `agent_def_dir` is (by design decision) not persisted in the manifest.

**Why it matters:** If the tmux session is still alive but the env var was somehow lost (tmux server restart, manual env clearing), re-publication on resume would be a useful self-healing mechanism. But the value needs to be available at resume time, which circles back to F1 (the carrier question).

**Recommendation:** The design should clarify whether resume re-publishes `AGENTSYS_AGENT_DEF_DIR` and, if so, where the value comes from at resume time. If not, document that loss of the tmux env var requires explicit `--agent-def-dir` to recover.

## Developer Disposition Of Findings

#### F1. Publication data-flow plumbing is unspecified

Decision: `Accept`, but not with manifest persistence.

Rationale: The review is right that the current design does not yet name the carrier that gets `agent_def_dir` from the CLI/runtime entry points down to tmux-env publication. I do not want to solve that by persisting a new `agent_def_dir` field in the session manifest or by redefining `LaunchPlan` as a persisted metadata carrier for this change. The better fit for the existing code is to thread the already-resolved `agent_def_dir` explicitly through `start_runtime_session()` / `resume_runtime_session()` into the tmux-backed backend session constructors and publish from backend-held state there.

#### F2. Resolution result type is unnamed and undefined

Decision: `Accept`, with refinement.

Rationale: The review is right that the change should not leave this as an ad-hoc tuple or dict. I do not think we need a brand new resolution type, though, because `runtime.py` already has `AgentIdentityResolution` for this exact call path. The change should extend that existing dataclass to carry the recovered `agent_def_dir` rather than inventing a second sibling type.

#### F3. CLI handler resolution-order refactoring is underspecified

Decision: `Accept`.

Rationale: The behavior difference between build/start flows and name-based control flows is real, and the design should spell out how `cli.py` will reflect that. I agree the change should use split resolution paths instead of hiding the difference behind a single mode-flagged helper.

#### F4. brain-launch-runtime spec lacks a send-prompt success scenario

Decision: `Accept`.

Rationale: `send-prompt` is the primary operator verb, and the spec should cover the happy path explicitly rather than implying it from `send-keys` and `stop-session`.

#### F5. Backward-compatibility scenario for legacy sessions is missing

Decision: `Accept`.

Rationale: Even though the change is small, this migration case is the most likely operator path immediately after rollout: an older tmux session that lacks `AGENTSYS_AGENT_DEF_DIR`, rescued by an explicit `--agent-def-dir` override.

#### F6. Task 1.2 validation helper could be named in the design

Decision: `Defer`.

Rationale: The review is helpful here, but I do not think helper naming needs to be locked into the design artifact. The design should define the behavior and placement constraints; the concrete helper name can stay an implementation choice unless that becomes confusing during implementation.

#### F7. Consider whether `AGENTSYS_AGENT_DEF_DIR` needs re-publication on resume

Decision: `Partial accept`.

Rationale: I agree the design should state the resume behavior explicitly. I do not want to expand this change into manifest persistence or a stronger self-healing guarantee than the current source-of-truth model supports. The intended contract should be: when resume reaches backend construction with an effective `agent_def_dir` available, tmux-backed backends re-publish `AGENTSYS_AGENT_DEF_DIR` alongside `AGENTSYS_MANIFEST_PATH`; when name-based fallback cannot recover the env var and no explicit override is provided, the command fails before resume by design.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should `agent_def_dir` be plumbed from CLI resolution to the backend tmux-env publication point?

- Decision: `Blocking`

#### Why this matters

- The design prescribes publishing `AGENTSYS_AGENT_DEF_DIR` into tmux session environment at launch time, but the value is resolved in `cli.py` and consumed in backend code (`headless_base.py`, `cao_rest.py`). No existing carrier object bridges this gap.
- The choice affects whether the value is available on resume (re-publication) and whether it touches the manifest schema (explicitly rejected by the design).

#### Examples

- **Start-session flow:** CLI resolves `agent_def_dir=/home/user/.agentsys/agents`, builds brain, starts session → backend needs this value to call `tmux set-environment ... AGENTSYS_AGENT_DEF_DIR /home/user/.agentsys/agents`.
- **Resume flow:** Backend re-publishes tmux env vars on reconnect (`_publish_tmux_manifest_pointer` is called on resume in headless_base.py) → where does `agent_def_dir` come from if not in the manifest?

#### Why this question arises in code

- `cli.py` resolves `agent_def_dir` (~line 372–380) and passes it to `start_runtime_session()`.
- `RuntimeSessionController` builds a `LaunchPlan` but `LaunchPlan` currently has no `agent_def_dir` field.
- `headless_base._publish_tmux_manifest_pointer()` (~line 297) publishes env vars but only has access to `self._plan` (the `LaunchPlan`) and `self._manifest_path`.
- On resume, the same function is called but the original `agent_def_dir` is not available from any persisted state.

```python
# headless_base.py ~line 297 (current)
def _publish_tmux_manifest_pointer(self) -> None:
    launch_env = {
        "AGENTSYS_MANIFEST_PATH": str(self._manifest_path),
        "AGENTSYS_TOOL": self._plan.tool_name,
        # ... where does AGENTSYS_AGENT_DEF_DIR come from?
    }
```

#### Options

**Option A: Add `agent_def_dir` to `LaunchPlan` as a metadata field**

- Code impact:
  - Add `agent_def_dir: str` field to `LaunchPlan` (likely in `launch_plan.py` or `models.py`).
  - `RuntimeSessionController` populates it from the CLI-provided value.
  - Backends read `self._plan.agent_def_dir` at publication time.
  - On resume, `LaunchPlan` is reconstructed from persisted manifest + brain manifest — `agent_def_dir` would need to be persisted somewhere for resume reconstruction.
  ```python
  # launch_plan.py
  @dataclass(frozen=True)
  class LaunchPlan:
      # ... existing fields ...
      agent_def_dir: str  # absolute path to agent-definition root
  ```
- User-facing difference:
  - None (internal plumbing).
- Pros:
  - Clean carrier — `LaunchPlan` already flows from CLI to backend.
  - Explicit field is greppable and type-checked.
- Cons:
  - Resume path requires `agent_def_dir` to be persisted somewhere (session manifest or alongside it), partially contradicting the "avoid manifest schema churn" non-goal.
  - `LaunchPlan` is currently a transient launch-time object; adding persistent metadata changes its semantics slightly.

**Option B: Inject `agent_def_dir` into `LaunchPlan.env` dict as `AGENTSYS_AGENT_DEF_DIR`**

- Code impact:
  - When building the `LaunchPlan`, inject `AGENTSYS_AGENT_DEF_DIR=<abs path>` into `plan.env`.
  - Backends already propagate `plan.env` into tmux session environment, so no backend changes needed for initial publication.
  - On resume, `plan.env` is reconstructed from the redacted launch plan in the session manifest — env vars may be filtered/redacted, so the value might not survive.
  ```python
  # In cli.py or runtime.py during launch plan construction:
  plan_env["AGENTSYS_AGENT_DEF_DIR"] = str(agent_def_dir)
  ```
- User-facing difference:
  - None.
- Pros:
  - Zero structural change — piggybacks on existing `plan.env` propagation.
  - Backends already publish `plan.env` to tmux, so `AGENTSYS_AGENT_DEF_DIR` appears automatically.
- Cons:
  - Relies on `plan.env` not being filtered/redacted for this key. The existing code redacts credentials from persisted launch plans; need to confirm `AGENTSYS_AGENT_DEF_DIR` would survive.
  - Semantically mixes a metadata pointer with credential/tool env vars.
  - Resume re-publication may fail if the redacted manifest drops non-credential env vars.

**Option C: Thread `agent_def_dir` through `RuntimeSessionController` as an explicit constructor parameter**

- Code impact:
  - Add `agent_def_dir: Path` to `RuntimeSessionController.__init__()`.
  - Store as `self._agent_def_dir` (following the `m_` prefix convention for stateful controllers, or without prefix if the controller doesn't follow that convention).
  - Pass to backends when calling `_publish_tmux_manifest_pointer()`.
  - On resume, `RuntimeSessionController` must receive `agent_def_dir` from the caller — which loops back to the recovery problem (tmux env → CLI → controller → backend).
- User-facing difference:
  - None.
- Pros:
  - Explicit parameter, clear ownership.
  - No semantic change to `LaunchPlan` or `plan.env`.
- Cons:
  - Requires signature changes across the controller → backend chain.
  - Resume path still needs the value from somewhere external.

> **PROPOSED: Option A (add `agent_def_dir` to `LaunchPlan` as a metadata field), with the acceptance that the value must also be persisted alongside the session manifest for resume re-publication.**
> Rationale: `LaunchPlan` already flows from CLI → `RuntimeSessionController` → backend and is the natural carrier. The resume concern is real but bounded: the design already accepts that `AGENTSYS_AGENT_DEF_DIR` is published into tmux env (which itself is volatile), so persisting the value in the session manifest's `launch_plan` section (even if redacted for credentials) is a minimal, targeted schema addition that doesn't conflict with the "avoid manifest schema churn" non-goal (that non-goal targets adding a top-level `agent_def_dir` field to the manifest payload, not a launch-plan metadata entry). Option B is viable but fragile due to env-var redaction uncertainty. Option C adds unnecessary signature churn.
>
> **DECISION: Option C, refined — thread `agent_def_dir` explicitly through the runtime start/resume path into tmux-backed backend session objects, and publish from backend-held state rather than persisting it in `LaunchPlan` or the session manifest.**
> Rationale: `start_runtime_session()` and `resume_runtime_session()` already compute or receive the effective `agent_def_dir`, and those are the only entry points that construct tmux-backed backend sessions. Passing the value directly to `_create_backend_session()` and then into `HeadlessInteractiveSession` / `CaoRestSession` keeps the carrier local to the publication site, avoids manifest or launch-plan schema churn, and still allows re-publication on resume whenever control successfully reaches backend construction. If name-based fallback cannot recover `AGENTSYS_AGENT_DEF_DIR`, the command should fail before resume, which is already the intended behavior of this change.

### Q2) Should the design define the resolution result type or leave it to implementation?

- Decision: `Deferrable`

#### Why this matters

- Tasks 2.1–2.2 require changing the return type of `resolve_agent_identity()` to carry both `manifest_path` and `agent_def_dir`. Without a named type, implementers may use inconsistent representations (tuple, dict, ad-hoc dataclass).
- The project convention is `@dataclass(frozen=True)` for internal value objects.

#### Examples

- **Current:** `resolve_agent_identity()` returns a manifest path (or loads a manifest directly).
- **Proposed:** needs to return `(manifest_path, agent_def_dir)` pair.

#### Why this question arises in code

- `runtime.py` `resolve_agent_identity()` (~line 274–306) currently calls `_resolve_manifest_path_from_tmux_session()`.
- The design's Decision 2 lists a 5-step flow that recovers both values but doesn't specify the return type.
- The Risks section mentions "centralize recovered manifest path and recovered agent-def-dir in one resolution result type" but doesn't name it.

#### Options

**Option A: Define a `@dataclass(frozen=True)` in the design**

- Code impact:
  ```python
  @dataclass(frozen=True)
  class TmuxSessionResolution:
      manifest_path: Path
      agent_def_dir: Path
  ```
  Placed in `agent_identity.py` or `runtime.py`.
- User-facing difference: None.
- Pros:
  - Consistent with project convention; type-checked by mypy.
  - Design-to-code traceability.
- Cons:
  - Slightly more prescriptive than typical OpenSpec design artifacts.

**Option B: Leave to implementation (name it in tasks only)**

- Code impact: Implementer chooses representation.
- User-facing difference: None.
- Pros:
  - Keeps design at appropriate abstraction level.
- Cons:
  - Risk of inconsistent implementation.

> **PROPOSED: Option A — define the type in the design.**
> Rationale: The project enforces `mypy --strict` and uses `@dataclass(frozen=True)` for value objects throughout `brain_launch_runtime/`. Naming the type in the design (even briefly, e.g., a one-line note under Decision 2) costs nothing and prevents the implementer from using an untyped tuple or dict. The existing `AgentIdentityNormalization` in `agent_identity.py` sets the precedent for named resolution result types.
>
> **DECISION: Accept the need for an explicit typed result, but use the existing `AgentIdentityResolution` dataclass rather than introducing a second resolution type.**
> Rationale: The current code already has `AgentIdentityResolution` in `runtime.py` as the return type for `resolve_agent_identity()`. Extending that dataclass with an optional `agent_def_dir: Path | None` field keeps one resolution object flowing through the control path, avoids type proliferation, and addresses the review's concern about ad-hoc tuples or dicts.

### Q3) How should the CLI handler split be structured for control commands vs. build/start commands?

- Decision: `Deferrable`

#### Why this matters

- Today, `cli.py` (~line 372) has a shared resolution pattern: resolve `agent_def_dir` → resolve identity → dispatch. The design changes the order for `send-prompt`/`send-keys`/`stop-session` but keeps the old order for `build-brain`/`start-session`.
- Without guidance, the implementer may duplicate resolution logic or introduce a confusing conditional path in the shared helper.

#### Examples

- **Build-brain:** always needs `agent_def_dir` upfront (no tmux session exists yet).
- **Stop-session with name:** should resolve tmux session first, then derive `agent_def_dir`.
- **Stop-session with manifest path:** should keep the old order.

#### Why this question arises in code

- `cli.py` line ~372–380 resolves `agent_def_dir` via a shared pattern used by all commands.
- Changing the order for three commands requires splitting or parameterizing this shared path.

#### Options

**Option A: Split into two resolution helpers**

- Code impact:
  - `_resolve_agent_def_dir_eager()` for build-brain/start-session (current behavior).
  - `_resolve_agent_def_dir_deferred(agent_identity, explicit_override)` for control commands (identity-first, tmux-fallback).
  - Each command handler calls the appropriate helper.
- User-facing difference: None.
- Pros:
  - Clear separation; each helper is independently testable.
  - No conditional logic pollution in a shared function.
- Cons:
  - Some code duplication for the explicit-override-present case.

**Option B: Parameterize the shared resolver with a mode flag**

- Code impact:
  - `_resolve_agent_def_dir(mode="eager" | "deferred", agent_identity=..., explicit_override=...)`.
  - Control commands pass `mode="deferred"`.
- User-facing difference: None.
- Pros:
  - Single entry point; less duplication.
- Cons:
  - Mode flag adds complexity; harder to reason about.

> **PROPOSED: Option A — split into two helpers.**
> Rationale: The existing `cli.py` already has per-command dispatch logic; adding a separate resolution helper for control commands keeps each path simple and testable. The `mode` flag approach (Option B) would add branching complexity to a function that is already doing path/env/default resolution. The split also makes the behavioral difference between build/start and control commands visible at the call site, which aids code review and onboarding.
>
> **DECISION: Option A — split the CLI resolution paths.**
> Rationale: The behavior really is different between build/start or manifest-path flows and name-based control flows. Two small helpers, or equivalent explicit per-command structure, keep that difference visible at the call sites and easier to test than a shared resolver with a mode flag.
