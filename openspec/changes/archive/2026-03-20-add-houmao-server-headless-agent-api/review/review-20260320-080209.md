## Summary

- The additive route split is directionally correct: keeping `/sessions/*`, `/terminals/*`, and `/houmao/terminals/{terminal_id}/*` intact while adding a smaller `/houmao/agents/*` surface matches the current app/service layering in `src/houmao/server/app.py` and `src/houmao/server/service.py`.
- Reusing runtime-owned headless helpers is the right architectural seam; the current headless runtime already centralizes turn artifacts, `session_id` handling, and tmux-backed execution in `src/houmao/agents/realm_controller/backends/headless_base.py` and `src/houmao/agents/realm_controller/backends/headless_runner.py`.
- The main gaps are not route naming but authority semantics: the change still needs an explicit persisted headless launch record, an explicit persisted active-turn record, and defined restart reconciliation rules.
- The native headless launch request is still too vague relative to the current pair workflow, which already separates pair-owned install from launch and uses different concepts for `agent_source`, installed profile names, and runtime launch-plan inputs.

## Findings

### Must Fix / Blocking

1. **The change promises restart-safe headless turn control without specifying any persisted active-turn authority, so `/turns`, `/interrupt`, and single-active-turn gating become undefined after a server restart.**

The design requires server restart rebuild for headless agents and also requires one-active-turn enforcement plus best-effort interrupt (`design.md:147-149`, `specs/houmao-server-agent-api/spec.md:89-91`, `specs/houmao-server-agent-api/spec.md:120-136`). But the reusable runtime state currently persists only `headless.session_id`, `turn_index`, `working_directory`, and tmux session name (`src/houmao/agents/realm_controller/boundary_models.py:151-164`, `src/houmao/agents/realm_controller/runtime.py:1094-1116`). Actual interrupt/terminate control is tied to in-memory runner fields such as `_active_process` and `_active_tmux_window_id` (`src/houmao/agents/realm_controller/backends/headless_runner.py:44-48`, `src/houmao/agents/realm_controller/backends/headless_runner.py:90-170`, `src/houmao/agents/realm_controller/backends/headless_runner.py:311-359`). After restart, the server can rebuild the session identity, but not the active turn authority it needs to reject overlapping turns or target `/interrupt` correctly.

> **DECISION: Accept.**
> Rationale: The current artifacts still do not define any persisted server-owned active-turn authority or restart reconciliation rule for native headless turns. That gap is real. The follow-up artifact revision should add a dedicated active-turn record under the native headless authority store, keyed by tracked-agent id and current turn id, so restart-time single-active-turn gating and `/interrupt` targeting do not depend on in-memory runner fields.

2. **The headless managed-agent registry is specified as “server-owned,” but the design never defines the persisted launch-record schema or storage root needed to satisfy that requirement.**

Current server authority is explicitly registration-file based: `houmao-server` persists delegated launches under `server_root/sessions/<session_name>/registration.json`, reloads them by globbing that directory, and drops anything without a real `terminal_id` (`src/houmao/server/config.py:138-184`, `src/houmao/server/service.py:405-469`, `src/houmao/server/tui/registry.py:63-79`, `src/houmao/server/tui/registry.py:123-174`). The change correctly says headless authority must be separate from that terminal-centric bridge (`design.md:68-79`, `specs/houmao-server/spec.md:19-34`), but it stops short of defining the replacement file shape, key, lifecycle cleanup, or boot-time validation rules. Without that contract, the startup rebuild requirement is not implementable in a way that preserves the current “explicit server-owned authority” boundary.

> **DECISION: Accept.**
> Rationale: The revised design now says headless authority is server-owned and native, but it still does not define the concrete persisted launch-record schema or storage root that makes that authority real. We should add a dedicated server-owned managed-agent authority store rather than leaving headless authority implied by manifest pointers alone.

### Should Fix / Important

3. **The native headless launch request is still too vague to implement consistently across `houmao-server` and `houmao-srv-ctrl`.**

The spec currently says `POST /houmao/agents/headless/launches` accepts “tool identity, provider or agent source selection, working directory, and optional identity hints” (`specs/houmao-server-agent-api/spec.md:37-39`, `design.md:147-149`, `design.md:209`). That is not enough to map cleanly onto the current pair workflow, which already separates pair-owned install (`POST /houmao/agent-profiles/install`) from launch (`src/houmao/server/app.py:251-262`, `src/houmao/server/service.py:246-307`), and where `houmao-srv-ctrl launch` currently delegates `--agents` as an installed profile name rather than an agent source (`src/houmao/srv_ctrl/commands/launch.py:53-80`, `src/houmao/srv_ctrl/commands/runtime_artifacts.py:51-203`). The design needs one normative request model with allowed field combinations and explicit validation errors, otherwise implementation will drift into ad hoc CLI-specific branching.

> **DECISION: Accept.**
> Rationale: The launch request remains too vague for a stable server and CLI contract. We should narrow it to a normative native request model around resolved runtime inputs and explicit validation rules, with any higher-level `--agents` or install convenience translation handled by `houmao-srv-ctrl` rather than by the raw HTTP route.

### Nice To Have

4. **The shared `/history` surface is named, but its retention semantics are still ambiguous enough to invite cross-transport mismatch.**

Current terminal history is bounded, in-memory recent transition history only (`src/houmao/server/models.py:280-285`, `src/houmao/server/service.py:477-481`, `src/houmao/server/tui/tracking.py:159-166`, `src/houmao/server/tui/tracking.py:748-751`). The new shared route adds `GET /houmao/agents/{agent_ref}/history`, but the artifacts do not yet say whether that route is likewise ephemeral/coarse or whether headless agents should project durable turn history into it (`design.md:47-52`, `design.md:112-123`, `specs/houmao-server-agent-api/spec.md:6-17`). Clarifying that now will keep the new route aligned with existing server behavior and avoid turning it into a second per-turn log surface beside `/turns/{turn_id}`.

> **DECISION: Accept.**
> Rationale: `/history` should not quietly become a second durable turn log for headless agents. In v1 it should stay a bounded coarse recent-history surface across both transports, while durable headless detail remains on `/turns/{turn_id}`.

## Design Alignment With Existing System

- Existing pattern(s) observed: `src/houmao/server/app.py` keeps FastAPI routes thin, `src/houmao/server/service.py` owns lifecycle and registry logic, `src/houmao/server/models.py` defines strict public contracts, and `src/houmao/server/tui/registry.py` reloads authority from explicit server-owned files instead of broad discovery.
- Existing pattern(s) observed: headless execution and artifact parsing already live under `src/houmao/agents/realm_controller/backends/headless_base.py` and `src/houmao/agents/realm_controller/backends/headless_runner.py`, while runtime manifest/state recovery is centralized in `src/houmao/agents/realm_controller/runtime.py`.
- Proposed design alignment: the route split and “reuse runtime helper seam” fit the codebase well, but the persistence plan needs to look more like a first-class server registry than a loose collection of manifest pointers. A native implementation should introduce a dedicated managed-agent storage contract under the server-owned state tree, keep route handlers thin, put alias resolution and boot reconciliation in service/registry classes, and treat runtime manifests/artifacts as evidence that is read through server authority rather than as the authority itself.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Where should the server-owned headless authority record live, and what should it key on?

- Decision: `Blocking`

#### Why this matters

- The current server can only rebuild what it has explicitly persisted under its own root.
- The change requires headless rebuild after restart without broad filesystem discovery.

#### Examples

- A headless agent launched through `POST /houmao/agents/headless/launches` survives a server restart and must reappear in `/houmao/agents`.
- A stale manifest still exists on disk, but the server should not re-admit it unless there is matching server-owned authority.

#### Why this question arises in code

- Current authority is terminal-registration based and stored under `sessions/<session_name>/registration.json`.
- That path is coupled to `terminal_id` admission and TUI reload rules.

```markdown
server_root/
  sessions/<session_name>/registration.json   # current TUI authority
  state/...                                   # available for native server-owned state
```

#### Options

**Option A: Reuse `sessions/<key>/registration.json` for both TUI and headless authority.**

- Code impact:
  - Extend `HoumaoRegisterLaunchRequest`-style records with optional headless-only fields and relax the current `terminal_id` expectations in registry loading.
- User-facing difference:
  - One storage family backs all managed agents, but the record shape becomes mixed and more ambiguous.
- Pros:
  - Fewer new paths.
  - Reuses some existing loading scaffolding.
- Cons:
  - Works against the current TUI-specific semantics in `src/houmao/server/tui/registry.py`.
  - Makes it easier to accidentally reintroduce fake `terminal_id` assumptions.

**Option B: Add a dedicated managed-agent authority store under the server-owned state tree, keyed by server-owned tracked-agent id.**

- Code impact:
  - Introduce a new headless launch-record schema plus loader/reconciler owned by `houmao-server`.
  - Keep `sessions/<session_name>/registration.json` as TUI compatibility-only authority.
- User-facing difference:
  - The API gains a cleaner server-owned identity boundary that does not depend on TUI naming.
- Pros:
  - Matches the design goal of separating TUI registration from headless authority.
  - Fits the existing `app -> service -> registry` layering and keeps boot recovery explicit.
- Cons:
  - Requires new storage and migration logic for the server runtime.

> **PROPOSED: Use a dedicated managed-agent authority store under the server-owned state tree, keyed by server-owned tracked-agent id.**
> Rationale: The current registry loader is intentionally terminal-centric and rejects records without a real `terminal_id` (`src/houmao/server/tui/registry.py:123-174`), while the design explicitly says headless agents must not be forced into fake terminal semantics (`design.md:21-24`, `design.md:74-79`). A separate server-owned record keeps authority explicit, lets restart rebuild remain deterministic, and avoids contaminating the existing TUI compatibility bridge with headless-only invariants.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Use a dedicated managed-agent authority store under the server-owned state tree, keyed by server-owned tracked-agent id. Keep `sessions/<session_name>/registration.json` TUI-only. The same per-agent subtree should also carry the persisted active-turn record needed for restart-safe turn gating and interrupt targeting.

### Q2) Should native headless launch accept resolved runtime inputs or high-level pair convenience inputs?

- Decision: `Blocking`

#### Why this matters

- The route cannot get a stable request/validation model until the launch responsibility boundary is clear.
- `houmao-srv-ctrl launch --headless` also cannot be specified cleanly until it knows what the server expects.

#### Examples

- A caller wants to launch Claude headless from an already prepared role/brain definition.
- A caller wants one convenience command that both installs an agent source and launches it.

#### Why this question arises in code

- The current server already exposes pair-owned install as a separate capability.
- The runtime launch-plan builder consumes resolved runtime artifacts rather than raw CAO profile names.

```markdown
Current split:
- install: agent_source + provider
- launch: delegated profile/session flow
- runtime: resolved launch plan + manifest + role package
```

#### Options

**Option A: Keep launch native and explicit by accepting resolved runtime inputs, with any install/convenience translation handled by `houmao-srv-ctrl`.**

- Code impact:
  - Define a request around fields such as `tool`, `working_directory`, and a resolved agent-definition or brain reference.
  - Keep install as a separate pre-step when needed.
- User-facing difference:
  - The raw HTTP API is lower-level and more explicit; the CLI can still offer higher-level ergonomics.
- Pros:
  - Aligns with current runtime seams and keeps launch side effects smaller.
  - Avoids mixing install and launch responsibilities inside one route.
- Cons:
  - Direct API clients must know more about Houmao-native runtime inputs.

**Option B: Let launch accept pair-style convenience inputs such as `provider` plus `agent_source` or installed profile name, and resolve/install internally.**

- Code impact:
  - The server route takes on install-or-resolve branching and more validation combinations.
- User-facing difference:
  - Simpler first-use API calls, but more hidden server-side behavior.
- Pros:
  - More convenient for operators coming from current pair workflows.
  - Makes `houmao-srv-ctrl launch --headless` look closer to the delegated TUI path.
- Cons:
  - Blurs the boundary between install and launch.
  - Risks coupling the new native API back to CAO/profile concepts it is trying to leave behind.

> **PROPOSED: Keep the HTTP launch route native and explicit, and let `houmao-srv-ctrl` provide any higher-level install/convenience translation.**
> Rationale: The repository already separates pair-owned install from launch (`src/houmao/server/app.py:251-262`, `src/houmao/server/service.py:246-307`), and the runtime launch machinery works from resolved artifacts and launch plans rather than raw delegated-session metadata (`src/houmao/srv_ctrl/commands/runtime_artifacts.py:122-173`). Keeping the route native preserves that separation, reduces side effects in the server route, and still allows the CLI to preserve operator ergonomics.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Keep the HTTP launch route native and explicit around resolved runtime inputs, and let `houmao-srv-ctrl` translate higher-level `--agents` / `--provider` convenience into that request. That preserves the current install-vs-launch separation and avoids coupling the native headless API back to CAO-profile concepts.

### Q3) What should the shared `/houmao/agents/{agent_ref}/history` route mean in v1?

- Decision: `Deferrable`

#### Why this matters

- The route exists in the proposed read surface, but its semantics affect retention, restart behavior, and client expectations.
- Headless agents already have a more natural durable inspection surface under per-turn resources.

#### Examples

- A TUI client asks for the last few coarse posture transitions.
- A headless client wants durable turn-by-turn records after a server restart.

#### Why this question arises in code

- Current terminal history is recent, bounded, and in-memory only.
- The new headless design also introduces explicit `/turns/{turn_id}` resources for durable inspection.

```markdown
Possible split:
- /history        -> coarse recent managed-agent posture
- /turns/{turn_id} -> durable per-turn status/events/artifacts
```

#### Options

**Option A: Keep `/history` as bounded coarse recent state history across both transports, and rely on `/turns/{turn_id}` for durable headless detail.**

- Code impact:
  - Reuse the existing “recent transitions” mental model for the shared surface.
  - Keep headless durability concentrated in turn records/artifacts.
- User-facing difference:
  - `/history` stays lightweight and comparable across transports.
- Pros:
  - Best match for current `houmao-server` behavior.
  - Avoids duplicating durable turn history in two APIs.
- Cons:
  - Cross-restart history may be sparse or empty.

**Option B: Make `/history` a durable cross-restart turn/history log for headless agents while keeping TUI history coarse.**

- Code impact:
  - Introduce transport-divergent semantics behind one shared route.
- User-facing difference:
  - More useful headless history, but less transport-neutral behavior.
- Pros:
  - Richer headless inspection from one route.
- Cons:
  - Works against the stated goal of a small honest cross-transport contract.
  - Risks overlap and confusion with `/turns/{turn_id}`.

> **PROPOSED: Keep `/history` as bounded coarse recent managed-agent history and use `/turns/{turn_id}` for durable headless inspection.**
> Rationale: Existing terminal history is already modeled as recent transition history rather than durable logs (`src/houmao/server/models.py:280-285`, `src/houmao/server/service.py:477-481`), and the new change already introduces explicit per-turn status/event/artifact routes (`specs/houmao-server-agent-api/spec.md:77-120`). Preserving that split keeps the shared surface honest and avoids making `/history` mean something different for headless and TUI agents.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Keep `/history` as bounded coarse recent managed-agent history across both transports, and use `/turns/{turn_id}` for durable headless status, events, and artifacts. That keeps the shared surface honest and avoids divergent semantics behind one route.
