## Summary

- **Strong narrative coherence**: Proposal, design, specs, and tasks all tell the same story. The rationale for removing public install is clear, well-grounded, and consistent across all artifacts.
- **Five delta specs cover all impacted capabilities** with concrete requirements, scenario coverage, and migration notes. The removal surface is precisely scoped.
- **Existing codebase confirms feasibility**: The `cao_rest.py` runtime-profile-rendering precedent (`render_cao_profile()` at `backends/cao_rest.py:362`) validates the launch-time projection model. Native headless resolution in `runtime_artifacts.py` already works and can be generalized.
- **The shared native resolver (Task 2.1) is under-specified for its cross-layer placement**: It must be usable from both the CLI layer (`srv_ctrl/commands/`) and the server-side control core (`server/control_core/`), but the design and tasks do not specify where it lives or how the dependency flows.
- **Provider adapter rework (Task 3.2) is the hardest task**: Current adapters consume `PreparedCompatibilityProfile` objects from the profile store. Reworking them to consume native brain-home / launch-plan data is a deep change to `_initialize_terminal()` and all `build_command()` implementations.
- **Brain-only launch has no current test coverage or code path**: `role_name` is always synthesized in `materialize_headless_launch_request()` (line 308 of `runtime_artifacts.py`), and the control core always loads a profile with a `system_prompt` field. The empty-prompt path is entirely new.

## Findings

### Must Fix / Blocking

1. **Shared native resolver placement is unspecified.** Task 2.1 introduces "one shared native launch-target resolver" but neither the design nor the task describes where the resolver code lives. The current native resolution logic is in `src/houmao/srv_ctrl/commands/runtime_artifacts.py` (CLI layer). The control core that needs to call it is in `src/houmao/server/control_core/core.py` (server layer). If the resolver lives in the CLI layer, the server cannot import it without an inverted dependency. If it lives in the server layer, the CLI must depend on the server for basic launch resolution. A shared module (e.g., under `src/houmao/agents/` or a new `src/houmao/launch/` namespace) is likely needed, but this should be specified before implementation to avoid a messy refactor.

> **DECISION: Accept.**
> Rationale: The gap is real and should be resolved before implementation. The shared resolver should live under `src/houmao/agents/` near `brain_builder.py` and launch-plan composition, not under `srv_ctrl` or `server/control_core`, so both caller layers can depend on it without inverting the dependency graph.

2. **Blueprint resolution is designed but has no existing code path or fixture coverage.** The design (Decision 2) specifies a deterministic selector resolution order: "first resolves a blueprint by name when one exists, otherwise resolves a tool-lane recipe." The current resolver (`_resolve_headless_recipe_path` at `runtime_artifacts.py:351`) only searches `brains/brain-recipes/<tool>/` and does not check `blueprints/` at all. While the existing `tests/fixtures/agents/blueprints/` directory has blueprint files (e.g., `gpu-kernel-coder-claude.yaml`), there is no code that resolves a selector to a blueprint first. The task (2.1) should clarify whether blueprint-first resolution is required for the first cut or can be deferred.

> **DECISION: Accept.**
> Rationale: The current artifacts overreach by promising blueprint-first selector resolution in the first cut without existing code or tests. The revision should narrow v1 to recipe-based native resolution for the supported pair and treat blueprint-by-name resolution as a later follow-up once the matching rules are specified.

3. **The demo spec still references `houmao-mgr cao launch --headless` which depends on `create_session()`.** The demo spec requirement "Demo startup SHALL ... launch one detached TUI session through the explicit compatibility surface `houmao-mgr cao launch --headless`" means the demo still goes through the session-backed CAO-compatible path. This is intentional (the demo shows TUI, not native headless), but it means Task 4.1 cannot be implemented until Task 3.1 is complete (the control core must already resolve native inputs at `create_session` time). The task ordering implies this but does not make the hard dependency explicit.

> **DECISION: Refine.**
> Rationale: The dependency is real, but the task order already places `3.*` before `4.1`. The later artifact revision should make that dependency explicit in the task wording rather than change the overall sequencing model.

### Should Fix / Important

4. **`role_name` is currently always populated in native headless launch.** In `materialize_headless_launch_request()` (line 308 of `runtime_artifacts.py`), `role_name` is always set to `_safe_role_name(agent_profile)` regardless of whether a role package actually exists. The design says role should be genuinely optional: "When the selected launch target has no role binding or no matching role package, that launch is still valid." The current code never checks whether a role directory exists under `roles/<name>/system-prompt.md`; it just manufactures a role name from the profile name. Task 2.3 should address this: the shared resolver must distinguish "role found and loaded" from "no role, emit empty prompt" rather than always synthesizing a name.

> **DECISION: Accept.**
> Rationale: This is a core contract gap, not a minor implementation detail. The later revision should stop synthesizing role identity from the selector string and instead model role presence as an actual resolved property so brain-only launch can map cleanly to an empty prompt.

5. **Provider adapter `build_command()` signatures are tightly coupled to `PreparedCompatibilityProfile`.** All six provider adapters in `provider_adapters.py` consume profile-shaped data (system_prompt, mcpServers, tools, allowedTools, etc.) through `PreparedCompatibilityProfile`. Task 3.2 says to "use native brain-home / launch-plan environment and launch-time compatibility sidecars rather than the compatibility home as the launch authority." This implies either: (a) the adapters get a new input shape derived from native launch data, or (b) a compatibility projection produces a `PreparedCompatibilityProfile`-shaped object from native inputs. Option (b) is lower risk and matches the design's "synthesized compatibility profile model" language, but neither the design nor the tasks explicitly commit to one approach. Clarifying this would reduce implementation ambiguity.

> **DECISION: Refine.**
> Rationale: The reviewer is right that adapter coupling matters, but the lower-risk direction is to keep the adapter surface profile-shaped and change the producer. The later revision should state that launch-time projection yields a `PreparedCompatibilityProfile`-like object or equivalent projection object from native inputs rather than pushing raw launch-plan data into every adapter directly.

6. **Tasks do not specify what happens to the `CompatibilityProfileStore` class itself.** Task 3.3 says "Remove or internalize compatibility profile-store code that only exists for the retired public install workflow, keeping only internal launch-scoped artifact generation where a provider still needs it." The profile store (`profile_store.py`, ~460 lines) does source resolution, markdown parsing, Q CLI/Kiro CLI artifact materialization, and profile index persistence. Some of that (markdown parsing, Q CLI JSON materialization) may still be needed for launch-time projection if the projection synthesizes profile-shaped artifacts. The task should indicate which parts of the store survive as internal utilities vs. which are fully removed.

> **DECISION: Refine.**
> Rationale: Not all of `CompatibilityProfileStore` should disappear mechanically. The public install, source-resolution, and index-authority responsibilities should go away, but any remaining materialization helpers needed for launch-time sidecars can survive as internal utilities after the store is broken apart.

7. **Demo migration needs tracked native agent-definition assets for the `gpu-kernel-coder` selector.** The current demo uses `scripts/demo/houmao-server-interactive-full-pipeline-demo/profiles/gpu-kernel-coder.md` as its tracked input. After migration, the demo needs tracked native assets (a recipe or blueprint for `gpu-kernel-coder` under the effective agent-definition root). The test fixtures already have `gpu-kernel-coder-claude.yaml` and `gpu-kernel-coder-codex.yaml` blueprints and corresponding recipes, but the demo spec says the demo "SHALL own or resolve the tracked native agent-definition inputs needed for pair-managed startup." It is unclear whether the demo will point at the test fixture directory (`tests/fixtures/agents/`) or ship its own agent-definition assets. This should be decided to avoid fixture coupling.

> **DECISION: Accept.**
> Rationale: The demo needs an explicit non-test native asset source. The later revision should keep the supported demo contract off `tests/fixtures/agents/` and point startup at demo-owned or otherwise non-test tracked agent-definition assets to avoid fixture coupling in a documented operator flow.

## Design Alignment With Existing System

- **Existing pattern: native headless resolution** (`runtime_artifacts.py:286-370`). The current `_resolve_headless_agent_def_dir()` + `_resolve_headless_recipe_path()` + `materialize_headless_launch_request()` chain is the closest existing pattern to the proposed shared resolver. The proposed design correctly identifies this as the foundation but broadens it with blueprint resolution and optional role semantics.

- **Existing pattern: runtime profile rendering** (`backends/cao_rest.py:362-424`). The `render_cao_profile()` function already renders compatibility profiles from role data at session time. The proposed launch-time projection model is a generalization of this precedent, which is well-aligned.

- **Existing pattern: control core profile loading** (`core.py:553-558`). The `_load_prepared_profile()` method is a thin delegation to `m_profile_store.load_profile()`. The proposed change replaces this with native resolution + launch-time projection. This is the primary coupling point that changes.

- **Misalignment: layering**. The current architecture places all native resolution logic in the CLI layer (`srv_ctrl/commands/runtime_artifacts.py`). The server layer (`server/control_core/`) has no native resolution capability; it only understands compatibility profiles. The proposed design needs native resolution in the server layer (for `create_session()` to resolve native inputs), which requires either moving the resolution code to a shared location or duplicating it. This is the main architectural tension.

- **Alignment: brain builder reuse**. The design's intent to "reuse native brain-home projection and launch-plan data" aligns with the existing `build_brain_home()` in `agents/brain_builder.py`. The `BuildRequest` dataclass already accepts all the inputs the shared resolver would produce (agent_def_dir, tool, skills, config_profile, credential_profile, recipe_path, etc.).

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Where should the shared native launch-target resolver live?

- Decision: `Blocking`

#### Why this matters

- The resolver must be importable from both `src/houmao/srv_ctrl/commands/launch.py` (CLI) and `src/houmao/server/control_core/core.py` (server). If it lands in the wrong layer, it creates a circular or inverted dependency.

#### Examples

- CLI calls `houmao-mgr launch --agents gpu-kernel-coder --provider codex` and needs the resolver to produce a `ResolvedNativeLaunchTarget`.
- Server's `create_session(agent_profile="gpu-kernel-coder", provider="codex")` also needs the same resolver to produce the same target, replacing `_load_prepared_profile()`.

#### Why this question arises in code

- `_resolve_headless_recipe_path()` lives in `src/houmao/srv_ctrl/commands/runtime_artifacts.py` (CLI layer, lines 351-370).
- `_load_prepared_profile()` lives in `src/houmao/server/control_core/core.py` (server layer, line 553).
- Both need to be replaced by the same shared resolver, but they are in different dependency trees.

#### Options

**Option A: Place the resolver in `src/houmao/agents/` (existing shared namespace)**

- Code impact:
  - New module like `src/houmao/agents/launch_resolver.py`
  - Both CLI and server already import from `houmao.agents`
- User-facing difference:
  - None (internal)
- Pros:
  - `agents/` is already the shared namespace for brain_builder, realm_controller, etc.
  - No new top-level package needed
- Cons:
  - `agents/` is getting large; resolver is conceptually launch-resolution, not agent-management

**Option B: Create a new `src/houmao/launch/` namespace**

- Code impact:
  - New package `src/houmao/launch/` with `resolver.py`, `models.py`
  - CLI and server both gain a new import dependency
- User-facing difference:
  - None (internal)
- Pros:
  - Clean separation; launch resolution is a distinct concern
  - Room for future launch-related utilities
- Cons:
  - Introduces a new top-level package for what may be a single module

> **PROPOSED: Option A — place the resolver in `src/houmao/agents/`.**
> Rationale: Both `srv_ctrl/commands/` and `server/control_core/` already import from `houmao.agents` (e.g., `brain_builder`, `realm_controller`). Adding a `launch_resolver.py` alongside `brain_builder.py` keeps the dependency graph unchanged and avoids a new top-level namespace for what is initially one module. The resolver produces inputs that feed directly into `build_brain_home()` in the same package, making co-location natural.
>
> **DECISION: Accept PROPOSED.**
> Rationale: `src/houmao/agents/` is already the shared dependency layer for launch construction and brain projection. Co-locating the resolver there keeps the dependency graph stable and avoids inventing a new namespace before we know we need one.

### Q2) Should blueprint-first selector resolution be required in the first cut, or deferred?

- Decision: `Deferrable`

#### Why this matters

- The design specifies deterministic resolution order (blueprint first, then recipe), but no code currently resolves blueprints from a selector name. Implementing blueprint resolution adds complexity to the shared resolver and requires defining how a blueprint name maps to a file under `blueprints/`.

#### Examples

- `--agents gpu-kernel-coder`: Currently resolves to `brains/brain-recipes/<tool>/gpu-kernel-coder-default.yaml`. Under the new design, should first check `blueprints/gpu-kernel-coder-*.yaml` (but which tool variant?).
- `--agents server-api-smoke-claude`: Could match `blueprints/server-api-smoke-claude.yaml` directly.

#### Why this question arises in code

- `_resolve_headless_recipe_path()` at `runtime_artifacts.py:351` only searches `brains/brain-recipes/<tool>/`.
- Blueprint files like `gpu-kernel-coder-claude.yaml` exist in `tests/fixtures/agents/blueprints/` but nothing resolves them by selector name.
- The design says "first resolves a blueprint by name when one exists" but doesn't specify the matching rule when the tool is already known from `--provider`.

#### Options

**Option A: Require blueprint-first resolution in the first cut**

- Code impact:
  - Resolver checks `blueprints/<selector>.yaml` and `blueprints/<selector>-<tool>.yaml` before falling back to recipe resolution
  - Blueprint file must be parsed to extract recipe + role bindings
- User-facing difference:
  - `--agents gpu-kernel-coder` could resolve differently if a blueprint exists
- Pros:
  - Matches the design as written
  - Blueprints become usable from session-backed launch immediately
- Cons:
  - Adds complexity to the first cut; blueprint matching rules with a known `--provider` need definition

**Option B: Defer blueprint resolution; support recipe-only resolution first**

- Code impact:
  - Resolver only checks `brains/brain-recipes/<tool>/` (generalized from current code)
  - Blueprint resolution added as a follow-up
- User-facing difference:
  - Blueprints cannot be targeted by selector name in the first cut
- Pros:
  - Simpler first cut; matches what already works
  - Blueprint matching rules can be designed separately
- Cons:
  - Partial implementation of the design; selector semantics change later

> **PROPOSED: Option B — defer blueprint resolution to a follow-up.**
> Rationale: The current demo and live suite both use recipe-based selectors (`gpu-kernel-coder`, `server-api-smoke`), and no existing caller needs blueprint-by-name resolution. Deferring keeps the first cut focused on the core goal (removing public install) without introducing an untested selector-matching rule. The design's blueprint-first resolution order can be layered in once recipe-based resolution is proven end-to-end.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Recipe-based resolution is enough to remove public install and land the native-first model. Blueprint-by-name resolution is useful, but it should not complicate the first cut until its matching rules are specified and tested.

### Q3) Should launch-time compatibility sidecars be purely ephemeral or cached for provider reuse across sessions?

- Decision: `Deferrable`

#### Why this matters

- This is raised as an open question in the design itself. If sidecars are cached, a stale cache could cause provider startup to use outdated config. If purely ephemeral, every session launch pays the materialization cost.

#### Examples

- Q CLI and Kiro CLI require on-disk JSON agent files. If a user launches two sessions with the same selector, should the second session reuse the first session's materialized JSON, or create its own?

#### Why this question arises in code

- `profile_store.py` currently persists materialized artifacts (Q CLI JSON, Kiro CLI JSON) under the compatibility home and indexes them in `profiles.json`.
- `cao_rest.py:render_cao_profile()` creates a new profile per session (ephemeral pattern).
- The two patterns are at odds: the store persists, the precedent is ephemeral.

#### Options

**Option A: Purely ephemeral under the session root**

- Code impact:
  - Sidecars written to `<session_root>/compat-sidecars/` or similar
  - No cross-session cache or index
- User-facing difference:
  - Slightly slower repeated launches (milliseconds, not seconds)
- Pros:
  - No stale-cache risk; simpler lifecycle
  - Matches `cao_rest.py` precedent
- Cons:
  - Repeated I/O for identical selectors

**Option B: Internal cache keyed by resolved native target**

- Code impact:
  - Cache keyed by `(tool, recipe_path_hash, role_hash)` or similar
  - Invalidation logic needed when source files change
- User-facing difference:
  - Marginally faster repeated launches
- Pros:
  - Avoids repeated materialization
- Cons:
  - Cache invalidation complexity; risk of stale artifacts

> **PROPOSED: Option A — purely ephemeral sidecars under the session root.**
> Rationale: The materialization cost for compatibility sidecars (writing a small JSON or Markdown file) is negligible compared to provider startup time (seconds to minutes). The `cao_rest.py` `render_cao_profile()` precedent already uses ephemeral per-session profiles successfully. Avoiding a cache eliminates stale-artifact risks and simplifies the lifecycle, which aligns with the design's goal of keeping compatibility artifacts as "ephemeral or cache-like" implementation details.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Session-root-scoped sidecars are the simpler and safer first cut. The cost is negligible relative to provider startup, and avoiding cross-session cache invalidation is more valuable than squeezing out a small amount of repeated file generation.

### Q4) Should session-backed launch expose an explicit `--agent-def-dir` flag?

- Decision: `Deferrable`

#### Why this matters

- This is raised as an open question in the design. The native headless path already supports `AGENTSYS_AGENT_DEF_DIR` as an env var override, but there is no CLI flag for it.

#### Examples

- An operator with agent definitions in `/opt/agents/` currently must set `AGENTSYS_AGENT_DEF_DIR=/opt/agents/` before launching. A `--agent-def-dir` flag would make this explicit in the command.

#### Why this question arises in code

- `_resolve_headless_agent_def_dir()` at `runtime_artifacts.py:342` checks `AGENTSYS_AGENT_DEF_DIR` then falls back to `.agentsys/agents`. No CLI flag exists.
- The demo overrides this via env var in its shell wrappers.

#### Options

**Option A: Add `--agent-def-dir` flag for the first cut**

- Code impact:
  - New click option on `launch_command()` and `cao_launch_command()`
  - Passed through to the shared resolver
- User-facing difference:
  - Explicit, discoverable override without env var
- Pros:
  - More ergonomic for operators
  - Explicit is better than implicit
- Cons:
  - Adds surface area; env var already works

**Option B: Rely on env var for the first cut; add flag later if needed**

- Code impact:
  - No CLI changes
- User-facing difference:
  - Operators must use env var
- Pros:
  - Simpler; smaller change surface
  - Env var is already documented and used
- Cons:
  - Less discoverable

> **PROPOSED: Option B — rely on the env var for the first cut.**
> Rationale: The `AGENTSYS_AGENT_DEF_DIR` env var is already the documented override mechanism used by both the headless path (`runtime_artifacts.py:345`) and the demo shell wrappers. Adding a CLI flag can be done in a follow-up without breaking anything. Keeping the first cut focused on the core removal avoids expanding the CLI surface prematurely.
>
> **DECISION: Accept PROPOSED.**
> Rationale: The env-var override is already part of the native resolution model and is sufficient for the first cut. Adding a new CLI flag would broaden the surface area of the change without helping the core install-removal contract land more cleanly.
