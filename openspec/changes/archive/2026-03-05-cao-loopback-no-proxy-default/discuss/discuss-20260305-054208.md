# Discussion: CAO Loopback No-Proxy Open Questions

- Feature/Change: `cao-loopback-no-proxy-default`
- Timestamp: `2026-03-05T05:42:08Z`
- Context: This doc captures unresolved decisions before implementing loopback no-proxy defaults. Inputs reviewed: `proposal.md`, `design.md`, `tasks.md`, delta specs under this change, and current launcher/runtime code paths.

Constraint reminder: do not modify upstream CAO sources under `extern/`; enforce behavior only in repository-owned launcher/runtime layers.
Working assumption after recorded decisions: for supported loopback CAO base URLs, loopback no-proxy is enforced by default by injecting loopback entries into `NO_PROXY`/`no_proxy`. The same loopback `NO_PROXY`/`no_proxy` injection contract applies to `brain_launch_runtime`-launched agent subprocesses (including non-CAO backends). Opt-out: set `AGENTSYS_PRESERVE_NO_PROXY_ENV=1` to preserve caller-provided `NO_PROXY`/`no_proxy` untouched (for example, to route localhost traffic through mitmproxy).

---

## Summary of recommended defaults

1) Keep no-proxy as the default for loopback CAO via `NO_PROXY`/`no_proxy` injection, with an explicit opt-out (`AGENTSYS_PRESERVE_NO_PROXY_ENV=1`) for traffic-watching proxies (Q1).
2) Keep loopback matching aligned with current supported launcher URLs for this change, with one shared helper to avoid drift (Q2).
3) Enforce loopback behavior via `NO_PROXY`/`no_proxy` injection, not specialized per-request HTTP proxy handlers (Q3).
4) For tmux env, preserve proxy vars for agent egress; inject loopback `NO_PROXY`/`no_proxy` entries by default unless preserve mode is enabled (Q4).
5) When injecting, merge+dedupe and append loopback tokens rather than overwriting; in preserve mode, do nothing to `NO_PROXY`/`no_proxy` (Q5).
6) Treat richer diagnostics for `inject` vs `preserve` mode as deferrable after behavior+tests land (Q6).
7) Apply the same loopback `NO_PROXY`/`no_proxy` injection contract to non-CAO `brain_launch_runtime` subprocess backends (for example, Codex app-server and headless CLIs), unless preserve mode is enabled.

## Q1) Should loopback `NO_PROXY` injection be mandatory, or allow an explicit preserve opt-out?

- Decision: `Blocking`

### Why this matters

- This determines the default safety contract vs operator flexibility for unusual enterprise setups.
- It affects config/API surface design and test matrix (default path + override path).

### Examples

- A developer shell has `HTTP_PROXY=http://corp-proxy:8080`; `status/start` against `http://localhost:9889` must remain direct by default.
- A developer wants to route localhost traffic through a traffic-watching proxy (for example, mitmproxy) and needs an explicit opt-out from loopback `NO_PROXY` injection.

### Why this question arises in code

- The design treats preserve opt-out as an explicit operator-facing behavior (`openspec/changes/cao-loopback-no-proxy-default/design.md`).
- Launcher has a proxy policy for spawned CAO process env (`src/agent_system_dissect/cao/server_launcher.py:352`) but probe HTTP calls still use default urllib behavior (`src/agent_system_dissect/cao/server_launcher.py:395`).
- Runtime REST paths currently expose no proxy policy knob (`src/agent_system_dissect/cao/rest_client.py:65`).

### Options

**Option A: Strict injection (no opt-out)**

- Code impact (pseudo code):

```python
if is_supported_loopback_cao_base_url(base_url):
    ensure_loopback_no_proxy_env(os.environ)  # always appends loopback tokens
return request.urlopen(request_obj, timeout=timeout)
```

- User-facing difference:
  - Loopback always bypasses proxies; users cannot intentionally proxy localhost loopback.
- Pros:
  - Smallest surface and least ambiguity.
  - Strongly prevents regression to accidental proxied loopback.
- Cons:
  - No escape hatch for special enterprise/debug scenarios.

**Option B: Default injection, with explicit preserve opt-out (`AGENTSYS_PRESERVE_NO_PROXY_ENV=1`)**

- Code impact (pseudo code):

```python
if (
    is_supported_loopback_cao_base_url(base_url)
    and os.getenv("AGENTSYS_PRESERVE_NO_PROXY_ENV") != "1"
):
    ensure_loopback_no_proxy_env(os.environ)
return request.urlopen(request_obj, timeout=timeout)
```

- User-facing difference:
  - Default remains safe/no-proxy; advanced users can opt out to preserve caller `NO_PROXY` and proxy localhost (traffic-watching).
- Pros:
  - Preserves default reliability while supporting exceptional infrastructure.
  - Keeps policy explicit instead of accidental environment coupling.
- Cons:
  - Adds one more config/behavior branch to document and test.

### Proposal (Recommended)

Adopt **Option B**: keep loopback no-proxy as default behavior by injecting loopback entries into `NO_PROXY`/`no_proxy`, and add one explicit preserve opt-out (`AGENTSYS_PRESERVE_NO_PROXY_ENV=1`) for traffic-watching and unusual enterprise setups.

### Pros / Cons (Proposal)

- Pros:
  - Aligns with the stated default contract and still supports non-standard operations.
  - Keeps "proxy usage" a conscious policy choice, not a shell-side accident.
- Cons:
  - Slightly broader implementation/test scope than strict-only mode.

> **DECISION: Accept Option B (default loopback `NO_PROXY` injection, with explicit preserve opt-out via `AGENTSYS_PRESERVE_NO_PROXY_ENV=1`).**
> Rationale: This change exists because loopback CAO traffic can be accidentally proxied when `NO_PROXY` is missing, causing flakiness (`openspec/changes/cao-loopback-no-proxy-default/proposal.md`). Using env-driven injection keeps behavior easy to understand and avoids specialized per-request proxy handlers, while preserve opt-out supports traffic-watching development proxies (mitmproxy) and other intentional proxied-loopback setups.

## Q2) How broad should loopback detection be for this change?

- Decision: `Blocking`

### Why this matters

- Detection scope directly controls where proxy bypass activates.
- Overly broad matching could unexpectedly disable proxies for non-target deployments; overly narrow matching can miss intended loopback traffic.

### Examples

- `http://localhost:9889` and `http://127.0.0.1:9889` should be classified as loopback policy scope; by default they bypass proxy via `NO_PROXY` injection, but preserve mode (`AGENTSYS_PRESERVE_NO_PROXY_ENV=1`) can allow proxied loopback.
- A remote URL like `http://cao.internal:9889` should preserve current behavior.

### Why this question arises in code

- Launcher currently hard-restricts supported base URLs to two exact values (`src/agent_system_dissect/cao/server_launcher.py:20`, `src/agent_system_dissect/cao/server_launcher.py:57`, `src/agent_system_dissect/cao/server_launcher.py:917`).
- Tasks ask for a shared detector across launcher and runtime (`openspec/changes/cao-loopback-no-proxy-default/tasks.md:3`), but runtime currently accepts any `api_base_url` string (`src/agent_system_dissect/agents/brain_launch_runtime/backends/cao_rest.py:464`).

### Options

**Option A: Match only currently supported explicit URLs**

- Code impact (pseudo code):

```python
SUPPORTED = {"http://localhost:9889", "http://127.0.0.1:9889"}
def is_supported_loopback_cao_base_url(base_url: str) -> bool:
    return normalize_base_url(base_url) in SUPPORTED
```

- User-facing difference:
  - Behavior is tightly scoped and mirrors current launcher contract.
- Pros:
  - Minimal surprise and minimal policy expansion.
  - Strong alignment with existing validation/docs.
- Cons:
  - Does not generalize to other loopback ports/hosts without future change.

**Option B: Host-based loopback detection (`localhost`, `127.0.0.1`, `::1`) regardless of port**

- Code impact (pseudo code):

```python
parts = parse.urlsplit(base_url)
return parts.hostname in {"localhost", "127.0.0.1", "::1"}
```

- User-facing difference:
  - More URLs receive automatic no-proxy bypass.
- Pros:
  - More future-proof and less fragile to port variations.
- Cons:
  - Broader behavior shift than currently specified; can surprise operators.

### Proposal (Recommended)

Adopt **Option A** for this change and centralize it in one shared helper used by launcher probes, REST client, and tmux env policy checks. Revisit broader host-based logic only in a follow-up change that updates launcher/base-url contract explicitly.

### Pros / Cons (Proposal)

- Pros:
  - Keeps scope aligned with current specs and launcher validation.
  - Reduces accidental behavior drift between components.
- Cons:
  - Follow-up change required if product direction later broadens loopback support.

> **DECISION: Accept Option A (loopback detection matches current supported explicit CAO URLs only).**
> Rationale: Launcher validation and supported URL contract are already exact-value based (`src/agent_system_dissect/cao/server_launcher.py:20`, `src/agent_system_dissect/cao/server_launcher.py:57`, `src/agent_system_dissect/cao/server_launcher.py:917`), and the change tasks call for a shared detector (`openspec/changes/cao-loopback-no-proxy-default/tasks.md:3`). Keeping exact URL scope limits behavioral expansion while still enabling the Q1 policy branch (default injection vs preserve opt-out) for those loopback URLs.

## Q3) What is the mechanism to enforce loopback no-proxy without specialized proxy handlers?

- Decision: `Blocking`

### Why this matters

- We want a simple and debuggable rule that matches common tooling: proxies are
  configured via `HTTP_PROXY`/`HTTPS_PROXY`/`ALL_PROXY`, and bypass is controlled
  via `NO_PROXY`.
- The mechanism must support an explicit opt-out for traffic-watching proxies.

### Examples

- Default: with `HTTP_PROXY` set, loopback CAO probes and REST calls should still
  reach `http://localhost:9889` directly.
- Preserve: with `AGENTSYS_PRESERVE_NO_PROXY_ENV=1`, the system should not change
  `NO_PROXY`, so a developer can intentionally proxy localhost traffic (for
  example, via mitmproxy).

### Why this question arises in code

- Launcher probes call `urllib.request.urlopen` directly (`src/agent_system_dissect/cao/server_launcher.py:419`).
- REST client calls `urllib.request.urlopen` directly (`src/agent_system_dissect/cao/rest_client.py:302`).
- Both paths currently depend on urllib's default proxy discovery from environment.

### Options

**Option A: Inject loopback `NO_PROXY/no_proxy` entries and rely on default urllib behavior**

- Code impact (pseudo code):

```python
def maybe_inject_loopback_no_proxy(env: dict[str, str], *, base_url: str) -> None:
    if not is_supported_loopback_cao_base_url(base_url):
        return
    if env.get("AGENTSYS_PRESERVE_NO_PROXY_ENV") == "1":
        return
    merged = merge_no_proxy(env.get("NO_PROXY"), env.get("no_proxy"))
    env["NO_PROXY"] = merged
    env["no_proxy"] = merged
```

- User-facing difference:
  - Default loopback reliability is independent of missing/misconfigured `NO_PROXY`,
    while preserve mode enables intentional proxied-loopback setups.
- Pros:
  - Uses standard proxy semantics; no specialized per-request proxy handler APIs.
  - One helper can be reused by launcher probes, REST client calls, and tmux env building.
- Cons:
  - Requires careful placement so injection happens before loopback HTTP calls.

**Option B: Use a dedicated no-proxy opener / ProxyHandler for loopback**

- Code impact (pseudo code):

```python
NO_PROXY_OPENER = request.build_opener(request.ProxyHandler({}))
return NO_PROXY_OPENER.open(req, timeout=timeout)
```

- User-facing difference:
  - Similar default behavior, but preserve-mode support becomes less obvious.
- Pros:
  - Localized to call sites.
- Cons:
  - Uses specialized Python proxy handler APIs (explicitly avoided in this change).
  - Easier to diverge between launcher/runtime/tmux paths over time.

### Proposal (Recommended)

Adopt **Option A**. Implement loopback no-proxy via `NO_PROXY/no_proxy` injection
and rely on default urllib behavior. Provide preserve opt-out via
`AGENTSYS_PRESERVE_NO_PROXY_ENV=1`.

### Pros / Cons (Proposal)

- Pros:
  - Simple mental model: `NO_PROXY` governs loopback bypass.
  - Avoids specialized per-request proxy handler APIs.
- Cons:
  - Requires consistent injection points across launcher probes, REST client calls, and tmux env building.

> **DECISION: Accept Option A (default loopback `NO_PROXY` injection with preserve opt-out; no ProxyHandler transport).**
> Rationale: Launcher probes and CAO REST client already rely on default `urllib.request.urlopen` behavior (`src/agent_system_dissect/cao/server_launcher.py:419`, `src/agent_system_dissect/cao/rest_client.py:302`). Injecting loopback entries into `NO_PROXY/no_proxy` makes the default behavior deterministic in proxy-heavy shells without introducing specialized proxy handler code, while `AGENTSYS_PRESERVE_NO_PROXY_ENV=1` provides an explicit escape hatch for mitmproxy and other intentional proxied-loopback workflows.

## Q4) How should tmux env enforce loopback policy without breaking agent egress proxy behavior?

- Decision: `Blocking`

### Why this matters

- CAO-backed agents often need outbound proxy settings for external network access.
- Loopback safety must target control-plane paths without disabling intended egress proxy behavior.

### Examples

- Default: tmux session should keep `HTTP_PROXY`/`HTTPS_PROXY`/`ALL_PROXY` for
  agent egress, and inject loopback tokens into `NO_PROXY/no_proxy` so localhost
  traffic bypasses proxies.
- Preserve: with `AGENTSYS_PRESERVE_NO_PROXY_ENV=1`, tmux session should preserve
  caller proxy and no-proxy variables as-is so localhost traffic can be proxied
  intentionally (for example, mitmproxy).
- Allowlisted credentials from `vars.env` and `plan.env` must still be present.

### Why this question arises in code

- Current tmux env composition starts from full caller env (`src/agent_system_dissect/agents/brain_launch_runtime/backends/cao_rest.py:1316`), then overlays credentials and launch env.
- `_start_terminal` has the CAO base URL context (`src/agent_system_dissect/agents/brain_launch_runtime/backends/cao_rest.py:1145`) needed to gate loopback-only `NO_PROXY` injection.
- Upstream CAO tmux session creation inherits process environment (`extern/orphan/cli-agent-orchestrator/src/cli_agent_orchestrator/clients/tmux.py:146`), so stripping proxy vars in runtime-managed tmux env can remove required agent egress proxy behavior.

### Options

**Option A: Control-plane-only enforcement (keep proxy vars, default `NO_PROXY` injection with preserve opt-out)**

- Code impact (pseudo code):

```python
def _compose_tmux_launch_env(plan, *, api_base_url):
    env = dict(os.environ)
    env.update(_credential_env_overlay(plan))
    env.update(plan.env)
    env[plan.home_env_var] = str(plan.home_path)
    if (
        is_supported_loopback_cao_base_url(api_base_url)
        and env.get("AGENTSYS_PRESERVE_NO_PROXY_ENV") != "1"
    ):
        merged = merge_no_proxy(env.get("NO_PROXY"), env.get("no_proxy"))
        env["NO_PROXY"] = merged
        env["no_proxy"] = merged
    # keep HTTP(S)_PROXY/ALL_PROXY for agent egress
    return env
```

- User-facing difference:
  - Default keeps agent outbound proxy behavior while enforcing loopback bypass;
    preserve opt-out keeps localhost proxying possible for traffic-watching.
- Pros:
  - Preserves agent external connectivity patterns that depend on proxies.
  - Implements loopback policy without overreaching into unrelated egress traffic.
- Cons:
  - Requires precise gating to avoid accidental `NO_PROXY` injection when preserve mode is enabled.

**Option B: Full sanitize (remove proxy vars + set `NO_PROXY`)**

- Code impact (pseudo code):

```python
env = _compose_tmux_launch_env(plan)
if (
    is_supported_loopback_cao_base_url(self._api_base_url)
    and os.getenv("AGENTSYS_PRESERVE_NO_PROXY_ENV") != "1"
):
    env = sanitize_proxy_env_for_loopback(env)
```

- User-facing difference:
  - Loopback safety is strong, but agent outbound traffic may stop using proxies.
- Pros:
  - Simple mental model for guaranteed local bypass.
- Cons:
  - Conflicts with environments where agents require proxy egress.

**Option C: No tmux policy changes; rely on caller env only**

- Code impact (pseudo code):

```python
def _compose_tmux_launch_env(plan, *, api_base_url):
    env = dict(os.environ)
    env.update(_credential_env_overlay(plan))
    env.update(plan.env)
    env[plan.home_env_var] = str(plan.home_path)
    return env
```

- User-facing difference:
  - Agent behavior mirrors caller env exactly; no repo-owned loopback guarantees inside tmux path.
- Pros:
  - Lowest implementation risk for existing proxy-dependent workflows.
- Cons:
  - Weakens consistency with loopback policy goals and increases drift risk.

### Proposal (Recommended)

Adopt **Option A**. Treat tmux as an agent-egress surface, not a control-plane
transport surface: keep proxy vars intact, and enforce loopback bypass by
injecting loopback entries into `NO_PROXY/no_proxy` by default. In preserve mode
(`AGENTSYS_PRESERVE_NO_PROXY_ENV=1`), do not modify `NO_PROXY/no_proxy`.

### Pros / Cons (Proposal)

- Pros:
  - Preserves required outbound proxy behavior for CAO-managed agents.
  - Keeps loopback policy enforcement targeted to intended control-plane behavior.
- Cons:
  - Requires clear docs so operators understand default injection vs preserve-mode impacts on localhost traffic.

> **DECISION: Accept Option A (keep proxy vars; inject loopback `NO_PROXY/no_proxy` by default; preserve opt-out via `AGENTSYS_PRESERVE_NO_PROXY_ENV=1`).**
> Rationale: Runtime tmux launch env starts from full caller env and is used to propagate credentials/env for CAO-managed agent execution (`src/agent_system_dissect/agents/brain_launch_runtime/backends/cao_rest.py:1316`, `src/agent_system_dissect/agents/brain_launch_runtime/backends/cao_rest.py:1146`), while upstream CAO tmux creation also inherits process environment (`extern/orphan/cli-agent-orchestrator/src/cli_agent_orchestrator/clients/tmux.py:146`). Stripping proxy vars at this boundary would break legitimate agent egress proxy usage. Therefore, tmux policy should preserve proxy vars and enforce loopback no-proxy via default `NO_PROXY/no_proxy` injection, with an explicit preserve opt-out for intentional proxied-loopback workflows.

## Q5) Should loopback `NO_PROXY/no_proxy` injection overwrite existing values or merge with them?

- Decision: `Blocking`

### Why this matters

- Overwriting can break operator-defined bypasses for internal domains.
- Merging incorrectly can duplicate/mangle entries and complicate debugging.
- Preserve interaction matters: when `AGENTSYS_PRESERVE_NO_PROXY_ENV=1`, the
  system must not modify `NO_PROXY`/`no_proxy`.

### Examples

- Caller env has `NO_PROXY=.corp.internal`; loopback enforcement should not remove it.
- Caller env has mixed-case duplicate loopback tokens; output should be deduplicated and stable.

### Why this question arises in code

- Launcher already uses merge semantics and deduplication via `_merge_no_proxy` (`src/agent_system_dissect/cao/server_launcher.py:888`).
- Launcher currently performs `NO_PROXY` merge regardless of other policy gates (`src/agent_system_dissect/cao/server_launcher.py:375`, `src/agent_system_dissect/cao/server_launcher.py:382`), which conflicts with preserve mode (`AGENTSYS_PRESERVE_NO_PROXY_ENV=1`) where we must not modify `NO_PROXY`/`no_proxy`.
- tmux loopback sanitization will likely need the same behavior to avoid policy drift between launcher and runtime.

### Options

**Option A: Overwrite with canonical loopback-only list**

- Code impact (pseudo code):

```python
env["NO_PROXY"] = "localhost,127.0.0.1,::1"
env["no_proxy"] = "localhost,127.0.0.1,::1"
```

- User-facing difference:
  - Simpler final value, but prior non-loopback exclusions are dropped.
- Pros:
  - Very predictable final string.
  - Small implementation.
- Cons:
  - Potentially destructive for existing enterprise configs.

**Option B: Merge + dedupe existing values, then append loopback entries**

- Code impact (pseudo code):

```python
merged = merge_no_proxy(env.get("NO_PROXY"), env.get("no_proxy"))
env["NO_PROXY"] = merged
env["no_proxy"] = merged
```

- User-facing difference:
  - In injection mode, existing operator entries remain, with guaranteed loopback coverage.
- Pros:
  - Backward-compatible with current launcher behavior.
  - Less likely to surprise users with removed exclusions.
- Cons:
  - Slightly more code/test surface than overwrite-only.

### Proposal (Recommended)

Adopt **Option B** for default injection and gate enforcement by preserve mode:
when `AGENTSYS_PRESERVE_NO_PROXY_ENV=1`, do nothing to `NO_PROXY`/`no_proxy`;
otherwise merge+dedupe and ensure loopback tokens.

### Pros / Cons (Proposal)

- Pros:
  - Consistent with existing launcher contract and docs.
  - Preserves user intent while enforcing loopback safety.
- Cons:
  - Requires shared utility extraction or careful helper reuse.

## Q6) Do we add user-visible diagnostics for effective proxy mode now?

- Decision: `Deferrable`

### Why this matters

- Better diagnostics reduce operator confusion when proxy-heavy shells behave differently than expected.
- Adding fields to structured outputs can affect downstream tooling expectations.

### Examples

- `start` succeeds under proxy-heavy shell; operator wants confirmation that loopback no-proxy was applied.
- `start` intentionally uses `AGENTSYS_PRESERVE_NO_PROXY_ENV=1` for mitmproxy;
  operator wants confirmation that preserve mode is enabled.
- Runtime support issue asks whether tmux env had loopback `NO_PROXY` injected for a specific session.

### Why this question arises in code

- Launcher results currently expose health/process fields but no explicit proxy-mode field (`src/agent_system_dissect/cao/server_launcher.py:140`, `src/agent_system_dissect/cao/server_launcher.py:152`).
- Runtime session state focuses on CAO identity/parsing metadata, not transport policy reporting (`src/agent_system_dissect/agents/brain_launch_runtime/backends/cao_rest.py:69`).

### Options

**Option A: Add explicit diagnostics fields now**

- Code impact (pseudo code):

```python
result.no_proxy_mode = "inject" | "preserve"
events.append({"kind": "diagnostic", "no_proxy_mode": no_proxy_mode})
```

- User-facing difference:
  - Easier debugging and support triage from structured outputs.
- Pros:
  - Immediate observability.
  - Faster incident triage.
- Cons:
  - Expands output/manifest contracts and test obligations.

**Option B: Defer schema changes; rely on behavior/tests/docs first**

- Code impact (pseudo code):

```python
# No contract field additions in this change.
# Validate behavior with unit tests and smoke flow under proxy-heavy env.
```

- User-facing difference:
  - No new output fields yet; users infer behavior from successful loopback health and docs.
- Pros:
  - Keeps current change tightly scoped.
  - Avoids accidental compatibility impact on tooling consumers.
- Cons:
  - Lower immediate observability for troubleshooting.

### Proposal (Recommended)

Adopt **Option B** for this change, with an assumption that behavior is validated by targeted tests and smoke/demo coverage (`tasks.md:25`). Track richer diagnostics as a follow-up if support burden remains high.

### Pros / Cons (Proposal)

- Pros:
  - Keeps implementation focused on contract behavior.
  - Avoids enlarging CLI/runtime schema surfaces mid-change.
- Cons:
  - Debugging still relies on indirect signals until follow-up lands.
