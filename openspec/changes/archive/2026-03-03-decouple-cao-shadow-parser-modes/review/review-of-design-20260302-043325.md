# Design Review: `decouple-cao-shadow-parser-modes/design.md`

## Findings

### HIGH-1: Design open question contradicts delta spec that already answers it

**Evidence:** `design.md:133` lists as an open question:
> "Should `claude + cao_only` be allowed as an explicit advanced mode, or rejected to preserve current reliability guarantees?"

However, the delta spec `specs/cao-claude-code-output-extraction/spec.md:14` already codifies behavior for this combination:
> "In `parsing_mode=cao_only`, the runtime SHALL NOT invoke Claude shadow parsing for turn gating or extraction."

And spec line 21-24 provides an explicit scenario for it:
> **Scenario: CAO-only mode does not invoke Claude shadow parser** — "WHEN a Claude CAO-backed session runs with `parsing_mode=cao_only`…"

The spec has decided that `claude + cao_only` IS allowed and has defined its behavior. The design's open question is stale — it was resolved downstream but not propagated back. This matters because the engine routing logic, validation rules (task 1.3), and test matrix all depend on this answer. An implementer following the design could build fail-fast rejection for `(claude, cao_only)` while the spec requires it to work.

**Severity justification:** Directly derails implementation of validation (task 1.3) and engine routing (task 2.3).

---

### HIGH-2: Engine interface contract has no defined error model

**Evidence:** `design.md:52` states the shared engine contract covers "readiness wait, submit, completion wait, answer extraction, and diagnostics snapshot," but defines no error return contract.

Meanwhile, all three delta specs require specific failure semantics:
- `brain-launch-runtime/spec.md:29`: "the turn SHALL fail without invoking the other mode in the same turn"
- `cao-claude-code-output-extraction/spec.md:33`: "the turn fails with a shadow-mode error"
- `cao-codex-output-extraction/spec.md:38`: "the runtime returns an explicit waiting-user-answer error"

Each mode must produce *mode-tagged, tool-specific* errors. Without an error taxonomy in the engine contract, the two engine implementations will likely diverge in error representation, making the "strict non-mixing" enforcement at the `CaoRestSession` level inconsistent and harder to test.

**Severity justification:** Non-mixing enforcement (Decision 3, task 2.4) requires the session to reason about whether an error came from the "correct" parser family. Without a structured error model, this is fragile.

---

### MEDIUM-1: Decision 4 omits `waiting-user-answer` as a lifecycle primitive

**Evidence:** `design.md:76-78` lists Codex shadow parser lifecycle primitives as:
> baseline capture, shadow status classification, answer extraction

But the Codex delta spec (`cao-codex-output-extraction/spec.md:34-40`) adds an explicit fourth requirement:
> **Requirement: Codex waiting-user-answer is explicit in shadow mode** — "SHALL fail with an explicit error rather than being classified as completed."

The Claude shadow parser already handles this (main spec line 82-83), and the design's Decision 4 claims the Codex parser follows "the same lifecycle primitives already used by Claude shadow parsing." Since `waiting-user-answer` detection is part of Claude's lifecycle, the omission from the explicit list in Decision 4 is a gap — an implementer reading only the design could miss this requirement for the Codex parser.

---

### MEDIUM-2: "Required" mode vs. "required with configurable default" ambiguity

**Evidence:** `design.md:34` says:
> "CAO-backed sessions carry a **required** `parsing_mode`"

But the brain-launch-runtime delta spec (`spec.md:12-14`) says:
> "WHEN a caller starts a CAO-backed session without specifying a parsing mode **and no default is configured** THEN the system rejects"

This implies a default CAN satisfy the requirement — the mode must be *resolved*, not necessarily *caller-provided*. The migration plan (`design.md:128`) reinforces this with "Keep a compatibility default (Codex `cao_only`, Claude `shadow_only`) during rollout."

The design should explicitly state that a tool-specific default satisfies the "required" constraint. Without this, task 1.1 implementers may build a model that demands an explicit caller value and breaks the rollout strategy.

---

### MEDIUM-3: Post-processing scope is undefined

**Evidence:** `design.md:86-88` (Decision 5) says:
> "Apply a parser-agnostic post-processing step to user-visible output in both modes."

But the design never specifies what this step includes. The specs define mode-specific output hygiene:
- Claude: ANSI stripping, prompt-line exclusion, separator-line trimming (main spec lines 63-76)
- Codex: "strips prompt/UI chrome from output" (`cao-codex-output-extraction/spec.md:30`)

These are parser-specific transformations, not "parser-agnostic." A truly parser-agnostic step (e.g., final whitespace normalization, encoding normalization) would be different from the parser-specific extraction cleanup. The design conflates two concerns:
1. Parser-specific extraction cleanup (already inside each engine)
2. Post-engine output normalization (the shared step)

This ambiguity could lead to duplicated or misplaced logic during implementation (tasks 4.1).

---

### MEDIUM-4: Codex `mode=full` output format is an uncharacterized dependency

**Evidence:** `design.md:109` lists "Codex shadow parser drift as Codex UI evolves" as a risk, but the design assumes the Codex `mode=full` output is sufficiently structured for shadow parsing without documenting what that structure looks like. The Claude shadow parser has versioned presets (`main spec:20-36`) because Claude's output format is known to change. No equivalent discovery/characterization step is mentioned for Codex.

The Codex delta spec (`cao-codex-output-extraction/spec.md:29`) requires stripping "Codex prompts, footer chrome, or spinner lines around assistant content" — but these patterns are undefined. If no Codex output samples exist, the implementation will begin with reverse-engineering rather than implementing a known contract, which is a schedule risk.

---

### LOW-1: No contract-test strategy for the engine interface

**Evidence:** `tasks.md` section 5 tests each mode independently (5.2 for `cao_only`, 5.3 for `shadow_only`), but there's no task for verifying that both engines satisfy the same shared contract. Without shared contract tests, engine behaviors can drift silently — e.g., one engine might return a different shape for diagnostics snapshots.

---

### LOW-2: No test task for provenance metadata correctness

**Evidence:** `design.md:121` (Migration Plan step 6) requires "Add parser provenance fields in turn event payloads." `tasks.md:22` (task 4.2) covers implementation. But `tasks.md` section 5 (tests) has no entry validating that provenance fields are correctly emitted in each mode. Provenance is listed as a design goal for "observability and debugging" (`design.md:23`) but has no test coverage plan.

---

### LOW-3: Open Question 3 (mode-specific timeouts) has spec implications not flagged

**Evidence:** `design.md:135` asks:
> "Do we want mode-specific timeout defaults (e.g., longer for shadow polling) or one shared default?"

The brain-launch-runtime delta spec mentions "timeout" (`spec.md:33`: "polls CAO terminal status until it becomes `idle|completed` (or timeout)") but doesn't parameterize it by mode. If the answer is "mode-specific," it requires a spec amendment. The design should flag this dependency.

---

## Open Questions

1. **Is `claude + cao_only` a supported combination?** The delta spec already defines its behavior — if the design intent is different, the spec needs correcting. If the design agrees, close the open question and remove it.

2. **What constitutes the "parser-agnostic post-processing" step?** Is it ANSI stripping? Whitespace normalization? Encoding? The design should distinguish this from parser-specific extraction cleanup to avoid duplication.

3. **Do Codex `mode=full` output samples exist?** If not, an upfront characterization task should be added before the parser implementation task (3.1), or risk should be re-rated.

4. **Should the engine interface be formalized as a Protocol/ABC before implementation?** The design mentions a "shared interface" but doesn't commit to whether this is a Python Protocol, ABC, or informal convention. This affects testability (contract tests) and refactor safety.

## Suggested Revisions

### SR-1 (addresses HIGH-1): Close the `claude + cao_only` open question

In `design.md`, **remove** from Open Questions (line 133):
> Should `claude + cao_only` be allowed as an explicit advanced mode, or rejected to preserve current reliability guarantees?

And **add** to Decision 1 (after line 38):
> Both `(claude, cao_only)` and `(claude, shadow_only)` are valid combinations; `(codex, cao_only)` and `(codex, shadow_only)` are valid combinations. All four `(tool, parsing_mode)` pairs are supported. Tool-specific defaults during rollout are `claude → shadow_only`, `codex → cao_only`.

### SR-2 (addresses HIGH-2): Add error contract to Decision 2

After `design.md:52`, add:
> Each engine method returns either a success result or a mode-tagged structured error. Mode-tagged errors carry the `parsing_mode` and `parser_family` that produced them, so `CaoRestSession` can enforce non-mixing without inspecting error internals. Error categories include at minimum: `timeout`, `extraction_failure`, `waiting_user_answer`, and `backend_error`.

### SR-3 (addresses MEDIUM-1): Add `waiting-user-answer` to Decision 4 lifecycle

In `design.md:77`, change:
```
- baseline capture
- shadow status classification
- answer extraction
```
to:
```
- baseline capture
- shadow status classification (including `waiting_user_answer` detection)
- answer extraction
- waiting-user-answer error surfacing
```

### SR-4 (addresses MEDIUM-2): Clarify "required" semantics in Decision 1

In `design.md:34`, change:
> CAO-backed sessions carry a required `parsing_mode`

to:
> CAO-backed sessions carry a resolved `parsing_mode` — either explicitly specified by the caller or derived from a tool-specific default. If neither is available, session start is rejected.

### SR-5 (addresses MEDIUM-3): Clarify post-processing scope in Decision 5

Replace `design.md:86-88` with:
> **Decision:** Each engine is responsible for its own extraction cleanup (ANSI stripping, marker removal, prompt-line exclusion). After engine extraction, apply a minimal shared normalization step (trailing whitespace trim, encoding normalization) that is identical for both modes. In `cao_only`, the shared step also sanitizes CAO-returned text that may contain residual formatting.

### SR-6 (addresses LOW-1): Add contract-test task to tasks.md

Add to section 5:
> - [ ] 5.6 Add shared engine-interface contract tests verifying both `CaoOnlyTurnEngine` and `ShadowOnlyTurnEngine` satisfy the same behavioral contract (method signatures, error shapes, provenance fields).

### SR-7 (addresses LOW-2): Add provenance test task to tasks.md

Add to section 5:
> - [ ] 5.7 Add unit tests verifying turn event provenance fields are correctly populated for each `(tool, parsing_mode)` combination.
