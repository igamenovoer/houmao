# Design Review: decouple-cao-shadow-parser-modes

## Summary

This change introduces an explicit `parsing_mode` (`cao_only` | `shadow_only`) for CAO-backed sessions, replacing the current implicit tool-based branching in `CaoRestSession`. It refactors turn handling into two mode-specific engines behind a shared interface, adds a new Codex shadow parser for `shadow_only` Codex sessions, removes all fallback/chaining behavior, and bumps the session manifest schema with no legacy compatibility path.

The change set is internally consistent: proposal, design, delta specs, and tasks all reinforce the same decisions without contradictions. The scope is well-bounded — parsing semantics only, with an explicit carve-out preserving AGENTSYS identity/addressing contracts.

## Strengths

- **Clear binary constraint.** Exactly two modes, no hybrid, no fallback. This makes the state space small and testable.
- **Strong artifact coherence.** The proposal's "What Changes" bullets map 1:1 onto design decisions, which map onto spec requirements, which map onto task items. Traceability is high.
- **Explicit non-goals.** The design explicitly rejects hybrid modes, per-turn overrides, and legacy compatibility — reducing ambiguity for implementers.
- **Consistent scenario style.** All three delta specs use the same WHEN/THEN/AND THEN structure and cover both the happy path and the failure/non-mixing path. Easy to turn into test cases directly.
- **Migration plan is sequenced sensibly.** Extract-without-behavior-change first (steps 3-4), then wire mode routing (step 5), then extend (step 6). This reduces regression risk during implementation.
- **AGENTSYS isolation is called out explicitly.** Dedicating a design decision, a spec requirement section, and a task group (5.8) to non-regression of identity/addressing contracts prevents scope creep.

## Findings

### HIGH: No guidance on what `cao_only` mode does for Claude sessions

The default mapping is `claude -> shadow_only`, but nothing prevents a caller from explicitly requesting `cao_only` for Claude. Design decision 3 and the `brain-launch-runtime` delta spec define `cao_only` behavior (use CAO status + `mode=last`), but the `cao-claude-code-output-extraction` delta spec only says "Claude shadow parser logic is not used" in `cao_only` — it does not specify what *is* used or whether runtime post-processing still applies. The design doc (decision 5) says post-processing is mandatory in `cao_only`, but this is not reflected in the Claude extraction spec.

**Risk:** An implementer working from the Claude extraction spec alone may skip post-processing for `cao_only` Claude sessions, or may not realize `cao_only` for Claude is a supported combination at all.

**Recommendation:** Add a scenario to `cao-claude-code-output-extraction/spec.md` that explicitly covers `cao_only` Claude turns: what extraction path is used (CAO `mode=last`), and that shared post-processing still applies.

**Related:** `../design.md`, `../specs/brain-launch-runtime/spec.md`, `../specs/cao-claude-code-output-extraction/spec.md`

> DECISION: For `(claude, cao_only)`, behavior matches `(codex, cao_only)`: use CAO endpoint calls with `mode=last` and do not fetch full text for runtime-side shadow parsing. In the current version this can still yield unexpected errors; treat that as an upstream limitation and assume upstream will become capable of parsing Claude Code output correctly. Product guidance should advise users to avoid `cao_only` for Claude in the current version, and the default for Claude remains `shadow_only`.

### HIGH: No definition of "shared post-processing" content or interface

Design decision 5 introduces parser-agnostic post-processing that is "mandatory" in `cao_only` mode and applied in both modes. However, no spec or task defines what this post-processing does, what its interface looks like, or where it lives. Task 4.1 says "Add shared parser-agnostic output post-processing" but gives no acceptance criteria.

**Risk:** Implementers will either skip it, implement inconsistent behavior across modes, or scope-creep it into a catch-all sanitizer.

**Recommendation:** Either add a dedicated requirement/scenario in the `brain-launch-runtime` delta spec defining post-processing behavior (at minimum: what transformations, when it runs relative to extraction, what its input/output contract is), or scope it out and note it as a follow-up.

**Related:** `../design.md`, `../specs/brain-launch-runtime/spec.md`, `../tasks.md`

> DECISION: Post-processing is not about text sanitization. In `cao_only` mode, extracted text is a passthrough — no transformation. The purpose of the shared post-processing step is to present a stable runtime-canonical contract to downstream consumers regardless of parsing mode. Specifically: (1) remap CAO-native statuses into runtime-canonical status values so downstream applications are insulated from CAO API changes (new statuses added, old ones removed, renamed), and (2) log agent-specific status for observability. Both modes run this step; downstream callers get the same behavioral contract whether `cao_only` or `shadow_only` is used. Update task 4.1 acceptance criteria accordingly and remove the "mandatory sanitization" framing from design decision 5.

### MEDIUM: Task 1.3 validation scope is ambiguous

Task 1.3 says "Add fail-fast validation for unsupported parsing modes and unsupported `(tool, parsing_mode)` combinations." The design does not enumerate which `(tool, parsing_mode)` pairs are unsupported. The default mapping implies all four combinations are possible (`claude x {cao_only, shadow_only}`, `codex x {cao_only, shadow_only}`). If some are unsupported, they need to be listed. If all are supported, task 1.3's second clause is vacuous.

**Risk:** Implementers may block valid combinations or allow invalid ones without clear guidance.

**Recommendation:** Explicitly enumerate which `(tool, parsing_mode)` pairs are supported vs. unsupported in the design doc's "Decisions" section 1. If all four are valid, remove the "unsupported combinations" language from task 1.3 and the design risks section.

**Related:** `../design.md`, `../tasks.md`

> DECISION: All four `(tool, parsing_mode)` combinations are supported: `(claude, shadow_only)`, `(claude, cao_only)`, `(codex, cao_only)`, `(codex, shadow_only)`. Task 1.3 validation should fail-fast only on unknown `parsing_mode` values (i.e., values other than `cao_only` or `shadow_only`), not on specific `(tool, parsing_mode)` pairs. Remove the "unsupported combinations" language from task 1.3 and the design risks section. Enumerate the four supported pairs explicitly in design decision 1. Optionally log a warning when a non-default `(tool, parsing_mode)` pair is selected.

### MEDIUM: Codex shadow parser has no versioning or format contract

The design risks section mentions "Codex shadow parser drift as Codex UI evolves" and mitigates with "versioned parser presets/tests." However, neither the `cao-codex-output-extraction` spec nor the tasks define what versioning looks like, how parser presets are selected, or how the parser detects/handles unknown Codex output formats.

**Risk:** The Codex shadow parser will be built against current Codex output format with no structured way to handle format changes, leading to silent extraction failures.

**Recommendation:** Add a requirement or task item covering: (a) how the parser identifies the Codex output format version it expects, and (b) what happens when the actual output does not match (explicit error vs. best-effort extraction).

**Related:** `../design.md`, `../specs/cao-codex-output-extraction/spec.md`, `../tasks.md`

> DECISION: Define an explicit Codex shadow format contract and fail-fast behavior in this change. Introduce a versioned parser preset id `codex_shadow_v1` and require `shadow_only` Codex turns to emit parser provenance (`parser_preset_id`, `format_detected`, `format_match=true|false`) in turn events. The parser first runs format detection against the expected v1 markers; if detection fails, return a structured `unsupported_output_format` error (no best-effort extraction, no silent fallback to `cao_only`). Add fixture-based tests for both known-good v1 output and unknown/changed output, and require the unknown-format path to be explicit and actionable.

### MEDIUM: No spec coverage for `cao_only` Codex turns

The `cao-codex-output-extraction` spec only covers `shadow_only` behavior. There is no delta spec for `cao_only` Codex behavior. The `brain-launch-runtime` spec mentions `cao_only` uses CAO status + `mode=last`, but this is generic — it does not address Codex-specific `cao_only` concerns (e.g., whether `mode=last` is currently reliable for Codex, whether post-processing applies).

**Risk:** Implementers may assume `cao_only` for Codex "just works" with CAO-native parsing and skip post-processing or Codex-specific handling.

**Recommendation:** Either add a brief `cao_only` section to the Codex extraction spec confirming the expected behavior, or add a note in the design doc that `cao_only` Codex is the existing baseline behavior and needs no delta spec.

**Related:** `../specs/brain-launch-runtime/spec.md`, `../specs/cao-codex-output-extraction/spec.md`

> DECISION: Add explicit `cao_only` Codex coverage to `specs/cao-codex-output-extraction/spec.md` so behavior is unambiguous. Scenario contract: for `(codex, cao_only)`, runtime uses CAO terminal polling + output fetch with `mode=last`, does not invoke any Codex shadow parser, and then applies the shared runtime post-processing/canonical status mapping before returning user-visible output. Keep `codex -> cao_only` as the default mapping and document that this path is the baseline production mode for current versions.

### LOW: Breaking schema change has no version bump strategy documented

The proposal marks the manifest schema change as **BREAKING** and the design says "bump session manifest schema for CAO payload changes." Task 1.2 says "Revise CAO session manifest schema." But there is no mention of what the new schema version identifier will be, or how the version bump mechanism works (e.g., `v1` -> `v2`, or a date-based version).

**Recommendation:** Add a note in the design or task 1.2 specifying the version bump convention (e.g., "bump `session_manifest.v1.schema.json` to `session_manifest.v2.schema.json`").

**Related:** `../proposal.md`, `../design.md`, `../tasks.md`

> DECISION: Adopt a hard-break policy while the library is unstable. Introduce `session_manifest.v2.schema.json` and emit only `schema_version: 2`; do not add migration logic, compatibility shims, fallback parsing, or downgrade support for `v1`. If updates or revised design break old manifests/flows, that is acceptable and should be fixed forward in the current code/design without preserving behavior for existing users.

### LOW: Task numbering may mislead on dependencies

Tasks are grouped into 5 sections but no dependency ordering is stated between groups. For example, group 3 (Codex shadow parser) depends on group 2 (turn engine refactor), but this is only implicit.

**Recommendation:** Add a brief note at the top of `tasks.md` stating the intended implementation order (e.g., "Groups 1-4 should be implemented in order; group 5 tests should be written alongside or immediately after their corresponding group").

**Related:** `../tasks.md`

> DECISION: Add an explicit dependency note at the top of `tasks.md` and treat groups as ordered milestones: `1 (schema+validation) -> 2 (engine split/refactor) -> 3 (Codex shadow implementation) -> 4 (mode routing + provenance/post-processing) -> 5 (tests/verification)`. Clarify that tests in group 5 should be added alongside each milestone when practical, but final completion requires all group-5 checks after group 4 is integrated. This removes ambiguity without changing task scope.

## Misuse Risks

| Risk | Scenario | Guard |
|------|----------|-------|
| **Wrong mode silently degrades output quality** | Operator sets `cao_only` for Claude, gets lower-quality extraction from CAO `mode=last` without realizing shadow parsing is disabled | Log a warning at session start when a non-default `(tool, parsing_mode)` pair is selected. Include the active mode in all turn event payloads (already planned via task 4.2). |
| **Stale manifests cause cryptic resume failures** | Operator creates sessions under old schema, upgrades runtime, tries to resume | Task 1.4 covers this. Ensure the error message includes the exact action needed (e.g., "Session manifest version X is unsupported. Start a new session."). |
| **Mode mismatch across coordinating systems** | External orchestrator creates a session with one mode but sends prompts assuming the other mode's output format | Expose `parsing_mode` in the `start-session` CLI output (alongside the already-planned agent identity). Include it in the manifest so orchestrators can validate before sending prompts. |
| **Codex `shadow_only` used before parser is mature** | Operator explicitly selects `shadow_only` for Codex when the shadow parser has limited coverage | Consider gating Codex `shadow_only` behind an explicit opt-in flag or configuration key (not just allowing it by value), with documentation noting it is less battle-tested than Claude `shadow_only`. |
| **Post-processing assumed optional** | Implementer skips post-processing for `shadow_only` because the design says it's "mandatory in `cao_only`" — implying it's optional elsewhere | Clarify in the spec that post-processing runs unconditionally in both modes, not just `cao_only`. |

## Recommended Revisions

1. **Add `cao_only` scenarios to both extraction specs** (Claude and Codex) — even if brief — to close the coverage gap for non-default mode combinations.
2. **Define the post-processing contract** as a spec requirement or explicitly defer it with a tracked follow-up item.
3. **Enumerate supported `(tool, parsing_mode)` pairs** in the design doc and align task 1.3 validation scope accordingly.
4. **Add Codex shadow parser format detection/versioning** guidance in the spec or as a task prerequisite.
5. **State task group dependencies** at the top of `tasks.md`.
6. **Include `parsing_mode` in `start-session` CLI output** alongside agent identity (minor addition to task 4.2 or a new sub-task).
