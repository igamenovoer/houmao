## Context

`add-agent-config-drafts` introduced `houmao-mgr internals config-drafts` as a smaller alternative to command-template schema output. The implementation is concise in output, but the input registry still exposes much of the full command/model override surface: model, env, mailbox, skills, prompt overlays, memo seed, gateway, and credential material can all be copied into drafts when supplied.

That still asks agents to reason about a broad option catalog. The intended abstraction is narrower: a draft id is an opinionated partial application of a full Houmao config model. It binds selected model fields to fixed values and exposes only a few required holes. Fixed values may be ordinary domain defaults, or they may be non-default choices that define the draft's lane.

## Goals / Non-Goals

**Goals:**
- Make initial config drafts expose fewer specifiable fields than the underlying data model and full project subcommands.
- Require credential as a minimal input for each initial draft that produces a credential-bearing specialist/profile config.
- Reject unsupported full-model optional override fields instead of copying them into draft YAML.
- Keep `config-drafts list` compact and honest about the minimal required intent fields.
- Preserve the distinction: drafts are concise authoring aids; full customization belongs to maintained project subcommands.

**Non-Goals:**
- Do not add a dynamic template language, field-selection DSL, or schema export.
- Do not teach config drafts to accept every model field through a different namespace.
- Do not change project catalog persistence or add config import/apply.
- Do not move credential material creation into config drafts; credential references are required, but credential CRUD/material still belongs to credential/project subcommands.

## Decisions

### Treat draft ids as model partial applications

Each draft id should define three sets:

```text
fixed fields      values generated by the draft, supplied by neither CLI nor intent
required holes    small caller-supplied values needed to complete the draft
hidden fields     valid full-model fields intentionally not accepted by this draft
```

For example, `project.easy.profile` fixes `profile_lane: easy_profile` and `source.kind: specialist`, requires `name`, `specialist`, and `credential`, and hides the rest of the launch-profile override model. If a caller needs model/env/mailbox/skills/memo/gateway customization, the skill guidance should route them to the full project subcommand.

Alternative considered: keep accepting all optional fields but document which ones are common. That keeps the registry large and re-creates the command-template problem in a new shape.

### Require credential references, not credential material

Credential should be required because generated drafts should not silently invent or guess the credential-bearing path. The field is a reference:

- `project.easy.specialist`: `credential` renders as the specialist credential reference.
- `project.easy.profile`: `credential` renders as a launch/profile auth default.
- `project.agents.launch-profile`: `credential` renders as a launch/profile auth default.

Config drafts should not accept `api_key`, `base_url`, OAuth tokens, auth JSON paths, or other material fields. Those remain credential creation concerns handled by maintained credential/project commands.

Alternative considered: preserve the previous `<specialist>-creds` derived default. That is convenient, but it hides a mutation-relevant naming decision and makes the draft less explicit.

### Keep the initial allowed field sets tiny

Initial allowed fields should be:

```text
project.easy.specialist:
  name, tool, credential

project.easy.profile:
  name, specialist, credential

project.agents.launch-profile:
  name, recipe, credential
```

The resulting YAML should still use concrete domain shapes and fixed draft values. For specialists, that likely includes `config_kind: project.easy.specialist`, `setup: default`, and `launch.prompt_mode: unattended` for the supported easy tools. For profile-like drafts, that includes fixed `profile_lane`, fixed `source.kind`, caller-provided `source.name`, and `defaults.auth` from credential.

Alternative considered: allow prompt text/file for specialists. Existing specialist creation allows an empty prompt, and prompt authoring tends to be large natural-language content, so it is better kept out of the minimal draft unless a later change scopes a prompt-specific draft.

### Use unsupported-field blockers as the guardrail

The existing validation path already reports unsupported fields. Narrowing the registry field definitions is enough to make full-model extras fail clearly. Tests should assert representative hidden fields are rejected so the registry does not drift back into full command coverage.

Alternative considered: silently ignore hidden fields. That would make agent mistakes harder to detect and could lead to false confidence that a requested customization was captured.

## Risks / Trade-offs

- Less convenient one-shot customization -> Mitigate through skill text that points full customization to project subcommands.
- Generated drafts may look too sparse -> Mitigate by explaining that sparseness is intentional: fixed fields and required holes only.
- Credential requirement may feel repetitive for profile drafts sourced from specialists -> Mitigate by treating it as an explicit launch/auth default decision; future drafts can specialize further if needed.
- Active-change layering with `add-agent-config-drafts` -> Mitigate by keeping this change focused on the same capability and applying it before archiving or syncing the earlier draft behavior.

## Migration Plan

1. Narrow config-draft registry fields to the minimal allowed sets.
2. Update draft renderers so hidden optional fields are no longer read and credential is rendered in each supported draft.
3. Update list output and tests for required intent keys.
4. Update packaged skill guidance to say config drafts are minimal opinionated drafts and full customization uses project subcommands.
5. Run focused config-draft tests, system-skill text tests, and the standard lint/typecheck/unit validation expected for the touched code.
