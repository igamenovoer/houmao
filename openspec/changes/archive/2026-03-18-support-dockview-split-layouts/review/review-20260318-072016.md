## Summary

- **Strong foundation:** The change builds naturally on the existing Dockview integration. The current codebase already stores layout JSON, uses `always` rendering, and keeps one DOM root per session â€” all of which the design correctly identifies and reuses.
- **Well-scoped non-goals:** Floating groups, pop-out windows, and duplicate views are explicitly deferred, keeping the change focused on the core split-layout value.
- **Clean removal path:** The three enforcement points (`ensureSingleVisibleGroup`, `isAllowedDropEvent`, and the `onWillShowOverlay` filter) are clearly identified in the implementation and the design correctly targets them.
- **Risk: resize path is under-specified.** The design says "fit every visible panel" but the current codebase only fits `activeSessionId` on window resize (app.js:1944â€“1947). The tasks don't specify how to discover all visible panels or debounce the resize cascade.
- **Risk: focus routing has a gap.** The design says interaction inside terminal content should forward to the owning Dockview panel, but the current component's `focus()` hook (app.js:676â€“679) only calls `focusSession()` â€” it does not call `setActiveSession()`. The reverse path (terminal interaction â†’ Dockview panel activation) is not wired.
- **Risk: `getOrderedSessionIds()` across multiple groups may produce a fragile ordering.** The current implementation (app.js:203â€“219) iterates groups and their panels, but group iteration order in Dockview's API is not guaranteed to be stable across layout changes. The design and tasks are silent on ordering semantics for multi-group layouts.

## Findings

### Must Fix / Blocking

1. **Terminal-to-panel focus forwarding is missing from the current implementation and under-specified in tasks.** The design (Decision 3) says "Tailmux will also forward direct interaction inside terminal content back to the owning panel so toolbar and keyboard actions follow the pane the user is actually working in." However, the current `createTerminalComponent` (app.js:634â€“685) only has a `focus()` hook that calls `focusSession()` â†’ `term.focus()`. There is no reverse path: when a user clicks into an xterm surface, the owning Dockview panel does not become the active panel. Task 2.2 says "Forward interaction inside terminal content back to the owning Dockview panel" but does not describe the mechanism. Without this, clicking into a non-active pane's terminal will type into the correct xterm but toolbar/keyboard/tmux actions will still target the previously active panel â€” violating the spec scenario "Clicking inside a visible pane updates the action target."

   Implementation hint: xterm's `term.textarea` fires `focus` events. The terminal component can listen for that and call `panel.api.setActive()` or equivalent to sync Dockview's active panel state.

> **DECISION: Refine.**
> Rationale: The gap is real and blocking, but I do not want the artifacts to hard-code a single xterm-internal hook as the contract. The revision should require terminal-content interaction to activate the owning Dockview panel, with `term.textarea` focus as one viable implementation path rather than the only prescribed mechanism.

2. **Resize handling for multi-pane layouts needs explicit design.** The current window resize handler (app.js:1944â€“1947) only fits `activeSessionId`. The design (Decision 4) says "Tailmux will fit every visible panel after workspace layout changes and window resize events" and "use per-panel dimension and visibility hooks for more local resizes." The tasks (3.1) say "Update resize and fit handling so visible panels are re-fit after split creation, divider moves, visibility changes, and window resize events" but don't specify whether this means:
   - (a) iterating `workspace.groups` and fitting each group's active panel, or
   - (b) relying on Dockview's per-component `layout()` callback (which is already wired at app.js:671â€“674 and calls `scheduleFitSession`), or
   - (c) both.

   The component `layout()` hook already exists and fires on dimension changes â€” this may already cover most of the multi-pane resize case once the single-group restriction is removed. The task should clarify whether the window-resize handler at app.js:1944 needs to change at all, or whether `layout()` callbacks are sufficient. The `toggleKeyboard` and `toggleTmuxPanel` flows (app.js:1708â€“1723) also only fit `activeSessionId` and would need updating.

> **DECISION: Accept.**
> Rationale: The current artifacts do not yet make the multi-pane resize path explicit enough. The revision should state that container-level size changes trigger `workspace.layout()` while per-panel fitting continues to happen in the component `layout()` hook, and it should cover the keyboard/tmux panel toggles that still assume a single active session.

### Should Fix / Important

3. **`getOrderedSessionIds()` ordering semantics are undefined for multi-group layouts.** The current function (app.js:203â€“219) iterates `workspace.groups` then each group's `panels`. In a split layout, the iteration order across groups is determined by Dockview's internal data structures and may not match the visual left-to-right or top-to-bottom order the user sees. This matters for: (a) the `orderedSessionIds` array persisted in workspace state, (b) the dashboard tab list ordering, and (c) the fallback active session on restore (app.js:2010). Task 1.3 says "Update workspace-derived ordering and dashboard/session enumeration helpers" but does not define what the correct multi-group ordering should be.

> **DECISION: Refine.**
> Rationale: Order semantics do need to be defined for dashboard display and fallback activation, but the review overstates the impact on persisted arrangement because the design already makes Dockview JSON the primary layout source. The revision should define a stable flattened enumeration for list and fallback use without turning `orderedSessionIds` back into the source of truth.

4. **The `isAllowedDropEvent` filter needs updating, not just removal.** The current function (app.js:532â€“538) restricts drops to `tab`, `header_space`, and `content` with `position === 'center'`. To allow edge docking, this must accept `position` values like `left`, `right`, `top`, `bottom` for `content` kind events. Task 1.1 says "Remove the Tailmux-specific single-group Dockview restrictions so edge docking can create horizontal and vertical split groups" â€” the word "remove" could be misread as deleting the filter entirely, which would also re-enable floating groups. Since floating groups are separately disabled via `disableFloatingGroups: true` (app.js:692), full removal of `isAllowedDropEvent` may be safe, but the task should be precise about the intent.

> **DECISION: Refine.**
> Rationale: The ambiguity is real, but I am not committing to preserving a custom drop filter. The revision should explicitly choose one behavior: either remove Tailmux's custom drop filtering entirely or define the allowed split positions precisely, while keeping `disableFloatingGroups: true` as the no-floating guard.

5. **The delta spec does not mention the `disableFloatingGroups: true` setting.** The design (Decision 1) and non-goals both state that floating groups stay out of scope. The current code already sets `disableFloatingGroups: true` (app.js:692), so no change is needed. However, the delta spec should acknowledge this invariant explicitly â€” especially since the `onWillShowOverlay` filter is being relaxed â€” to prevent accidental regression during implementation.

> **DECISION: Refine.**
> Rationale: The spec should state the behavioral invariant that this phase does not create floating groups or popouts, but it does not need to name the implementation flag `disableFloatingGroups: true`. I want the capability contract expressed in user-visible terms, not bound to a specific API option.

6. **The workspace state version bump (v1 â†’ v2) and storage key change are mentioned in design but not reflected in tasks.** Design Decision 5 says "bump the workspace state version" and the migration plan says "on version mismatch, discard the previous stored workspace state." Task 3.2 mentions bumping the version. The current restore logic (app.js:1970) checks `payload.version !== WORKSPACE_STATE_VERSION` â€” so bumping the constant alone should work. The task should be explicit that the storage key name can stay the same while only the version field changes.

> **DECISION: Refine.**
> Rationale: The version-bump issue is real, but the required artifact change is narrower than the finding implies. 7. **Focused-pane styling (Task 2.3) has no design detail.** The design (Decision 6) says "the focused pane must be visually obvious" and Task 2.3 says "Add visible focused-pane styling." The existing CSS (styles.css:158â€“159) already styles `.dv-active-group > .dv-tabs-and-actions-container` with a bottom border, but there's no `.dv-active-group` styling on the content area itself. In a split layout, users need to see which pane owns global actions at a glance. The task should specify whether the styling is a border, background tint, glow, or header highlight â€” and whether it applies to the content container or the tab strip or both.

> **DECISION: Refine.**
> Rationale: The artifacts need minimum acceptance criteria for focused-pane styling, but I do not want to overdesign the exact look in the review response. The revision should require an unmistakable active-pane signal on the visible panel content area, with the specific border or highlight treatment left to implementation.

## Design Alignment With Existing System

- **Existing pattern: one session = one panel = one DOM root.** The current implementation (app.js:634â€“685, `createTerminalComponent`) attaches a session's `root` DOM element into the panel host via `element.replaceChildren(session.root)`. The design correctly preserves this invariant (Decision 2). No misalignment.

- **Existing pattern: `always` renderer for DOM-stateful content.** The workspace uses `defaultRenderer: 'always'` (app.js:691) and `panel.api.setRenderer('always')` (app.js:459). The design correctly notes this as the foundation for cross-group panel moves without session recreation. No misalignment.

- **Existing pattern: debounced workspace save.** `scheduleWorkspaceSave()` (app.js:282â€“295) uses a 120ms debounce timer. The design mentions debouncing layout saves. This aligns well â€” the existing debounce should handle the increased save frequency from split events. No misalignment.

- **Existing pattern: `onDidActivePanelChange` â†’ `setActiveSession`.** The current handler (app.js:704â€“706) sets `activeSessionId` from the panel's `id`. ## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the Dockview component `layout()` callback be the primary resize mechanism for multi-pane, or should the window resize handler also iterate visible panels?

- Decision: `Deferrable`

#### Why this matters

- Getting resize wrong causes xterm to render with incorrect row/column counts, which breaks terminal UX (garbled output, misaligned prompts).
- Over-resizing (fitting all sessions including hidden ones) creates unnecessary WebSocket resize traffic.

#### Examples

- User drags a split divider: both affected panels change size. If only `layout()` fires, both panels refit automatically. No window resize handler change needed.
- User resizes the browser window: all visible panels change size. If `layout()` fires for each visible component, all panels refit. But if `layout()` only fires for the "active" component, the others get stale dimensions.
- User toggles the mobile keyboard/tmux panel: `terminal-container` height changes. The current code (app.js:1708â€“1709) only fits `activeSessionId`. All visible panels need refitting.

#### Why this question arises in code

- `createTerminalComponent.layout()` (app.js:671â€“674) already calls `scheduleFitSession` for its own session.
- `window.addEventListener('resize', ...)` (app.js:1944â€“1947) only fits `activeSessionId`.
- `toggleKeyboard()` (app.js:1708â€“1709) and `toggleTmuxPanel()` (app.js:1722â€“1723) only fit `activeSessionId`.
- Dockview's `layout()` callback is documented to fire on panel dimension changes, which should cover divider moves and window resizes â€” but the mobile keyboard/tmux panel resize changes `#terminal-container` height, which may require calling `workspace.layout()` to trigger Dockview's internal relayout.

#### Options

**Option A: Rely on `layout()` callbacks and call `workspace.layout()` where the container dimensions change externally.**

- Code impact:
  - Keep `createTerminalComponent.layout()` as the per-panel fit hook.
  - In window resize, keyboard toggle, and tmux panel toggle, call `workspace.layout()` instead of `scheduleFitSession(activeSessionId)`. Dockview propagates the layout event to all visible components.
  ```js
  window.addEventListener('resize', () => {
    workspace?.layout();
  });
  ```
- User-facing difference:
  - All visible panes refit simultaneously on any container size change.
- Pros:
  - Single responsibility: each component manages its own fit via `layout()`.
  - No need to enumerate visible panels manually.
- Cons:
  - Depends on Dockview firing `layout()` for all visible components (needs verification).

**Option B: Manually iterate visible panels in the resize handler.**

- Code impact:
  - Replace `scheduleFitSession(activeSessionId)` with a loop over `workspace.groups` â†’ `group.activePanel` â†’ `scheduleFitSession`.
  ```js
  window.addEventListener('resize', () => {
    workspace?.groups.forEach((group) => {
      if (group.activePanel) {
        scheduleFitSession(group.activePanel.id);
      }
    });
  });
  ```
- User-facing difference:
  - Same as Option A: all visible panes refit.
- Pros:
  - Explicit control, no dependency on Dockview's internal layout propagation.
- Cons:
  - Duplicates the per-panel fit logic that `layout()` already provides.
  - Must be updated if Dockview's group/panel API changes.

> **PROPOSED: Option A â€” rely on `layout()` callbacks and call `workspace.layout()` at the container level.**
> Rationale: The component `layout()` hook (app.js:671â€“674) already calls `scheduleFitSession` for its session. Dockview is designed to propagate `layout()` to all visible components when the container dimensions change. Calling `workspace.layout()` from the window resize handler and mobile toggle flows is the idiomatic Dockview approach and avoids manually enumerating groups. This is also forward-compatible with any future group additions.
>
> **DECISION: Accept PROPOSED with a verification note.**
> Rationale: This fits the current component structure best: per-panel fitting already lives in `createTerminalComponent.layout()`, and container-level changes should trigger `workspace.layout()` instead of duplicating visible-panel iteration. Implementation should still verify that this covers browser resize and toolbar-height changes; only then use manual iteration as a fallback.

### Q2) How should the dashboard present sessions from multiple groups?

- Decision: `Deferrable`

#### Why this matters

- The dashboard is the primary session management surface outside the tab strip. If it shows a flat list, users lose the visual correspondence to their split layout. If it shows groups, the UI becomes more complex.

#### Examples

- User has 3 sessions: A and B in the left group, C in the right group. The dashboard currently shows "A, B, C" as a flat list. After this change, should it still be flat, or show "Left: A, B | Right: C"?
- User clicks "Switch" on session C in the dashboard. The spec says the tab activates in its group (scenario "Dashboard switch activates a tab in another visible group"). The flat list is sufficient for this behavior.

#### Why this question arises in code

- `updateDashboard()` iterates `getOrderedSessionIds()` (which walks groups â†’ panels). The dashboard DOM (index.html:103â€“141) has a single `#dashboard-tabs-list` container.
- The design's open question (design.md:145) asks exactly this: "Should the dashboard remain a flat session list in this phase, or should it show which workspace group each session belongs to?"

#### Options

**Option A: Keep a flat session list for this change.**

- Code impact:
  - No dashboard DOM or styling changes needed.
  - `getOrderedSessionIds()` continues to produce a flat list.
- User-facing difference:
  - Dashboard does not reflect split arrangement. Users manage sessions by name, not by group.
- Pros:
  - Simpler implementation, smaller change scope.
  - Consistent with the design's current assumption ("Current design assumes a flat list is acceptable").
- Cons:
  - With many sessions across groups, users may not know which group a session belongs to.

**Option B: Group sessions by workspace group in the dashboard.**

- Code impact:
  - Dashboard DOM needs group headers or separators.
  - `updateDashboard()` needs to iterate groups, not a flat list.
- User-facing difference:
  - Dashboard mirrors the split layout structure.
- Pros:
  - Better spatial awareness for complex layouts.
- Cons:
  - More UI work, more CSS, more testing surface.
  - Group identity in Dockview is transient (groups are created/destroyed as tabs move).

> **PROPOSED: Option A â€” keep a flat session list for this change.**
> Rationale: The design already states "Current design assumes a flat list is acceptable" (design.md:145). The dashboard's "Switch" action just calls `activateSession()` which will activate the correct tab in whatever group it lives in. Adding group structure to the dashboard is additive and can be layered in a follow-up change without breaking the split-layout core. Keeping the dashboard flat reduces the scope and testing surface of this already-substantial change.
>
> **DECISION: Accept PROPOSED.**
> Rationale: A flat dashboard keeps this change focused on split workspace behavior and matches the current design assumption. Group-aware dashboard structure is additive and can be revisited separately once multi-pane behavior is proven.

### Q3) Should the tasks explicitly enumerate all `ensureSingleVisibleGroup` call sites to remove?

- Decision: `Blocking`

#### Why this matters

- Missing a call site means the split layout silently collapses back into a single group, which is the exact behavior this change is removing. This is the highest-risk implementation mistake.

#### Examples

- An implementer removes `ensureSingleVisibleGroup` from `onDidMovePanel` and `onDidLayoutChange` but forgets the `onDidLayoutFromJSON` hook. Result: restored layouts always collapse to one group, making the persist/restore feature appear broken.
- An implementer removes it from all event hooks but forgets the explicit call in `restoreWorkspaceState()`. Result: fresh sessions split fine, but any restored layout collapses.

#### Why this question arises in code

- `ensureSingleVisibleGroup` is called at four locations in app.js:
  - `workspace.onDidMovePanel` (line 709)
  - `workspace.onDidLayoutChange` (line 715)
  - `workspace.onDidLayoutFromJSON` (line 721)
  - `restoreWorkspaceState` (line 2006)
- Task 1.1 says "Remove the Tailmux-specific single-group Dockview restrictions" but does not list these call sites.
- Task 1.2 says "Update tab add, move, and close flows to preserve valid multi-group layouts instead of collapsing everything back into one visible group" â€” this covers the event hooks but not the restore path.
- Task 3.3 covers restore but doesn't mention removing `ensureSingleVisibleGroup` from the restore flow.

#### Options

**Option A: Add an explicit sub-bullet to Task 1.1 listing all four call sites.**

- Code impact:
  - Task text change only; no code impact.
- User-facing difference:
  - None â€” this is an implementation guidance improvement.
- Pros:
  - Eliminates the highest-risk implementation mistake for this change.
  - Cheap to add.
- Cons:
  - Tasks become slightly more verbose.

**Option B: Keep the current task wording and rely on the implementer to grep for `ensureSingleVisibleGroup`.**

- Code impact:
  - None.
- User-facing difference:
  - None.
- Pros:
  - Tasks stay concise.
- Cons:
  - Risk of partial removal, which produces the most confusing bug: splits work on fresh sessions but collapse on restore or on certain drag events.

> **PROPOSED: Option A â€” add explicit call-site enumeration to Task 1.1.**
> Rationale: `ensureSingleVisibleGroup` is called at four distinct locations (app.js lines 709, 715, 721, 2006), spanning three different tasks (1.1, 1.2, 3.3). The function is the core mechanism being removed by this change. Enumerating all call sites in one place (Task 1.1) costs one sub-bullet and eliminates the highest-probability implementation mistake. The function itself and `isAllowedDropEvent` together form the complete "single-group restriction" and should be addressed as a unit.
>
> **DECISION: Refine PROPOSED.**
> Rationale: All `ensureSingleVisibleGroup` call sites must be enumerated explicitly, but I do not want to force every restore-path detail into Task 1.1. The revision should make the live workspace hooks explicit in Task 1.1 and the restore-path removals explicit in the restore-related task so ownership stays aligned with the implementation sections.

### Q4) Should the UI expose explicit split commands (buttons/keyboard shortcuts) in addition to drag/drop?

- Decision: `Deferrable`

#### Why this matters

- Drag/drop is the only split mechanism in this change. If it's hard to discover or use (especially on non-desktop browsers), users may not realize splits are available.

#### Examples

- A user wants to put a terminal side-by-side but doesn't know they can drag a tab to the edge of the content area.
- A power user wants a keyboard shortcut to split the current terminal left/right.

#### Why this question arises in code

- The design's open question (design.md:146) asks: "Should the UI expose explicit split commands later in addition to drag/drop, or is drag/drop alone sufficient for the first multi-pane release?"
- No split button or keyboard shortcut exists in the current index.html or app.js.

#### Options

**Option A: Defer explicit split commands; rely on drag/drop only for this change.**

- Code impact:
  - No additional UI elements or keyboard handlers.
- User-facing difference:
  - Splits require mouse drag to an edge docking zone. No button or shortcut alternative.
- Pros:
  - Keeps the change scope focused.
  - Drag/drop is the standard Dockview interaction model.
- Cons:
  - Discoverability is lower; new users may not find the feature.

**Option B: Add a "Split Right" / "Split Down" button or context menu in this change.**

- Code impact:
  - New button(s) in the workspace toolbar or tab context menu.
  - New handler that calls Dockview's `addPanel` or `movePanel` with a directional position.
- User-facing difference:
  - Users can split without drag/drop.
- Pros:
  - Better discoverability.
  - Accessible for users who struggle with drag/drop.
- Cons:
  - Increases scope and testing surface.
  - Design of button placement needs its own UX decision.

> **PROPOSED: Option A â€” defer explicit split commands to a follow-up.**
> Rationale: The design already lists this as an open question and the non-goals exclude "browser-level 2-view or 4-view presets separate from Dockview drag/drop behavior." Drag/drop is the native Dockview interaction and is sufficient for a first release. Adding split buttons is additive and can be a separate change with its own UX design. The README update (Task 4.1) should mention that splits are created via drag/drop to aid discoverability.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Drag/drop is enough for this phase and keeps the change aligned with Dockview's native interaction model. Discoverability can be handled in docs now, and explicit split commands can be proposed later as a separate UX addition.
