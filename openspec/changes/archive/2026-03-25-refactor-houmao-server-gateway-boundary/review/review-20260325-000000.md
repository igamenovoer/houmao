## Summary

- **Solid three-layer architecture.** The shared coordination plane / per-agent control plane / execution substrate split is well-reasoned, matches the existing code topology, and aligns with the natural fault and ownership boundaries already present in the codebase.
- **Comprehensive spec coverage.** All six modified capabilities have delta specs with concrete scenarios. The proposal, design, tasks, and specs form a coherent chain.
- **Missing control-provider protocol spec.** The most critical new abstraction introduced by this change — the internal control-provider seam that selects between gateway-backed and direct fallback — has no delta spec defining its shape. It appears only as a design concept and a task title.
- **TUI tracking ownership transfer is underspecified.** Moving tracker authority from server to gateway requires resolving package-boundary, transport, and handoff questions that the current specs leave implicit.
- **Gateway-to-server state projection transport is unspecified.** The specs require the server to project gateway-owned state, but never define how the server reads that state from the gateway (HTTP poll, filesystem, push, in-process call).
- **Risk of scope creep.** 16 tasks across 5 phases with no task complete yet; this is a large refactor touching the most central ownership boundaries in the system. Cross-phase dependencies are implicit, and the attach/detach transition protocol is not specified.

## Findings

### Must Fix / Blocking

1. **The control-provider seam — the central abstraction of this change — has no spec, no protocol shape, and no delta spec defining its contract.** Task 1.1 says "introduce an internal managed-agent control-provider seam" and the design says "one internal control-provider contract," but no artifact defines what that protocol looks like. This is the single most important new interface in the change and implementers have no specification to build against. The existing inline branching in `HoumaoServerService._managed_gateway_context()` (service.py ~lines 2360–2397) is the natural replacement site, but the target abstraction needs at least: method signatures for state projection, request submission, interrupt, and lifecycle control; the rules for provider selection and fallback; and how provider identity is threaded through alias resolution. Without this, tasks 1.1–1.3 and all downstream phases that depend on the seam are underspecified.

> **DECISION: Refine.**
> Rationale: The gap is real, but the fix should be an explicit internal contract in the design and Phase 1 tasks rather than a new public capability spec with Python method signatures. The public contract remains the managed-agent route family; the control-plane seam is an internal implementation boundary that still needs to be spelled out before apply.

2. **The transport/protocol by which `houmao-server` reads gateway-owned per-agent live control state is never specified.** The `agent-gateway` delta spec adds "read-optimized per-agent live control snapshots" and the `official-tui-state-tracking` spec says the server "projects gateway-owned tracked state," but no artifact specifies the mechanism. The gateway already exposes an HTTP surface via `GatewayService` (in `gateway_service.py`) and the server already has a `GatewayClient` (in `gateway_client.py`) for health/status/request calls. But the delta specs don't say whether the server should: (a) call existing or new gateway HTTP endpoints, (b) read gateway filesystem artifacts directly from the shared session root, (c) use an in-process call when co-located, or (d) receive push notifications. This decision affects latency, coupling, error handling, and testing strategy for every Phase 3 task.

> **DECISION: Accept.**
> Rationale: Gateway-owned live state should be read through the existing HTTP boundary via `GatewayClient`, extended with explicit read endpoints. Files under the gateway root can remain durability and capability artifacts, but the authoritative live state must stay in gateway memory and be projected over the gateway API.

3. **TUI tracking ownership transfer across package boundaries is architecturally unresolved.** The `official-tui-state-tracking` delta spec requires the gateway to become the "active per-agent control plane" for tracked-state reduction using the "shared TUI tracking core." Currently, that tracking core lives in `src/houmao/server/tui/tracking.py` (`LiveSessionTracker`) and `src/houmao/server/tui/supervisor.py` (`TuiTrackingSupervisor`), both under the server package. The gateway lives in `src/houmao/agents/realm_controller/gateway_service.py`. For the gateway to run the shared tracking core, either: (a) the tracking core must be extracted to a shared package importable by both server and gateway, (b) the gateway must depend on the server package, or (c) the gateway must reimplement tracking. None of these options is stated in the specs or tasks. Task 2.2 says "move attached-agent TUI tracking ownership into the gateway" without addressing the package-level restructuring required.

> **DECISION: Accept.**
> Rationale: The package boundary is not resolved in the current artifacts. `LiveSessionTracker` and `TuiTrackingSupervisor` live under `houmao.server.tui`, so Phase 2 needs an explicit extraction of the reusable tracking runtime or core into a neutral module that both the gateway and the direct server fallback can use without making the gateway depend on `houmao.server`.

### Should Fix / Important

4. **Attach/detach transition protocol is unspecified.** The design and specs describe a steady-state model (gateway attached → gateway owns; no gateway → direct fallback) but never define the transition protocol. Key unanswered questions: What happens to in-flight requests when a gateway attaches mid-tracking? Does the server drain its direct tracker before handing off? What happens to queued gateway requests when the gateway detaches? Is there an atomic handoff or a brief gap? The `TrackingRuntime` protocol in `tui/supervisor.py` has `release_known_session()` which could serve as a handoff hook, but the specs don't reference it.

> **DECISION: Refine.**
> Rationale: A transition protocol is needed, but v1 does not need atomic state transfer. The first revision should explicitly allow a brief handoff gap with last-known-state caching and single-owner semantics, then leave coordinated state handoff as later work.

5. **Headless turn record creation ownership during gateway-backed admission is ambiguous.** The design says the server remains the "durable turn catalog" while the gateway owns "live admission." But the specs don't clarify: (a) who creates the `ManagedHeadlessTurnRecord` and `ManagedHeadlessActiveTurnRecord` when a prompt is admitted through the gateway — does the server create the record before forwarding to the gateway, or does the gateway create it and push it back? (b) how turn events flow from gateway-admitted headless execution back to the server's turn persistence layer (currently `ManagedHeadlessTurnRecord` in `managed_agents.py`); (c) how the server knows a gateway-admitted turn has completed so it can transition the active turn record to terminal state.

> **DECISION: Accept.**
> Rationale: The durable server-owned turn catalog needs an explicit creation point. The server should create the durable turn identity and active-turn record before handing live execution to the gateway, then reconcile terminal status back into the same store.

6. **Registry publisher identity storage and selection mechanism is not specified.** Task 4.1 says "add explicit publisher selection" and the `agent-discovery-registry` delta spec says publication "follows agent creation authority," but no artifact defines: (a) where the publisher identity is persisted (in the session manifest? a new field on the authority record? the registry record itself?); (b) how runtime knows at publication time whether to defer to `houmao-server`; (c) what happens if a server-managed agent's runtime tries to refresh the registry independently (race, conflict, or silent no-op). The `brain-launch-runtime` delta spec adds scenarios for "server-managed session continues pointer publication while deferring registry publication" but doesn't specify the selection signal.

> **DECISION: Accept.**
> Rationale: Publisher selection must be persisted in a runtime-readable place so runtime knows whether to publish or defer. That signal should not live only in prose or be inferred from the registry record itself; it needs an explicit field in server or runtime-managed session metadata.

7. **`CompatibilityProviderAdapter` naming overlap with the new "control provider" concept.** The existing `control_core/provider_adapters.py` defines `CompatibilityProviderAdapter` for CAO-compatible tmux session control. The new change introduces a "control provider" seam for gateway-backed vs. direct fallback selection. These are architecturally distinct concepts, but the "provider" naming overlap will cause confusion during implementation and code review. Consider a distinct name for the new abstraction (e.g., `ManagedAgentControlPlane`, `ControlAuthority`, or `AgentControlBackend`).

> **DECISION: Accept.**
> Rationale: Reusing the term `provider` will collide with existing CAO compatibility terminology and make code review harder. The new abstraction should use a distinct name such as `ManagedAgentControlPlane` or `ManagedAgentControlBackend`.

8. **Tasks lack explicit acceptance criteria.** The 16 tasks have good descriptive titles but no measurable acceptance criteria. For a refactor of this scope, each task should specify: what tests must pass, what observable behavior changes (or must not change), and what artifacts are produced. This is especially important for Phase 1 (control-provider seam) and Phase 2 (gateway ownership transfer), where the boundary between "done" and "not done" is architectural rather than feature-visible.

> **DECISION: Refine.**
> Rationale: The concern is valid, especially for Phases 1 and 2, but the right fix is targeted acceptance criteria for the high-risk tasks rather than turning every task into a long test checklist. The artifacts should say what observable invariants and regression coverage define completion.

### Nice To Have

9. **The `managed-agent-detailed-state` delta spec adds headless detail projection from gateway but doesn't address freshness.** When the server projects gateway-backed headless detail, the spec says "projected from the gateway-backed per-agent control state" but doesn't specify staleness bounds. If the gateway is unreachable for a brief period, should the detail route return cached stale state, return an error, or fall back to direct state? This is relevant for operational resilience.

> **DECISION: Refine.**
> Rationale: Freshness matters, but it should be handled as part of gateway-degradation semantics rather than a separate caching spec. The intended v1 rule is "project current gateway state when healthy; otherwise degrade to the selected fallback path or return unavailable," not "serve indefinite stale snapshots."

10. **The `houmao-server-agent-api` delta spec modifies requirements for gateway-backed control but doesn't add a new scenario for gateway health degradation during request submission.** The existing spec has `503` for "managed-agent unavailable" and `409` for admission conflicts, but there's no explicit scenario for: "gateway is attached but unhealthy — should the server fall back to direct, reject with 503, or retry?" The design's rollback strategy mentions "disable gateway-backed provider selection and fall back," but the API spec doesn't define the runtime degradation behavior.

> **DECISION: Accept.**
> Rationale: The API artifacts need an explicit degraded-gateway rule. Provider selection should prefer an attached gateway only when it is healthy and admissible; otherwise the server should fall back only when direct control is safe for that agent, and return `503` or `409` when it is not.

11. **Phase 5 documentation tasks (5.1, 5.2) are vague relative to the other phases.** They say "update docs" and "update CLI guidance" without specifying which documentation files to update. The repo has `docs/` with reference material (`agents_brains.md`, `cli.md`), `context/` and `magic-context/` with agent instructions, and inline docstrings. Specifying the target files would make these tasks actionable.

> **DECISION: Refine.**
> Rationale: The doc tasks should name the minimum target docs, but they do not need a full file-by-file migration matrix yet. At minimum the change should call out the gateway reference, state-tracking developer doc, shared-registry reference, and managed-agent or CLI reference surfaces.

12. **No explicit scenario for CAO compatibility layer interaction with the new control-provider seam.** The server has `CompatibilityControlCore` (in `control_core/core.py`) as a third ownership path alongside TUI tracking and headless management. The design and specs focus on TUI and headless but don't clarify whether CAO-compatible sessions should also go through the new control-provider seam, remain on their existing separate path, or be explicitly excluded. Since `m_cao_core` is a distinct ownership structure in `service.py`, this should be stated.

> **DECISION: Refine.**
> Rationale: The change should explicitly exclude CAO compatibility from this seam in Phase 1 rather than adding full scenario coverage for it. `CompatibilityControlCore` is a separate, already-deprecated ownership path and should remain out of scope unless a later change chooses to unify it.

## Design Alignment With Existing System

### Existing patterns observed

- **Server ownership fragmentation.** `HoumaoServerService` (service.py) maintains three separate ownership structures: `m_trackers` (TUI), `m_headless_agents` (headless), and `m_cao_core` (CAO compatibility). The proposed control-provider seam would unify the first two under a common dispatch mechanism, which aligns with the existing `_managed_gateway_context()` resolution pattern.
- **Gateway infrastructure is mature.** `GatewayService`, `GatewayClient`, `GatewayStatusV1`, `GatewayStoragePaths`, and the full attach/detach lifecycle are already implemented in `src/houmao/agents/realm_controller/gateway_*`. The gateway already owns durable queue, live status, mailbox facade, and notifier state. Extending it to own TUI tracking and headless admission is a natural progression.
- **`TrackingRuntime` protocol exists.** `tui/supervisor.py` already defines a `TrackingRuntime` protocol for tracker supervision with methods like `load_live_known_sessions()`, `poll_known_session()`, and `release_known_session()`. This is a good seam for the gateway to implement or delegate through.
- **Registry publication is centralized in `RuntimeSessionController`.** The `refresh_shared_registry_record()` method and `registry_storage.py` utilities handle all current publication. The proposed split (runtime vs. server publisher) requires adding publisher selection logic to this path.
- **Route definitions in `app.py` delegate cleanly to `service.py`.** All managed-agent routes (lines 300–449 in `app.py`) delegate to `HoumaoServerService` methods. The control-provider seam should be introduced at the service layer, not the route layer — the change design aligns with this.

### Proposed design alignment assessment

- **Aligns well:** The three-layer model, the stable public route contract, and the pointer-oriented registry model all match the existing architecture and code layout.
- **Partial mismatch — TUI tracking package location.** The shared TUI tracking core is in `src/houmao/server/tui/`. The gateway is in `src/houmao/agents/realm_controller/`. For the gateway to run the tracking core, a package restructuring is needed that isn't addressed by the change artifacts.
- **Partial mismatch — CAO compatibility path.** The change focuses on TUI and headless agents but doesn't clarify how the `CompatibilityControlCore` path interacts with the new control-provider seam.
- **Naming.** The existing `CompatibilityProviderAdapter` in `control_core/provider_adapters.py` uses "provider" for a different concept than the proposed "control provider." A distinct name for the new abstraction would improve clarity.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What is the shape of the control-provider protocol?

- Decision: `Blocking`

#### Why this matters

The control-provider seam is the central abstraction of this entire change. Every subsequent task depends on it. Without a defined protocol shape, implementers must invent the interface during implementation, risking inconsistency across the 16 tasks and rework when the interface doesn't fit later phases.

#### Examples

- Task 3.1 needs the provider to expose a state-projection method for `/state` and `/state/detail`.
- Task 3.2 needs the provider to expose a request-submission method for `/requests`.
- Task 2.2 needs the provider to own TUI tracking authority.
- Task 2.3 needs the provider to own headless admission authority.
- All of these must share one protocol, but no artifact defines what that protocol looks like.

#### Why this question arises in code

The existing inline resolution in `_managed_gateway_context()` (service.py ~lines 2360–2397) returns a `dict[str, object]` with `"controller"`, `"status"`, etc. This ad-hoc structure is the current implicit protocol. The refactor needs to replace it with a typed protocol, but the target shape isn't specified.

```python
# Current pattern (service.py):
def _managed_gateway_context(self, agent_ref: str) -> dict[str, object]:
    resolved = self._resolve_managed_agent_ref(agent_ref)
    if resolved["transport"] == "headless":
        handle = self._require_headless_handle(resolved["tracked_agent_id"])
        controller = self._resume_or_get_headless_controller(handle)
    else:  # "tui"
        tracker = self._tracker_for_session_id(resolved["tracked_agent_id"])
        controller = self._controller_for_tui_tracker(tracker)
    return {"controller": controller, ...}
```

#### Options

**Option A: Define a `ManagedAgentControlPlane` protocol with per-concern methods**

- Code impact:
  - New protocol in `src/houmao/server/models.py` or a new `control_plane.py` module
  - Methods: `project_state()`, `project_detail()`, `submit_request()`, `interrupt()`, `project_gateway_summary()`
  - Two implementations: `GatewayBackedControlPlane` and `DirectFallbackControlPlane`
  - `_managed_gateway_context()` replaced with provider lookup returning the protocol
  ```python
  class ManagedAgentControlPlane(Protocol):
      def project_state(self) -> ManagedAgentStateResponse: ...
      def project_detail(self) -> ManagedAgentDetailResponse: ...
      def submit_request(self, req: ManagedAgentRequest) -> AcceptedResponse: ...
      def interrupt(self) -> AcceptedResponse: ...
  ```
- User-facing difference:
  - None — same public routes and response shapes
- Pros:
  - Explicit contract; implementers know exactly what to build
  - Testable in isolation (mock either implementation)
  - Clean separation of concerns
- Cons:
  - Up-front design effort before implementation can start
  - May need iteration as Phase 2 and 3 reveal additional methods

**Option B: Let the protocol emerge during implementation, formalize after Phase 1**

- Code impact:
  - Start with the existing `dict[str, object]` pattern, gradually replace
  - Protocol defined after tasks 1.1–1.3 are implemented
- User-facing difference:
  - None
- Pros:
  - Faster start; avoids premature abstraction
  - Protocol shaped by actual implementation needs
- Cons:
  - Risk of inconsistency across tasks
  - Rework when the protocol is formalized
  - Harder for parallel implementers to work on Phase 2/3 concurrently

> **PROPOSED: Option A — define the protocol shape as part of task 1.1 before starting implementation.**
> Rationale: The existing `_managed_gateway_context()` in service.py already reveals the rough method surface (state projection, request submission, interrupt, gateway summary). The three server route groups in `app.py` (state/detail routes, request/turn routes, terminal-facing TUI routes) map cleanly to protocol methods. Defining the protocol up front avoids rework and enables parallel implementation of Phase 2 and Phase 3. The protocol can be extended as needed, but the initial shape should be captured in a delta spec or at minimum in the task 1.1 description.
>
> **DECISION: Refine PROPOSED.**
> Rationale: The protocol should be defined before implementation and should replace the ad-hoc `_managed_gateway_context()` shape, but it should be captured as an internal design and task contract rather than a standalone public capability spec. The artifacts need to state the control-plane responsibilities, provider selection rules, and fallback semantics explicitly before apply.

### Q2) How does the server read gateway-owned tracked state for attached TUI agents?

- Decision: `Blocking`

#### Why this matters

The entire Phase 3 (server route projection) depends on this. If the server polls gateway HTTP endpoints, that's a network dependency on every state request. If it reads filesystem artifacts, that couples the server to the gateway's disk layout. If it uses in-process calls, the gateway must be co-located. Each choice has different latency, error handling, and testing implications.

#### Examples

- A dashboard client calls `GET /houmao/agents/{agent_ref}/state/detail` for 10 attached TUI agents every 2 seconds. If each call requires an HTTP round-trip to 10 separate gateway sidecars, that's 5 req/s per gateway.
- A gateway sidecar restarts. The server needs to know whether to return stale cached state, fall back to direct tracking, or return an error.

#### Why this question arises in code

The `GatewayClient` in `gateway_client.py` already supports HTTP calls to the gateway (`health()`, `status()`, `submit_request()`). The gateway filesystem layout in `gateway_storage.py` publishes artifacts under `<session-root>/gateway/`. But the delta specs add "read-optimized per-agent live control snapshots" without specifying which transport delivers them to the server.

#### Options

**Option A: HTTP via `GatewayClient` — extend the existing gateway HTTP surface with tracked-state endpoints**

- Code impact:
  - Add `GET /state/tracked` and `GET /state/tracked/history` to `GatewayService`
  - Add corresponding methods to `GatewayClient`
  - Server calls `GatewayClient` from the control-provider implementation
  ```python
  # gateway_client.py
  def tracked_state(self) -> TrackedStateSnapshot: ...
  def tracked_state_history(self) -> list[TrackedStateEntry]: ...
  ```
- User-facing difference:
  - Slightly higher latency per state request (HTTP round-trip to local gateway)
- Pros:
  - Clean separation; gateway and server are truly independent processes
  - Aligns with existing `GatewayClient` pattern
  - Testable with mock HTTP server
- Cons:
  - Latency overhead on frequently polled state routes
  - Gateway unavailability requires fallback or error handling on every request

**Option B: Filesystem artifacts — gateway writes tracked-state snapshots to the session root, server reads them**

- Code impact:
  - Gateway writes JSON snapshots under `<session-root>/gateway/tracked-state/`
  - Server reads those files when projecting state
- User-facing difference:
  - Lower latency (local file read), but staleness depends on gateway write frequency
- Pros:
  - No network dependency
  - Works even if gateway HTTP is temporarily unavailable
- Cons:
  - Tight coupling to gateway's disk layout
  - Staleness is hard to bound
  - Contradicts the spec requirement that "authoritative tracked state lives in memory"

> **PROPOSED: Option A — extend `GatewayClient` with tracked-state HTTP endpoints.**
> Rationale: The existing `GatewayClient` in `gateway_client.py` already implements the HTTP transport pattern for gateway communication (`health()`, `status()`, `submit_request()`). Adding tracked-state endpoints follows the established pattern. The `official-tui-state-tracking` delta spec explicitly requires that "authoritative tracked state lives in memory of the active per-agent control plane," which rules out filesystem artifacts as the authoritative source. HTTP polling latency can be mitigated with server-side caching with short TTL. The gateway is a local sidecar (127.0.0.1), so round-trip overhead is minimal.
>
> **DECISION: Accept PROPOSED.**
> Rationale: The repository already has a live HTTP boundary in `GatewayClient` and `GatewayService`, and the change explicitly keeps authoritative live state in the active control plane's memory. Extending the gateway HTTP surface with tracked and live-control read endpoints matches the existing boundary better than filesystem or in-process coupling.

### Q3) What happens during the attach/detach transition — is there an atomic handoff or a gap?

- Decision: `Deferrable`

#### Why this matters

During gateway attach, the server currently owns TUI tracking for that agent. After attach, the gateway should own it. The transition must either be atomic (no tracking gap) or explicitly allow a brief gap with defined recovery. Without this, the system could lose tracking state, double-track, or return inconsistent results during transitions.

#### Examples

- An operator attaches a gateway to a TUI agent that is mid-turn. The server's direct tracker sees `turn.phase=active`. After attach, the gateway's tracker must pick up from the same state, or the system briefly reports `turn.phase=unknown`.
- A gateway detaches (crash or explicit detach). The server must resume direct tracking without losing the last known state.

#### Why this question arises in code

`RuntimeSessionController.attach_gateway()` (runtime.py ~line 488) and `HoumaoServerService.attach_managed_agent_gateway()` (service.py ~line 1176) handle attach. `TuiTrackingSupervisor` (supervisor.py) manages the tracking worker lifecycle. But there's no coordination between attach/detach and tracker start/stop.

#### Options

**Option A: Accept a brief tracking gap during transitions, with last-known-state caching**

- Code impact:
  - Server stops direct tracking on attach, caches last state
  - Server resumes direct tracking on detach, using cached state as seed
  - State routes return cached state during the transition window
- User-facing difference:
  - Brief window (~1 poll cycle) where state may be slightly stale during attach/detach
- Pros:
  - Simpler implementation
  - No distributed coordination needed
  - Acceptable for a v1 where gateway is optional
- Cons:
  - Observable gap in tracking freshness during transitions
  - Cached state could be stale if transition takes longer than expected

**Option B: Coordinated handoff — server and gateway exchange tracking state during transition**

- Code impact:
  - Attach flow: server snapshots current tracked state → passes to gateway → gateway seeds its tracker → server stops direct tracking
  - Detach flow: gateway snapshots current tracked state → passes to server → server seeds its direct tracker → gateway stops
- User-facing difference:
  - No observable tracking gap
- Pros:
  - Continuous tracking accuracy
  - Clean state transfer
- Cons:
  - More complex implementation
  - Requires a handoff protocol (HTTP or in-process)
  - More failure modes during the handoff itself

> **PROPOSED: Option A for v1 — accept a brief gap with last-known-state caching, document the transition behavior.**
> Rationale: The design already accepts optional gateway as a "richer behavior upgrade." A brief tracking gap (~1 poll cycle, typically 1–2 seconds based on `watch_poll_interval_seconds()` in `TrackingRuntime`) is acceptable for a v1 boundary refactor. The `LiveSessionTracker` in `tui/tracking.py` already maintains in-memory state that can be serialized as a seed. Coordinated handoff (Option B) is a natural follow-up once the basic ownership split is stable. The current `release_known_session()` in the `TrackingRuntime` protocol provides a natural hook for the server side of the transition.
>
> **DECISION: Accept PROPOSED, with one constraint.**
> Rationale: A brief stale window is acceptable for v1, but there must never be dual authoritative trackers for the same agent during handoff. The active owner flips on attach, detach, or health transition, the server may serve last-known state during the gap, and coordinated state transfer can wait for a later change.

### Q4) Who creates the durable headless turn record when admission is gateway-backed?

- Decision: `Blocking`

#### Why this matters

The design separates "live admission" (gateway) from "durable turn catalog" (server). But creating a `ManagedHeadlessTurnRecord` is the bridge between them. If the server creates the record before forwarding to the gateway, it owns the turn lifecycle from the start. If the gateway creates it, the server needs a callback to persist it. This affects data consistency, failure handling, and the turn-inspection route contract.

#### Examples

- A prompt arrives at `POST /houmao/agents/{agent_ref}/turns`. The server must return a `turn_id` in the accepted response. If the gateway creates the turn record, the server needs to know the `turn_id` before responding.
- The gateway crashes mid-execution. The server's durable turn record must reflect whether the turn completed, failed, or is in an unknown state.

#### Why this question arises in code

Currently, `HoumaoServerService` creates `ManagedHeadlessActiveTurnRecord` and `ManagedHeadlessTurnRecord` directly in its headless launch/turn-submission handlers (service.py). The proposed gateway-backed path would need to either keep this server-side creation or delegate it.

#### Options

**Option A: Server creates the durable turn record before forwarding live admission to the gateway**

- Code impact:
  - Server creates `ManagedHeadlessActiveTurnRecord` with `turn_id`
  - Server forwards the prompt + `turn_id` to gateway for live admission
  - Gateway owns queueing, readiness gating, and execution for that turn
  - Server polls or receives callback for turn completion
  ```python
  # service.py (simplified)
  turn_record = self._create_active_turn(agent_ref, prompt)
  gateway_client.submit_prompt(prompt, turn_id=turn_record.turn_id)
  return AcceptedResponse(turn_id=turn_record.turn_id)
  ```
- User-facing difference:
  - `turn_id` is immediately available in the accepted response (same as today)
- Pros:
  - Server always owns turn identity — simpler data consistency
  - Gateway failure doesn't lose the turn record
  - Same response contract as current direct path
- Cons:
  - Server creates a record for a turn that the gateway might reject (admission conflict)
  - Requires cleanup if gateway rejects admission

**Option B: Gateway creates the turn record and pushes it back to the server**

- Code impact:
  - Server forwards prompt to gateway without a pre-created turn record
  - Gateway creates the turn, starts execution, and notifies the server of the `turn_id`
  - Server persists the turn record after notification
- User-facing difference:
  - `turn_id` might not be available immediately in the accepted response (needs async flow)
- Pros:
  - Gateway has full admission authority
  - No speculative record creation
- Cons:
  - Requires a callback/notification mechanism from gateway to server
  - Breaks the current synchronous `turn_id` response contract
  - More complex failure handling

> **PROPOSED: Option A — server creates the durable turn record before forwarding to the gateway.**
> Rationale: The existing `POST /houmao/agents/{agent_ref}/turns` handler in service.py creates the `ManagedHeadlessActiveTurnRecord` synchronously and returns `turn_id` in the accepted response. The `houmao-server-agent-api` delta spec requires the accepted response to "include enough metadata to relate the accepted request to the created headless turn." Server-side creation preserves this contract. The gateway can validate admission separately (returning 409 to the server if busy), and the server can clean up the speculative record on gateway rejection. This keeps the durable catalog fully server-owned as stated in the design.
>
> **DECISION: Accept PROPOSED.**
> Rationale: The server must preserve synchronous `turn_id` issuance and remain the durable turn catalog. Server-side turn record creation before gateway-backed live admission matches the current API contract and keeps reconciliation one-directional back into the server store.

### Q5) Should the CAO compatibility path (`CompatibilityControlCore`) be integrated into the new control-provider seam or remain separate?

- Decision: `Deferrable`

#### Why this matters

The server currently has three distinct ownership paths: TUI tracking, headless management, and CAO compatibility. The change focuses on unifying the first two under the control-provider seam but doesn't mention the third. If CAO-compatible sessions remain on a separate path, that's a deliberate architectural choice that should be documented. If they should eventually go through the seam, the protocol design should accommodate them.

#### Why this question arises in code

`HoumaoServerService` maintains `m_cao_core: CompatibilityControlCore` separately from `m_trackers` and `m_headless_agents`. The `CompatibilityControlCore` in `control_core/core.py` has its own session management, provider adapters, and tmux controller. It doesn't use gateway attach/detach or managed-agent resolution.

#### Options

**Option A: Explicitly exclude CAO compatibility from the control-provider seam in this phase**

- Code impact:
  - No changes to `CompatibilityControlCore`
  - Document that CAO compatibility remains a separate execution path
- User-facing difference:
  - CAO-compatible routes continue working as-is
- Pros:
  - Reduces scope
  - CAO compatibility is being deprecated (public deprecation guard exists)
- Cons:
  - Three ownership paths remain in service.py

**Option B: Include CAO compatibility as a third control-provider implementation**

- Code impact:
  - `CompatibilityControlPlane` wrapping `CompatibilityControlCore`
  - Unified dispatch through the seam for all three transport types
- User-facing difference:
  - None
- Pros:
  - Truly unified dispatch
  - Cleaner service.py
- Cons:
  - Increases scope significantly
  - CAO path is being deprecated — effort may be wasted

> **PROPOSED: Option A — exclude CAO compatibility from the seam in this phase, add a non-goal statement.**
> Rationale: The existing code already has a `public deprecation guard` that rejects deprecated `backend=cao_rest` before session construction (per the `brain-launch-runtime` baseline spec). Investing in unifying a deprecated path through the new control-provider seam is low-value. The design's non-goals section should add an explicit exclusion: "Integrating the CAO compatibility control core into the new control-provider seam."
>
> **DECISION: Accept PROPOSED.**
> Rationale: `CompatibilityControlCore` is a distinct and already-deprecated path, while this change is about managed TUI and headless ownership. Phase 1 should explicitly exclude CAO compatibility from the seam so the refactor does not broaden into an unrelated unification effort.
