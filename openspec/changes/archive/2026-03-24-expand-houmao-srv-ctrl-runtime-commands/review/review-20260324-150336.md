## Summary

- **Solid authority split.** The three-family native tree (`agents` → server-authoritative, `brains` → local build, `admin` → local maintenance) directly mirrors the existing codebase's responsibility boundaries and keeps the pair-native CLI coherent.
- **Most `agents` commands are thin CLI wrappers over existing server routes.** Discovery, state, history, prompt, interrupt, and all turn commands already have server routes and client methods in `HoumaoServerClient`. Only three targeted server expansions are needed: TUI stop, gateway request proxy, and mail routes. This significantly lowers implementation risk.
- **Spec inconsistency on mail surface.** The server API spec lists a `POST /houmao/agents/{agent_ref}/mail/state` route that has no corresponding CLI command and no scenario coverage, while the CLI spec and design only mention `status`, `check`, `send`, `reply`.
- **TUI stop is a non-trivial behavior change.** The current `stop_managed_agent()` in `service.py` (lines 879–920) explicitly rejects TUI agents with a 400 error and a v1 comment. Extending this requires defining how TUI teardown interacts with the CAO session layer and the tmux lifecycle.
- **Relationship between existing `gateway/mail-notifier` routes and new `mail/*` routes is undefined.** The server already has three mail-notifier CRUD routes; the new mail surface is operationally different (send/check/reply vs configure), but the design does not acknowledge the existing routes or clarify the boundary.
- **Task ordering and dependency structure are sound.** Server contract expansion (group 1) before CLI wiring (groups 2–4) before docs and verification (group 5) is the right sequencing.

## Findings

### Must Fix / Blocking

1. **Server API spec lists `POST /houmao/agents/{agent_ref}/mail/state` with no corresponding CLI command, no scenario, and no design mention.** The server spec (line 54) includes five mail routes including `mail/state`, but the CLI spec's `agents mail` family (lines 68–93) only defines commands for `status`, `check`, `send`, `reply`. The design's command tree (line 73) also omits `mail state`. Either `mail/state` is a typo for `mail/status` (in which case the spec has a duplicate under a different HTTP method), or it is a distinct route that needs a CLI command and scenario coverage.

> **DECISION: Refine.**
> Rationale: The inconsistency is real, but `mail/state` is not a typo. The existing gateway surface already defines `POST /v1/mail/state` via `GatewayMailStateRequestV1` / `GatewayMailStateResponseV1` and uses it to mark one processed `message_ref` read. For this change, however, the native CLI shape is intentionally `status`, `check`, `send`, and `reply`. The follow-up artifact revision should therefore either add an explicit native command for that state-update operation or, preferably for v1 scope, remove `mail/state` from the new server delta so the public pair contract matches the designed CLI.

### Should Fix / Important

2. **Design and specs do not clarify the relationship between existing `gateway/mail-notifier` CRUD routes and the new `mail/*` operational routes.** The server currently exposes `GET/PUT/DELETE /houmao/agents/{agent_ref}/gateway/mail-notifier` (for mail-notifier configuration) via `GatewayMailNotifierStatusV1` and `GatewayMailNotifierPutV1` models. The new `mail/*` routes are operationally different (send, check, reply, status), but the design does not acknowledge the existing routes or explain whether they coexist, are superseded, or serve a different layer. Without this clarification, implementers may conflict the two surfaces or duplicate capability. A brief note in the design's Decision 3 section would suffice.

> **DECISION: Accept.**
> Rationale: The reviewer is correct that the artifacts should distinguish the two layers explicitly. The existing `gateway/mail-notifier` routes are configuration for background notification behavior, while the proposed `mail/*` routes are foreground mailbox operations. They should coexist, not replace each other, and the design should say that directly so implementers do not collapse them into one surface.

3. **TUI stop implementation strategy is under-specified.** The current `stop_managed_agent()` in `service.py` (lines 879–920) guards against TUI agents with an explicit v1 rejection. The design says "transport-neutral for both TUI and headless managed agents" but does not specify the TUI teardown mechanism. In the existing codebase, TUI agents are backed by CAO sessions that can be deleted via `DELETE /cao/sessions/{session_name}`, and the tmux session may need separate cleanup. Task 1.2 says "extend service logic so managed-agent stop supports both TUI and headless" but doesn't specify whether TUI stop should (a) proxy a CAO session delete, (b) send tmux kill-session, (c) use a combination, or (d) introduce a new server-side teardown path. Adding a brief implementation note to the design or expanding the task description would reduce ambiguity during implementation.

> **DECISION: Refine.**
> Rationale: The ambiguity is real, but v1 does not need a new tmux-specific teardown mechanism. The intended TUI stop path should be server-owned managed-agent resolution followed by the existing CAO-compatible session-delete lifecycle, with the server reusing its existing registration and tracker cleanup hooks. The later artifact revision should state that `agents stop` for TUI wraps pair-owned session deletion rather than inventing a second teardown path.

4. **`agents show` lacks a clear mapping to an existing server route.** The design says "`show` is the detail-oriented inspection verb" (line 93) and distinguishes it from `state` (lightweight). The server currently exposes three relevant GET routes: `/houmao/agents/{agent_ref}` → `HoumaoManagedAgentIdentity`, `/houmao/agents/{agent_ref}/state` → `HoumaoManagedAgentStateResponse`, and `/houmao/agents/{agent_ref}/state/detail` → `HoumaoManagedAgentDetailResponse`. The CLI spec has a scenario for `agents state` but none for `agents show`. Which server route does `show` map to? If it maps to `state/detail`, that should be stated. If it maps to the identity route, then "detail-oriented" is misleading since `HoumaoManagedAgentIdentity` is a subset of `HoumaoManagedAgentStateResponse`.

> **DECISION: Accept.**
> Rationale: `show` should map to `GET /houmao/agents/{agent_ref}/state/detail`, and `state` should map to `GET /houmao/agents/{agent_ref}/state`. That is the only mapping that matches the current design language of "detail-oriented" versus "lightweight" while reusing existing server routes without ambiguity.

5. **Tasks do not mention extending `HoumaoServerClient` with new methods for the three new server route families.** The existing client (`src/houmao/server/client.py`) already has methods for all currently-existing routes. The three new route families (gateway request proxy, mail operations, TUI stop extension) will each need new client methods. Task group 1 mentions "client helpers" (task 1.3) for server-side work, but the CLI tasks (groups 2–3) don't explicitly call out the client-side additions that the `agents gateway prompt`, `agents mail *`, etc. commands will consume. This should be clarified — either task 1.3 covers all new client methods, or the CLI tasks should reference the client work they depend on.

> **DECISION: Reject.**
> Rationale: Task 1.3 already covers this work: it explicitly says to add `houmao-server` routes and client helpers for the new managed-agent gateway and mail operations. In this repository, those client helpers live on `HoumaoServerClient`, so the dependency is already captured in the right phase before CLI task groups 2 and 3. I do not think a separate task is required just to restate that pattern.

6. **The `agents` command family is large enough to warrant a module-organization decision.** The existing `src/houmao/srv_ctrl/commands/` directory uses one file per command group (`agent_gateway.py`, `launch.py`, `install.py`, `cao.py`, `local_compat.py`). The `agents` family alone has ~20 commands across 4 sub-groups (core, gateway, mail, turn). Putting all of these in a single `agents.py` file would be inconsistent with the current module granularity. The tasks (2.1) mention wiring the groups but don't address file layout. A `commands/agents/` sub-package with per-subgroup modules (e.g., `core.py`, `gateway.py`, `mail.py`, `turn.py`) would be more maintainable and consistent with the existing pattern.

> **DECISION: Accept.**
> Rationale: The reviewer is right about module scale. The existing `srv_ctrl/commands/` layout uses small command-group modules, and the planned `agents` family is materially larger than any current single module. The follow-up artifact revision should record a `commands/agents/` sub-package with per-subgroup modules such as `core.py`, `gateway.py`, `mail.py`, and `turn.py`, plus shared helpers where needed.

### Nice To Have

7. **Design open question about `agents launch` as a future route family could be resolved now.** The design asks whether a future `agents launch` should exist or whether top-level `launch` is the long-term shape. Deciding this now would prevent the `agents` namespace from growing in a direction that later conflicts. Since `launch` already involves local artifact materialization (brain build + tmux session creation + server registration), it straddles the local/server boundary in a way that doesn't fit cleanly under server-authoritative `agents`. Keeping it at top level is the more natural long-term shape.

> **DECISION: Accept.**
> Rationale: I want to resolve this now rather than leave the namespace direction open. `launch` should remain a top-level long-term command because it spans local artifact materialization, compatibility or server lifecycle orchestration, and registration concerns that do not fit the server-authoritative `agents` family cleanly. This change should not plan a future `agents launch` family.

8. **No explicit error response model for capability-unavailable rejections.** Both specs use phrasing like "fails with explicit availability guidance" and "rejects the request explicitly" for scenarios where mail capability is unavailable or no live gateway is attached. The existing codebase uses `CaoApiError` (HTTP 4xx with `detail` string). The specs don't specify whether a new structured error model is needed or whether the existing error pattern suffices. Specifying this would help implementers produce consistent error responses across the new routes.

> **DECISION: Reject.**
> Rationale: A new structured error model is unnecessary for this change. The existing server contract already uses ordinary HTTP error statuses with a `detail` string, and the CLI already maps those through `CaoApiError` and `click.ClickException`. "Explicit availability guidance" can and should be expressed through that existing error pattern rather than by introducing a second error-response schema.

9. **`brains build` CLI argument mapping is unspecified.** The existing `houmao-cli build-brain` command uses argparse with 11 options (`--recipe`, `--blueprint`, `--tool`, `--skill`, `--config-profile`, `--cred-profile`, `--launch-overrides`, `--home-id`, `--reuse-home`, `--agent-def-dir`, `--runtime-root`). Task 4.1 says "local wrapper over the existing brain-build artifact workflow" but doesn't specify which subset of arguments to expose on the Click-based `brains build` command, or whether it should mirror the full argparse interface. Since `brains build` calls `build_brain_home()` directly (not via subprocess), the Click command needs explicit argument design.

> **DECISION: Refine.**
> Rationale: The reviewer is right that the argument surface needs to be made explicit, but I do not want this change to promise a verbatim copy of every `houmao-cli build-brain` convenience flag. The follow-up artifact revision should state that `brains build` is a Click wrapper over the lower-level builder inputs that map directly to `BuildRequest` and `brain_builder.py` semantics, with any extra convenience flags such as `--blueprint` added only if we intentionally choose them.

The design doesn't name Click explicitly, but the tasks' wiring language (2.1: "wire into the top-level command tree") is consistent with Click group registration.

- **Command module organization: ⚠️** Current layout is one `.py` file per command group under `src/houmao/srv_ctrl/commands/`. The `agents` family is significantly larger than any existing group. See Finding 6.

- **Client pattern alignment: ✅** The existing `HoumaoServerClient` in `src/houmao/server/client.py` already provides typed methods for all current routes. The new server routes should follow the same pattern (add methods to the existing client class). Task 1.3 mentions "client helpers" which is consistent.

- **Server base URL resolution: ✅** All existing `houmao-srv-ctrl` commands use `resolve_server_base_url()` from `src/houmao/srv_ctrl/commands/common.py` (defaults to `http://127.0.0.1:{port}`). The design's task 2.2 mentions "pair base-URL resolution" as a shared helper, which aligns with this existing pattern.

- **Error handling pattern: ✅** Existing commands raise `click.ClickException` for user-facing errors and catch `CaoApiError` from client calls. The new commands should follow this pattern. The spec scenarios describe explicit failure without specifying mechanism, which is compatible.

- **Agent reference addressing: ✅** The design uses `<agent-ref>` as a positional argument. The existing server routes already accept `{agent_ref}` for managed-agent alias resolution. The existing `agent_gateway.py` uses `--agent` as a Click option; the shift to positional for the native tree is a deliberate design decision documented in the design (line 90).

- **JSON output pattern: ✅** Existing commands output JSON via `json.dumps(response.model_dump(), indent=2, sort_keys=True)`. The design's task 2.2 mentions "JSON rendering" as a shared helper, which could standardize this pattern.

- **Local command pattern (brains, admin): ✅** The `brains build` command wrapping `build_brain_home()` from `src/houmao/agents/brain_builder.py` and `admin cleanup-registry` wrapping `cleanup_stale_live_agent_records()` from `src/houmao/agents/realm_controller/registry_storage.py` is consistent with how `houmao-cli` already wraps these functions (see `realm_controller/cli.py` lines 419–490 for build-brain, lines 790–818 for cleanup-registry).

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What is the intended semantic of `POST /houmao/agents/{agent_ref}/mail/state` in the server spec?

- Decision: `Blocking`

#### Why this matters

The server API spec lists five mail routes but the CLI spec and design only surface four CLI commands (`status`, `check`, `send`, `reply`). If `mail/state` is a distinct route, it needs a CLI command and scenario. If it is a typo or duplicate, it should be removed from the spec to avoid implementer confusion.

#### Examples

- If `mail/state` means "get mailbox state" and is synonymous with `GET /mail/status`, then only one should exist (and they have different HTTP methods — GET vs POST).
- If `mail/state` means "set/mutate mailbox state" (e.g., enable/disable), it is a fundamentally different operation that needs its own design rationale and CLI verb.

#### Why this question arises in code

- The server spec (lines 50–55) lists both `GET /houmao/agents/{agent_ref}/mail/status` and `POST /houmao/agents/{agent_ref}/mail/state` without scenario coverage for the latter.
- The CLI spec (lines 68–77) lists only `status`, `check`, `send`, `reply` — no `state`.
- The design command tree (lines 72–75) also has no `mail state` verb.

#### Options

**Option A: `mail/state` is a typo for `mail/status` — remove it from the server spec.**

- Code impact:
  - Remove line 54 from `specs/houmao-server-agent-api/spec.md`.
  - No implementation change needed.
- User-facing difference:
  - None; `GET /mail/status` covers status inspection.
- Pros:
  - Simplest resolution, eliminates ambiguity.
- Cons:
  - Loses the route if it was intentionally distinct.

**Option B: `mail/state` is a distinct mutating route — add CLI command and scenario.**

- Code impact:
  - Add `agents mail state <agent-ref>` CLI command.
  - Add scenario in both specs.
  - Add server route implementation and client method.
- User-facing difference:
  - Operators get a way to mutate mailbox state (e.g., enable/disable mail follow-up).
- Pros:
  - Complete API surface for mailbox lifecycle management.
- Cons:
  - Increases scope; may not be needed in v1 if the existing `PUT/DELETE gateway/mail-notifier` routes already cover configuration.

> **PROPOSED: Option A — treat `mail/state` as a typo and remove it from the server spec.**
> Rationale: The existing `PUT/DELETE /houmao/agents/{agent_ref}/gateway/mail-notifier` routes already handle mail-notifier configuration. The design and CLI spec consistently omit `mail state`, and the server spec provides no scenario for it. Introducing a new mutating route without design rationale would increase scope and conflict with the existing mail-notifier CRUD surface.
>
> **DECISION: Refine the proposal. `mail/state` is a distinct mark-read operation, but it should not stay in this change unless the native CLI also exposes it.**
> Rationale: The codebase already gives `mail/state` a concrete meaning on the gateway surface: `POST /v1/mail/state` marks one processed message read and returns `GatewayMailStateResponseV1`. That means Option A is directionally right about narrowing the current change, but wrong about it being a typo. For this change, I want the later artifact revision to remove `mail/state` from the new server delta rather than broaden the native CLI immediately.

### Q2) How should `agents gateway prompt` and `agents prompt` be distinguished for the operator?

- Decision: `Deferrable`

#### Why this matters

The design defines two prompt paths: `agents prompt <agent-ref>` (transport-neutral, via `POST /houmao/agents/{agent_ref}/requests`) and `agents gateway prompt <agent-ref>` (gateway-mediated, via the new `POST /houmao/agents/{agent_ref}/gateway/requests`). Both accept `--prompt "..."`. An operator who doesn't understand the gateway architecture may not know which to use, and the difference in behavior (direct vs gateway-queued, different admission and execution semantics) is not surfaced in the CLI spec scenarios.

#### Examples

- Operator runs `agents prompt agent-1 --prompt "fix the bug"` → goes through transport-neutral path (direct to headless controller or TUI send-keys).
- Operator runs `agents gateway prompt agent-1 --prompt "fix the bug"` → goes through live gateway queue with admission control, execution tracking, and potential mail notification.
- If the operator just wants to send a prompt and doesn't care about the path, which should they use?

#### Why this question arises in code

- The existing `POST /houmao/agents/{agent_ref}/requests` route (`submit_managed_agent_request` in `service.py`) already handles prompt submission via `HoumaoManagedAgentRequestEnvelope`.
- The new gateway request proxy adds a second path with different semantics (gateway admission, queuing, execution lifecycle).
- Neither the CLI spec nor the design explains when an operator should prefer one over the other.

#### Options

**Option A: Document the distinction in help text and let operators choose.**

- Code impact:
  - Add clear help strings: `agents prompt` = "direct transport-neutral submission", `agents gateway prompt` = "gateway-mediated with admission control and execution tracking".
- User-facing difference:
  - Operators see explanatory help text; choice is explicit.
- Pros:
  - Transparent; no hidden behavior.
- Cons:
  - Requires operators to understand the gateway model.

**Option B: Make `agents prompt` the default recommendation and document `agents gateway prompt` as the advanced/explicit path.**

- Code impact:
  - Same as Option A, but with stronger wording in docs: "use `agents prompt` unless you specifically need gateway-mediated request lifecycle".
- User-facing difference:
  - Clear default recommendation reduces decision fatigue.
- Pros:
  - Simpler mental model for most operators.
- Cons:
  - May hide the gateway path from operators who would benefit from it.

> **PROPOSED: Option B — make `agents prompt` the documented default and position `agents gateway prompt` as the explicit gateway-mediated path.**
> Rationale: The design already positions `agents` as the primary pair-native surface. `agents prompt` maps to the existing transport-neutral `POST /requests` route which already handles both TUI and headless. Gateway-mediated submission is a specialized path for operators who need admission control, execution lifecycle tracking, and mail notification. Help text and docs should make this layering clear without requiring operators to understand gateway internals for the common case.
>
> **DECISION: Accept the proposed solution. `agents prompt` is the default documented path, and `agents gateway prompt` is the explicit gateway-mediated path.**
> Rationale: This keeps the common operator workflow on the existing transport-neutral managed-agent request route while still exposing the gateway queue and execution lifecycle when an operator explicitly wants that behavior. The distinction should be carried in help text and docs, not left implicit.

### Q3) Should `agents show` map to `/state/detail` or to the identity route?

- Decision: `Blocking`

#### Why this matters

The implementation needs a clear mapping from CLI verb to server route. The design says `show` is "detail-oriented" and `state` is "lightweight," but the server has three relevant routes with different response payloads:

| Route | Response Model | Content |
|-------|---------------|---------|
| `GET /houmao/agents/{agent_ref}` | `HoumaoManagedAgentIdentity` | IDs, transport, tool, paths |
| `GET /houmao/agents/{agent_ref}/state` | `HoumaoManagedAgentStateResponse` | Identity + availability, turn, last_turn, diagnostics, mailbox, gateway |
| `GET /houmao/agents/{agent_ref}/state/detail` | `HoumaoManagedAgentDetailResponse` | Full detail view |

#### Examples

- `agents show agent-1` should return a rich overview — but is "rich" the identity record or the full detail view?
- `agents state agent-1` should return a lightweight status — but `HoumaoManagedAgentStateResponse` already includes turn info, diagnostics, mailbox, and gateway summaries, which may not feel "lightweight."

#### Why this question arises in code

- The design (line 93) describes the semantic difference but doesn't map to routes.
- The CLI spec has a scenario for `agents state` but none for `agents show`.
- Implementation will need to pick a route and format the output.

#### Options

**Option A: `show` → `/state/detail`, `state` → `/state`**

- Code impact:
  - `agents show` calls `client.get_managed_agent_state_detail()`.
  - `agents state` calls `client.get_managed_agent_state()`.
- User-facing difference:
  - `show` gives the most complete view; `state` gives the operational snapshot.
- Pros:
  - Clean mapping; both routes already exist.
  - Matches the "detail-oriented" vs "lightweight" language.
- Cons:
  - The identity route (`GET /houmao/agents/{agent_ref}`) has no CLI command.

**Option B: `show` → `/houmao/agents/{agent_ref}` (identity), `state` → `/state`**

- Code impact:
  - `agents show` calls `client.get_managed_agent()`.
  - `agents state` calls `client.get_managed_agent_state()`.
- User-facing difference:
  - `show` gives the static identity record; `state` gives the dynamic status.
- Pros:
  - Clear separation of concerns (identity vs runtime state).
- Cons:
  - `show` returning less data than `state` contradicts "detail-oriented."

> **PROPOSED: Option A — map `show` to `/state/detail` and `state` to `/state`.**
> Rationale: The design explicitly describes `show` as "detail-oriented" (line 93), which maps naturally to the existing `HoumaoManagedAgentDetailResponse` from `GET /state/detail`. The `HoumaoManagedAgentStateResponse` from `GET /state` serves as the operational snapshot. Both routes already exist with distinct response models in the server and client. The identity route can remain accessible via `agents list` (which returns `HoumaoManagedAgentIdentity` for each agent).
>
> **DECISION: Accept the proposed solution. `show` maps to `/state/detail`, and `state` maps to `/state`.**
> Rationale: That mapping matches the existing server models, preserves the intended semantic split in the design, and avoids making `show` return a less informative identity payload than `state`.

### Q4) Should the v1 server mail routes require an attached live gateway, or should the server implement a direct runtime-backed fallback?

- Decision: `Deferrable`

#### Why this matters

This is also listed as an open question in the design. The answer affects implementation scope and the server's failure semantics for mail operations. If v1 requires a live gateway, mail commands fail when no gateway is attached — which may be the common state for headless agents that were launched without a gateway.

#### Examples

- Operator runs `agents mail check agent-1` but agent-1 has no gateway attached → should this fail immediately or should the server have a fallback path?
- Headless agent with filesystem mailbox configured at launch time → the server could read the mailbox directly without a gateway.

#### Why this question arises in code

- The existing mail-notifier routes (`gateway/mail-notifier`) are explicitly gateway-scoped and require an attached gateway.
- The new mail routes are under `/houmao/agents/{agent_ref}/mail/*` — a server-level path, not a gateway sub-path.
- The design (Decision 3) says the server "MAY satisfy those routes by proxying an attached live gateway or by using another pair-owned runtime-backed mail path."

#### Options

**Option A: v1 requires an attached live gateway; fail explicitly otherwise.**

- Code impact:
  - Server resolves managed agent → checks for attached gateway → proxies mail request through gateway.
  - Returns 409/503 if no gateway attached.
- User-facing difference:
  - Mail commands only work when gateway is running.
- Pros:
  - Simpler v1 implementation; reuses existing gateway mail infrastructure.
  - Clear failure semantics.
- Cons:
  - Limits mail utility for headless agents without persistent gateways.

**Option B: v1 implements direct runtime-backed mail path for filesystem mailboxes.**

- Code impact:
  - Server reads mailbox config from managed-agent metadata.
  - For filesystem transport, reads/writes mailbox directory directly.
  - For other transports, may still require gateway proxy.
- User-facing difference:
  - Mail commands work for agents with filesystem mailboxes even without a gateway.
- Pros:
  - More useful for headless-only workflows.
- Cons:
  - Larger implementation scope; server needs to understand mailbox transport details.

> **PROPOSED: Option A — v1 requires an attached live gateway for mail routes; fail explicitly otherwise.**
> Rationale: The existing gateway mailbox infrastructure (`gateway_mailbox.py`, `gateway_service.py`) already handles mail operations through the gateway process. Implementing a direct runtime-backed path in v1 would require the server to duplicate mailbox transport resolution (filesystem, JMAP/Stalwart, etc.) that currently lives in the gateway layer. Starting with gateway-proxied mail and clear 409/503 errors when no gateway is attached keeps v1 scope manageable. A direct path can be added in a follow-up if operators need mail on gateway-less agents.
>
> **DECISION: Accept the proposed solution. v1 mail routes require an attached live gateway and fail explicitly otherwise.**
> Rationale: The current gateway layer already owns mailbox status, check, send, reply, and mark-read behavior. Re-implementing mailbox transport handling directly in `houmao-server` would broaden scope substantially and duplicate gateway logic. A gateway-required v1 keeps the pair contract coherent and gives clear failure semantics; direct runtime-backed mail can be a later change if needed.
