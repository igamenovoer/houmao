## Summary

- **Solid:** The scope is well-defined and the gateway-versus-transport boundary is the right place to cut. Proposal, design, specs, and tasks are internally coherent and mutually reinforcing. The decision to keep `/v1/mail/*` separate from `POST /v1/requests` is architecturally sound.
- **Solid:** The fabricated-fixture testing strategy is pragmatic and avoids requiring full launched-agent flows for the first proof of the gateway mail facade.
- **Solid:** The seven design decisions are well-reasoned with clear alternatives-considered sections.
- **Risky:** The existing `MailboxResolvedConfig` dataclass requires `filesystem_root: Path` as a non-optional field (line 29 of `mailbox_runtime_models.py`). Every function in `mailbox_runtime_support.py` that touches transport bindings hard-rejects non-filesystem transports. The design does not specify how to restructure these data models for transport polymorphism, and implementation will stall immediately on this gap.
- **Risky:** No JMAP Python library exists in current dependencies, and the design does not specify which one to use. This is a foundational technical choice that affects the Stalwart adapter's implementation shape.
- **Risky:** The `message_ref` shape (the change's core abstraction contract) is deferred to an open question in the design. Implementation of all four `/v1/mail/*` routes depends on this decision.

## Findings

### Must Fix / Blocking

1. **`MailboxResolvedConfig` requires `filesystem_root: Path` as a non-optional field and cannot represent Stalwart bindings.** The current dataclass at `src/houmao/agents/mailbox_runtime_models.py:23-31` has `filesystem_root: Path` as mandatory. `MailboxDeclarativeConfig` (line 13-19) also lacks fields for JMAP endpoint, login identity, or credential references. The design says the manifest will persist "transport-safe endpoint or credential-reference metadata" but never specifies how these models change. Every consumer in `mailbox_runtime_support.py` — `mailbox_env_bindings()` (line 272), `mailbox_env_var_names()` (line 301), `bootstrap_resolved_mailbox()` (line 311), `resolved_mailbox_config_from_payload()` (line 111), `resolve_effective_mailbox_config()` (line 161), `parse_declarative_mailbox_config()` (line 55) — hard-rejects non-filesystem transports with `raise ValueError`. The design must specify the concrete model restructuring strategy before implementation can begin: whether to make `filesystem_root` optional and add Stalwart-specific optional fields, or to split into a transport-polymorphic union, or to use a metadata dict for transport-specific bindings.

> **DECISION: Accept.**
> Rationale: This is a real blocker. The current runtime mailbox binding model and its manifest round-trip are filesystem-only, so implementation cannot proceed cleanly until the binding model becomes transport-polymorphic. This is the central abstraction of the change and needs a concrete interface definition in the design or a dedicated spec.

> **DECISION: Refine.**
> Rationale: The gap is real, but it does not require a separate top-level spec. ## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should `MailboxResolvedConfig` be restructured to support both filesystem and Stalwart transport bindings?

- Decision: `Blocking`

#### Why this matters

- `MailboxResolvedConfig` is consumed by `LaunchPlan`, `mailbox_env_bindings()`, `prepare_mail_prompt()`, `ensure_mailbox_command_ready()`, `bootstrap_resolved_mailbox()`, the gateway notifier, and manifest serialization/deserialization. Every one of these consumers currently assumes `filesystem_root: Path` exists and is valid.

#### Examples

- A Stalwart session needs to persist JMAP endpoint URL, login identity, and a credential reference — none of which fit in `filesystem_root: Path`.
- `resolved_mailbox_config_from_payload()` (line 111-158) requires `filesystem_root` to be non-None and rejects any transport that isn't `filesystem`.

#### Why this question arises in code

- `MailboxResolvedConfig` at `mailbox_runtime_models.py:23-31` is a frozen dataclass with `filesystem_root: Path` as a required positional-style field.
- `redacted_payload()` serializes `filesystem_root` unconditionally.
- `mailbox_env_bindings()` at `mailbox_runtime_support.py:272-298` calls `resolve_active_mailbox_dir()` against `config.filesystem_root`, which will fail for Stalwart configs.

#### Options

**Option A: Make `filesystem_root` optional and add transport-specific optional fields**

- Code impact:
  - Change `filesystem_root: Path` to `filesystem_root: Path | None = None`.
  - Add optional fields: `jmap_url: str | None = None`, `login_identity: str | None = None`, `credential_ref: str | None = None`.
  - Each consumer dispatches on `transport` and accesses the relevant fields.
  - ```python
    @dataclass(frozen=True)
    class MailboxResolvedConfig:
        transport: MailboxTransport
        principal_id: str
        address: str
        bindings_version: str
        filesystem_root: Path | None = None
        jmap_url: str | None = None
        login_identity: str | None = None
        credential_ref: str | None = None
    ```
- User-facing difference:
  - No user-facing change; internal model restructuring only.
- Pros:
  - Minimal structural change to the existing codebase.
  - Single config type avoids union-dispatch complexity.
- Cons:
  - Optional fields that are semantically required for a given transport are not type-enforced.
  - As more transports are added, the config accumulates unrelated optional fields.

**Option B: Use a transport-discriminated union of frozen dataclasses**

- Code impact:
  - Define `FilesystemMailboxConfig` and `StalwartMailboxConfig` as separate frozen dataclasses sharing common fields.
  - Define `MailboxResolvedConfig = FilesystemMailboxConfig | StalwartMailboxConfig` as a union type.
  - Each consumer narrows on `transport` and gets type-safe access to transport-specific fields.
  - ```python
    @dataclass(frozen=True)
    class _MailboxConfigBase:
        transport: MailboxTransport
        principal_id: str
        address: str
        bindings_version: str

    @dataclass(frozen=True)
    class FilesystemMailboxConfig(_MailboxConfigBase):
        filesystem_root: Path

    @dataclass(frozen=True)
    class StalwartMailboxConfig(_MailboxConfigBase):
        jmap_url: str
        login_identity: str
        credential_ref: str

    MailboxResolvedConfig = FilesystemMailboxConfig | StalwartMailboxConfig
    ```
- User-facing difference:
  - No user-facing change.
- Pros:
  - Type-safe: transport-specific fields are guaranteed present for the right transport.
  - Aligns with `mypy --strict` enforcement.
  - Clean extension point for future transports.
- Cons:
  - More structural change required.
  - Union narrowing with `isinstance()` at every consumer site.
  - `redacted_payload()` needs per-type implementation.

**Option C: Keep a flat dataclass with a transport-specific metadata dict**

- Code impact:
  - Add `transport_metadata: dict[str, Any] = field(default_factory=dict)` to the existing config.
  - Transport-specific values go into the metadata dict.
  - ```python
    @dataclass(frozen=True)
    class MailboxResolvedConfig:
        transport: MailboxTransport
        principal_id: str
        address: str
        bindings_version: str
        filesystem_root: Path | None = None
        transport_metadata: dict[str, Any] = field(default_factory=dict)
    ```
- User-facing difference:
  - No user-facing change.
- Pros:
  - Least structural change.
  - Flexible for future transport additions.
- Cons:
  - Loses type safety entirely for transport-specific fields.
  - Contradicts the project's `mypy --strict` convention and the preference for typed models.
  - Runtime key-access errors instead of type errors.

> **PROPOSED: Option B — Use a transport-discriminated union of frozen dataclasses.**
> Rationale: The codebase enforces `mypy --strict` and uses `@dataclass(frozen=True)` for internal value objects (`models.py:56`, `mailbox_runtime_models.py:12`). A discriminated union lets `isinstance()` narrowing provide type-safe field access, which aligns with the existing `BackendKind`-based dispatch patterns in `launch_plan.py` and the `GatewayAttachBackendMetadata*` union in `gateway_models.py:172-176`. Option A's bag of optional fields would accumulate over time and degrade type safety. Option C's metadata dict directly contradicts the strict typing philosophy.
>
> **DECISION: Accept Option B.**
> Rationale: Accept Option B, with the same union pattern applied to mailbox config modeling more broadly where needed. A transport-discriminated union of frozen dataclasses fits the repository's strict typing conventions better than a flat bag of optional fields, keeps transport-specific requirements type-safe, and gives each transport binding its own redacted serialization path.

### Q2) Which JMAP client approach should the Stalwart adapter use?

- Decision: `Blocking`

#### Why this matters

- The Stalwart adapter's implementation shape, error handling, and test mock strategy all depend on the JMAP client choice.

#### Examples

- `jmapc` provides typed Python models for JMAP methods but may lag behind Stalwart-specific extensions.
- A raw-HTTP client using `urllib.request` (matching the pattern in `gateway_client.py:141-155`) gives full control but requires manual JMAP method/response modeling.

#### Why this question arises in code

- The current codebase uses only `urllib.request` for HTTP (no `requests`, no `httpx`, no `aiohttp`). See `gateway_client.py` and `cao/rest_client.py`.
- Adding a third-party JMAP library introduces a new dependency class.

#### Options

**Option A: Use a thin raw-HTTP JMAP client built on `urllib.request`**

- Code impact:
  - New module (e.g., `src/houmao/agents/stalwart_jmap_client.py`) with typed request/response models using Pydantic, wrapping JMAP method calls over HTTP POST.
- User-facing difference:
  - No new pip dependency.
- Pros:
  - Consistent with existing HTTP client patterns (`gateway_client.py`, `cao/rest_client.py`).
  - No external dependency risk.
  - Full control over JMAP method modeling.
- Cons:
  - More boilerplate code.
  - Must manually model JMAP `Email/query`, `Email/get`, `Email/set`, `EmailSubmission/set`.

**Option B: Use `jmapc` library**

- Code impact:
  - Add `jmapc` to `pyproject.toml` dependencies.
  - Stalwart adapter wraps `jmapc.Client` methods.
- User-facing difference:
  - New pip dependency.
- Pros:
  - Reduces JMAP boilerplate significantly.
  - Typed JMAP models are pre-built.
- Cons:
  - Introduces a new third-party dependency.
  - May not cover Stalwart management API (account provisioning is separate from JMAP).
  - Library maturity and maintenance risk.

> **PROPOSED: Option A — Use a thin raw-HTTP JMAP client built on `urllib.request`.**
> Rationale: The codebase exclusively uses `urllib.request` for HTTP interactions (see `gateway_client.py:135-155`, `cao/rest_client.py`). Adding a third-party JMAP library would introduce an inconsistent dependency pattern. The set of JMAP methods needed is small (Email/query, Email/get, Email/set, EmailSubmission/set) and the Stalwart management API (domain/account provisioning) is a separate REST surface that `jmapc` wouldn't cover anyway. A thin client with typed Pydantic request/response models matches the project's existing patterns.
>
> **DECISION: Accept Option A.**
> Rationale: Accept Option A. The repository already standardizes on `urllib.request` for HTTP client work, the JMAP method set needed in this change is small, and Stalwart provisioning still requires separate management API calls that a third-party JMAP library would not eliminate.

### Q3) What shape should `message_ref` take in the shared gateway mailbox contract?

- Decision: `Blocking`

#### Why this matters

- `message_ref` is the opaque handle used by managed agents and runtime callers for later `reply` targeting. Its shape affects the `/v1/mail/reply` route payload, the normalized message metadata returned by `/v1/mail/check`, the filesystem adapter's ref-from-message-id mapping, and the Stalwart adapter's ref-from-JMAP-id mapping.

#### Examples

- Filesystem transport: a `message_ref` might encode the existing `msg-{timestamp}-{uuid}` message id.
- Stalwart transport: a `message_ref` might encode a JMAP email id or an RFC 5322 `Message-ID` header value.
- A mixed environment where agents receive refs from both transports needs unambiguous ref identity.

#### Why this question arises in code

- The existing `MailboxMessage.message_id` (protocol.py:131) uses `msg-{YYYYMMDDTHHMMSSZ}-{uuid4}` format validated by `MESSAGE_ID_PATTERN`.
- The shared gateway contract needs a ref that works for both transports without exposing transport-native id formats.

#### Options

**Option A: Opaque string with transport-prefix convention**

- Code impact:
  - `message_ref` is a plain `str` in all gateway models.
  - Convention: `fs:{message_id}` for filesystem, `stalwart:{jmap_email_id}` for Stalwart.
  - Each adapter parses/generates its own prefix.
  - ```python
    class MailboxCheckMessageV1(_StrictGatewayModel):
        message_ref: str
        thread_ref: str | None = None
        ...
    ```
- User-facing difference:
  - Agents see refs like `fs:msg-20260319...` or `stalwart:abc123`.
- Pros:
  - Simplest implementation.
  - Easy to parse in adapters.
- Cons:
  - Prefix convention leaks transport identity into the "opaque" ref.
  - Agents might start pattern-matching on the prefix.

**Option B: Opaque string without convention, adapter-owned mapping**

- Code impact:
  - `message_ref` is a plain `str` — adapters generate whatever opaque string they want.
  - Filesystem adapter might use the message id directly; Stalwart adapter might use a base64-encoded compound key.
  - No prefix convention; adapters maintain internal lookup from ref → transport-native id.
- User-facing difference:
  - Agents see truly opaque refs like `ZnM6bXNnLTIwMjYwMzE5...`.
- Pros:
  - Truly opaque — no transport identity leakage.
  - Agents cannot pattern-match.
- Cons:
  - Harder to debug.
  - Adapters need internal decode/lookup logic.

**Option C: Small structured object with transport and id fields**

- Code impact:
  - `message_ref` is a small Pydantic model serialized as a JSON object in the gateway response.
  - ```python
    class MessageRef(_StrictGatewayModel):
        transport: str
        id: str
    ```
  - Gateway routes accept/return this structured ref.
- User-facing difference:
  - Agents handle a structured object instead of a string.
- Pros:
  - Explicit transport discrimination.
  - Self-describing.
- Cons:
  - More complex API surface.
  - Violates the "opaque ref" principle — transport identity is explicitly exposed.

> **PROPOSED: Option A — Opaque string with transport-prefix convention.**
> Rationale: The simplicity of a plain string ref is important for the first version. The design explicitly calls for "opaque `message_ref`" values in the shared contract (`design.md` Decision 3), and a transport-prefix convention (`fs:` / `stalwart:`) gives adapters a deterministic, zero-state way to recognize and decode their own refs without requiring an external lookup table. The prefix does not expose transport-internal identities (the suffix is still adapter-owned), and agents are instructed by projected skills to treat refs as opaque handles. The prefix also makes debugging and audit log inspection easier, which matters for a first implementation. If prefix leakage becomes a real problem later, a migration to fully opaque refs is straightforward.
>
> **DECISION: Refine Option A.**
> Rationale: Refine Option A. The shared contract should keep `message_ref` as a plain string, and in v1 adapters may encode that string with a transport-prefixed convention so the gateway can decode it statelessly. Callers still must treat the entire value as opaque, and no caller behavior should depend on the prefix shape.

### Q4) Should runtime `mail check/send/reply` prefer the live gateway facade immediately in this change, or defer that preference?

- Decision: `Deferrable`

#### Why this matters

- The design's Decision 5 and the `brain-launch-runtime` spec say projected mailbox skills "SHALL prefer" the gateway facade when available. But the design's open questions also ask whether this preference should wait for "a later cleanup." The answer determines how much of `mail_commands.py`'s prompt construction changes in this change versus a follow-up.

#### Examples

- If preference is immediate: `prepare_mail_prompt()` gains a transport-aware branch that emits gateway-first instructions when a live gateway endpoint is discoverable from env vars.
- If preference is deferred: `prepare_mail_prompt()` only gets Stalwart-specific direct-transport prompts in this change; gateway-first preference lands later.

#### Why this question arises in code

- `prepare_mail_prompt()` at `mail_commands.py:62-115` currently always emits filesystem skill references and filesystem-specific instructions.
- The function takes a `LaunchPlan` but has no access to live gateway endpoint information.

#### Options

**Option A: Implement gateway preference immediately in this change**

- Code impact:
  - `prepare_mail_prompt()` checks for gateway env vars (e.g., `AGENTSYS_GATEWAY_BASE_URL`) in the launch plan env.
  - When present, the prompt directs the agent to use the gateway `/v1/mail/*` surface.
  - When absent, the prompt falls back to transport-specific direct behavior.
- User-facing difference:
  - Agents with a live gateway will automatically use the shared mail facade for `check/send/reply`.
- Pros:
  - Completes the gateway abstraction in one change.
  - Matches what the specs say.
- Cons:
  - Increases scope of `mail_commands.py` changes.
  - Gateway preference depends on gateway mailbox routes being implemented first.

**Option B: Defer gateway preference to a follow-up change**

- Code impact:
  - This change adds Stalwart-specific direct-transport prompts only.
  - Gateway preference is a separate change.
- User-facing difference:
  - Agents always use direct transport paths in this change.
- Pros:
  - Keeps this change focused on the gateway-side adapter and routes.
  - Reduces risk of cross-cutting prompt regressions.
- Cons:
  - Specs promise gateway preference but this change doesn't deliver it.
  - Two-step deployment required for the full abstraction.

> **PROPOSED: Option A — Implement gateway preference immediately in this change.**
> Rationale: The `brain-launch-runtime` spec and `agent-mailbox-system-skills` spec both use normative "SHALL prefer" language for gateway-first mailbox interaction. Deferring this to a follow-up change would leave the specs technically unsatisfied and create a state where the gateway routes exist but are never used by the runtime mail commands. The gateway endpoint information is already available through the existing gateway env binding contract used by the runtime, so the prompt construction change is bounded.
>
> **DECISION: Accept Option A.**
> Rationale: Accept Option A. Gateway preference remains in scope for this change because the current artifacts already use normative "SHALL prefer" language for runtime prompts and projected skills. Direct transport behavior remains the fallback when no live gateway facade is available, and the focused verification strategy does not require full launched-agent proof to keep this behavior in scope.

### Q5) How should Stalwart credentials flow from provisioning to runtime mailbox access?

- Decision: `Blocking`

#### Why this matters

- The Stalwart adapter needs authenticated access to both the management API (for provisioning) and JMAP (for mailbox operations). Decision 7 says credentials stay out of the manifest. The runtime needs to provide credentials to launched agents for direct-transport fallback when no gateway is attached. This end-to-end credential flow needs to be concrete enough for implementation.

#### Examples

- During `start-session`, the runtime provisions a Stalwart account and generates a password. Where does that password go?
- During `resume-session`, the runtime restores the session. How does it recover the password?
- When the gateway constructs a Stalwart adapter from the manifest, how does it authenticate to JMAP?

#### Why this question arises in code

- `MailboxResolvedConfig.redacted_payload()` (line 32-41) is the persistence contract — it explicitly omits secrets.
- The manifest at `manifest.py` stores `launch_plan.mailbox` as the redacted payload.
- Gateway loads mailbox config from manifest via `_load_notifier_mailbox_config()` at `gateway_service.py:450-476`.

#### Options

**Option A: Runtime writes a session-scoped credentials file; manifest stores a path reference**

- Code impact:
  - During provisioning, the runtime writes `<session_root>/mailbox-credentials.json` with the Stalwart password/token.
  - The manifest stores `credential_ref: "<session_root>/mailbox-credentials.json"` (a path, not the secret).
  - The gateway and runtime read the credentials file at adapter construction time.
  - Env var `AGENTSYS_MAILBOX_EMAIL_CREDENTIAL_FILE` points to the file for agent direct-transport fallback.
- User-facing difference:
  - Credentials are stored in a gitignored session-local file.
- Pros:
  - Clear separation between manifest (secret-free) and credential storage.
  - Gateway can read the file without manifest changes.
  - Consistent with the `api-creds/` pattern in agent definition directories.
- Cons:
  - File-based credential management.
  - Credentials persist on disk.

**Option B: Runtime injects credentials via env vars only, no persisted credential file**

- Code impact:
  - During provisioning, the runtime generates and holds the password in memory.
  - The password is injected into `LaunchPlan.env` as `AGENTSYS_MAILBOX_EMAIL_PASSWORD`.
  - The gateway cannot recover credentials from the manifest alone — it needs the runtime to pass them.
- User-facing difference:
  - Credentials exist only in process memory and env vars.
- Pros:
  - No credential files on disk.
- Cons:
  - Gateway cannot independently construct a Stalwart adapter from manifest + credential lookup.
  - Resume requires re-provisioning or a separate credential recovery mechanism.
  - Contradicts the gateway's manifest-backed adapter construction model.

> **PROPOSED: Option A — Runtime writes a session-scoped credentials file; manifest stores a path reference.**
> Rationale: The gateway's adapter construction model reads from `attach.json -> manifest_path -> manifest.json -> launch_plan.mailbox` (see `gateway_service.py:450-476`). The gateway process runs independently from the runtime and needs to construct a Stalwart adapter without runtime assistance. A session-scoped credentials file lets the gateway resolve `credential_ref` from the manifest and read the file independently. This is consistent with the `api-creds/` pattern already used for tool credentials in agent definition directories. The file should be created with restrictive permissions and excluded from any commit paths via `.gitignore` patterns.
>
> **DECISION: Refine Option A.**
> Rationale: Refine Option A. The runtime should materialize session-scoped credential material and persist only a secret-free `credential_ref` in the manifest. In v1, that `credential_ref` may resolve to a runtime-owned file path under the session root, which keeps the manifest secret-free while still letting gateway-backed and direct fallback paths recover the same credentials.
