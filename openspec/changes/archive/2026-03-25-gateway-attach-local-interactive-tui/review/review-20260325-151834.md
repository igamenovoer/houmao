## Summary

- The problem statement is precise and the core fix is a one-liner in `_attach_gateway_for_controller()` — just add `"local_interactive"` to the allowlist. The adapter layer (`_build_gateway_execution_adapter`, `_LocalHeadlessGatewayAdapter`) already fully supports `local_interactive`.
- Design decisions are well-reasoned; the choice to reuse the existing adapter rather than proliferating new adapter classes is justified by `LocalInteractiveSession`'s inheritance from `HeadlessInteractiveSession` and the shared tmux resume contract.
- Task 1.2 ("rename/clarify local gateway adapter contract") has no concrete target name in spec or tasks, leaving implementors to choose freely. A `_LocalTmuxBackedGatewayAdapter` or a broader docstring clarification are both plausible, but the ambiguity is a real risk.
- A residual `isinstance(backend_session, HeadlessInteractiveSession)` guard in `_LocalHeadlessGatewayAdapter.submit_prompt()` is correct today (because `LocalInteractiveSession` subclasses `HeadlessInteractiveSession`), but will silently become wrong if that class hierarchy ever changes. The task list does not call this out for documentation.
- Test coverage plan (task 3.1) is scoped correctly but the tasks stop short of prescribing which fixture helpers to reuse, leaving risk that new tests duplicate rather than extend the existing `_FakeGatewayHeadlessSession`/`_seed_headless_gateway_root` infrastructure.

## Findings

### Must Fix / Blocking

1. **Error message in `_attach_gateway_for_controller()` hardcodes the current (incomplete) backend set as a quoted string.**
   The detail string at `src/houmao/agents/realm_controller/runtime.py:2229` reads:
   ```python
   detail = (
       "Gateway attach is only implemented for runtime-owned tmux-backed backends "
       "{'cao_rest', 'houmao_server_rest', 'claude_headless', "
       "'codex_headless', 'gemini_headless'} in v1, got "
       f"backend={controller.launch_plan.backend!r}."
   )
   ```
   After the allowlist is extended with `"local_interactive"`, this hardcoded set will be stale and actively misleading to operators debugging attach failures on other unsupported backends. Task 1.1 does not explicitly mention updating the error message.

> **DECISION: Accept.**
> Rationale: Once `local_interactive` is admitted, the quoted backend set becomes stale and operator-facing failure text would be inaccurate. The change explicitly keeps unsupported-backend failures visible, so the implementation and any later artifact revision should update this message alongside the admission gate.

### Should Fix / Important

2. **`_LocalHeadlessGatewayAdapter.submit_prompt()` silently relies on `LocalInteractiveSession` subclassing `HeadlessInteractiveSession`.**
   The guard at `gateway_service.py:370` is:
   ```python
   if not isinstance(backend_session, HeadlessInteractiveSession):
       raise GatewayError("Resumed headless controller is missing a headless backend.")
   ```
   `LocalInteractiveSession` (line 36 of `backends/local_interactive.py`) does subclass `HeadlessInteractiveSession`, so this works today. But this coupling is invisible from the adapter's point of view: if the class hierarchy changes, `local_interactive` sessions silently start failing the isinstance check. A short comment or a wider union check (`HeadlessInteractiveSession | LocalInteractiveSession`) near the guard would make the invariant explicit.

> **DECISION: Refine.**
> Rationale: The coupling is real, but the artifact-level requirement is that the local adapter stop reading as headless-only while continuing to support `local_interactive`; we do not need to lock the implementation to one exact union type or comment pattern in the artifacts. Task 1.2 can stay outcome-focused as long as the implementation makes the invariant explicit in a local, readable way.

3. **`_require_live_tmux_session()` error message calls all sessions "Headless tmux session".**
   After task 1.2 broadens the adapter, callers attaching a `local_interactive` gateway will see:
   ```
   Headless tmux session `<name>` is unavailable.
   ```
   This is confusing for interactive TUI sessions. The message should be backend-neutral (e.g., "Local tmux session `{name}` is unavailable").

> **DECISION: Accept.**
> Rationale: Supporting `local_interactive` while still reporting "Headless tmux session" would be confusing to operators and inconsistent with the proposal's documentation and wording goals. This should be corrected when the local adapter wording is broadened.

4. **Task 1.2 does not specify the new adapter name, leaving scope ambiguity.**
   The design says "The implementation may rename the current `_LocalHeadlessGatewayAdapter` for clarity" but uses "local tmux-backed adapter" conceptually throughout. Neither the spec nor the tasks prescribe a concrete new symbol name. Without a shared target, reviewers of the implementation cannot tell if the rename was intentional or incomplete.

> **DECISION: Reject.**
> Rationale: The design deliberately makes renaming optional; the required outcome is semantic clarity for the local tmux-backed adapter family, not one mandated class name. For this change, implementation review should focus on whether the adapter contract is explicit and backend-neutral, not on whether it picked a predetermined symbol.

### Nice To Have

5. **`runtime_resumable` flag at `gateway_service.py:695` uses an `isinstance` check against the current class name.**
   ```python
   runtime_resumable=isinstance(self.m_adapter, _LocalHeadlessGatewayAdapter),
   ```
   If task 1.2 renames the class, this line still compiles, but if the refactor ever splits the adapter into two classes, this check would need updating. A named constant (`_LOCAL_TMUX_BACKED_ADAPTER_TYPES = (...)`) would be more resilient. Low urgency unless the rename results in an adapter split.

> **DECISION: Refine.**
> Rationale: This is sensible implementation hygiene if task 1.2 renames or reshapes the adapter, but it is derivative of the broader adapter-clarification work rather than a separate artifact gap. The current artifacts do not need a new constant or adapter-type bucket just for this line.

6. **The `specs/**/*.md` glob in the `openspec` metadata does not surface the spec files during standard `openspec instructions apply` output**, even though both spec files exist. This is a tooling behavior issue and does not block the change, but it means implementers relying on the JSON output to discover specs may miss them.

> **DECISION: Defer.**
> Rationale: The observed `openspec` glob behavior looks like a tooling issue rather than a problem with this change's proposal, design, specs, or tasks. It does not block coherent implementation of this change and should be handled separately if it recurs.

## Open Questions

### Q1) Should `_attach_gateway_for_controller()` derive its admission set from a shared constant rather than duplicating it?

- Decision: `Deferrable`

#### Why this matters

The current code has two independent places that enumerate the local tmux-backed backend set: the allowlist in `_attach_gateway_for_controller()` (runtime.py) and the routing condition in `_build_gateway_execution_adapter()` (gateway_service.py). Adding `"local_interactive"` to the runtime-side allowlist while it was already in the service-side set is exactly the kind of divergence that caused the current bug. Without a shared constant, the same divergence can re-emerge the next time a new local backend is added.

#### Examples

- A developer adds a new `"local_codex_tui"` backend to `_build_gateway_execution_adapter()` but forgets to also add it to `_attach_gateway_for_controller()` → same bug recurs.

#### Why this question arises in code

- The admission set at `runtime.py:2214` and the adapter-routing set at `gateway_service.py:499` are identical after this fix but live in separate modules with no shared reference.
- A `_GATEWAY_LOCAL_ADAPTER_BACKENDS` constant in `gateway_service.py` could be imported by `runtime.py` to close the gap.

#### Options

**Option A: Keep the two independent sets, document the invariant in a comment**

- Code impact:
  ```python
  # In runtime.py _attach_gateway_for_controller — keep in sync with
  # _build_gateway_execution_adapter() in gateway_service.py.
  if controller.launch_plan.backend not in {
      "cao_rest", "houmao_server_rest",
      "claude_headless", "codex_headless", "gemini_headless", "local_interactive",
  }:
  ```
- User-facing difference: None; error messages unchanged.
- Pros: Zero refactor scope; change remains surgical.
- Cons: Comment-only coupling; future divergence still possible.

**Option B: Extract a shared `_GATEWAY_ADAPTER_SUPPORTED_BACKENDS` constant**

- Code impact:
  ```python
  # In gateway_service.py
  _GATEWAY_ADAPTER_SUPPORTED_BACKENDS: frozenset[BackendKind] = frozenset({
      "cao_rest", "houmao_server_rest",
      "claude_headless", "codex_headless", "gemini_headless", "local_interactive",
  })

  # In runtime.py
  from .gateway_service import _GATEWAY_ADAPTER_SUPPORTED_BACKENDS
  if controller.launch_plan.backend not in _GATEWAY_ADAPTER_SUPPORTED_BACKENDS:
  ```
- User-facing difference: None; error messages would reference the constant.
- Pros: Single source of truth; future additions are automatically reflected in the attach gate.
- Cons: Cross-module import of a private symbol; slightly wider diff for this narrowly scoped fix.

> **PROPOSED: Defer shared-constant extraction to a follow-up; use a comment in this change.**
> Rationale: The change is scoped as a minimal correctness fix (`design.md`, Decision 1). The two-set divergence existed before this change and only one line needs to be added now. Extracting a shared constant is a good refactor but increases diff width beyond the correctness fix, and the comment approach (Option A) makes the coupling visible without widening scope. `gateway_service.py` is also a 1700+ line module — a cross-module import of a private constant should be reviewed as part of a broader `gateway_service` modularization pass, not bundled here.
>
> **DECISION: Defer shared-constant extraction for this change.**
> Rationale: Keep this change scoped to enabling `local_interactive`; any small sync comment or helper added during implementation is fine, but the artifacts do not need to mandate a cross-module refactor. The current change is coherent as a narrow correctness fix plus local-adapter clarification.

### Q2) Should the test fixture for `local_interactive` gateway attach reuse `_seed_headless_gateway_root` or define a parallel `_seed_local_interactive_gateway_root` helper?

- Decision: `Deferrable`

#### Why this matters

Task 3.1 requires regression coverage for gateway attach, status, prompt, and interrupt on `local_interactive` sessions. The existing `_seed_headless_gateway_root` fixture in `test_gateway_support.py:923` is hardcoded to `"claude_headless"` (backend, session_id, tmux_session_name). Reusing it for `local_interactive` tests would require either parameterizing it or copying it.

#### Examples

- `test_attach_gateway_supports_runtime_owned_headless_backend` at line 375 could be paired with a `test_attach_gateway_supports_runtime_owned_local_interactive_backend` using the same structure.

#### Why this question arises in code

- `_seed_headless_gateway_root` hardcodes `backend="claude_headless"` and `session_id="claude-headless-1"` at `test_gateway_support.py:936`. A `local_interactive` variant would need `backend="local_interactive"` and a matching `local_interactive` manifest state (a `local_interactive` manifest entry in `backend_state` rather than the bare `{}` that the headless seed writes).
- The local_interactive manifest state shape is defined in `models.py` and populated by `_resume_local_interactive_state()` at `runtime.py:1207`. Tests need a valid stub of that state to exercise the resume path.

#### Options

**Option A: Parameterize `_seed_headless_gateway_root` to accept a `backend` parameter**

- Code impact:
  ```python
  def _seed_headless_gateway_root(tmp_path, *, backend: BackendKind = "claude_headless", ...):
      manifest_path = default_manifest_path(tmp_path, backend, f"{backend}-1")
      _write(manifest_path, ...)  # write minimal valid state for the backend
      paths = ensure_gateway_capability(GatewayCapabilityPublication(backend=backend, ...))
  ```
- User-facing difference: None; test scaffolding only.
- Pros: Single helper, no duplication.
- Cons: The manifest state content differs for `local_interactive` vs headless; parameterization may not be clean without a per-backend state factory.

**Option B: Add a separate `_seed_local_interactive_gateway_root` helper**

- Code impact: Parallel helper function, ~20 lines, using a minimal but valid `local_interactive` manifest state stub.
- User-facing difference: None; test scaffolding only.
- Pros: Each helper is self-contained and readable; no conditional logic.
- Cons: Some duplication with `_seed_headless_gateway_root`.

> **PROPOSED: Use a separate `_seed_local_interactive_gateway_root` helper (Option B).**
> Rationale: `local_interactive` sessions require a different manifest state shape (the `local_interactive` payload block in `models.py`), and the headless seed writes `"{}\n"` as a placeholder manifest (line 935). Sharing the helper would require conditional backend-state logic that makes both the parameterized version and its callers harder to read. Option B keeps the test intent explicit and matches the existing pattern where `_seed_headless_gateway_root` and `_seed_cao_gateway_root_with_stalwart_mailbox` are separate helpers (lines 923 and 1045+). The marginal duplication is acceptable.
>
> **DECISION: Prefer a separate `local_interactive` seed helper if resume-path coverage needs persisted local-interactive state.**
> Rationale: Keep this as implementation guidance rather than a task-level naming requirement. `_resume_local_interactive_state()` requires a `local_interactive` payload block plus `backend_state.tmux_session_name`, so a dedicated helper is clearer than stretching the existing headless seed into backend-conditional setup.
