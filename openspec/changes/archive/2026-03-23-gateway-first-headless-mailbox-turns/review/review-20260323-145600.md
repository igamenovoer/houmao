## Summary

- **Solid motivation and grounding.** The change clearly traces back to concrete hack-through findings (HT-02, HT-03), and the three-gap analysis (filesystem leakage, notifier not actionable, facade stops short of read-state) is persuasive and well-evidenced from existing code.
- **Clean architectural direction.** The gateway-first principle is consistent with the existing documented product direction visible in `docs/reference/gateway/operations/mailbox-facade.md` ("Direct Versus Gateway Mailbox Paths") and the existing mailbox system skill contract.
- **Delta specs are well-scoped and follow existing patterns.** Each delta spec modifies exactly one spec boundary and keeps scenario coverage focused.
- **Risk area: `GatewayMailboxAdapter` protocol extension requires careful transport parity handling.** The adapter protocol today has `status`, `check`, `send`, `reply`; adding a state-update method touches both filesystem and Stalwart adapters and must be kept narrow.
- **Risk area: notifier prompt restructuring could regress deduplication logic.** The current `_mail_notifier_digest` hashes all unread `message_ref` values; nominating one target changes what the prompt looks like but the digest contract should remain stable.
- **Risk area: projected skill content rewrite is the hardest to verify automatically.** The SKILL.md template change from "filesystem-first" to "gateway-first with fallback" is prompt engineering that depends on model compliance, and the spec only covers it with instructional scenarios.

## Findings

### Must Fix / Blocking

1. **The `GatewayMailboxAdapter` protocol (in `gateway_mailbox.py:40-76`) does not include a `state_update` or `mark_read` method.** The design commits to `POST /v1/mail/state` dispatching through the existing adapter boundary, but the adapter `Protocol` class will need a new method. The delta spec for `agent-gateway` says "the gateway SHALL resolve that request through the same manifest-backed mailbox adapter boundary," but the tasks (1.1 and 1.2) don't explicitly call out adding the method to the `Protocol` class versus just adding transport implementations. Ensure Task 1.2 includes the protocol-level method signature, not just the transport adapter implementations, to avoid a structural gap during implementation.

> **DECISION: Accept.**
> Rationale: The reviewer is right that the change needs an explicit protocol-level adapter method, not just per-transport implementations. The revision pass will tighten Task 1.2 and the design/spec language around one narrow single-message read-state update method on `GatewayMailboxAdapter`.

2. **The notifier prompt restructuring (Task 2.2) needs to preserve the existing deduplication digest contract.** The current `_mail_notifier_digest` in `gateway_service.py:1027-1031` hashes all unread `message_ref` values to detect whether the unread set has changed since the last notification. The design says the notifier will "nominate one actionable unread message" — but the digest must still cover the full unread set, not just the nominated target, or stale nominations will repeat. The design and spec don't explicitly address whether deduplication semantics change. This should be clarified before implementation, because a regression here causes repeated wake-ups for the same unread message.

> **DECISION: Accept.**
> Rationale: Prompt nomination and deduplication are separate concerns. The notifier may nominate one actionable target, but the dedup digest must continue to represent the full unread set so prompt wording changes do not cause repeated reminders for unchanged unread state.

This helps future readers understand the spec evolution.

> **DECISION: Accept.**
> Rationale: The delta should say explicitly that this change adds a fourth shared routine operation category, read-state update, alongside `check`, `send`, and `reply`, so the protocol evolution is discoverable instead of implied.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the `POST /v1/mail/state` request model accept `message_ref` as a required field or as one of multiple possible targeting modes?

- Decision: `Deferrable`

#### Why this matters

The design commits to opaque `message_ref` targeting, but the existing gateway models show that `GatewayMailReplyRequestV1` accepts `message_ref` as a required field directly. If `POST /v1/mail/state` uses the same single-target shape, the model is simple and consistent. But if the design later wants to support batch updates (e.g., mark multiple messages read after a multi-message turn), a list-based targeting shape would be cheaper to add now.

#### Examples

- Single target: `{"schema_version": 1, "message_ref": "filesystem:msg-...", "read": true}`
- List target: `{"schema_version": 1, "updates": [{"message_ref": "filesystem:msg-...", "read": true}]}`

#### Why this question arises in code

- `gateway_models.py:715` shows `GatewayMailReplyRequestV1` uses a single `message_ref: str` field.
- The current notifier nominates one target, but the design non-goal says "Add broad mailbox flag editing for every possible mailbox state in one step" — which rejects breadth of state, not breadth of targets.
- If the route only accepts one target, marking multiple processed messages read requires multiple HTTP calls per turn.

#### Options

**Option A: Single `message_ref` field, consistent with `reply` route.**

- Code impact:
  - Simple Pydantic model: `class GatewayMailStateRequestV1(_StrictGatewayModel): message_ref: str; read: bool`
  - One adapter call per request.
- User-facing difference:
  - Agent must call the endpoint once per message it wants to mark read.
- Pros:
  - Consistent with the existing `reply` model shape.
  - Simple adapter interface: one method that takes a single ref.
  - Easier to test and audit.
- Cons:
  - Multiple HTTP calls if an agent processes several messages in one turn.

**Option B: List-based `updates` field for batch support.**

- Code impact:
  - Model carries `updates: list[GatewayMailStateUpdateV1]`, where each entry has `message_ref` and `read`.
  - Adapter method accepts a sequence.
  - Atomicity semantics must be decided (all-or-nothing vs partial success).
- User-facing difference:
  - Agent can mark multiple messages read in one call.
- Pros:
  - Fewer HTTP round-trips for multi-message turns.
  - Forward-compatible with batch patterns.
- Cons:
  - Extra complexity for the current single-target use case.
  - Partial-failure semantics add design surface.
  - Inconsistent with the existing single-ref `reply` shape.

> **PROPOSED: Use single `message_ref` field (Option A).**
> Rationale: The current bounded-turn contract processes one message at a time, and the existing `GatewayMailReplyRequestV1` in `gateway_models.py:715` already establishes the single-`message_ref` pattern for adapter dispatch. Adding batch support introduces atomicity questions that are outside the stated scope. If batch marking becomes necessary later, a separate `POST /v1/mail/state/batch` route can be added without breaking the single-target contract.
>
> **DECISION: Choose Option A.**
> Rationale: Accept the proposed single-target shape. One required `message_ref` plus explicit `read` state matches the bounded one-message turn contract, stays consistent with `POST /v1/mail/reply`, and avoids introducing batch atomicity semantics that are outside this change.

### Q2) Should the notifier wake-up prompt include `thread_ref` for the nominated target, and should it also include the remaining unread count?

- Decision: `Blocking`

#### Why this matters

The design says the wake-up prompt "will include the target `message_ref`, optional `thread_ref`, sender and subject context, whether additional unread messages remain queued behind the nominated target." The `_UnreadMailboxMessage` dataclass already has `thread_ref` as `str | None`. Including `thread_ref` in the prompt helps the agent maintain threading context when replying through the gateway facade, but it also adds information that could encourage the agent to do thread-level reasoning instead of the bounded single-action turn the design intends.

#### Examples

- Prompt with thread: "Action: reply to message_ref=`filesystem:msg-...` (thread_ref=`filesystem:msg-...`), from `AGENTSYS-responder@agents.localhost`, subject: 'Ping round 2'. 1 more unread message remains."
- Prompt without thread: "Action: reply to message_ref=`filesystem:msg-...`, from `AGENTSYS-responder@agents.localhost`, subject: 'Ping round 2'. 1 more unread message remains."

#### Why this question arises in code

- `_UnreadMailboxMessage` in `gateway_service.py:118-125` already carries `thread_ref: str | None`.
- The reply route `POST /v1/mail/reply` targets by `message_ref`, not `thread_ref`, so `thread_ref` in the prompt is context, not a targeting parameter.
- The design explicitly lists `optional thread_ref` in the prompt contents, so including it seems intentional, but the reason is unstated.

#### Options

**Option A: Include `thread_ref` and remaining unread count, as the design specifies.**

- Code impact:
  - Prompt construction uses all `_UnreadMailboxMessage` fields plus a count.
  - No structural change to deduplication (digest still covers all unread refs).
- User-facing difference:
  - Agent sees richer context per wake-up, may use thread_ref for conversation awareness.
- Pros:
  - Follows the design as written.
  - Gives the agent enough context to decide whether it is continuing a thread or starting a new one.
- Cons:
  - Slightly more information in the prompt; risk of thread-level reasoning consuming turn budget.

**Option B: Omit `thread_ref`, include only `message_ref`, sender, subject, and remaining count.**

- Code impact:
  - Simpler prompt template.
- User-facing difference:
  - Agent knows which message to process but has less threading context.
- Pros:
  - Minimizes prompt surface.
  - Keeps the turn focused on one action.
- Cons:
  - Deviates from the design document.
  - Agent may lack context to produce a correct reply (especially for thread continuity).

> **PROPOSED: Include `thread_ref` and remaining unread count (Option A), as the design specifies.**
> Rationale: The `reply` action through the gateway facade (`POST /v1/mail/reply`) already targets by `message_ref` for ancestry resolution, but `thread_ref` gives the agent enough conversational context to compose a coherent reply body. The design explicitly calls for it. The remaining-unread count lets the agent (and later audit) distinguish "last unread" from "more work queued." Risk of thread-level over-reasoning is mitigated by the skill contract change (Task 2.1) which frames the turn as bounded to one action.
>
> **DECISION: Choose Option A with the stated refinement.**
> Rationale: Accept the proposed content with one refinement: the wake-up prompt should include optional `thread_ref`, sender, subject, and the remaining unread count, but still frame the work as one actionable target identified by `message_ref`. That gives enough context for coherent replies without turning the prompt back into a multi-message digest.

### Q4) Should the `POST /v1/mail/state` route return the updated message state, or just acknowledge success?

- Decision: `Deferrable`

#### Why this matters

The existing `POST /v1/mail/send` and `POST /v1/mail/reply` routes return a `GatewayMailActionResponseV1` wrapping the full `GatewayMailboxMessageV1` for the delivered message. A state-update route could follow the same pattern (return the updated message with new read state) or use a minimal acknowledgment shape. This affects model size and what information the agent gets back.

#### Examples

- Full message return: `{"schema_version": 1, "message": {"message_ref": "...", "unread": false, ...}}`
- Minimal ack: `{"schema_version": 1, "ok": true, "message_ref": "...", "read": true}`

#### Why this question arises in code

- `GatewayMailActionResponseV1` in `gateway_models.py:744` wraps a `GatewayMailboxMessageV1`.
- For a read-state update, the caller already knows the `message_ref` and the target state; returning the full message metadata is mostly confirmation, not new information.
- The filesystem adapter can easily return the full message, but the Stalwart adapter's JMAP `Email/set` response shape may not include the full message body.

#### Options

**Option A: Return the full `GatewayMailboxMessageV1` wrapped in an action response, consistent with send/reply.**

- Code impact:
  - Reuses `GatewayMailActionResponseV1` or a similar wrapper.
  - Both adapters must be able to fetch the full message after the state update.
- User-facing difference:
  - Agent gets full message confirmation.
- Pros:
  - Consistent with existing action response pattern.
  - Agent can verify the message content matches expectations.
- Cons:
  - Stalwart adapter may need an extra fetch after the state update.
  - Heavier response for a simple flag change.

**Option B: Return a minimal acknowledgment with just `message_ref` and the resulting state.**

- Code impact:
  - New response model, e.g., `GatewayMailStateResponseV1` with `message_ref`, `read`, and `ok` fields.
  - Simpler adapter implementation.
- User-facing difference:
  - Agent gets confirmation that the state change succeeded but no full message body.
- Pros:
  - Lighter response, appropriate for the operation scope.
  - Avoids the extra fetch in Stalwart adapter.
- Cons:
  - Introduces a new response pattern distinct from send/reply.
  - Agent cannot verify message content in the same call.

> **PROPOSED: Return a minimal acknowledgment (Option B).**
> Rationale: A read-state update is a flag mutation, not a content-producing operation. The caller already has the `message_ref` and target state. The existing `send`/`reply` routes return full messages because those are content-creating operations where the caller needs the new `message_ref` and thread state. For state updates, a minimal ack avoids an extra JMAP fetch in the Stalwart adapter and keeps the route lightweight. A new `GatewayMailStateResponseV1` with `schema_version`, `message_ref`, and `read` fields is small and self-documenting.
>
> **DECISION: Choose Option B.**
> Rationale: Accept the minimal-ack direction. A state update mutates mailbox state but does not create new message content, so the route should return a narrow success contract centered on the target `message_ref` and resulting `read` state rather than forcing both transports to fetch and return a full `GatewayMailboxMessageV1`.

### Q5) How should the notifier select which unread message to nominate when multiple are unread?

- Decision: `Deferrable`

#### Why this matters

The design says "nominate one actionable unread message" but does not specify the selection policy. For the ping-pong demo, this is likely oldest-first (FIFO), but in general use cases the policy could matter (e.g., most recent, or highest priority).

#### Examples

- Three unread messages from different senders, oldest first: nominate the oldest.
- Two unread messages, one is a reply to a previously processed thread: nominate the in-thread reply for continuity.

#### Why this question arises in code

- `_run_notifier_cycle` in `gateway_service.py:872` calls `adapter.check(unread_only=True, limit=None, since=None)` and gets all unread messages.
- The current prompt lists up to 10 unread messages; the new design nominates one.
- Selection policy is unstated in the design, the delta spec, and the tasks.

#### Options

**Option A: Oldest-first (FIFO by `created_at_utc`).**

- Code impact:
  - Sort unread messages by `created_at_utc` ascending, pick first.
  - Trivial implementation.
- User-facing difference:
  - Agent always processes the oldest unread message first.
- Pros:
  - Simple, predictable, deterministic.
  - Natural fit for the ping-pong demo's round-by-round progression.
- Cons:
  - May not be optimal for all future use cases.

**Option B: Most recent first.**

- Code impact:
  - Sort descending, pick first.
- User-facing difference:
  - Agent processes the newest message first, which may be more relevant but breaks round ordering.
- Pros:
  - Common in email UX.
- Cons:
  - Breaks ping-pong round ordering.
  - Counterintuitive for sequential conversation processing.

> **PROPOSED: Oldest-first / FIFO (Option A).**
> Rationale: The current `adapter.check` for filesystem returns messages ordered by creation time via the SQLite query in `gateway_mailbox.py`. The ping-pong demo requires round-sequential processing, and the bounded-turn contract assumes one actionable target is the next in sequence. FIFO is the simplest deterministic policy, and it can be overridden by a future notifier configuration parameter if needed without changing the base contract.
>
> **DECISION: Choose Option A with an explicit tie-breaker.**
> Rationale: Choose oldest-first, but not for the reason stated in the proposal. The current filesystem adapter returns newest-first order, so notifier nomination must not rely on adapter iteration order. We will make nomination policy explicit: pick the oldest unread message by `created_at_utc` using a stable tie-breaker so bounded turns drain backlog deterministically across transports.
