## Summary

- The overall direction is sound: collapsing recipe + blueprint into path-derived presets, renaming config-profile → setup and credential-profile → auth, and removing duplicated identity fields genuinely reduces conceptual surface.
- Specs are well-structured with concrete scenarios. The component-agent-construction delta spec is thorough and covers the independent setup×auth axis clearly.
- The `--agent-name` CLI option is silently dropped from the houmao-mgr-agents-launch spec's launch-options list without an explicit removal/migration entry, creating ambiguity about whether managed-agent identity at launch time is still supported.
- Three design open questions remain unresolved with no corresponding task decision points, risking implementation drift.
- The preset `launch` field conflates two existing distinct models (`LaunchOverrides` and `OperatorPromptMode`) without specifying the new schema shape.
- No brain manifest schema version bump is specified despite breaking field changes.

## Design Alignment With Existing System

- **Existing pattern: path-derived identity in native resolver.** The current `_resolve_recipe_path()` at `native_launch_resolver.py:101-124` already derives tool from provider and searches `brains/brain-recipes/<tool>/` by selector name. The proposed preset resolution is a natural extension of this pattern — mapping `<role>` + `<tool>` (from provider) → `roles/<role>/presets/<tool>/default.yaml`. The design aligns well.

- **Existing pattern: secret-free / local-only split.** The repo enforces this split via `.gitignore` and `pyproject.toml` exclusions. The proposed `agents/tools/<tool>/setups/` (checked-in) vs `agents/tools/<tool>/auth/` (local-only) preserves this split and makes it structurally clearer by colocating both under the same tool namespace. Aligns well.

- **Existing pattern: frozen dataclasses for value objects.** `BrainRecipe`, `ToolAdapter`, `BuildRequest`, `ResolvedNativeLaunchTarget` are all `@dataclass(frozen=True)`. The new preset model should follow the same pattern. Design is consistent.

- **Existing pattern: Pydantic for boundary validation.** `_BlueprintPayloadV1` in `loaders.py:53-68` uses Pydantic BaseModel with strict validation for YAML parsing. The new preset loader should use the same pattern (e.g., `_PresetPayloadV1`). This isn't stated explicitly but is implied by task 1.2's "define and implement the minimal preset schema."

- **Potential misalignment: adapter relocation depth.** Current adapters live at depth 1 under `brains/tool-adapters/claude.yaml`. New layout puts them at `tools/<tool>/adapter.yaml`, which is the same depth but under a different parent. However, the `_load_tool_adapter()` function at `brain_builder.py:173-266` takes a direct file path. The relocation is transparent to the loader if callers pass the correct new path, but the path construction in `build_brain_home()` at line ~550 uses `agent_def_dir / "brains" / "tool-adapters" / f"{tool}.yaml"` — this hardcoded path will need updating in the build pipeline.

- **Potential misalignment: `loaders.py` manifest version check.** `load_brain_manifest()` at `loaders.py:99-104` hard-checks `schema_version == 2`. A manifest field change without a version bump would silently break manifest loading. The change should bump to v3 and update the check.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should `--agent-name` remain a launch option (now optional) or be removed entirely from the preset-backed launch surface?

- Decision: `Blocking`

#### Why this matters

- The baseline spec treats `--agent-name` as required. The design says identity is launch-time-only, but "may pass" is ambiguous about whether the CLI still exposes it. Existing code at `core.py:114-224` treats it as required. Removing it changes the CLI contract; keeping it optional preserves backward compat for scripts.

#### Examples

- An operator currently runs: `houmao-mgr agents launch --agents gpu-kernel-coder --provider claude_code --agent-name my-agent`. Under the new model, if `--agent-name` is removed, the runtime must auto-derive a name. If kept as optional, the current command still works.
- A multi-instance scenario: launching the same preset twice requires distinct names. Without `--agent-name`, the runtime's auto-derivation must handle uniqueness.

#### Why this question arises in code

- `src/houmao/srv_ctrl/commands/agents/core.py:114-224` defines `agent_name` as a required CLI parameter.
- `src/houmao/agents/realm_controller/agent_identity.py` provides `derive_auto_agent_name_base(tool, role_name)` as a fallback.
- `src/houmao/agents/realm_controller/runtime.py:987-1008` calls `_resolve_start_session_identity()` which handles both explicit and fallback identity.

#### Options

**Option A: Keep `--agent-name` as optional with auto-derivation fallback**

- Code impact:
  - Change `agent_name` from required to optional in `core.py`
  - Fallback to `derive_auto_agent_name_base(tool, role_name)` when omitted
  - Add uniqueness suffix (e.g., short UUID) to prevent collisions on multi-launch
- User-facing difference:
  - Existing scripts keep working
  - Simple launches become shorter: `houmao-mgr agents launch --agents gpu-kernel-coder --provider claude_code`
- Pros:
  - Consistent with design intent ("operators may pass agent_name")
  - Non-breaking for existing automation
- Cons:
  - Auto-derived names may be less memorable

**Option B: Remove `--agent-name` entirely, always auto-derive**

- Code impact:
  - Remove `agent_name` parameter from `launch_agents_command()`
  - Always use `derive_auto_agent_name_base()`
  - Remove `--agent-name` from CLI help and docs
- User-facing difference:
  - Existing scripts using `--agent-name` break
  - All names are system-generated
- Pros:
  - Simplest model, no optional parameter to document
- Cons:
  - Breaking for existing users
  - Less ergonomic for operators who want meaningful names

> **PROPOSED: Keep `--agent-name` as optional with auto-derivation fallback (Option A).**
> Rationale: The existing `derive_auto_agent_name_base()` at `agent_identity.py` already provides the fallback mechanism, and `_resolve_start_session_identity()` at `runtime.py:987-1008` already handles the None case. Making it optional rather than removing it is a one-line change in the CLI definition and avoids breaking existing operator scripts while aligning with the design's "may pass" language. The design's goal of removing preset-owned identity is achieved either way since the default no longer comes from a recipe field.
>

> Rationale: _To be filled by developer._

### Q2) What is the concrete schema shape of the preset `launch` field?

- Decision: `Blocking`

#### Why this matters

- Task 1.2 must define this schema. The current codebase has two separate concepts: `LaunchOverrides` (`args` + `tool_params`) and `OperatorPromptMode`. The spec references `launch.prompt_mode` but doesn't define whether `launch` also carries `args`/`tool_params` or only `prompt_mode`.

#### Examples

- A preset requesting unattended mode with custom tool params:
  ```yaml
  launch:
    prompt_mode: unattended
    tool_params:
      include_partial_messages: true
  ```
  vs. only:
  ```yaml
  launch:
    prompt_mode: unattended
  ```

#### Why this question arises in code

- `src/houmao/agents/launch_overrides/models.py:84-136` defines `LaunchOverrides` with `args` and `tool_params`.
- `src/houmao/agents/brain_builder.py:74-84` has `operator_prompt_mode` as a separate field on `BrainRecipe`.
- `src/houmao/agents/realm_controller/launch_plan.py:112-126` merges both into the final launch plan.

#### Options

**Option A: Unified `launch` with `prompt_mode` + optional `overrides`**

- Code impact:
  ```python
  class PresetLaunchSettings:
      prompt_mode: OperatorPromptMode | None = None
      overrides: LaunchOverrides | None = None
  ```
- User-facing difference:
  - One `launch:` block in preset YAML covers both policy and overrides
- Pros:
  - Single location for all launch-affecting preset metadata
  - Clear mapping from current `BrainRecipe` fields
- Cons:
  - Slightly larger schema

**Option B: `launch` contains only `prompt_mode`; tool_params/args stay in `extra`**

- Code impact:
  - `launch` is a simple object with `prompt_mode` only
  - `extra.launch_overrides` carries args/tool_params
- User-facing difference:
  - Preset YAML separates policy from tool-specific overrides
- Pros:
  - Minimal core schema
- Cons:
  - Splits related launch metadata across two locations

> **PROPOSED: Unified `launch` field with `prompt_mode` + optional `overrides` (Option A).**
> Rationale: The current `BrainRecipe` already groups `launch_overrides` and `operator_prompt_mode` as sibling fields that both affect the same build→launch pipeline path at `launch_plan.py:147-170`. Keeping them together in the preset mirrors the existing data flow. Splitting `overrides` into `extra` would require the build pipeline to look in two places for launch-affecting data, contradicting the simplification goal. The existing `LaunchOverrides` model is already minimal (just `args` + `tool_params`), so the combined `launch` block stays small.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._

### Q3) Should the `--agents` selector support both shorthand role names and explicit preset file paths?

- Decision: `Deferrable`

#### Why this matters

- The design lists this as an open question. Task 2.3 says "explicit path-like preset selectors" but doesn't define the boundary. The current `_resolve_recipe_path()` at `native_launch_resolver.py:101-124` already distinguishes path-like selectors (containing `/`, `\`, `.`, `~`, or ending `.yaml`) from name-based selectors.

#### Examples

- Shorthand: `--agents gpu-kernel-coder` → resolves `roles/gpu-kernel-coder/presets/claude/default.yaml`
- Explicit: `--agents roles/gpu-kernel-coder/presets/claude/research.yaml` → uses that file directly
- Hybrid: `--agents gpu-kernel-coder/research` → could mean role=gpu-kernel-coder, setup=research (if supported)

#### Why this question arises in code

- `src/houmao/agents/native_launch_resolver.py:101-124` already has the path-like vs name-based distinction
- The new preset layout `roles/<role>/presets/<tool>/<setup>.yaml` introduces a third potential pattern: `<role>/<setup>` when tool is already known from `--provider`

#### Options

**Option A: Support shorthand role name + explicit file path only (no hybrid)**

- Code impact:
  - Simple: name-based → `roles/<role>/presets/<tool>/default.yaml`; path-like → use as-is
- User-facing difference:
  - Non-default setups require full path: `--agents roles/gpu-kernel-coder/presets/claude/research.yaml`
- Pros:
  - No ambiguity in selector parsing
  - Follows current implementation pattern
- Cons:
  - Verbose for non-default setups

**Option B: Support shorthand role name, `<role>/<setup>` hybrid, and explicit file path**

- Code impact:
  - Add pattern matching for `<role>/<setup>` when provider determines tool
  - More complex selector grammar
- User-facing difference:
  - Shorter non-default: `--agents gpu-kernel-coder/research --provider claude_code`
- Pros:
  - Ergonomic for power users
- Cons:
  - More complex, potential ambiguity (is `foo/bar` a role/setup or a relative path?)

> **PROPOSED: Support shorthand role name + explicit file path only, no hybrid (Option A).**
> Rationale: The current resolver at `native_launch_resolver.py:101-124` cleanly separates name-based and path-like selectors. Adding a third hybrid grammar introduces ambiguity (is `a/b` a role/setup pair or a relative path?) that would require heuristics to resolve. Since non-default setups are the minority case (the fixture tree shows only `yunwu-openai` as a non-default setup for codex), the verbosity cost is low. The hybrid can be added later if demand emerges without breaking the simpler grammar.
>

> Rationale: _To be filled by developer._

### Q4) Should unknown top-level preset fields be rejected immediately or tolerated with a deprecation warning during migration?

- Decision: `Deferrable`

#### Why this matters

- This is the second design open question. Once `extra` exists as the extension surface, strict validation prevents schema drift. But during the migration window, strict rejection could make iterative testing harder.

#### Examples

- A preset with a typo: `skill: [...]` instead of `skills: [...]` — strict mode catches this immediately.
- A preset with a legacy field during migration: `config_profile: default` — strict mode blocks loading.

#### Why this question arises in code

- The current `_BlueprintPayloadV1` at `loaders.py:53-68` uses Pydantic's strict model validation, which already rejects unknown fields.
- Pydantic v2 supports `model_config = ConfigDict(extra='forbid')` for strict validation.

#### Options

**Option A: Strict rejection from day one**

- Code impact:
  - Pydantic model with `extra='forbid'`
- User-facing difference:
  - Any unknown field causes an immediate load error
- Pros:
  - Clean schema enforcement, catches typos
  - Consistent with existing blueprint validation pattern
- Cons:
  - Slightly less forgiving during migration

**Option B: Warn-and-ignore during migration, strict after**

- Code impact:
  - Pydantic model with `extra='ignore'` + custom warning emission
  - Second commit switches to `extra='forbid'`
- User-facing difference:
  - Unknown fields produce warnings but don't block
- Pros:
  - Smoother migration
- Cons:
  - Extra code, temporary state

> **PROPOSED: Strict rejection from day one (Option A).**
> Rationale: The existing `_BlueprintPayloadV1` at `loaders.py:53-68` already uses strict Pydantic validation. The repo is explicitly breaking-change friendly and the migration plan calls for a one-shot cut without parallel legacy/new support. Strict validation from the start catches errors early and avoids the temporary dual-mode code that contradicts the design's stated preference for a clean break. Fixtures and tests will be migrated in the same commit set (tasks 4.1-4.2).
>

### Q5) Should existing blueprint gateway defaults migrate to `extra.gateway` or become launch-time-only?

- Decision: `Deferrable`

#### Why this matters

- This is the third design open question. The current `BlueprintBinding` at `loaders.py:18-35` carries `gateway: BlueprintGatewayDefaults | None` with `host` and `port`. Task 3.3 mentions migrating blueprint metadata but doesn't resolve whether gateway defaults belong in presets.

#### Examples

- Current blueprint: `gateway: { host: "0.0.0.0", port: 8090 }` — used by gateway-aware demo scenarios.
- Under the new model, this either moves to `extra.gateway` in the preset or requires explicit `--gateway-host`/`--gateway-port` at launch time.

#### Why this question arises in code

- `src/houmao/agents/realm_controller/loaders.py:18-35` defines `BlueprintGatewayDefaults`
- Gateway attach is a runtime operation in `src/houmao/agents/realm_controller/gateway_service.py`
- Demo scripts may reference blueprint gateway defaults

#### Options

**Option A: Migrate to `extra.gateway` in preset**

- Code impact:
  - Gateway defaults preserved as `extra: { gateway: { host: "...", port: ... } }` in preset YAML
  - Launch pipeline reads `extra.gateway` when gateway attach is requested
- User-facing difference:
  - Presets continue to carry sensible gateway defaults
- Pros:
  - Non-breaking for demo scripts that rely on defaults
  - Gateway config stays declarative
- Cons:
  - Adds semantic meaning to `extra`, slightly contradicting "non-core extension"

**Option B: Gateway defaults become launch-time-only**

- Code impact:
  - Remove gateway from presets entirely
  - Require `--gateway-host`/`--gateway-port` at launch or in gateway attach command
- User-facing difference:
  - Demo scripts must pass explicit gateway options
- Pros:
  - Cleaner preset schema
  - Gateway is a runtime concern, not a build concern
- Cons:
  - More verbose demo commands

> **PROPOSED: Migrate to `extra.gateway` in preset (Option A).**
> Rationale: Gateway attach defaults are currently used by several demo blueprints (e.g., `gateway-mail-wakeup-demo-claude.yaml` under `tests/fixtures/agents/blueprints/`). Moving them to `extra.gateway` preserves the declarative demo experience without promoting gateway to a core schema field. The `extra` field is explicitly designed for "secret-free subsystem-specific metadata" per the spec, and gateway defaults fit that description. The launch pipeline at `launch_plan.py` already reads metadata from the manifest, so reading `extra.gateway` is a minor extension.
>
