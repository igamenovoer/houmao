## Summary

- **Solid**: The four-artifact set (proposal, design, spec, tasks) is internally consistent: the 2N-message / (2N+1)-turn conversation contract, `outputs/` ownership model, and stepwise command surface are coherently described across all documents.
- **Solid**: The design correctly identifies the three adjacent packs (mailbox-roundtrip, gateway-wakeup, dual-shadow-watch) and justifies a new standalone pack rather than extending them.
- **Solid**: Environment variable redirection via `AGENTSYS_GLOBAL_RUNTIME_DIR` / `AGENTSYS_GLOBAL_REGISTRY_DIR` / `AGENTSYS_GLOBAL_MAILBOX_DIR` / `AGENTSYS_LOCAL_JOBS_DIR` is well-aligned with the existing `owned_paths.py` resolution chain.
- **Risk**: The design omits several critical runtime details — agent role prompts, brain recipe selection, kickoff prompt content, and how agents autonomously discover and handle mail threads — without which the 5-round conversation cannot actually progress.
- **Risk**: State schemas (`demo_state.json`, `report.json`, `inspect.json`, `conversation_events.jsonl`) are referenced but not defined, creating ambiguity for implementers and test authors.
- **Risk**: The task list has no explicit task for the `run_demo.sh` shell wrapper, and several implicit inter-task dependencies are not called out.

## Findings

### Must Fix / Blocking

1. **Design does not specify which role prompts or system prompt packages are used, nor their mail-handling instructions.** The entire conversation flow depends on agents autonomously reading unread mail, composing context-appropriate replies, respecting the round limit, and sending the final stop decision. The gateway mail-notifier only surfaces "you have unread mail" digests (see `gateway_service.py` lines 714–763); the agent must know *how* to act on that. Without a defined role (or at minimum a role contract) in the design, implementers have no specification for the most critical behavioral surface of the demo. Existing packs that use roles reference them explicitly — e.g., `cao_interactive_demo` uses `DEFAULT_ROLE_NAME="gpu-kernel-coder"` (`src/houmao/demo/cao_interactive_demo/commands.py`). The design should specify or constrain the role name, the required mail-handling skill(s), and the round-limit instruction contract.

> **DECISION: Accept.**
> Rationale: The current artifacts name the headless Claude/Codex pair and the conversation contract, but they do not yet define the role-layer behavior that turns a gateway reminder into mailbox work plus a round-limit-aware stop decision. The runtime-owned mailbox skill exists, but the design still needs an explicit demo role contract because the tracked `mailbox-demo` role only narrows scope; it does not define this ping-pong workflow.

2. **Design does not specify brain recipe selection or the headless launch bootstrap path.** The `POST /houmao/agents/headless/launches` route requires `brain_manifest_path`, `agent_def_dir`, `role_name`, and `working_directory` (`src/houmao/server/service.py` lines 737–800). The design mentions "launch one managed headless Claude Code managed agent and one managed headless Codex managed agent" but does not describe how brain recipes are loaded, which `agent_def_dir` is used (default `AGENTSYS_AGENT_DEF_DIR`? pack-local? fixture-based?), or whether the demo performs a `build-brain` phase before calling the headless launch API. Existing packs that launch agents either perform explicit brain building (e.g., `cao_interactive_demo/brain_recipes.py`) or reference pre-built manifest paths. The design should specify this bootstrap chain.

> **DECISION: Accept.**
> Rationale: `launch_headless_agent()` requires a real `working_directory`, `agent_def_dir`, `brain_manifest_path`, and `role_name`, so the change needs an explicit bootstrap chain rather than only "launch managed headless agents." The artifact revision should spell out the tracked fixture source, brain-build step, and manifest handoff before the headless launch API call.

3. **The kickoff prompt content and thread-key injection mechanism are unspecified.** The design states the pack "create[s] one run-specific thread key" and submits "one kickoff prompt to the initiator," but does not define: (a) how the thread key is generated, (b) how it is embedded in the kickoff prompt or communicated to both agents, (c) what the kickoff prompt text says, or (d) how the responder learns the thread key so it can reply in the same thread. This is the causal root of the entire conversation — without it, there is no concrete contract for implementers to code against.

> **DECISION: Accept.**
> Rationale: The kickoff prompt and thread-key propagation are the causal root of the whole demo, and the current design only says "run-specific thread key" without defining how it is generated or surfaced to both participants. The revised artifacts should define the kickoff prompt contract and the thread-key injection path explicitly.

### Should Fix / Important

4. **State schemas are referenced but not defined.** The design and spec reference `control/demo_state.json`, `control/inspect.json`, `control/report.json`, `control/report.sanitized.json`, and `conversation_events.jsonl` as stable artifacts, but none of these schemas appear in the design or spec. The spec requires the sanitized report to "mask non-deterministic values" and the event stream to include specific event types (`kickoff_accepted`, `mail_sent`, etc.), but without a schema the implementer must invent the structure. Even a minimal JSON-schema or Pydantic model sketch in the design would prevent divergence between the implementation and the verification/test tasks (3.1–3.4, 4.2–4.3).

> **DECISION: Refine.**
> Rationale: The artifacts do need a minimum contract for the stable demo-owned files because `demo_state.json`, `inspect.json`, `report.json`, and the event stream are referenced as durable interfaces by the spec and tasks. But repo precedent keeps the full schema in code, so the follow-up should add required fields and ownership boundaries in the design, not a full JSON-schema or Pydantic dump.

5. **Task 1.1 does not explicitly include creating `run_demo.sh` or choosing the shell-wrapper pattern.** The repository has two established `run_demo.sh` patterns: Pattern A (thin dispatcher → `scripts/demo_driver.py` → `src/houmao/demo/<pack>/driver.py`, used by `houmao-server-dual-shadow-watch` and `shared-tui-tracking-demo-pack`) and Pattern B (full bash orchestration with inline logic, used by `mailbox-roundtrip-tutorial-pack`). The design implies Pattern A by proposing a backing module under `src/houmao/demo/mail_ping_pong_gateway_demo_pack/`, but this is not made explicit in the task list. Task 1.1 says "runner, helper-script" without naming `run_demo.sh` or specifying which pattern.

> **DECISION: Accept.**
> Rationale: The new capability spec explicitly requires `run_demo.sh`, and the design already points toward the newer thin-wrapper plus backing-module pattern. Task 1.1 should name that deliverable and pattern directly so implementation does not have to infer it.

6. **Task dependencies are implicit and may cause implementation confusion.** Several tasks have ordering requirements that are not stated: Task 2.2 (launch agents) requires Task 1.4 (project provisioning) and Task 2.1 (server startup). Task 2.3 (gateway attach) requires Task 2.2 (agent launch). Task 2.4 (kickoff + wait) requires Tasks 2.2 and 2.3. Task 3.1 (event capture) is needed by Tasks 3.2, 3.3, and 3.4. Task 4.2/4.3 (tests) need the schemas from Tasks 3.1–3.3. The task numbering implies a rough ordering via section grouping, but making dependencies explicit would reduce implementation risk, especially if tasks are parallelized across agents.

> **DECISION: Reject.**
> Rationale: The current numbered task groups already encode the intended order and match the repo's normal OpenSpec task style; adding explicit "depends on" annotations would add a second dependency notation that the other task artifacts do not use. The real task-list gap is missing concrete deliverables such as `run_demo.sh`, not missing dependency metadata.

7. **The `houmao-server` startup configuration is underspecified.** The design says "start one demo-owned `houmao-server` under `<output-root>/server/`" but does not specify: the port (or port-selection strategy for avoiding conflicts), whether `--startup-child` is used to co-launch the CAO subprocess, the `--runtime-root` override, or health-check polling parameters. The existing `houmao-server-dual-shadow-watch` demo pack has an established pattern for this in `src/houmao/demo/houmao_server_dual_shadow_watch/driver.py`. The design should either reference that pattern or define its own.

> **DECISION: Refine.**
> Rationale: The design should specify demo-owned loopback listener selection, runtime-root override, and bounded health polling, ideally by referencing the established `houmao-server-dual-shadow-watch` startup pattern. It does not need to commit to `--startup-child`, because this headless-only demo does not depend on CAO child startup.

8. **The spec's dummy-project provisioning requirement is vague.** The spec requires "copied dummy-project workdirs under `<output-root>/projects/initiator/` and `<output-root>/projects/responder/`" but does not define what files a dummy project must contain or whether it should reference existing fixtures. The repo has a `runtime-agent-dummy-project-fixtures` spec (`openspec/specs/runtime-agent-dummy-project-fixtures/`) that likely defines a standard. The design should reference that spec or describe the minimum project content needed for a headless agent to launch successfully (e.g., a `.git` directory, a `README.md`, etc.).

> **DECISION: Refine.**
> Rationale: The design should explicitly reuse the existing dummy-project fixture family instead of leaving the source fixture implicit. It does not need to restate the minimum project contents in full because that contract already exists in `runtime-agent-dummy-project-fixtures`.

### Nice To Have

> **DECISION: Accept.**
> Rationale: This question should not remain open in an apply-ready change. The follow-up artifact revision should explicitly defer a pack-local live-agent `autotest/` harness and keep v1 regression coverage pytest-based, with live HTT left for a later change if the pack contract stabilizes.

10. **The transport-neutral abstraction boundary is aspirational but untested.** The design says "keep the pack contract transport-neutral while implementing headless/headless only in v1." In practice, headless-specific concepts (turn IDs, headless turn terminal states, `/turns/*` inspection) are deeply embedded in the evidence model. A future TUI or mixed-mode extension would need a different evidence collection path. Consider adding a brief note in the design about which specific interfaces would need transport abstraction versus which are inherently headless-only, so a future change has a clearer starting point.

> **DECISION: Refine.**
> Rationale: The design can keep a transport-neutral operator/report vocabulary while admitting that v1 evidence collection is headless-specific. A short note separating transport-neutral artifact names from headless-only collectors is sufficient; a broader abstraction layer is not needed now.

11. **Conversation timeout and polling parameters are not specified.** The design mentions a "wait loop for conversation completion or timeout" but does not specify the default timeout duration, the polling interval, or the diagnostic output when a timeout occurs. These are important for operator experience — a 5-round conversation through gateway wake-ups could take several minutes, and the operator needs feedback during the wait.

> **DECISION: Refine.**
> Rationale: The design should define that `wait` uses bounded polling with visible progress and timeout diagnostics, and it should name where those defaults live. It does not need to freeze every numeric value in the spec, but leaving timeout behavior implicit is too loose for operator-facing workflow design.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should agents discover mail-handling behavior — via role prompt, skill injection, or both?

- Decision: `Blocking`

#### Why this matters

- The entire demo conversation depends on agents knowing how to: (a) check for unread mail when woken by the gateway notifier, (b) read a thread and compose a contextually appropriate reply, (c) track the round count and stop when the limit is reached, (d) use the correct mailbox send/reply primitives. Without this, the gateway notifier will surface "you have unread mail" but the agent will not know what to do with it.

#### Examples

- The initiator receives a kickoff prompt saying "start a ping-pong conversation." It must know to use the mailbox `send` primitive, not just print a response.
- The responder wakes on a gateway notifier digest. It must know to call `mail check`, read the thread, compose a reply, and call `mail reply`.
- On round 5, the initiator reads reply 5 and must decide to stop — it needs a round-limit instruction.

#### Why this question arises in code

- The gateway mail-notifier (`gateway_service.py` lines 714–763) generates a text digest prompt describing unread messages but does not inject mail-handling instructions.
- The agent's behavioral contract comes from the role system prompt (`roles/<role>/system-prompt.md`) and optional skill injections (`brains/skills/<name>/SKILL.md`).
- Existing mailbox-capable roles in `context/` or `.agentsys/agents/roles/` may already include mail-handling instructions, but the design doesn't reference any.

#### Options

**Option A: Define a new demo-specific role that embeds mail-handling instructions and the round-limit contract directly in the system prompt.**

- Code impact:
  - Create `roles/mail-ping-pong-initiator/system-prompt.md` and `roles/mail-ping-pong-responder/system-prompt.md` under the agent definition directory.
  - Each role includes explicit instructions for mailbox interaction and round-limit awareness.
- User-facing difference:
  - Self-contained demo — all behavioral contracts are visible in the role prompt.
- Pros:
  - Maximum clarity and debuggability.
  - No dependency on external skill definitions.
- Cons:
  - Duplicates mail-handling instructions that may exist in reusable skills.

**Option B: Use an existing mail-handling skill (e.g., `agent-mailbox-system-skills`) plus a thin demo-specific role overlay.**

- Code impact:
  - Reference existing `brains/skills/` entries in the brain recipe.
  - Role prompt only adds the round-limit and ping-pong conversation contract.
- User-facing difference:
  - Demonstrates the composable skill + role pattern, which is the intended architectural approach.
- Pros:
  - Follows the skill-composition pattern the framework is designed around.
  - Reuses proven mail-handling instructions.
- Cons:
  - Adds a dependency on a skill that must exist and be correct.
  - Harder to debug if the skill's instructions don't align with the demo's expectations.

> **PROPOSED: Option B — use existing mail-handling skills plus a thin demo-specific role.**
> Rationale: The framework's architecture separates reusable capabilities (skills) from behavioral policy (roles). The `agent-mailbox-system-skills` spec exists in `openspec/specs/`, indicating the system already has a mail-handling skill contract. Using it teaches the intended composition pattern and avoids duplicating mail instructions. The demo-specific role only needs to add the ping-pong thread semantics and round-limit contract, keeping it focused on what's unique to this demo.
>
> **DECISION: Accept PROPOSED with refinement — use the runtime-owned mailbox system skill plus thin demo-specific initiator and responder role overlays, not the bare tracked `mailbox-demo` role alone.**
> Rationale: The existing mailbox skill contract and gateway reminder prompt already tell the agent to use the runtime-owned mailbox capability, so reusing that system-owned behavior is the right base. But the current tracked `mailbox-demo` role only narrows scope; the demo still needs a thin role overlay that adds the ping-pong thread semantics, reply policy, and round-limit stop rule.

### Q2) Should the design specify state schemas for `demo_state.json`, `report.json`, and `conversation_events.jsonl`, or leave them to the implementer?

- Decision: `Deferrable`

#### Why this matters

- Tasks 3.1–3.4 (event capture, inspect, report, snapshot-refresh) and Tasks 4.2–4.3 (automated tests) all depend on these schemas. Without a shared schema definition, the implementer of the event-capture task and the implementer of the test task may produce incompatible structures. The sanitized report comparison (Task 3.3) specifically needs a stable, well-defined masking contract.

#### Examples

- `demo_state.json` must persist enough to support `inspect`, `verify`, `pause`, `continue`, and `stop` commands — but what fields? Agent refs? Server PID? Gateway ports? Thread key?
- `conversation_events.jsonl` lists event types (`kickoff_accepted`, `mail_sent`, etc.) but does not define the event record shape (timestamp? agent_ref? round_number? message_id?).

#### Why this question arises in code

- Existing demo packs define their state schemas implicitly in code (e.g., `cao_interactive_demo/models.py` has `DemoState` and `VerificationReport` as Pydantic models) rather than in the design document.
- The change's design doc has a detailed event-type vocabulary but no field-level schema.

#### Options

**Option A: Add minimal Pydantic model sketches to the design document for the three key artifacts.**

- Code impact:
  - Design doc grows by ~50–80 lines.
  - Implementers have a concrete starting point.
- User-facing difference:
  - Review and implementation feedback can validate the schema early.
- Pros:
  - Reduces implementation ambiguity.
  - Enables parallel task work (event capture and tests can be developed against the same schema).
- Cons:
  - Design doc becomes more prescriptive; may over-constrain implementation.

**Option B: Leave schemas to the implementer, following the existing demo pack precedent where models are defined in code.**

- Code impact:
  - No design doc change.
  - Implementer defines schemas in `src/houmao/demo/mail_ping_pong_gateway_demo_pack/models.py`.
- User-facing difference:
  - Faster design finalization; schema evolves during implementation.
- Pros:
  - Matches existing precedent (`cao_interactive_demo/models.py`).
  - Less design doc maintenance burden.
- Cons:
  - Risk of schema drift if tasks 3.x and 4.x are implemented by different agents or in different sessions.

> **PROPOSED: Option B — leave schemas to the implementer, following existing precedent.**
> Rationale: All existing demo packs in `src/houmao/demo/` define their state models in code (`models.py`) rather than in design documents. The `cao_interactive_demo/models.py` defines `DemoState`, `TurnRecord`, and `VerificationReport` as Pydantic models without a prior design-level schema. Since the change is in active unstable development and the tasks are likely to be implemented sequentially (3.1 before 3.3, 3.3 before 4.2), schema drift risk is low. However, the finding (Finding #4) should still be acknowledged — implementers should create `models.py` early in the scaffolding phase (Task 1.1) to serve as the shared contract.
>
> **DECISION: Reject PROPOSED — add a minimal design-level field contract, then keep the concrete models in code.**
> Rationale: These artifacts are treated as stable demo-owned interfaces by the spec and tasks, so leaving them entirely to the implementer is too loose for later verification and test work. Existing demo packs still define the real Pydantic models in `models.py`, so the revision should capture required fields and ownership boundaries in the design without bloating it into full schema listings.

### Q3) Should the design reference or reuse the existing dummy-project fixture pattern, or define its own?

- Code impact:
  - Import or copy from the existing fixture location.
  - Task 1.4 implementation follows established fixture structure.

  - Reuse reduces maintenance burden.

> **PROPOSED: Option A — reference and reuse the existing `runtime-agent-dummy-project-fixtures` pattern.**

Reusing it keeps the demo aligned with the testing infrastructure and avoids duplicating fixture maintenance. The design should add a brief reference to this spec in the dummy-project provisioning section.

> **DECISION: Accept PROPOSED.**
> Rationale: The repository already has a standard dummy-project fixture family and lightweight mailbox-demo fixture guidance for narrow runtime flows. Reusing that pattern keeps this demo aligned with existing launch and test infrastructure and avoids inventing a second minimal-project convention.
