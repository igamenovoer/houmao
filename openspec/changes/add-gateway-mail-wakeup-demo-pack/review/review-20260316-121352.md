## Findings

### Must Fix / Blocking

- **No blocking issues found.** The change is internally consistent and the spec requirements are achievable against the current codebase. However, the implementation path for structured notifier auditing (Finding S1 below) needs careful attention because it is the primary new runtime code contribution and has no precedent in the existing notifier code path.

### Should Fix / Important

- **S1: Structured notifier-decision audit storage surface is unspecified.** The agent-gateway delta spec requires "one structured notifier-decision audit record for each enabled poll cycle in gateway-owned durable history" with fields including poll time, unread count, unread-set identity, admission state, execution state, queue depth, decision outcome, request id, and skip detail. The current codebase has two candidate surfaces:

  1. `events.jsonl` — currently used only for request lifecycle events (`accepted`, `accepted_internal`, `completed`, `failed`; see `gateway_service.py:336,672,830` and `gateway_storage.py:546`).
  2. A new table in `queue.sqlite` — where the singleton `gateway_mail_notifier` row already lives (see `gateway_storage.py:670`).

  Neither the delta spec nor the design explicitly prescribes one surface over the other. Design Decision 7 says "it should favor the existing append-only structured history surface over abusing the human log for machine verification" — which points toward `events.jsonl`, but introducing high-frequency notifier poll events (one per `interval_seconds`, potentially every 1 second per the demo spec) into the same file that records queue lifecycle events could make `events.jsonl` significantly noisier and harder to query for request-level debugging.

  This deserves an explicit choice in the design or spec before implementation begins. See Open Question Q1.

- **S2: Demo spec mandates `expected_report/report.json` but notifier audit data is time-varying.** The gateway-mail-wakeup-demo-pack spec (Scenario: "Verification report explains why wake-up succeeded or was skipped") says the sanitized report should be "compared reproducibly against `expected_report/report.json`." Gateway decision-audit evidence includes timestamps, request IDs, poll counts, and digest hashes. The sanitization contract for these fields is unspecified. The mailbox roundtrip tutorial pack achieves reproducibility by masking `<DEMO_OUTPUT_DIR>`, `<PROJECT_WORKDIR>`, `<MESSAGE_ID>`, etc. (see `scripts/demo/mailbox-roundtrip-tutorial-pack/README.md:160-175` and `scripts/sanitize_report.py`). The wake-up demo will need a similarly explicit sanitization contract for notifier-specific fields. This should be acknowledged in the spec or the tasks.

- **S3: The `GatewayMailNotifierStatusV1` model does not expose audit history summary.** The delta spec in `specs/agent-gateway/spec.md` focuses on durable per-poll history records. Design Open Question 2 asks whether `GET /v1/mail-notifier` should expose a summary of the last structured decision. The current `GatewayMailNotifierStatusV1` model (see `gateway_models.py:372-419`) returns `last_poll_at_utc`, `last_notification_at_utc`, and `last_error`, but has no field for `last_decision_outcome` or `last_unread_count`. The demo's inspect/verify steps (Task 3.3) will need either an API extension or a direct file read of the audit history. The spec should clarify the expected inspection path.

- **S4: Task dependency ordering is implicit.** Task 1.1 (structured notifier auditing) must land before Task 3.3 (inspection/verification of notifier audit evidence) and Task 4.3 (running targeted tests for auditing). The tasks file organizes work into four numbered groups, but does not declare cross-group dependencies. The current two-tier structure (groups 1–4) implies sequential execution, which is likely correct, but an explicit dependency note from Task 3.3 → Task 1.1 would prevent a contributor from attempting the demo verification flow before the audit contract exists.

### Nice To Have

- **N1: Consider aligning demo output layout documentation with the stabilization change.** The `stabilize-mailbox-roundtrip-tutorial-pack-after-local-state` design (Decision 2) proposes a `cao/` subtree under `<demo-output-dir>` for launcher-managed CAO state. The new gateway demo pack should document whether it follows the same layout convention or has a different structure, especially for the `gateway/` subtree that holds the audit artifacts the demo needs to inspect.

- **N2: The idle-check injector interval (3 seconds) and the notifier interval (1 second) create a subtle timing window.** The automatic workflow spec says the injector checks every 3 seconds and the notifier polls every 1 second. After mail injection, the next notifier poll should find unread mail within ~1 second. But if the agent processes the wake-up task very quickly (< 3 seconds), the injector's "idle" check might not observe the busy window. This is cosmetic for the demo (the output file still proves the wake-up happened), but it is worth a note in the README so operators do not misread the idle-check interval as the notifier's own timing.

- **N3: The 10-message cap in `_build_mail_notifier_prompt` is undocumented.** The current `gateway_service.py:636` truncates the unread summary to 10 messages with a "and N more" trailer. This is reasonable behavior but is not part of the notifier contract in the delta spec. The burst test scenario in the demo spec mentions verifying "all of those messages remain represented in unread mailbox state until processed explicitly" — the demo should verify unread state via mailbox SQLite rather than relying on prompt content, which is already the stated intent, but it is worth confirming this boundary during implementation.

## Design Alignment With Existing System

- **Existing tutorial-pack pattern:** The new pack follows the established pattern from `scripts/demo/mailbox-roundtrip-tutorial-pack/` (README, `run_demo.sh`, `inputs/`, `expected_report/report.json`, helper scripts under `scripts/`). The design's Decision 3 (automatic + manual workflow) appropriately borrows from both the mailbox roundtrip pack (one-shot `run_demo.sh`) and the interactive CAO demo pack (stateful subcommands like `start`, `inspect`, `stop`). This hybrid shape fits the gateway wake-up use case well.

- **Mail injection boundary:** The design correctly uses `deliver_message.py` as the managed mailbox mutation boundary (Decision 4) rather than direct SQLite writes. This aligns with the mailbox module architecture at `src/houmao/mailbox/managed.py:1172` (`deliver_message`) and `src/houmao/mailbox/assets/rules/scripts/deliver_message.py`.

- **Gateway HTTP surface:** The change adds no new public request kinds. Notifier control remains on `GET/PUT/DELETE /v1/mail-notifier`, and internal notifier prompts use `mail_notifier_prompt` as an internal stored-request kind (`GatewayStoredRequestKind` at `gateway_models.py:19`). This is consistent with `docs/reference/gateway/contracts/protocol-and-state.md:161` which states "The notifier reminder path does not add a new public request kind."

- **Gateway notifier code path:** The existing `_run_notifier_cycle` in `gateway_service.py:492-582` already implements the core unread-set semantics: digest-based deduplication (`_mail_notifier_digest` at line 620), busy-session deferral (lines 550-568), and single-prompt summarization (`_build_mail_notifier_prompt` at line 626). The new structured audit requirement overlays this existing logic without changing its decision semantics — it adds observability, not behavior. This is a clean alignment.

- **Event model:** The existing `events.jsonl` append-only log uses `kind` as a discriminator (`accepted`, `accepted_internal`, `completed`, `failed`). The proposed notifier poll events would add new kinds (e.g., `notifier_poll_empty`, `notifier_poll_dedup_skip`, `notifier_poll_busy_skip`, `notifier_poll_enqueued`, `notifier_poll_error`). This is structurally consistent but increases event volume significantly when the notifier polls at 1-second intervals. See Q1.

- **Mismatch/gap — `GatewayMailNotifierRecord` lacks structured audit fields:** The durable notifier record (`gateway_storage.py:102`) stores only operational bookkeeping (`enabled`, `interval_seconds`, `last_poll_at_utc`, `last_notification_at_utc`, `last_notified_digest`, `last_error`). It does not store per-poll decision history. The spec requires durable per-poll records, which means either `events.jsonl` or a new SQLite table is needed. The current `gateway_mail_notifier` singleton table in `queue.sqlite` is not sufficient.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Where should structured notifier-decision audit records be persisted?

- Decision: `Blocking`

#### Why this matters

- The delta spec for agent-gateway requires "one structured notifier-decision audit record for each enabled poll cycle in gateway-owned durable history." This is the foundational data contract that Task 1.1, Task 3.3, and the demo verification flow depend on. If the storage surface is wrong, the implementation will need rework.

#### Examples

- A demo run with `interval_seconds=1` running for 60 seconds generates ~60 poll events. If stored in `events.jsonl` alongside queue lifecycle events, the file gains 60 notifier lines per minute. A 10-minute demo session with a busy gateway (lots of request lifecycle events) would mix notifier polls into the same stream, making both harder to grep.
- A demo verify step needs to query "show me all poll decisions where outcome=busy_skip" — this is trivial with SQLite `WHERE` clauses but requires line-by-line JSON parsing of `events.jsonl`.

#### Why this question arises in code

- `gateway_storage.py:546` defines `append_gateway_event()` which writes to `events.jsonl`, currently only for request lifecycle events.
- `gateway_storage.py:670` defines the `gateway_mail_notifier` table in `queue.sqlite`, but as a singleton config row, not a history table.
- `gateway_service.py:492-582` (`_run_notifier_cycle`) is the implementation site where audit records would be emitted, but the method currently uses only `write_gateway_mail_notifier_record` (singleton update) and `_log_rate_limited` (human log).

#### Options

**Option A: Append notifier poll events to `events.jsonl`**

- Code impact:
  - Add new event kinds (`notifier_poll_empty`, `notifier_poll_dedup_skip`, etc.) to `_run_notifier_cycle`.
  - Each poll cycle calls `append_gateway_event(self.m_paths, {...})` with the required fields.
  - No schema migration needed.
  ```python
  append_gateway_event(self.m_paths, {
      "kind": "notifier_poll",
      "poll_time_utc": poll_time_utc,
      "unread_count": len(unread_messages),
      "unread_digest": unread_digest,
      "admission": status.request_admission,
      "execution": status.active_execution,
      "queue_depth": status.queue_depth,
      "outcome": "enqueued",
      "request_id": request_id,
  })
  ```
- User-facing difference:
  - `events.jsonl` becomes a mixed stream of request events and notifier poll events.
  - Demo verify scripts must parse JSONL and filter by `kind`.
- Pros:
  - Reuses an existing append-only surface; no new table or file format to maintain.
  - Consistent with the design's language about "existing append-only structured history surface."
- Cons:
  - High-frequency poll events (1/s) make `events.jsonl` noisier for request-level debugging.
  - JSONL line-scan is less efficient than SQLite queries for demo verification.

**Option B: Add a `gateway_notifier_audit` table in `queue.sqlite`**

- Code impact:
  - Add `CREATE TABLE IF NOT EXISTS gateway_notifier_audit (...)` in `ensure_gateway_sqlite_schema()` at `gateway_storage.py:~660`.
  - Add `insert_notifier_audit_record()` and `query_notifier_audit_records()` helpers in `gateway_storage.py`.
  - `_run_notifier_cycle` inserts one row per poll cycle via the new helper.
  ```python
  insert_notifier_audit_record(self.m_paths.queue_path, {
      "poll_time_utc": poll_time_utc,
      "unread_count": len(unread_messages),
      "unread_digest": unread_digest,
      "admission": status.request_admission,
      "execution": status.active_execution,
      "queue_depth": status.queue_depth,
      "outcome": "enqueued",
      "request_id": request_id,
  })
  ```
- User-facing difference:
  - `events.jsonl` stays clean for request lifecycle events.
  - Demo verify scripts can use SQL queries directly against `queue.sqlite`.
- Pros:
  - Queryable: the demo verify step can `SELECT * FROM gateway_notifier_audit WHERE outcome = 'busy_skip'`.
  - Does not pollute the request event stream.
  - Aligns with how the existing notifier config is already in `queue.sqlite`.
- Cons:
  - The design says "favor the existing append-only structured history surface" — this diverges from that language.
  - Adds a new table and helper functions.

**Option C: Both — SQLite for queryable history, events.jsonl for notable events only**

- Code impact:
  - SQLite table for full per-poll audit (as in Option B).
  - `events.jsonl` only for notable notifier events (`enqueued`, `poll_error`), not for every empty/dedup/busy poll.
- User-facing difference:
  - `events.jsonl` gains a few new event kinds but stays low-volume.
  - SQLite provides full audit history for demo verification.
- Pros:
  - Best of both: queryable audit + clean event stream + notable events still in the unified log.
- Cons:
  - More implementation work; two write paths for overlapping data.

> **PROPOSED: Option B — add a `gateway_notifier_audit` table in `queue.sqlite`.**
> Rationale: The demo verification flow (Task 3.3) needs to query poll decisions by outcome, unread count, and timing. SQLite is already the gateway's durable structured store (`queue.sqlite` holds `gateway_requests` and `gateway_mail_notifier` per `gateway_storage.py:660-695`), and a queryable table directly serves the demo's `inspect` and `verify` commands without JSONL line-scanning. The design's "favor the existing append-only structured history surface" language was written before the demo verification queryability requirement was fully specified — the intent was "not just `gateway.log`," and SQLite satisfies that intent while being more practical for the demo use case. Notable events (`enqueued`, `poll_error`) can still optionally appear in `events.jsonl` as a low-cost enhancement later without blocking the audit contract.
>
> **DECISION: Accept Option B for v1 — add a dedicated `gateway_notifier_audit` table in `queue.sqlite`, and keep `events.jsonl` request-lifecycle only for now.**
> Rationale: The current gateway already treats `queue.sqlite` as its durable operational store for requests and notifier state in `gateway_storage.py`, while `events.jsonl` stays focused on request lifecycle transitions. The demo's `inspect` and `verify` steps need queryable per-poll history, and pushing 1-second notifier polls into `events.jsonl` would make the request event stream noisier without helping the tutorial workflow. We can revisit low-volume notifier event mirroring later, but v1 should choose one primary durable audit surface and keep that choice simple.

### Q3) Should `GET /v1/mail-notifier` be extended with a last-decision summary, or should the demo rely solely on the durable audit history?

- Decision: `Deferrable`

#### Why this matters

- The demo's `inspect` subcommand needs to show the operator what the notifier just decided and why. The current `GatewayMailNotifierStatusV1` response (see `gateway_models.py:372-419`) includes `last_poll_at_utc`, `last_notification_at_utc`, and `last_error`, but has no `last_decision_outcome` (e.g., `empty`, `dedup_skip`, `busy_skip`, `enqueued`) or `last_unread_count`. Without that, `inspect` must read the audit history directly.

#### Examples

- An operator runs `inspect` and sees `last_poll_at_utc: 2026-03-16T12:00:05Z`, `last_error: null`, but cannot tell whether the last poll found unread mail and deferred because of a busy session, or found no unread mail at all.
- With a `last_decision_outcome: busy_skip` field, `inspect` could show the operator the immediate answer without querying SQLite.

#### Why this question arises in code

- `GatewayMailNotifierStatusV1` at `gateway_models.py:372` is the HTTP response model.
- `build_gateway_mail_notifier_status` at `gateway_storage.py:470` constructs it from the singleton record.
- Adding fields to the HTTP model is a contract change visible to external callers.

#### Options

**Option A: Do not extend the HTTP model; demo reads audit history directly**

- Code impact:
  - No HTTP model change.
  - Demo `inspect` queries `gateway_notifier_audit` table (or parses `events.jsonl`) from the gateway root.
- User-facing difference:
  - `GET /v1/mail-notifier` stays compact.
  - `inspect` needs filesystem access to the gateway root.
- Pros:
  - Avoids a contract change to the HTTP surface.
  - Keeps the HTTP model minimal.
- Cons:
  - Demo `inspect` must know how to find and query the gateway root.
  - Less convenient for external tooling that only has HTTP access.

**Option B: Add optional last-decision fields to `GatewayMailNotifierStatusV1`**

- Code impact:
  - Add `last_decision_outcome: str | None = None` and `last_unread_count: int | None = None` to the model.
  - Update `build_gateway_mail_notifier_status` to populate them from the singleton record.
  - Extend `GatewayMailNotifierRecord` in `gateway_storage.py` with those fields.
- User-facing difference:
  - `GET /v1/mail-notifier` includes the last poll's decision context.
- Pros:
  - `inspect` can show decision context from HTTP alone.
  - More useful for any external tool watching notifier state.
- Cons:
  - Adds fields to a strict `extra="forbid"` Pydantic model — any consumer that uses strict validation against the old schema will break.

> **PROPOSED: Option A — do not extend the HTTP model now; the demo reads audit history directly from the gateway root.**
> Rationale: The gateway HTTP models use `extra="forbid"` strict validation (`gateway_models.py:54`), so adding optional fields is technically backward-compatible for JSON consumers but could still surprise strict Pydantic consumers. The demo already needs filesystem access to the gateway root for report building (Task 3.3 captures "gateway decision-audit evidence"), so adding a direct audit-table query path is natural. The HTTP extension can be considered in a follow-up change once the audit contract is stable and has been exercised in the demo.
>
> **DECISION: Accept Option A — keep `GET /v1/mail-notifier` as a compact status snapshot and have the demo inspect durable audit history directly.**
> Rationale: This change is about adding durable audit visibility and a tutorial pack, not expanding the public HTTP contract. The current notifier status model in `gateway_models.py` is intentionally small, and the demo already has filesystem access to the gateway root for report building and verification. We should exercise the new audit surface directly first, then revisit HTTP summary fields later if operators still need them.

### Q4) What sanitization rules should the demo report use for notifier audit data?

- Decision: `Blocking`

#### Why this matters

- The demo spec requires that the sanitized report be "compared reproducibly against `expected_report/report.json`." Notifier audit records contain timestamps, request IDs, digest hashes, and per-poll counts that vary on every run. Without explicit sanitization rules, the `expected_report/report.json` contract will either be too brittle (exact match fails on every run) or too loose (masks so much data that the report proves nothing).

#### Examples

- A raw report might include `"last_poll_at_utc": "2026-03-16T12:05:03+00:00"` — this must be masked to a placeholder.
- A raw report might include `"notifier_decisions": [{"outcome": "empty", ...}, {"outcome": "enqueued", "request_id": "gwreq-20260316-..."}]` — should the sanitized report preserve the outcome sequence but mask timestamps and IDs, or should it reduce to summary counts like `{"enqueued_count": 1, "empty_count": 3}`?

#### Why this question arises in code

- The mailbox roundtrip pack sanitizes by replacing known variable values with placeholders (see `scripts/demo/mailbox-roundtrip-tutorial-pack/scripts/sanitize_report.py`).
- The gateway demo needs a more complex sanitization strategy because notifier audit data has both structural content (decision sequence, outcome types) and variable content (timestamps, IDs, counts).

#### Options

**Option A: Summarize audit data to counts and outcome presence**

- Code impact:
  - Sanitization reduces the audit history to `{"total_polls": "<COUNT>", "outcomes": {"enqueued": true, "empty": true, "busy_skip": false, ...}}`.
  - The `expected_report` asserts which outcome types occurred, not how many times.
- User-facing difference:
  - Report proves "the notifier did enqueue a wake-up" and "no poll errors occurred" without timing sensitivity.
- Pros:
  - Highly reproducible across runs with varying timing.
  - Still proves the key behavioral contract.
- Cons:
  - Loses the audit sequence detail that makes the report useful for debugging.

**Option B: Preserve audit sequence with masked variable fields**

- Code impact:
  - Sanitization replaces timestamps with `<TIMESTAMP>`, request IDs with `<REQUEST_ID>`, digests with `<DIGEST>`, but preserves the outcome field and the count of records.
  - The `expected_report` asserts the exact sequence of outcomes (e.g., `["empty", "empty", "enqueued"]`).
- User-facing difference:
  - Report shows the precise decision sequence.
- Pros:
  - Richer verification — proves the exact notifier behavior path.
- Cons:
  - Outcome sequence depends on timing (number of empty polls before mail injection varies by run).

> **PROPOSED: Option A — summarize audit data to outcome presence plus a few key structural assertions.**
> Rationale: The mailbox roundtrip pack's existing sanitization strategy (see `scripts/demo/mailbox-roundtrip-tutorial-pack/scripts/sanitize_report.py`) focuses on replacing variable values with stable placeholders while preserving structural checks. For notifier audit data, the exact poll count and sequence depend on timing that varies across machines and runs. The gateway demo should assert: (1) notifier was enabled, (2) at least one `enqueued` outcome occurred in the automatic case, (3) no `poll_error` outcomes occurred, (4) output file exists. This matches the design's guidance that "the golden assertion [should be] narrow" (Design, Risks section) and that transcript/model-specific evidence is supplemental. Sequence-level assertions can be added later as optional debug output if needed.
>
> **DECISION: Accept Option A, refined — the sanitized golden report should reduce notifier audit data to stable outcome-summary assertions rather than preserving per-poll sequence.**
> Rationale: The existing tutorial-pack flow compares sanitized JSON for exact equality, so sequence-level notifier assertions would be fragile across runs because poll timing is intentionally variable. For v1, `expected_report/report.json` should preserve stable facts such as notifier enabled, which outcome types were observed, whether any `poll_error` occurred, whether an `enqueued` decision appeared when expected, and whether the output file evidence exists. Full audit rows can still remain available in raw or debug artifacts for investigation, but the golden snapshot should stay summary-shaped and reproducible.
