- Proposed design alignment:
  - Publishing `AGENTSYS_AGENT_GATEWAY_HOST/PORT`, `AGENTSYS_GATEWAY_ROOT`, etc. fits the existing tmux discovery style and should reuse the same error patterns as `_read_tmux_session_env_var()` (`src/gig_agents/agents/brain_launch_runtime/runtime.py:820`).

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What explicitly makes a session “gateway-enabled”?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- The change claims gateway integration is additive and must preserve non-gateway sessions (`openspec/changes/add-agent-gateway-sidecar/design.md:280`), but the specs do not define the enablement switch.
- Without a crisp enablement rule, implementation will either accidentally start a gateway for all tmux-backed sessions (high impact) or never start it (no effect).

#### Examples

- A developer runs `gig-agents-cli brain-launch-runtime start-session ...` today; should that default to “legacy” or “gateway-enabled” behavior?
- A blueprint declares `gateway.host/port`; is that an enablement signal or only defaults used if enabled elsewhere?

#### Why this question arises in code

- `start-session` currently has no gateway-related flags (`src/gig_agents/agents/brain_launch_runtime/cli.py:105`), and `start_runtime_session()` has no gateway parameters (`src/gig_agents/agents/brain_launch_runtime/runtime.py:242`).
- Blueprint parsing currently ignores any `gateway:` section (`src/gig_agents/agents/brain_launch_runtime/loaders.py:85`).

#### Options

**Option A: Explicit enablement flag (recommended)**

- Code impact:
  - Add `--enable-gateway` (or `--gateway`) to `start-session` and persist a `gateway_enabled: true` field in the session manifest.
  - Allow `--gateway-host/--gateway-port` only when enabled; otherwise reject with an explicit error.
  - Blueprint may include `gateway.*` defaults that only apply when enabled.
- User-facing difference:
  - Existing sessions continue to behave as today unless the user opts in.

**Option B: Implicit enablement by presence of gateway config**

- Code impact:
  - Treat any of `--gateway-host`, `--gateway-port`, env `AGENTSYS_AGENT_GATEWAY_*`, or blueprint `gateway.*` as “enable gateway”.
  - Risk: free-port fallback becomes an implicit enablement if not carefully guarded.
- User-facing difference:
  - A user with ambient env vars or a blueprint with gateway defaults could “turn on” a network service without noticing.

> **PROPOSED: Use explicit session enablement via a boolean flag (Option A), defaulting to disabled.**
> Rationale: The design explicitly calls out additive integration and preserving non-gateway sessions (`openspec/changes/add-agent-gateway-sidecar/design.md:280`), and the current runtime CLI/control surface has no notion of gateway enablement (`src/gig_agents/agents/brain_launch_runtime/cli.py:105`). Making enablement explicit avoids accidental activation via free-port fallback or ambient env vars and gives the manifest a clear source of truth for resume/control routing (`src/gig_agents/agents/brain_launch_runtime/manifest.py:62`).
>
> **DECISION: Accept the explicit opt-in direction, but persist it as a first-class gateway launch mode instead of inferring enablement from host/port presence alone.**
> Rationale: This change is explicitly additive, so legacy sessions must remain non-gateway by default. `gateway.host` and `gateway.port` should remain defaults that apply only after explicit opt-in, and the persisted session state should carry one authoritative gateway-enabled marker for resume and gateway-aware control routing.

### Q2) What is the first backend to support, and how will it satisfy the “same tmux window” sidecar constraint?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- “Same tmux window, no visible second surface” is a core constraint (design: `openspec/changes/add-agent-gateway-sidecar/design.md:48`; agent-gateway spec: `openspec/changes/add-agent-gateway-sidecar/specs/agent-gateway/spec.md:6`).
- Today, the only interactive TUI backend is CAO (`backend=cao_rest`), and CAO creates the tool terminal window itself; gig-agents does not control the exact foreground command running in that window.

#### Examples

- Start a CAO session: `cao_rest` creates a tmux session and then asks CAO server to create a terminal window (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1479`). Where does `gatewayd ... & exec <tui>` run?
- If we instead target headless tmux-backed sessions first, what is the “managed TUI” surface the gateway is meant to control?

#### Why this question arises in code

- CAO client `create_terminal()` accepts only `provider`, `agent_profile`, and `working_directory` — there is no hook for a wrapper command (`src/gig_agents/cao/rest_client.py:120`).
- CAO-backed runtime startup currently creates the tmux session, sets environment, then creates the CAO terminal (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1494`).

#### Options

**Option A: Extend CAO to support a wrapper/entrypoint for terminal creation**

- Code impact:
  - Update CAO server API to accept a wrapper/entrypoint (or provider variant) that runs `gatewayd ... & exec provider`.
  - Update `CaoRestClient.create_terminal()` and `CaoRestSession._start_terminal()` to supply that wrapper for gateway-enabled sessions.
  - Add integration tests covering “bind failure prevents terminal start”.
- User-facing difference:
  - Gateway-enabled CAO sessions keep the existing interactive UX; gateway is truly same-window.

**Option B: Add a new non-CAO interactive tmux backend owned by gig-agents**

- Code impact:
  - Implement an interactive tmux backend that launches the provider TUI directly under tmux and uses the wrapper contract from the design.
  - Keep CAO backend as-is for legacy sessions.
- User-facing difference:
  - New backend path may have different behavior and status semantics than CAO; higher maintenance surface.

> **PROPOSED: Target CAO-backed interactive sessions first, and extend CAO terminal creation to accept a wrapper/entrypoint (Option A).**
> Rationale: The current runtime’s interactive path is CAO (`backend=cao_rest`), and the design’s “wrapper backgrounds gateway then execs TUI” requires control over the foreground command (`openspec/changes/add-agent-gateway-sidecar/design.md:54`) which gig-agents does not have with `create_terminal()` today (`src/gig_agents/cao/rest_client.py:120`). Extending CAO to accept a wrapper keeps the operator UX stable while satisfying the same-window requirement, and avoids introducing a parallel interactive backend inside gig-agents before the gateway contract is proven.
>
> **DECISION: Accept the CAO-first direction and scope the first backend implementation to `cao_rest`, with an explicit CAO wrapper or entrypoint extension as a required dependency.**
> Rationale: The current interactive TUI path already runs through CAO, and adding a second gig-agents-owned interactive backend now would create extra divergence before the gateway contract is validated. The artifacts should state clearly that non-CAO same-window integrations are deferred until the CAO-backed path works end to end.

### Q3) What is the concrete v1 HTTP API surface (paths + models + error semantics)?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- The specs require a “structured HTTP API” (agent-gateway spec: `openspec/changes/add-agent-gateway-sidecar/specs/agent-gateway/spec.md:72`), but without explicit route and payload definitions, implementation and tests are underdetermined.
- A stable API surface is needed for in-repo control tooling to submit work without raw tmux mutation (brain-launch-runtime spec: `openspec/changes/add-agent-gateway-sidecar/specs/brain-launch-runtime/spec.md:134`).

#### Examples

- A local tool wants to check liveness: does it call `GET /health`, `GET /`, or something else?
- A tool submits queued work: what endpoint, what body fields, and what response contains the `request_id`?

#### Why this question arises in code

- There is no existing gateway client/server module in `src/gig_agents/` to anchor API shapes; this will be new surface area.

#### Options

**Option A: Minimal REST-ish v1 with typed models (recommended)**

- Code impact:
  - Define `GET /health` (no auth, fast) and `GET /v1/status` (returns the normalized status model).
  - Define `POST /v1/requests` to enqueue work and return `{request_id, status, queued_at_utc}`.
  - Use pydantic models for request/response validation and explicit error codes (e.g. 409 for conflicts, 403 for policy reject, 422 for validation).
- User-facing difference:
  - Predictable endpoints and schemas for tooling and tests.

**Option B: Single “command” endpoint**

- Code impact:
  - `POST /v1/command` with `{action: "...", payload: {...}}`.
  - Simpler routing but weaker typing and harder to evolve.
- User-facing difference:
  - Less discoverable; errors often become “generic”.

> **PROPOSED: Define a minimal typed v1 API with explicit `GET /health`, `GET /v1/status`, and `POST /v1/requests` endpoints (Option A).**
> Rationale: The change explicitly wants a “well-typed local HTTP contract” (design: `openspec/changes/add-agent-gateway-sidecar/design.md:68`) and already depends on pydantic + FastAPI (`pyproject.toml`). A small stable endpoint set is enough to unblock the runtime integration requirement (“submit managed work through the gateway”: `openspec/changes/add-agent-gateway-sidecar/specs/brain-launch-runtime/spec.md:134`) while keeping v1 scope controlled.
>
> **DECISION: Accept the minimal typed API and make the v1 surface `GET /health`, `GET /v1/status`, and `POST /v1/requests`, with `POST` returning the accepted queued request record.**
> Rationale: This is enough surface area for liveness, normalized state reads, and durable request submission without collapsing everything into a generic command endpoint. The artifact revision should pin the request and response models plus core error semantics: `422` for validation errors, `403` for policy rejection, and `409` for request-state conflicts.

### Q4) Is `state.json` part of the stable gateway protocol contract, and what is its schema/versioning?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- Multiple specs treat `state.json` as a first-class “current state artifact” and publish its path via tmux env (`AGENTSYS_GATEWAY_STATE_PATH`) (`openspec/changes/add-agent-gateway-sidecar/specs/agent-identity/spec.md:8`, agent-gateway spec: `openspec/changes/add-agent-gateway-sidecar/specs/agent-gateway/spec.md:28`).
- If external tools are expected to read it, it needs a schema, compatibility rules, and tests.

#### Examples

- A tool reads `state.json` to show queue depth and whether surface is submit-ready. Which fields are required and what do the enums mean?
- A gateway upgrade changes field names. Is that allowed when `protocol-version.txt` changes?

#### Why this question arises in code

- Current manifests are schema-validated and versioned (`src/gig_agents/agents/brain_launch_runtime/manifest.py:22`), but there is no equivalent defined for gateway state artifacts yet.

#### Options

**Option A: `state.json` is a stable local contract, versioned by protocol version (recommended)**

- Code impact:
  - Define a pydantic model for the `state.json` payload and write/read it atomically.
  - Tie schema to `AGENTSYS_GATEWAY_PROTOCOL_VERSION` and `protocol-version.txt` and document compatibility.
  - Serve the same model via `GET /v1/status` to avoid drift.
- User-facing difference:
  - Tools can reliably read either HTTP status or the file pointer for fast local access.

**Option B: `state.json` is internal-only; external tools must use HTTP**

- Code impact:
  - `state.json` can change without compat guarantees, but then `AGENTSYS_GATEWAY_STATE_PATH` becomes misleading and should be treated as debug-only.
- User-facing difference:
  - Tools must call HTTP even for local reads.

> **PROPOSED: Treat `state.json` as a stable, versioned local contract (Option A), sharing the same model as `GET /v1/status`.**
> Rationale: The design explicitly calls out external readers inspecting “the current gateway state artifact” without replay (`openspec/changes/add-agent-gateway-sidecar/specs/agent-gateway/spec.md:40`) and requires publishing `AGENTSYS_GATEWAY_STATE_PATH` (`openspec/changes/add-agent-gateway-sidecar/specs/agent-identity/spec.md:8`). A pydantic model + protocol version marker matches the repo’s existing “schema + versioned payload” style for manifests (`src/gig_agents/agents/brain_launch_runtime/manifest.py:22`) and reduces drift risk between HTTP and file-based reads.
>
> **DECISION: Accept PROPOSED: `state.json` is a stable read contract at the gateway protocol-version level and must stay schema-aligned with `GET /v1/status`.**
> Rationale: The current specs already publish `AGENTSYS_GATEWAY_STATE_PATH` as a discovery pointer, so treating the file as debug-only would contradict the intended local-tooling contract. The file should be written atomically and remain read-only for callers, while `queue.sqlite` and other storage details stay internal implementation choices.

### Q5) What request kinds are first-class in v1, and how much should `payload_json` be relied on?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- The proposal/design list many behaviors (queueing, timers, wakeups, recovery actions), but the runtime integration should start with a minimal executable slice to avoid an untestable v1.
- Request kinds determine what the backend adapter must support and what tests need to exist.

#### Examples

- Is “send a prompt turn” the only v1 request, or do we also need `interrupt`, `wakeup`, and `restart`?
- If a human changes the surface mid-request, what “completion” means depends on request kind.

#### Why this question arises in code

- Today, `send-prompt` and CAO submission happen directly in the backend (`CaoRestSession.send_prompt()`), not via a queue (`src/gig_agents/agents/brain_launch_runtime/runtime.py:120`, `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:746`).

#### Options

**Option A: Minimal v1 kinds: `submit_prompt` only**

- Code impact:
  - Implement durable enqueue + single execution slot for prompt submission.
  - Defer timers/wakeups/restart to follow-up changes.
- User-facing difference:
  - Gateway v1 solves concurrency and basic scheduling for prompt turns first.

**Option B: Small set of v1 kinds: `submit_prompt`, `interrupt`, `wakeup_nudge`**

- Code impact:
  - Add safe interrupt semantics and a coalescable “nudge” type.
  - Still defer aggressive recovery actions.
- User-facing difference:
  - Better operational control, slightly more surface area to test.

> **PROPOSED: Start with a minimal, testable v1 set: `submit_prompt` plus (optionally) `interrupt` as the only additional control action (closest to Option B).**
> Rationale: The runtime already has clear “prompt submission” and “interrupt” semantics in its control model (`RuntimeSessionController.send_prompt()` / `.interrupt()` in `src/gig_agents/agents/brain_launch_runtime/runtime.py:120`) and CAO backend (`src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:746`). Adding wakeups/timers/restart in the same change expands ambiguity around readiness/reconciliation before the gateway contract is proven; starting small keeps the first integration slice implementable and testable while still achieving the core “serialize terminal mutation” goal (`openspec/changes/add-agent-gateway-sidecar/design.md:233`).
>
> **DECISION: Modify PROPOSED to make the v1 public request kinds exactly `submit_prompt` and `interrupt`, and keep `payload_json` as an internal serialized form of typed per-kind payloads rather than the public contract.**
> Rationale: Those are the only request semantics the runtime already owns cleanly today, so they give the first queueing slice a concrete execution target without overextending the API. Timers, wakeup nudges, and restart or recovery actions still belong in the design, but they should remain internal scheduler or recovery behavior for now instead of additional externally submitted request kinds.

### Q6) How will “fail fast if bind fails” be enforced while still running the gateway as a background same-window sidecar?

<!-- Classification: mark every question -->
- Decision: `Blocking`

#### Why this matters

- The change requires: if the resolved host/port cannot be bound, “the agent session is not started” (design: `openspec/changes/add-agent-gateway-sidecar/design.md:115`; agent-gateway spec: `openspec/changes/add-agent-gateway-sidecar/specs/agent-gateway/spec.md:54`).
- The design’s conceptual wrapper backgrounds the sidecar and then `exec`s the TUI, but that shape by itself cannot guarantee bind success before the foreground takes over.

#### Examples

- Free-port selection race: runtime resolves a free port, but another process binds it before the gateway does.
- The gateway process crashes on startup; the TUI still starts if we don’t gate.

#### Why this question arises in code

- There is no existing “sidecar + readiness gate” pattern in the runtime launch paths today (e.g., CAO creates the terminal after tmux env is set: `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1504`).

#### Options

**Option A: Wrapper starts gateway, waits for readiness, then execs TUI (recommended)**

- Code impact:
  - Implement a small launcher wrapper that:
    1) starts the gateway process,
    2) polls a local readiness condition (`/health` or a pidfile/socket) with a short timeout,
    3) exits non-zero (and cleans up) if readiness fails,
    4) only then `exec`s the TUI.
- User-facing difference:
  - Startup may take a small additional delay, but failure is clean and deterministic.

**Option B: Runtime pre-binds the socket and passes it to the gateway**

- Code impact:
  - More complex process plumbing (FD passing) but avoids free-port race.
- User-facing difference:
  - Strongest correctness, higher implementation complexity.

> **PROPOSED: Use a wrapper that blocks TUI startup on gateway readiness (Option A), and treat free-port selection as “best effort” with deterministic failure on race.**
> Rationale: This matches the explicit requirement that bind failure prevents the session from launching (`openspec/changes/add-agent-gateway-sidecar/design.md:115`) without introducing FD-passing complexity into the first implementation slice. It also aligns with the repo’s existing “fail explicitly with actionable error” style in tmux/session startup paths (e.g. tmux env pointer validation in `src/gig_agents/agents/brain_launch_runtime/runtime.py:868`).
>
> **DECISION: Accept the readiness-gated wrapper approach, with readiness defined as successful listener bind plus a positive local health check before the foreground TUI `exec`s.**
> Rationale: This is the smallest mechanism that actually satisfies the requirement that bind failure prevents session start while preserving the same-window sidecar model. A pidfile or child-process spawn alone is not enough; the wrapper should clean up the sidecar on timeout or failed readiness and only hand off to the TUI once the resolved listener is verifiably up.
