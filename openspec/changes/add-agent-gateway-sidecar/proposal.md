## Why

Current multi-agent control in this project is centered on direct runtime commands, direct tmux interaction, and backend-specific status surfaces. That is workable for simple operator-driven flows, but it does not give each live agent a durable per-agent control plane for admission, queueing, wakeups, heartbeats, and crash recovery.

We need that control plane now because agent sessions are becoming longer-lived and more collaborative. A per-agent gateway lets the system add policy, scheduling, and recovery without making CAO the architectural center, while still preserving a key feature of this system: humans can interact with the live TUI directly and the system remains coherent.

## What Changes

- Add an optional per-agent gateway companion that introduces no visible tmux pane or window, can be auto-attached at session start, and can also attach later to a running tmux-backed session without restarting the managed agent.
- Define a secret-free gateway attach contract, discoverable from tmux session environment, so newly started runtime-owned tmux-backed sessions become gateway-capable by default through both `AGENTSYS_GATEWAY_ATTACH_PATH` and `AGENTSYS_GATEWAY_ROOT`, while future manual launchers can opt into the same contract later.
- Resolve one effective gateway listener host and port when a gateway instance is started, with the host defaulting to `127.0.0.1` and optionally switching to `0.0.0.0` when explicitly configured, and with the port resolved from lifecycle CLI overrides, `AGENTSYS_AGENT_GATEWAY_PORT`, optional blueprint configuration, or a system-selected free port when no explicit setting is supplied; once a gateway instance starts successfully, that resolved listener becomes the desired listener reused on later restarts unless explicitly overridden.
- Define a minimal HTTP gateway API, implemented by a FastAPI gateway companion process bound by default on `127.0.0.1:<port>` and optionally on `0.0.0.0:<port>` when explicitly configured, with explicit `GET /health`, `GET /v1/status`, and `POST /v1/requests` routes; v1 public request kinds are limited to `submit_prompt` and `interrupt`.
- Add a durable gateway state and queue store under the runtime root so concurrent senders can interact with one managed agent without racing through raw tmux input, and so restart or attach-later flows can recover from stable per-agent state; runtime-owned capability publication seeds `state.json` with an offline or not-attached snapshot before the first live gateway attach.
- Normalize backend-specific readiness and health signals into gateway-owned status axes such as gateway health, agent state, active work, queue depth, last heartbeat, and recovery state.
- Define gateway behavior so direct human interaction with the managed TUI remains a supported feature rather than a protocol violation; the gateway must tolerate and reconcile operator-driven TUI changes instead of assuming exclusive terminal ownership.
- Add recovery expectations for agent crash, stalled work, first attach to an already-running session, and gateway restart, while explicitly scoping full tmux-session loss to an outer supervisor layer rather than the gateway process itself.
- Fail fast when a requested gateway listener cannot be bound; port conflicts do not trigger silent port reselection and instead prevent that gateway instance from starting.
- Keep gateway v1 independent from the mailbox system; mailbox-triggered submission, mailbox-aware policy, and mailbox-specific bindings are deferred until the mailbox contract is mature enough for a follow-up integration.
- Keep the gateway attach contract future-backend-extensible, publish it for runtime-owned tmux-backed sessions, and scope the first live attach implementation to runtime-owned `cao_rest` sessions while leaving manual launchers and other backends as follow-up adopters of the same contract.

## Capabilities

### New Capabilities
- `agent-gateway`: Per-agent FastAPI HTTP gateway companion with a stable attach contract, resolved bind host and port per running gateway instance, a stable versioned `state.json` / `GET /v1/status` contract, durable queueing and logging, normalized status reporting, permission and priority policy, and recovery behavior that tolerates direct human TUI interaction.

### Modified Capabilities
- `brain-launch-runtime`: Session startup, resume, and control flows gain default gateway-capability publication for runtime-owned tmux-backed sessions, optional launch-time auto-attach, explicit attach or detach lifecycle surfaces, gateway runtime-root storage, stable attach metadata publication, live gateway env publication while a gateway is running, and gateway-aware request or status surfaces for attached sessions that do not auto-attach a gateway implicitly.
- `agent-identity`: Tmux-backed agent identity resolution gains gateway attach metadata and live gateway env discovery so control tooling can distinguish gateway-capable sessions from sessions with a currently running gateway.
- `component-agent-construction`: Optional agent blueprints gain a secret-free gateway configuration surface that can declare default gateway host and port values for sessions launched from that blueprint without implicitly enabling the gateway.

## Impact

This change is expected to affect runtime session startup and shutdown, tmux session environment publication, persisted runtime state under the runtime root, agent control surfaces, operator workflows, optional blueprint configuration, and local recovery behavior. It will introduce a secret-free gateway attach contract for runtime-owned tmux-backed sessions, default publication of `AGENTSYS_GATEWAY_ATTACH_PATH` plus `AGENTSYS_GATEWAY_ROOT` for newly started runtime-owned tmux sessions, runtime-owned gateway roots keyed by the runtime-generated session id in v1, an optional FastAPI HTTP gateway companion for attached sessions with resolved bind host and port per live gateway instance, gateway-local durable queue state plus append-only event logging, launch-time auto-attach plus explicit attach or detach lifecycle controls, optional blueprint-declared gateway host and port defaults that do not auto-attach the gateway by themselves, a stable `state.json` plus `GET /v1/status` contract seeded offline before first attach, persisted desired listener reuse across gateway restarts unless explicitly overridden, new live gateway env bindings for currently attached sessions, backend-to-gateway status normalization over CAO signals in the first live attach implementation, explicit attach failure when the requested gateway listener conflicts with another process, and explicit failure for gateway-aware control commands when no live gateway is attached. This revision does not require mailbox enablement, mailbox env bindings, mailbox-triggered workflows, same-window gateway injection, or manual-session gateway support in v1, but it intentionally leaves room for manual launchers to adopt the same attach contract later.
