## Summary

- The change is pointed at the right failure mode: both current parsers still scan the whole tail as a bag of booleans (`claude_code_shadow.py:427-498`, `codex_shadow.py:384-447`), so stale history can pollute active-surface classification.
- Keeping Rx/runtime consumers and dialog projection out of scope matches the existing architecture well (`design.md:7`, `design.md:76-80`, `shadow_parser_core.py:77-112`).
- The current boundary rule is internally contradictory with the existing `working + freeform` contract and tests; as written, it would throw away the still-open prompt line whenever the newest anchor is a spinner/progress line.
- The shared `SnapshotSignalSet` shape leaks Claude-specific blocked-surface names into core even though Codex uses different provider-native concepts today.
- The proposal says `_detect_output_variant()` changes, but the design/spec/tasks never define what that change is or how it should be verified.

## Findings

### Must Fix / Blocking

1. **The proposed prompt-boundary rule cannot represent the repository's existing `working + freeform` surfaces.** `design.md:31-32` and `cursor-anchored-signal-extraction/spec.md:42-45` say the bottommost spinner/progress line becomes the `prompt_boundary_index`, and `cursor-anchored-signal-extraction/spec.md:83-93` says non-idle anchors produce no `active_prompt_payload`. That contradicts both provider specs, which explicitly require `working` snapshots to remain `input_mode = freeform` when a generic prompt is still open (`specs/cao-claude-code-output-extraction/spec.md:31-32`, `specs/cao-codex-output-extraction/spec.md:30-31`), and it contradicts the existing unit tests built around `prompt-above-spinner` layouts (`tests/unit/agents/realm_controller/test_claude_code_shadow_parser.py:167-186`, `tests/unit/agents/realm_controller/test_codex_shadow_parser.py:209-232`). With the design as written, snapshots like `❯ summarize repo` followed by `✽ Razzmatazzing…` or `› summarize repo` followed by `• Thinking ...` lose the prompt line from the active zone and can only classify as `working + closed|unknown`.

## Design Alignment With Existing System

- Existing pattern(s) observed: `shadow_parser_core.py:20-24` and `shadow_parser_core.py:46-87` keep the shared core provider-agnostic, while provider parsers own their detailed heuristics and vocabularies. Claude owns spinner/prompt/trust detection in `claude_code_shadow.py:631-799`; Codex owns its separate prompt/progress/approval/login heuristics in `codex_shadow.py:573-742`. Dialog projection remains full-snapshot and independent from state classification in both providers (`claude_code_shadow.py:511-551`, `codex_shadow.py:460-500`).
- Proposed design alignment: keeping Rx/runtime and dialog projection untouched fits the existing layering well. The main mismatch is that D2/D5 push provider-specific blocked-state names and a regex-only anchor API down into the shared core, while the current codebase keeps those semantics provider-owned. A more native approach would keep shared core responsibilities to zone geometry and generic signal plumbing, then let each provider resolve its richer anchor/blocking semantics on top.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What should define the start of the active zone when processing is visible but the prompt is still typeable?

- Decision: `Blocking`

#### Why this matters

- This is the main place where the current design contradicts the existing parser contract. If the active zone starts at the spinner line, the change breaks `working + freeform`.

#### Examples

- Claude working-freeform snapshot:

```text
❯ summarize repo
✽ Razzmatazzing…
```

- Codex working-freeform snapshot:

```text
› summarize repo
• Thinking about files (2s • esc to interrupt)
```

#### Why this question arises in code

- Current prompt recovery scans backward for the latest prompt payload (`claude_code_shadow.py:697-706`, `codex_shadow.py:607-612`), and the tests assert that working snapshots can still be freeform (`tests/unit/agents/realm_controller/test_claude_code_shadow_parser.py:167-186`, `tests/unit/agents/realm_controller/test_codex_shadow_parser.py:209-232`).
- The design currently changes that rule to "spinner is the boundary" (`design.md:31-32`, `cursor-anchored-signal-extraction/spec.md:42-45`), which excludes the prompt line above it.

#### Options

**Option A: Keep the spinner as the boundary, but add a second upward search for the still-open prompt.**

- Code impact:
  - `find_prompt_boundary()` remains simple.
  - `_extract_signals()` needs a second local search from the spinner/menu anchor back toward the nearest compatible prompt line before deriving `active_prompt_payload` and `has_idle_prompt`.
- User-facing difference:
  - Preserves `working + freeform`, but the stored `prompt_boundary_index` still points at the spinner rather than at the actual start of the active interaction block.
- Pros:
  - Minimal change to the current design text.
  - Keeps spinner/menu lines as explicit anchor hits.
- Cons:
  - Creates two competing "starts" of the active surface.
  - Makes `prompt_boundary_index` less diagnostic because it is no longer where the freeform prompt actually begins.

**Option B: Redefine the boundary as the start of the latest active interaction block, with spinner/progress lines treated as evidence inside that block rather than the block boundary.**

- Code impact:
  - Boundary selection must prefer the latest prompt/block-start anchor that still owns the visible spinner/menu state.
  - `active_prompt_payload` stays anchored to the active prompt line when one exists.
  - Spinner/progress remains a signal inside `SnapshotSignalSet`, not necessarily the boundary-defining anchor.
- User-facing difference:
  - Working surfaces with an open prompt continue to report `input_mode = freeform` without special-case repair logic.
- Pros:
  - Matches the current provider specs and tests directly.
  - Makes `prompt_boundary_index` describe the actual start of the current active surface.
- Cons:
  - The boundary finder becomes more semantic than "first regex hit from the bottom."
  - Requires more precise wording in the design/specs.

> **PROPOSED: Use Option B and define the boundary as the start of the latest active interaction block, not simply the bottommost anchor line.**
> Rationale: The repository already treats `working` and `input_mode` as coexisting axes rather than mutually exclusive states (`docs/developer/tui-parsing/claude.md:108`, `docs/developer/tui-parsing/codex.md:111-128`). The current working-freeform tests in both providers require the prompt line above a spinner/progress line to stay part of the active surface. Option B keeps `prompt_boundary_index` meaningful, preserves current contracts, and avoids adding a second hidden "effective prompt boundary" rule on top of the spec's visible boundary.
>

### Q2) How much provider-specific signal vocabulary should the shared core own?

- Decision: `Blocking`

#### Why this matters

- The change introduces a new shared data model in `shadow_parser_core.py`. If that shared shape bakes in Claude-specific blocked states or assumes regex-only anchor matchers, the Codex implementation will be forced into awkward field overloading and the core module will stop being provider-neutral.

#### Examples

- Claude currently distinguishes trust/setup surfaces:

```text
Allow Claude to work in this folder? [y/n]
Complete setup to continue
```

- Codex currently distinguishes approval/login surfaces:

```text
Approve this command? [y/n]
Sign in to continue in browser
```

#### Why this question arises in code

- Shared core contracts today are intentionally generic (`shadow_parser_core.py:20-24`, `shadow_parser_core.py:46-87`).
- Claude and Codex already diverge in the actual heuristics they need:
  - Claude blocked detection and prompt detection: `claude_code_shadow.py:631-746`
  - Codex progress detection and blocked detection: `codex_shadow.py:573-684`

#### Options

**Option A: Keep one fully shared `SnapshotSignalSet` with fixed booleans like `has_trust_prompt` / `has_setup_block`, and keep `find_prompt_boundary()` on raw regex lists only.**

- Code impact:
  - Faster to implement.
  - Codex must map approval/login semantics into Claude-shaped field names, or add undocumented side channels.
- User-facing difference:
  - No intentional external API change, but internal reasoning/debugging terms become misleading for Codex.
- Pros:
  - Lowest artifact churn right now.
  - Keeps the core type very explicit.
- Cons:
  - Misaligns with current provider-specific parser logic.
  - Makes the shared core semantically biased toward one provider.
  - Regex-only matching is less expressive than the current block-level/menu heuristics.

**Option B: Keep the shared core limited to zone geometry and provider-neutral common signals, and model provider-specific blocked/anchor semantics through generic enums or provider-owned extension logic.**

- Code impact:
  - `SnapshotSignalSet` would carry shared fields such as zone slices, prompt payload, slash/error/processing flags, and perhaps a generic `blocked_surface_kind` / `anchor_kind`.
  - Provider parsers can still own nuanced heuristics such as Codex progress detection or menu-block recognition.
- User-facing difference:
  - No external contract change, but internal diagnostics stay truthful across both providers.
- Pros:
  - Matches the existing repo pattern where the shared core is generic and provider parsers own provider-specific semantics.
  - Avoids Claude-specific field names in Codex code.
  - Leaves room for richer anchor matchers than a flat regex list.
- Cons:
  - Slightly more design work now.
  - Requires clearer documentation for which fields are shared vs provider-owned.

> **PROPOSED: Use Option B and keep `shadow_parser_core.py` provider-neutral, with provider-specific blocked/anchor semantics represented through generic kinds or provider-owned extension logic.**
> Rationale: The current architecture already follows that split: shared types in `shadow_parser_core.py` are generic, while `ClaudeCodeShadowParser` and `CodexShadowParser` each own their own heuristics and provider vocabularies. Option B preserves that pattern and avoids forcing Codex approval/login semantics into Claude-specific field names such as `has_trust_prompt` and `has_setup_block`.
>

> Rationale: _To be filled by developer._
