## Summary

- **Strong architectural clarity**: the summary/detail state split, execution-centric headless model, and explicit adapter-kind gateway refactor are well-reasoned and well-aligned with existing codebase patterns (the `GatewayAttachContractV1` discriminated union already models headless, cao, and houmao-server backend metadata variants).
- **Design decisions are thoughtful and alternatives are documented**: each decision includes a rejected alternative with justification, which makes the reasoning verifiable.
- **Good non-goals discipline**: explicitly excluding ping-pong orchestration, `/v1/mail/*` proxy, and `/turns` elimination keeps the scope focused on infrastructure contracts.
- **Risk: the gateway adapter refactor is the hardest task**: `CaoGatewayAdapter` is currently a concrete class with no protocol/ABC and hardcoded REST-backed assumptions throughout. Splitting it into three adapters will require a protocol definition and substantial refactoring of `GatewayServiceRuntime` (~1400 lines).
- **Risk: no error contract specified for the new managed-agent request API**: the spec defines `submit_prompt` and `interrupt` but does not specify error response shapes, admission-rejection semantics, or idempotency behavior.
- **Risk: tasks may underestimate the gateway adapter refactoring effort** — task 3.1 is a single line item for what could be the largest and most complex code change in the entire plan.

## Findings

### Must Fix / Blocking

1. **The managed-agent request spec does not define error or admission-rejection response models.** The `houmao-server-agent-api` spec says the request API accepts `submit_prompt` and `interrupt` and returns "enough metadata to relate the accepted request to the created headless turn." But it does not specify what happens when: (a) the agent is already running a turn and cannot accept a new prompt, (b) the agent is unavailable or degraded, (c) the request payload is malformed, or (d) an interrupt targets an agent with no active turn. The existing `HoumaoManagedAgentActionResponse` model (`src/houmao/server/models.py`) uses a `success: bool` + `detail: str` pattern, but the spec does not commit to or extend that pattern for the new request surface. Without a defined admission-rejection or error contract, implementers will have to invent one and consumers (especially the gateway server-managed-agent adapter, task 3.2) will not have a stable contract to program against.

> **DECISION: Accept.**
> Rationale: This is a real contract gap. The new request route is intended to become the official callback surface for server-managed execution, so it needs explicit validation and admission semantics rather than implementation-defined behavior. A later artifact revision should add concrete rejected/unavailable/conflict scenarios and align them with the existing server and gateway HTTP status patterns instead of leaving the gateway adapter to infer failure modes ad hoc.

2. **The gateway adapter refactor (task 3.1) has no corresponding spec requirement defining the adapter protocol.** The `agent-gateway` delta spec adds requirements for three adapter kinds and their execution semantics, but does not define the common adapter interface — what methods it must expose, what its error contract is, or how the `GatewayServiceRuntime` selects and instantiates adapters. Currently `CaoGatewayAdapter` is a concrete class (not implementing any protocol) with methods `submit_prompt`, `interrupt`, `read_current_terminal_id`, and `inspect_connectivity` (`gateway_service.py:118-211`). The spec should either define the adapter protocol shape or explicitly state that the adapter protocol is an implementation detail below the spec boundary.

> **DECISION: Reject.**
> Rationale: The externally visible contract here is adapter behavior, selection inputs, and error semantics, not the Python typing shape of the adapter boundary. The spec should stay at that level. Whether implementation uses a `Protocol`, an ABC, or another internal boundary belongs in design and implementation, not in the normative spec text.

### Should Fix / Important

3. **The headless detail spec lists "active turn metadata" and "last-turn detail" fields but does not reference the existing `HoumaoManagedAgentTurnView` and `HoumaoManagedAgentLastTurnView` models.** The existing summary state already has `turn: HoumaoManagedAgentTurnView` and `last_turn: HoumaoManagedAgentLastTurnView` (`models.py:479-498`). The headless detail spec (`managed-agent-detailed-state/spec.md:31-40`) independently lists similar fields (active turn metadata, last-turn detail including turn identity and terminal result evidence). It should clarify whether the detail payload reuses, extends, or replaces those existing coarse turn models. Without this, two similar but incompatible turn representations could emerge — one in the summary and one in the detail.

> **DECISION: Accept.**
> Rationale: The review is right that the relationship between summary turn fields and detailed headless turn fields should be explicit. The intended direction is reuse and extension, not a second incompatible turn vocabulary. A later artifact revision should say that detail state carries the same coarse `turn` and `last_turn` concepts and adds headless-specific supporting fields around them.

4. **The `brain-launch-runtime` delta spec says runtime SHALL publish attach metadata "by default for newly started runtime-owned tmux-backed sessions" but does not address existing sessions.** Lines 12-13 of the delta spec say the runtime publishes by default for new sessions, but existing sessions started before this change will not have headless attach metadata. The migration plan (design.md:183-191) says the change is additive but does not say whether re-start or resume of existing sessions will retroactively gain headless attach metadata. The existing `ensure_gateway_capability` in `runtime.py:425` already runs on start and resume for tmux-backed sessions — if it's extended to cover headless backends, existing session resumes would get the new metadata naturally. The spec should confirm this is the intended behavior.

> **DECISION: Refine.**
> Rationale: The concern is valid, but it is better framed as a resume-behavior clarification than as a missing migration mechanism. The intended behavior is that existing runtime-owned tmux-backed sessions gain the new attach metadata on resume once `ensure_gateway_capability` covers the implemented headless backends. A later artifact revision should say that explicitly instead of implying a one-time migration or restart-only path.

5. **The server-managed gateway routes spec does not specify whether `POST /houmao/agents/{agent_ref}/gateway/attach` is idempotent or what happens on double-attach.** The existing gateway model uses epoch-based reconciliation (`gateway_service.py` uses `m_current_epoch` to detect stale requests). The spec should clarify: if a gateway is already attached and a second attach is requested, does it return the current gateway status, fail with a conflict error, or detach-then-reattach? This matters because the gateway is a companion process — double-attaching could start a second process or conflict with the existing one.

> **DECISION: Refine.**
> Rationale: This should be clarified at the spec level, but the right answer is narrower than "define any double-attach behavior." The intended contract should be idempotent attach when a healthy live gateway is already attached for the same managed agent, with explicit conflict or reconciliation semantics only when the live state is inconsistent or stale. That keeps automation safe without forcing detach-then-reattach as the normal path.

6. **Task granularity is uneven: task 3.1 ("Refactor gateway execution around explicit adapter types") is substantially larger than other tasks.** The current `CaoGatewayAdapter` has REST-specific logic coupled to `CaoRestClient` in its constructor, `read_current_terminal_id`, `inspect_connectivity`, `submit_prompt`, and `interrupt`. Extracting a protocol, creating three implementations, updating `GatewayServiceRuntime` to select adapters based on `GatewayAttachContractV1.backend`, and updating tests, is easily 2-3x the effort of most other individual tasks. Consider splitting 3.1 into: (a) define and extract the adapter protocol from the existing concrete class, (b) refactor `GatewayServiceRuntime` to use the protocol, (c) verify existing REST-backed behavior still passes.

> **DECISION: Accept.**
> Rationale: This is a good task-planning correction. The adapter refactor is the highest-risk implementation slice in the change, and splitting it into protocol extraction, runtime selection plumbing, and REST-path verification will make the implementation sequence safer and easier to review.

7. **The `managed-agent-detailed-state` spec does not specify how "structured diagnostics" are shaped.** It requires "structured diagnostics for degraded or unavailable conditions" (spec line 40) but does not reference the existing `HoumaoErrorDetail` model already used in `HoumaoManagedAgentStateResponse.diagnostics` (`models.py:498`). The detail route should either reuse `HoumaoErrorDetail` or define its own diagnostic shape. If it reuses the existing model, the spec should say so.

> **DECISION: Accept.**
> Rationale: The detail route should not invent a parallel diagnostics model unless it has a strong reason to do so. Reusing the existing `HoumaoErrorDetail` family is the consistent direction and should be stated explicitly in a later artifact revision.

## Design Alignment With Existing System

- **Gateway attach metadata model**: The existing `GatewayAttachContractV1` (`gateway_models.py:174-254`) already has a discriminated union of backend metadata types including `GatewayAttachBackendMetadataHeadlessV1`, `GatewayAttachBackendMetadataCaoV1`, and `GatewayAttachBackendMetadataHoumaoServerV1`. The model validator at line 222 already maps backend kinds to metadata types, including headless backends. This is well-aligned with the proposed adapter-kind refactoring — the attach contract already carries the information needed to select the correct adapter. The design correctly builds on this existing foundation.

- **Gateway execution pattern**: The current `CaoGatewayAdapter` (`gateway_service.py:118-211`) is a concrete class, not a protocol. It hardcodes REST-client construction in `__init__` and uses `CaoRestClient` for both `cao_rest` and `houmao_server_rest` backends. The design proposes three adapter kinds. The existing code structure supports this but will require extracting a common protocol (at minimum: `submit_prompt`, `interrupt`, `read_current_terminal_id`, `inspect_connectivity`). The `GatewayServiceRuntime` class (`gateway_service.py:213+`, ~1200 lines) currently constructs a `CaoGatewayAdapter` in its `__init__` — this coupling will need to be broken.

- **Server model patterns**: Existing managed-agent models follow `_HoumaoModel` base class with Pydantic v2. Response models pair an identity object with state fields. The proposed summary additions (gateway/mailbox posture fields on `HoumaoManagedAgentStateResponse`) and new detail route fit this pattern cleanly. New models for request payloads should follow the same `_HoumaoModel` base.

- **Runtime gateway publish flow**: `runtime.py:ensure_gateway_capability` already calls `gateway_storage.ensure_gateway_capability` and `publish_stable_gateway_env` for tmux-backed sessions. It currently guards behind `_is_tmux_backed()` which returns `True` for headless backends (they use tmux under the hood). The `GatewayCapabilityPublication` dataclass in `gateway_storage.py` already accepts `BackendKind` and `backend_state: GatewayJsonObject`. This means publishing headless attach metadata is mostly a matter of constructing the correct `GatewayAttachBackendMetadataHeadlessV1` payload and ensuring `ensure_gateway_capability` populates it — which aligns well with the proposed spec.

- **Server route naming**: Existing routes follow `/houmao/agents/{agent_ref}/...` naming. The proposed new routes (`/state/detail`, `/requests`, `/gateway/attach`, etc.) follow the same pattern. The gateway sub-routes use a `/gateway/` segment which is a clean namespace extension.

- **`houmao-srv-ctrl` CLI pattern**: The CLI uses Click with `@click.group` and `add_command`. New commands for gateway and state operations would follow the same pattern in `src/houmao/srv_ctrl/commands/`.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the managed-agent request API return an accepted-request envelope with a request ID, or directly return the headless turn response?

- Decision: `Blocking`

#### Why this matters

- The request API is the callback target for gateway server-managed-agent adapters. If the gateway receives an accepted-request envelope, it can track request lifecycle independently of turn lifecycle. If it receives a headless turn directly, the gateway must treat turn-id as the request-id, which couples request and turn semantics.

#### Examples

- A `submit_prompt` request is accepted but turn creation is asynchronous (e.g., the server queues it before admission). An envelope with a request ID allows the caller to poll request status, while a direct turn response would force synchronous turn creation.
- An `interrupt` request has no corresponding turn. With an envelope, both request kinds return the same shape; without it, interrupt returns a different shape than prompt.

#### Why this question arises in code

- The existing `POST /houmao/agents/{agent_ref}/turns` returns `HoumaoHeadlessTurnAcceptedResponse` directly (`app.py:330-335`), which is a turn-specific shape (contains `turn_id`, `turn_index`, `turn_artifact_dir`). The new request API needs to decide whether it wraps this in a transport-neutral request envelope or passes it through.
- The gateway `_execute_request` in `gateway_service.py:957-1018` currently does not inspect the response from `submit_prompt` — it only checks for exceptions. If the request API returns an envelope, the gateway adapter will need to parse it for status/linkage.

#### Options

**Option A: Return a transport-neutral request envelope with a request ID, embedding turn linkage as optional metadata.**

- Code impact:
  - New `HoumaoManagedAgentRequestAcceptedResponse` model with `request_id`, `request_kind`, `status`, and optional `headless_turn_linkage` field.
  - Both `submit_prompt` and `interrupt` return the same response shape.
  ```python
  class HoumaoManagedAgentRequestAcceptedResponse(_HoumaoModel):
      request_id: str
      request_kind: str
      status: Literal["accepted", "rejected"]
      headless_turn_linkage: HoumaoHeadlessTurnLinkage | None = None
  ```
- User-facing difference:
  - Uniform shape for all request kinds; callers parse one model.
  - Turn details are optional metadata, not the primary response.
- Pros:
  - Decouples request lifecycle from turn lifecycle.
  - Uniform shape for gateway adapter parsing.
  - Extensible for future request kinds without shape changes.
- Cons:
  - More complex than a direct pass-through.
  - Callers who only care about the turn must unwrap the envelope.

**Option B: Return transport-specific responses directly (headless turn response for prompts, action response for interrupts).**

- Code impact:
  - Reuse existing `HoumaoHeadlessTurnAcceptedResponse` for prompt requests, `HoumaoManagedAgentActionResponse` for interrupt.
  - The route handler returns a `Union` type or uses separate sub-routes.
  ```python
  # /requests returns different shapes per request_kind
  response: HoumaoHeadlessTurnAcceptedResponse | HoumaoManagedAgentActionResponse
  ```
- User-facing difference:
  - Callers must switch on request kind to parse the response.
  - Turn details are the primary response for prompts.
- Pros:
  - Simpler, reuses existing models.
  - No new abstraction layer.
- Cons:
  - Union response shapes complicate client code and OpenAPI schema.
  - Couples request surface to transport-specific response models.
  - Gateway adapter must handle multiple response shapes.

> **PROPOSED: Option A — return a transport-neutral request envelope with optional turn linkage.**
> Rationale: The design (decision 3) explicitly says "the gateway needs one official callback surface" and "server-owned authority should remain centralized." A uniform request envelope preserves that centralization. The existing `HoumaoManagedAgentActionResponse` already uses a `success` + `detail` pattern that is envelope-like. Option A makes the request surface stable for future request kinds (the spec already says "at minimum" for `submit_prompt` and `interrupt`). The gateway server-managed-agent adapter (task 3.2) would benefit from a single response shape to parse.
>
> **DECISION: Adopt Option A.**
> Rationale: A transport-neutral accepted-request envelope is the right long-term shape for `/houmao/agents/{agent_ref}/requests`, even if this change does not yet add a durable server-side request-status resource. It keeps the callback surface uniform for `submit_prompt` and `interrupt`, lets headless responses carry optional turn linkage rather than exposing turn-specific models directly, and gives the gateway adapter one response family to handle. A request ID can still serve as correlation metadata even before a later `GET /requests/{id}` exists.

### Q2) Should the gateway adapter protocol be defined as a Python `Protocol` (structural subtyping) or as an `ABC` (nominal subtyping)?

- Decision: `Deferrable`

#### Why this matters

- The codebase already uses both patterns: `GatewayMailboxAdapter` in `gateway_mailbox.py` is a `Protocol`, while other patterns use concrete classes. The choice affects how implementations are validated and how type-checking works.

#### Examples

- If `Protocol`, adapters are structurally typed — any class with `submit_prompt`, `interrupt`, etc. methods matches, without explicit inheritance.
- If `ABC`, adapters must explicitly inherit and implement all abstract methods, getting IDE support for missing implementations.

#### Why this question arises in code

- `gateway_mailbox.py` already defines `GatewayMailboxAdapter` as a `Protocol` — there's precedent for protocols in the gateway subsystem.
- The existing `CaoGatewayAdapter` is a plain class with no inheritance.
- `mypy --strict` requires either approach to be fully typed.

#### Options

**Option A: Python `Protocol` (structural subtyping).**

- Code impact:
  - Define `GatewayExecutionAdapter(Protocol)` with typed method signatures.
  - Existing `CaoGatewayAdapter` works without modification (already has matching methods).
- User-facing difference:
  - None — implementation detail.
- Pros:
  - Consistent with `GatewayMailboxAdapter` precedent in the same subsystem.
  - No inheritance coupling.
  - Existing concrete class already matches.
- Cons:
  - Missing methods caught at call site, not at class definition.

**Option B: ABC (nominal subtyping).**

- Code impact:
  - Define `GatewayExecutionAdapter(ABC)` with abstract methods.
  - `CaoGatewayAdapter` must be changed to inherit from it.
- User-facing difference:
  - None — implementation detail.
- Pros:
  - Missing methods caught at class definition.
  - Clearer intent in IDE.
- Cons:
  - Breaks from `Protocol` precedent in the gateway subsystem.
  - Adds inheritance coupling.

> **PROPOSED: Option A — use `Protocol`, consistent with the existing `GatewayMailboxAdapter` pattern.**
> Rationale: The gateway subsystem already uses `Protocol` for its mailbox adapter boundary (`gateway_mailbox.py`). Using the same typing approach for execution adapters keeps the subsystem internally consistent. The existing `CaoGatewayAdapter` already has the right method signatures, so a `Protocol` definition would retroactively type-check it without requiring inheritance changes. `mypy --strict` supports both patterns equivalently.
>
> **DECISION: Adopt Option A.**
> Rationale: The gateway subsystem already has clear precedent for `Protocol`-based boundaries in `GatewayMailboxAdapter`, and the broader repo also uses `Protocol` for behavioral interfaces such as `InteractiveSession`. Using a `Protocol` for execution adapters keeps the typing style consistent and avoids turning the public design into a nominal class hierarchy commitment. This remains an implementation decision rather than a spec requirement.

### Q3) Should the managed-agent detail route for TUI agents embed the full canonical TUI terminal-state payload or return a reference/URL to the canonical route plus a summarized projection?

- Decision: `Blocking`

#### Why this matters

- This is one of the three open questions from `design.md:202`. The answer affects response payload size, client complexity, and whether the detail route is self-contained.

#### Examples

- Full embed: `GET /houmao/agents/{agent_ref}/state/detail` returns a `TuiDetailPayload` containing all fields from the existing terminal-keyed TUI route. One request gets everything.
- Reference + summary: the response contains a `tui_canonical_route` URL plus a trimmed projection (e.g., operator status, parser state name, connection status). Two requests needed for full TUI detail.

#### Why this question arises in code

- The existing TUI state route (`GET /cao/terminals/{terminal_id}/state/live`, exposed by CAO compatibility layer) returns a rich parsed-surface payload. Embedding it in the managed-agent detail response couples the managed-agent model to the TUI parsing model.
- The design (decision 7) says "project or embed" but does not commit to one.

#### Options

**Option A: Embed a projection — include key TUI state fields in the detail response, not the full canonical payload.**

- Code impact:
  - `TuiDetailPayload` model with selected fields (operator status, parser state, connection info, summary of parsed surface).
  - Server `service.py` reads TUI tracking state and projects it into the detail model.
- User-facing difference:
  - Detail route is self-contained for common queries. Full canonical detail still available through the TUI-specific route.
- Pros:
  - Self-contained response for discovery/orchestration use cases.
  - Avoids coupling managed-agent models to the full TUI parsing model.
  - Keeps payload size reasonable.
- Cons:
  - Two models representing TUI state (canonical and projected) — drift risk.

**Option B: Full embed — return the entire canonical TUI terminal-state payload inside the detail response.**

- Code impact:
  - Reuse the existing TUI state response model directly inside the transport-discriminated detail payload.
- User-facing difference:
  - Detail route returns everything; no need for a second request.
- Pros:
  - No projection logic; no drift risk.
  - One request for complete information.
- Cons:
  - Couples managed-agent models to TUI parsing internals.
  - Larger payload; may include parser-specific fields irrelevant to orchestration callers.

> **PROPOSED: Option A — embed a curated projection of key TUI state fields.**
> Rationale: The design (decision 7) says the managed-agent detail route should "unify discovery, not fork the TUI model." A projection achieves this: it gives orchestration callers the key fields they need (operator status, availability, connection) without coupling the managed-agent model to the full TUI parsing contract (which includes prompt-area content, parser state machines, and surface classification). The canonical TUI route remains the authoritative deep-inspection surface, consistent with design decision 7. The projection can reference the canonical route URL so callers know where to go for full detail.
>
> **DECISION: Adopt Option A.**
> Rationale: The managed-agent detail route should be self-contained for common discovery and orchestration use, but it should not embed the entire canonical TUI payload and thereby couple managed-agent models to every parser-facing TUI field. The right v1 contract is a curated TUI projection plus a pointer to the canonical detailed route, which in the current server surface is the existing `/houmao/terminals/{terminal_id}/state` family.

### Q4) Should the managed-agent request API define a durable request-status resource from the start?

- Decision: `Deferrable`

#### Why this matters

- This is the first open question from `design.md:200`. If a durable request-status resource exists, callers can poll `GET /houmao/agents/{agent_ref}/requests/{request_id}` to track request lifecycle. Without it, the accepted response is fire-and-forget.

#### Examples

- With durable status: a gateway adapter submits a prompt, receives request ID `req-123`, and can later poll to check if it completed, failed, or is still running.
- Without: the gateway adapter relies on headless turn polling (`GET /turns/{turn_id}`) to understand what happened.

#### Why this question arises in code

- The gateway already has its own durable request queue (SQLite in `gateway_service.py`). Adding a second request-status resource on the server side could create redundant state tracking.
- The headless turn surface (`/turns/{turn_id}`) already provides durable per-turn status. For headless agents, request status and turn status overlap significantly.

#### Options

**Option A: Accept-only in this change; defer durable request status.**

- Code impact:
  - `POST /requests` returns accepted/rejected with optional turn linkage. No `GET /requests/{id}` route.
  - Callers use turn-specific routes for headless detail.
- User-facing difference:
  - Simpler API surface. Callers must know to use `/turns` for headless follow-up.
- Pros:
  - Avoids redundant state tracking between gateway queue and server request store.
  - Keeps the change focused on the infrastructure contracts.
  - Can be added later if a use case demands it.
- Cons:
  - Asymmetry: TUI callers have no follow-up surface after request acceptance.

**Option B: Include a durable `GET /requests/{id}` from the start.**

- Code impact:
  - New server-side request store, new route, new status model.
  - More implementation work in this already large change.
- User-facing difference:
  - Uniform request lifecycle tracking for all transports.
- Pros:
  - Complete request lifecycle from day one.
  - Uniform for TUI and headless.
- Cons:
  - Adds significant scope and a new persistence concern.
  - Risk of redundancy with gateway queue and headless turn store.

> **PROPOSED: Option A — defer durable request status to a follow-on change.**
> Rationale: This change already has 15 tasks across 5 major groups. Adding a server-side request store would increase scope materially. The headless turn surface (`/turns/{turn_id}`) already covers the primary follow-up use case for headless agents. The gateway has its own durable queue for request tracking. Deferring is low-risk because the request API can be extended with `GET /requests/{id}` later without breaking the `POST /requests` contract.

> **DECISION: Adopt Option A.**
> Rationale: Durable server-side request status is valuable, but it is a separate persistence concern and would materially widen the change. For this change, the accepted-request envelope plus optional headless turn linkage is enough. Headless callers can use `/turns/{turn_id}` for durable follow-up, and the gateway already has its own queue-level request tracking. A later `GET /requests/{id}` can be added without breaking the accepted-request contract.
