## Summary

- **Solid:** The proposal clearly articulates the discovery gap (cross-root, cross-project agent lookup) and the design preserves the existing authority boundary — registry is a locator layer with secret-free pointers, not a state replacement. This is well-motivated and architecturally sound.
- **Solid:** Design decisions are thorough with alternatives considered and rejected for concrete reasons. The lease-based freshness model, atomic replace semantics, and per-agent hashed directories are all reasonable choices that avoid shared mutable coordination.
- **Solid:** The delta spec for `brain-launch-runtime` correctly scopes registry publication to the runtime lifecycle layer (not backends), aligning with how `RuntimeSessionController` already owns manifest persistence, gateway capability publication, and mailbox binding refresh.
- **Risky:** Several critical implementation parameters are left unspecified: hash algorithm, lease duration defaults, refresh intervals, grace period bounds, record schema shape, and error model. These gaps could lead to inconsistent implementations or require follow-up design changes.
- **Risky:** The module placement for new registry code is unspecified. The existing codebase has clear layering (`brain_builder.py` for build phase, `realm_controller/` for runtime, `realm_controller/backends/` for backend-specific code), and the registry code needs explicit placement guidance.
- **Risky:** Two open questions from the proposal (CLI exposure and non-tmux session scope) remain unresolved and will affect task scope and API surface.

## Findings

### Must Fix / Blocking

1. **Hash algorithm for `agent-key` is unspecified.** The design and spec both say "deterministic hash of the globally unique published agent name" but never specify the algorithm. This is a blocking interoperability gap: if two implementations (or a future version) use different hashes, records become unreachable. The hash function, truncation length, and encoding (hex vs base64url) must be pinned in the spec.

2. **Lease duration and refresh interval have no defaults.** The design makes freshness lease-based and central to the trust model (readers validate expiry, cleanup tooling uses grace periods), but neither the design nor the spec defines: (a) default lease TTL, (b) recommended refresh interval, or (c) the grace period bound for cleanup. Without these, the `agent-discovery-registry` spec's scenarios ("lease has expired", "expired beyond the cleanup grace period") are not testable against concrete values.

3. **Record schema shape is unspecified.** The design lists the conceptual fields (agent_name, agent_key, generation_id, timestamps, pointers) and the spec says "one strict versioned schema," but no JSON schema, field names, or version identifier is provided. Task 1.1 mentions "versioned `record.json` payloads" — this needs a concrete schema before implementation can begin, or the task description must explicitly delegate schema design to the implementer with constraints.

### Should Fix / Important

4. **Module placement for registry code is unspecified.** The existing codebase has a clear module hierarchy: `src/houmao/agents/brain_builder.py` (build phase), `src/houmao/agents/realm_controller/` (runtime phase with `runtime.py`, `launch_plan.py`, `gateway_storage.py`, `agent_identity.py`, `mailbox_runtime_support.py`). The design mentions "registry helpers alongside the current runtime-owned manifest and gateway helpers" but doesn't specify whether these live in a new `src/houmao/agents/realm_controller/registry.py` or `registry_storage.py`, a new subpackage `src/houmao/agents/registry/`, or elsewhere. Given the gateway precedent (`gateway_storage.py` in `realm_controller/`), guidance here avoids implementer guesswork.

5. **Error model for registry operations is undefined.** The design says publication "will fail" on duplicate fresh generation and resolution returns "explicit not-found or stale-record result," but no error types, exception hierarchy, or result-type contract is specified. Existing patterns in the codebase use `SessionControlResult` / `GatewayControlResult` dataclasses with status enums for control operations. The registry should follow a consistent pattern.

6. **`platformdirs` usage scope is ambiguous.** The design says "resolve home via `platformdirs`-aware path handling" but the actual intent is just `Path.home()` or `platformdirs.user_home_path()` — not a full `platformdirs` state/data/cache directory. The dependency exists in `pyproject.toml` (line 20: `"platformdirs>=4.9.4,<5"`) but is currently unused in runtime code. The spec should clarify whether this means `platformdirs.user_home_path()` specifically, or just Python's `Path.home()`, since both achieve the same effect for the stated goal and `platformdirs` would be a new runtime import.

7. **`generation_id` semantics need tightening.** The design says it "changes per process start or per publication generation" — these are different things. Per-process-start means a UUID generated once when the session starts; per-publication-generation means it could change on every refresh. The duplicate-detection logic ("different fresh generation already owns the same agent name") depends on which interpretation is used. The spec should pin one interpretation.

8. **The delta spec for `brain-launch-runtime` doesn't reference integration points with the existing 94KB main spec.** The main spec at `openspec/specs/brain-launch-runtime/spec.md` is substantial. The delta adds ADDED requirements but doesn't indicate where these requirements fit relative to existing ones (e.g., after which requirement section, or as a new capability subsection). This makes merging ambiguous.

9. **No scenario for concurrent publication race.** The spec covers "fresh duplicate publication for one agent name is rejected" but doesn't address the race where two publishers both check-then-write simultaneously. The atomic replace prevents partial reads, but two publishers could both see "no fresh record" and both attempt to publish. A scenario clarifying the expected outcome (last-writer-wins after the check window, or filesystem-level locking) would strengthen the contract.

### Nice To Have

10. **Logging and observability for registry operations are not mentioned.** The existing codebase uses standard `logging` throughout `runtime.py` and `gateway_storage.py`. Registry publication, refresh, cleanup, and resolution should emit structured log messages for debuggability, especially since stale-record scenarios will be common operational events.

11. **Performance bounds for cleanup scanning are unaddressed.** Cleanup tooling must scan all `live_agents/` directories. If many agents are published over time with crashes, this directory could grow. A note on expected scale or an optimization (e.g., batch cleanup, age-based directory naming) would be useful.

12. **The two open questions from the proposal should be resolved or explicitly deferred in the design.** The proposal asks whether v1 exposes operator-facing CLI commands and whether non-tmux sessions publish. The design's non-goals section partially addresses the second (non-tmux) but doesn't make a definitive call, and the tasks don't include CLI work. Making these deferral decisions explicit would prevent scope creep during implementation.

### Proposed design alignment

- **Aligns well:** The design correctly places registry publication at the `RuntimeSessionController` level, matching how gateway and mailbox publication are already integrated. The pointer-only, secret-free approach matches `gateway_storage.py`'s `AttachContract` pattern.

- **Suggestion — follow `gateway_storage.py` module placement:** Place registry storage/helpers in `src/houmao/agents/realm_controller/registry_storage.py` (or `registry.py`), co-located with `gateway_storage.py` and `agent_identity.py`. This keeps all runtime-phase publication logic in the same package.
- **Mismatch to watch:** The design mentions `RuntimeSessionController` integration but doesn't mention `start_runtime_session()` (the factory function in `runtime.py` that creates controllers). Publication on session start likely belongs in that factory or in the controller's post-construction initialization, not in the constructor itself.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What hash algorithm, truncation, and encoding should be used for `agent-key` directory names?

- Decision: `Blocking`

#### Why this matters

- The hash algorithm determines directory name length, collision probability, and cross-implementation interoperability. If two codepaths or future versions use different algorithms, records become orphaned.

#### Examples

- Agent name `AGENTSYS-gpu` → `agent-key` could be `8f6c1f9d5a4c...` (SHA-256 hex, truncated to 12 chars) or `jGwfnVpM...` (SHA-256 base64url, truncated to 12 chars) or the full 64-char hex digest.
- Very long agent names (e.g., `AGENTSYS-my-very-long-project-specific-agent-name-for-ci`) must produce directory names within filesystem limits.

#### Why this question arises in code

- `gateway_storage.py` uses deterministic path derivation from session roots (not hashed). The registry introduces hashing as a new pattern with no precedent in the codebase.
- The spec says "deterministic hash" but the implementation needs a pinned algorithm to ensure all publishers and readers agree on the same directory.

#### Options

**Option A: SHA-256, hex-encoded, truncated to first 16 characters**

- Code impact:
  - ```python
    import hashlib
    agent_key = hashlib.sha256(canonical_name.encode()).hexdigest()[:16]
    ```
- User-facing difference:
  - Short directory names like `8f6c1f9d5a4c3b2e/` — easy to inspect manually.
- Pros:
  - Short, human-scannable directory names.
  - 16 hex chars = 64 bits of entropy, collision probability negligible for expected agent counts (< 1000).
- Cons:
  - Theoretical collision risk at very large scale (birthday bound ~4 billion).
  - Cannot recover agent name from hash alone (need `record.json` for reverse lookup).

**Option B: SHA-256, full 64-character hex digest**

- Code impact:
  - ```python
    agent_key = hashlib.sha256(canonical_name.encode()).hexdigest()
    ```
- User-facing difference:
  - Long directory names like `8f6c1f9d5a4c3b2e7d8a9f0b1c2d3e4f5a6b7c8d9e0f1a2b3c4d5e6f7a8b9c0d/` — harder to scan but zero collision risk.
- Pros:
  - Zero practical collision risk.
  - Standard, no truncation decisions needed.
- Cons:
  - Unwieldy directory names for manual inspection.

**Option C: SHA-256, hex-encoded, truncated to first 24 characters**

- Code impact:
  - ```python
    agent_key = hashlib.sha256(canonical_name.encode()).hexdigest()[:24]
    ```
- User-facing difference:
  - Medium-length directory names balancing inspectability and collision resistance.
- Pros:
  - 96 bits of entropy — effectively zero collision risk for any practical scale.
  - Reasonably compact for manual inspection.
- Cons:
  - Still an arbitrary truncation choice.

> **PROPOSED: Option A — SHA-256, hex-encoded, truncated to 16 characters.**
> Rationale: The expected agent population per user is in the low hundreds at most, making 64 bits of hash entropy more than sufficient. Short directory names improve operator debuggability when inspecting `~/.houmao/registry/live_agents/` manually, which aligns with the design's stated goal of making the registry "easy to inspect, debug, and clean manually" (design.md, Decision 1 rationale). The `record.json` inside each directory already stores the full `agent_name` and `agent_key`, providing a reverse-lookup path when needed. If future scale requires it, a version bump to the record schema can lengthen the hash without breaking existing readers (who validate `agent_name` inside the record, not just the directory name).
>

### Q2) What should the default lease TTL and refresh interval be?

- Decision: `Blocking`

#### Why this matters

- The lease TTL determines how quickly a crashed agent becomes invisible to readers. Too short and healthy agents risk appearing stale between refreshes; too long and crashed agents remain discoverable for an extended period.
- The refresh interval must be shorter than the TTL to maintain freshness, but frequent refreshes add filesystem I/O.

#### Examples

- A 5-minute lease with 2-minute refresh: agent crash is detected within 5 minutes, but healthy agents write to disk every 2 minutes.
- A 30-minute lease with 10-minute refresh: longer visibility window for crashes, but less filesystem churn.

#### Why this question arises in code

- The spec requires readers to validate lease freshness and cleanup tooling to use a grace period, but no concrete durations are specified.
- The runtime lifecycle in `runtime.py` already has natural refresh points (gateway attach/detach, mailbox refresh) that could serve as implicit refresh triggers, but there's no periodic heartbeat mechanism.

#### Options

**Option A: 10-minute lease TTL, refresh on every lifecycle event only (no periodic heartbeat)**

- Code impact:
  - Record published with `lease_expires_at = now + 10min` on every lifecycle event (start, resume, gateway attach/detach, mailbox refresh, prompt).
  - No background refresh thread needed.
- User-facing difference:
  - Agents with no activity for >10 minutes appear stale even if the tmux session is alive.
- Pros:
  - Simple — no background threads or timers.
  - Natural fit with existing lifecycle hooks in `RuntimeSessionController`.
- Cons:
  - Idle agents (no prompts, no gateway changes) go stale. This may be acceptable for v1 since discovery is primarily for active collaboration.

**Option B: 30-minute lease TTL, refresh on lifecycle events, with optional periodic heartbeat**

- Code impact:
  - Longer TTL gives more breathing room for idle sessions.
  - Optional background heartbeat (e.g., via a simple timer in RuntimeSessionController) can be added later without changing the contract.
- User-facing difference:
  - Crashed agents visible for up to 30 minutes, but idle agents stay discoverable longer.
- Pros:
  - More forgiving for idle-but-alive sessions.
  - Leaves room for a periodic heartbeat to be added without contract changes.
- Cons:
  - Crashed agents linger longer in discovery.

> **PROPOSED: Option A — 10-minute lease TTL, lifecycle-event-only refresh for v1.**
> Rationale: The runtime already has well-defined lifecycle hooks in `RuntimeSessionController` (start, resume, gateway attach/detach, mailbox refresh, stop) that naturally trigger publication. Adding a periodic heartbeat would require a background thread or async timer — complexity not justified for v1 where the primary use case is active agent-to-agent collaboration. The 10-minute window is short enough to limit stale-crash visibility while being long enough to survive brief periods between lifecycle events. If idle-agent discovery becomes important, a future change can add a heartbeat and extend the TTL without breaking the record contract.
>

> Rationale: _To be filled by developer._

### Q4) What should `generation_id` represent — per-process-start or per-publication-event?

- Decision: `Blocking`

#### Why this matters

- The duplicate-detection contract depends on `generation_id` semantics. If `generation_id` changes per publication refresh, then every refresh looks like a "new generation" and duplicate detection becomes meaningless. If it's per-process-start, it correctly identifies "same live process" vs "replacement process."

#### Examples

- Agent `AGENTSYS-gpu` starts (gen=`abc123`), publishes record, refreshes 5 times → all refreshes have gen=`abc123`.
- Agent crashes, new instance starts (gen=`def456`) → publication sees gen=`abc123` is stale (expired lease) or different, proceeds.
- If gen changed per-refresh: gen=`abc123`, then `abc124`, then `abc125`... → duplicate detection can't distinguish "same process refreshing" from "new process replacing."

#### Why this question arises in code

- Design says generation_id "changes per process start or per publication generation" — these are contradictory options presented as equivalent.
- The duplicate-detection scenario in the spec ("different `generation_id`") only makes sense if generation is stable within one session lifecycle.

#### Options

**Option A: `generation_id` is a UUID generated once per session start, stable across all refreshes**

- Code impact:
  - ```python
    generation_id = str(uuid.uuid4())  # generated once in RuntimeSessionController
    ```
  - Stored as a field on the controller, passed to every registry publish/refresh call.
- User-facing difference:
  - Duplicate detection works correctly: same gen = same live process, different gen = different process.
- Pros:
  - Clean semantics for duplicate detection.
  - Simple to implement — one UUID per session lifecycle.
- Cons:
  - None significant.

**Option B: `generation_id` is a monotonic counter or timestamp that changes per publication**

- Code impact:
  - Each publish/refresh increments or regenerates the generation.
- User-facing difference:
  - Every refresh looks like a "new generation."
- Pros:
  - Could detect stale-but-not-expired records from the same process.
- Cons:
  - Breaks duplicate-detection contract as designed.
  - Adds complexity with no clear benefit.

> **PROPOSED: Option A — `generation_id` is a UUID generated once per session start.**
> Rationale: The design's duplicate-detection contract ("if a fresh record already exists for the same `agent_name` but a different `generation_id`, publication will fail") only makes sense when `generation_id` is stable within one session lifecycle. This matches the natural lifecycle boundary already managed by `RuntimeSessionController` in `src/houmao/agents/realm_controller/runtime.py` — the controller is constructed once per session start/resume and persists for the session's lifetime. A per-start UUID stored on the controller gives clear ownership semantics and enables the designed duplicate-rejection behavior.
>

> Rationale: _To be filled by developer._

### Q6) How should concurrent publication races be handled when two publishers both pass the duplicate check?

- Decision: `Deferrable`

#### Why this matters

- The spec says "if a fresh record already exists for a different `generation_id`, publication SHALL reject." But with filesystem-based check-then-write, two publishers can both see "no fresh record" and both proceed to write. Atomic replace (`os.replace()`) means one wins, but the loser doesn't know it lost.

#### Examples

- Publisher A checks: no record. Publisher B checks: no record. Both write atomically. Publisher B's `os.replace()` silently overwrites A's record. A believes it published successfully but is no longer the discoverable owner.

#### Why this question arises in code

- `gateway_storage.py` uses atomic writes (`_atomic_json_write` pattern) but doesn't face this exact race because gateway state is per-session-root, not shared across processes.
- The registry is the first shared-across-processes writable surface in the system.

#### Options

**Option A: Accept last-writer-wins within a small race window, rely on lease refresh to self-heal**

- Code impact:
  - Check-then-write with atomic replace. No file locking.
  - On the next refresh cycle, the "losing" publisher re-checks and detects the conflict.
- User-facing difference:
  - Brief window where the wrong publisher is discoverable. Self-heals on next refresh.
- Pros:
  - Simple, no locking complexity.
  - The race window is tiny (milliseconds between check and write).
  - Practical risk is low: two agents with the same canonical name starting simultaneously is an operator error.
- Cons:
  - Not strictly serializable.

**Option B: Use advisory file locking (`fcntl.flock`) around the check-and-write sequence**

- Code impact:
  - Lock the agent-key directory or a lockfile before check+write.
  - Adds OS-level locking dependency.
- User-facing difference:
  - Strictly serialized publication — no race window.
- Pros:
  - Correct under all concurrent scenarios.
- Cons:
  - NFS and some network filesystems don't honor `flock` reliably.
  - Adds complexity and potential deadlock risk.
  - Overkill for the expected deployment model (single-user, local filesystem).

> **PROPOSED: Option A — accept last-writer-wins within the race window, rely on lease refresh to self-heal.**
> Rationale: The registry lives on a local per-user filesystem (`~/.houmao/registry`), not a network share. Two different agents claiming the same `AGENTSYS-<name>` identity simultaneously is an operator configuration error, not a normal operating condition. The design already requires publishers to check for fresh records before writing (design.md, Decision 5), which handles the common case. The remaining race window (both check simultaneously) is milliseconds wide. Adding file locking would introduce complexity and platform-specific failure modes (`fcntl.flock` behavior varies on NFS) disproportionate to the risk. If the race does occur, the next refresh cycle (within the lease TTL) will detect the conflict and surface it.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._
