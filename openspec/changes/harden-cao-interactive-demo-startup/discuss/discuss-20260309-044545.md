# Discussion: Open Questions for Interactive CAO Demo Startup Hardening

- Feature/Change: `harden-cao-interactive-demo-startup`
- Timestamp: `2026-03-09T04:45:45Z`
- Context: This doc captures the unresolved decisions I found while reviewing the active change artifacts plus the current interactive demo and CAO launcher code. The goal is to make the startup-hardening scope explicit before implementation work begins.

The change already has a strong direction: make the demo runnable by default, from normal shells, without manual CAO cleanup or launcher-home overrides. The open questions below are the places where that intent still needs sharper decisions.

---

## Summary of recommended defaults

1. Default the interactive demo to create a per-run git worktree at `<repo-root>/tmp/demo/<what-demo>/<ts>/wktree` and use `<repo-root>/tmp/demo/<what-demo>/<ts>` as the CAO trusted home. See Q1.
2. Keep repo-root-anchored defaults scoped to this demo rather than broadening shared runtime-wide default behavior. See Q2.
3. On `127.0.0.1:9889`, prompt before replacing an existing CAO server, and make `-y` a consistent yes-to-all flag across all demo scripts. See Q3.
4. Keep fixed-port recycle semantics inside this demo; do not change the launcher as part of this work. See Q4.
5. Reset `AGENTSYS-alice` to a clean slate before start: kill any existing session and clean leftover demo state/artifacts so startup behaves like a new run. See Q5.

## Q1) What should the default CAO trusted home be for the interactive demo?

- Decision: `Blocking`

### Why this matters

- The current default wrapper picks a launcher home that is incompatible with the default repo workdir, which directly caused startup failure in a real run.
- The trusted home choice also determines where CAO writes `HOME/.aws/cli-agent-orchestrator/...`, so it affects repo cleanliness and operator surprise.

### Examples

- A developer runs `scripts/demo/cao-interactive-full-pipeline-demo/launch_alice.sh` from this repo and expects the default workdir to be the repo root. If launcher home stays under the workspace, CAO rejects the workdir.
- A developer runs the same demo repeatedly and later notices `.aws/cli-agent-orchestrator/` inside the repo root. That is acceptable only if we explicitly choose that behavior.

### Why this question arises in code

- The interactive wrapper currently defaults:
  - `WORKSPACE_ROOT="$REPO_ROOT/tmp/cao_interactive_full_pipeline_demo"`
  - `LAUNCHER_HOME_DIR="${CAO_LAUNCHER_HOME_DIR:-$WORKSPACE_ROOT}"`
  - `--workdir "$REPO_ROOT"`
  - Source: `scripts/demo/cao-interactive-full-pipeline-demo/run_demo.sh:6-9,62-69`
- That means the default trusted home can exclude the default workdir.
- An older local CAO demo already uses a broader home:
  - `CAO_LAUNCHER_HOME_DIR="${CAO_LAUNCHER_HOME_DIR:-$(dirname "$REPO_ROOT")}"`
  - Source: `scripts/demo/cao-claude-esc-interrupt/run_demo.sh:13-15`

### Options

**Option A: Default launcher home to repo root**

- Code impact:
  - Change the wrapper default and the Python demo default to use `repo_root`.
- User-facing difference:
  - The demo works by default and CAO writes its launcher-managed state under the repo tree.
- Pros:
  - Simple mental model: everything is rooted in this checkout.
  - Guaranteed to contain the repo workdir.
- Cons:
  - Leaves repo-local CAO state under `.aws/`.
  - Can make the repo feel “dirtier” during tutorial use.

**Option B: Default launcher home to the parent of repo root**

- Code impact:
  - Resolve `launcher_home_dir` to `repo_root.parent` when the user does not override it.
- User-facing difference:
  - The demo still works by default, but CAO state lands outside the repo while still trusting the repo tree.
- Pros:
  - Compatible with the repo workdir.
  - Avoids repo-local `.aws` churn.
  - Matches an existing CAO demo pattern already in the repo.
- Cons:
  - Slightly less obvious than “just use repo root.”
  - Uses a broader trust boundary than the repo itself.

**Option C: Keep launcher home under the demo workspace**

- Code impact:
  - No change from current interactive wrapper behavior.
- User-facing difference:
  - The demo continues to fail unless the user manually overrides `CAO_LAUNCHER_HOME_DIR`.
- Pros:
  - Keeps CAO state tightly scoped under the demo workspace.
- Cons:
  - Known-bad with the current default workdir.
  - Violates the “easy to run” tutorial goal.

### Proposal (Recommended)

Use the parent of repo root as the default launcher home for the interactive demo. It preserves the easy-run behavior by containing the repo workdir, avoids repo-local `.aws` state by default, and matches prior demo precedent already used elsewhere in `scripts/demo/`.

### Pros / Cons (Proposal)

- Pros:
  - Fixes the workdir rejection problem with a default-only change.
  - Avoids surprising repo-local CAO state.
  - Reuses an established pattern already present in another CAO demo.
- Cons:
  - Slightly broadens the trusted home beyond the repo root.
  - Needs clear documentation so the default does not feel “magical.”

> **DECISION: Default the interactive demo to create a git worktree at `<repo-root>/tmp/demo/<what-demo>/<ts>/wktree` and use `<repo-root>/tmp/demo/<what-demo>/<ts>` as the CAO trusted home.**
> Rationale: The current interactive wrapper defaults the launcher home to the demo workspace while also passing the repository root as `--workdir`, which produces an incompatible home/workdir pair (`scripts/demo/cao-interactive-full-pipeline-demo/run_demo.sh:6-9,62-69`). Using a timestamped demo root under `tmp/demo/` gives the demo a dedicated CAO home directory that contains the effective workdir via the nested worktree path, avoids writing launcher-managed state into the main repository root, and keeps tutorial artifacts scoped under a per-run directory instead of a shared global location.

## Q2) Should repo-root-anchored defaults be a wrapper-only behavior or an engine-level contract?

- Decision: `Blocking`

### Why this matters

- If repo-root anchoring lives only in `run_demo.sh`, the wrapper works but the Python demo CLI still behaves differently when invoked directly.
- The tasks already imply engine-level behavior, but the current parser defaults remain cwd-sensitive.

### Examples

- A developer runs `scripts/demo/cao-interactive-full-pipeline-demo/launch_alice.sh` from a shell outside the repo. The wrapper can still find repo root and pass explicit flags.
- A maintainer later runs `pixi run python -m gig_agents.demo.cao_interactive_full_pipeline_demo start` without all path flags and gets cwd-based defaults instead of repo-based defaults.

### Why this question arises in code

- The wrapper always computes repo-root-derived values before invoking Python:
  - `git -C "$SCRIPT_DIR" rev-parse --show-toplevel`
  - Source: `scripts/demo/cao-interactive-full-pipeline-demo/run_demo.sh:4-9`
- But the Python CLI parser still defaults everything from `Path.cwd()`:
  - `--repo-root`, `--workspace-root`, `--agent-def-dir`, `--launcher-home-dir`, `--workdir`
  - Source: `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:720-748`

### Options

**Option A: Wrapper-only anchoring**

- Code impact:
  - Fix shell defaults only; leave Python parser defaults as `Path.cwd()`.
- User-facing difference:
  - Wrapper flow is reliable, but direct CLI invocation remains context-sensitive.
- Pros:
  - Smallest code change.
  - Keeps the Python module generic.
- Cons:
  - Two different default models for the same demo.
  - Easier for future maintenance drift to reintroduce the bug.

**Option B: Engine-level anchoring**

- Code impact:

```python
repo_root = resolve_repo_root(args.repo_root)
workspace_root = args.workspace_root or repo_root / "tmp" / "cao_interactive_full_pipeline_demo"
agent_def_dir = args.agent_def_dir or repo_root / "tests" / "fixtures" / "agents"
workdir = args.workdir or repo_root
```

- User-facing difference:
  - Both wrapper-driven and lower-level interactive demo entrypoints behave consistently.
- Pros:
  - One coherent default model.
  - More robust against future wrapper drift.
- Cons:
  - Requires a slightly more deliberate parsing model to distinguish omitted defaults from explicit values.
  - Makes the interactive demo module more repo-aware.

### Proposal (Recommended)

Make repo-root anchoring an engine-level contract for this interactive demo module. The wrapper should remain the main UX, but the Python engine should not secretly fall back to a different cwd-driven path model.

### Pros / Cons (Proposal)

- Pros:
  - Aligns the wrapper and engine behavior.
  - Matches the change intent already captured in `tasks.md`.
  - Reduces the number of places future maintainers need to keep in sync.
- Cons:
  - Slightly more refactoring than a shell-only fix.
  - Needs careful handling of explicit overrides versus omitted defaults.

> **DECISION: Keep repo-root-anchored defaults scoped to this demo only; do not generalize them into shared runtime-wide default behavior.**
> Rationale: The change scope is explicitly the interactive CAO demo pack, not a general `brain_launch_runtime` contract (`openspec/changes/harden-cao-interactive-demo-startup/proposal.md`, `openspec/changes/harden-cao-interactive-demo-startup/tasks.md`). The current cwd-sensitive defaults that need correction live in the demo wrapper and demo module (`scripts/demo/cao-interactive-full-pipeline-demo/run_demo.sh:4-9,62-69`, `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:720-748`), while shared runtime path resolution continues to operate through generic cwd-based CLI semantics in `src/gig_agents/agents/brain_launch_runtime/cli.py:232-249,313-320`. Constraining the behavior to the demo avoids widening the contract of shared runtime entrypoints that are not part of this startup-hardening change.

## Q3) What should startup do when an existing CAO server is already using `127.0.0.1:9889`?

- Decision: `Blocking`

### Why this matters

- This is the core “replace existing CAO server” question for the demo's fixed local endpoint.
- If we keep reusing any healthy server, the demo can inherit stale CAO `HOME` policy and fail even though the port looks healthy.
- If we replace automatically, we risk interrupting another local workflow without warning.
- The demo also needs a non-interactive path so wrapper-driven automation can still proceed without hanging on prompts.
- The demo exposes several wrapper scripts, so `-y` should be a shared interface contract even on commands that do not currently prompt.

### Examples

- A previous demo run left a healthy `cao-server` on `127.0.0.1:9889` with `HOME` pointing somewhere that does not contain this repo. The next interactive demo run starts against the same server and gets `Working directory not allowed`.
- A developer is using a local CAO server for unrelated work on the same fixed port. The interactive demo should ask before taking over that server.
- CI-style or scripted usage wants the replacement behavior, but without an interactive confirmation prompt. A `-y` flag can make that path explicit.
- A future extension adds a second confirmation step to `stop_demo.sh` or `send_prompt.sh`. If every wrapper already accepts `-y`, that extension does not need a one-off CLI contract.

### Why this question arises in code

- The launcher currently reuses any healthy server on the supported loopback target:
  - `if initial_status.healthy: ... reused_existing_process=True`
  - Source: `src/gig_agents/cao/server_launcher.py:512-529`
- The interactive demo currently retries only when launcher output implies one specific ownership mismatch shape:
  - `reused_existing_process` and `pid is None`
  - Source: `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:900-937`
- The current wrapper scripts do not share a pass-through `-y` surface:
  - `launch_alice.sh` and `stop_demo.sh` currently reject any extra flag, while `send_prompt.sh` only accepts `--prompt <text>`
  - Source: `scripts/demo/cao-interactive-full-pipeline-demo/launch_alice.sh:18-28`, `scripts/demo/cao-interactive-full-pipeline-demo/send_prompt.sh:18-28`, `scripts/demo/cao-interactive-full-pipeline-demo/stop_demo.sh:18-28`

### Options

**Option A: Always trust a healthy local server**

- Code impact:
  - Keep current launcher and demo behavior with minor diagnostics improvements only.
- User-facing difference:
  - Startup may still fail later in session creation with confusing CAO workdir-policy errors.
- Pros:
  - Minimal implementation work.
  - Lowest risk of interfering with other local CAO usage.
- Cons:
  - Does not actually solve the failure we observed.
  - Keeps the demo fragile.

**Option B: Prompt before replacing an existing `cao-server`, with demo-wide `-y` to auto-confirm**

- Code impact:

```python
if existing_verified_cao_server():
    if assume_yes or confirm("Replace existing CAO server on 127.0.0.1:9889?"):
        stop_existing_cao_server()
        start_fresh_cao_server()
    else:
        raise DemoWorkflowError("user declined to replace existing CAO server")
```

- User-facing difference:
  - Interactive runs get a clear safety prompt; scripted runs can opt into non-interactive replacement with `-y`, and every demo wrapper accepts the same flag even if that command has no prompt today.
- Pros:
  - Directly addresses the stale-home problem.
  - Preserves a safe default for developer-facing usage.
  - Still supports easy reruns and automation.
  - Establishes one extension-friendly CLI convention across the demo surface.
- Cons:
  - Slightly more orchestration work than silent replacement.
  - Requires flag plumbing through all wrapper scripts, not just the start path.

**Option C: Always replace an existing `cao-server` without prompting**

- Code impact:
  - Add an unconditional verified-stop-and-restart path.
- User-facing difference:
  - The demo silently takes over the fixed port whenever a `cao-server` is already there.
- Pros:
  - Maximum convenience for repeated tutorial runs.
  - Simplest non-interactive operator experience.
- Cons:
  - Can unexpectedly interrupt another developer's CAO session.
  - Too destructive as the default behavior for a local tutorial tool.

### Proposal (Recommended)

When the demo finds an existing verified `cao-server` on the fixed loopback target, prompt before replacing it and offer `-y` as an explicit “yes to all” override for non-interactive use. Treat `-y` as a demo-wide wrapper contract so `run_demo.sh`, `launch_alice.sh`, `send_prompt.sh`, and `stop_demo.sh` all accept it consistently, even when a specific command has no prompt yet. This fixes the stale-home startup problem while keeping the default developer experience safe and legible.

### Pros / Cons (Proposal)

- Pros:
  - Solves the observed stale-server problem.
  - Gives interactive users an explicit confirmation point before takeover.
  - Supports scripted reruns through a clear `-y` contract.
  - Keeps future prompt additions from fragmenting the wrapper UX.
- Cons:
  - Adds prompt and flag plumbing to the demo UX.
  - Still intentionally disrupts another local CAO workflow once confirmed.

> **DECISION: If an existing CAO server is already using the interactive demo's fixed local target, prompt before killing it, and make `-y` a consistent yes-to-all flag across all demo scripts.**
> Rationale: The tutorial goal is still to make the demo easy to run, but unconditional fixed-port takeover is too destructive for a developer-facing workflow. The current launcher/demo path can leave startup blocked by an already-healthy CAO server on the fixed port (`src/gig_agents/cao/server_launcher.py:512-529`, `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:866-937`), so the demo needs an explicit replacement path. At the same time, the current wrapper surface is inconsistent: `launch_alice.sh` and `stop_demo.sh` reject any extra flags and `send_prompt.sh` only accepts `--prompt <text>` (`scripts/demo/cao-interactive-full-pipeline-demo/launch_alice.sh:18-28`, `scripts/demo/cao-interactive-full-pipeline-demo/send_prompt.sh:18-28`, `scripts/demo/cao-interactive-full-pipeline-demo/stop_demo.sh:18-28`). Standardizing `-y` across `run_demo.sh` and all helper wrappers preserves a safe interactive default, supports non-interactive runs, and avoids having future prompt-bearing extensions invent per-script flag behavior.

### Notes / Follow-ups

- This decision covers replacing an existing verified `cao-server`. The behavior for a non-CAO occupant on the fixed port should either remain an explicit failure or be decided separately if the change later widens scope beyond CAO-only takeover.

## Q4) Should fixed-port recycle semantics live in the interactive demo or in the launcher?

- Decision: `Deferrable`

### Why this matters

- The behavior contract in Q3 must be decided before implementation.
- The code-placement choice determines whether this change stays narrowly scoped to the demo or expands into shared launcher behavior.

### Examples

- A demo-local implementation could perform `status -> prompt -> stop -> start` before launching `alice`.
- A launcher change would introduce new shared semantics for server replacement that other commands and demos might later rely on, even though this change only aims to harden one tutorial path.

### Why this question arises in code

- The interactive demo already wraps launcher `status` / `start` / `stop` through `_ensure_cao_server()` in `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:866-937`.
- The launcher already owns health checks, pid/log artifacts, and verified stop behavior in `src/gig_agents/cao/server_launcher.py:512-710`.

### Options

**Option A: Implement recycle logic in the interactive demo only**

- Code impact:
  - Extend `_ensure_cao_server()` in the demo module.
- User-facing difference:
  - Only this tutorial gets the new replacement behavior.
- Pros:
  - Fastest path to fixing the immediate demo problem.
  - Lowest short-term scope.
- Cons:
  - Pushes more orchestration policy into one demo.
  - Harder to reuse across other CAO demos later.

**Option B: Extend the launcher with explicit replace/recycle behavior**

- Code impact:
  - Add a launcher-level control path or mode that can recycle a verified healthy local server before returning success.
- User-facing difference:
  - The interactive demo can call a more expressive launcher primitive.
- Pros:
  - Cleaner ownership boundary.
  - Reusable for other fixed-port CAO demos.
- Cons:
  - Broader scope than the immediate tutorial fix.
  - Requires updating launcher contract/tests in addition to demo tests.

### Proposal (Recommended)

Keep the recycle and confirmation logic in the interactive demo only. The existing demo module already orchestrates launcher `status` / `start` / `stop`, so this change can stay within the tutorial surface without widening the launcher contract.

### Pros / Cons (Proposal)

- Pros:
  - Keeps the change tightly scoped to the tutorial that is actually broken.
  - Avoids introducing new shared launcher behavior and test surface.
  - Reuses the demo's existing orchestration point in `_ensure_cao_server()`.
- Cons:
  - Leaves similar lifecycle needs in other demos to be solved separately later.
  - Concentrates more policy in the demo module.

> **DECISION: Keep fixed-port recycle semantics in the interactive demo only; do not touch the launcher in this change.**
> Rationale: The scope of `harden-cao-interactive-demo-startup` is to make this specific tutorial demo easy to run, not to broaden shared CAO lifecycle behavior across the repo (`openspec/changes/harden-cao-interactive-demo-startup/proposal.md`, `openspec/changes/harden-cao-interactive-demo-startup/tasks.md`). The interactive demo already owns a purpose-built orchestration path in `_ensure_cao_server()` (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:866-937`) that wraps launcher `status`, `start`, and `stop`, so the replacement prompt and `-y` behavior can be implemented there without modifying the shared launcher code in `src/gig_agents/cao/server_launcher.py:512-710`.

### Notes / Follow-ups

- If other CAO demos later need the same behavior, we can revisit whether to extract a shared launcher-level capability after there is evidence of repeated duplication.

## Q5) How should startup clear stale `AGENTSYS-alice` state?

- Decision: `Blocking`

### Why this matters

- The current startup flow only stops a previous session when `state.json` says it is active.
- Real stale state can exist outside that state file: tmux session still present, manifest pointer stale, CAO session already gone, or local state inactive after a partial failure.
- The startup contract should feel like “start new,” not “hope the previous run cleaned up perfectly.”

### Examples

- A previous run left a tmux session named `AGENTSYS-alice`, but local state is inactive or missing. The next `launch_alice.sh` fails with “agent identity already in use.”
- A user tries to clear `AGENTSYS-alice` with `stop-session --force-cleanup`, but for CAO sessions that flag does not actually give a tmux fallback path.
- A prior run left turn artifacts or a report in the workspace. The next `start` should clear them so verification only reflects the new session.

### Why this question arises in code

- Interactive demo startup only cleans the previously recorded active state:
  - `if previous_state is not None and previous_state.active: ... stop-session`
  - Source: `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:361-372`
- `stop-session --force-cleanup` only configures force cleanup for `HeadlessInteractiveSession`, not for `CaoRestSession`:
  - Source: `src/gig_agents/agents/brain_launch_runtime/runtime.py:117-125`
- Raw tmux cleanup primitives are available:
  - `kill_tmux_session()` in `src/gig_agents/agents/brain_launch_runtime/backends/tmux_runtime.py:71-81`
- The demo already clears turn/report artifacts on replacement start, which is part of the desired fresh-start behavior:
  - `_reset_turn_artifacts(paths)` removes `turns/` and `report.json`
  - Source: `src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:374-376,824-831`

### Options

**Option A: State-file-only cleanup**

- Code impact:
  - Keep the current `previous_state.active` check as the only cleanup path.
- User-facing difference:
  - Stale tmux sessions outside persisted state continue to block startup.
- Pros:
  - Very simple.
- Cons:
  - Known insufficient.

**Option B: Always use `brain_launch_runtime stop-session --agent-identity AGENTSYS-alice`**

- Code impact:
  - Add unconditional name-based stop before startup.
- User-facing difference:
  - Better than state-only cleanup, but still vulnerable when name-based manifest resolution is stale.
- Pros:
  - Reuses runtime-owned stop behavior.
  - Keeps logic higher-level than raw tmux management.
- Cons:
  - Can still fail when tmux manifest pointers are missing or stale.
  - Does not guarantee tmux cleanup for CAO-specific stale-state cases.

**Option C: Clean-slate reset: kill the existing session, clean leftovers, then start fresh**

- Code impact:

```python
try:
    stop_session(agent_identity="AGENTSYS-alice")
except ExpectedStaleIdentityErrors:
    pass

if tmux_session_exists("AGENTSYS-alice"):
    kill_tmux_session("AGENTSYS-alice")

reset_demo_local_artifacts_and_state()
```

- User-facing difference:
  - Startup behaves like a fresh run even after partial failures or stale leftovers.
- Pros:
  - Handles both live CAO-backed sessions and stale tmux leftovers.
  - Matches the user expectation that `start` means “start new.”
  - Keeps verification/report artifacts scoped to the new run only.
- Cons:
  - More orchestration logic in the demo.
  - Needs careful cleanup ordering so the new run does not erase diagnostics too early.

### Proposal (Recommended)

Use a clean-slate reset. Before launching `alice`, always try to stop any existing session for the fixed demo identity, then kill leftover tmux state if needed, then clear demo-local artifacts that should not survive into the next run. The effect should be the same as starting from a fresh workspace.

### Pros / Cons (Proposal)

- Pros:
  - Most reliable startup recovery for the fixed tutorial identity.
  - Preserves runtime semantics when they still work, while still cleaning leftovers.
  - Explicitly handles the stale tmux session we already observed.
  - Aligns startup behavior with the mental model of “replace with new.”
- Cons:
  - More complex than state-only or runtime-only cleanup.
  - Needs explicit logging so operators understand what was recycled and what was wiped.

> **DECISION: Before start, reset `AGENTSYS-alice` to a clean slate by killing any existing session and cleaning leftover demo state/artifacts so the run behaves like new.**
> Rationale: The current `start_demo()` path only attempts remote cleanup when `state.json` exists and is still marked active (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:361-372`), which leaves stale identity collisions possible after partial failures. At the same time, the demo already treats replacement starts as a fresh-run boundary by clearing turn artifacts and `report.json` through `_reset_turn_artifacts(paths)` (`src/gig_agents/demo/cao_interactive_full_pipeline_demo.py:374-376,824-831`). Combined with the available tmux cleanup primitive in `kill_tmux_session()` (`src/gig_agents/agents/brain_launch_runtime/backends/tmux_runtime.py:71-81`) and the fact that CAO stop force-cleanup is not generalized through `runtime.py` for `CaoRestSession` (`src/gig_agents/agents/brain_launch_runtime/runtime.py:117-125`), the safest demo-only contract is to treat every replacement start as “kill the old session, clean what is left, then launch fresh.”
