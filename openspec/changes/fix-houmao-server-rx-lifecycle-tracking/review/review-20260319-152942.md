## Summary

- **The problem diagnosis is accurate and well-grounded.** The server's `_SurfaceStateReducer` in `tracking.py` (13 mutable fields, manual `monotonic_ts` arithmetic) is structurally the same design shape that failed in the CAO `_TurnMonitor` and was fixed by the Rx rewrite. The motivating issue correctly identifies both failure directions (false completion from churn, missed fast turns).
- **The core design decisions (D1–D6) are sound.** Shared Rx kernel, continuous-watch vs turn-anchored split, explicit lifecycle authority metadata, deterministic test schedulers — all directly address the root causes and align with lessons from issue-002.
- **However, the "shared kernel" extraction is under-specified in a way that could produce an awkward API or a second kind of drift.** The CAO Rx monitor is fundamentally a terminal-result pipeline (it emits one `CompletedResult` and stops), while the server needs continuous state emissions that never terminate. The design says "separate adapters" but does not show how the kernel's completion pipeline works when it must be restarted or reset per turn anchor while the readiness pipeline runs forever.
- **The turn anchor model has a critical gap: what happens after completion?** Once an anchored turn completes, does the anchor expire? Does the system fall back to unanchored background mode until the next `POST /terminals/{id}/input`? The design and specs are silent on anchor lifecycle after completion, which is exactly where the next batch of false-completion bugs will emerge.
- **The existing `note_prompt_submission` hook is a no-op today** (`service.py:348-352` — both args are discarded). This validates that the anchor-from-input-submission architecture has an existing insertion point, but the tasks don't explicitly call out wiring this existing hook.
- **Task scope is ambitious (12 tasks across 4 groups) for what could be phased more incrementally.** The highest-value immediate fix (suppress unanchored `candidate_complete`/`completed`) does not require the full shared Rx kernel extraction. A phased approach could deliver safety sooner.

## Findings

### Must Fix / Blocking

1. **Anchor lifecycle after completion is unspecified — this is the next false-completion vector.** The design says turn-anchored monitoring "may emit `candidate_complete` and `completed`" but never specifies what happens to the anchor after `completed` fires. If the anchor stays active, the next ready-surface churn after a completed turn will immediately re-enter `candidate_complete` (because `saw_working` / `saw_projection_change` evidence is still accumulated from the prior turn). If the anchor expires, clients lose visibility into whether subsequent activity is related to the same session. The CAO Rx monitor avoids this because it terminates its observable after one `CompletedResult` — the server cannot do that in continuous-watch mode.

   Concretely: in `cao_rx_monitor.py`, `_reduce_completion_event` (line ~760) emits a terminal `CompletedResult` and the subscription ends. In `tracking.py`, the reducer runs forever — `m_cycle_saw_working` and `m_cycle_saw_projection_change` persist until the next `working` transition resets them (line 342-350). After an anchored turn completes, if no new `working` is observed, the old evidence remains hot and the next submit-ready observation will immediately re-enter `candidate_complete`.

   The design must specify anchor expiry semantics: does the anchor auto-expire on `completed`, requiring a new submission to re-arm? Or does it stay armed until explicitly released?

2. **The "shared kernel" must reconcile terminal-result vs continuous-emission semantics before implementation, not during.** The CAO Rx completion pipeline is designed to emit exactly one terminal result (`CompletedResult`, `StalledResult`, `FailedResult`, `BlockedResult`) and then the caller disposes the subscription. The server needs a lifecycle stream that emits state continuously and never terminates. This is not just an adapter concern — it affects whether the kernel's `_reduce_completion_event` reducer can be reused at all, because its `result` field in `_CompletionPipelineState` is a "latch" (once set, it stays set and gets filtered into terminal output).

Either the kernel needs a reset/re-arm mechanism (which changes its current API), or the "shared" code is really just the classification functions and evidence accumulator — not the full pipeline reducer. The design should be explicit about which layer of `cao_rx_monitor.py` is the shared kernel vs which parts remain CAO-specific.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) How should the shared Rx kernel handle completion pipeline reset across multiple turns in continuous-watch mode?

- Decision: `Blocking`

#### Why this matters

The CAO Rx completion pipeline is single-shot: it accumulates evidence from a fixed baseline, emits one `CompletedResult`, and terminates. The server runs forever across many potential turns. The shared kernel must support multi-turn lifecycle without requiring subscription disposal and re-creation per turn.

#### Examples

- Session receives input via `POST /terminals/{terminal_id}/input` → anchor armed → agent works → prompt returns to ready → stability window passes → `completed` emitted. What now?
- Next input arrives 30 seconds later → should evidence from the previous turn carry over or be reset?
- Agent starts working again without a new `POST /input` (e.g., operator typed directly in tmux) → no anchor → should completion monitoring stay armed from the old anchor?

#### Why this question arises in code

- In `cao_rx_monitor.py`, `_CompletionPipelineState.result` is a latch — once set to `CompletedResult`, the pipeline emits it and terminates via `ops.filter(_is_pipeline_result)`. There is no reset path.
- In `tracking.py`, evidence resets on working→non-working transition (lines 342-350), providing implicit turn separation. But this reset is tied to observing `business_state == "working"`, not to an anchor event.
- The shared kernel must decouple turn boundaries from observation-based heuristics and tie them to explicit anchor events instead.

#### Options

**Option A: Anchor-scoped completion pipelines — create a new Rx completion subscription per anchor event, dispose the previous one.**

- Code impact:
  - Server adapter maintains a `Disposable` for the current completion pipeline
  - On anchor event: dispose old subscription, create new pipeline with fresh `_CompletionAccumulatorState` seeded from current baseline
  - Readiness pipeline remains continuous and unaffected
  ```python
  def arm_turn_anchor(self, baseline_projection: str) -> None:
      if self.m_completion_subscription is not None:
          self.m_completion_subscription.dispose()
      self.m_completion_subscription = build_completion_pipeline(
          source=self.m_observation_subject,
          baseline_projection=baseline_projection,
          ...
      ).subscribe(self._on_completion_result)
  ```
- User-facing difference:
  - Clean turn separation; each anchor gets its own evidence accumulation
  - After completion, no stale evidence can trigger false re-completion
- Pros:
  - Reuses the existing CAO kernel completion pipeline exactly as-is (single-shot per subscription)
  - Clean evidence lifecycle — no reset logic needed inside the reducer
- Cons:
  - Subscription churn; must ensure thread-safe disposal
  - Brief gap between disposal and new subscription where completion state is undefined

**Option B: Add a reset/re-arm event type to the shared completion reducer so it can be driven from anchor events without subscription churn.**

- Code impact:
  - Add `_CompletionAnchorResetEvent` to the event union
  - `_reduce_completion_event` handles it by clearing evidence, resetting result latch, setting new baseline
  ```python
  if isinstance(event, _CompletionAnchorResetEvent):
      return _CompletionPipelineState(
          evidence=PostSubmitEvidence(baseline=event.baseline, ...),
          result=None,
      )
  ```
- User-facing difference:
  - Same clean turn separation, but completion state transitions atomically from `completed` back to `inactive` on anchor event
- Pros:
  - No subscription churn; single long-lived completion pipeline
  - Atomic state transitions visible in the continuous stream
- Cons:
  - Modifies the shared kernel's reducer contract (adds a new event type the CAO runtime doesn't use)
  - Must ensure reset events are properly ordered relative to observations

> **PROPOSED: Option A — anchor-scoped completion pipelines.**
> Rationale: Option A reuses the existing `build_completion_pipeline` without modification, keeping the shared kernel's single-shot semantics intact. The server adapter manages subscription lifecycle externally, which is the natural responsibility boundary for the continuous-watch owner. Thread-safe disposal is straightforward with RxPY's `SerialDisposable`. The brief gap between disposal and re-subscription is acceptable because the readiness pipeline (which runs continuously) still provides authoritative state. This avoids expanding the shared kernel's event contract with server-specific reset semantics that the CAO runtime would never exercise — keeping the kernel genuinely shared rather than weighted toward one consumer.
>

### Q2) Should the immediate safety fix (suppress unanchored completion) be decoupled from the full Rx kernel extraction?

- Decision: `Deferrable`

#### Why this matters

The change proposes 12 tasks across 4 groups, coupling the safety fix (stop false completions) with the architectural improvement (shared Rx kernel). The safety fix could be shipped much sooner as a focused change — the server already has all the information it needs to suppress unanchored completion states.

#### Examples

- A minimal patch in `_SurfaceStateReducer._reduce_completion()` that returns `"inactive"` instead of `"candidate_complete"` / `"completed"` when no turn anchor exists would fix the immediate false-completion problem in <50 lines of code.
- The full Rx kernel extraction is a valuable architectural investment but doesn't need to gate the safety fix.

#### Why this question arises in code

- The false completion bug is live now (per the motivating issue: "the server tracker showed both failure directions" during interactive testing).
- The Rx kernel extraction (Tasks 1.1–1.3) is the most complex and risky part of the change — it touches a working subsystem (CAO runtime) and introduces cross-module refactoring.
- Suppressing unanchored completion is a local change to `tracking.py` that could ship independently with the lifecycle authority metadata.

#### Options

**Option A: Ship the full change as designed — Rx kernel + anchor model + server migration together.**

- Code impact:
  - All 12 tasks in one change
- Pros:
  - Complete architectural fix in one pass
  - No intermediate states where server has partial Rx and partial imperative
- Cons:
  - Longer time to safety fix
  - Higher risk per change (touching CAO runtime too)
  - Harder to review and verify

**Option B: Split into two changes — (1) suppress unanchored completion + add lifecycle authority metadata, then (2) Rx kernel extraction as a follow-up.**

- Code impact:
  - Change 1: Tasks 2.1, 2.2, 3.2, 3.4, 4.2 (anchor model + suppression + consumer updates)
  - Change 2: Tasks 1.1, 1.2, 1.3, 3.1, 3.3, 4.1, 4.3 (Rx kernel + server Rx migration)
- Pros:
  - Safety fix ships immediately
  - Rx extraction gets its own verification cycle
  - CAO runtime stays untouched until the Rx kernel is proven
- Cons:
  - Two changes instead of one
  - Server keeps imperative timing for stall/unknown until change 2 ships

> **PROPOSED: Option B — split into a focused safety fix and a follow-up Rx kernel extraction.**
> Rationale: The motivating issue is P1 and the false completion bug is actively impacting interactive testing of the dual shadow-watch demo. The fastest path to safety is suppressing unanchored completion in the existing imperative reducer (adding an anchor flag to `_SurfaceStateReducer` and gating `candidate_complete`/`completed` on it) plus adding `HoumaoLifecycleAuthorityMetadata` to the response model. This can ship in 2-3 tasks. The Rx kernel extraction is valuable but its risk profile (touching `cao_rx_monitor.py` which currently passes 11 deterministic tests) argues for a separate change with its own test verification gate. The imperative stall/unknown timing (which is less buggy than the completion timing) can tolerate waiting for the Rx migration.
>

### Q3) Where should the shared Rx lifecycle kernel module live?

- Decision: `Deferrable`

#### Why this matters

The kernel is consumed by two separate subsystems: `src/houmao/agents/realm_controller/backends/` (CAO runtime) and `src/houmao/server/tui/` (server tracking). Its module location should not create awkward cross-layer imports.

#### Why this question arises in code

- `cao_rx_monitor.py` currently lives at `src/houmao/agents/realm_controller/backends/cao_rx_monitor.py` — deep in the backend tree.
- `tracking.py` lives at `src/houmao/server/tui/tracking.py` — in the server tree.
- A shared kernel under `backends/` would make the server import from a backend-specific path. A shared kernel under `server/` would make the CAO runtime import from a server-specific path.

#### Options

**Option A: `src/houmao/agents/realm_controller/backends/` — adjacent to `cao_rx_monitor.py` with a neutral module name like `rx_lifecycle_kernel.py`.**

- Pros: Minimal file moves; the Rx operator expertise is already in this directory.
- Cons: Server importing from `agents/realm_controller/backends/` is an unusual dependency direction.

**Option B: `src/houmao/lifecycle/` — a new neutral subtree.**

- Pros: Clean dependency graph; neither consumer "owns" the kernel.
- Cons: New top-level package for one module; may be premature.

**Option C: `src/houmao/agents/lifecycle/` — under agents but not under backends.**

- Pros: Acknowledges that lifecycle monitoring is an agent-adjacent concern; avoids deep backend nesting.
- Cons: Server still imports from `agents/`.

> **PROPOSED: Option B — `src/houmao/lifecycle/rx_lifecycle_kernel.py`.**
> Rationale: The kernel is genuinely shared infrastructure, not an agent-specific or server-specific concern. A neutral path avoids the server importing from `agents/realm_controller/backends/` (which would be an architectural layer violation) and avoids the agents importing from `server/`. The directory may start with one module, but lifecycle timing is likely to attract related utilities (observation types, test fixtures, timing constants). This follows the same pattern as `src/houmao/cao/` which exists as neutral CAO integration infrastructure consumed by both server and agents.
