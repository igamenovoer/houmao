## Summary

- **Solid architectural alignment:** The design correctly leverages the existing `DetectorProfileRegistry` version-floor resolution and the Claude Code version-split precedent (`ClaudeCodeSignalDetectorV2_1_X` + `FallbackClaudeDetector`), rather than inventing a parallel mechanism.
- **Well-motivated problem statement:** The coupling between `_CODEX_PLACEHOLDER_TEXTS` (9 hardcoded literals in `ready.py`) and `_normalize_prompt_text()` (dim-SGR regex matching) is real and brittle; the proposal correctly identifies this as the primary risk surface.
- **Conservative degradation policy is strong:** Defaulting to `unknown` instead of manufacturing `editing=yes` from ambiguous evidence directly addresses the sticky-editing failure mode that motivates the change.
- **Underspecified variant contract shape:** The `CodexPromptBehaviorVariant` interface, its input model (`PromptAreaSnapshot`), and its output model are described conceptually but not structurally — the design lacks enough detail for an implementer to produce a first draft without making significant ad-hoc decisions.
- **Open question on `unknown` → `editing_input` mapping required a developer decision:** This is the most user-visible behavioral decision in the change; that decision is now recorded below and still needs to be pushed into the artifacts.
- **Delta spec for `versioned-tui-signal-profiles` is thin:** It introduces "behavior variant delegation" as a concept but doesn't constrain whether variants are registry-visible or profile-private, leaving an architectural seam underspecified.

## Design Alignment With Existing System

- **Existing pattern: version-split detector profiles.** The Claude Code tracker already implements the exact pattern this change proposes for Codex: `ClaudeCodeSignalDetectorV2_1_X` (with `minimum_supported_version=(2, 1, 0)`) plus `FallbackClaudeDetector` (with `minimum_supported_version=None`), both registered via `DetectorProfileRegistration` in `registry.py`. The Codex change follows this pattern correctly — no architectural mismatch.

- **Existing pattern: `DetectedTurnSignals.notes` for debug metadata.** The design plans to surface variant identity and unrecognized-prompt markers through diagnostic notes. This aligns with the existing `notes: tuple[str, ...]` field on `DetectedTurnSignals` and `TrackedStateSnapshot`, which is already used for debugging evidence. No new debug plumbing is needed.

- **Existing pattern: temporal frame storage is profile-private.** The current `_CodexTuiFrame` dataclass is already private to the Codex profile (prefixed with `_`). Adding variant classification to the frame or to per-snapshot metadata fits this encapsulation model.

- **Potential misalignment: variant as a new abstraction layer.** The existing codebase has no "behavior variant" abstraction — detector profiles directly implement all signal logic. Introducing `CodexPromptBehaviorVariant` as a separate composable unit within a profile is a new pattern. While the design justifies it well (isolating drift-prone prompt interpretation), the implementation should be careful not to over-abstract. The Claude Code detector handles all its signal logic in one class without sub-variants. If the Codex variant contract ends up being a single method on a single class, a simpler approach (a classmethod or module-level function selected by version) might be more aligned with the existing codebase's style.

- **Alignment confirmed: `@dataclass(frozen=True)` for internal value objects.** The project convention (per `CLAUDE.md` and `pyproject.toml`) uses frozen dataclasses for internal value objects. The prompt-area snapshot model and prompt classification result should follow this pattern, not Pydantic.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should `CodexPromptBehaviorVariant` be a `Protocol`, an ABC, or a plain callable?

- Decision: `Deferrable`

#### Why this matters

The variant contract is the central new abstraction in this change. Its shape determines how many classes are needed, how they compose with detector profiles, and whether the pattern is reusable for future non-Codex apps that may also need drift-prone region variants.

#### Examples

- If a `Protocol`, any object with a matching `classify(snapshot: PromptAreaSnapshot) -> PromptClassification` method works, including lambdas or module-level functions.
- If an ABC, subclasses must explicitly inherit, which adds ceremony but makes the hierarchy visible in type-checking and IDE navigation.
- If a plain callable (`Callable[[PromptAreaSnapshot], PromptClassification]`), the "variant" is just a function reference stored on the profile — simplest, but least self-documenting.

#### Why this question arises in code

- The existing codebase uses `BaseTrackedTurnSignalDetector` (in `models.py`) as the abstract base for full detector profiles. But behavior variants are narrower — they classify one region, not the full surface.
- The Claude Code detector has no sub-variant abstraction; it handles everything directly. Introducing a new ABC just for Codex prompt classification could feel over-engineered if only one concrete implementation exists initially.
- The design says "behavior-oriented rather than mechanism-oriented" which suggests a Protocol or callable might be more aligned than a rigid class hierarchy.

#### Options

**Option A: `Protocol` with a single `classify` method**

- Code impact:
  - Define `class CodexPromptBehaviorVariant(Protocol)` in a types module under `codex_tui/`.
  - Each variant is a frozen dataclass that implements the protocol.
  - Profile stores `m_prompt_variant: CodexPromptBehaviorVariant`.
- User-facing difference:
  - None (internal implementation detail).
- Pros:
  - Structural typing — any conforming object works without inheritance.
  - Aligns with the "behavior-oriented" design intent.
  - Easy to test with stub implementations.
- Cons:
  - Protocol methods aren't as discoverable via IDE "go to implementations."

**Option B: ABC inheriting from a narrow base**

- Code impact:
  - Define `class CodexPromptBehaviorVariant(abc.ABC)` with `@abstractmethod classify(...)`.
  - Concrete variants inherit explicitly.
- User-facing difference:
  - None.
- Pros:
  - Explicit hierarchy, easy to find all implementations.
  - Consistent with `BaseTrackedTurnSignalDetector` pattern.
- Cons:
  - Heavier than needed for a one-method contract.
  - Might encourage over-engineering if future apps add their own variant ABCs.

**Option C: Plain `Callable[[PromptAreaSnapshot], PromptClassification]` type alias**

- Code impact:
  - No class needed — just a type alias and module-level functions.
  - Profile stores a callable directly.
- User-facing difference:
  - None.
- Pros:
  - Simplest, lightest.
  - Matches existing codebase style where module-level functions (like `editing_input_state()`) already do this work.
- Cons:
  - No place to hang shared variant metadata (name, version, debug identity).
  - Harder to attach diagnostic notes to the variant itself.

> **PROPOSED: Option A — `Protocol` with a single `classify` method.**
> Rationale: The existing `editing_input_state()` in `ready.py` is already a stateless function, so the refactoring target is naturally functional. However, the design requires variant identity and debug notes to be surfaceable (design decision 4: "emit notes that include the selected detector/variant identity"), which means the variant needs to carry metadata beyond a raw callable. A `Protocol` with `classify` plus a `variant_name` property provides the minimal contract that supports both classification and identity, without imposing ABC inheritance overhead. This also aligns with the design's "behavior-oriented rather than mechanism-oriented" principle. The risk is slightly weaker IDE discoverability compared to ABC, but with only 2–3 concrete implementations this is negligible.
>
> **DECISION: Accept PROPOSED.**
> Rationale: A narrow `Protocol` fits this repo's existing use of Protocol-based internal seams better than introducing another ABC hierarchy for a one-method prompt classifier, and it still allows concrete frozen dataclass implementations to carry variant identity. That keeps the contract behavior-oriented while avoiding raw `Callable` ambiguity around metadata and debug naming.

### Q2) What Codex version floor should define the initial concrete version family?

- Decision: `Blocking`

#### Why this matters

Task 2.1 requires splitting Codex into "at minimum one concrete current-version family plus fallback." The version floor determines which observed Codex versions route to the new prompt behavior variant vs the conservative fallback. Getting this wrong means either (a) the fallback handles too many versions conservatively, reducing tracking quality, or (b) the concrete variant handles versions it wasn't tested against, risking false classifications.

#### Examples

- Codex CLI currently at `0.116.0`. If a user runs `0.110.0`, does it get the concrete variant or fallback?
- If Codex releases `0.120.0` with the same prompt rendering, does it automatically get the concrete variant (desirable) or fall to fallback (safe but unnecessary)?

#### Why this question arises in code

- `registry.py` `_select_best_registration()` uses `minimum_supported_version` as a semver floor. The Claude precedent uses `(2, 1, 0)` — a minor-version boundary.
- For Codex at `0.116.x`, the minor version `116` is unusually high and may not correspond to a meaningful "family" boundary the way `2.1` does for Claude.
- The `_parse_version()` function in `registry.py` extracts `(major, minor, patch)` from strings like `"0.116.0"`, so `(0, 116, 0)` is technically valid but semantically questionable as a "family."

#### Options

**Option A: Floor at `(0, 116, 0)` — tight binding to current observed version**

- Code impact:
  - `DetectorProfileRegistration(app_id="codex_tui", minimum_supported_version=(0, 116, 0), ...)`
  - Only `0.116.x` and above route to the concrete variant.
- User-facing difference:
  - Older Codex versions (e.g., `0.110.0`) get fallback behavior immediately.
  - Future versions (e.g., `0.120.0`) get the concrete variant automatically.
- Pros:
  - Safe: only tested versions use the concrete variant.
- Cons:
  - May be too narrow if nearby older versions have the same prompt rendering.

**Option B: Floor at `(0, 100, 0)` — broader family covering recent Codex 0.1xx releases**

- Code impact:
  - `DetectorProfileRegistration(app_id="codex_tui", minimum_supported_version=(0, 100, 0), ...)`
- User-facing difference:
  - Wider range of Codex versions get the concrete variant.
- Pros:
  - Covers more users without requiring version-specific profiles.
- Cons:
  - Risk of applying the variant to untested versions with different prompt rendering.

**Option C: Floor at `(0, 1, 0)` — catch-all for any versioned Codex, fallback only for `None`**

- Code impact:
  - `DetectorProfileRegistration(app_id="codex_tui", minimum_supported_version=(0, 1, 0), ...)`
  - Fallback only activates when no version is observed.
- User-facing difference:
  - All versioned Codex instances get the concrete variant.
- Pros:
  - Maximizes coverage.
- Cons:
  - Defeats the purpose of conservative fallback for genuinely old/different versions.

> **PROPOSED: Option A — floor at `(0, 116, 0)`, the earliest version whose prompt rendering has been directly validated.**
> Rationale: The change's core principle is conservative degradation for unvalidated prompt presentations (design decision 4). Using a tight floor means only versions with validated prompt behavior get the concrete variant, while untested older versions get the fallback. The fallback already exists (current `builtin` behavior). As new Codex versions are validated, the floor can be left as-is (new versions automatically resolve to the highest compatible floor) or a new version-family profile can be added. This mirrors how Claude uses `(2, 1, 0)` — the floor of the earliest version where the v2.1.x detection suite was tested.
>
> **DECISION: Accept PROPOSED.**
> Rationale: The initial concrete Codex family should use `minimum_supported_version=(0, 116, 0)`. That is the earliest directly validated version family we have in hand, it fits the shared semver-floor selector in [registry.py](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/registry.py#L143), and it preserves the intended conservative fallback path for older or unvalidated Codex versions.

### Q3) Should the prompt behavior variant live as a separate composable object on the profile, or as a set of overridable methods on the detector class itself?

- Decision: `Deferrable`

#### Why this matters

This determines the structural relationship between the detector profile and the prompt variant. A composable object means the variant is injected/owned by the profile; overridable methods mean the variant logic is part of the detector's class hierarchy.

#### Examples

- Composable: `self.m_prompt_variant.classify(snapshot)` in the detector's `detect()` method.
- Overridable: `self._classify_prompt(snapshot)` as a method on the detector base class, overridden per version family.

#### Why this question arises in code

- The Claude Code detector uses class inheritance for version differences (`ClaudeCodeSignalDetectorV2_1_X` vs `FallbackClaudeDetector` both inheriting `BaseVersionedClaudeDetector`). The version-specific behavior is embedded in the class, not in a separate composable.
- The design says "the selected Codex profile may directly own its prompt behavior variant or construct it from a profile-private selector" — it explicitly permits both approaches.
- If only prompt behavior varies between Codex version families (and the rest of the detection logic is shared), a composable variant avoids duplicating the full detector class.

#### Options

**Option A: Composable object — variant as a separate frozen dataclass/Protocol owned by the profile**

- Code impact:
  - Codex detector stores `m_prompt_variant: CodexPromptBehaviorVariant`.
  - One Codex detector class handles all version families; only the variant instance differs.
  - Factory function selects the variant based on version.
- User-facing difference:
  - None.
- Pros:
  - Avoids duplicating detector logic across version-family subclasses.
  - Aligns with the design's "behavior variant" language.
  - Easier to unit-test variants in isolation.
- Cons:
  - New abstraction pattern not present in existing codebase.
  - If other signal dimensions also need version-varying behavior later, this may lead to multiple composable objects.

**Option B: Overridable methods — version-family subclasses of a Codex detector base**

- Code impact:
  - `BaseCodexTuiDetector` with shared logic, `CodexTuiDetectorV0_116_X` and `FallbackCodexTuiDetector` as subclasses.
  - Prompt classification is a method override.
  - Each subclass is a separate `DetectorProfileRegistration`.
- User-facing difference:
  - None.
- Pros:
  - Consistent with the Claude Code pattern (`BaseVersionedClaudeDetector` → subclasses).
  - No new abstraction — just class hierarchy.
- Cons:
  - Duplicates the full detector class structure even when only prompt behavior differs.
  - If only 1 method differs between subclasses, inheritance is overkill.

> **PROPOSED: Option A — composable variant object owned by the profile, with one detector class parameterized by variant.**
> Rationale: The key difference from the Claude Code pattern is that this change explicitly scopes the version-varying behavior to one surface region (prompt area), not the entire detection suite. In `profile.py`, the Codex `detect()` method (lines 70–166) contains ~100 lines of shared logic (activity detection, overlay checks, error scanning, posture derivation) and only ~10 lines of prompt/editing logic. Duplicating the full detector class to vary only prompt classification would be disproportionate. A composable variant isolates exactly the varying dimension. If future Codex versions also vary in activity or overlay detection, those can become additional composable variants or the design can pivot to full subclasses — but starting with the narrower approach matches the stated scope. The detector would still be registered as separate `DetectorProfileRegistration` entries (one per version family), each constructing the same detector class with a different variant instance.
>
> **DECISION: Accept PROPOSED.**
> Rationale: This is the structure that best matches the actual scope of variation. The current Codex detector keeps most of its logic in one shared `detect()` implementation at [profile.py](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/apps/codex_tui/profile.py#L70), and only the prompt-area interpretation is intended to vary. A composable profile-owned variant preserves version-visible detector registrations while avoiding subclass duplication across mostly identical detectors.

### Q4) Should `editing_input=unknown` be emitted when prompt IS visible but unrecognized, or should the profile use `editing_input=no` with a diagnostic note?

- Decision: `Blocking`

#### Why this matters

This is the design's own flagged open question. It changes the observable contract of `TrackedStateSnapshot.surface_editing_input`. Currently `"unknown"` only appears when the prompt is not visible at all (`prompt_visible=False` in `editing_input_state()`, `ready.py` line 96). Emitting `"unknown"` when the prompt IS visible but the variant cannot classify it is a semantic expansion that downstream consumers may interpret differently.

#### Examples

- A dashboard that shows "editing: unknown" to the user — is that confusing when the prompt is clearly visible?
- A downstream automation that treats `editing_input="unknown"` as "not ready to send input" — false negative when the user IS actually editing.
- A debugging session where `editing_input="unknown"` plus a note like `"unrecognized_prompt_presentation"` helps diagnose drift quickly.

#### Why this question arises in code

- `editing_input_state()` in `ready.py` (lines 88–99):
  ```python
  if prompt_visible and prompt_text:
      return "yes"
  if prompt_visible:
      return "no"
  return "unknown"
  ```
- The `Tristate` type (`Literal["yes", "no", "unknown"]`) is already defined and used throughout the tracker.
- `TrackedStateSnapshot.surface_editing_input` is part of the public state contract consumed by live-watch, demo pack, and replay tools.

#### Options

**Option A: Emit `editing_input=unknown` for unrecognized prompt presentation**

- Code impact:
  - When the variant returns `PromptClassification.UNKNOWN`, the profile sets `editing_input="unknown"` and adds a diagnostic note.
  - This creates a new path to `"unknown"` that didn't exist before (prompt visible but unrecognized).
- User-facing difference:
  - Live-watch / demo dashboard shows "editing: unknown" in drift scenarios.
  - Downstream consumers that branch on `editing_input` see a new state they may not have handled.
- Pros:
  - Honest: doesn't pretend to know something it doesn't.
  - Drift is immediately visible in all consumers.
  - Aligns with the design's "false certainty is worse than temporary ambiguity" principle.
- Cons:
  - Could cause unexpected behavior in consumers that assume `prompt_visible → editing ∈ {yes, no}`.

**Option B: Emit `editing_input=no` with a diagnostic note for unrecognized prompt presentation**

- Code impact:
  - When the variant returns `PromptClassification.UNKNOWN`, the profile sets `editing_input="no"` and adds a note like `"prompt_classification_uncertain"`.
  - No new `"unknown"` path is introduced for visible prompts.
- User-facing difference:
  - Live-watch shows "editing: no" — indistinguishable from a real empty prompt unless notes are inspected.
  - Consumers that don't read notes won't see drift at all.
- Pros:
  - Non-breaking: downstream consumers see no new states.
  - "No editing" is a safe default for most automation (won't try to read draft text that may not exist).
- Cons:
  - Hides drift from consumers that don't inspect notes.
  - Contradicts the design's stated preference for explicit uncertainty.

> **PROPOSED: Option A — emit `editing_input=unknown` for unrecognized prompt presentation, with a diagnostic note.**
> Rationale: The design's decision 4 explicitly states "false certainty is worse than temporary ambiguity for a drift-prone TUI region." Option B contradicts this by masking uncertainty behind `"no"`. The `Tristate` type already includes `"unknown"` as a first-class value, and `TrackedStateSnapshot` already carries it. The risk of breaking downstream consumers is real but manageable: (1) the `"unknown"` state already exists in the type contract, so well-written consumers should already handle it; (2) the diagnostic note provides an actionable signal for operators; (3) this only triggers for genuinely unrecognized prompt presentations, which should be rare outside version drift windows. If specific consumers need to treat `"unknown"` as `"no"` for their use case, they can do so locally.
>
> **DECISION: Accept PROPOSED.**
> Rationale: Unrecognized visible-prompt presentation should map to `editing_input=unknown` with a diagnostic note. That is the most honest interpretation of the existing `Tristate` contract in [models.py](/data1/huangzhe/code/houmao/src/houmao/shared_tui_tracking/models.py#L9), and it is consistent with the change's core design principle that false certainty is worse than temporary ambiguity for this drift-prone region.
