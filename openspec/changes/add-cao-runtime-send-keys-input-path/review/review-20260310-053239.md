## Summary

- **Solid:** The proposal correctly identifies the fundamental abstraction gap between turn-shaped `send-prompt` and raw terminal input, and the decision to keep them as separate commands is well justified by the existing codebase architecture.
- **Solid:** The `<[key-name]>` mixed-sequence grammar is compact, deterministic, and directly maps to tmux key-name semantics. The exact-token + global escape approach avoids parser ambiguity.
- **Solid:** Scoping the first release to `backend=cao_rest` is pragmatic and matches the originating use case (`interrupt_driver.py` line 136–148 already does ad hoc `tmux send-keys`).
- **Risk:** The design proposes persisting optional tmux window metadata in the manifest but leaves the persistence format, schema impact, and backward compatibility details unspecified — this is the most under-specified area of the change.
- **Risk:** The tmux delivery mechanism for literal text segments is described as "tmux literal-text behavior" without specifying the exact tmux primitive (raw `send-keys -l` vs. `load-buffer`/`paste-buffer`). The choice matters for TUI interaction fidelity.
- **Risk:** Task 4.3 (update the interrupt demo) couples the change to a demo script that directly uses the CAO REST client and shadow parser, and its current wording could be read as a larger refactor than the change actually needs.

## Findings

### Must Fix / Blocking

1. **MF-1: Tmux window metadata persistence format is unspecified.**

   Design decision §5 states the runtime will "persist optional tmux window metadata when available for fast reuse" in the manifest, but neither the spec nor the tasks define:
   - What fields (`tmux_window_id`, `tmux_window_name`, `tmux_window_index`)?
   - Where in the manifest structure (inside `CaoSectionV2`? in `backend_state`?)?
   - Whether this requires a schema version bump (the design says "no" but the rationale is thin given `CaoSectionV2` uses `extra="forbid"` in `boundary_models.py:102`).

   The `CaoSectionV2` Pydantic model has `extra="forbid"` (inherited from `_StrictBoundaryModel`), meaning any new field added without updating the model and schema will cause validation failures on load. This is a hard incompatibility, not a soft one.

   **Impact:** Task 3.1 ("Extend CAO session state and manifest persistence with the metadata needed for manifest-driven tmux target resolution") cannot be implemented without resolving this. Older manifests that lack the new field will continue to validate, but the spec needs to explicitly confirm the field is `Optional` with a default and that the JSON schema `session_manifest.v2.schema.json` is updated accordingly.

   **Recommendation:** Add a concrete field specification to the `brain-launch-runtime` spec: define the exact field name(s), their placement in `CaoSectionV2`, their optionality, and the corresponding JSON schema update path (likely additive optional fields, no version bump needed).

2. **MF-2: Missing specification of the control-input result model.**

   The `brain-launch-runtime` spec says the control-input command "return[s] a control-input result without waiting for turn completion or output extraction." But the existing result models are:
   - `SessionResult` — turn-shaped (has `turn_index`, `completed`, `interrupted`, `output_text`)
   - `SessionControlResult` — interrupt/terminate-shaped (has `action: Literal["interrupt", "terminate"]`)
   - `SessionEvent` — streaming event

   None of these fit a "raw input delivered, no turn state change" result. The spec and tasks mention a "control-input result" but do not define its shape, and `models.py` has no placeholder for it.

   **Recommendation:** Define a `ControlInputResult` dataclass in the spec (or explicitly state that `SessionControlResult` will be extended with a new `action` literal like `"control_input"`). Task 1.1 should include "add result model" as an explicit subtask.

### Should Fix / Important

3. **SF-1: Tmux delivery primitive for literal text is ambiguous.**

   Design decision §4 says literal text uses "tmux literal-text behavior" and special keys use "tmux key names." However, the codebase has three distinct tmux text delivery strategies:
   - `tmux send-keys -l` (raw literal, character-by-character — used by tmux natively, not present in current codebase)
   - `tmux load-buffer` + `paste-buffer -p` (bracketed paste — `cao_rest.py` private helpers and `tmux.py:199–252`)
   - `tmux send-keys` without `-l` (key-name interpretation — `tmux_runtime.py:146–162`)

   For `send-keys` style raw input, `send-keys -l` is the natural fit (it preserves literal characters without triggering tmux key-name interpretation). Bracketed paste (`paste-buffer -p`) would be wrong because it signals "pasted text" to the TUI and adds submit behavior. The spec should be explicit.

   **Recommendation:** Task 2.2 should specify: literal text segments → `tmux send-keys -l -t <target>`, special-key tokens → `tmux send-keys -t <target> <key-name>`. Both without implicit trailing `Enter`.

4. **SF-2: Supported key-name allowlist is absent.**

   The `runtime-tmux-control-input` spec requires rejecting unsupported exact key tokens, but there is no definition of which key names are supported. The scenario shows `Enter`, `Down`, and `Escape`, and the demo uses `Escape`, but no exhaustive or minimum allowlist is provided.

   Tmux supports a large key-name vocabulary (`Enter`, `Escape`, `Up`, `Down`, `Left`, `Right`, `Tab`, `BSpace`, `C-a` through `C-z`, `M-a` through `M-z`, `F1`–`F12`, `Home`, `End`, `PgUp`, `PgDn`, `DC`, `IC`, etc.). Without a defined allowlist, the implementation must choose between:
   - Pass-through to tmux and let tmux reject unknown names (lenient but opaque errors)
   - Curated allowlist in the runtime (safe but requires maintenance)

   **Recommendation:** Define a minimum guaranteed set in the spec (at least: `Enter`, `Escape`, `Up`, `Down`, `Left`, `Right`, `Tab`, `BSpace`, `C-c`, `C-d`, `C-z`). Document that the implementation may support a larger curated allowlist in future releases, while still rejecting unsupported exact tokens explicitly. Task 2.3 should include the guaranteed minimum set and the explicit-rejection rule.

5. **SF-3: `InteractiveSession` protocol may need extension.**

   The `InteractiveSession` protocol in `models.py:179–212` defines `send_prompt()`, `interrupt()`, `terminate()`, `close()`. The new `send-keys` capability needs a corresponding protocol method (e.g., `send_input_ex()`), but this is not mentioned in the tasks.

   If the new method is only on `CaoRestSession` (not part of the protocol), the `RuntimeSessionController` will need backend-type checks (`isinstance`) to dispatch it, which is the pattern already used for `stop()` → `configure_stop_force_cleanup()` (`runtime.py:120–123`). This is acceptable but should be an explicit design choice.

   **Recommendation:** Either extend `InteractiveSession` with an optional `send_input_ex()` method (with a default `NotImplementedError` raise for non-CAO backends), or document that `RuntimeSessionController` will use `isinstance` dispatch. Task 1.1 should address this.

6. **SF-4: CLI argument parsing for `send-keys` command needs specification.**

   The CLI currently has four commands: `build-brain`, `start-session`, `send-prompt`, `stop-session` (`cli.py:45–52`). Task 1.2 says "Add a new CLI command for mixed raw control input, including the global `--escape-special-keys` flag." However, the spec does not define:
   - Command name (likely `send-keys`, but not stated)
   - Required arguments (e.g., `--agent-identity`, `--sequence` or positional?)
   - Whether `--agent-def-dir` follows the same precedence pattern
   - Output format (JSON like other commands?)

   **Recommendation:** Add a concrete CLI interface definition to the spec or design document. Follow the pattern of `send-prompt` (`cli.py:131–148`): `send-keys --agent-identity <name-or-manifest> --sequence <string> [--escape-special-keys] [--agent-def-dir <path>]`, and specify that the command returns a single JSON control-action result object rather than streaming turn events.

### Nice To Have

7. **NTH-1: The `<[key-name]>` syntax uses bracket-angle delimiters that may conflict with shell quoting.**

   Sequences like `"/model<[Enter]><[Down]><[Enter]>"` require shell quoting to prevent `<` and `>` from being interpreted as redirections. While this is standard shell usage, the docs/spec could include a usage example showing proper quoting in Bash:
   ```bash
   gig-agents-cli send-keys --agent-identity my-agent --sequence '/model<[Enter]><[Down]><[Enter]>'
   ```

8. **NTH-2: Consider adding a `--dry-run` flag for debugging.**

   Parsing complex mixed sequences is error-prone in practice. A `--dry-run` flag that parses the sequence and prints the parsed segments (literal vs. key-name) without sending them would improve developer experience. This can be deferred.

9. **NTH-3: Task 4.3 (interrupt demo refactor) scope may be larger than expected.**

   The interrupt demo (`interrupt_driver.py`) does not just send an Escape key — it orchestrates a multi-step flow: initial readiness polling → first prompt submission via CAO → processing observation → Escape injection → second prompt → completion extraction. Refactoring this to use the new runtime `send-keys` command would only replace the `_send_escape_key()` call (`interrupt_driver.py:136–148`), while the rest of the flow uses the CAO REST client and shadow parser directly. The task should be scoped to just replacing the raw tmux call, not rewriting the entire demo.

## Design Alignment With Existing System

### Existing Patterns Observed

- **CLI command dispatch:** `cli.py:44–56` uses `args.command` string matching for dispatch, with a shared `_build_parser()` for subcommand definition. Each command has its own handler (`_cmd_*`).
- **Agent identity resolution:** `runtime.py:238–274` resolves `--agent-identity` via `resolve_agent_identity()` → manifest path. Both name-based (tmux env lookup) and path-based (direct file) modes are supported.
- **Backend session creation:** `runtime.py:336–449` uses a factory pattern with backend-specific branches and `cast(InteractiveSession, ...)` returns.
- **Controller pattern:** `RuntimeSessionController` (`runtime.py:89–146`) wraps backend sessions and handles manifest persistence after each operation.
- **Error hierarchy:** `errors.py` defines `BrainLaunchRuntimeError` → `BackendExecutionError` / `SessionManifestError` / etc. The CLI catches `BrainLaunchRuntimeError` at the top level (`cli.py:53–55`).
- **Manifest persistence:** `manifest.py` + `boundary_models.py` use Pydantic strict validation with `extra="forbid"`. Schema changes require model + JSON schema alignment.
- **Tmux primitives:** `tmux_runtime.py` provides shared primitives (`run_tmux`, `list_tmux_sessions`, etc.) used by `cao_rest.py` via aliased imports (`cao_rest.py:49–58`).

### Proposed Design Alignment

**Aligns well:**
- Adding a separate CLI command follows the existing subcommand pattern in `cli.py`.
- Using `resolve_agent_identity()` for the new command matches the `send-prompt` and `stop-session` patterns.
- Scoping to `cao_rest` backend with explicit rejection for others follows the existing `_create_backend_session()` factory pattern.
- The error strategy (explicit failures for unsupported backends, invalid tokens, unresolved targets) matches the existing error hierarchy.

**Potential mismatches:**
- The new tmux `send-keys` delivery will bypass `cao_rest.py`'s existing CAO REST client entirely, sending commands directly via `tmux_runtime.py:run_tmux()`. This is architecturally correct (raw control input should not go through CAO's paste-buffer + submit flow) but creates a second tmux interaction path in `CaoRestSession`. The implementation should use the shared `run_tmux()` from `tmux_runtime.py` rather than adding new `subprocess.run(["tmux", ...])` calls in `cao_rest.py`, to stay consistent with the existing abstraction layer.
- The tmux window resolution for control input needs to discover the `window_name` from CAO terminal metadata (`CaoTerminal.name`), which the interrupt demo already does (`interrupt_driver.py:185–188`). The runtime should encapsulate this resolution in a reusable helper in `cao_rest.py`, not leave it to individual callers.

## Open Questions (With Proposed Defaults)

### Q1) Should `send-keys` be a new method on the `InteractiveSession` protocol or a backend-specific method with `isinstance` dispatch?

- Decision: `Deferrable`

#### Why this matters

- If added to the protocol, all backends must implement it (even as `raise NotImplementedError`), and the type system enforces the contract.
- If left as backend-specific, `RuntimeSessionController` needs `isinstance` checks, creating an implicit dispatch that is harder to audit.

#### Examples

- `send-prompt` is a protocol method — all backends implement it.
- `configure_stop_force_cleanup()` is backend-specific — `RuntimeSessionController.stop()` uses `isinstance(self.backend_session, HeadlessInteractiveSession)` dispatch (`runtime.py:120–123`).

#### Why this question arises in code

- The `InteractiveSession` protocol (`models.py:179–212`) currently has exactly four methods: `send_prompt`, `interrupt`, `terminate`, `close`.
- Adding a fifth optional method changes the protocol surface. Backends that don't support it would need a stub.
- The controller (`runtime.py:89–146`) currently uses `isinstance` checks sparingly (only for `stop` force-cleanup).

```python
# Option A: Protocol extension
class InteractiveSession(Protocol):
    def send_input_ex(self, sequence: str, *, escape_special_keys: bool = False) -> ControlInputResult:
        ...

# Option B: isinstance dispatch in controller
def send_input_ex(self, sequence: str, ...) -> ControlInputResult:
    if not isinstance(self.backend_session, CaoRestSession):
        raise BackendExecutionError(f"send-keys not supported for {self.launch_plan.backend}")
    return self.backend_session.send_input_ex(sequence, ...)
```

#### Options

**Option A: Add to `InteractiveSession` protocol**

- Code impact:
  - Add `send_input_ex()` to `InteractiveSession` protocol in `models.py`.
  - All backends (`CaoRestSession`, `ClaudeHeadlessSession`, `CodexHeadlessSession`, `GeminiHeadlessSession`, `CodexAppServerSession`) must implement or stub the method.
  - `RuntimeSessionController.send_input_ex()` delegates directly without type checks.
- User-facing difference:
  - Cleaner contract; future backends that support control input get it "for free."
- Pros:
  - Type-safe, auditable, follows existing protocol pattern.
  - Easier to extend to non-CAO backends later.
- Cons:
  - Five stub implementations that only raise `NotImplementedError`.
  - Protocol grows for a capability that currently applies to one backend.

**Option B: Backend-specific `isinstance` dispatch**

- Code impact:
  - Add `send_input_ex()` only to `CaoRestSession`.
  - `RuntimeSessionController` uses `isinstance` check before calling.
  - No changes to other backends.
- User-facing difference:
  - Identical behavior. The `isinstance` check produces the same unsupported-backend error.
- Pros:
  - Minimal blast radius; no changes to backends that don't support it.
  - Matches the existing `configure_stop_force_cleanup` precedent.
- Cons:
  - Implicit dispatch pattern; harder to discover the capability from the protocol.
  - If multiple backends gain control input later, the `isinstance` cascade grows.

#### Proposal (Recommended)

Use Option B (`isinstance` dispatch) for the initial release. This matches the existing `configure_stop_force_cleanup` pattern and avoids five no-op stubs for a single-backend capability. Migrate to protocol method when a second backend gains support.

#### Pros / Cons (Proposal)

- Pros:
  - Minimal change to existing backends and protocol.
  - Consistent with established `isinstance` precedent in `runtime.py:120`.
- Cons:
  - Implicit dispatch; may need migration later.

> **DECISION: Use `isinstance` dispatch for initial release, defer protocol extension.**
> Rationale: The existing codebase already uses `isinstance(self.backend_session, HeadlessInteractiveSession)` for backend-specific behavior in `runtime.py:120–123`. Adding five stub implementations for a single-backend capability would add noise without type-safety benefit since the capability is CAO-scoped. When a second backend gains `send_input_ex()`, the method should be promoted to the protocol.

### Q2) What is the exact tmux delivery primitive for literal text segments?

- Decision: `Resolved`

#### Why this matters

- The choice between `send-keys -l`, `load-buffer`/`paste-buffer`, and `send-keys` (key-name mode) fundamentally changes how the TUI receives the text and whether bracketed paste mode is engaged.
- The existing CAO input path intentionally uses bracketed paste (`paste-buffer -p`) plus explicit `Enter` to handle multi-line input and TUI hotkey bypass. The new raw control-input path must not replicate that behavior.

#### Examples

- `tmux send-keys -l -t target "hello"` → types `h`, `e`, `l`, `l`, `o` literally, no key-name interpretation, no bracketed paste, no `Enter`.
- `tmux send-keys -t target "hello"` → tmux interprets `hello` as a key sequence (safe for simple alphanumeric but risky for punctuation that overlaps tmux key names).
- `tmux load-buffer` + `paste-buffer -p` → bracketed paste mode, TUI sees paste start/end markers.

#### Why this question arises in code

- The CAO `tmux.py:199–252` uses `load-buffer` + `paste-buffer -p` + explicit `Enter` — this is intentionally paste-like.
- The CAO `tmux.py:309–337` `send_special_key()` uses `pane.send_keys(key, enter=False)` — this is for tmux key names.
- `tmux_runtime.py:146–162` `run_tmux()` is a low-level subprocess wrapper with no opinion on `-l` vs key-name mode.
- The new path needs to mix literal and key-name delivery in the same ordered stream.

```python
# For a sequence like "/model<[Enter]><[Down]><[Enter]>"
# Parsed segments: [Literal("/model"), Key("Enter"), Key("Down"), Key("Enter")]

# Delivery:
run_tmux(["send-keys", "-l", "-t", target, "/model"])   # literal text
run_tmux(["send-keys", "-t", target, "Enter"])           # key name
run_tmux(["send-keys", "-t", target, "Down"])            # key name
run_tmux(["send-keys", "-t", target, "Enter"])           # key name
```

#### Options

**Option A: `send-keys -l` for text, `send-keys` (no `-l`) for keys**

- Code impact:
  - Implement in `tmux_runtime.py` as a new `send_control_sequence()` helper.
  - Iterate over parsed segments: text → `["send-keys", "-l", "-t", target, text]`, key → `["send-keys", "-t", target, key_name]`.
- User-facing difference:
  - True character-by-character literal typing for text segments.
  - No bracketed paste markers, no implicit Enter.
- Pros:
  - Exact raw typing semantics — matches the design intent for partial input.
  - Simple, no buffer management.
  - Characters are typed individually which matches TUI interaction expectations.
- Cons:
  - Slower than paste-buffer for long text (character-by-character tmux delivery).
  - Each segment is a separate `subprocess.run()` call.

**Option B: `load-buffer`/`paste-buffer` (without `-p`) for text, `send-keys` for keys**

- Code impact:
  - Use `load-buffer` + `paste-buffer` (no `-p` flag → no bracketed paste) for text segments.
  - Use `send-keys` for key-name segments.
- User-facing difference:
  - Text appears instantly in the pane (pasted), but without bracketed paste markers.
- Pros:
  - Faster for long text.
- Cons:
  - `paste-buffer` without `-p` still inserts all text at once, which may not match TUI expectations for partial typing.
  - Some TUIs behave differently with pasted text vs. typed text even without bracketed paste.

#### Proposal (Recommended)

Use Option A: `send-keys -l` for literal text, `send-keys` (no `-l`) for key names. This is the natural tmux primitive for raw key injection and directly matches the design goal of "preserve partial typing semantics rather than bracketed-paste semantics" (design §4).

#### Pros / Cons (Proposal)

- Pros:
  - Exact semantic match for raw typing behavior.
  - No paste-buffer lifecycle management.
  - Aligns with the demo's existing `send-keys` usage (`interrupt_driver.py:138–139`).
- Cons:
  - Slower for long literal text segments (acceptable for the control-input use case which targets short sequences like `/model`, slash commands, and menu navigation).

> **DECISION: Use `tmux send-keys -l` for literal text and `tmux send-keys` for key names.**
> Rationale: The design explicitly states "literal text segments are delivered using tmux literal-text behavior" (design §4), and `send-keys -l` is tmux's native literal-text primitive. The existing CAO paste-buffer path (`tmux.py:199–252`) is intentionally paste-oriented with Enter appended, which contradicts the "no implicit Enter" requirement. The interrupt demo already uses bare `send-keys` for `Escape` injection (`interrupt_driver.py:139`), establishing the precedent. Control-input sequences are expected to be short (slash commands, menu keys), so character-by-character speed is not a concern.

### Q3) Should the new tmux window resolution helper be placed in `cao_rest.py` or factored into `tmux_runtime.py`?

- Decision: `Deferrable`

#### Why this matters

- The interrupt demo (`interrupt_driver.py:184–188`) already resolves tmux targets by calling `CaoRestClient.get_terminal()` and using `terminal.name` as the window name. The new runtime control-input path needs to do the same thing.
- Placing the helper determines whether non-CAO backends can reuse it later.

#### Examples

- Current: `interrupt_driver.py:184–188` does ad hoc resolution:
  ```python
  terminal = client.get_terminal(terminal_id)
  window_name = terminal.name.strip()
  tmux_target = f"{session_name}:{window_name}"
  ```
- New: The runtime should encapsulate this in a reusable function.

#### Why this question arises in code

- `cao_rest.py` already has private tmux window helpers (`_resolve_tmux_window_by_name`, `_select_tmux_window`, `_read_bootstrap_tmux_window`) for CAO-specific window management during session startup.
- `tmux_runtime.py` has generic tmux primitives but no CAO-specific window resolution.
- The window resolution requires a CAO REST API call (`get_terminal`), which is CAO-specific.

#### Options

**Option A: Place in `cao_rest.py` as a method on `CaoRestSession`**

- Code impact:
  - Add `resolve_tmux_target()` method to `CaoRestSession` that uses `self._client.get_terminal()` + `self._session_name`.
  - `RuntimeSessionController` calls `self.backend_session.resolve_tmux_target()` via `isinstance` check.
- Pros:
  - Co-located with existing CAO tmux helpers.
  - Has direct access to `self._client`, `self._session_name`, `self._terminal_id`.
- Cons:
  - Not reusable outside `CaoRestSession`.

**Option B: Factor target resolution into a standalone function in `cao_rest.py`**

- Code impact:
  - Add `resolve_cao_tmux_target(client, terminal_id, session_name) -> str` as a module-level function.
  - Both `CaoRestSession` and the demo script can call it.
- Pros:
  - Reusable by the interrupt demo and other scripts.
  - Clean function signature with explicit dependencies.
- Cons:
  - Still CAO-specific; would need a CAO REST client instance.

#### Proposal (Recommended)

Use Option A for the initial implementation (method on `CaoRestSession`), but extract to a standalone function if Task 4.3 (demo refactor) also needs it. The window resolution depends on the CAO REST client and is inherently CAO-specific.

#### Pros / Cons (Proposal)

- Pros:
  - Minimal initial change.
  - Co-located with related tmux window logic.
- Cons:
  - Demo refactor may still need a standalone version.

> **DECISION: Implement as `CaoRestSession` method initially; extract if demo needs it.**
> Rationale: The existing CAO tmux window helpers (`_resolve_tmux_window_by_name`, `_list_tmux_windows`) in `cao_rest.py:88–91` are already private methods of the session class. Adding target resolution as a method follows this pattern. The demo script (`interrupt_driver.py`) currently constructs its own target from raw API responses; if Task 4.3 needs to share this logic, a module-level function can be extracted at that point without disrupting the runtime path.

### Q4) Should the optional tmux window metadata in `CaoSectionV2` trigger a schema version bump to V3?

- Decision: `Resolved`

#### Why this matters

- `CaoSectionV2` uses `extra="forbid"` (`boundary_models.py:22–25` via `_StrictBoundaryModel`), meaning any unrecognized field in a persisted manifest will cause a `ValidationError` on load.
- Adding a new optional field to `CaoSectionV2` requires updating both the Pydantic model and the JSON schema to accept it. If an older runtime version tries to load a manifest with the new field, it will fail.
- Conversely, if the new field is optional with a default, newer runtimes can load older manifests that lack it.

#### Examples

- Current `CaoSectionV2` fields: `api_base_url`, `session_name`, `terminal_id`, `profile_name`, `profile_path`, `parsing_mode`, `turn_index`.
- Proposed addition: `tmux_window_name: str | None = None` (or similar).

#### Why this question arises in code

- `boundary_models.py:102–124` defines `CaoSectionV2` with strict validation.
- `session_manifest.v2.schema.json` at `$defs.CaoSectionV2` lists the exact required/optional fields.
- `manifest.py:12` defines `SESSION_MANIFEST_SCHEMA_VERSION = 2` — bumping this requires updating the version constant, schema file, and all validation paths.

#### Options

**Option A: Add optional field to `CaoSectionV2`, no schema version bump**

- Code impact:
  - Add `tmux_window_name: str | None = None` to `CaoSectionV2`.
  - Update `session_manifest.v2.schema.json` to include the optional field.
  - No version bump — `schema_version` remains 2.
- Pros:
  - Simpler; no migration logic needed.
  - Newer manifests are readable by any V2-compatible runtime that has the updated model.
- Cons:
  - An older runtime (without the updated model) will reject manifests that include `tmux_window_name` because `extra="forbid"` will fire.
  - This is acceptable only if backward compatibility means "new runtime reads old manifests" (not "old runtime reads new manifests").

**Option B: Bump to `CaoSectionV3` / `SessionManifestPayloadV3`**

- Code impact:
  - Create `CaoSectionV3` with the new field.
  - Create `SessionManifestPayloadV3` referencing `CaoSectionV3`.

### Q5) How should the parser handle adjacent or nested `<[` sequences?

- Decision: `Deferrable`

#### Why this matters

- The grammar defines `<[key-name]>` as exact tokens, but edge cases like `<[<[Enter]>]>` or `<[Enter]><[Enter]>` (adjacent) need deterministic handling.

#### Examples

- `<[Enter]><[Enter]>` — should parse as two `Enter` keys (straightforward).
- `<[<[Enter]>` — ambiguous: is this literal `<[` followed by key `Enter` inside another `<[...]>`?
- `text<[Enter]>more<[Down]>` — should parse as: literal `text`, key `Enter`, literal `more`, key `Down`.

#### Why this question arises in code

- A regex like `<\[([A-Za-z0-9_-]+)\]>` handles the clean cases. Nested `<[` is an edge case the spec doesn't address.

#### Options

**Option A: Left-to-right greedy scan with non-greedy regex**

- Use a non-greedy regex `<\[([A-Za-z0-9_-]+?)\]>` that matches the shortest valid key-name.
- `<[<[Enter]>` → literal `<[`, then key `Enter` (the regex would match the first valid `<[Enter]>`).
- Adjacent sequences parse naturally.

**Option B: Fail on nested `<[` inside an open token**

- If the parser encounters `<[` while scanning for `]>`, fail with a parse error.

#### Proposal (Recommended)

Use Option A. A left-to-right scan with `re.split()` or `re.finditer()` on the pattern `<\[([A-Za-z0-9_-]+)\]>` naturally handles adjacency and treats stray `<[` as literal text (per the spec: "preserve non-matching substrings as literal text"). Nested `<[` is literal because it doesn't match the exact pattern.

#### Pros / Cons (Proposal)

- Pros:
  - Simple regex, deterministic, handles all edge cases.
  - Matches spec: "Exact token recognition requires the precise `<[key-name]>` form."
- Cons:
  - None significant.

> **DECISION: Use left-to-right regex scan; stray `<[` is literal text.**
> Rationale: The spec explicitly states "recognize a special-key token only when the substring uses the exact `<[key-name]>` form" (spec `runtime-tmux-control-input` requirement 1). A regex like `<\[([A-Za-z0-9_-]+)\]>` applied left-to-right inherently treats non-matching fragments as literal text. This is deterministic and matches the spec's intent without additional parser state.

### Q7) What result shape should `send_input_ex()` return?

- Decision: `Resolved`

#### Why this question arises in code

- `stop-session` emits a single `SessionControlResult` JSON object.

**Option A: Introduce a dedicated `ControlInputResult` model**

- Cons:

#### Pros / Cons (Proposal)

  - Keeps `send-keys` as a single control-action response instead of inventing a third result family.
- Cons:

> Rationale: The current runtime already distinguishes turn-style responses (`SessionEvent`) from single-object control responses (`SessionControlResult`). Raw terminal input fits the latter family more naturally than a new turn result, and extending the action literal keeps the CLI output uniform for `stop-session` and `send-keys`.

### Q9) How should supported key names be specified?
