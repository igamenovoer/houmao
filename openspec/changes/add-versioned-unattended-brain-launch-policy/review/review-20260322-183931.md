## Summary

- **Solid problem framing**: The proposal correctly identifies that `launch_args_override = ["--dangerously-skip-permissions"]` scattered across demo packs (`src/houmao/demo/shared_tui_tracking_demo_pack/tooling.py`), CAO providers, and config profiles is fragile and does not capture user intent.
- **Sound two-phase split**: Persisting abstract intent at build time and resolving concrete strategy at launch time aligns well with the existing `BuildRequest` → `BrainManifest` → `LaunchPlan` pipeline already in the codebase.
- **Good fail-closed default**: Refusing to launch when unattended mode cannot be guaranteed is the right call for an orchestration framework that manages real processes.
- **Registry data/code hybrid carries implementation risk**: The descriptive-data-plus-Python-hooks design is well-motivated but underspecified — the data file format, action vocabulary, and hook contract are not pinned down in specs, leaving room for scope creep or a mini-DSL that becomes hard to test.
- **Claude bootstrap replacement is the highest-risk path**: Replacing the 123-line create-only `claude_bootstrap.py` with idempotent strategy-managed state is operationally sensitive and needs clear invariant documentation and integration test coverage for reused homes.
- **Gemini backend is absent from all artifacts**: `gemini_headless` is a supported `BackendKind` but is never mentioned in registry, specs, or tasks — even as an explicit non-goal or deferred item.

## Findings

### Must Fix / Blocking

1. **Registry data file format and action vocabulary are not specified.** The design (Decision 3) describes a "hybrid version launch policy registry" with "descriptive registry entries stored as repo-owned data files" and lists example generic actions (ensure/remove CLI args, set JSON/TOML keys, require files/values, record provenance). However, no spec or design artifact defines: (a) the schema of a registry entry data file, (b) the canonical action type identifiers, (c) how provider-specific Python hooks are declared/discovered, or (d) whether registry entries are YAML, JSON, or Python. Task 2.1 says "implement the registry format and loader" but the spec `versioned-launch-policy-registry/spec.md` only describes behavioral requirements (version-scoped strategies, ordered actions, fail-closed) without constraining the file format. This leaves the implementer to design a significant data schema with no review checkpoint.

> **DECISION: _(accept / refine / reject / defer this finding)_**
> Rationale: Refine. The reviewer correctly identified that the artifacts still do not pin the concrete entry schema, action identifiers, or hook-declaration contract. However, the gap is narrower than in the original review snapshot because the current specs now already require minimal-input and evidence metadata. The next artifact revision should define the concrete registry entry structure and action vocabulary while keeping the hybrid data-plus-Python design.

2. **No spec constrains how `operator_prompt_mode` flows from recipe YAML through `BuildRequest` to manifest.** The `component-agent-construction/spec.md` says "the system SHALL let callers declare an operator prompt policy" and "the manifest records that requested policy," but it does not specify the field name, allowed values, or where the field lives in the recipe schema vs. `BuildRequest` dataclass vs. manifest dict. The current `BrainRecipe` (`brain_builder.py:61-68`) and `BuildRequest` (`brain_builder.py:72-84`) have no such field. The spec should nail down:
   - The exact field name and type (e.g., `operator_prompt_mode: Literal["interactive", "unattended"] | None`)
   - Whether it appears in `BrainRecipe`, `BuildRequest`, or both
   - The manifest key path (e.g., `inputs.operator_prompt_mode` or a new top-level `launch_policy` block)

> **DECISION: _(accept / refine / reject / defer this finding)_**
> Rationale: Refine. The current artifacts now consistently use the field name `operator_prompt_mode`, so that part is no longer open. The remaining gap is that the artifacts still need to pin where the field lives in recipe YAML, `BuildRequest`, and the resolved manifest path.

### Should Fix / Important

3. **`LaunchPlan` model extension is not described precisely enough for task 1.2.** Task 1.2 says "extend launch-plan and session-manifest models to carry requested policy plus detected-version and selected-strategy provenance." The current `LaunchPlan` (`models.py:56-98`) is a frozen dataclass. The design needs to clarify whether launch policy provenance goes into:
   - New typed fields on `LaunchPlan` (breaking the current signature)
   - The existing `metadata: dict[str, Any]` bag (untyped, consistent with current `cao_parsing_mode_config` pattern at `launch_plan.py:32-36`)
   - A new nested dataclass (e.g., `LaunchPolicyProvenance`)

   The current codebase uses `metadata` for CAO-specific config, so using it for policy provenance would be consistent but untyped. A typed approach would be cleaner but breaks the existing pattern. The spec should pick one.

> **DECISION: _(accept / refine / reject / defer this finding)_**
> Rationale: Accept. Launch-policy provenance is first-class cross-backend state, not just backend-specific implementation detail, so the artifacts should pick a typed provenance structure on `LaunchPlan` and persisted manifests rather than relying only on the existing untyped `metadata` bag. Backend-specific applied-action traces can still use `metadata` where appropriate.

4. **Task 1.3 ("route raw launch helpers and runtime-managed launch paths through one shared entrypoint") has unclear scope.** The "raw launch helper" is the generated `launch.sh` script that `BrainBuilder` emits at build time. This is a shell script, not Python. Making it call the same Python strategy engine at launch time means either (a) the shell script calls a Python CLI entrypoint for strategy resolution before exec-ing the tool, or (b) the shell script is replaced by a Python launcher. Neither approach is described. The current `launch.sh` is a simple `exec` wrapper generated by `brain_builder.py` — converting it to a two-phase Python-then-exec pipeline is a non-trivial architecture change that should be explicitly addressed in the design.

> **DECISION: _(accept / refine / reject / defer this finding)_**
> Rationale: Refine. The scope issue is real, but the architectural risk is overstated because the generated `launch.sh` helper already shells out to Python bootstrap code before the final `exec`. The artifact revision should explicitly say that raw helpers remain shell wrappers and invoke the shared Python launch-policy entrypoint before launching the tool.

5. **Version detection mechanism is not specified.** Task 2.2 says "implement tool-version detection" but no artifact describes how the version is obtained. The codebase currently has zero version-detection code. Key questions: Does the system run `claude --version` / `codex --version` as a subprocess? Does it parse version from a known file in the tool's home? Does it cache the result? How does it handle tools that are not installed or that output unexpected version formats? The `brain-launch-runtime/spec.md` says "the runtime SHALL resolve that policy against the actual installed CLI tool version" but does not specify the detection mechanism.

> **DECISION: _(accept / refine / reject / defer this finding)_**
> Rationale: Accept. The launch contract should specify subprocess-based version detection against the actually installed CLI (for example `claude --version` / `codex --version`), plus tool-specific parsing and explicit failure when the executable is missing or the version output cannot be parsed in unattended mode.

6. **Idempotent Claude state patching needs explicit field-ownership documentation.** Design Decision 6 says "update only strategy-owned portions when state already exists" and "preserve unrelated prior/template-derived state." The `claude-cli-noninteractive-startup/spec.md` adds scenarios for this. However, neither artifact lists which specific `.claude.json` keys are strategy-owned vs. unmanaged. Currently `claude_bootstrap.py` manages `hasCompletedOnboarding`, `numStartups`, and `customApiKeyResponses` (lines 104-112). The spec should explicitly enumerate strategy-owned keys (or define a convention for declaring ownership per strategy entry) so that future Claude version updates don't accidentally merge-conflict with unrelated fields.

> **DECISION: _(accept / refine / reject / defer this finding)_**
> Rationale: Refine. Explicit field ownership is needed, but the ownership should be strategy-specific rather than a single static Claude key list because different Claude versions may require different state paths. The next artifact revision should require each Claude strategy to declare the exact JSON/settings paths it owns.

7. **`gemini_headless` backend is silently excluded.** `BackendKind` (`models.py:16-23`) includes `gemini_headless`, and the design's Non-Goal 4 says "initial implementation may focus on the currently exercised providers while keeping the registry extensible." But Gemini is never mentioned in registry specs, design decisions, or tasks — not even as an explicit deferral. Task 3 only covers Codex (3.1) and Claude (3.2). The design should add one sentence acknowledging Gemini as deferred and confirming it will continue to launch in interactive/default mode without registry involvement.

> **DECISION: _(accept / refine / reject / defer this finding)_**
> Rationale: Accept. Gemini should be called out explicitly as deferred so the first implementation scope is unambiguous and users do not infer silent registry coverage for `gemini_headless`.

8. **No task covers `houmao_server_rest` or `cao_rest` backend integration.** Design Decision 5 says "CAO-backed runtime startup" shares the same engine, and the proposal Impact says "shared demo launch code" is affected. But there is no task for wiring the strategy engine into the `cao_rest.py` backend (2200+ lines) or the `houmao_server_rest` path. If these backends rely on the external CAO server to handle unattended behavior, that delegation model should be documented. If not, they need tasks.

> **DECISION: _(accept / refine / reject / defer this finding)_**

The reviewer is right that CAO-backed paths need explicit coverage, but `houmao_server_rest` is built on the same CAO-compatible session core, so one design/task update for CAO-backed launch-policy application should cover both `cao_rest` and `houmao_server_rest` instead of treating them as two unrelated integrations.

The strategy engine's "apply actions before provider start" must integrate at the runner level, between launch-plan composition and process start. ## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should `operator_prompt_mode = unattended` cover only startup-blocking prompts, or also mid-turn approval flows?

- Decision: `Deferrable`

#### Why this matters

The current Claude `--dangerously-skip-permissions` flag suppresses both startup prompts and mid-turn tool approval. The current Codex `approval_policy = "never"` suppresses mid-turn approval. If `unattended` only covers startup, the existing `approval_policy`/`--dangerously-skip-permissions` injection must remain as separate concerns. If it covers both, the strategy engine becomes responsible for a broader surface.

#### Examples

- Startup-only: The strategy seeds workspace trust and onboarding state but does not inject `--dangerously-skip-permissions`. Claude would still ask before running dangerous commands mid-turn.
- Full-unattended: The strategy also injects the permissions flag and sets `approval_policy = "never"`, covering both startup and runtime approval.

#### Why this question arises in code

- `claude_bootstrap.py:92` validates `skipDangerousModePermissionPrompt` in `settings.json` — this is a runtime approval concern, not just startup.
- `codex_bootstrap.py:59-66` copies `approval_policy` from config profile — this is a mid-turn approval concern.
- The design's term "operator-facing trust or permission prompts" is ambiguous about whether "permission prompts" includes mid-turn tool approval or only first-run setup.

#### Options

**Option A: Startup-only scope — `unattended` covers only first-run/session-start blocking surfaces.**

- Code impact:
  - Strategy engine only manages workspace trust, onboarding, and bootstrap state.
  - Mid-turn approval (`approval_policy`, `--dangerously-skip-permissions`) remains managed by config profiles and `launch_args_override` as today.
- User-facing difference:
  - Users must still configure mid-turn approval separately per tool.
  - `unattended` is a narrower, more predictable guarantee.
- Pros:
  - Simpler first implementation.
  - Does not conflate two distinct trust surfaces.
- Cons:
  - Users still need to know about tool-specific approval config.
  - "Unattended" is misleading if the session can still block mid-turn.

**Option B: Full-unattended scope — `unattended` suppresses both startup and mid-turn operator prompts.**

- Code impact:
  - Strategy actions must also inject `--dangerously-skip-permissions` (Claude) and `approval_policy = "never"` (Codex).
  - `settings.json` validation in `claude_bootstrap.py` becomes strategy-owned.
  - Config-profile-based `approval_policy` injection in `codex_bootstrap.py:59-66` moves into the strategy engine.
- User-facing difference:
  - Single intent captures the full "never ask me anything" posture.
  - Existing config profile overrides for approval may conflict with strategy-injected values.
- Pros:
  - "Unattended" means what users expect.
  - Eliminates the remaining tool-specific override patterns.
- Cons:
  - Broader scope means more actions per strategy, more version-specific divergence.
  - Harder to test exhaustively.

> **PROPOSED: Option B — full-unattended scope covering both startup and mid-turn prompts.**
> Rationale: The design's stated goal is "forbid operator prompts," and both `claude_bootstrap.py` (line 92, `skipDangerousModePermissionPrompt`) and `codex_bootstrap.py` (lines 59-66, `approval_policy`) already manage mid-turn approval as part of non-interactive bootstrap. Moving these into the strategy engine is a natural consolidation. Leaving mid-turn approval out would make `unattended` a half-measure that still requires per-tool config knowledge.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Reject PROPOSED. Choose Option A for v1: startup-only scope. The current artifacts were revised after this review to scope `operator_prompt_mode = unattended` to startup-blocking operator prompts, while explicitly leaving mid-turn approval behavior as provider-specific follow-up work. That keeps the contract honest and aligned with the fresh-home blockers we actually observed.

### Q2) Should the registry data format be YAML files under a well-known directory, or Python-declared registry tables?

- Decision: `Blocking`

#### Why this matters

The design says "descriptive registry entries stored as repo-owned data files" but does not commit to a format. This is a blocking architectural choice that affects how strategies are authored, validated, and tested.

#### Examples

- YAML approach: A file like `src/houmao/agents/launch_policy/registry/claude-unattended-v1.yaml` defining strategy metadata, version ranges, and action lists.
- Python approach: A module like `src/houmao/agents/launch_policy/registry.py` with frozen dataclass entries registered in a list/dict.

#### Why this question arises in code

- The existing codebase uses YAML for external-facing declarative config (brain recipes, tool adapters) but Python dataclasses for internal value objects.
- The design says "most compatibility rules auditable in data" but the action vocabulary and hook references are inherently code-coupled.

#### Options

**Option A: YAML files in a well-known directory.**

- Code impact:
  - Need a YAML schema, loader, and validator.
  - Hook references must use string identifiers mapped to Python functions.
  - Registry entries are decoupled from code.
- User-facing difference:
  - New strategies can be added by dropping a YAML file (but hooks still need code).
- Pros:
  - Auditable without reading Python.
  - Separates version-range data from execution logic.
- Cons:
  - Indirection between YAML action names and Python implementations.
  - Harder to type-check and refactor.
  - Adds a schema/loader/validator layer.

**Option B: Python-declared registry as frozen dataclass entries in a module.**

- Code impact:
  - Registry is a Python module with a list/dict of `StrategyEntry` dataclasses.
  - Hooks are direct function references — no string-to-function mapping needed.
  - Standard `mypy --strict` catches type errors.
- User-facing difference:
  - Adding a strategy requires editing Python, not dropping a file.
- Pros:
  - Type-safe and refactorable.
  - Consistent with existing internal patterns (frozen dataclasses, no external YAML for internal logic).
  - Hooks are first-class Python references.
  - No loader/validator layer needed.
- Cons:
  - Version-range data mixed with code (less auditable at a glance).
  - Adding strategies requires a code change, not a config change.

> **PROPOSED: Option B — Python-declared registry with frozen dataclass entries.**
> Rationale: The existing codebase uses YAML only for user-facing declarative artifacts (brain recipes at `tests/fixtures/agents/brains/brain-recipes/`, tool adapters). Internal value objects are frozen dataclasses (`brain_builder.py:35-94`, `models.py:33-98`). The registry is runtime-internal logic, not user-facing config — strategy entries reference Python hook functions and have complex resolution semantics that benefit from `mypy --strict` coverage. A YAML layer would add loader/validator complexity without clear benefit, since adding a new strategy already requires writing the Python hook code. Keep the data auditable by using clear, well-named dataclass entries with explicit `version_range` and `actions` fields.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Reject PROPOSED. Keep the hybrid direction: repo-owned data entries plus a Python action engine and provider-specific hooks. The remaining artifact gap is to pin the concrete serialized entry schema and hook/action contract, not to collapse the design into a Python-only registry.

### Q3) Where should the strategy resolution and action engine live in the module tree?

- Decision: `Blocking`

#### Why this matters

The engine must be importable by both `brain_builder.py` (build-time, to validate policy) and `backends/headless_runner.py` (launch-time, to apply strategy). Its placement affects import layering and circular dependency risk.

#### Examples

- Under `realm_controller/`: `src/houmao/agents/realm_controller/launch_policy/` — co-located with launch_plan and backends.
- Under `agents/`: `src/houmao/agents/launch_policy/` — sibling to `brain_builder.py` and `realm_controller/`.

#### Why this question arises in code

- `brain_builder.py` lives at `src/houmao/agents/brain_builder.py`.
- `launch_plan.py` lives at `src/houmao/agents/realm_controller/launch_plan.py`.
- The engine is consumed by both layers, so it must be importable from both without circular deps.

#### Options

**Option A: `src/houmao/agents/launch_policy/` — sibling to `brain_builder.py`.**

- Code impact:
  - Both `brain_builder.py` and `realm_controller/` can import from `agents.launch_policy` without circular deps.
  - Clean layering: `agents.launch_policy` is a shared utility.
- Pros:
  - Avoids nesting inside `realm_controller` while being consumed by it.
  - Build-time code in `brain_builder.py` can import policy types without pulling in runtime backend code.
- Cons:
  - New package at `agents/` level increases top-level module count.

**Option B: `src/houmao/agents/realm_controller/launch_policy/` — nested inside realm_controller.**

- Code impact:
  - `brain_builder.py` would need to import from `realm_controller.launch_policy`, creating a build→runtime import direction.
- Pros:
  - Co-located with the launch plan and backends that consume it most.
- Cons:
  - `brain_builder.py` importing from `realm_controller/` reverses the current dependency direction (runtime imports build artifacts, not vice versa).

> **PROPOSED: Option A — `src/houmao/agents/launch_policy/` as a sibling package.**
> Rationale: Currently `brain_builder.py` is at `src/houmao/agents/` and does not import from `realm_controller/`. The `realm_controller/` layer imports brain manifests (dict) but not the builder module. Placing the policy engine under `realm_controller/` would force `brain_builder.py` to import from a downstream layer, reversing the dependency direction. A sibling `agents/launch_policy/` package keeps the import graph clean: `brain_builder` → `launch_policy` ← `realm_controller`.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Accept PROPOSED. A sibling package under `src/houmao/agents/launch_policy/` keeps the dependency direction clean: build-time code and runtime code can both import it without making `brain_builder.py` depend on `realm_controller/`.

### Q4) What is the narrowest useful override surface for controlled experiments with unsupported versions?

- Decision: `Deferrable`

#### Why this matters

Design Decision 4 says "the design may still allow an explicit developer override" but does not specify the mechanism. The open question in the design also asks this. Pinning it now avoids ad-hoc workarounds later.

#### Examples

- A developer has Claude Code v3.0 (not yet in the registry) and wants to test whether the v2.x strategy still works.
- A CI environment pins a specific version for reproducibility.

#### Why this question arises in code

- The fail-closed behavior means any new upstream auto-update can block all unattended launches until the registry is updated.
- The override must be explicit and noisy (per Design Decision 4) but the mechanism (env var, manifest field, CLI flag) is unspecified.

#### Options

**Option A: Pinned strategy ID via an environment variable (e.g., `HOUMAO_LAUNCH_POLICY_OVERRIDE_STRATEGY`).**

- Code impact:
  - Resolution engine checks env var before version-based lookup.
  - Override is per-process, not persisted in manifest.
- User-facing difference:
  - Quick escape hatch for testing; does not require manifest rebuild.
- Pros:
  - Simple, non-persistent, easy to audit in logs.
  - Does not pollute manifest or recipe schema.
- Cons:
  - Per-process only; not transferable across machines.

**Option B: Pinned strategy ID as an optional field in `BuildRequest` / brain recipe.**

- Code impact:
  - New field `launch_policy_strategy_override: str | None` in `BuildRequest` and recipe YAML.
  - Persisted in manifest, used at launch time.
- User-facing difference:
  - Override is declarative and reproducible.
- Pros:
  - Explicit, reviewable in recipe YAML.
  - Persisted for debugging.
- Cons:
  - Adds a field that should rarely be used.
  - Risk of override persisting longer than intended.

> **PROPOSED: Option A — environment variable override, not persisted in manifest.**
> Rationale: The override is explicitly described as a controlled-experiment mechanism, not a production feature. An env var is the simplest, noisiest (logged in provenance), and most transient form. It avoids polluting the recipe/manifest schema with a field that should rarely be used. The resolution engine can log a warning when the override is active, satisfying the "noisy provenance" requirement from Design Decision 4.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: Accept PROPOSED. An environment-variable override is the narrowest useful controlled-experiment surface: transient, noisy, non-persistent, and easy to surface in diagnostics without polluting recipes or manifests.
