## Summary

- **Solid overall structure**: the proposal–design–specs–tasks chain is internally coherent, and the manifest-first adoption philosophy aligns well with the existing runtime model where every control path resolves through a persisted session manifest.
- **Good failure-closed posture**: the design explicitly enumerates all join failure cases and the specs carry them through as testable scenarios.
- **Risk: manifest schema version impact is unaddressed**: extending `SessionManifestAgentLaunchAuthorityV1` with session-origin and relaunch-posture fields likely requires a manifest schema bump (v4→v5), JSON schema regeneration, and `test_schema_consistency` updates — none of which appear in the tasks.
- **Risk: placeholder brain manifest must satisfy downstream readers**: `build_launch_plan()` enforces brain manifest schema v2 and several runtime paths read brain manifest fields; the "placeholder" contract is underspecified.
- **Risk: the relaunch posture model shape is not yet decided**: four named postures are described in prose but neither the specs nor design specify whether this is a discriminated union, an enum + optional fields, or inferred from field presence — this decision cascades into serialization, schema, and every runtime consumer.

## Findings

None of the 10 tasks mention a schema version bump, schema regeneration, or consistency test update. If this is missed, CI will break on the first PR that touches the boundary model.

> **DECISION: Accept.**
> Rationale: This finding is still valid. Even if we keep manifest v4, any boundary-model extension here requires packaged schema regeneration and explicit `test_schema_consistency` coverage. The current tasks should call that work out directly in a later artifact revision pass.

2. **Placeholder brain manifest structural contract is unspecified.** The design (Decision 1) and the `brain-launch-runtime` delta spec both say "placeholder `brain_manifest.json`" without defining what fields it must contain. Multiple runtime paths read brain manifest fields to resolve tool name, home path, env var names, and profile info (`launch_plan.py:60-195`). If the join path materializes a structurally incomplete brain manifest, those readers will raise `KeyError` or validation errors. The specs should define the minimum required fields for a join-originated placeholder brain manifest, or the runtime code must be given a join-aware fallback path.

> **DECISION: Accept.**
> Rationale: This gap remains. The current artifacts still describe a placeholder brain manifest without yet stating the runtime-safe structural contract. The decided direction below is to build join-derived launch-plan state directly and keep the placeholder brain manifest out of runtime behavioral truth.

### Should Fix / Important

This is a design-level decision that should be resolved before task 2.2 begins, because it affects boundary models, JSON schemas, manifest writing, and every relaunch consumer.

> **DECISION: Refine.**
> Rationale: The concern was valid, but it is now partly addressed by the current artifacts. The design explicitly names `posture_kind`, and the runtime spec now carries structured launch-option postures. > **DECISION: Accept.**
> Rationale: Task 2.3 is still too broad for clean implementation planning. A later artifact revision should split adoption-construction work from post-construction control wiring while leaving the exact implementation shape (`from_joined_session()` factory, flag, or subclass) to the implementation design.

If the lease expires, registry cleanup (`houmao-mgr admin cleanup-registry`) will remove the record even though the session is still alive. The design or specs should address this: either the join command starts a lightweight lease-renewal daemon, or joined sessions use a sentinel lease value, or the operator is expected to run a companion process.

> **DECISION: Accept.**
> Rationale: The lease issue is still real for a one-shot join command. The decision below is to use a long sentinel lease for joined sessions rather than background renewal, and the artifacts should be revised later to state that explicitly.

## Design Alignment With Existing System

- **Manifest-first control model**: The design correctly identifies that all existing control paths (state, show, prompt, interrupt, gateway attach, relaunch) resolve through a persisted session manifest, and explicitly commits to materializing the same envelope for joined sessions. This is the right call — inventing a join-only lightweight record would create a parallel discovery path that every consumer would need to handle.

- **CLI patterns in `srv_ctrl/commands/agents/`**: The existing commands in `core.py` consistently use `@managed_agent_selector_options` (providing `--agent-id` / `--agent-name`) and `@pair_port_option()`. The proposed `join` command adds `--provider`, `--launch-cmd`, `--working-directory`, `--headless`, and `--resume-id`. The identity options (`--agent-name`, optional `--agent-id`) align with existing patterns. The join-specific options are structurally new but follow the same Click decorator style.

- **`LocalInteractiveSession` construction**: The existing constructor (`local_interactive.py:43-71`) assumes a `LaunchPlan` is always available and uses it to spawn the provider. The join path needs to construct a `LocalInteractiveSession` (or equivalent) without spawning, which is a construction-path extension, not a refactor of the existing model. The codebase already has `HeadlessInteractiveSession` as a base class that `LocalInteractiveSession` extends, so a `@classmethod` factory like `from_joined_session()` would align with the repo's preference for zero-arg constructors + factory classmethods (`magic-context/instructions/python-coding-guide.md`).

- **Registry publication**: The existing `LiveAgentRegistryRecordV2` model and `registry_storage.py` write/read pattern are well-defined. The join path can reuse `write_registry_record()` directly. The only gap is the lease renewal lifecycle (see finding 5).

- **Gateway artifact materialization**: `ensure_gateway_capability()` in `gateway_storage.py:212-256` already creates the full gateway directory structure from a manifest handle. The join path can call it after writing the session manifest, which aligns with the existing materialization order.

- **Tmux env publishing**: The existing pattern in `headless_base.py:363-387` publishes env vars via `set_tmux_session_environment_shared()`. The join path should use the same helper but must explicitly include `AGENTSYS_AGENT_ID` and `AGENTSYS_JOB_DIR` which are currently published by separate code paths in `runtime.py`.

- **Schema consistency testing**: The `test_schema_consistency.py` test regenerates schemas from models and compares against packaged `.schema.json` files. Any boundary model extension must be followed by schema regeneration. This is a well-established pattern but is not mentioned in the change tasks.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should extending `SessionManifestAgentLaunchAuthorityV1` require a session manifest schema version bump to v5?

- Decision: `Blocking`

#### Why this matters

- Schema version determines which manifest reader path is used. If the authority model gains new required fields or changes its shape, older v4 manifests without those fields will fail validation unless the changes are backward-compatible (all new fields have defaults).
- `test_schema_consistency` will fail if the packaged JSON schema doesn't match the updated model.

#### Examples

- Adding a required `session_origin` field without a default to `SessionManifestAgentLaunchAuthorityV1` would make all existing v4 manifests fail re-validation.
- Adding an optional `relaunch_posture` field with a `None` default would be backward-compatible within v4.

#### Why this question arises in code

- `SessionManifestPayloadV4` (`boundary_models.py:687-800`) includes `agent_launch_authority: SessionManifestAgentLaunchAuthorityV1 | None = None` — the authority itself is already optional at the manifest level.
- `_resolve_manifest_relaunch_authority()` (`runtime.py:1450-1486`) reads the authority from the manifest and must handle any new fields.
- The packaged schema `session_manifest.v4.schema.json` must be regenerated if the model changes.

#### Options

**Option A: Keep v4 with backward-compatible extensions only (all new fields optional with defaults).**

- Code impact:
  - Add optional fields (`session_origin: str | None = None`, `relaunch_posture: RelaunchPosture | None = None`) to `SessionManifestAgentLaunchAuthorityV1`.
  - Regenerate `session_manifest.v4.schema.json`.
  - Existing v4 manifests remain valid — new fields are absent but default to `None`.
- User-facing difference:
  - No manifest migration needed. Existing sessions continue to work.
- Pros:
  - Simpler, no manifest reader branching.
  - All existing v4 manifests stay valid.
- Cons:
  - The authority model carries optional fields that are always present for joined sessions and always absent for native launches, which is a mild type-safety loss.

**Option B: Bump to v5 with the new authority shape as a required section for joined sessions.**

- Code impact:
  - Create `SessionManifestPayloadV5` with a richer `agent_launch_authority` model.
  - Add v5 schema, update `SESSION_MANIFEST_SCHEMA_VERSION`, update consistency test.
  - Add a v4→v5 manifest reader fallback in `_resolve_manifest_relaunch_authority()`.
- User-facing difference:
  - Existing sessions stay on v4 and continue working; joined sessions are v5.
- Pros:
  - Cleaner type boundary: v5 authority model can enforce joined-session fields.
- Cons:
  - More code: reader branching, new schema file, migration consideration for tooling that reads manifests.

> **PROPOSED: Option A — keep v4 with backward-compatible optional extensions.**
> Rationale: The `agent_launch_authority` field is already `| None` at the manifest level (`boundary_models.py:~780`), so the authority model itself is naturally optional. Adding `session_origin` and `relaunch_posture` as optional fields with `None` defaults keeps all existing v4 manifests valid, avoids reader branching, and matches the repo pattern where authority sections grow incrementally (e.g., `session_id` and `profile_name` were added to the same model without a version bump). The packaged JSON schema still needs regeneration and the consistency test must pass, but no v5 is needed.
>
> **DECISION: Accept Option A, refined to the current tagged launch-options model.**
> Rationale: We should stay on manifest v4 and extend it backward-compatibly. The current artifacts now point to optional tagged relaunch metadata such as `posture_kind`, structured `launch_args`, structured `launch_env`, and join-origin metadata, all of which can be added as optional/defaulted fields without forcing a v5 reader split. This still requires schema regeneration and `test_schema_consistency` coverage, but not a manifest version bump.

### Q2) How should the placeholder brain manifest satisfy downstream readers?

- Decision: `Blocking`

#### Why this matters

- `build_launch_plan()` reads brain manifest fields (`tool`, `home_path`, `env_var_names`, `executable`, `args`, etc.) and enforces schema v2.
- If the join path bypasses the brain builder entirely, the placeholder brain manifest must either satisfy the same schema or the join path must avoid calling `build_launch_plan()` altogether.

#### Examples

- A joined TUI session that later needs `relaunch` would go through `_resolve_manifest_relaunch_authority()` → relaunch backend, which may or may not re-read the brain manifest depending on the backend.
- A `gateway attach` for a joined session calls `ensure_gateway_capability()` which reads the session manifest but not the brain manifest directly.

#### Why this question arises in code

- `build_launch_plan()` (`launch_plan.py:34`) asserts brain manifest schema version.
- `_build_manifest_agent_launch_authority()` (`manifest.py:535-553`) extracts tool, backend, working dir, etc. from the `LaunchPlan` which was built from the brain manifest.
- The join path creates no `LaunchPlan` through the normal flow.

#### Options

**Option A: Materialize a structurally valid but minimal brain manifest v2 with join-derived fields.**

- Code impact:
  - The join materialization path writes a brain manifest with `schema_version: 2`, `tool`, `home_path`, `env_var_name`, and empty/placeholder values for fields the join path doesn't own (like `skills`, `config_profile`).
  - Downstream readers that parse the brain manifest get valid but sparse data.
- User-facing difference:
  - `houmao-mgr agents show` would display a brain manifest that looks mostly empty.
- Pros:
  - All existing brain manifest readers work unchanged.
- Cons:
  - The placeholder content is partially fictitious — it claims the brain was "built" when it wasn't.

**Option B: Skip brain manifest materialization; have the join path construct a `LaunchPlan`-equivalent directly from join inputs.**

- Code impact:
  - The join path constructs a minimal `LaunchPlan` (or `LaunchPlan`-like dataclass) directly from `--provider`, `--launch-cmd`, detected working directory, etc., without going through `build_launch_plan()`.
  - The session manifest's `launch_plan` section is populated from this join-derived plan.
  - A placeholder brain manifest is still written (for path existence) but is never read by runtime code.
- User-facing difference:
  - Same as Option A for operators; internally cleaner because the code doesn't pretend a brain was built.
- Pros:
  - Honest: the join path doesn't claim build-phase artifacts it doesn't have.
  - Less risk of downstream readers misinterpreting sparse brain manifest fields.
- Cons:
  - Needs a new join-specific `LaunchPlan` construction path that duplicates some of `build_launch_plan()`'s field assembly.

> **PROPOSED: Option B — construct a join-derived `LaunchPlan`-equivalent directly, write a path-placeholder brain manifest that is not read by runtime code.**
> Rationale: The design already states (Decision 6) that "joined sessions get control-plane parity, not retroactive build-time parity" and that "placeholder role and brain-manifest artifacts exist only to satisfy current manifest and controller invariants." A join-specific `LaunchPlan` factory (e.g., `LaunchPlan.for_joined_session(...)`) that takes provider, backend, working directory, and structured launch options would be a cleaner boundary than trying to fabricate a brain manifest that passes schema v2 validation. The existing `LaunchPlan` is a frozen dataclass (`models.py:57-101`) and adding a `@classmethod` factory aligns with the repo pattern of zero-arg constructors plus factory classmethods.
>
> **DECISION: Accept Option B, refined to the structured launch-options model.**
> Rationale: The runtime should construct join-derived launch-plan state directly from provider, backend, working directory, structured `--launch-args`, structured `--launch-env`, and headless resume-selection state. The placeholder brain manifest may still exist for path and invariant purposes, but runtime control paths should not depend on it for behavioral truth.

### Q3) Should the relaunch posture be a discriminated union with an explicit type tag, or inferred from field presence?

- Decision: `Blocking`

#### Why this matters

- The relaunch posture determines what `houmao-mgr agents relaunch` does for a joined session. If the model shape is ambiguous, the relaunch dispatch code must guess which posture applies, leading to fragile conditionals.
- The choice also affects JSON schema generation and the `test_schema_consistency` contract.

#### Examples

- A joined TUI without `--launch-cmd` should have posture `unavailable`. If this is inferred from "launch_cmd is None and session_origin is join", the inference logic is spread across multiple fields.
- A joined headless session should have posture `headless_prefix_command` with both `launch_cmd` and `resume_id` present. If we later add a fifth posture, inference-based dispatch becomes increasingly fragile.

#### Why this question arises in code

- `_resolve_manifest_relaunch_authority()` (`runtime.py:1450-1486`) currently checks for `agent_launch_authority` presence and reads its fields. Adding 4 postures without a discriminator means this function needs a multi-field inference chain.
- `_relaunch_backend_session()` (`runtime.py:1489-1507`) dispatches across backend types and would need to further dispatch on posture.

#### Options

**Option A: Discriminated union with explicit `posture_kind` literal field.**

- Code impact:
  ```python
  RelaunchPostureKind = Literal[
      "runtime_launch_plan",
      "literal_tui_command",
      "headless_prefix_command",
      "unavailable",
  ]
  # Added to SessionManifestAgentLaunchAuthorityV1:
  relaunch_posture_kind: RelaunchPostureKind | None = None
  relaunch_literal_command: str | None = None
  relaunch_resume_id: str | None = None
  ```
- User-facing difference:
  - `relaunch` error messages can say "relaunch posture is 'unavailable'" rather than inferring it.
- Pros:
  - Explicit, exhaustive `match` dispatch in runtime code.
  - Serializes cleanly to JSON schema with enum constraint.
  - Easy to add future postures.
- Cons:
  - Slightly more verbose model.

**Option B: Infer posture from field presence (no explicit tag).**

- Code impact:
  - Add `session_origin`, `relaunch_literal_command`, `relaunch_resume_id` as optional fields.
  - Runtime infers posture: if `session_origin != "join"` → `runtime_launch_plan`; if join + `relaunch_literal_command` + no `resume_id` → `literal_tui_command`; etc.
- User-facing difference:
  - Same behavior, but inference errors are possible.
- Pros:
  - Fewer fields in the model.
- Cons:
  - Fragile inference chain; adding a fifth posture requires updating all inference sites.

> **PROPOSED: Option A — discriminated union with explicit `relaunch_posture_kind` literal field.**
> Rationale: The repo already uses `Literal` types extensively for discriminated dispatch (e.g., `BackendKind` in `models.py:17-25`, `RegistryTerminalKind` in `registry_models.py`). An explicit `RelaunchPostureKind` literal follows that pattern, gives exhaustive `match` dispatch in `_resolve_manifest_relaunch_authority()` and `_relaunch_backend_session()`, and serializes cleanly to JSON schema. The four postures are architecturally distinct (not just presence/absence of data) so they deserve an explicit discriminator.
>
> **DECISION: Accept Option A, refined to the current posture names and structured fields.**
> Rationale: We should use an explicit discriminator rather than field-presence inference. The current artifacts already lean in that direction via `posture_kind`, and the concrete tagged variants should match the structured launch-options model now in the change (`runtime_launch_plan`, `tui_launch_options`, `headless_launch_options`, `unavailable`) rather than the earlier command-string terminology used in this review.

### Q4) What is the lease renewal strategy for joined session registry records?

- Decision: `Deferrable`

#### Why this matters

- `LiveAgentRegistryRecordV2` requires `lease_expires_at` (validated at `registry_models.py:211-212`). For native launches, the `RuntimeSessionController` stays resident and can renew leases. For `join`, the CLI command is a one-shot invocation that exits after publishing.
- If the lease expires, `houmao-mgr admin cleanup-registry` will remove the record even though the joined session is still alive.

#### Examples

- Operator joins a Claude Code TUI, walks away for 2 hours. If the default lease is 30 minutes, the record is eligible for cleanup even though the TUI is still running.
- Operator joins a headless session and runs `houmao-mgr agents turn submit` an hour later — if the record was cleaned up, the turn submit can't discover the agent.

#### Why this question arises in code

- `write_registry_record()` sets `lease_expires_at` from a caller-provided value.
- `cleanup_registry()` removes records past their lease.
- No background renewal process exists for one-shot CLI commands.

#### Options

**Option A: Use a long sentinel lease (e.g., 30 days) for joined sessions.**

- Code impact:
  - The join command sets `lease_expires_at` far in the future.
  - Cleanup is operator-initiated via `houmao-mgr agents stop` or explicit removal.
- User-facing difference:
  - Joined sessions appear in the registry indefinitely until explicitly stopped.
- Pros:
  - Simple, no background process needed.
- Cons:
  - Stale records accumulate if operators forget to stop.

**Option B: Start a lightweight background lease-renewal thread or daemon.**

- Code impact:
  - The join command spawns a background process that periodically refreshes the lease.
  - Requires lifecycle management (PID file, cleanup on stop).
- User-facing difference:
  - Lease stays fresh automatically; cleanup works normally.
- Pros:
  - Registry hygiene is automatic.
- Cons:
  - Adds process management complexity to a one-shot CLI command.

**Option C: Defer lease renewal to the first control command that touches the joined session.**

- Code impact:
  - `resolve_managed_agent_target()` refreshes the lease whenever it resolves a joined session.
  - The initial join sets a moderate lease (e.g., 2 hours).
- User-facing difference:
  - Joined sessions that are actively used stay alive; unused ones expire.
- Pros:
  - No background process; lease tracks actual usage.
- Cons:
  - If no control command runs within the lease window, the record expires even though the session is alive.

> **PROPOSED: Option A — use a long sentinel lease for joined sessions.**
> Rationale: The join command is explicitly a one-shot adoption flow (design context, paragraph 3). Adding a background daemon contradicts the one-shot contract. Option C is fragile because operators may not interact with the joined session within the lease window. A long sentinel lease (e.g., 30 days or a year) is simple and honest — the joined session record persists until the operator explicitly stops it or runs cleanup. The existing `houmao-mgr agents stop` already removes registry records, which provides the intended cleanup path. Stale record accumulation is mitigated by the fact that `cleanup-registry` is already an operator-initiated maintenance command.
>
> **DECISION: Accept Option A.**
> Rationale: Joined sessions are a one-shot adoption flow, so adding a lease-renewal daemon would blur ownership and add unnecessary lifecycle machinery. The cleaner v1 contract is a long sentinel lease for joined records, with explicit `agents stop` or operator cleanup as the removal path. Opportunistic lease refresh on later control commands is acceptable as an implementation detail, but the base contract should not require a resident renewal process.
