## Summary

- **Solid framing:** The five design decisions are well-motivated, mutually consistent, and each maps cleanly to either a delta spec requirement or a documentation task. The distinction between "forward-compatible reads" and "intentionally limited launches" is the right boundary.
- **Clean delta specs:** Both delta specs are additive to their main-spec counterparts and do not introduce contradictions. The new scenarios (unknown provider parsing, workdir pass-through, shadow_only rejection for unsupported parsers) test exactly the new behavioral contract.
- **Existing code confirms the direction:** `_CaoModel` already sets `extra="ignore"`, the `_CAO_PROVIDER_BY_TOOL` dict already maps by string value, and `create_terminal()` already accepts `provider: str` at the call boundary — the enum narrowing is localized to the Pydantic model field on `CaoTerminal`.
- **Tasks are under-specified for docs and test impact:** Task group 2 does not list target files, and task 2.3 does not call out the large test file (`test_cao_client_and_profile.py`, ~90KB) that likely asserts `CaoProvider` enum values extensively.
- **Open question on `CaoProvider` enum removal scope:** The enum currently contains `Q_CLI` and `KIRO_CLI` entries that are not launch-supported — the proposal doesn't address whether these survive as string constants, are deleted, or are left for call-site discovery.
- **Missing spec clause for shadow parser support predicate:** The delta brain-launch-runtime spec adds a `shadow_only` rejection scenario but does not codify *how* the runtime determines "has a shadow parser family" beyond the existing default mapping.

## Findings

### Must Fix / Blocking

1. **Delta brain-launch-runtime spec omits the mechanism for determining shadow parser availability.**

   The new scenario (line 84-86 of the delta spec) says: "`shadow_only` is rejected when no runtime shadow parser exists." But the spec does not define how the runtime determines this. The current code relies on the `_CAO_PARSING_MODE_DEFAULT_BY_TOOL` dict in `cao_rest.py`, which implicitly encodes which tools have parsers. However, the `shadow_only` rejection scenario implies a validation step *at session start* that must happen for any tool — including future ones added to `_CAO_PROVIDER_BY_TOOL` but not to `_CAO_PARSING_MODE_DEFAULT_BY_TOOL`.

   **Suggestion:** Add a normative statement in the spec clarifying that the runtime SHALL maintain an explicit set of tools with shadow parser support and SHALL reject `shadow_only` for any tool not in that set. This gives the scenario a grounded mechanism and prevents accidental widening.

2. **Tasks do not account for call-site impact of removing `CaoProvider` enum.**

   `CaoTerminal.provider` is currently typed as `CaoProvider` (a `str, Enum`). Changing it to `str` will break any code path or test that compares against `CaoProvider.CODEX`, `CaoProvider.CLAUDE_CODE`, etc. The runtime backend (`cao_rest.py`) primarily uses the `_CAO_PROVIDER_BY_TOOL` dict with string values, so launch code is likely safe. But the test file `tests/unit/agents/realm_controller/test_cao_client_and_profile.py` (~90KB) almost certainly asserts on `CaoProvider` enum members in parsed responses.

   **Suggestion:** Add a sub-task under 1.1 that explicitly audits and updates all `CaoProvider` member references across `src/` and `tests/`, or state that the enum is *replaced* (not just loosened) and that a migration grep for `CaoProvider` is required.

### Should Fix / Important

3. **Task 2.1–2.2 should list target doc files explicitly.**

   The proposal's Impact section lists "CAO launcher/reference docs, CAO troubleshooting docs, and interactive demo operator docs/README surfaces" but the tasks don't name specific paths. The affected files (found by codebase search) are at minimum:
   - `docs/reference/cao_server_launcher.md` — contains "Working directory must be inside HOME tree" and the nested-home layout recommendation
   - `docs/reference/cao_interactive_demo.md` — frames `<launcher-home>/wktree` as a structural requirement
   - `scripts/demo/cao-interactive-full-pipeline-demo/` — demo scripts reference the wktree layout
   - Any demo README surfaces under `scripts/demo/`

   Listing these in the tasks reduces the risk of partial doc updates.

4. **The delta `cao-rest-client-contract` spec should explicitly state that `CaoProvider` enum is removed (or replaced) as the model field type.**

   The delta requirement says: "The terminal `provider` field SHALL be treated as a non-empty provider identifier string returned by CAO rather than a repo-owned closed enum." This is clear directionally but ambiguous about whether `CaoProvider` enum survives in any form (e.g., as a non-model utility type, as string constants for the two launch-supported providers, etc.).

   **Suggestion:** Add a sentence such as: "The `CaoProvider` enum type SHALL NOT be used as the Pydantic field type for terminal provider in response models." This makes the implementation intent unambiguous.

5. **Task 1.2 conflates two concerns that should be separately testable.**

   Task 1.2 says: "Keep the runtime CAO launch-time tool-to-provider mapping explicit, and add tests that parsed upstream provider ids do not automatically widen runtime launch support." This is two distinct testing goals: (a) the positive mapping for codex/claude works, and (b) parsing `kimi_cli` in a response does not create launch support for kimi. These should be separate assertions because (a) already exists and (b) is the new behavioral contract.

   **Suggestion:** Split into 1.2a (verify existing launch mapping assertions still pass) and 1.2b (add negative test: parsed provider ≠ launch support).

6. **Task 2.3 is vague about what "golden text" means in this context.**

   "Refresh any affected unit tests, fixtures, or golden text that currently assert the old provider enum surface or stale workdir guidance." Without specifying which fixtures or golden text files, an implementer could miss test expectations that embed `CaoProvider` enum values in mock response payloads or snapshot-style assertions.

   **Suggestion:** Reference the specific test file `tests/unit/agents/realm_controller/test_cao_client_and_profile.py` and any test fixtures under `tests/fixtures/` that construct `CaoTerminal` payloads.

## Design Alignment With Existing System

- **Existing pattern: `_CaoModel` base with `extra="ignore"` (`src/houmao/cao/models.py:15-18`).** The proposed forward-compatible provider string aligns with this existing tolerance for unknown response fields. The enum → string change on `CaoTerminal.provider` is the natural next step when the enum itself becomes the brittleness source.

- **Existing pattern: explicit tool-to-provider dict (`src/houmao/agents/realm_controller/backends/cao_rest.py`, `_CAO_PROVIDER_BY_TOOL`).** The design's decision 2 (keep launch mapping explicit) perfectly matches the existing code shape. No structural change needed — just preserving it and adding documentation/test coverage for the "parsing ≠ launch support" invariant.

- **Existing pattern: `CaoParsingMode` as a Literal type (`src/houmao/agents/realm_controller/models.py`).** The parsing mode is already well-typed and the delta spec's `shadow_only` rejection scenario fits the existing `resolve_cao_parsing_mode()` function in `launch_plan.py`, which already handles tool-default resolution. The gap is only that the rejection path for "tool without shadow parser" is not yet tested or explicitly specified.

- **Existing pattern: workdir pass-through without local validation (`src/houmao/agents/realm_controller/backends/cao_rest.py`, `_start_terminal`).** The code already passes `working_directory=str(self._plan.working_directory)` directly to `create_terminal()` without a home-tree check. - **Potential misalignment: `CaoProvider` enum removal scope.** The enum is imported in `src/houmao/cao/__init__.py` as a public export. Removing it or making it non-Pydantic will change the package's public API surface. The change should decide whether to deprecate (keep enum as utility constants), remove (delete entirely), or replace (use a `KNOWN_CAO_PROVIDERS` dict/set for documentation purposes). The current artifacts do not address this.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) What happens to the `CaoProvider` enum type after the provider field becomes a string?

- Decision: `Blocking`

#### Why this matters

The `CaoProvider` enum (`src/houmao/cao/models.py:21-27`) is a public export from the `houmao.cao` package (via `__init__.py`) with four members: `Q_CLI`, `KIRO_CLI`, `CLAUDE_CODE`, `CODEX`. Changing `CaoTerminal.provider` from `CaoProvider` to `str` removes the primary consumer of this enum. But string constants like `"codex"` and `"claude_code"` are still needed at launch time (`_CAO_PROVIDER_BY_TOOL`) and in tests that construct mock CAO responses. The question is whether the enum survives as a utility type, gets replaced by plain string constants, or is deleted entirely.

#### Examples

- Test code currently writes: `assert terminal.provider == CaoProvider.CODEX` → needs updating regardless
- Launch code uses: `_CAO_PROVIDER_BY_TOOL = {"codex": "codex", "claude": "claude_code"}` → already string-based, no enum dependency
- If enum is deleted, the `Q_CLI` and `KIRO_CLI` members vanish — they were never used in launch code but documented the upstream surface

#### Why this question arises in code

- `src/houmao/cao/models.py:21-27` — defines the enum
- `src/houmao/cao/models.py:80` — `CaoTerminal.provider: CaoProvider` field
- `src/houmao/cao/__init__.py` — exports `CaoProvider` as public API
- `src/houmao/agents/realm_controller/backends/cao_rest.py` — `_CAO_PROVIDER_BY_TOOL` uses string values, not enum members

```python
# Current state: enum used as Pydantic field type
class CaoTerminal(_CaoModel):
    provider: CaoProvider  # ← this becomes `str`
    # ...

# Launch code already uses strings, not enum:
_CAO_PROVIDER_BY_TOOL: Final[dict[str, str]] = {
    "codex": "codex",
    "claude": "claude_code",
}
```

#### Options

**Option A: Delete `CaoProvider` enum entirely**

- Code impact:
  - Remove class from `models.py`, remove from `__init__.py` exports
  - Change `CaoTerminal.provider` type to `str` (with a Pydantic `min_length=1` validator for the "non-empty" requirement)
  - Grep for `CaoProvider` across `src/` and `tests/` and replace all references with string literals
- User-facing difference:
  - Cleaner API surface; no misleading enum suggesting a closed set
- Pros:
  - Simplest; no vestigial type to maintain
  - Aligns with the design's "open string" intent
- Cons:
  - Loses documentation value of `Q_CLI`, `KIRO_CLI` as known upstream identifiers
  - Magic string literals in tests unless a constant dict is introduced

**Option B: Keep `CaoProvider` as a non-model utility enum or constant set**

- Code impact:
  - Keep the enum (or replace with `KNOWN_CAO_PROVIDERS: Final[set[str]] = {...}`) for documentation/logging
  - Change `CaoTerminal.provider` to `str` with `min_length=1`
  - Update call sites that compare against enum members to compare against strings or the constant set
- User-facing difference:
  - Known providers remain discoverable in code; new providers just aren't listed until someone adds them
- Pros:
  - Preserves discoverability of the current upstream provider vocabulary
  - Utility type can be used in logging/diagnostics ("known provider" vs "unknown provider")
- Cons:
  - Keeping a non-model enum that tracks upstream vocabulary partially recreates the maintenance burden the change is trying to remove
  - Dual representation (enum for docs, string for parsing) can confuse contributors

> **PROPOSED: Option A — delete `CaoProvider` enum entirely.**
> Rationale: The launch-time mapping (`_CAO_PROVIDER_BY_TOOL` in `cao_rest.py`) already uses plain strings and is the only place where provider identity matters operationally. The `Q_CLI`/`KIRO_CLI` entries were added to keep parsing from failing, not because any runtime code uses them. Deleting the enum fully delivers on the "open string" intent and avoids a vestigial type that could re-attract maintenance churn. A `min_length=1` Pydantic validator on the `str` field satisfies the "non-empty" spec requirement. If future diagnostics need a "known provider" check, a `frozenset` constant in `cao_rest.py` (co-located with the launch mapping) is cheaper than a standalone enum.
>
> **DECISION: Accept PROPOSED Option A — delete `CaoProvider` enum entirely.**
> Rationale: This change is explicitly about removing the repo-owned closed provider surface, and the runtime launch path already uses plain strings in `_CAO_PROVIDER_BY_TOOL` rather than enum members. Because this repository currently prioritizes clarity over backward compatibility, keeping `CaoProvider` as a public utility enum would preserve exactly the stale vocabulary-tracking burden the change is meant to remove. If we later need a curated set of known identifiers for diagnostics, that should be a small local constant near the launch mapping, not a public exported enum.

### Q2) How should the runtime determine "tool has a shadow parser family" for the `shadow_only` rejection scenario?

- Decision: `Blocking`

#### Why this matters

The delta brain-launch-runtime spec (scenario at line 84-86) requires the runtime to reject `shadow_only` for tools without a shadow parser. Today, the implicit mechanism is that `_CAO_PARSING_MODE_DEFAULT_BY_TOOL` only lists `claude` and `codex`. But this is a *default mapping* (what tool gets if no explicit mode is requested), not a *capability gate* (what modes a tool is allowed to use). A future contributor could add a tool to `_CAO_PROVIDER_BY_TOOL` and request `shadow_only` explicitly — the current code would not reject it, despite no parser existing.

#### Examples

- Operator requests: `tool=kimi, parsing_mode=shadow_only` — should fail because no kimi shadow parser exists
- Operator requests: `tool=codex, parsing_mode=cao_only` — should succeed (explicit override of default)
- Operator requests: `tool=kimi, parsing_mode=cao_only` — should succeed (once kimi is added to the provider mapping)

#### Why this question arises in code

- `src/houmao/agents/realm_controller/backends/cao_rest.py` — `_CAO_PARSING_MODE_DEFAULT_BY_TOOL` is the implicit shadow-capability set
- `src/houmao/agents/realm_controller/launch_plan.py` — `resolve_cao_parsing_mode()` resolves the mode but doesn't have an explicit "tool supports shadow" gate

```python
# Current: default mapping doubles as implicit capability set
_CAO_PARSING_MODE_DEFAULT_BY_TOOL: Final[dict[str, CaoParsingMode]] = {
    "claude": "shadow_only",
    "codex": "shadow_only",
}

# But resolve_cao_parsing_mode() doesn't reject shadow_only for unlisted tools
```

#### Options

**Option A: Reuse default mapping as the shadow capability gate**

- Code impact:
  - In `resolve_cao_parsing_mode()`, if `requested_mode == "shadow_only"` and `tool` is not in `_CAO_PARSING_MODE_DEFAULT_BY_TOOL`, raise a validation error
  - Simple, no new data structures
- User-facing difference:
  - `shadow_only` is only available for tools that have a default shadow mapping; others must use `cao_only`
- Pros:
  - Minimal change; reuses existing data
  - Single source of truth for "which tools have shadow parsers"
- Cons:
  - Couples the *default* mapping with the *capability* gate — a tool could theoretically have a shadow parser but default to `cao_only`

**Option B: Add an explicit `_TOOLS_WITH_SHADOW_PARSER: Final[frozenset[str]]` capability set**

- Code impact:
  - Add `_TOOLS_WITH_SHADOW_PARSER = frozenset({"claude", "codex"})` alongside the default mapping
  - In `resolve_cao_parsing_mode()`, gate `shadow_only` against this set
- User-facing difference:
  - Same as Option A for current tools; future tools can be in the capability set without having `shadow_only` as default
- Pros:
  - Clean separation between "default mode" and "supported modes"
  - Future-proof for a tool that supports shadow parsing but defaults to `cao_only`
- Cons:
  - Two data structures to maintain instead of one; extra indirection for a scenario that doesn't exist today

> **PROPOSED: Option A — reuse the default mapping as the shadow capability gate.**
> Rationale: The current tool set (`claude`, `codex`) has a 1:1 relationship between "has shadow parser" and "defaults to shadow_only". Introducing a separate capability set adds indirection for a future scenario (tool with shadow parser that defaults to `cao_only`) that has no concrete design driver today. The spec scenario can be satisfied by a guard in `resolve_cao_parsing_mode()` that checks whether the tool appears in the default mapping before accepting `shadow_only`. If the coupling becomes a problem later, extracting a `frozenset` is a trivial refactor.
>
> **DECISION: Reject PROPOSED Option A; add one explicit shared shadow-parser capability helper/set and use it for validation.**
> Rationale: The repo already has two distinct notions today: `_CAO_PARSING_MODE_DEFAULT_BY_TOOL` in `launch_plan.py` and the explicit `launch_plan.tool in {"claude", "codex"}` check when constructing `ShadowParserStack` in `cao_rest.py`. Reusing the default mapping as the capability gate would keep that duplication implicit instead of naming the real contract. This change should introduce one small repo-owned capability predicate or constant set for "tool supports shadow parser", use it for `shadow_only` validation, and keep default-mode selection separate from capability.

### Q3) Should the `min_length=1` non-empty constraint on provider be a Pydantic field validator or a model validator?

- Decision: `Deferrable`

#### Why this matters

The delta `cao-rest-client-contract` spec requires the provider to be "a non-empty provider identifier string." Pydantic v2 supports both `Field(min_length=1)` inline and `@field_validator` methods. The choice affects error messages and whether the constraint is visible in schema export.

#### Examples

- CAO returns `{"provider": ""}` → should fail validation
- CAO returns `{"provider": "kimi_cli"}` → should pass

#### Why this question arises in code

- `src/houmao/cao/models.py:80` — `CaoTerminal.provider` field type change

```python
# Option A: Field annotation
class CaoTerminal(_CaoModel):
    provider: str = Field(min_length=1)

# Option B: Field validator
class CaoTerminal(_CaoModel):
    provider: str

    @field_validator("provider")
    @classmethod
    def _provider_not_empty(cls, v: str) -> str:
        if not v:
            raise ValueError("provider must be a non-empty string")
        return v
```

#### Options

**Option A: `Field(min_length=1)` inline annotation**

- Code impact:
  - One-line change to the field definition
- User-facing difference:
  - Standard Pydantic validation error message
- Pros:
  - Concise; visible in JSON Schema export
  - Follows Pydantic v2 idiomatic style
- Cons:
  - Error message is generic ("String should have at least 1 character")

**Option B: `@field_validator` method**

- Code impact:
  - Additional method on the model class
- User-facing difference:
  - Custom error message
- Pros:
  - More descriptive error message
- Cons:
  - More verbose for a simple constraint

> **PROPOSED: Option A — use `Field(min_length=1)` inline annotation.**
> Rationale: The existing `_CaoModel`-derived models in `src/houmao/cao/models.py` use Pydantic field annotations (`Literal["full", "last", "tail"]` on `CaoTerminalOutputResponse.mode`, `str` on `CaoHealthResponse.status`) without custom validators for simple constraints. `Field(min_length=1)` matches this convention and keeps the model concise. If a more descriptive error is needed later, it can be added without changing the field type.
>
> **DECISION: Accept PROPOSED Option A — use `Field(min_length=1)` inline annotation.**
> Rationale: The CAO response models already rely on straightforward Pydantic field typing rather than custom validators for simple constraints, and this change only needs a non-empty string guarantee on `CaoTerminal.provider`. An inline field constraint keeps the model small and makes the contract visible at the field site. A custom validator would add noise without buying us a materially better contract right now.
