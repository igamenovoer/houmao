## Summary

- The change direction is still right, but the remaining gap is narrower than when the change was created because several runtime and demo docs are already aligned with the newer CAO contract.
- The largest unsynchronized source of truth is now the launcher side: both `docs/reference/cao_server_launcher.md` and `openspec/specs/cao-server-launcher/spec.md` still encode the old “workdir must be under home” story.
- The typed provider boundary is still stale in live code: `CaoTerminal.provider` remains `CaoProvider`, and `houmao.cao.__all__` still exports that closed enum even though launch-time mapping already uses plain strings.
- The shadow parser support contract is still duplicated implicitly between `launch_plan.py` default-mode logic and the hard-coded `{"claude", "codex"}` parser-stack gate in `cao_rest.py`; the change docs should promote one explicit helper/set into planned work.
- Tasks should name the concrete current targets rather than the original broad doc buckets so implementation work is easier to verify.

## Findings

### Must Fix / Blocking

1. The change currently omits the `cao-server-launcher` capability even though the repo’s remaining stale workdir/home-tree contract now lives there most explicitly.

   `openspec/specs/cao-server-launcher/spec.md` still says the launcher’s trusted home exists to control CAO workdir acceptance and that CAO restricts workdirs to the home tree. `docs/reference/cao_server_launcher.md` repeats the same guidance. If the change only updates `brain-launch-runtime` and `cao-rest-client-contract`, the implementation could update docs and code while leaving the launcher spec as contradictory source-of-truth.

2. The design should explicitly state that the response-model contract drops `CaoProvider` as a public closed enum surface, not just that one field becomes forward-compatible.

   Today `src/houmao/cao/models.py` still defines `CaoProvider`, `CaoTerminal.provider` still uses it, and `src/houmao/cao/__init__.py` still re-exports it. The planned change should name that public-surface adjustment so the implementation and tests are scoped correctly.

3. The shadow-parser-support mechanism is still described too loosely for the current repo shape.

   `src/houmao/agents/realm_controller/launch_plan.py` owns `_CAO_PARSING_MODE_DEFAULT_BY_TOOL`, while `src/houmao/agents/realm_controller/backends/cao_rest.py` separately gates parser-stack creation with `launch_plan.tool in {"claude", "codex"}`. The change docs should now require one explicit shared shadow-parser-support helper or capability set so future provider additions do not widen `shadow_only` accidentally.

### Should Fix / Important

4. The doc-cleanup scope should be narrowed to the surfaces that are still stale today.

   `docs/reference/realm_controller.md`, `docs/reference/cao_interactive_demo.md`, and much of the interactive demo tutorial pack already describe parsing modes and worktree defaults in the newer terms. The remaining clearly stale surfaces are `docs/reference/cao_server_launcher.md`, `docs/reference/cao_shadow_parser_troubleshooting.md`, `src/houmao/demo/cao_interactive_demo/cli.py`, and launcher/demo help text that still reads as normative containment guidance.

5. Task wording should call out the actual high-churn test file and the export boundary explicitly.

   The code impact is concentrated in `tests/unit/agents/realm_controller/test_cao_client_and_profile.py` plus any importers of `houmao.cao.CaoProvider`. Naming those targets in the task list will make the implementation easier to audit.

## Design Alignment With Existing System

- Existing pattern observed: the runtime launch path already keeps provider mapping explicit via `_CAO_PROVIDER_BY_TOOL` in `src/houmao/agents/realm_controller/backends/cao_rest.py`, and `src/houmao/cao/rest_client.py` already accepts provider ids as plain strings on requests. Revising the change toward an open-string response boundary aligns with that code naturally.
- Existing pattern observed: parsing mode defaults and parser capability are currently separate ideas in code, even though they are not yet named that way. `launch_plan.py` holds default selection, while `cao_rest.py` holds parser availability. A small shared helper or capability set is the more native repo shape than continuing to rely on duplicated ad hoc conditionals.
- Existing pattern observed: several higher-level docs have already converged on the newer contract. The change should now steer implementation toward the remaining mismatches instead of re-sweeping already-aligned references.

## Suggested Next Steps

1. Revise the change artifacts so the modified capabilities are `brain-launch-runtime`, `cao-rest-client-contract`, and `cao-server-launcher`.
2. Update the design to make `CaoProvider` removal from the public response-model surface explicit and to require one shared shadow-parser-support helper/set.
3. Refresh the delta specs and tasks so they name the current remaining doc/code/test targets precisely.
