Inputs reviewed: `contracts/claude-state-contracts.md`, `contracts/codex-state-contracts.md`, `contracts/turn-monitor-contracts.md`, `design.md`, `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py`, `src/gig_agents/agents/brain_launch_runtime/backends/claude_code_shadow.py`, `src/gig_agents/agents/brain_launch_runtime/backends/codex_shadow.py`, and `src/gig_agents/cao/models.py`.

These questions matter because they affect the public-ish runtime contract for `shadow_only`, the boundary between provider parsing and runtime lifecycle, and the amount of raw CAO/TUI detail that leaks to callers.

---

## Summary of recommended defaults

1. Keep `activity=waiting_user_answer` as one combined blocking state, and use `ui_context` to distinguish `selection_menu` vs `trust_prompt`. See Q1.
2. Standardize `slash_command` in the shared `CommonUiContext`, and revise the Codex TUI parser/contract within this change so the shared pattern is implemented consistently. See Q2.
3. Keep raw CAO `tail` available only through non-obvious internal/debugging paths, not as a first-class caller-facing result field. See Q3.
4. Mark a turn success-terminal only when the surface returns to `ready_for_input` and runtime has observed either `evt_projection_changed` or post-submit `working`. See Q4.

## Q1) Should `waiting_user_answer` stay combined, or should trust/onboarding become its own activity state?

- Decision: `Blocking`

### Why this matters

- This changes the shared `activity` contract, which affects provider parser outputs, runtime turn-monitor branching, and tests.
- It determines whether callers look at one coarse blocking state plus `ui_context`, or whether they branch directly on multiple blocking activity enums.
- If we get this wrong now, we create churn in the base `SurfaceAssessment` shape right as it is being introduced.

### Examples

- Example 1: Claude shows a numbered selection menu where the operator must choose one option. This is clearly a "waiting for user" state.
- Example 2: Claude shows a trust/onboarding prompt for the current folder. This is also blocked on the user, but it is semantically different from a normal selection menu.

### Why this question arises in code

- The Claude contract currently combines both cases in `activity=waiting_user_answer`; `WAITING_MENU_BLOCK(V)` and `TRUST_PROMPT_BLOCK(V)` both map there in `contracts/claude-state-contracts.md:151-167`.
- The same document already distinguishes the finer-grained contexts in `ui_context`; `selection_menu` and `trust_prompt` are separate values in `contracts/claude-state-contracts.md:169-189`.
- The runtime contract currently treats all blocked states as one coarse branch. `contracts/turn-monitor-contracts.md:112-123` maps `waiting_user_answer` to `blocked_waiting_user`, and the existing runtime code already raises one blocked-turn error path in `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:734-742` and `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:818-826`.
- The current Claude parser also emits one waiting excerpt path, not distinct trust/menu result types, in `src/gig_agents/agents/brain_launch_runtime/backends/claude_code_shadow.py:299-307` and `src/gig_agents/agents/brain_launch_runtime/backends/claude_code_shadow.py:642`.

### Options

**Option A: Split activity into separate blocking states**

- Code impact:

```python
Activity = Literal[
    "ready_for_input",
    "working",
    "waiting_selection",
    "waiting_trust",
    "unknown",
]
```

```python
if TRUST_PROMPT_BLOCK(V):
    activity = "waiting_trust"
elif WAITING_MENU_BLOCK(V):
    activity = "waiting_selection"
```

- User-facing difference:
  - Callers can branch on blocking reason directly from `activity`.
- Pros:
  - Makes trust/onboarding a first-class runtime-visible state.
  - Avoids forcing every caller to inspect `ui_context` for blocked-state detail.
- Cons:
  - Expands the shared `activity` vocabulary for a distinction that is already present in `ui_context`.
  - Pushes Claude-specific blocking nuance into the common lifecycle state machine.
  - Creates extra cross-provider coordination pressure even though Codex currently models its finer blocking reason through `approval_prompt` in `ui_context`, not in `activity`.

**Option B: Keep one blocking activity state and use `ui_context` for the reason**

- Code impact:

```python
if TRUST_PROMPT_BLOCK(V) or WAITING_MENU_BLOCK(V):
    activity = "waiting_user_answer"

if TRUST_PROMPT_BLOCK(V):
    ui_context = "trust_prompt"
elif WAITING_MENU_BLOCK(V):
    ui_context = "selection_menu"
```

- User-facing difference:
  - Generic callers see one blocked activity state, while provider-aware callers can inspect `ui_context` for the specific kind of block.
- Pros:
  - Preserves a compact shared `activity` enum for `TurnMonitor`.
  - Reuses the distinction the Claude contract already defines.
  - Aligns with the current runtime shape, where blocked turns are handled the same way regardless of which prompt produced them.
- Cons:
  - Callers that need precise blocked-state remediation must read two fields.
  - Future provider-specific blocking modes may continue to accumulate in `ui_context` instead of `activity`.

### Proposal (Recommended)

Use Option B. Keep `activity=waiting_user_answer` as the coarse blocking state and use `ui_context` to distinguish `selection_menu` vs `trust_prompt`.

This matches the current contract split in `contracts/claude-state-contracts.md:151-189`, keeps `TurnMonitor` generic, and avoids polluting the shared `activity` vocabulary with provider-specific variants too early.

### Pros / Cons (Proposal)

- Pros:
  - The runtime lifecycle remains simple: blocked is blocked.
  - Claude-specific nuance still remains available to callers via `ui_context`.
  - The same pattern already exists in the Codex-side design, where provider-specific blocking detail lives in `ui_context`.
- Cons:
  - Some callers may initially expect `activity` alone to explain the exact blocked reason.
  - Error rendering code may need to include both `activity` and `ui_context` to stay operator-friendly.

> **DECISION: Q1 uses Option B; keep one generic blocking activity state and distinguish the reason through `ui_context`.**
> Rationale: This is more generic and easier to understand. It also matches the existing contract split where Claude maps both `WAITING_MENU_BLOCK(V)` and `TRUST_PROMPT_BLOCK(V)` into `activity=waiting_user_answer` while keeping `selection_menu` and `trust_prompt` separate in `ui_context` in `contracts/claude-state-contracts.md:164-189`, and it fits the current runtime model where blocked turns share one lifecycle branch in `contracts/turn-monitor-contracts.md:112-123`.

### Notes / Follow-ups

- If later experience shows trust/onboarding needs distinct runtime semantics from ordinary blocking, we can promote it from `ui_context` into `activity` in a later versioned contract update.

## Q2) Should `slash_command` be standardized in the shared `CommonUiContext`, or remain provider-specific?

- Decision: `Deferrable`

### Why this matters

- This determines whether the first cross-provider `ui_context` vocabulary stays small and generic, or whether it bakes Claude-specific UI concepts into the shared base type immediately.
- It affects the type signature for the new frozen `SurfaceAssessment` base object and every future provider implementation.
- Standardizing too early makes the base contract harder to evolve if other providers do not have a real equivalent.

### Examples

- Example 1: Claude enters a slash-command palette or command-like UI where the terminal is still inside Claude but not at an ordinary prompt.
- Example 2: Codex currently has no parallel `slash_command` concept in the proposed state contract; its provider-specific extension uses `approval_prompt` instead.

### Why this question arises in code

- The design deliberately keeps the common base small: `CommonUiContext = Literal["normal_prompt", "selection_menu", "unknown"]` in `openspec/changes/decouple-shadow-state-from-answer-association/design.md:55-66`.
- Claude then extends that base with `slash_command`, `trust_prompt`, and `error_banner` in `openspec/changes/decouple-shadow-state-from-answer-association/design.md:70-91`.
- Codex extends the same base differently, adding `approval_prompt` and `error_banner` instead of `slash_command` in `openspec/changes/decouple-shadow-state-from-answer-association/design.md:80-96`.
- The Claude contract already models `slash_command` as a provider-specific context in `contracts/claude-state-contracts.md:169-189`, while the Codex contract does not try to mirror it in `contracts/codex-state-contracts.md:135-151`.

### Options

**Option A: Promote `slash_command` into the shared `CommonUiContext`**

- Code impact:

```python
CommonUiContext = Literal[
    "normal_prompt",
    "selection_menu",
    "slash_command",
    "unknown",
]
```

- User-facing difference:
  - Generic cross-provider callers can see `slash_command` without doing provider-specific type narrowing.
- Pros:
  - Exposes a potentially useful UI mode to all callers through one common field.
  - Avoids one special-case provider branch for Claude-specific code.
- Cons:
  - Codex and future providers inherit a base value that may be meaningless for them.
  - Turns one provider-specific UI detail into a permanent shared contract concept very early.
  - Encourages the base vocabulary to grow around Claude rather than around real cross-provider commonality.

**Option B: Keep `slash_command` provider-specific for v1**

- Code impact:

```python
CommonUiContext = Literal["normal_prompt", "selection_menu", "unknown"]

ClaudeUiContext = Literal[
    "normal_prompt",
    "selection_menu",
    "slash_command",
    "trust_prompt",
    "error_banner",
    "unknown",
]
```

- User-facing difference:
  - Generic callers only depend on a minimal cross-provider context vocabulary; Claude-aware callers can inspect the richer subclass type.
- Pros:
  - Preserves a clean separation between shared and provider-specific semantics.
  - Matches the current design and both contract docs.
  - Avoids over-committing the base model before we see a second provider need the same concept.
- Cons:
  - Generic callers cannot treat `slash_command` as a uniform cross-provider state.
  - Some consumer code will need provider narrowing when it cares about slash-command UI specifically.

### Proposal (Recommended)

Use Option A. Promote `slash_command` into the shared `CommonUiContext`, and make the Codex parser/contract changes inside this same OpenSpec change.

The decision rationale is that slash-command style UI is common enough across modern CLI agents that it is worth naming in the shared base contract now, rather than treating it as a Claude-only extension. Since this change already covers Codex parser refactoring, Codex should be brought into alignment here instead of leaving the shared contract half-adopted until a later update.

### Pros / Cons (Proposal)

- Pros:
  - Cross-provider callers can reason about slash-command UI using one shared field.
  - The shared contract better reflects the reality that slash-command surfaces are common in CLI agent tooling.
  - It keeps the current change internally consistent by aligning Codex in the same implementation wave.
- Cons:
  - The current Codex contract and parser do not yet expose a matching context, so this change's implementation scope must absorb that work.
  - The base `CommonUiContext` grows earlier than originally planned.

> **DECISION: Q2 uses Option A; standardize `slash_command` in the shared `CommonUiContext`.**
> Rationale: Slash-command UI is common across modern CLI tools, including Claude Code, Codex, and Gemini, so it is worth modeling as a shared contract concept instead of a Claude-only extension. The current design already centralizes `CommonUiContext` in `openspec/changes/decouple-shadow-state-from-answer-association/design.md:55-66`, and this change already includes Codex parser refactoring work, so Codex should be revised here rather than deferred. The current Codex contract still exposes `approval_prompt` but no `slash_command` in `openspec/changes/decouple-shadow-state-from-answer-association/contracts/codex-state-contracts.md:135-153`, which is exactly the consistency gap this change should close.

### Notes / Follow-ups

- This should be reflected back into the implementation tasks/specs so Codex parser work in this change includes shared-`slash_command` alignment rather than leaving it as a later follow-up.

## Q3) Should the runtime expose raw CAO `tail` snapshots directly alongside projected dialog slices?

- Decision: `Deferrable`

### Why this matters

- This is a contract-shape question: exposing raw CAO `tail` makes the runtime result surface more powerful for debugging, but it also risks reintroducing raw-TUI parsing as a caller habit.
- It affects whether the new projection contract stays cleanly layered, or whether the runtime also promises access to transport-level raw data.
- It changes the likely migration path for callers moving off `output_text`.

### Examples

- Example 1: A debugging or operator tool wants the exact raw tail of the tmux/CAO output when projection or parsing drifts.
- Example 2: A feature caller starts using raw tail directly to parse answers, bypassing the projector and recreating the same parser drift the change is trying to eliminate.

### Why this question arises in code

- The CAO response model already supports `mode="tail"` at the transport layer in `src/gig_agents/cao/models.py:87-91`.
- The current runtime already uses tail excerpts heavily for diagnostics and error messages in `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:226-232`, `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:767-770`, and `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1126-1175`.
- The new design deliberately says head/tail slices are defined over normalized projected dialog, not raw CAO output, in `openspec/changes/decouple-shadow-state-from-answer-association/design.md:171-183`.
- The Claude contract already defines projection responsibility as keeping dialog, dropping chrome, and exposing `head`/`tail` slices at the projected-dialog layer in `contracts/claude-state-contracts.md:396-410`.

### Options

**Option A: Expose raw CAO `tail` as a first-class runtime field**

- Code impact:

```python
class ShadowResult:
    dialog_projection: DialogProjection
    surface_assessment: SurfaceAssessment
    raw_tail_output: str
```

- User-facing difference:
  - Callers receive both the normalized projection and the raw transport tail in the success payload.
- Pros:
  - Improves debuggability for tools that need exact transport evidence.
  - Makes it easier to build advanced tooling without a second CAO fetch.
- Cons:
  - Encourages callers to parse raw tail directly instead of relying on the projector.
  - Expands the runtime contract with transport-specific detail that the change is trying to hide.
  - Makes the result surface less provider-neutral and more tightly coupled to CAO/tmux.

**Option B: Keep raw tail internal or diagnostics-only**

- Code impact:

```python
class ShadowResult:
    dialog_projection: DialogProjection
    surface_assessment: SurfaceAssessment
    diagnostics: dict[str, object]  # may contain excerpts or debug-only payloads
```

- User-facing difference:
  - Callers use projected `head`/`tail` for normal logic; raw tail remains an internal debug mechanism or a future opt-in diagnostics field.
- Pros:
  - Preserves the core contract boundary: callers consume projection, not transport.
  - Matches the current design recommendation in `design.md:171-183`.
  - Keeps room for a future opt-in diagnostic surface without making it part of the first-class runtime payload now.
- Cons:
  - Some debugging tools may need an additional path to retrieve raw transport evidence.
  - Advanced tooling loses a potentially convenient raw signal in the initial contract.

### Proposal (Recommended)

Use Option B. Do not expose raw CAO `tail` as a first-class runtime success field in v1.

The design goal of this change is to move callers away from raw TUI parsing and toward provider-owned projection. The runtime can still keep raw tail excerpts for errors and debugging, which it already does today, but that is a very different promise from surfacing raw tail alongside `dialog_projection` in the primary result contract.

### Pros / Cons (Proposal)

- Pros:
  - Keeps the new runtime contract aligned with the parser/projection boundary.
  - Reduces the chance that callers will recreate answer association on top of raw tail.
  - Leaves room for a future opt-in debug surface if real users need it.
- Cons:
  - Some operator/debug use cases will require a separate diagnostics path.
  - The contract becomes slightly less "power user friendly" in the first version.

> **DECISION: Q3 uses Option B; keep raw CAO `tail` only in non-obvious internal/debugging paths rather than as a first-class result field.**
> Rationale: Raw tail is still useful for debugging, but callers should need to know the runtime internals to work with it rather than seeing it as part of the normal contract surface. That fits the current design in `openspec/changes/decouple-shadow-state-from-answer-association/design.md:171-183`, which defines `head` and `tail` over normalized projected dialog, while preserving the existing runtime practice of using tail excerpts for diagnostics in `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:226-232`, `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:767-770`, and `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:1126-1175`.

### Notes / Follow-ups

- Assumption while deferred: projected `head` and `tail` are the only obvious first-class transcript slices; raw tail may still exist in diagnostics or internal/debugging paths that require caller knowledge of runtime internals.
- If later exposed, raw tail should probably live under an explicitly diagnostic field rather than alongside `dialog_projection` as a peer contract object.

## Q4) Should `ready_for_input` require observed `evt_projection_changed` after submit before runtime marks a turn terminal?

- Decision: `Blocking`

### Why this matters

- This determines the success condition for `shadow_only` turns and directly affects whether stale transcript snapshots can be misreported as fresh completion.
- It is the core runtime answer to the baseline-reset class of bugs that motivated the change.
- The decision changes the `TurnMonitor` state machine, test expectations, and caller-visible completion timing.

### Examples

- Example 1: Claude returns to an idle-looking prompt before any poll catches `working`. If runtime treats `ready_for_input` as sufficient by itself, the turn can terminate with stale projected dialog.
- Example 2: Claude visibly enters `working` and later returns to `ready_for_input`. In that case, requiring an additional projection change check on top of observed `working` can add unnecessary latency or fragility for identical-looking outputs.

### Why this question arises in code

- The current runtime uses `_ShadowLifecycleTracker` only for unknown→stalled handling in `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:312-370`; it is not a submit-aware terminality engine.
- The current completion loop still carries a `saw_processing` flag and only accepts `idle` after processing plus successful extraction in `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:784-860`.
- The Claude contract still lists this as an open question in `contracts/claude-state-contracts.md:426-431`.
- The new runtime contract already leans toward an answer: `contracts/turn-monitor-contracts.md:99-110` defines success terminality as `ready_for_input` plus either `evt_projection_changed` or `evt_working_seen`.

### Options

**Option A: Require `evt_projection_changed` for all successful terminality**

- Code impact:

```python
if ready_for_input and saw_projection_change_after_submit:
    return completed
```

- User-facing difference:
  - Completion is withheld until transcript text visibly changes after submit.
- Pros:
  - Strong protection against stale completions.
  - Simple success rule.
- Cons:
  - Can over-wait or fail when the tool does real work but leaves a nearly identical projection.
  - Turns text-diff visibility into a hard requirement even when runtime saw unambiguous post-submit activity.

**Option B: Treat `ready_for_input` alone as success-terminal**

- Code impact:

```python
if ready_for_input:
    return completed
```

- User-facing difference:
  - Completion is faster and simpler, but more likely to terminate on stale or unchanged snapshots.
- Pros:
  - Minimal runtime state memory.
  - Lowest latency in the common path.
- Cons:
  - Reintroduces the stale-idle bug class this change is meant to avoid.
  - Gives the runtime no causal evidence that something happened after submit.

**Option C: Require `ready_for_input` plus (`evt_projection_changed` or post-submit `working`)**

- Code impact:

```python
if ready_for_input and (
    saw_projection_change_after_submit or saw_working_after_submit
):
    return completed
```

- User-facing difference:
  - The runtime completes quickly when it has seen real activity, but still protects the fast-return edge case where no poll captured `working`.
- Pros:
  - Protects against stale terminality without making visible text change mandatory in every successful turn.
  - Matches the `TurnMonitor` contract already drafted in `contracts/turn-monitor-contracts.md:99-110`.
  - Preserves the spirit of the current `saw_processing` safeguard in `cao_rest.py:784-860`.
- Cons:
  - Slightly more runtime state to track.
  - The success rule is more complex to explain than a single-condition completion test.

### Proposal (Recommended)

Use Option C.

This is the best fit with both the current runtime history and the new contract direction. It keeps the stale-protection goal of requiring some post-submit evidence, but it does not make visible projection change mandatory when runtime has already seen `working`.

### Pros / Cons (Proposal)

- Pros:
  - Strong stale-completion protection.
  - Good latency in the common path.
  - Directly aligns the Claude contract with the new TurnMonitor contract.
- Cons:
  - Requires explicit runtime memory of post-submit events.
  - Test cases need to cover both the "saw working" and "projection changed without working observation" paths.

> **DECISION: Q4 uses Option C; treat a turn as success-terminal only after `ready_for_input` plus (`evt_projection_changed` or post-submit `working`).**
> Rationale: This keeps the stale-protection goal without making visible projection change mandatory for every successful turn. It also aligns directly with the drafted TurnMonitor contract in `openspec/changes/decouple-shadow-state-from-answer-association/contracts/turn-monitor-contracts.md:99-110` and preserves the spirit of the current runtime safeguard in `src/gig_agents/agents/brain_launch_runtime/backends/cao_rest.py:784-860`, where completion is not accepted on idle alone.

### Notes / Follow-ups

- This question should probably stop living as an open question in `contracts/claude-state-contracts.md` once the change adopts the `TurnMonitor` contract as authoritative.
