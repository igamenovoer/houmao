## Summary

- **Strong conceptual core.** The separation of state detection / dialog projection / answer association is well-motivated and directly addresses the documented baseline-invalidation problems in the Claude shadow parser. The layered architecture (SurfaceParser → DialogProjector → TurnMonitor → optional AnswerAssociator) is clean.
- **Spec coverage is thorough.** Delta specs cover all five affected capabilities with MODIFIED/ADDED/REMOVED sections, concrete scenarios, and explicit migration notes for removed requirements.
- **State contract document is excellent.** `contracts/state-contracts.md` provides a formally structured state model with priority rules, transition events, a Mermaid transition graph, and clear responsibility boundaries. This is the strongest artifact in the change.
- **Risk: new abstractions without concrete model definitions.** `SurfaceAssessment` and `DialogProjection` are described conceptually but no frozen-dataclass shapes appear in the specs or tasks. Implementation latitude here could cause provider divergence.
- **Risk: `TurnMonitor` complexity is under-specified.** The design describes what it does but does not define its state machine, timeouts, or relationship to the existing `_ShadowLifecycleTracker` in `cao_rest.py`.
- **Per-provider state classes need an explicit inheritance plan.** The state contract currently defines one shared state model. Claude and Codex backends should have their own concrete state classes (inheriting common facets), since backend-specific states will evolve independently.

## Findings

### Must Fix / Blocking

- **MF-1: `SurfaceAssessment` and `DialogProjection` need concrete model specs with per-provider inheritance.**
  The design (§1, §3) and `contracts/state-contracts.md` (§Core Contract Objects) list field names but the delta specs do not include a formal model definition (types, optionality, defaults). Additionally, the state model should be structured as: a common base `SurfaceAssessment` frozen dataclass with shared fields (`availability`, `activity`, `accepts_input`, `parser_metadata`, `anomalies`) plus provider-specific subclasses (e.g., `ClaudeSurfaceAssessment`, `CodexSurfaceAssessment`) that add backend-specific `ui_context` values and evidence fields. Same for `DialogProjection`. Different backends are fundamentally different things — they share a common interface but their internal state vocabularies evolve independently.
  **Suggestion:** Add a model spec section that defines (a) base frozen dataclasses with common fields and (b) provider-specific subclasses. The base class should use a `Literal` for common states, while subclasses extend with provider-specific literals. This follows the existing pattern where `ClaudeCodeShadowParser` and `CodexShadowParser` are separate classes with no shared base, and `ClaudeCodeShadowStatus`/`CodexShadowStatus` are separate types.

- **MF-2: Delta specs for `versioned-shadow-parser-stack` only add; they don't modify.**
  The main spec (`openspec/specs/versioned-shadow-parser-stack/spec.md`) currently says the stack returns "structured parser metadata" including "baseline invalidation status." The delta spec adds two new requirements (returns surface assessment + dialog projection; does not own prompt-to-answer association) but does not include a MODIFIED section to update the existing requirement's wording to align with the new contract. This creates a post-merge inconsistency.
  **Suggestion:** Add a MODIFIED block that updates the existing "Runtime-owned shadow parser stack is version-aware and reports match/anomalies" requirement to reference surface assessment / dialog projection instead of extraction-oriented metadata.

- **MF-3: Tasks lack dependency ordering and are missing a key task for `TurnMonitor` design.**
  Task 3.1 says "Add a runtime-owned shadow_only turn-monitor flow" but the `TurnMonitor` state machine, its relationship to the existing `_ShadowLifecycleTracker`, and its timeout/recovery semantics are not specified anywhere except as a conceptual sketch in `contracts/state-contracts.md` (§Runtime Turn Lifecycle Responsibility). A dedicated design sub-section or contract for the turn monitor state machine would prevent implementation drift.
  **Suggestion:** Either (a) add a `contracts/turn-monitor-contracts.md` with the TurnMonitor state machine, or (b) expand Task 3.1's description to include the explicit state machine and its relationship to `_ShadowLifecycleTracker`.

- **MF-4: State contracts document defines only Claude-specific predicates; Codex needs its own.**
  `contracts/state-contracts.md` defines Claude-specific detection predicates (`IDLE_PROMPT_LINE(V)`, `PROCESSING_SPINNER_LINE(V)`, etc.) and a Claude-specific transition graph. There is no equivalent Codex state contract. Since backends are fundamentally different things with independent evolution, Codex needs its own predicate table and state contract — not a generalized "cross-provider" document.
  **Suggestion:** Create `contracts/codex-state-contracts.md` with Codex-specific detection predicates (e.g., `CODEX_LABEL_ASSISTANT_LINE(V)`, `CODEX_TUI_BULLET_LINE(V)`, `CODEX_APPROVAL_PROMPT(V)`, `CODEX_IDLE_PROMPT(V)`) and rename the existing contract to `contracts/claude-state-contracts.md` for clarity.

### Should Fix / Important

- **SF-1: Existing `_ShadowLifecycleTracker` is not mentioned anywhere.**
  `cao_rest.py` already contains `_ShadowLifecycleTracker` which manages the unknown→stalled lifecycle, anomaly recording (`STALLED_ENTERED`, `STALLED_RECOVERED`), and terminal-stalled failure. The change introduces a `TurnMonitor` concept that overlaps significantly. Neither the design nor the tasks describe whether `_ShadowLifecycleTracker` is being replaced, wrapped, or evolved.
  **Suggestion:** Add an explicit note in design §2 or tasks §3.1 about the disposition of `_ShadowLifecycleTracker`: absorb into `TurnMonitor`, rename, or deprecate.

- **SF-2: `shadow-dialog-projection` is a new capability with no main spec to merge into.**
  The delta spec at `specs/shadow-dialog-projection/spec.md` is all ADDED requirements. This is correct since there is no existing main spec at `openspec/specs/shadow-dialog-projection/`. However, the tasks do not include creating the main spec directory during the docs/tests phase.
  **Suggestion:** Add an explicit subtask under 4.1 (or a new task) to create `openspec/specs/shadow-dialog-projection/spec.md` from the delta after implementation.

- **SF-3: Task 4.2 should list the specific test files/fixtures affected.**
  The removed Claude requirement ("Answer extraction from `mode=full` is preset-scoped, ANSI-stripped, and prompt-bounded") currently drives `extract_last_answer()` in `claude_code_shadow.py` (~100 lines) and is directly tested by multiple test cases. Task 4.2 says "Replace or rewrite unit tests" but doesn't name the affected test files.
  **Suggestion:** Make Task 4.2 concrete: list `tests/unit/agents/test_claude_code_shadow_parser.py`, `tests/unit/agents/test_codex_shadow_parser.py`, and `tests/fixtures/shadow_parser/` as the specific targets.

- **SF-4: A `TailRegexExtractAssociator` should be included in this change as the concrete associator example.**
  Design §4 introduces the answer-association layer as an explicit optional boundary above the core parser, but defers all implementation. The change should ship a `TailRegexExtractAssociator` that accepts constructor parameters `n: int` (tail window size in characters) and `pattern: re.Pattern` (regex to match against the tail window). It receives projected dialog text through the association interface, takes the last `n` characters, searches for `pattern`, and returns the matched portion as the associated output. This is more useful than a plain tail slice because it demonstrates the core associator value proposition: applying caller-specific extraction logic on top of projected dialog. It establishes the protocol concretely and gives callers a real pattern to follow for their own associators.
  **Suggestion:** Add a task (e.g., 3.4) to implement `TailRegexExtractAssociator(n, pattern)`, implement the associator protocol, and add a unit test. Reference it in design §4 as the shipped example.

### Nice To Have

- **NTH-1: Mermaid diagrams in `contracts/state-contracts.md` are excellent.** Consider adding a similar diagram for the `DialogProjector` pipeline.

- **NTH-2: Task ordering could benefit from explicit dependencies.** The four task groups are logically ordered, but dependencies between groups are implicit (1.1 → 2.x → 3.x → 4.x). A simple DAG note in `tasks.md` would reduce coordination risk.

## Design Alignment With Existing System

- **Existing patterns observed:**
  - `shadow_parser_core.py`: Frozen dataclasses for `ShadowParserMetadata`, `ShadowParserAnomaly`, `PresetResolution`. Uses `@dataclass(frozen=True)` for value objects.
  - `shadow_parser_stack.py`: Thin adapter that delegates to provider-specific parsers and wraps errors.
  - `cao_rest.py` / `ShadowOnlyTurnEngine`: Owns the turn lifecycle (ready→submit→poll→extract), uses `_ShadowLifecycleTracker` for stalled-state management, captures baseline before submit.
  - Provider parsers (`claude_code_shadow.py`, `codex_shadow.py`): Completely separate classes with no shared base. Each exposes `resolve_preset_version`, `detect_output_format`, `capture_baseline_pos`, `classify_shadow_status`, `extract_last_answer`. Uses `VersionedPresetRegistry` for version-aware preset selection.
  - `ClaudeCodeShadowStatus` and `CodexShadowStatus`: Separate types, not shared.
  - `models.py`: `BackendKind` literal type, `CaoParsingMode`, `LaunchPlan`, `InteractiveSession` protocol.

- **Proposed design alignment:**
  - **Aligns well:** The `SurfaceAssessment` / `DialogProjection` split maps naturally onto the existing separation between `classify_shadow_status` and `extract_last_answer` in provider parsers. Replacing the extraction method with a projection method is a clean refactor path.
  - **Needs adjustment (per-provider state classes):** The design and state contracts define a single `SurfaceAssessment` with a shared `ui_context` vocabulary. The existing codebase treats Claude and Codex as completely independent backends — `ClaudeCodeShadowParser` and `CodexShadowParser` share no base class, and their status types (`ClaudeCodeShadowStatus`, `CodexShadowStatus`) are separate. The new model should follow the same pattern: a base `SurfaceAssessment` with common fields, plus `ClaudeSurfaceAssessment` / `CodexSurfaceAssessment` subclasses with provider-specific `ui_context` values and evidence. Common state names (like `ready_for_input`, `working`) can live in the base; provider-specific states (like `slash_command`, `trust_prompt` for Claude) live in subclasses.
  - **Needs adjustment (`TurnMonitor` vs `_ShadowLifecycleTracker`):** The design introduces a `TurnMonitor` as a new architectural layer, but `_ShadowLifecycleTracker` in `cao_rest.py` already manages unknown→stalled lifecycle. The design should clarify the relationship.
  - **Needs adjustment (`DialogProjector` placement):** The existing architecture has no separate projector — ANSI stripping and chrome removal are done inline in `extract_last_answer()`. Creating a standalone `DialogProjector` is a new module boundary that needs file placement (in `shadow_parser_core.py`? A new file?). The tasks don't specify.
  - **Convention alignment:** `SurfaceAssessment` and `DialogProjection` should be `@dataclass(frozen=True)` per the project convention for internal value objects (matching `ShadowParserMetadata` in `shadow_parser_core.py`), not Pydantic models.

## Open Questions (With Proposed Defaults)

### Q1) Should `output_text` remain populated in `shadow_only` as a compatibility alias to projected dialog text?

- Decision: `Blocking`

#### Why this matters

- `output_text` is consumed downstream of `ShadowOnlyTurnEngine._EngineTurnResult`. Changing its semantics affects callers.

#### Examples

- A caller sends a prompt, gets back `output_text = "<full projected dialog including prior turns>"` instead of the extracted answer.
- A caller gets `output_text = ""` and their processing breaks.

#### Why this question arises in code

- `cao_rest.py` `ShadowOnlyTurnEngine` returns `_EngineTurnResult.output_text` which is currently the extracted answer from `extract_last_answer()`.
- The change removes `extract_last_answer()` from the core contract, so something must populate (or stop populating) `output_text`.

#### Options

**Option A: Replace `output_text` with projection payload fields; drop `output_text` outright**

- Code impact:
  - Remove `output_text` from `_EngineTurnResult`
  - Add `dialog_projection: DialogProjection` and `surface_assessment: SurfaceAssessment` as first-class fields
  - All callers must update
- User-facing difference:
  - Clean break; no ambiguity about what the field means
- Pros:
  - No legacy field semantics to maintain
  - Forces callers to adopt the new contract
- Cons:
  - Immediate breakage for all callers

**Option B: Keep `output_text` populated with projected dialog tail slice**

- Code impact:
  - `output_text = dialog_projection.tail`
  - New fields added alongside
- User-facing difference:
  - Callers see some text but it may include prior-turn content
- Pros:
  - Softer migration
- Cons:
  - Preserves ambiguity the change is trying to remove

#### Proposal (Recommended)

Replace `output_text` with the new projection payload fields. The project is unstable by nature and callers expect breaking changes. Keeping `output_text` as a compatibility alias undermines the architectural clarity this change is meant to establish. Pair the break with the `TailRegexExtractAssociator` (see SF-4) so callers who want extracted text have a named, explicit tool.

#### Pros / Cons (Proposal)

- Pros:
  - Clean break with no ambiguity
  - `TailRegexExtractAssociator` provides a concrete migration path for callers that need extracted text
  - No legacy field to maintain or eventually remove
- Cons:
  - All callers must update (acceptable given project stability posture)

> **DECISION: Drop `output_text` outright; replace with `dialog_projection` + `surface_assessment` fields; ship `TailRegexExtractAssociator` as the escape hatch.**
> Rationale: The project is explicitly unstable and breaking changes are expected. `_EngineTurnResult` in `cao_rest.py` is an internal type (prefixed with `_`), not a public API. Keeping a compatibility alias contradicts the change's core thesis that projected dialog is not an authoritative answer. The `TailRegexExtractAssociator(n=..., pattern=...)` provides a concrete, named migration path for callers that need to extract specific content from projected dialog.

### Q2) What is the minimal cross-provider `ui_context` vocabulary to standardize now?

- Decision: `Deferrable`

#### Why this matters

- `ui_context` values appear in `SurfaceAssessment` and will be consumed by the `TurnMonitor`. If the vocabulary is shared, providers must agree on semantics. If per-provider, each can evolve independently.

#### Examples

- Claude has `slash_command`, `trust_prompt` contexts. Codex does not.
- Both have menu/approval prompts that map to `selection_menu`.

#### Why this question arises in code

- `contracts/state-contracts.md` defines Claude-specific `ui_context` values but no Codex-specific ones.
- `ClaudeCodeShadowParser` and `CodexShadowParser` are completely separate classes — there is no shared state type today.

#### Options

**Option A: Single shared `ui_context` type for all providers**

- Code impact:
  - One `Literal` type with the union of all provider values
  - Codex must handle Claude-specific values (always returning `unknown`/`normal_prompt`)
- Pros:
  - Single vocabulary
- Cons:
  - Couples provider evolution; Codex carries dead variants

**Option B: Common base `ui_context` plus provider-specific subtype extensions**

- Code impact:
  - Base: `Literal["normal_prompt", "selection_menu", "unknown"]` (shared cross-provider)
  - `ClaudeUiContext`: extends with `"slash_command" | "trust_prompt" | "error_banner"`
  - `CodexUiContext`: extends with provider-specific values as needed (e.g., `"approval_prompt"`)
  - `ClaudeSurfaceAssessment.ui_context: ClaudeUiContext`; `CodexSurfaceAssessment.ui_context: CodexUiContext`
- Pros:
  - Each backend evolves independently
  - Common states are shared for `TurnMonitor` logic that works across providers
  - Backend-specific states can change anytime without affecting the other
- Cons:
  - Slightly more type boilerplate

#### Proposal (Recommended)

Use Option B: common base plus provider-specific extensions. This matches the existing codebase pattern where `ClaudeCodeShadowParser` and `CodexShadowParser` are completely separate classes. Standardize common names (`normal_prompt`, `selection_menu`, `unknown`) but let each provider define its own full vocabulary. The `TurnMonitor` reasons over the common base; provider-specific `ui_context` values are only consumed by provider-aware code.

#### Pros / Cons (Proposal)

- Pros:
  - Follows existing pattern of separate provider types
  - Backend-specific states can change anytime without cross-provider coordination
  - Common states still enable shared TurnMonitor logic
- Cons:
  - More types to define (trivial cost)

> **DECISION: Common base `ui_context` with provider-specific subtype extensions; each backend gets its own `SurfaceAssessment` subclass.**
> Rationale: The codebase treats Claude and Codex as fundamentally different backends. `ClaudeCodeShadowParser` and `CodexShadowParser` share no base class; `ClaudeCodeShadowStatus` and `CodexShadowStatus` are separate types. Forcing a single shared `ui_context` type would couple their evolution. A common base (`normal_prompt`, `selection_menu`, `unknown`) lets `TurnMonitor` work generically, while `ClaudeSurfaceAssessment` adds `slash_command`/`trust_prompt`/`error_banner` and `CodexSurfaceAssessment` adds `approval_prompt` or future Codex-specific contexts.

### Q3) Should the runtime ship optional associator helpers in this change?

- Decision: `Blocking`

#### Why this matters

- Without a concrete associator, the associator layer exists only as a design concept. Shipping a working example establishes the protocol, demonstrates real extraction logic, and gives callers an immediate migration path.

#### Examples

- A caller currently reads the extracted answer. After this change, they get projected dialog (possibly multi-turn). The `TailRegexExtractAssociator(n=4000, pattern=re.compile(r"(?s)●\s+(.*?)(?=\n❯|\Z)"))` looks at the last 4000 characters and extracts the regex match — a real extraction pattern, not just a tail slice.

#### Why this question arises in code

- Design §4 says "The runtime may later provide helper associator interfaces" but lists it as an open question.
- No task in `tasks.md` covers building an associator.

#### Options

**Option A: Ship a `TailRegexExtractAssociator` as a concrete example**

- Code impact:
  - New class `TailRegexExtractAssociator` with constructor params `n: int` (tail window size) and `pattern: re.Pattern` (regex to match)
  - Implements the associator protocol: receives projected dialog text, takes the last `n` characters, searches for `pattern`, returns the matched portion as the associated output (or `None` / empty if no match)
  - Demonstrates real extraction logic — not just a tail slice — which is the whole point of the associator layer
  - New file (e.g., `answer_associator.py`) or section in `shadow_parser_core.py`
  - Unit test for match and no-match cases
- User-facing difference:
  - Callers have a named, working extraction tool
  - The associator protocol is established with a non-trivial example
- Pros:
  - Establishes the pattern with real extraction semantics
  - Small implementation (~30 lines)
  - Immediately useful: callers can plug in provider-specific regexes
- Cons:
  - Slightly more scope than a pure tail slice

**Option B: Defer all associator work**

- Code impact:
  - None
- Pros:
  - Less scope
- Cons:
  - Associator layer remains abstract; callers have no example

#### Proposal (Recommended)

Ship `TailRegexExtractAssociator` in this change. Constructor takes `n: int` (tail window size in characters) and `pattern: re.Pattern` (regex to search for). It receives projected dialog text through the associator interface, takes the last `n` characters as the search window, applies `pattern`, and returns the matched portion as the associated output. If no match is found, it returns `None`. This demonstrates the core associator value proposition — applying caller-specific extraction logic on top of projected dialog — rather than being a trivial tail slice.

#### Pros / Cons (Proposal)

- Pros:
  - Concrete protocol definition with real extraction semantics
  - Small implementation (~30 lines + test)
  - Immediately useful: callers plug in provider-specific or task-specific regexes
  - Serves as documentation-by-example for future associators
- Cons:
  - Minor additional scope (~30 lines + test)

> **DECISION: Ship `TailRegexExtractAssociator(n, pattern)` as the concrete associator example in this change.**
> Rationale: Design §4 establishes the associator boundary but without a concrete implementation it remains abstract. A `TailRegexExtractAssociator` with constructor params `n` and `pattern` that searches `projected_dialog[-n:]` for a regex match is ~30 lines of code and demonstrates the real value of the associator layer: caller-defined extraction logic applied to projected dialog. Combined with dropping `output_text` (Q1), it gives callers an explicit named tool: `TailRegexExtractAssociator(n=4000, pattern=my_regex).associate(projection.dialog_text)`. Add as Task 3.4.

### Q4) Should `ready_for_input` require observed `evt_projection_changed` after submit before runtime marks a turn terminal?

- Decision: `Blocking`

#### Why this matters

- If the runtime marks a turn terminal as soon as it sees `ready_for_input` after submission, it may catch a snapshot where the TUI has returned to idle but the projected dialog hasn't changed yet. This would produce a stale projection.

#### Examples

- A fast Claude response: prompt submitted, Claude processes and returns to idle in under one polling interval. Runtime sees `ready_for_input` on the next poll with stale projection.
- A slow response: Claude goes through `working` → `ready_for_input` with visible dialog change. No issue.

#### Why this question arises in code

- `contracts/state-contracts.md` defines `evt_projection_changed` as a transition event but does not prescribe whether the `TurnMonitor` requires it for terminality.
- The current `ShadowOnlyTurnEngine._wait_for_shadow_completion()` in `cao_rest.py` waits for `completed` status (which requires a post-baseline response marker), implicitly gating on content change. The new design removes `completed` as a parser state, so this implicit gate is lost.

#### Options

**Option A: Require `evt_projection_changed` before marking a turn terminal**

- Code impact:
  - `TurnMonitor` stores pre-submit projection hash and compares after `ready_for_input`
- Pros:
  - Prevents stale/empty projection on fast returns
- Cons:
  - If the tool returns identical text, the turn may stall until timeout

**Option B: Rely on `working` → `ready_for_input` transition only**

- Code impact:
  - Simpler TurnMonitor
- Pros:
  - Simple
- Cons:
  - Risk of stale projection in fast-return edge cases

#### Proposal (Recommended)

Gate terminality on `evt_projection_changed` OR observed `working` activity between submit and `ready_for_input`. If the runtime saw `working` at any point after submit, then `ready_for_input` is sufficient. If it never saw `working` (instant return), it must also see projection change before marking terminal. This handles the fast-return edge case without adding latency for the common case.

#### Pros / Cons (Proposal)

- Pros:
  - Common case (saw `working`) is not slowed down
  - Edge case (never saw `working`) is protected by projection check
- Cons:
  - More complex TurnMonitor logic (two conditions ORed)

> **DECISION: Gate terminality on `evt_projection_changed` OR observed `working` activity post-submit.**
> Rationale: The current `_wait_for_shadow_completion()` in `cao_rest.py` implicitly requires content change because `completed` status needs a post-baseline response marker. Removing that status eliminates the gate. The proposed OR-condition preserves the common-case fast path (where `working` is observed during polling) while protecting against the edge case where the TUI returns to idle before any poll captures the `working` state. This maps onto `_ShadowLifecycleTracker`'s existing phase tracking, which already distinguishes "pre-first-known-status" from "post-processing."
