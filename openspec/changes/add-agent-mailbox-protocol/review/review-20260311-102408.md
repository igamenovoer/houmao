## Summary

- The proposal and design are thorough, well-structured, and internally coherent across all five delta specs. The email-like mental model, filesystem-first transport, and explicit non-goals are well-calibrated for v1 scope.
- The canonical message model (YAML front matter + Markdown body, immutable content, mutable state in SQLite) is a strong design that separates concerns cleanly.
- The `tasks.md` artifact is missing; the change cannot proceed to implementation without concrete, ordered task definitions.
- The `mail` CLI subcommand design relies entirely on agent instruction-following for correctness (agent-mediated rather than runtime-direct), which is the highest-risk architectural choice and needs explicit fallback/validation strategy.
- The runtime skill injection mechanism ("reserved runtime-owned skill namespace") has no precedent in the current codebase; the existing `BrainBuilder` skill pipeline (`brain_builder.py:288-293`, `brain_builder.py:470-476`) only handles recipe-selected skills projected from `brains/skills/`. This change requires a new injection path that needs careful design to avoid conflicting with the existing skill projection model.
- The `rules/scripts/` approach (agents invoke shared Python/shell scripts for SQLite and lock operations) is a creative solution for correctness funneling, but agent compliance is soft-enforced and the scripts themselves need a clear authoring and versioning contract.

## Findings

### Must Fix / Blocking

1. **Missing `tasks.md` artifact.** The change status shows tasks as "ready but missing/not created yet." Without ordered implementation tasks, this change cannot progress to the apply phase. Tasks should reflect the seven-step migration plan in `design.md:565-573` and decompose each step into implementable units.

2. **No existing runtime skill injection path.** The current `BrainBuilder` (`brain_builder.py`) projects skills from `brains/skills/<name>/SKILL.md` into the brain home at build time via the tool adapter's `skills_destination` and `skills_mode` fields. There is no mechanism for the runtime to inject additional skills at session-start time. The design says mailbox system skills come from "platform-owned templates" and use a "reserved runtime-owned skill namespace," but the specs do not define:
   - Where platform-owned templates live on disk (inside the package? under `agent_def_dir`?).
   - How runtime-injected skills are merged with build-time-projected skills without namespace collisions.
   - Whether the injection happens during `start_runtime_session()` (in `runtime.py`) or during `build_launch_plan()` (in `launch_plan.py`), or requires a new phase between build and start.
   - How tool adapters for different backends (Codex, Claude, Gemini) surface runtime-injected skills differently.

   This needs at minimum a design note or spec scenario covering the injection mechanism.

3. **`mail` CLI subcommand integration point is underspecified.** The `brain-launch-runtime/spec.md` says the runtime CLI exposes `mail check|send|reply` and routes through `send-prompt`. But `cli.py` currently dispatches commands via `args.command` in `main()` (line 45-54). The spec does not clarify:
   - Whether `mail` is a top-level subcommand (`gig-agents-cli mail check`) or nested under an existing subparser.
   - Whether `mail` reuses `resume_runtime_session()` like `send-prompt` does, or needs its own resume path to inject the structured request template and parse the sentinel-delimited response.
   - How the runtime validates the `AGENTSYS_MAIL_RESULT_BEGIN/END` sentinel-delimited JSON response from the agent, given that `send_prompt()` returns `list[SessionEvent]` and the current event model (`SessionEvent` at `models.py:140-149`) has no structured payload parsing for sentinel-delimited agent output.

### Should Fix / Important

4. **Symlink semantics on non-Unix platforms.** The filesystem transport spec relies heavily on symlinks for mailbox projections (`inbox/`, `sent/` entries are symlinks to canonical messages) and for principal registration (`mailboxes/<principal>` can be a symlink). The design mentions hardlink fallback in one risk bullet (`design.md:553`) but the spec itself does not include a fallback scenario. If the project ever targets Windows or restricted filesystems, this could be a problem. At minimum, document that symlink support is a platform requirement for the filesystem transport.

5. **Lock ordering and deadlock prevention.** The design specifies acquiring per-principal locks in lexicographic order then `locks/index.lock` (`design.md:226-231`). The spec (`agent-mailbox-fs-transport/spec.md`) does not formally specify the lock acquisition order or timeout/retry semantics. Since agents will invoke shared scripts from `rules/scripts/` for locking, those scripts are the actual enforcement point. The spec should include a scenario covering lock ordering and deadlock avoidance, or explicitly delegate that to the shared script contract.

6. **SQLite journal mode constraint is design-only.** The design says "SQLite uses a non-WAL journal mode" (`design.md:274`), but this is not captured in any spec requirement or scenario. This is an important operational constraint (no `-wal`/`-shm` sidecar files) that should be formally specified in the fs-transport spec.

7. **`protocol-version.txt` is in the design layout but absent from specs.** The design shows `protocol-version.txt` at the mailbox root (`design.md:145`) but no spec requirement mentions it. If the protocol is versioned on disk, the spec should require it and define what happens when a participant encounters an unsupported version.

8. **Spec gap: how does `start-session` enable mailbox support?** The `brain-launch-runtime/spec.md` says "when filesystem mailbox support is enabled for a started session" but does not specify the enabling mechanism. Is it a CLI flag (`--enable-mailbox`), a recipe field, a blueprint field, or auto-detected from the agent definition directory? The existing `start-session` in `cli.py:88-123` has no mailbox-related arguments.

9. **Env var refresh mechanism is abstract.** The `agent-mailbox-system-skills/spec.md` specifies refreshable bindings but the `brain-launch-runtime/spec.md` only says "an explicit control path that refreshes." Is this a new CLI subcommand (`gig-agents-cli refresh-mailbox-bindings`)? An extension of `send-prompt`? A tmux environment update? The existing `RuntimeSessionController` (`runtime.py:89-101`) has no refresh/env-update method. The spec should commit to a concrete control surface.

10. **No spec coverage for `archive/` and `drafts/` folders.** The design layout shows `archive/` and `drafts/` as mailbox subfolders (`design.md:169-170`), but no spec scenario covers archiving or draft workflows. Either add scenarios or explicitly defer these to a follow-up.

### Nice To Have

11. **Consider a Pydantic model for the canonical message envelope.** The project already uses Pydantic v2 for boundary models (`boundary_models.py`). The canonical message envelope with its required fields, optional fields, and nested structures (sender/recipient principals, attachment entries) is a natural fit for a Pydantic model that can validate both in-memory representations and YAML front matter parsing.

12. **The `AGENTSYS_MAIL_REQUEST` / `AGENTSYS_MAIL_RESULT` sentinel contract could benefit from a formal schema.** The design shows the request as YAML-in-prompt and the result as JSON between sentinels. A JSON Schema for both would make validation implementable and testable without relying on ad hoc parsing.

13. **Consider adding a `--dry-run` mode to `mail send` and `mail reply`.** Since these are agent-mediated and involve filesystem mutations, a dry-run that shows the composed prompt without delivering it would help operators debug and build trust.

14. **The five delta specs have significant content overlap with the design.** This is common in spec-driven workflows, but during task creation, consider whether some spec requirements can be traced directly to design decisions to reduce maintenance burden.

## Design Alignment With Existing System

### Existing pattern(s) observed

- **Two-phase lifecycle (build then run):** `BrainBuilder` materializes a brain home at build time; `start_runtime_session()` / `resume_runtime_session()` handle the run phase. Skills are currently build-time artifacts projected into the brain home, not runtime-injected.
  - Files: `src/gig_agents/agents/brain_builder.py` (skill validation at line 288, skill projection at line 470), `src/gig_agents/agents/brain_launch_runtime/runtime.py`

- **CLI dispatch pattern:** `cli.py:main()` uses `argparse` subparsers with explicit `if args.command == "..."` dispatch (lines 45-54). Each subcommand gets its own `_cmd_*` handler.
  - File: `src/gig_agents/agents/brain_launch_runtime/cli.py`

- **`--agent-identity` resolution:** All session-control commands (`send-prompt`, `send-keys`, `stop-session`) use the same `resolve_agent_identity()` + `resume_runtime_session()` pattern.
  - File: `src/gig_agents/agents/brain_launch_runtime/runtime.py:274-306`

- **Backend-agnostic models:** `LaunchPlan`, `InteractiveSession`, `SessionEvent`, `SessionControlResult` in `models.py` form the backend-neutral interface. Backends implement `InteractiveSession` protocol.
  - File: `src/gig_agents/agents/brain_launch_runtime/models.py`

- **Agent identity namespace:** `AGENTSYS-...` identities are already used for tmux session naming and manifest resolution. The mailbox proposal reuses this namespace for `principal_id`, which aligns well.
  - File: `src/gig_agents/agents/brain_launch_runtime/agent_identity.py`

### Proposed design alignment assessment

**Aligns well:**

- Reusing `AGENTSYS-...` identities for mailbox principals is consistent with the existing identity model.
- The `mail` CLI following the `--agent-identity` + `resume_runtime_session()` pattern matches the existing CLI structure.
- Making the mailbox transport layer independent from any specific backend aligns with the backend-agnostic `InteractiveSession` protocol.
- Using the `runtime_root` as the default parent for the mailbox content root is consistent with how the runtime root is used for session manifests and brain homes.

**Mismatches requiring attention:**

- **Runtime skill injection breaks the build-time skill projection model.** The existing flow is: recipe selects skills -> `BrainBuilder` validates and projects them into the brain home -> backend reads them from the home. The proposed mailbox system skills are runtime-injected, not recipe-selected. This creates a second skill injection path that the `BrainBuilder`, tool adapters, and all five backends need to understand. Consider whether the runtime-owned skill templates should instead be projected during `build_brain_home()` as a special "system skill" category, which would keep the single-path model intact. If runtime injection is necessary (because the skill template depends on session-specific bindings), then `LaunchPlan` likely needs a new field or the injection should happen in `start_runtime_session()` before backend session creation.

- **`mail` command response parsing diverges from existing `send_prompt` output.** Today `send_prompt()` returns `list[SessionEvent]`, where each event has `kind`, `message`, `turn_index`, and optional `payload`. The `mail` command needs to parse a sentinel-delimited JSON blob from the agent's output text. This means either (a) the `mail` handler does its own post-processing on the `output_text` from the last `SessionEvent`, or (b) a new response parsing layer is added. Option (a) is simpler and more aligned with the current architecture.

- **No existing precedent for env var refresh on active sessions.** The current `RuntimeSessionController` does not modify session env after start. For tmux-backed sessions, env refresh could use `tmux set-environment`; for CAO, it would need an API call. The spec should clarify per-backend refresh semantics. The `LaunchPlan.env` dict (`models.py:89`) is frozen at build time and there is no mutable env surface on `RuntimeSessionController`.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Where should runtime-owned mailbox skill templates live, and how should they be injected into sessions?

- Decision: `Blocking`

#### Why this matters

- The existing skill pipeline is build-time only: `BrainBuilder` validates skill names against `brains/skills/` and projects them into the brain home. Runtime-injected skills are a new concept that needs a clear home and injection path.

#### Examples

- Agent session starts with recipe skills `[code-review, git-ops]` plus runtime-injected `[$email-via-filesystem]`. Where does `$email-via-filesystem` template live? How does it appear in the brain home alongside the projected recipe skills?

#### Why this question arises in code

- `brain_builder.py:470-476` projects recipe-selected skills by symlinking/copying from `brains/skills/<name>` into `home/<skills_destination>/<name>`. Runtime-owned templates are not in `brains/skills/`.
- `runtime.py:207-214` calls `build_launch_plan()` which does not handle skill injection.
- No existing code path adds skills between `build_brain_home()` and `start_runtime_session()`.

#### Options

**Option A: Inject at build time via a new "system skills" category in BrainBuilder**

- Code impact:
  - Add a `system_skills` field to `BuildRequest`. `BrainBuilder` projects system skills from a package-internal template directory into a separate subdirectory (e.g., `home/<skills_destination>/_system/`).
  ```python
  # brain_builder.py
  system_skills_root = Path(__file__).parent / "system_skill_templates"
  for sys_skill in request.system_skills:
      source = system_skills_root / sys_skill
      destination = skill_destination_dir / "_system" / sys_skill
      _project_path(source, destination, mode="copy")
  ```
- User-facing difference:
  - System skills appear in the brain home alongside recipe skills, visible in `build-brain` output.
- Pros:
  - Preserves single skill-projection path.
  - Skills are visible and inspectable in the brain home before session start.
- Cons:
  - Build time does not have session-specific mailbox bindings yet; templates must use env-var indirection (which the design already requires).
  - Requires rebuild to change system skill set.

**Option B: Inject at session-start time in RuntimeSessionController**

- Code impact:
  - Add a post-build skill injection step in `start_runtime_session()` that writes generated skill files into the brain home or a runtime-specific skill directory before creating the backend session.
  ```python
  # runtime.py, inside start_runtime_session()
  if mailbox_enabled:
      inject_mailbox_system_skills(
          brain_home=launch_plan.home_path,
          skills_destination=...,
          mailbox_config=...,
      )
  ```
- User-facing difference:
  - System skills are not visible in `build-brain` output but appear in the started session.
- Pros:
  - Can use session-specific context to parametrize templates.
  - Does not require BrainBuilder changes.
- Cons:
  - Creates a second skill injection path that bypasses BrainBuilder validation.
  - Skills written after build may not be visible to tool adapters that scan the home at build time.

> **PROPOSED: Option A (inject at build time via BrainBuilder system skills category).**
> Rationale: The existing skill lifecycle is build-time projection into the brain home (`brain_builder.py:470-476`). The design already requires env-var indirection for all mailbox bindings, so templates do not need session-specific literals. Keeping one skill projection path avoids a shadow injection channel that could cause confusion when debugging skill visibility. The `_system/` subdirectory namespace keeps runtime-owned skills separate from recipe-authored skills as the spec requires.
>

> Rationale: _To be filled by developer._

### Q2) How is filesystem mailbox support enabled for a session?

- Decision: `Blocking`

#### Why this matters

- The `brain-launch-runtime/spec.md` conditions all mailbox behavior on "when filesystem mailbox support is enabled" but no artifact defines the enabling mechanism. Without this, tasks cannot be written.

#### Examples

- Developer wants agent `AGENTSYS-research` to have mailbox. Do they add `--enable-mailbox` to `start-session`? Add `mailbox: { transport: filesystem }` to the recipe? Add it to the blueprint?

#### Why this question arises in code

- `cli.py:88-123` (`start-session` subparser) has no mailbox-related arguments.
- `LaunchPlanRequest` (`launch_plan.py:36-43`) has no mailbox field.
- `start_runtime_session()` (`runtime.py:185-271`) has no mailbox parameter.

#### Options

**Option A: Recipe-level opt-in (`brain-recipes/<tool>/*.yaml` gains a `mailbox` section)**

- Code impact:
  - `BrainRecipe` model gains `mailbox: { transport: filesystem, ... }`.
  - `BuildRequest` and `BrainBuilder` propagate mailbox config.
  - `start_runtime_session()` reads mailbox config from the brain manifest.
- User-facing difference:
  - Mailbox is declaratively bound to the agent recipe. Consistent across all sessions using that recipe.
- Pros:
  - Declarative, version-controlled, auditable.
  - Aligns with the existing recipe-driven configuration model.
- Cons:
  - Requires rebuild to change mailbox config.

**Option B: CLI flag on `start-session` (`--mailbox-transport filesystem`)**

- Code impact:
  - New CLI arguments on `start-session`.
  - `start_runtime_session()` accepts mailbox config.
- User-facing difference:
  - Mailbox is per-session, operator-controlled at start time.
- Pros:
  - Flexible, no rebuild needed.
- Cons:
  - Not declarative; easy to forget.
  - Diverges from recipe-driven convention.

**Option C: Blueprint-level binding (recipe + role + mailbox config)**

- Code impact:
  - Blueprint YAML gains a `mailbox` section.
  - `load_blueprint()` extracts mailbox config.
- User-facing difference:
  - Mailbox is bound at the blueprint level alongside recipe+role.
- Pros:
  - Keeps recipe reusable across mailbox/no-mailbox contexts.
  - Blueprint is already the composition layer.
- Cons:
  - Adds another config dimension to blueprints.

> **PROPOSED: Option A (recipe-level opt-in) with CLI override support.**
> Rationale: The existing system uses recipes as the primary declarative agent configuration (`brain_builder.py:214-222` loads recipe fields including tool, skills, config/cred profiles). Adding a `mailbox` section to recipes keeps configuration version-controlled and auditable while aligning with the existing pattern. A `--mailbox-transport` CLI flag can serve as an override or standalone path for ad hoc sessions, similar to how `--backend` overrides the recipe-default backend.

### Q3) Should the `mail` command parse sentinel-delimited agent output itself, or should a new structured-response model be added to the session event pipeline?

- Decision: `Deferrable`

#### Why this matters

- The `mail` command needs to extract a JSON result from between `AGENTSYS_MAIL_RESULT_BEGIN` and `AGENTSYS_MAIL_RESULT_END` sentinels in agent output. The current `SessionEvent` model has no structured response parsing.

#### Examples

- Agent returns a long response with reasoning text plus the sentinel-delimited JSON blob. The `mail` handler needs to find and extract just the JSON.

#### Why this question arises in code

- `send_prompt()` returns `list[SessionEvent]` (`models.py:186-198`). Each event has `message: str`.
- `_cmd_send_prompt()` (`cli.py:286-304`) serializes events as JSON lines. There is no post-processing.
- The `mail` handler would need to concatenate event messages, scan for sentinels, and parse the JSON between them.

#### Options

**Option A: Mail command handler does its own sentinel parsing**

- Code impact:
  - `_cmd_mail_check/send/reply()` in `cli.py` calls `controller.send_prompt(structured_mailbox_prompt)`, then scans `output_text` from events for the sentinel pair and extracts JSON.
  ```python
  events = controller.send_prompt(mailbox_prompt)
  full_output = "".join(e.message for e in events)
  result_json = extract_between_sentinels(full_output, "AGENTSYS_MAIL_RESULT_BEGIN", "AGENTSYS_MAIL_RESULT_END")
  ```
- User-facing difference:
  - No change to existing `send_prompt` contract. Mail-specific parsing is contained.
- Pros:
  - Minimal changes to existing code. Self-contained.
  - Does not impose sentinel parsing on non-mailbox callers.
- Cons:
  - If other structured-response commands are added later, each would need its own parsing.

**Option B: Add a `send_structured_prompt()` method to RuntimeSessionController**

- Code impact:
  - New method that wraps `send_prompt()` and extracts a structured payload based on configurable sentinels.
- User-facing difference:
  - Reusable for future structured-response patterns.
- Pros:
  - Reusable abstraction.
- Cons:
  - Speculative generalization for a single use case.

> **PROPOSED: Option A (mail command handler does its own sentinel parsing).**
> Rationale: The current codebase follows a pattern of command-specific handlers (`_cmd_send_prompt`, `_cmd_send_keys`, etc. in `cli.py:286-347`) that each handle their own response format. Adding sentinel parsing as a mail-specific concern keeps the blast radius small and avoids speculative abstraction. If a second structured-response command appears later, the pattern can be extracted then.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._

### Q4) Should mailbox principal bindings be persisted in the session manifest?

- Decision: `Deferrable`

#### Why this matters

- The design's open questions list this explicitly (`design.md:579`). The answer affects whether `resume_runtime_session()` can restore mailbox context and whether the `mail` command on a resumed session knows the bound principal.

#### Examples

- Developer starts a mailbox-enabled session, then later runs `gig-agents-cli mail check --agent-identity AGENTSYS-research`. The runtime resumes the session from its manifest. Does the manifest contain the mailbox principal ID and transport, or must the runtime re-derive them?

#### Why this question arises in code

- `SessionManifestPayloadV2` (`boundary_models.py`) currently has no mailbox-related fields.
- `persist_manifest()` (`runtime.py:170-182`) writes launch plan + backend state. No mailbox state.
- On resume (`resume_runtime_session()` at `runtime.py:309-365`), all session context comes from the persisted manifest.

#### Options

**Option A: Persist mailbox principal binding in session manifest**

- Code impact:
  - Add `mailbox` section to `SessionManifestPayloadV2` with `transport`, `principal_id`, `address`, `fs_root`.
  - `persist_manifest()` includes mailbox bindings.
  - `resume_runtime_session()` restores mailbox context.
- User-facing difference:
  - `mail` commands on resumed sessions work without re-specifying mailbox config.
- Pros:
  - Self-contained session state. Resume is complete.
  - Consistent with how backend state is already persisted.
- Cons:
  - Schema version bump for manifest format.

**Option B: Re-derive mailbox bindings from recipe/blueprint on resume**

- Code impact:
  - `resume_runtime_session()` re-reads recipe/blueprint to reconstruct mailbox config.
- User-facing difference:
  - Same UX, but mailbox config changes between start and resume would apply.
- Pros:
  - No manifest schema change.
  - Always reflects current config.
- Cons:
  - Requires recipe/blueprint to be available and unchanged on resume.
  - Inconsistent with the existing manifest-as-source-of-truth pattern.

> **PROPOSED: Option A (persist mailbox bindings in session manifest).**
> Rationale: The existing system treats the session manifest as the authoritative source of truth for resume operations. `SessionManifestPayloadV2` already includes backend-specific state sections (`headless`, `cao`) read by `resume_runtime_session()` (`runtime.py:309-365`). Adding a `mailbox` section follows the same pattern. A manifest schema version bump from v2 to v3 is straightforward given the existing `SESSION_MANIFEST_SCHEMA_VERSION` constant in `manifest.py:24`.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._

### Q5) What happens when an agent's `mail` response does not contain valid sentinel-delimited JSON?

- Decision: `Deferrable`

#### Why this matters

- Agent-mediated mailbox operations depend on the agent producing a correctly formatted response. Agent compliance is not guaranteed, especially across different LLM backends.

#### Examples

- Agent returns reasoning text but forgets the sentinels. Agent returns malformed JSON between sentinels. Agent returns multiple sentinel pairs.

#### Why this question arises in code

- The design shows a response contract (`design.md:534-538`) but no spec scenario covers validation failure or retry semantics.

#### Options

**Option A: Fail with a clear error, no retry**

- Code impact:
  - `_cmd_mail_*()` returns exit code 2 with a descriptive error when sentinel parsing fails.
- User-facing difference:
  - Operator sees "mailbox operation failed: agent did not return a valid structured result" and can retry manually.
- Pros:
  - Simple, predictable, matches v1 "fail fast" philosophy.
- Cons:
  - No automatic recovery.

**Option B: Retry once with a correction prompt**

- Code impact:
  - On parse failure, send a follow-up prompt asking the agent to re-emit the result in the correct format.
- User-facing difference:
  - Higher success rate at the cost of an extra turn.
- Pros:
  - More resilient to minor agent formatting errors.
- Cons:
  - Adds complexity. Second attempt may also fail. Two prompt turns for one logical operation.

> **PROPOSED: Option A (fail with clear error, no retry) for v1.**
> Rationale: The design explicitly states "fail fast" as the v1 philosophy for busy-session conflicts (`design.md:540`). The same principle applies to malformed responses. Retries add complexity and obscure whether the underlying agent can reliably follow the response contract. Operators can retry manually or adjust the mailbox skill template if failures are systematic.
>
> **DECISION: _(pick PROPOSED solution if not filled by developer)_**
> Rationale: _To be filled by developer._
