## Summary

- **Solid:** The change has a clear, well-articulated goal — decouple the live server tracking adapter from parser-derived authority and delegate TUI state reduction entirely to `shared_tui_tracking`. The proposal and design are internally coherent, and the existing codebase already supports this direction (the shared tracker is already instantiated and consulted for `surface`/`turn`/`last_turn` in `record_cycle`).
- **Solid:** The spec delta correctly modifies the existing `official-tui-state-tracking` capability with scenarios that codify the new raw-snapshot-only tracker input boundary, the parser-as-sidecar role, and observed-version-aware profile resolution.
- **Solid:** Tasks are ordered logically (identity/metadata → boundary refactor → verification → docs) and map cleanly to the design decisions.
- **Risk — synthetic snapshot fallback is the most complex removal:** The `_tracker_snapshot_text_from_host` function at `tracking.py:1262` is a significant mixed-authority seam; its `_synthetic_codex_snapshot` and `_synthetic_claude_snapshot` branches fabricate tracker input from parser fields. Removing this without breaking tests that rely on synthetic snapshots requires careful test migration (covered conceptually by Task 3.2 but not detailed).
- **Risk — `_should_infer_prompt_submission` still depends on parsed surface:** The surface-inference turn-arming path at `tracking.py:290–325` relies exclusively on `HoumaoParsedSurface` fields (`normalized_projection_text`, `_is_submit_ready`). The design says parser output should not be tracker authority, but this method feeds directly into anchor arming which is a tracker-authority action. This dependency is not addressed in the tasks.
- **Risk — lifecycle observation pipeline still parser-owned:** The `_lifecycle_observation_from_parsed_surface` function (tracking.py:1402) and the entire `build_readiness_pipeline` / `build_anchored_completion_pipeline` Rx chain are fed exclusively from parsed surface data. The design and tasks do not address whether this server-local lifecycle pipeline should remain, be removed, or be refactored to use tracker-owned state.

## Findings

### Must Fix / Blocking

1. **`_should_infer_prompt_submission` is a parser-to-tracker-authority bridge not addressed by the tasks.** This method at `tracking.py:290–325` uses `_is_submit_ready(previous_surface)` and `parsed_surface.normalized_projection_text` to decide whether to arm a turn anchor via `_arm_turn_anchor(source="surface_inference", ...)`. Arming a turn anchor is a tracker-authority action — it directly determines `turn.phase` and `last_turn.source`. The design says "parser output shall not be tracker authority" and the spec scenario "Parsed surface does not replace raw tracker input" says the adapter "does not synthesize or substitute tracker input from parser-owned fields during normal live execution." Yet this method does exactly that: it uses parser-owned readiness and projection-growth heuristics to create tracker-side turn semantics. The tasks need to either (a) migrate this inference to the shared tracker session (which already has its own turn-arming mechanism via `on_input_submitted()`), (b) replace the parser-derived inputs with raw-text equivalents from the shared tracker's signal detection, or (c) explicitly document this as a server-owned sidecar action that arms the shared tracker but uses server evidence. Without addressing this, the authority boundary is still violated after implementation.

> **DECISION: Accept, with a narrower fix.**
> Rationale: The gap is real, but the right artifact change is not to invent a new server-side raw-text heuristic. The shared tracker already arms `surface_inference` from raw snapshots in `src/houmao/shared_tui_tracking/session.py`, so this change should remove parser-derived `_should_infer_prompt_submission()` from tracker-authority flow and rely on the existing shared tracker inference while keeping server-local explicit-input anchoring only for server-owned lifecycle enrichment.

2. **The server-local lifecycle Rx pipeline (`build_readiness_pipeline` / `build_anchored_completion_pipeline`) remains entirely parser-fed, and the tasks do not address its fate.** `record_cycle` at tracking.py:508–513 feeds `_lifecycle_observation_from_parsed_surface(parsed_surface=...)` into `m_observation_subject`, which drives `m_readiness_snapshot_queue` and `m_completion_snapshot_queue`. These pipelines produce `SurfaceReduction` (readiness_state, completion_state) which feeds into `_build_operator_state`. The shared tracker session (`TuiTrackerSession`) has its own settle/timing logic. After this change, the server will have two parallel reduction paths: one from the shared tracker (authoritative for `surface`/`turn`/`last_turn`) and one from the server-local lifecycle Rx pipeline (feeding `operator_state`). The design and tasks should clarify whether `operator_state` and its underlying lifecycle pipelines remain as server-owned enrichment, or whether they should be simplified/removed now that the shared tracker handles timing. At minimum, the relationship between the two pipelines needs to be documented.

> **DECISION: Accept.**
> Rationale: The design/tasks currently leave the fate of the parser-fed Rx lifecycle pipeline implicit. This change will keep `build_readiness_pipeline` / `build_anchored_completion_pipeline` as server-owned enrichment for `operator_state`, `lifecycle_timing`, and `lifecycle_authority`, but the artifacts need to say explicitly that those fields are not authoritative for `surface`, `turn`, or `last_turn`.

### Should Fix / Important

3. **Task 2.1 is under-specified about what "raw tmux pane text is the normal input" means for the `_tracker_snapshot_text_from_host` compatibility fallback.** The current function (tracking.py:1262–1285) has three code paths: (a) raw text if it "looks tracker informative," (b) synthetic snapshot from parsed surface if raw text doesn't look informative, (c) raw text as-is if no parsed surface exists. Path (b) is the authority violation. Task 2.2 says "remove or isolate" synthetic tracker input, but it's unclear whether path (a)'s `_raw_text_looks_tracker_informative` heuristic also needs to change. If the raw text is always used directly (since `output_text` is already available at the `poll_known_session` call site for supported-tool cycles), the entire `_tracker_snapshot_text_from_host` function may reduce to a simple pass-through. The task should specify this.

> **DECISION: Accept.**
> Rationale: The normal live path already has raw `output_text` available in `service.py` and passes it into `record_cycle()`. The artifacts should say that supported live cycles pass raw tmux text straight through to the shared tracker, and that `_tracker_snapshot_text_from_host()` compatibility heuristics plus synthetic fallback are removed from normal live execution, with any remaining synthetic helper constrained to tests only if implementation still needs one.

4. **`KnownSessionRecord` and `_ManifestMetadata` in `registry.py` do not carry `observed_version`.** Task 1.1 says "extend server-owned tracked-session and registration metadata to carry optional observed tool-version." Task 1.2 says "load from manifest-backed launch/session provenance." The current `_ManifestMetadata` (registry.py:177–185) does not extract any version field from the manifest, and `KnownSessionRecord` (registry.py:24–36) has no `observed_version` field. Neither does `HoumaoTrackedSessionIdentity` in `models.py`. The tasks should explicitly list which data classes and models need the new field, since the wiring runs through at least: `_ManifestMetadata` → `KnownSessionRecord` → `HoumaoTrackedSessionIdentity` → `LiveSessionTracker.__init__` → `TuiTrackerSession.from_config(observed_version=...)`.

> **DECISION: Accept.**
> Rationale: The task list is too abstract here. The affected server-owned types should be named explicitly, and the manifest source should be grounded in the existing `launch_policy_provenance.detected_tool_version` path rather than left implicit.

5. **Task 1.3 says "update live tracker admission and rebuild logic so shared tracker profile resolution depends on tool identity plus observed version," but `_ensure_tracker_session_locked` only rebuilds when `app_id` changes.** Currently (tracking.py:178–191), the rebuild guard is `if tracker_app_id == self.m_tracker_app_id: return`. If observed version changes without changing the tool (e.g., Claude 2.0 → 2.1), the tracker session won't rebuild. The rebuild logic needs to also track and compare `observed_version`. This is an implementation detail but important enough that the task should mention it explicitly.

> **DECISION: Accept.**
> Rationale: This is a concrete implementation requirement, not an optional cleanup. The artifacts should call out that tracker-session rebuilds depend on both tool family and observed tool version, because the current `app_id`-only guard would miss profile changes within the same tool family.

6. **The `_should_infer_prompt_submission` method depends on `m_last_state.stability.stable`, which is computed from a signature that includes `parsed_surface` (tracking.py:616–625).** If the change intends to make stability independent of parser authority, the stability signature computation should also be reviewed. Currently `_build_stability` takes `parsed_surface` as input. This creates a circular authority concern: parser data feeds stability, stability gates surface-inference turn arming, and turn arming is tracker authority. Even if `_should_infer_prompt_submission` is refactored, the stability-to-parser coupling needs attention.

> **DECISION: Refine.**
> Rationale: Visible stability is intentionally server-owned over the published response, and while `parsed_surface` remains part of that response it is valid for `stability` to continue including parser-owned fields. The artifact change needed here is narrower: parser-derived stability must stop gating tracker-authority surface inference once `_should_infer_prompt_submission()` is removed from the tracking path.

### Nice To Have

7. **The design's "Migration Plan" step 4 says "confine to tests/debug seams" for synthetic tracker input, but there is no specification of what those seams look like.** Defining a clear test-only injection point (e.g., a constructor parameter or a test-only subclass) would make the boundary more maintainable than ad-hoc `if DEBUG`-style guards. This is a code-quality suggestion, not a correctness requirement.

> **DECISION: Reject.**
> Rationale: The OpenSpec change does not need to design a formal public test seam. The normative requirement is only that synthetic tracker input leaves the normal live path; if implementation later needs a tiny test-only helper, that can be introduced locally without elevating it into the design/spec surface.

8. **The `note_prompt_submission` method (tracking.py:327–407) uses `m_last_state.parsed_surface.normalized_projection_text` to set `baseline_projection_text` for the turn anchor.** This is another parser-to-tracker data path. While it's less critical than the surface-inference arming (it's used for anchor bookkeeping rather than state classification), it should be audited for whether a raw-text equivalent is available from the shared tracker session state.

> **DECISION: Refine.**
> Rationale: The reviewer correctly spotted another parser-derived data path, but `note_prompt_submission()` already calls `m_tracker_session.on_input_submitted()` separately and uses the parsed baseline only for server-owned completion monitoring. This change should document that boundary clearly, but it does not need to replace the lifecycle baseline with a raw-text equivalent.

9. **Spec scenario "Observed tool version informs live profile selection when available" does not specify graceful degradation behavior.** The design says "make the new metadata optional" and "graceful fallback," and the spec says "does not default to an unspecified tracker profile merely because the session is live rather than standalone." But the spec doesn't define what happens when version metadata is missing — does it resolve to the same fallback profile as `observed_version=None` today, or does it become a diagnostic warning? The current behavior (`observed_version=None` at tracking.py:117,188) produces a valid fallback. The spec should codify that this existing fallback remains acceptable.

> **DECISION: Accept.**
> Rationale: The design already intends graceful fallback when version metadata is absent, but the delta spec does not state it explicitly enough. The spec should say that missing observed-version metadata falls back to the shared tracker’s compatible default profile rather than becoming an error or warning-only state.

  - The `_tracker_snapshot_text_from_host` function (tracking.py:1262–1285) is the primary mixed-authority seam — it synthesizes tracker input from parser fields when raw text "doesn't look tracker-informative."

- **Proposed design alignment:**
  - The design aligns well with the existing shared tracker contract. The `TuiTrackerSession.on_snapshot(text)` and `on_input_submitted()` methods are exactly the tracker input boundary the design wants to enforce.
  - The `observed_version` plumbing aligns with existing patterns: the shared registry already supports version-aware resolution, and `KnownSessionRecord` already carries enrichment from manifest metadata.

The shared tracker has its own turn anchor mechanism. The design should clarify whether server-local anchors survive as a parallel system or are replaced by the shared tracker's anchor handling.

## Open Questions (Reviewer Proposal / Developer Decision)

### Q1) Should the server-local lifecycle Rx pipeline (`build_readiness_pipeline` / `build_anchored_completion_pipeline`) be retained, simplified, or removed?

- Decision: `Blocking`

#### Why this matters

The server currently runs two parallel state-reduction paths:
1. **Shared tracker** (`TuiTrackerSession`): owns `surface`/`turn`/`last_turn` from raw snapshots.
2. **Server-local lifecycle pipeline** (`LifecycleObservation` → readiness/completion Rx pipelines → `SurfaceReduction`): owns `operator_state` (readiness_state, completion_state) from parser-derived observations.

After this change, path 1 becomes the authority for TUI state. But path 2 still feeds `operator_state`, which the public route includes. If both paths remain, the two reduction systems may disagree on timing (e.g., the shared tracker says `turn.phase=ready` while the lifecycle pipeline says `completion_state=in_progress`), and consumers may be confused about which to trust.

#### Examples

- Shared tracker settles a turn to `last_turn.result=success` via its internal timer. Simultaneously, the lifecycle pipeline's `build_anchored_completion_pipeline` hasn't reached `candidate_complete` yet because its timer parameters differ. `operator_state.completion_state` would show `in_progress` while `last_turn.result` shows `success`.
- A parser failure cycle produces no `LifecycleObservation`, so the lifecycle pipeline stalls, while the shared tracker continues reducing from raw snapshots. The two paths diverge until the next successful parse.

#### Why this question arises in code

- `record_cycle` at tracking.py:508–513 feeds `_lifecycle_observation_from_parsed_surface` into the lifecycle Rx pipeline.
- tracking.py:515–518 computes `reduction = self._reduction_from_current_snapshots(...)` from the lifecycle pipeline output.
- tracking.py:534–543 builds `operator_state` from that `reduction`.
- tracking.py:571–594 reads the shared tracker state and builds `surface`/`turn`/`last_turn` from it.
- The two systems coexist in the same `record_cycle` call with no documented authority hierarchy.

#### Options

**Option A: Keep the lifecycle pipeline but explicitly document it as server-owned enrichment for `operator_state` only, separate from tracker authority.**

- Code impact:
  - No code removal. Add clear doc comments and update state-tracking.md to explain the two parallel paths.
  - `operator_state` continues to be derived from parser-fed lifecycle pipeline.
- User-facing difference:
  - No behavioral change. `operator_state` remains available but is explicitly documented as non-authoritative for turn semantics.
- Pros:
  - Minimal code churn. Preserves existing `operator_state` behavior for consumers that depend on it.
- Cons:
  - Two parallel reduction paths remain, with potential for timing disagreements. Future maintenance burden.

**Option B: Remove the lifecycle pipeline and derive `operator_state` from the shared tracker state.**

- Code impact:
  - Remove `build_readiness_pipeline`, `build_anchored_completion_pipeline`, `LifecycleObservation`, `SurfaceReduction` from the live tracking path.
  - Rewrite `_build_operator_state` to derive from `TrackedStateSnapshot` fields.
  - Remove or mark `houmao.lifecycle` as legacy/test-only.
- User-facing difference:
  - `operator_state` timing may change slightly. Any consumer relying on the specific readiness/completion lifecycle semantics would see different behavior.
- Pros:
  - Single reduction authority. Cleaner boundary. Easier to maintain.
- Cons:
  - Larger code change. May break existing `operator_state` consumers. Potentially out of scope for this change.

> **PROPOSED: Option A — keep the lifecycle pipeline as server-owned enrichment, with explicit documentation and a follow-up task to evaluate removal.**
> Rationale: The design explicitly says "replacing server-owned diagnostics, stability, or route shaping with shared-tracker-owned equivalents" is a non-goal. The `operator_state` fields (`readiness_state`, `completion_state`, `detail`) serve existing server routes and demo displays. Removing the lifecycle pipeline would be a larger change than what this proposal scopes. However, the dual-path confusion should be documented clearly in `state-tracking.md` and a follow-up change should evaluate whether `operator_state` can be simplified to derive from the shared tracker.
>
> **DECISION: Accept Option A with refinement.**
> Rationale: This change will keep the parser-fed lifecycle Rx pipeline as server-owned enrichment for `operator_state`, `lifecycle_timing`, and `lifecycle_authority`, because removing it would broaden scope beyond the requested boundary cleanup. The artifacts will be revised so that this pipeline is explicitly non-authoritative for `surface`, `turn`, and `last_turn`, and so any timing disagreement is documented as an ownership split rather than an undefined state model.

### Q2) Should `_should_infer_prompt_submission` be migrated to the shared tracker, refactored to use raw-text signals, or explicitly preserved as a server-owned bridge?

- Decision: `Blocking`

#### Why this matters

This is the primary remaining parser-to-tracker authority path after the synthetic snapshot removal. It uses parser-derived `_is_submit_ready()` and `normalized_projection_text` growth to arm turn anchors that directly determine `turn.phase` and `last_turn.source`. The spec says parsed surface "SHALL NOT replace raw snapshot text as the authoritative tracking source" and "does not synthesize or substitute tracker input from parser-owned fields during normal live execution." Surface-inference turn arming from parser fields is arguably a violation of this contract.

#### Examples

- The parser fails for one cycle, then succeeds the next cycle with a significantly grown projection. `_should_infer_prompt_submission` sees growth and arms a turn — but the raw snapshot may not have changed meaningfully (the parser may have recovered differently). The turn anchor is parser-derived, not raw-text-derived.
- The shared tracker already detects surface changes from raw snapshots and has its own turn-arming via `on_input_submitted()`. Duplicating the inference in the server adapter may cause disagreements.

#### Why this question arises in code

- `_should_infer_prompt_submission` at tracking.py:290–325 reads `self.m_last_state.parsed_surface` and the current `parsed_surface`.
- It calls `_is_submit_ready(previous_surface)` (tracking.py:1446–1453) which checks `parsed_surface.availability == "supported"` and `parsed_surface.business_state == "idle"`.
- It calls `normalize_projection_text(parsed_surface.normalized_projection_text)` to measure growth.
- On success, it calls `_arm_turn_anchor(source="surface_inference", ...)` at tracking.py:473–483.
- The shared tracker's `_handle_snapshot_event` in `session.py` already has its own surface-change detection logic.

#### Options

**Option A: Migrate surface inference into the shared tracker session.**

- Code impact:
  - Add a "surface inference" detection path to `TuiTrackerSession` that works from raw snapshots.
  - Remove `_should_infer_prompt_submission` from `LiveSessionTracker`.
  - The shared tracker's `on_snapshot` would handle both explicit and inferred turn arming.
- User-facing difference:
  - The inference threshold may change since it would use raw text metrics instead of parser-derived projection metrics.
- Pros:
  - Clean authority boundary. Single place for all turn-arming logic.
- Cons:
  - Requires changes to the shared tracker contract, which the design explicitly says is a non-goal ("Redesigning the standalone tracker contract itself beyond the live adapter boundary needed here").

**Option B: Refactor `_should_infer_prompt_submission` to use the shared tracker's signal state instead of parser fields.**

- Code impact:
  - Replace `_is_submit_ready(previous_surface)` check with `tracker_state.surface_ready_posture == "yes"`.
  - Replace projection-text growth check with a raw-text-based growth metric (e.g., comparing raw `output_text` lengths across cycles).
  - Keep the method in `LiveSessionTracker` but make it parser-independent.
- User-facing difference:
  - Inference may trigger on different boundaries since raw text metrics differ from parser-normalized projection metrics.
- Pros:
  - Preserves server-owned surface inference without violating tracker authority.
- Cons:
  - Raw text growth is noisier than parser-normalized projection growth. May need additional heuristics to avoid false positives.

**Option C: Explicitly preserve as a server-owned bridge with documentation that this is a known authority exception.**

- Code impact:
  - Add doc comments and spec wording that surface-inference turn arming is a server-owned bridge action that uses parser evidence to arm the shared tracker, distinct from direct tracker input.
  - No code change to the method itself.
- User-facing difference:
  - None.
- Pros:
  - Minimal code churn. Pragmatic.
- Cons:
  - Weakens the "parser is not tracker authority" contract. May confuse future maintainers.

> **PROPOSED: Option B — refactor `_should_infer_prompt_submission` to use shared tracker signals and raw text metrics.**
> Rationale: The design explicitly says "Redesigning the standalone tracker contract itself" is a non-goal, ruling out Option A. Option C undermines the core design intent. Option B preserves the server-owned inference mechanism while removing its parser dependency. The shared tracker's `current_state().surface_ready_posture` already provides the readiness check, and raw `output_text` (already available at the `record_cycle` call site via `service.py:1431`) provides a parser-independent growth metric. This keeps the inference in the server adapter where it belongs while honoring the raw-text authority boundary.
>
> **DECISION: Reject the proposed Option B and adopt a narrower variant: remove the server-side parser-derived inference and rely on the existing shared tracker surface-inference path.**
> Rationale: The shared tracker already performs raw-snapshot `surface_inference` in `src/houmao/shared_tui_tracking/session.py`, so there is no need to replace `_should_infer_prompt_submission()` with a second server-local raw-text heuristic. The artifacts will be revised so server-local explicit-input anchoring remains only for lifecycle enrichment, while shared tracker inference remains the sole source of public `surface_inference` turn authority.

### Q3) Where should observed tool version be extracted from in the manifest?

- Decision: `Deferrable`

#### Why this matters

Task 1.2 says "load observed tool-version metadata from manifest-backed launch/session provenance" but does not specify which manifest field contains the version. The manifest structure (inspected via `_load_manifest_metadata` at registry.py:188–225) has tool-name under `payload["tool"]` and backend state under `payload["backend_state"]`, but no existing version field. The manifest would need a new field, or the version would need to be extracted from an existing field (e.g., a semver in `backend_state`, or from the brain recipe metadata).

#### Examples

- A manifest contains `{"tool": "claude", "brain_recipe": {"tool_version": "2.1.3"}}` — the version is nested in recipe metadata.
- Alternatively, the version is probed at registration time and stored in the registration payload rather than the manifest.

#### Why this question arises in code

- `_load_manifest_metadata` at registry.py:188–225 currently extracts `tool`, `terminal_id`, `tmux_session_name`, `tmux_window_name`, and `session_root` from the manifest payload.
- No version field is currently extracted.
- The `HoumaoRegisterLaunchRequest` model in `models.py` is the registration payload — it also has no `observed_version` field currently.
- The shared tracker's `TuiTrackerSession.from_config(observed_version=...)` accepts a string.

#### Options

**Option A: Extract from manifest payload, adding a new optional `tool_version` or `observed_tool_version` field.**

- Code impact:
  - Add `observed_tool_version: str | None` to `_ManifestMetadata`.
  - Extract from `payload.get("tool_version")` or a nested path.
  - Thread through `KnownSessionRecord` → `LiveSessionTracker`.
- User-facing difference:
  - Version-aware profile selection works for sessions launched with version-aware manifests.
- Pros:
  - Manifest is already the primary provenance source. Consistent with existing enrichment pattern.
- Cons:
  - Requires upstream manifest producers to populate the field.

**Option B: Add to registration payload and extract from `HoumaoRegisterLaunchRequest`.**

- Code impact:
  - Add `observed_tool_version: str | None = None` to `HoumaoRegisterLaunchRequest`.
  - Use in `known_session_record_from_registration()`.
- User-facing difference:
  - Registration callers can supply version info. Works even without a manifest.
- Pros:
  - Registration is the server-owned admission path. Independent of manifest format.
- Cons:
  - Duplicates provenance if manifest also carries version.

> **PROPOSED: Support both sources with manifest as primary and registration as fallback, matching the existing enrichment pattern in `known_session_record_from_registration`.**
> Rationale: The existing `known_session_record_from_registration` function (registry.py:115–174) already follows this pattern: registration fields are used first, then enriched/overridden from manifest metadata and shared registry. Adding `observed_tool_version` to both `HoumaoRegisterLaunchRequest` and `_ManifestMetadata`, with manifest taking precedence, is consistent with how `terminal_id`, `tmux_window_name`, and `tool` are already handled at lines 149–157. This also aligns with the design's statement: "Manifest-backed launch/session provenance is the primary source. Registration payloads may carry an override or fallback when manifest provenance is absent."
>
> **DECISION: Refine the proposal: use existing manifest-backed `launch_policy_provenance.detected_tool_version` as the primary source, with registration fallback when manifest provenance is absent.**
> Rationale: The repository already persists `detected_tool_version` in session-manifest boundary models under `launch_policy_provenance`, so no new manifest field is needed. The artifacts will be revised to name that exact manifest path and keep registration payload support only as fallback enrichment when the manifest is missing or incomplete.

### Q4) Should the server-local turn anchor system coexist with the shared tracker's turn anchor handling?

- Decision: `Deferrable`

#### Why this matters

The `LiveSessionTracker` maintains its own turn anchor system (`m_active_turn_anchor`, `_arm_turn_anchor`, `_lose_turn_anchor`, anchor IDs, anchor expiry) in tracking.py:139–144 and related methods. The shared `TuiTrackerSession` also has turn-arming via `on_input_submitted()` and its own anchor tracking via `m_armed_turn_source` (session.py:91). After this change, both systems will be active simultaneously. The `record_cycle` method at tracking.py:572–583 uses `self.m_active_turn_anchor` (server-local) alongside `tracker_state.turn_phase` (shared tracker) in the same decision block, creating a hybrid anchor-management model.

#### Examples

- Server arms `m_active_turn_anchor` via `note_prompt_submission`, and also calls `m_tracker_session.on_input_submitted()`. Both systems now think a turn is active. If the shared tracker settles the turn first, the server-local anchor may still be alive, causing stale `lifecycle_authority.turn_anchor_state = "active"` while `turn.phase = "ready"`.

#### Why this question arises in code

- tracking.py:336–340: `note_prompt_submission` calls both `self._ensure_tracker_session_locked` and `self.m_tracker_session.on_input_submitted()`, then also `self._arm_turn_anchor(...)`.
- tracking.py:572–583: `record_cycle` checks `self.m_active_turn_anchor` and `tracker_state.last_turn_result` together.
- session.py:91–92: The shared tracker has `m_armed_turn_source` and `m_pending_success_signature`.

#### Options

**Option A: Keep both anchor systems and document the dual-anchor model explicitly.**

- Code impact:
  - Add documentation explaining that the server-local anchor drives `lifecycle_authority` and `operator_state`, while the shared tracker anchor drives `turn`/`last_turn`.
- User-facing difference:
  - `lifecycle_authority.turn_anchor_state` and `turn.phase` may briefly disagree during transitions.
- Pros:
  - Minimal code churn. Preserves existing server-owned lifecycle authority metadata.
- Cons:
  - Dual anchor systems are confusing. Timing disagreements are likely.

**Option B: Defer server-local anchor removal to a follow-up change.**

- Code impact:
  - Same as Option A for this change. Create a follow-up task to evaluate replacing server-local anchors with shared tracker anchor state.
- User-facing difference:
  - None for this change.
- Pros:
  - Keeps this change focused on the core authority split. Avoids scope creep.
- Cons:
  - Technical debt accumulates.

> **PROPOSED: Option B — defer to a follow-up change, but add a task to document the dual-anchor model in this change's docs update (Task 4.1).**
> Rationale: The design explicitly scopes out "redesigning the standalone tracker contract itself beyond the live adapter boundary needed here." Removing the server-local anchor system would be a significant refactor that changes `lifecycle_authority` semantics, which is a server-owned concept not covered by this change. However, Task 4.1's docs update should explicitly document the dual-anchor relationship so future maintainers understand the intended evolution path. A concrete follow-up task entry would also help track this technical debt.
>
> **DECISION: Accept Option B with refinement.**
> Rationale: Dual anchor systems will remain for this change, but the split will be narrowed and documented: shared tracker anchoring remains authoritative for public `surface`/`turn`/`last_turn`, while the server-local anchor remains a lifecycle-enrichment mechanism tied to explicit input and `lifecycle_authority`. Broader anchor unification is follow-up work, not part of this boundary cleanup.
