### Validation and business-rule `422` responses were expected

Initial malformed or shape-incomplete probes produced `422` on several routes. Those responses came from FastAPI/Pydantic request validation and confirmed that the routes were mounted.

The most important follow-up probe was:

- `POST /houmao/agents/headless/launches`
  - schema-valid payload:
    `{"tool":"codex","working_directory":"/tmp","agent_def_dir":"/tmp","brain_manifest_path":"/tmp/brain.json"}`
  - response:
    `422 Native headless launch requires an existing brain_manifest_path file.`

This is expected service behavior. The implementation explicitly validates:

- existing `working_directory`
- `working_directory` is a directory
- existing `agent_def_dir`
- existing `brain_manifest_path`
- successful manifest load
- request `tool` matches `brain_manifest_path.inputs.tool`
- resolved backend is one of the supported headless backends

That means the `422` observed here is a deliberate semantic rejection of an unusable launch request, not a reachability failure and not an implementation bug.

## Shutdown endpoint behavior

- `POST /houmao/server/shutdown` returned `{"success":true}`
- Follow-up `houmao-mgr server status --port 13444` confirmed the server stopped
- The server was then restarted in tmux session `houmao-server-13444`
- Final status check confirmed the server is running again on `127.0.0.1:13444`

## Conclusion

- All non-`/cao/*` endpoints exposed by the live `houmao-server` were reachable during this probe session.
- No bug was identified from the observed `422` on native headless launch.
- For this API surface:
  - `200` means the route completed successfully
  - `404` on dummy refs means the route exists and rejected an unknown resource
  - `422` means either request validation failed or the request violated explicit service-side business rules
