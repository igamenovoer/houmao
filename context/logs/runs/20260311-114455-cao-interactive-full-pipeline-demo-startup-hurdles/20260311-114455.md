## Summary

The demo eventually started successfully, but startup hit two separate environment-level hurdles before the final run succeeded:

1. A stale tmux session named `AGENTSYS-chris` was still present from an older run and did not publish the newer `AGENTSYS_AGENT_DEF_DIR` session env var.
2. A healthy local `cao-server` was already listening on `127.0.0.1:9889`, but the demo's launcher-managed stop path crashed before emitting JSON because its run-local artifact directory did not exist.

No code was modified during recovery. The session was brought up using manual cleanup steps and then the normal demo startup path.

## Hurdle 1: stale `AGENTSYS-chris` tmux session without `AGENTSYS_AGENT_DEF_DIR`

### Symptom

Initial launch attempts failed during the demo's "replace previous interactive session state" phase with:

```text
error: Failed to stop the interactive CAO session (see `.../logs/replacement-stop-1.stderr`).
```

The stderr log showed:

```text
error: Agent definition pointer missing: tmux session `AGENTSYS-chris` has no `AGENTSYS_AGENT_DEF_DIR` value.
```

### Diagnosis

- `tmux has-session -t AGENTSYS-chris` succeeded.
- An older persisted state file still marked `AGENTSYS-chris` active:
  `tmp/demo/cao-interactive-full-pipeline-demo/20260311-091236-865912Z/state.json`
- That earlier run used `codex/gpu-kernel-coder-yunwu-openai`.
- The session predated the new runtime behavior that rehydrates name-addressed control paths from tmux-published `AGENTSYS_AGENT_DEF_DIR`.

### Recovery

1. Stopped the remote runtime session with an explicit override:

```bash
pixi run python -m gig_agents.agents.brain_launch_runtime.cli stop-session \
  --agent-def-dir tests/fixtures/agents \
  --agent-identity AGENTSYS-chris
```

2. The stale tmux session still existed after the remote stop, so it was removed directly:

```bash
tmux kill-session -t AGENTSYS-chris
```

### Outcome

After tmux cleanup, the next startup attempt passed the old-session replacement phase and moved on to CAO server setup.

## Hurdle 2: launcher-managed `cao-server` stop crashed with missing artifact directory

### Symptom

Once the stale tmux session was cleared, startup advanced to CAO setup and then failed with:

```text
error: Invalid JSON in CAO launcher stop output.
```

### Diagnosis

- `ss -ltnp '( sport = :9889 )'` showed a live listener:
  `cao-server` on `127.0.0.1:9889`
- The demo's CAO status output confirmed the existing service was healthy and identified as `cli-agent-orchestrator`.
- The failing launcher stop stderr contained a Python traceback ending in:

```text
FileNotFoundError: [Errno 2] No such file or directory: '.../runtime/cao-server/127.0.0.1-9889/launcher_result.json'
```

This means the stop command crashed before producing JSON, so the demo reported it as "Invalid JSON in CAO launcher stop output."

### Recovery

The existing verified `cao-server` process was terminated directly instead of relying on the broken launcher-managed stop path:

```bash
kill 2147472
```

The port was then confirmed clear before retrying startup.

### Outcome

With `127.0.0.1:9889` free and `cao-server` available on `PATH`, the demo was able to launch a fresh local CAO server through its normal startup flow.

## Bug record

### Title

Interactive CAO demo cannot reliably replace an already-healthy fixed-port `cao-server`

### Expected behavior

When a healthy verified `cao-server` is already bound to `http://127.0.0.1:9889`, the interactive demo should stop it automatically and continue startup without manual intervention.

### Actual behavior

The demo detects the healthy server correctly, but its launcher-managed stop path crashes before returning JSON. The demo then aborts startup with:

```text
error: Invalid JSON in CAO launcher stop output.
```

### Root cause

The underlying launcher stop command raises:

```text
FileNotFoundError: [Errno 2] No such file or directory: '.../runtime/cao-server/127.0.0.1-9889/launcher_result.json'
```

This happens while `stop_cao_server()` attempts to write its result artifact. Because the write target's parent directory does not exist yet, the stop path crashes before producing structured JSON, and the demo treats that as malformed stop output.

### Scope

- The bug affects the interactive demo's startup replacement flow for pre-existing local `cao-server` instances on the demo's fixed loopback target.
- It does not prevent the demo from replacing an existing same-name interactive session.

---

## Proposed fix

### Primary fix

Make the CAO launcher stop path create the result artifact parent directory before writing `launcher_result.json`.

Concretely:

- In the CAO server launcher implementation, ensure `artifacts.launcher_result_file.parent.mkdir(parents=True, exist_ok=True)` happens before `_write_launcher_result(...)`.
- Treat launcher artifact directory creation as part of stop-path setup, not only start-path setup.

### Why this is the right fix

- The existing server is already verified as the expected service (`cli-agent-orchestrator`), so replacement should be routine.
- The current failure is not ambiguity in process identity or safety policy; it is a filesystem precondition bug in the stop command's reporting path.
- Fixing directory creation at the launcher layer benefits all callers, not just this demo wrapper.

### Defensive improvement in the demo layer

Even after fixing the launcher, the demo could be made more robust by adding a fallback path:

- If launcher-managed stop fails after a healthy verified `cao-server` status check, fall back to the existing PID-based verified termination logic already present in `_replace_existing_cao_server()`.
- When that fallback succeeds, continue startup rather than surfacing the JSON parsing failure as fatal.

### Suggested regression test

Add an integration-style test covering this sequence:

1. Start a real or stubbed healthy `cao-server` on `127.0.0.1:9889`.
2. Run the interactive demo `start`.
3. Assert the demo replaces the existing service and still reaches `session_status: active`.
4. Assert no `Invalid JSON in CAO launcher stop output` error is surfaced.

This should be paired with a narrower unit test for the launcher stop path that verifies result-file parent directories are created automatically before writing stop artifacts.
