# Agent Loop Correctness Evaluator Feature Requirement

## Goal

Provide an automatic evaluation capability for agent loops generated by `houmao-agent-loop-lite` and `houmao-agent-loop-pro`. The capability is delivered as a Houmao system skill that teaches the operator agent who writes an agent-team execplan to design test cases for that execplan and to run those test cases in a bounded, isolated mode. The skill surfaces correctness problems before or during live execution so that loop authors no longer depend on manual user debugging.

## Non-Goals

- It is not a generic debugger for arbitrary agent code outside the generated execplan.
- It does not replace human review of loop intent or high-level design.
- It does not provide formal proofs of correctness.
- It does not maintain regression tests for the loop-generation skills themselves.
- It does not execute tests against production live systems without isolation.

## Users And Workflows

- **Operator agent / loop author**: Reads generated execplan contracts, answers clarifying questions about intended behavior, receives test-case guidance, authors or approves a test-case manifest, runs the evaluator, and reviews the verdict report.
- **Houmao developer / maintainer**: Extends evaluation contracts, assertion patterns, and bounded-run harnesses.
- **Reviewer**: Inspects the evaluation report to decide whether the loop is ready to start.

## Functional Requirements

- The skill must be invocable as a Houmao system skill, for example `$houmao-agent-loop-evaluator`.
- It must read a generated execplan package (`<loop-dir>/execplan/specs/`, `execplan/skills/`, `execplan/agents/`) and the loop intention.
- It must guide the operator agent through deriving test cases from loop contracts: objective, participant roles, state transitions, message/tick handling, and success/failure criteria.
- It must generate or scaffold a test-case manifest that lists scenarios, inputs, and expected outputs or invariants in a machine-readable format.
- It must run test cases against the loop in an isolated, bounded mode without requiring full live deployment.
- It must capture observable outputs: state transitions, messages, agent actions, artifacts, and errors.
- It must compare observed behavior against expected invariants and classify each test case as `pass`, `fail`, `degraded`, or `inconclusive`.
- It must produce a human-readable evaluation report with per-test evidence, failure explanations, and recommended fixes.
- It must integrate with `houmao-agent-loop-lite` and `houmao-agent-loop-pro` without changing their core contracts.

## System Boundaries

- In scope: test-case design guidance, manifest generation, bounded loop execution harness, assertion checking, report generation, and integration with existing loop execplan artifacts.
- Out of scope: formal verification, live production traffic, modification of loop skill internals, automatic repair of execplans, and coverage of runtime infrastructure outside the loop package.

## Operational Constraints

- Tests must be isolated from live mail, gateway, and managed-agent lifecycle state.
- Test reports must not expose secrets or auth material.
- The skill must work with the existing `<loop-dir>` structure produced by the loop skills.
- It must be usable in unattended headless runs as well as interactive operator sessions.
- It must respect provider rate limits and auth bundle boundaries.

## Assumptions

- A generated execplan exists and is structurally valid enough to inspect.
- Loop contracts contain sufficient information to derive expected behavior.
- The operator agent can answer clarifying questions about intended behavior.
- A deterministic replay or sandbox run mechanism exists or will be built.

## Open Questions

- Should the evaluator run the loop directly, or replay recorded traces?
- What assertion language should test cases use (declarative YAML/JSON, natural-language invariants, or code)?
- How do we handle non-deterministic LLM outputs in assertions?
- Should the skill generate tests automatically from contracts, or only guide the operator to author them?
- Where do test-case manifests and reports live relative to `<loop-dir>`?
